複雜度分析

時間與空間複雜度

複雜度分析是評估演算法效率的數學工具，幫助開發者在問題規模增長時預測程式的執行時間與記憶體消耗。透過複雜度分析，我們能夠在實際執行程式之前，就對不同演算法方案進行客觀比較，從而選出最適合特定場景的解法。這項能力是系統設計與效能優化的基礎。

核心概念

漸進分析（Asymptotic Analysis） 是複雜度分析的核心框架，關注的是當輸入規模 n 趨近於無窮大時，演算法資源消耗的增長趨勢，而非精確的執行步驟數。

Big-O 表示法（O-notation） 描述演算法的上界（最差情況）。例如 O(n²) 表示演算法的執行時間最多與 n² 成正比。這是最常用的表示法，因為它保證了效能的下限承諾。

Big-Omega 表示法（Ω-notation） 描述演算法的下界（最佳情況）。Ω(n log n) 表示基於比較的排序至少需要 n log n 次操作，這是理論上無法突破的極限。

Big-Theta 表示法（Θ-notation） 描述演算法的精確界（平均情況）。Θ(n) 表示演算法的執行時間恰好與 n 成正比，同時是上界和下界。

時間複雜度衡量演算法執行所需的基本操作次數。常見的複雜度等級從快到慢依次為：O(1) 常數時間、O(log n) 對數時間、O(n) 線性時間、O(n log n) 線性對數時間、O(n²) 平方時間、O(2ⁿ) 指數時間、O(n!) 階乘時間。

空間複雜度衡量演算法執行過程中所需的額外記憶體空間。需要注意區分「輸入空間」與「輔助空間」——通常我們只計算輔助空間。遞迴演算法的空間複雜度需要考慮呼叫堆疊的深度。

架構總覽

複雜度分析的方法論可分為以下層次：

逐行分析法：最基礎的方法，計算每行程式碼的執行次數，然後取主導項。對於巢狀迴圈，複雜度通常是各層迴圈次數的乘積。

遞迴關係求解：對於遞迴演算法，先建立遞迴方程 T(n)，再透過主定理（Master Theorem）、遞迴樹法或代入法求解。主定理處理形如 T(n) = aT(n/b) + f(n) 的標準分治遞迴。

攤銷分析（Amortized Analysis）：當單次操作的成本差異極大時，透過分析一系列操作的平均成本來獲得更精確的界。三種主要方法為：聚集法（Aggregate Method）計算 n 次操作的總成本再平均；記帳法（Accounting Method）為每次操作預付「信用」；勢能法（Potential Method）定義資料結構的勢能函數，追蹤狀態變化。

機率分析：對於隨機化演算法或隨機輸入，使用期望值分析。例如 Quick Sort 的期望時間為 O(n log n)，雖然最差情況為 O(n²)，但透過隨機化樞紐選擇，最差情況發生的機率極低。

空間與時間之間通常存在取捨（Space-Time Tradeoff）：雜湊表以 O(n) 額外空間換取 O(1) 查詢時間；動態規劃以 O(n) 或 O(n²) 空間儲存中間結果來避免指數級重複計算；位元運算技巧犧牲可讀性換取常數級空間優化。

應用場景

系統設計面試：在設計大規模系統時，複雜度分析決定了架構選擇。例如，需要支撐百萬級 QPS 的服務，O(n) 的請求處理在 n 很大時不可接受，必須設計 O(1) 或 O(log n) 的查詢路徑。

資料庫查詢優化：了解 B-Tree 索引提供 O(log n) 查詢、全表掃描是 O(n)，能幫助 DBA 判斷何時建立索引。JOIN 操作的不同實現方式（Nested Loop O(n²)、Hash Join O(n+m)、Merge Join O(n log n)）直接影響查詢效能。

演算法選型：在實際開發中，常數因子和快取效能也很重要。例如理論上 O(n log n) 的 Merge Sort 由於記憶體存取模式不佳，在實務上常被快取友好的 Quick Sort 超越。攤銷分析讓我們理解為何 ArrayList 的動態擴容操作平均仍是 O(1)。

容量規劃：預測系統在不同資料量下的表現。若目前處理 10 萬筆資料需要 1 秒，使用 O(n log n) 演算法處理 1 億筆約需 2000 秒，而 O(n²) 則需要 10⁶ 秒——這種差異決定了是否需要改變演算法策略或增加運算資源。

記憶體管理：空間複雜度分析幫助預防 OOM（Out of Memory）問題。在處理大規模資料時，O(n) 空間的演算法可能需要改為 O(1) 的串流處理方式，或使用外部排序等磁碟友好的演算法。
