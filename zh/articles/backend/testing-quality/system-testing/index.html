系統測試（System Testing）在完整的系統環境中驗證整體功能是否符合需求規格。與單元測試和整合測試不同，系統測試從使用者的角度出發，驗證系統作為一個整體的行為，包括功能正確性與各種非功能性需求。

核心概念

系統測試的範疇

測試類型驗證內容範例
功能測試業務需求正確性使用者能完成購買流程
非功能測試品質屬性效能、安全性、可用性
端對端測試完整使用者旅程從註冊到第一次下單
回歸測試既有功能未被破壞修 bug 後驗證相關功能
冒煙測試核心功能基本可用部署後的快速驗證

測試環境層級

環境目的資料穩定性要求
Development開發自測Mock/少量低
Integration元件整合驗證測試資料中
Staging上線前驗證類生產資料高
Production生產環境驗證真實資料最高

端對端測試的取捨

優勢劣勢
最高的信心程度執行速度最慢
驗證真實使用者流程維護成本最高
發現整合問題失敗原因定位困難
驗證部署配置環境依賴問題
覆蓋跨服務場景可能不穩定（Flaky）

冒煙測試 vs 全面回歸

維度冒煙測試全面回歸測試
範圍核心功能（5-10 個場景）所有功能
時間5-15 分鐘數小時
觸發時機每次部署定期/發版前
目的確認基本功能正常確認無回歸
自動化程度必須 100% 自動化通常 70-90%

非功能性測試類型

測試類型衡量指標工具執行頻率
效能測試回應時間、吞吐量k6、JMeter每次發版
安全測試漏洞數量、風險等級OWASP ZAP、Burp Suite每週/每版
可用性測試使用者完成任務成功率用戶研究重大改版
兼容性測試跨瀏覽器/裝置支援BrowserStack重大改版
可靠性測試MTBF、恢復時間Chaos Engineering定期
可存取性測試WCAG 合規axe、Lighthouse每次發版

應用場景

微服務系統的 E2E 測試策略

微服務架構下的端對端測試挑戰：

挑戰解決方案
環境複雜使用容器編排（Docker Compose/K8s）
服務依賴多非關鍵服務使用 Mock
資料一致性每次測試前重置到已知狀態
執行時間長平行執行、優先執行關鍵路徑
結果不穩定重試機制、等待策略最佳化

Flaky Test 的處理

不穩定測試的常見原因與對策：

原因症狀對策
時間依賴某些時間點失敗使用固定時間或 Mock
順序依賴平行執行時失敗確保測試獨立
非同步等待有時太快驗證使用智能等待
環境問題特定機器上失敗容器化測試環境
資源競爭高負載時失敗資源隔離

測試報告與可視化

有效的測試報告應包含：

  • 通過/失敗/跳過的數量與比例
  • 失敗測試的錯誤訊息與截圖
  • 執行時間趨勢
  • 按功能模組分組的結果
  • 與上次執行的差異比較
