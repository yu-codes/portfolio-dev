整合測試

整合測試（Integration Testing）驗證多個元件或模組組合後能否正確協作。不同於單元測試的隔離性，整合測試關注的是元件間的互動、資料流轉與系統邊界的正確性。它是發現介面不匹配、資料格式錯誤和配置問題的重要手段。

核心概念

整合測試的範圍

整合層級測試對象範例
模組整合同一服務內的模組互動Service → Repository → Database
服務整合服務間的 API 呼叫OrderService → PaymentService
資料整合資料儲存的讀寫Application → Redis/PostgreSQL
外部整合與第三方系統的互動Application → Email/SMS Provider
訊息整合非同步訊息的發送與消費Producer → Kafka → Consumer

整合測試策略

策略說明優點缺點
大爆炸（Big Bang）所有元件一起測試覆蓋完整問題定位困難
自頂向下從上層開始，下層用 Stub早期驗證主流程需建立 Stub
自底向上從底層開始，上層用 Driver底層品質保證晚期才驗證全流程
三明治（Sandwich）上下同時進行平衡複雜度高

測試環境的選擇

方式真實度速度成本適用場景
In-Memory（H2）低極快低快速驗證邏輯
Testcontainers高中中CI/CD 整合
共享測試環境高中中多服務整合
生產環境副本最高慢高最終驗證

Testcontainers 工作原理

Testcontainers 為每次測試啟動真實的容器化依賴：

  • 測試開始前自動拉取並啟動容器
  • 提供動態的連接資訊給應用程式
  • 測試結束後自動清理容器
  • 確保測試間隔離性

關鍵比較

單元測試 vs 整合測試

維度單元測試整合測試
範圍單一函式/類別多個元件互動
依賴Mock/Stub 隔離使用真實依賴
速度極快（< 10ms）中等（100ms-數秒）
定位問題精確到函式縮小到互動層
維護成本低中
信心程度個別正確性組合正確性
數量大量適量

資料庫整合測試策略

策略隔離方式優點缺點
Transaction Rollback每個測試在交易中執行後回滾快速、乾淨無法測試 commit 行為
重建 Schema每個測試重建資料庫結構完全隔離較慢
資料清理測試前/後清除資料接近真實清理邏輯複雜
資料庫容器每個測試啟用新容器完全隔離啟動時間長

應用場景

Repository 層整合測試

驗證資料存取層與資料庫的正確互動：

測試重點：

  • SQL 查詢語法正確性
  • ORM 映射是否正確
  • 交易管理行為
  • 並行操作的處理
  • 資料庫遷移後的兼容性

訊息佇列整合測試

驗證非同步訊息系統的正確性：

測試面向驗證內容注意事項
訊息發送格式正確、投遞到正確 Topic序列化格式驗證
訊息消費正確反序列化並處理冪等性驗證
錯誤處理Dead Letter Queue重試邏輯驗證
順序保證Partition 內的順序Key 分配策略
效能消費速度跟得上批次消費驗證

契約測試（Contract Testing）

用於驗證服務間 API 契約的一致性：

角色責任產出
Consumer定義期望的 API 行為Consumer Contract
Provider驗證自身滿足 ContractProvider Verification
Broker存儲和管理 Contract版本管理、CI 整合

契約測試的價值：

  • 無需同時啟動所有服務
  • 及早發現 API 變更導致的不兼容
  • 比端對端測試快速且穩定
