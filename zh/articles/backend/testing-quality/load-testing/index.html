負載測試從品質保證的角度驗證系統在預期和超出預期負載下的行為表現。作為測試品質的一環，負載測試不僅關注系統是否能承受壓力，更重視如何將效能驗證系統性地融入品質保證流程中。

核心概念

負載測試在品質保證中的定位

測試類型主要問題發現時機
單元測試這段邏輯正確嗎？開發時
整合測試元件能協作嗎？整合時
負載測試系統能承受壓力嗎？發版前
混沌測試系統能容錯嗎？持續

效能需求的定義

效能需求應該是明確、可測量的：

指標類型範例需求衡量方式
回應時間95% 請求 < 200msP95 延遲
吞吐量支撐 10,000 RPS峰值 RPS
並行數5,000 同時在線連線數
資源使用CPU < 70%、記憶體 < 80%監控指標
可用性錯誤率 < 0.1%失敗比例
恢復時間壓力解除後 30 秒內恢復恢復延遲

負載模型設計原則

真實的負載模型需考慮：

面向說明資料來源
使用者旅程不同操作的比例分配生產日誌分析
時間分布尖峰/離峰的流量差異歷史流量資料
資料分布熱點資料 vs 均勻分布存取模式分析
成長預測未來 3-6 個月的流量預期業務成長計劃

應用場景

CI/CD 中的效能品質門檻

將效能測試嵌入持續交付流程：

階段測試類型通過標準失敗動作
PR 合併前輕量基準測試無顯著退步阻擋合併
部署到 Staging完整負載測試滿足 SLA阻擋部署
部署到 Production冒煙效能驗證基本正常自動回滾
定期排程長時間浸泡測試無資源洩漏告警通知

效能基準追蹤

持續追蹤效能趨勢的重要性：

  • 及早發現效能退化（在影響使用者前）
  • 量化每次變更的效能影響
  • 為容量規劃提供數據依據
  • 建立團隊的效能意識

追蹤的核心指標：

  • 各 API 的 P50/P95/P99 延遲趨勢
  • 系統吞吐量的天花板變化
  • 資源使用效率（每 RPS 消耗的 CPU/Memory）

負載測試報告模板

一份有效的負載測試報告應包含：

章節內容
測試目標驗證什麼場景、達到什麼標準
測試環境硬體規格、網路拓撲、軟體版本
測試場景使用者行為模型、負載曲線
結果摘要關鍵指標的數值與是否通過
詳細數據延遲分布圖、吞吐量曲線、資源使用
瓶頸分析識別到的瓶頸與根因
建議優化建議與優先順序
