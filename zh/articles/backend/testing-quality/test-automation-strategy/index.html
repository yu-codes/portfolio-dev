測試自動化是持續交付的基石。有效的自動化策略不是追求 100% 自動化，而是在成本與收益之間找到最佳平衡點。本文探討如何制定適合團隊與專案的測試自動化策略。

核心概念

自動化的價值主張

面向手動測試自動化測試
執行速度慢（人工操作）快（秒到分鐘）
重複執行成本每次都相同首次高，後續近零
一致性人為疏忽風險每次執行完全一致
回歸測試耗時且乏味最適合自動化
探索性測試人類更擅長不適合
回饋速度天到週分鐘到小時

自動化適合度評估

因素適合自動化不適合自動化
執行頻率高頻重複執行一次性驗證
穩定性需求穩定需求頻繁變動
判斷力明確的通過/失敗標準需要主觀判斷
ROI長期維護的功能即將棄用的功能
複雜度可預測的步驟高度創造性操作

測試自動化金字塔的經濟學

層級開發成本維護成本執行速度信心度建議佔比
單元測試低低極快低70%
整合測試中中中中20%
E2E 測試高高慢高10%

關鍵比較

測試框架選擇考量

考量因素選項A選項B決策依據
語言一致性與產品同語言專用測試語言團隊技能、維護性
BDD vs 傳統Cucumber/BehavexUnit 風格是否需要非技術人員參與

Page Object vs Screenplay 模式

維度Page ObjectScreenplay
抽象層級頁面使用者行為
可讀性中等高（接近自然語言）
可重用性頁面內跨頁面
維護性頁面變更影響大行為穩定
學習曲線低中
適用場景中小型專案大型複雜專案

CI/CD 整合模式

模式觸發條件測試範圍目的
Pre-commit本地提交前相關單元測試快速回饋
PR 檢查Pull Request 建立單元+相關整合品質門檻
合併後合併到主幹全面自動化確保主幹健康
部署後新版本部署冒煙測試驗證部署正確
排程定時觸發長時間測試深度品質檢查

變數手動成本自動化成本

開發時間04 小時
維護（每月）030 分鐘
執行頻率每週 2 次每天 5 次

測試資料管理策略

策略適用場景優點缺點
工廠模式動態產生測試資料靈活、隔離需維護工廠程式碼
快照恢復複雜的預設狀態一致性好快照過時風險
API 建立透過 API 準備資料接近真實依賴服務可用
資料庫種子直接插入資料庫快速繞過業務邏輯

處理不穩定測試的策略

策略說明適用情況
隔離標記標記 flaky，從主流程移除暫時無法修復
自動重試失敗後自動重試 1-2 次環境偶發問題
根因修復分析並修復不穩定原因長期解決方案
監控追蹤追蹤 flaky rate 趨勢量化問題嚴重度
降級為手動若無法穩定則改為手動自動化 ROI 不足
