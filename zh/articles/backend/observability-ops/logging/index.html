日誌是軟體系統中最基本也最重要的可觀測性支柱之一。一個設計良好的日誌系統能讓開發者在分散式環境中快速定位問題、追蹤請求流程，並為告警與分析提供可靠的數據基礎。本文將深入探討現代日誌管理的核心概念、架構設計以及最佳實踐。

核心概念

結構化日誌 (Structured Logging)

傳統的純文字日誌難以被機器解析與查詢。結構化日誌以 JSON 或其他結構化格式記錄事件，使每條日誌都包含明確的欄位與數值，大幅提升可搜尋性與分析能力。

結構化日誌的關鍵要素包括：

  • 時間戳 (Timestamp)：精確到毫秒的事件發生時間，建議使用 UTC 時區
  • 日誌等級 (Level)：DEBUG、INFO、WARN、ERROR、FATAL 等分級
  • 訊息 (Message)：人類可讀的事件描述
  • 上下文欄位 (Context Fields)：請求 ID、使用者 ID、服務名稱等結構化元資料
  • 堆疊追蹤 (Stack Trace)：錯誤發生時的完整呼叫堆疊

日誌聚合 (Log Aggregation)

在微服務架構中，日誌分散在數十甚至上百個服務實例中。日誌聚合系統將所有來源的日誌集中收集、索引並提供統一的查詢介面。常見的聚合管線模式為 Agent → Buffer → Processor → Storage → Query Engine。

日誌採樣與保留策略

並非所有日誌都需要永久保存。高流量系統需要實作智慧採樣策略，在成本與可觀測性之間取得平衡。常見做法包括尾部採樣（只保留出錯的請求完整日誌）與基於規則的過濾。

架構總覽

現代日誌系統的典型架構由以下層次組成：

採集層 (Collection Layer)：透過 Sidecar 容器、DaemonSet 或應用內建 SDK 收集日誌。常見工具包括 Fluentd、Fluent Bit、Vector 和 Filebeat。

傳輸層 (Transport Layer)：使用訊息佇列（如 Kafka、NATS）作為緩衝，解耦生產者與消費者，防止下游系統過載時造成日誌遺失。

處理層 (Processing Layer)：進行日誌解析、欄位擷取、格式轉換、脫敏處理以及路由分發。

儲存層 (Storage Layer)：根據查詢需求選擇適當的儲存引擎。熱資料使用 Elasticsearch 或 ClickHouse 提供即時查詢；冷資料歸檔至物件儲存（如 S3）以降低成本。

查詢與視覺化層：提供統一的搜尋介面、儀表板和告警整合。Grafana Loki、Kibana、Datadog Logs 都是常見選擇。

ELK Stack（Elasticsearch + Logstash + Kibana） 提供強大的全文搜尋能力，但 Elasticsearch 叢集的運維成本高，且儲存空間需求大。

Grafana Loki 採用類似 Prometheus 的標籤索引策略，不對日誌內容建立全文索引，因此儲存成本極低，適合雲原生環境。

Datadog Logs 作為 SaaS 方案，免除了運維負擔，但在大規模場景下費用可能較高。

應用場景

場景一：微服務錯誤排查

當使用者回報某個 API 回傳 500 錯誤時，透過 Correlation ID 在日誌系統中搜尋，可以快速串聯請求經過的所有服務，定位根本原因。結構化日誌中的 trace_id 欄位使這種跨服務追蹤成為可能。

場景二：安全事件調查

日誌系統是安全團隊調查可疑活動的主要資料來源。透過分析登入失敗次數、異常存取模式或權限提升事件，可以及早發現潛在攻擊。此場景要求日誌具備防竄改保護和足夠的保留期限。

場景三：效能基準線建立

透過聚合與分析請求延遲、錯誤率等指標的日誌，團隊可以建立系統的正常行為基準線。當指標偏離基準線時，自動觸發告警。這是 SRE 實踐中「基於症狀的告警」的基礎。

場景四：合規與審計

金融、醫療等受監管行業需要完整的操作審計軌跡。日誌系統提供不可變的事件記錄，滿足 SOX、HIPAA、GDPR 等法規的審計要求。
