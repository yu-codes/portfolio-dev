分散式追蹤

請求追蹤與鏈路分析

在微服務架構中，一個使用者請求可能經過數十個服務的處理。當延遲升高或錯誤發生時，如何快速定位問題出在哪個服務？分散式追蹤（Distributed Tracing）正是為解決這個問題而生的技術。本文探討分散式追蹤的原理、架構設計及實踐要點。

核心概念

Trace、Span 與 Context

分散式追蹤的數據模型由三個核心元素構成：

Trace（追蹤）：代表一個完整的請求從進入系統到回應的端到端路徑。每個 Trace 擁有唯一的 Trace ID。

Span（跨度）：Trace 中的一個操作單元，代表一次服務呼叫或處理步驟。每個 Span 包含：

  • Span ID：唯一識別碼
  • Parent Span ID：父級 Span，建立因果關係
  • 操作名稱：描述此 Span 的動作
  • 開始與結束時間：精確測量延遲
  • Tags/Attributes：鍵值對形式的元資料
  • Events/Logs：Span 執行期間的事件紀錄
  • Status：成功或錯誤狀態

Context Propagation（上下文傳播）：在服務之間傳遞 Trace ID 和 Span ID 的機制。通常透過 HTTP Headers（如 traceparent）、gRPC Metadata 或訊息佇列的 Message Headers 傳遞。

採樣策略 (Sampling)

在高流量系統中，記錄每一個請求的追蹤資料既不經濟也不必要。採樣策略決定哪些請求需要被完整追蹤：

  • 頭部採樣 (Head-based Sampling)：在請求入口處決定是否追蹤。簡單但可能遺漏有價值的錯誤請求。
  • 尾部採樣 (Tail-based Sampling)：在請求完成後根據結果決定是否保留。可以確保所有錯誤和慢請求都被記錄，但實作複雜度高。
  • 自適應採樣 (Adaptive Sampling)：根據當前流量動態調整採樣率，在高峰期降低採樣以控制成本。

W3C Trace Context 標準

W3C Trace Context 是業界標準的追蹤上下文傳播格式，定義了兩個 HTTP Headers：

  • traceparent：包含版本、Trace ID、Parent Span ID 和採樣標記
  • tracestate：供各廠商攜帶額外的追蹤元資料

採用標準格式確保不同追蹤系統之間的互通性。

架構總覽

分散式追蹤系統的架構包含以下關鍵元件：

Instrumentation（儀器化）：在應用程式碼中插入追蹤邏輯。可分為自動儀器化（透過 Agent 或 Middleware 自動注入）和手動儀器化（開發者顯式建立 Span）。OpenTelemetry SDK 同時支援兩種方式。

Collector（收集器）：接收、處理和轉發追蹤資料的中間元件。OpenTelemetry Collector 支援多種接收器（OTLP、Jaeger、Zipkin 格式）、處理器（批次、採樣、屬性修改）和匯出器（多種後端）。

Storage Backend（儲存後端）：追蹤資料的持久化層。需要支援高寫入吞吐量和基於 Trace ID 的快速查詢。常見選擇包括 Elasticsearch、Cassandra、ClickHouse 和雲端託管服務。

Query & Visualization（查詢與視覺化）：提供追蹤搜尋、時序圖（Timeline View）、服務拓撲圖（Service Map）和延遲分析的使用者介面。

應用場景

場景一：延遲瓶頸定位

使用者反映某 API 回應變慢。透過追蹤系統查看該 API 的 Trace，發現一個下游資料庫查詢的 Span 佔據了 80% 的總延遲。進一步檢查該 Span 的 Tags，發現是未加索引的查詢導致全表掃描。

場景二：服務依賴關係發現

透過聚合追蹤資料，自動建立服務拓撲圖。這對新加入團隊的成員理解系統架構特別有價值，也能揭露未預期的服務依賴（如某服務透過共享資料庫間接耦合）。

場景三：跨團隊協作除錯

當問題涉及多個團隊負責的服務時，一條 Trace 提供了所有團隊都能理解的共同事實基礎。不再需要互相推測問題在對方的服務中——追蹤資料清楚顯示了延遲和錯誤的確切位置。

場景四：Canary 發布驗證

新版本以 Canary 方式發布後，比較新舊版本的追蹤延遲分佈。如果新版本的 P99 延遲顯著高於舊版本，可以在影響擴大前及時回滾。
