分散式追蹤

分散式追蹤的核心概念與實踐，涵蓋 Trace/Span 模型、上下文傳遞與 OpenTelemetry 標準

分散式追蹤（Distributed Tracing）記錄一個請求在分散式系統中穿越多個服務的完整路徑。它讓工程師能夠理解跨服務的因果關係、定位效能瓶頸，並分析故障的傳播路徑。在微服務架構中，分散式追蹤是不可或缺的可觀測性工具。

核心概念

Trace 與 Span

概念定義類比
Trace一個請求的完整生命週期記錄一次完整的旅程
SpanTrace 中的單一操作旅程中的一個站點
Parent Span發起呼叫的 Span呼叫者
Child Span被呼叫產生的 Span被呼叫者
Root SpanTrace 中的第一個 Span旅程的起點

Span 的核心屬性

屬性說明範例
Trace ID全域唯一，串聯整個請求abc123def456
Span ID當前操作的唯一標識span789
Parent Span ID父級 Spanspan456
Operation Name操作名稱GET /api/orders
Start/End Time開始和結束時間精確到微秒
Status操作結果OK/ERROR
Attributes附加的鍵值對http.method=GET
EventsSpan 中的事件異常、日誌

上下文傳遞（Context Propagation）

傳遞方式適用場景標頭格式
W3C Trace Context標準（推薦）traceparent, tracestate
B3 (Zipkin)Zipkin 生態X-B3-TraceId
JaegerJaeger 原生uber-trace-id

上下文傳遞確保跨服務的 Span 能正確關聯到同一個 Trace。

取樣策略

策略原理優點缺點
Head-based在入口決定是否取樣簡單、一致可能遺漏有趣的 Trace
Tail-based收集完整 Trace 後決定可保留異常 Trace需要更多資源
固定比例固定 N% 取樣簡單低流量時樣本不足
動態比例依負載調整平衡成本和覆蓋實作複雜
優先取樣錯誤/慢請求優先保留聚焦問題需要 Tail-based

OpenTelemetry 的角色

功能說明取代的工具
API統一的追蹤 API各家專用 SDK
SDK跨語言的實作Jaeger client、Zipkin SDK
Collector統一的收集器各後端的 Agent
協議(OTLP)標準傳輸協議各家專用協議
語意約定標準化的屬性名各家不同命名

追蹤 vs 日誌 vs 指標

維度追蹤日誌指標
資料模型樹狀 Span 結構離散事件數值時間序列
粒度請求級事件級聚合級
關聯性天然跨服務關聯需 trace_id 關聯標籤分組
成本中-高高低
適合發現延遲瓶頸、依賴問題具體錯誤詳情趨勢、異常

追蹤在除錯中的應用

典型的問題定位流程：

步驟動作使用工具
1告警顯示延遲上升指標+告警
2找到慢請求的 Trace追蹤系統按延遲排序
3查看 Span 樹，找到最慢的 SpanTrace 視覺化
4查看該 Span 的屬性和事件Span 詳情
5跳轉到對應的日誌日誌關聯
6確認根因並修復程式碼修改
