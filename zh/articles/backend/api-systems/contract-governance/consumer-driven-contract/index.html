消費者驅動契約

消費者驅動契約（Consumer-Driven Contracts, CDC）是一種由 API 消費者定義期望的測試方法論。它翻轉了傳統由提供者定義 API 的模式，確保 API 的變更不會破壞任何已知消費者的需求。本文深入探討 CDC 的設計理念與實作方式。

核心概念

傳統整合測試 vs 契約測試

特性端對端整合測試契約測試
測試範圍整個系統服務間介面
執行速度慢快
環境需求需要所有服務啟動獨立執行
維護成本高（環境不穩定）低
故障定位困難精確到介面
反饋速度慢（通常 CI 後期）快（單元測試級）
假陽性多（環境問題）少

消費者驅動契約的角色

角色描述職責
消費者 (Consumer)呼叫 API 的服務定義期望的互動
提供者 (Provider)提供 API 的服務驗證滿足所有消費者期望
契約 (Contract/Pact)消費者的期望集合可驗證的介面規格
Broker契約中介儲存與分享契約

CDC 的核心原則

原則描述效益
消費者定義需求消費者說明需要什麼避免過度設計
提供者驗證滿足提供者確認能滿足保證不破壞
獨立可驗證不需要對方在線快速反饋
契約即文件契約描述真實互動活的文件
最小化依賴只驗證用到的部分減少耦合

架構總覽

CDC 測試流程

步驟執行者動作產出
1消費者撰寫消費者測試Pact 檔案（契約）
2消費者發布契約到 Broker契約版本化儲存
3提供者從 Broker 取得契約最新消費者期望
4提供者對提供者執行契約驗證驗證結果
5Broker記錄驗證結果相容性矩陣
6CI/CD基於矩陣決定是否可部署部署決策

Pact 框架的互動模型

組件消費者端提供者端
測試對象消費者的 API 呼叫程式碼提供者的 API 實作
Mock/StubPact Mock Server真實服務
驗證方式發出請求、比對回應重放消費者請求、比對回應
產出Pact 檔案 (JSON)驗證報告

Pact 契約結構

欄位描述範例
consumer消費者名稱{“name”: “OrderService”}
provider提供者名稱{“name”: “UserService”}
interactions互動列表請求-回應對
interaction.description互動描述“取得使用者資料”
interaction.request期望的請求method, path, headers, body
interaction.response期望的回應status, headers, body
metadata元資料Pact 版本等

匹配規則 (Matchers)

匹配類型描述範例
精確匹配值必須完全相同“status”: “active”
型別匹配只要型別相同like(123) → 任何數字
正則匹配符合正則表達式regex("\d{4}-\d{2}")
陣列最小長度至少有 N 個元素eachLike({}, {min: 1})
包含匹配包含特定值includes(“error”)
日期格式符合日期格式dateFormatOf(“yyyy-MM-dd”)

工具語言支援特點適用場景

Provider-Driven vs Consumer-Driven

方式定義者優點缺點
Provider-Driven提供者定義 API提供者自主可能不符消費者需求
Consumer-Driven消費者定義期望保護消費者提供者需滿足多方
混合式雙方協商平衡溝通成本

CDC vs Schema 驗證 vs E2E 測試

維度CDCSchema 驗證E2E 測試
測試什麼實際互動行為格式正確性完整業務流程
速度快極快慢
環境不需對方不需對方需要所有服務
覆蓋面介面層格式層全面
假陽性少極少多
維護成本中低高
適合階段開發/CI設計/CI預發布

應用場景

微服務架構中的 CDC

場景CDC 的價值實作方式
服務拆分確保拆分不破壞下游消費者先定義期望
API 演化安全地修改 API驗證所有消費者仍通過
獨立部署確認可安全部署Can-I-Deploy 檢查
新消費者接入明確溝通需求消費者提交 Pact
提供者重構確認行為不變所有 Pact 仍通過

Can-I-Deploy 決策流程

步驟檢查結果
1消費者最新 Pact 是否已驗證已驗證/未驗證
2提供者是否通過所有消費者的 Pact全通過/有失敗
3相容性矩陣是否全綠可部署/不可部署

實作最佳實踐

實踐說明
最小化契約只包含實際使用的欄位
使用 Matcher避免過於嚴格的精確匹配
具名互動清楚描述每個互動的目的
Provider States定義提供者的前置狀態
版本標記使用 Git commit/tag 標記版本
Webhook 整合新 Pact 自動觸發提供者驗證
定期清理移除不再使用的消費者契約

常見挑戰與解決方案

挑戰描述解決方案
初始導入既有服務導入 CDC漸進式，從關鍵介面開始
團隊配合需要消費者團隊配合建立共同實踐
契約過於嚴格小改變就失敗善用 Matcher
Provider States 管理狀態設定複雜共用測試資料工廠
非同步通訊訊息佇列的契約Message Pact
GraphQL 服務動態查詢的契約針對特定查詢建立契約
