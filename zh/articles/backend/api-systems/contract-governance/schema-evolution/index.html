Schema 演化策略

深入探討 API 與資料庫 Schema 的演化策略，確保系統在持續變更中保持穩定

Schema 演化是軟體系統長期運營中不可避免的挑戰。無論是 API 回應格式、資料庫結構還是訊息格式，都需要在不中斷服務的情況下安全地演進。本文深入探討各種 Schema 演化策略與最佳實踐。

核心概念

Schema 演化的類型

層面範例影響範圍
API SchemaJSON 回應結構變更所有 API 客戶端
資料庫 Schema表結構變更應用程式 + 資料
訊息 Schema事件格式變更所有消費者
設定 Schema設定檔格式變更部署流程
序列化格式Protobuf/Avro 變更序列化/反序列化

Schema 變更分類

變更類型描述相容性範例
新增選填欄位加入新的非必要欄位✅ 向後相容加入 nickname 欄位
移除欄位刪除既有欄位❌ 破壞性移除 phone 欄位
重新命名改變欄位名稱❌ 破壞性name → full_name
型別變更改變欄位型別❌ 破壞性string → number
新增必填欄位加入必要欄位❌ 破壞性要求提供 email
改變預設值修改預設行為⚠️ 可能破壞預設 limit 改變
嵌套結構變更改變巢狀結構❌ 破壞性扁平化嵌套

相容性的維度

相容性定義保護對象
向後相容 (Backward)新 Schema 可讀舊資料消費者
向前相容 (Forward)舊 Schema 可讀新資料生產者
完全相容 (Full)同時向前向後相容所有方
無相容不保證任何相容性無

架構總覽

資料庫 Schema 遷移策略

策略方法風險停機時間
直接 ALTER直接修改表結構高（鎖表）可能很長
擴展-遷移-收縮新增→資料遷移→移除舊低零
影子表建新表、同步資料、切換中極短
雙寫同時寫新舊中零
版本表每版本一張表低零

擴展-遷移-收縮模式 (Expand-Migrate-Contract)

階段動作狀態
1. 擴展 (Expand)新增欄位/表，不移除舊的新舊共存
2. 遷移 (Migrate)將舊資料填入新結構資料同步
3. 過渡期應用程式讀寫新結構驗證新結構
4. 收縮 (Contract)移除舊欄位/表清理完成

訊息序列化格式的演化能力

格式向後相容向前相容Schema Registry演化規則
Protocol Buffers✅ (欄位編號)✅ (忽略未知)可選不改編號、不改型別
Avro✅ (Schema 解析)✅Confluent SR有明確規則
JSON⚠️ (需手動)⚠️ (需手動)無標準無強制
Thrift✅ (欄位 ID)✅無類似 Protobuf
FlatBuffers✅⚠️無欄位追加

關鍵比較

Protobuf Schema 演化規則

操作允許條件
新增欄位✅使用新的欄位編號
移除欄位✅使用 reserved 標記
重新命名欄位✅不影響（靠編號）
改變型別⚠️僅相容型別 (int32→int64)
改變欄位編號❌會破壞相容性
改 required→optional✅ (proto3)proto3 全部 optional
新增 enum 值✅舊版本會看到未知值

Avro Schema 演化規則

操作向後相容向前相容完全相容
新增有預設值的欄位✅✅✅
新增無預設值的欄位✅❌❌
移除有預設值的欄位❌✅❌
移除無預設值的欄位❌✅❌

應用場景

零停機 Schema 遷移步驟

步驟動作風險控制
1新增新欄位（nullable/有預設值）不影響現有功能
2部署寫入新欄位的程式碼讀取仍用舊欄位
3背景遷移既有資料批次處理避免鎖
4部署讀取新欄位的程式碼驗證新欄位資料正確
5移除舊欄位的讀寫邏輯確認無依賴
6移除舊欄位最後清理

大型表的 Schema 變更策略

表大小建議策略原因
< 100MB直接 ALTER足夠快，影響小
100MB - 10GBpt-online-schema-change避免長時間鎖表
> 10GBgh-ost / 影子表無觸發器、可暫停
分片表逐片遷移減少影響範圍

Schema Registry 的使用

功能描述效益
註冊 Schema新版本 Schema 註冊集中管理
相容性檢查自動驗證相容性防止破壞性變更
Schema 解析讀取時取得正確 Schema版本正確對應
版本歷史追蹤所有版本審計與回溯

演化策略的選擇

場景推薦策略理由
公開 API版本化 + 棄用期保護外部客戶端
內部 API擴展-收縮可控制所有客戶端
事件串流Schema Registry + Avro自動相容性驗證
資料庫 (小)遷移腳本簡單直接
資料庫 (大)線上 DDL 工具零停機
微服務間Protobuf + 規則編譯期保證
