JWT 與 Session 認證

認證（Authentication）是確認使用者身份的過程。JWT（JSON Web Token）與 Session 是兩種最主流的認證機制，它們代表了無狀態與有狀態兩種不同的設計哲學。本文深入比較這兩種機制的設計原理、安全性與適用場景。

核心概念

認證 vs 授權

概念英文描述問題
認證 (Authentication)AuthN你是誰？身份確認
授權 (Authorization)AuthZ你能做什麼？權限確認

Session 認證流程

步驟客戶端伺服器
1發送帳密驗證帳密
2-建立 Session，存入記憶體/DB
3收到 Set-Cookie: session_id=xxx回傳 Session ID
4後續請求帶 Cookie查找 Session 驗證身份
5登出銷毀 Session

JWT 認證流程

步驟客戶端伺服器
1發送帳密驗證帳密
2-生成 JWT（簽名）
3儲存 JWT回傳 JWT
4Authorization: Bearer 驗證簽名、解析內容
5丟棄 Token無狀態（不需操作）

JWT 的結構

部分內容範例
Header演算法、類型{“alg”:“HS256”,“typ”:“JWT”}
Payload聲明（Claims）{“sub”:“123”,“exp”:1234567890}
Signature簽名驗證HMACSHA256(header.payload, secret)

常見 Claims：

Claim名稱用途
subSubject使用者 ID
expExpiration過期時間
iatIssued At簽發時間
issIssuer簽發者
audAudience接收方
jtiJWT ID唯一識別

JWT 簽名演算法

演算法類型安全性適用場景
HS256對稱 (HMAC)中單一服務
HS384對稱 (HMAC)中高單一服務
RS256非對稱 (RSA)高多服務驗證
RS512非對稱 (RSA)極高高安全需求
ES256非對稱 (ECDSA)高高效能 + 安全
EdDSA非對稱 (Ed25519)極高現代推薦

Session 儲存方案

方案效能持久性擴展性適用場景
記憶體 (In-memory)極高無差（單機）開發環境
Redis高可配置好生產環境主流
資料庫中高中需持久化
Cookie (加密)高隨客戶端好簡單場景
Memcached高無好大規模快取

Token 刷新機制

組件用途生命週期儲存位置
Access Token存取資源短 (15min-1h)記憶體/Cookie
Refresh Token取得新 Access Token長 (7-30天)HttpOnly Cookie

JWT vs Session 全面比較

特性JWTSession
狀態無狀態有狀態
儲存位置客戶端伺服器
擴展性天然分散式需共享 Session Store
撤銷能力困難（需黑名單）容易（刪除 Session）
效能每次解碼驗證一次查找
大小較大（含 Payload）小（僅 Session ID）

安全風險Token 洩漏無法即時撤銷Session 劫持

微服務✅ 適合❌ 需要集中式 Store

選擇指南

場景推薦方案理由
傳統 Web 應用Session簡單、可控
SPA + APIJWT (HttpOnly Cookie)跨域、無狀態
行動 AppJWT無 Cookie 環境
微服務JWT服務間無需共享狀態
需要即時撤銷Session 或 JWT + 黑名單可立即失效

高安全需求Session + 多因子伺服器端完全控制

JWT 最佳實踐

實踐說明
短過期時間Access Token 15 分鐘
使用 Refresh Token長期登入不需長效 Access Token
不存敏感資料Payload 可被解碼（Base64）
使用強演算法RS256 或 ES256
驗證所有 Claimsexp, iss, aud 都要檢查
使用黑名單關鍵場景支援即時撤銷
HttpOnly Cookie防 XSS 讀取

Session 最佳實踐

實踐說明
使用安全的 Session ID足夠長度的隨機值
設定合理的逾時閒置逾時 + 絕對逾時
登入後換 ID防 Session Fixation
HTTPS OnlyCookie 設定 Secure 旗標
Redis 儲存效能好、支援 TTL
分散式一致多節點共享 Session Store
