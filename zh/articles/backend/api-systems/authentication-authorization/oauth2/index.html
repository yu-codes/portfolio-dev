OAuth 2.0 授權框架

OAuth 2.0 是目前最廣泛使用的授權框架，允許第三方應用在不取得使用者密碼的情況下存取使用者的資源。本文深入解析 OAuth 2.0 的核心概念、各種授權流程與安全最佳實踐。

核心概念

OAuth 2.0 的角色

角色描述範例
Resource Owner資源擁有者（使用者）使用者本人
Client第三方應用程式需要存取使用者資料的 App
Authorization Server授權伺服器Google Auth, Auth0
Resource Server資源伺服器Google API, GitHub API

OAuth 2.0 vs OAuth 1.0 vs OIDC

特性OAuth 1.0OAuth 2.0OpenID Connect
目的授權授權認證 + 授權
複雜度高（簽名）中中
簽名每次請求需簽名不需要（靠 TLS）不需要
Token 類型唯一Bearer, MAC 等ID Token + Access Token
行動支援差好好
標準化完整框架（較鬆散）完整

Token 類型

Token用途格式生命週期
Authorization Code交換 Token短隨機字串極短（分鐘）
Access Token存取資源JWT 或 opaque短（小時）
Refresh Token更新 Access Tokenopaque長（天/週）
ID Token (OIDC)攜帶使用者資訊JWT短

架構總覽

授權碼流程 (Authorization Code)

步驟動作說明
1Client 導向 Auth Server帶 client_id, redirect_uri, scope
2使用者登入並同意在 Auth Server 上操作
3Auth Server 重導回 Client帶 authorization_code
4Client 用 code 換 token後端 → Auth Server (帶 client_secret)
5Auth Server 回傳 tokensaccess_token + refresh_token
6Client 用 access_token 存取資源Authorization: Bearer token

Grant Types 比較

Grant Type適用場景安全性是否推薦
Authorization CodeWeb 後端應用高✅ 推薦
Authorization Code + PKCESPA、行動 App高✅ 強烈推薦
ImplicitSPA (舊式)低❌ 已棄用
Client Credentials服務對服務高✅ 推薦
Resource Owner Password信任的第一方 App低❌ 不推薦
Device CodeIoT、智慧電視中✅ 特定場景
Refresh Token更新 Access Token中高✅ 搭配使用

PKCE (Proof Key for Code Exchange)

步驟動作說明
1生成 code_verifier隨機字串 (43-128 字元)
2計算 code_challengeSHA256(code_verifier) → Base64URL
3授權請求帶 challenge?code_challenge=xxx&code_challenge_method=S256
4Token 請求帶 verifiercode_verifier=原始值
5Auth Server 驗證SHA256(verifier) == challenge

範疇範例用途

實作最佳實踐

實踐說明
永遠使用 HTTPS防止 token 被竊聽
使用 PKCE即使有後端也建議使用
驗證 state 參數防止 CSRF 攻擊
最小權限 scope只請求必要的權限
Token 安全儲存HttpOnly Cookie 或安全儲存
Refresh Token 輪替每次使用後更換
嚴格的 redirect_uri不使用萬用字元
定期輪替密鑰簽名密鑰定期更換
