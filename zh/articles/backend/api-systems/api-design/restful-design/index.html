RESTful API 設計原則

深入理解 RESTful API 的設計哲學、最佳實踐與常見反模式，構建一致且易用的 API

RESTful API 是目前最廣泛使用的 Web API 設計風格。一個好的 REST API 應該直覺、一致且易於使用。本文深入探討 REST 的設計原則、URL 設計規範、回應格式與常見的設計反模式。

核心概念

REST 的六大架構約束

約束描述效果
客戶端-伺服器分離前後端獨立演化職責分離
無狀態 (Stateless)每次請求包含所有資訊可擴展性
可快取 (Cacheable)回應可被快取效能提升
統一介面 (Uniform Interface)一致的資源操作方式簡化架構
分層系統 (Layered System)客戶端不知道連接的是哪層中間層可加入
按需代碼 (Code on Demand)可選，伺服器傳送可執行程式碼擴展客戶端功能

HTTP 方法語意

方法操作安全冪等請求體典型用途
GET讀取✅✅無取得資源
POST建立❌❌有建立新資源
PUT全量替換❌✅有更新整個資源
PATCH部分更新❌❌*有更新部分欄位
DELETE刪除❌✅無/有刪除資源
HEAD取得標頭✅✅無檢查資源存在
OPTIONS取得支援方法✅✅無CORS 預檢

HTTP 狀態碼分類

範圍類別常用碼含義
2xx成功200 OK請求成功
201 Created資源已建立
204 No Content成功但無內容
3xx重新導向301 Moved永久移動
304 Not Modified未修改（快取）
4xx客戶端錯誤400 Bad Request請求格式錯誤
401 Unauthorized未認證
403 Forbidden無權限
404 Not Found資源不存在
409 Conflict衝突（如版本）
429 Too Many Requests限流
5xx伺服器錯誤500 Internal Error伺服器內部錯誤
502 Bad Gateway閘道錯誤
503 Service Unavailable服務不可用

架構總覽

URL 設計規範

規則好的設計不好的設計
使用名詞，非動詞/users/getUsers
使用複數/articles/article
層級表示從屬關係/users/123/orders/getUserOrders?id=123
使用 kebab-case/user-profiles/userProfiles
不超過 3 層嵌套/shops/123/products/shops/123/shelves/456/products/789
使用查詢參數篩選/users?role=admin/admin-users

資源命名模式

模式URL方法說明
列表GET /usersGET取得集合
詳情GET /users/123GET取得單個資源
建立POST /usersPOST新建資源
全量更新PUT /users/123PUT替換資源
部分更新PATCH /users/123PATCH修改部分
刪除DELETE /users/123DELETE移除資源
子資源GET /users/123/ordersGET從屬資源
動作POST /users/123/activatePOST非 CRUD 操作

回應格式設計

{
  "data": {
    "id": "123",
    "type": "user",
    "attributes": { ... }
  },
  "meta": {
    "total": 100,
    "page": 1,
    "per_page": 20
  },
  "links": {
    "self": "/users?page=1",
    "next": "/users?page=2"
  }
}

錯誤回應格式：

欄位描述必要性
error.code機器可讀的錯誤碼必要
error.message人類可讀的描述必要
error.details詳細錯誤資訊選填
error.target錯誤相關的欄位選填

關鍵比較

分頁策略

策略方式優點缺點
Offset-based?page=2&size=20簡單、可跳頁大偏移效能差
Cursor-based?cursor=abc123&size=20效能穩定不可跳頁
Keyset?after_id=100&size=20效能好需要唯一排序鍵
Time-based?since=2024-01-01適合時間序列需要時間欄位

篩選與排序

功能方式範例
篩選查詢參數?status=active&role=admin
排序sort 參數?sort=-created_at,name
欄位選擇fields 參數?fields=id,name,email
搜尋q 或 search 參數?q=keyword
嵌入關聯include/expand?include=orders,profile

應用場景

API 設計檢查清單

類別檢查項目
URL 設計使用名詞、複數、層級合理
HTTP 方法正確對應 CRUD 操作
狀態碼精確反映操作結果
錯誤處理統一格式、資訊充足
分頁大列表必須分頁
版本控制有版本策略
安全HTTPS、認證、授權
文件OpenAPI 規範

常見反模式

反模式問題正確做法
動詞 URL/getUserGET /users/:id
深度嵌套/a/1/b/2/c/3/d扁平化 + 查詢參數
忽略狀態碼永遠回 200使用語意化狀態碼
無分頁回傳所有資料預設分頁
敏感資訊在 URL/users?token=xxx放在標頭
不一致的命名混用 camelCase 與 snake_case統一一種風格
