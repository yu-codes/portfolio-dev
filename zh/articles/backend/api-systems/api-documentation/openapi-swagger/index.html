OpenAPI 與 Swagger

OpenAPI 規範（前身為 Swagger Specification）是描述 RESTful API 的業界標準。它提供了一種機器可讀的格式來定義 API 的結構、請求/回應格式與認證方式，使得 API 的文件化、測試與程式碼生成都能自動化。本文深入探討 OpenAPI 的設計理念與實際應用。

架構總覽

Path 定義的關鍵元素

元素描述範例
summary簡短描述“取得使用者列表”
description詳細說明Markdown 格式長文
operationId操作唯一識別listUsers
parameters路徑/查詢/標頭參數page, limit, filter
requestBody請求體定義JSON Schema
responses回應定義200, 400, 404, 500
security安全要求Bearer Token
tags分類標籤[“Users”]

Component 可復用元件

元件類型用途使用方式
schemas資料模型定義$ref: ‘#/components/schemas/User’
responses通用回應$ref: ‘#/components/responses/NotFound’
parameters通用參數$ref: ‘#/components/parameters/PageParam’
requestBodies通用請求體$ref: ‘#/components/requestBodies/UserInput’
securitySchemes認證方式bearerAuth, apiKey, oauth2
headers通用標頭X-Rate-Limit
examples範例資料完整的請求/回應範例

關鍵比較

Code First vs Spec First

方式描述優點缺點
Code First從程式碼生成規範程式碼即真相可能生成不整潔的規範
Spec First先寫規範再實作設計優先、一致性好規範與實作可能不同步
混合式設計先行、註解補充平衡工具鏈複雜

最佳實踐

實踐說明
使用 $ref 避免重複共用的 Schema、Response 抽到 components
提供豐富的 examples每個操作都有請求/回應範例
定義所有錯誤回應不只定義成功情況
使用 tags 分類讓文件有清晰的結構
語意化 operationId方便程式碼生成使用
維護 CHANGELOG追蹤規範的變更歷史
CI 中驗證每次提交都驗證規範正確性
