契約優先設計

契約優先設計（Contract-First Design）是一種在實作 API 之前先定義 API 契約（規範）的開發方法論。這種方法確保前後端團隊能並行開發、減少整合問題，並提供一致性的 API 體驗。本文深入探討契約優先設計的原則與實踐。

核心概念

契約優先 vs 程式碼優先

特性契約優先 (Contract-First)程式碼優先 (Code-First)
開發順序先寫規範，再實作先寫程式碼，再生規範
設計品質高（深思熟慮）中（可能有技術偏見）
前後端並行✅ 立即可用 Mock❌ 需等後端完成
一致性高（單一來源）中（可能偏離）
重構成本低（早期發現問題）高（晚期才發現）
適合場景多團隊、公開 API原型、內部工具
學習曲線中（需學規範語言）低（直接寫程式碼）

契約優先的核心原則

原則描述效益
設計先行API 設計在實作之前更好的設計決策
單一真相來源規範文件是唯一的定義避免不一致
消費者導向從使用者需求出發更好的開發體驗
可驗證實作必須符合規範持續合規
可演化規範支持版本演進長期維護性

契約的組成要素

要素描述範例
端點定義URL 路徑與方法GET /users/{id}
請求格式參數與請求體Query params, JSON body
回應格式回應結構與狀態碼200: User, 404: Error
認證方式安全需求Bearer Token
錯誤格式統一的錯誤回應{code, message, details}
約束條件欄位驗證規則min, max, pattern
範例請求/回應範例完整的 JSON 範例

架構總覽

契約優先開發工作流

階段活動參與者產出
1. 需求分析確定 API 需求產品 + 前後端需求文件
2. 契約設計撰寫 OpenAPI 規範API 設計師OpenAPI YAML
3. 契約審查團隊審查規範全團隊修訂後的規範
4. Mock 生成從規範生成 Mock自動化Mock Server
5. 並行開發前端用 Mock、後端實作前後端分別程式碼
6. 合約測試驗證實作符合規範CI/CD測試結果
7. 整合測試前後端整合QA整合報告

設計審查的檢查項目

類別檢查項目重要性
一致性命名風格是否統一高
完整性所有錯誤情況都有定義高
可用性是否符合消費者需求極高
安全性認證/授權是否完整高
效能是否有適當的分頁/篩選中
向後相容是否會破壞現有客戶端高
文件品質描述和範例是否充足中

契約驗證的層級

層級驗證內容工具
語法層規範文件語法正確YAML/JSON 解析器
結構層符合 OpenAPI 規範Spectral, swagger-cli
語意層設計規則（命名、格式）自定義 Linting 規則
實作層程式碼符合規範Dredd, Schemathesis
相容層新版本相容舊版本openapi-diff

微服務中的契約管理

模式描述適用場景
集中式規範統一管理所有服務的規範小型微服務
去中心化每個服務管理自己的規範大型組織
API Gateway 整合閘道匯集各服務規範統一入口
Schema Registry中央 Schema 註冊表事件驅動

契約優先的最佳實踐

實踐說明
使用設計系統統一的命名、格式、錯誤處理規範
自動化驗證CI 中驗證實作符合規範
版本管理規範規範文件與程式碼一起版控
Mock 先行規範完成後立即提供 Mock
利益相關者參與設計階段就讓前端/行動參與
漸進式採用新 API 先採用，舊 API 逐步遷移
文件即規範規範本身就是最佳文件

常見反模式

反模式問題解決方式
規範過時實作改了但規範沒更新合約測試 CI
複製貼上不用 $ref，到處重複提取 components
過度設計初版就想設計完美 API迭代式設計
忽略錯誤情況只定義 happy path規範 Lint 規則
無範例規範沒有 examples必須包含範例
