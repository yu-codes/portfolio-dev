Docker 容器化

Docker 容器化的核心概念與最佳實踐，涵蓋映像建置、多階段構建與生產環境最佳化

Docker 容器化技術徹底改變了軟體的打包與部署方式。透過將應用程式及其所有依賴打包成標準化的容器映像，Docker 消除了「在我的機器上可以跑」的問題，實現了開發、測試、生產環境的一致性。

核心概念

容器 vs 虛擬機器

維度容器虛擬機器（VM）
隔離層級程序層級（共享核心）硬體層級（獨立核心）
啟動時間秒級分鐘級
資源佔用MB 級GB 級
效能接近原生有虛擬化開銷
安全隔離較弱強
密度同一主機可跑數百個容器通常 10-20 個 VM
可攜性極高中

Docker 映像層概念

Docker 映像是由多個唯讀層疊加而成：

層內容快取效果
基礎層作業系統基礎（alpine、debian）高度共享
依賴層套件管理器安裝的系統依賴較少變更
應用依賴層npm/pip/maven 安裝的套件偶爾變更
應用程式碼層實際的應用程式碼頻繁變更

Dockerfile 指令

指令用途注意事項
FROM指定基礎映像優先使用官方映像
RUN執行命令（產生新層）合併多個 RUN 減少層數
COPY複製檔案到映像善用 .dockerignore
WORKDIR設定工作目錄避免使用 cd
ENV設定環境變數區分建置時和執行時
EXPOSE宣告端口（文件用途）不等於開放端口
ENTRYPOINT容器的主程序不可被覆蓋
CMD預設命令/參數可被 docker run 覆蓋

架構總覽

多階段建置（Multi-stage Build）

# 階段 1：建置
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# 階段 2：執行
FROM node:20-alpine AS runner
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
USER node
CMD ["node", "dist/main.js"]

多階段建置的優勢：

  • 最終映像不包含建置工具（如 TypeScript 編譯器）
  • 映像大小顯著減少
  • 攻擊面減少

映像大小優化策略

策略效果範例
Alpine 基礎映像從 ~900MB 降到 ~50MBnode:20-alpine
多階段建置去除建置依賴只複製產出和執行依賴
.dockerignore排除不必要檔案node_modules、.git
合併 RUN 指令減少層數和中間快取RUN apt-get update && apt-get install && rm -rf
使用 distroless極致精簡gcr.io/distroless/nodejs

關鍵比較

基礎映像選擇

映像大小包含內容適用場景
ubuntu/debian~100MB完整套件管理需要系統工具
alpine~5MB最小 Linux通用推薦
slim 變體~50MB精簡版 Debian平衡大小與兼容
distroless~20MB僅語言運行時生產環境
scratch0MB完全空白Go 靜態二進位

容器安全最佳實踐

實踐目的方法
非 root 執行減少權限USER 1001
唯讀檔案系統防止篡改--read-only
映像掃描發現漏洞Trivy、Snyk Container
簽名驗證確保完整性cosign、Docker Content Trust
最小權限減少攻擊面不安裝不需要的套件
固定標籤避免意外更新使用 SHA256 digest

Docker Compose vs Kubernetes

維度Docker ComposeKubernetes
適用規模單機開發/小型部署大規模生產
學習曲線低高
服務發現內建 DNSService + Ingress
自動修復基本 restart完整健康檢查
擴展手動 scale自動水平擴展
配置管理環境變數/檔案ConfigMap/Secret

應用場景

開發環境容器化

使用 Docker Compose 統一開發環境：

  • 資料庫：PostgreSQL、Redis、MongoDB
  • 訊息佇列：Kafka、RabbitMQ
  • 監控：Prometheus、Grafana
  • 應用服務：熱重載的開發容器

優勢：新成員 docker compose up 即可開始開發。

生產環境的健康檢查

檢查類型用途配置
HEALTHCHECKDocker 內建健康檢查Dockerfile 中定義
Liveness判斷容器是否需要重啟HTTP/TCP/Command
Readiness判斷容器是否可接收流量HTTP endpoint
Startup等待應用啟動完成長啟動時間的應用

映像版本管理策略

策略格式優點缺點
Git SHAapp:abc1234精確對應程式碼不易讀
SemVerapp:1.2.3語意化、清晰需維護版本
日期app:20250101簡單不對應程式碼
latestapp:latest方便不可預測（禁止生產使用）
