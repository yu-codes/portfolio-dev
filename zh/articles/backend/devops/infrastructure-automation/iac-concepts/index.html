IaC 概念

基礎設施即程式碼（Infrastructure as Code, IaC）是以程式碼的方式定義和管理 IT 基礎設施的實踐。透過將基礎設施配置寫成可版本控制、可測試、可重複執行的程式碼，IaC 消除了手動配置的不一致性和不可重複性問題。

核心概念

IaC 的核心原則

原則說明價值
版本控制所有配置存放在 Git變更追蹤、審計、回滾
可重複性相同程式碼產生相同結果消除環境差異
冪等性重複執行不會產生副作用安全地重新套用
自文件化程式碼本身描述基礎設施不需額外文件
可測試性可以驗證配置的正確性部署前發現問題

宣告式 vs 命令式

方式描述方式範例工具
宣告式描述「目標狀態是什麼」“我需要 3 台 VM”Terraform、CloudFormation
命令式描述「如何達到目標」“建立第一台 VM, 建立第二台…”Ansible playbook、Shell script
維度宣告式命令式
冪等性天然冪等需自行確保
可讀性描述目標，直觀描述步驟，較長
漂移修復自動修正到期望狀態需額外邏輯
複雜邏輯較難表達靈活
學習曲線需學習 DSL類似腳本

可變 vs 不可變基礎設施

模式更新方式優點缺點
可變（Mutable）在現有伺服器上修改快速更新漂移、雪花伺服器
不可變（Immutable）建新的、砍舊的一致性高部署時間較長

不可變基礎設施確保每台伺服器都是從相同的映像建立，消除了「設定漂移」的問題。

IaC 的分層管理

層級內容變更頻率管理者
基礎層網路、VPC、IAM極少平台團隊
平台層K8s 叢集、資料庫、快取少平台團隊
應用層服務部署、配置頻繁開發團隊
環境層環境特定配置中DevOps

狀態管理

IaC 工具需要追蹤資源的當前狀態：

方式工具優點風險
狀態檔案Terraform精確追蹤狀態檔案需安全管理
API 查詢CloudFormation無需本地狀態受限於 API 能力
代理式Puppet/Chef持續監控需安裝代理

IaC 工具生態比較

工具類型語言雲端支援狀態管理
Terraform配置工具HCL多雲狀態檔案
Pulumi配置工具程式語言多雲狀態後端
CloudFormation配置工具JSON/YAMLAWS onlyAWS 管理
Ansible配置+部署YAML多平台無狀態

Crossplane配置工具YAML/CRD多雲Kubernetes

Terraform vs Pulumi

維度TerraformPulumi
語言HCL（專用DSL）Python/TS/Go/C#
學習曲線需學 HCL用熟悉的語言
IDE 支援中等完整（程式語言）
測試有限程式語言原生測試
複雜邏輯HCL 較受限完整程式語言能力
社群非常大成長中
Provider極豐富可用 Terraform Provider

IaC 測試策略

測試層級目的工具
靜態分析語法、格式、安全規則tflint、checkov
單元測試驗證模組邏輯terratest、pytest
整合測試在真實環境中驗證terratest
合規檢查確保符合組織策略OPA、Sentinel
Plan 審查預覽變更影響terraform plan

應用場景

多環境管理

使用 IaC 管理多個環境的策略：

策略方式優點缺點
目錄分離每個環境一個目錄完全獨立配置重複
WorkspaceTerraform workspace輕量切換狀態共享風險
變數檔案共用程式碼 + 環境變數DRY需小心管理變數
模組化共用模組 + 環境調用最佳實踐初期設計成本

漂移偵測與修復

配置漂移的處理流程：

  1. 偵測：定期執行 terraform plan 檢查差異
  2. 分類：區分有意的修改和意外的漂移
  3. 修復：
    • 如果程式碼正確 → terraform apply 修復實際環境
    • 如果實際環境正確 → 更新程式碼或 terraform import
  4. 預防：限制手動修改的權限

IaC 的安全考量

考量風險對策
狀態檔案中的敏感資料密碼、Token 明文存在加密狀態後端、限制存取
Provider 憑證高權限帳號暴露環境變數、短期憑證
程式碼中的密文版本控制中可見使用 Vault/KMS 引用
過度權限IaC 帳號權限太大最小權限、分離角色
