基礎設施自動化

IaC 與組態管理

基礎設施自動化（Infrastructure Automation）是將基礎設施的配置、部署與管理從手動操作轉變為可程式化、可版本控制的自動化流程。Infrastructure as Code（IaC）讓基礎設施定義像應用程式碼一樣被審查、測試與部署，而 GitOps 則將 Git 倉庫作為基礎設施狀態的唯一真實來源。本篇將介紹 Terraform、Ansible 與 GitOps 三大核心實踐及其整合運用。

核心概念

基礎設施自動化的三大支柱為：Terraform（基礎設施配置）、Ansible（組態管理）、以及 GitOps（交付流程）。

Terraform 是 HashiCorp 出品的 IaC 工具，使用聲明式語言（HCL）描述期望的基礎設施狀態。使用者定義「我要什麼」，Terraform 負責計算「如何達成」。它維護一個 State File 記錄當前基礎設施的真實狀態，透過比對期望與現實的差異生成執行計劃（Plan），經確認後套用（Apply）變更。

Ansible 是一個無代理（Agentless）的自動化工具，透過 SSH 連線至目標機器執行任務。它使用 YAML 格式的 Playbook 定義操作步驟，適合用於軟體安裝、系統配置、應用部署等「配置後」（Day-2）操作。其冪等性（Idempotency）保證多次執行結果一致。

GitOps 是一種以 Git 為核心的運維模式。所有基礎設施與應用的期望狀態都定義在 Git 倉庫中，透過自動化代理（如 ArgoCD、Flux）持續監控倉庫變更並同步至目標環境。變更必須透過 Pull Request 提交，確保有審查記錄與回滾能力。

架構總覽

一個完整的基礎設施自動化架構通常分為以下層次：

  1. 基礎設施層（Terraform）：VPC、子網路、安全群組、負載均衡器、資料庫實例
  2. 平台層（Terraform + Ansible）：Kubernetes 叢集、中介軟體部署、監控系統
  3. 應用層（GitOps）：Kubernetes Manifests、Helm Charts、Kustomize Overlays
  4. 策略層（OPA/Kyverno）：合規檢查、資源限制、命名規範

Terraform 的模組化設計支援將基礎設施拆分為可重用的元件：

infrastructure/
├── modules/
│   ├── networking/    # VPC, Subnets, Route Tables
│   ├── compute/       # EC2, ASG, Launch Templates
│   ├── database/      # RDS, ElastiCache
│   └── kubernetes/    # EKS Cluster, Node Groups
├── environments/
│   ├── dev/
│   ├── staging/
│   └── production/
└── global/            # IAM, DNS, S3 Backends

GitOps 的運作模式則為：Developer Push → Git Repository → ArgoCD 偵測差異 → 自動同步至 Kubernetes → 健康檢查 → 完成/回滾。

應用場景

多環境一致性：使用 Terraform Module 搭配不同的 terraform.tfvars 為 Dev/Staging/Production 環境建立結構相同但規格不同的基礎設施。環境間的差異僅在變數層面，消除配置漂移（Configuration Drift）。

災難復原自動化：將整套生產環境的 Terraform 定義存放在 Git，災難發生時只需在備援區域執行 terraform apply 即可在分鐘內重建完整環境，實現低 RTO。

合規即程式碼：利用 OPA（Open Policy Agent）或 Sentinel 定義基礎設施策略（如「所有 S3 Bucket 必須啟用加密」），在 Terraform Plan 階段自動攔截違規變更，實現 Policy as Code。

混合雲管理：Terraform 的多 Provider 架構允許在同一套程式碼中管理 AWS、GCP 與地端 VMware 資源，搭配 Terragrunt 管理跨環境的模組引用與狀態隔離。

自動化 Day-2 維運：Ansible 在節點加入叢集後自動完成安全加固（SSH 配置、防火牆規則、Audit Log 啟用）、監控代理安裝與應用配置部署，確保每台新機器都符合組織標準。
