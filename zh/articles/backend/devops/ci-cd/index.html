CI/CD

持續整合與持續部署

持續整合與持續部署（CI/CD）是現代軟體工程的基礎實踐，它將程式碼從開發者的本機推進至生產環境的過程完全自動化。CI 確保每次變更都經過驗證，CD 確保通過驗證的變更能快速且安全地到達使用者手中。本篇將深入介紹 CI/CD 管線的設計原則、測試整合策略與產物管理實務。

核心概念

CI/CD 涵蓋三個關鍵領域：管線設計（Pipelines）、CI 中的測試策略（Testing in CI）、以及產物管理（Artifact Management）。

管線（Pipeline） 是 CI/CD 的核心抽象，定義了程式碼從提交到部署的完整自動化流程。一條管線由多個階段（Stage）組成，每個階段包含一或多個工作（Job），工作之間可並行或串行執行。良好的管線設計應遵循「快速回饋」原則——最可能失敗的檢查放在最前面。

CI 中的測試不僅包含單元測試，還涵蓋靜態分析（Lint）、安全掃描（SAST/DAST）、整合測試與端對端測試。測試金字塔（Test Pyramid）的概念指導我們在速度與信心之間取得平衡：大量快速的單元測試為底層，少量慢速但高價值的 E2E 測試為頂層。

產物管理處理建構輸出（Artifact）的儲存、版本化與分發。Docker Image、JAR 檔、NPM Package 等產物需要統一管理，確保部署時使用的是經過完整驗證的確切版本。

架構總覽

一條典型的 CI/CD 管線架構如下：

  1. 觸發階段：Git Push / Pull Request / Schedule / Manual Trigger
  2. 驗證階段：程式碼格式檢查 → 靜態分析 → 安全掃描
  3. 建構階段：編譯 → 打包 → 產物生成
  4. 測試階段：單元測試 → 整合測試 → E2E 測試（可並行）
  5. 發布階段：產物推送至 Registry → 版本標記
  6. 部署階段：Staging 部署 → 冒煙測試 → Production 部署
  7. 驗證階段：健康檢查 → 效能基準 → 自動回滾判斷

管線即程式碼（Pipeline as Code）是現代 CI/CD 的標準做法，管線定義檔（如 .github/workflows/, Jenkinsfile, .gitlab-ci.yml）與應用程式碼存放在同一 Repository 中，享有版本控制與 Code Review 的保護。

進階模式包括：矩陣建構（Matrix Build）支援多平台/多版本測試、快取策略（Caching）加速重複建構、以及條件式執行（Conditional Execution）避免不必要的工作。

應用場景

前端 Monorepo：大型前端專案使用 Turborepo 或 Nx 管理 Monorepo，CI 管線需支援影響分析（Affected Analysis），只建構和測試受變更影響的套件，大幅節省管線執行時間。

多環境漸進式部署：管線設計為 Dev → QA → Staging → Production 的漸進式推進，每個環境之間設置 Gate（人工審批或自動化品質門檻），確保只有高品質的變更能到達生產環境。

安全合規管線：在管線中整合 Snyk、Trivy 或 SonarQube 進行漏洞掃描與程式碼品質檢查，不通過安全檢查的建構自動阻止合併，實現 Shift-Left Security。

產物版本追溯：每個部署版本都對應唯一的產物 Hash 與 Git Commit，支援快速定位問題版本並回滾至已知正確的版本。搭配 SBOM（Software Bill of Materials）滿足供應鏈安全需求。

跨團隊共享管線：透過可重用工作流程（Reusable Workflows）或共享程式庫（Shared Libraries），標準化組織內的 CI/CD 實踐，避免各團隊重複造輪子。
