發布策略（Release Strategy）定義了軟體如何從開發階段安全地交付到使用者手中。良好的發布策略能在發布速度與風險控制之間取得平衡，確保每次發布都是可預測的、可逆的、低風險的。

核心概念

發布 vs 部署

概念定義關注點
部署（Deploy）將程式碼放到伺服器上技術層面的安裝
發布（Release）讓使用者能使用新功能業務層面的曝光

分離部署與發布是現代發布策略的核心思想。程式碼可以先部署到生產環境但功能隱藏（Dark Launch），確認穩定後再對使用者開放。

語意化版本控制（SemVer）

位置名稱意義變更時機
X.y.zMajor不兼容的 API 變更破壞性變更
x.Y.zMinor向下兼容的新功能新增功能
x.y.ZPatch向下兼容的修復Bug 修復

發布節奏模型

模型特點適用場景風險
持續發布隨時發布，一天多次成熟的 CD 團隊低（小量變更）
固定週期每週/雙週發布一次中型團隊中
發布火車固定時間發車，趕上就上大型組織中
里程碑發布功能完成時發布套裝軟體高（大量變更）

應用場景

漸進式發布策略組合

結合多種技術實現安全發布：

  1. Dark Launch：部署到生產但不對外
  2. 內部測試：僅內部員工可見
  3. Beta 測試：開放給申請的使用者
  4. 百分比放量：1% → 5% → 25% → 50% → 100%
  5. 全量發布：所有使用者可見

每個階段的監控指標：

  • 錯誤率是否上升
  • 效能指標是否劣化
  • 業務指標是否異常
  • 使用者回饋情況
