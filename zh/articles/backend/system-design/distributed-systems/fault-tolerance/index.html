容錯設計是分散式系統工程的核心能力，目標是讓系統在部分元件發生故障時仍能繼續提供服務。在大規模系統中，故障不是「是否會發生」的問題，而是「何時發生」的問題。

核心概念

故障類型

分散式系統面臨的故障可分為以下類別：

  • 崩潰故障（Crash Failure）：節點停止運作，不再回應任何請求
  • 遺漏故障（Omission Failure）：節點未能發送或接收訊息
  • 時序故障（Timing Failure）：節點回應超出時間限制
  • 拜占庭故障（Byzantine Failure）：節點表現出任意行為，包括發送錯誤資訊

容錯策略

容錯的核心策略基於冗餘（Redundancy）：

  • 資訊冗餘：額外的位元用於錯誤偵測與修正（如校驗碼）
  • 時間冗餘：操作可重試，使用冪等性保證安全重試
  • 物理冗餘：額外的硬體或軟體元件承擔故障轉移

故障偵測

準確的故障偵測是容錯的前提：

  • 心跳機制（Heartbeat）：定期發送存活訊號
  • Phi Accrual Failure Detector：基於歷史回應時間計算故障機率
  • Gossip-based Detection：多個節點交叉確認，減少誤判

主要容錯模式

  1. 主動-被動（Active-Passive）：一個主節點處理請求，備援節點待命接管
  2. 主動-主動（Active-Active）：多個節點同時處理請求，故障時流量重新分配
  3. N+1 冗餘：N 個節點處理正常流量，額外 1 個節點作為緩衝
  4. Circuit Breaker：偵測下游服務故障，快速失敗避免級聯崩潰

關鍵比較

容錯模式恢復時間資源成本複雜度資料一致性
Active-Passive秒級2x低強（同步複製）
Active-Active毫秒級2x+高需要衝突解決
Multi-Region秒~分鐘3x+很高通常最終一致
Circuit Breaker即時1x中降級服務

Failover vs Failback

  • Failover：當主節點故障時，自動切換到備援節點
  • Failback：當原主節點恢復後，是否自動切回
  • 自動 Failback 需要謹慎：可能導致反覆切換（Flapping）

同步複製 vs 非同步複製

  • 同步複製：寫入確認前等待所有副本，零資料丟失但延遲高
  • 非同步複製：寫入後背景同步，延遲低但可能丟失最近寫入
  • 半同步複製：等待至少一個副本確認，折衷方案

應用場景

資料庫高可用

MySQL Group Replication 或 PostgreSQL Patroni 提供自動故障轉移。主節點故障時，系統自動選舉新主節點，應用層通過虛擬 IP 或 DNS 切換無感知遷移。典型 RTO（Recovery Time Objective）在 10-30 秒。

微服務容錯

Netflix Hystrix（已停維，現用 Resilience4j）實現 Circuit Breaker 模式。當下游服務錯誤率超過閾值，熔斷器打開，直接返回降級回應。搭配重試、逾時、艙壁隔離形成完整的容錯體系。

全球多區域部署

跨區域部署提供最高等級的容錯能力。AWS 的 Multi-AZ 保護單一資料中心故障，Multi-Region 保護整個區域故障。CockroachDB 的 Multi-Region 配置可以設定存活區域偏好和資料地理位置約束。

Chaos Engineering

Netflix 的 Chaos Monkey 主動注入故障測試系統韌性。透過定期隨機終止生產環境的服務實例，團隊被迫建構真正具有容錯能力的系統。更進階的有 Chaos Kong（模擬整個區域故障）。

消息佇列容錯

Kafka 通過分區副本（Replication Factor）和 ISR（In-Sync Replicas）機制保證訊息不丟失。當 Broker 故障時，Partition Leader 自動切換到 ISR 中的副本。
