Bounded Context 是領域驅動設計中最重要的戰略模式，定義了模型的適用範圍與語意邊界。每個 Bounded Context 內擁有一致的領域語言和模型，不同 Context 之間的同名概念可以有完全不同的含義。

核心概念

什麼是 Bounded Context

Bounded Context 是一個明確的邊界，在這個邊界內，領域模型的每個術語都有精確且唯一的定義。它解決了大型系統中「同一個詞在不同部門代表不同含義」的問題。

例如在電商系統中：

  • 銷售 Context：「Product」包含價格、促銷、描述
  • 倉儲 Context：「Product」包含重量、尺寸、庫位
  • 物流 Context：「Product」包含包裹大小、運送限制

Context Map

Context Map 描述了系統中所有 Bounded Context 之間的關係與整合模式：

  • Shared Kernel：兩個 Context 共享一小部分模型，任何修改需雙方同意
  • Customer-Supplier：上游（Supplier）提供服務，下游（Customer）消費，上游需考慮下游需求
  • Conformist：下游完全遵循上游的模型，不做任何轉換
  • Anti-Corruption Layer (ACL)：下游建立轉換層，隔離上游模型的影響
  • Open Host Service：上游提供公開的協議供多個下游使用
  • Published Language：使用標準化的語言格式進行交流（如 JSON Schema）
  • Separate Ways：兩個 Context 完全獨立，不進行整合

邊界劃分原則

  1. 語言邊界：當同一個術語在不同團隊有不同含義時
  2. 業務流程邊界：不同的業務流程通常屬於不同 Context
  3. 資料所有權：誰擁有資料的最終權威版本
  4. 團隊邊界：Conway’s Law — 組織結構影響系統結構

整合模式實現

Context 之間的整合通常通過以下技術實現：

  • 同步：REST API、gRPC 呼叫
  • 非同步：Domain Events + Message Queue
  • 共享資料庫（反模式）：應避免，破壞邊界獨立性

關鍵比較

整合模式耦合程度適用場景複雜度
Shared Kernel高緊密合作的團隊低
Customer-Supplier中明確上下游關係中
Conformist中高無法影響上游低
ACL低隔離遺留系統高
Open Host Service低多消費者場景中
Separate Ways無完全獨立業務最低

Bounded Context vs 微服務

面向Bounded Context微服務
本質邏輯邊界（設計概念）物理部署單元
粒度可大可小通常較小
關注點語言一致性、模型邊界獨立部署、技術異構
關係一個 BC 可包含多個微服務一個微服務屬於一個 BC

常見誤區：一個 Bounded Context 不一定等於一個微服務。初期可以一個 Context 對應一個模組（Modular Monolith），成熟後再拆分為獨立服務。

過大 vs 過小的 Context

  • 過大：模型內部出現歧義，同一類別承載多個不同含義
  • 過小：過度分散，簡單操作需要跨越多個 Context 協調

應用場景

電商平台 Context 劃分

大型電商通常包含：商品目錄 Context、訂單 Context、支付 Context、庫存 Context、物流 Context、用戶 Context、促銷 Context。每個 Context 有獨立的資料庫和團隊所有權，通過領域事件實現最終一致性。

遺留系統現代化

使用 Anti-Corruption Layer 隔離遺留系統。新的 Bounded Context 通過 ACL 與舊系統互動，ACL 負責模型轉換和資料映射。逐步將遺留系統的功能遷移到新 Context，最終淘汰舊系統。

SaaS 多租戶架構

租戶管理、計費、權限各自為獨立 Context。核心業務邏輯作為另一個 Context，通過 Published Language（如 JSON API 規格）與其他 Context 整合。不同租戶可能啟用不同的 Context 組合。

事件驅動的 Context 整合

訂單 Context 發布 OrderConfirmed 事件，庫存 Context 訂閱並扣減庫存，物流 Context 訂閱並建立配送任務。每個 Context 只關心自己需要的事件資料，通過事件實現鬆耦合的整合。

Modular Monolith

在微服務之前，先以模組化單體的形式實現 Bounded Context。每個模組有獨立的程式碼目錄、資料庫 Schema 和公開介面。驗證邊界劃分正確後，再根據擴展需求逐步拆分為獨立服務。
