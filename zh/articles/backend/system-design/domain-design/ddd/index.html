領域驅動設計（Domain-Driven Design, DDD）是一種軟體設計方法論，強調以業務領域為核心來驅動軟體架構的設計。它提供了一套戰略與戰術模式，幫助團隊應對複雜業務系統的開發挑戰。

核心概念

戰略設計（Strategic Design）

戰略設計關注的是如何劃分和組織整個系統的大架構：

  • 領域（Domain）：軟體系統要解決的業務問題空間
  • 子域（Subdomain）：領域的分解，分為核心域、支撐域、通用域
  • Bounded Context：明確的語意邊界，每個 Context 內有一致的語言模型
  • Context Map：描述 Bounded Context 之間的關係與整合方式

戰術設計（Tactical Design）

戰術設計提供具體的程式碼層級建模工具：

  • Entity（實體）：具有唯一身份標識的物件，生命週期內身份不變
  • Value Object（值物件）：沒有身份標識，由其屬性值定義相等性
  • Aggregate（聚合）：一組相關物件的集合，有一個 Aggregate Root 作為存取入口
  • Domain Event（領域事件）：記錄領域中發生的重要事實
  • Repository（倉儲）：提供聚合的持久化與查詢介面
  • Domain Service（領域服務）：不屬於任何實體或值物件的領域邏輯

統一語言（Ubiquitous Language）

DDD 最重要的核心原則是建立統一語言——開發團隊與業務專家共用的精確術語。程式碼中的類別名、方法名應直接反映業務概念，消除溝通障礙。

聚合設計原則

  1. 一致性邊界：聚合內部保證事務一致性，聚合之間通過最終一致性
  2. 小聚合原則：聚合應盡量小，只包含真正需要強一致的資料
  3. 通過 ID 引用：聚合之間通過 ID 引用而非物件引用
  4. 單一 Aggregate Root：外部只能通過 Root 存取聚合內部物件

關鍵比較

模式適用時機複雜度好處
Transaction Script簡單 CRUD低快速開發
Active Record中等複雜度中低ORM 友善
Domain Model (DDD)高複雜業務高精確建模
Event Sourcing需要審計追蹤很高完整歷史

DDD vs 傳統三層架構

面向DDD傳統三層
驅動力業務領域技術分層
模型位置核心（領域層）Service 層邏輯
資料庫角色實現細節設計起點
業務規則聚合內封裝Service/Controller 散落
測試重點領域邏輯單元測試整合測試為主

Entity vs Value Object

  • Entity：Order(id=123) — 同一訂單在不同時間點仍是同一訂單
  • Value Object：Money(100, "USD") — 兩個 100 USD 完全相等可互換

應用場景

電商訂單系統

訂單聚合包含 Order（Root）、OrderLine、ShippingAddress（Value Object）。下單時驗證庫存、計算價格的業務規則封裝在 Order 聚合內。庫存和支付作為獨立聚合，通過領域事件 OrderPlaced 觸發後續流程。

保險理賠系統

理賠是典型的複雜領域：包含理賠申請、審核流程、賠付計算等多個子域。每個子域有自己的 Bounded Context 和領域模型。Policy Context 中的「保單」和 Claims Context 中的「保單」可能有不同的屬性和行為。

物流排程系統

運輸路線規劃涉及車輛（Entity）、路線（Value Object）、排程規則（Domain Service）。聚合邊界劃分在單一配送任務，跨任務的最佳化通過領域服務協調。

CQRS 與 Event Sourcing

DDD 天然適合與 CQRS（Command Query Responsibility Segregation）結合。命令側使用豐富的領域模型處理業務邏輯，查詢側使用扁平化的讀模型最佳化查詢效能。Event Sourcing 將所有狀態變更記錄為事件序列，提供完整的審計追蹤。
