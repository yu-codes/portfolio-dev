系統拆分策略決定了如何將一個大型系統分解為多個獨立且可管理的部分。正確的拆分能夠提升團隊自主性、部署靈活性和系統可維護性，而錯誤的拆分則會導致分散式單體的反模式。

核心概念

拆分的動機

系統拆分通常出於以下原因：

  • 團隊擴展：單一程式碼庫無法支撐多個團隊並行開發
  • 獨立部署：不同模組有不同的發布節奏和風險等級
  • 技術異構：不同功能適合不同的技術棧
  • 獨立擴展：不同元件有不同的資源需求和流量模式
  • 故障隔離：防止單一模組的故障影響整個系統

拆分原則

  1. 高內聚、低耦合：相關的功能放在一起，減少跨服務呼叫
  2. 業務能力對齊：根據業務能力而非技術層級拆分
  3. 資料所有權：每個服務擁有並管理自己的資料
  4. 團隊邊界對齊：服務邊界與團隊邊界一致（Conway’s Law）
  5. 最小化分散式事務：避免跨服務的強一致性需求

拆分方法

按業務能力拆分（Business Capability）：識別組織的核心業務能力，每個能力對應一個服務。例如：訂單管理、庫存管理、支付處理。

按子域拆分（Subdomain）：基於 DDD 的子域分析，識別核心域、支撐域和通用域，以此為基礎劃分服務邊界。

按數據分區拆分：根據資料的所有權和存取模式劃分。相同資料集的讀寫操作應在同一個服務內。

Strangler Fig 模式

從單體逐步遷移到微服務的實施策略：

  1. 在單體外圍建立 Proxy/Gateway
  2. 新功能作為獨立微服務開發
  3. 逐步將舊功能從單體中提取到微服務
  4. 單體逐漸縮小直至完全淘汰

關鍵比較

拆分策略適用階段複雜度團隊規模風險
模組化單體早期/中期低1-3 團隊低
按業務能力拆分中期中3-10 團隊中
按子域拆分 (DDD)中後期中高5+ 團隊中
事件驅動拆分後期高10+ 團隊中高

正確拆分 vs 分散式單體

健康的微服務分散式單體（反模式）
獨立部署必須同時部署多個服務
獨立資料庫共享資料庫
非同步通訊為主大量同步串聯呼叫
團隊自主跨團隊協調頻繁
局部故障不擴散一個服務掛全部掛

何時不該拆分

  • 團隊少於 5 人
  • 業務領域尚未充分理解
  • 沒有獨立部署的需求
  • 缺乏分散式系統的維運能力
  • 系統流量規模不需要獨立擴展

應用場景

電商單體拆分實例

一個成長中的電商平台拆分路徑：

  1. 第一刀：將支付獨立（合規需求、安全隔離）
  2. 第二刀：將商品搜尋獨立（獨立擴展需求、技術異構）
  3. 第三刀：將訂單與庫存分離（不同團隊、不同發布節奏）
  4. 後續根據團隊成長和業務需求繼續細分

EventStorming 驅動的拆分

通過 EventStorming 工作坊識別領域事件和聚合，自然形成服務邊界：

  • 識別所有領域事件（橙色便利貼）
  • 將事件分群為相關的業務流程
  • 識別每群事件的聚合和命令
  • 群組邊界即為潛在的服務邊界

資料網格（Data Mesh）思維

將資料所有權下放到各個領域團隊：

  • 每個團隊擁有自己領域的資料產品
  • 提供標準化的資料存取介面
  • 聯邦式治理而非中央集權
  • 自助式資料基礎設施平台

漸進式拆分策略

避免大爆炸式重寫，採用漸進式方法：

  • Branch by Abstraction：在單體內引入抽象層
  • Feature Toggle：控制新舊實現的切換
  • Parallel Run：新舊系統同時運行並比對結果
  • Dark Launch：新服務接收流量但不影響回應
