負載平衡（Load Balancing）是分散式系統實現高可用與高效能的基石。透過將流量分配到多個後端伺服器，負載平衡器確保系統能夠處理大量並發請求，同時在節點故障時自動轉移流量。本文深入探討負載平衡的原理、演算法與架構設計。

核心概念

負載平衡的層級

依據 OSI 模型，負載平衡器主要運作在兩個層級：

  • L4 負載平衡（傳輸層）：基於 IP 位址與 TCP/UDP Port 進行流量分配，不解析應用層內容
  • L7 負載平衡（應用層）：可解析 HTTP 標頭、URL、Cookie 等應用層資訊做智慧路由
  • DNS 負載平衡：透過 DNS 回應不同 IP 實現最基本的流量分配

負載分配演算法

  • Round Robin（輪詢）：依序分配給每個後端節點
  • Weighted Round Robin（加權輪詢）：依據權重分配更多流量給高性能節點
  • Least Connections（最少連線）：分配給當前連線數最少的節點
  • Least Response Time（最快回應）：分配給回應時間最短的節點
  • IP Hash：依據客戶端 IP 雜湊值固定分配，實現會話黏著
  • Consistent Hashing（一致性雜湊）：節點增減時最小化重新分配

健康檢查

  • 主動健康檢查：定期向後端發送探測請求（HTTP GET、TCP Connect）
  • 被動健康檢查：監控實際請求的回應狀態判斷健康度
  • 複合檢查：結合主動與被動檢查提高準確性
  • 優雅下線：支援 drain 模式讓現有連線完成後再移除節點

Session 持久性

  • Source IP Affinity：相同 IP 導向相同後端
  • Cookie-based：透過 Cookie 標記分配到的後端
  • Header-based：依據自訂 HTTP 標頭路由

演算法適用場景

演算法適用場景優點缺點
Round Robin節點性能一致簡單公平不考慮負載
Least Connections請求處理時間差異大動態適應新節點易過載
IP Hash需要 Session 黏著一致性分布不均
Weighted節點性能不一致精準控制需手動配置

應用場景

場景一：Web 應用多層負載平衡

  • DNS 層：GeoDNS 將用戶導向最近的機房
  • L4 層：ECMP/BGP 在多個 L7 LB 間分流
  • L7 層：Nginx 根據 URL 路由到不同服務叢集
  • 健康檢查自動移除故障節點

場景二：微服務內部負載平衡

  • Service Mesh（Envoy Sidecar）實現客戶端負載平衡
  • 支援 Circuit Breaker 與 Retry 策略
  • 金絲雀部署透過權重控制流量比例
  • 灰度發布依據 Header 路由到新版本

場景三：全球化部署

  • GSLB 依據延遲與可用性選擇最佳區域
  • 各區域內部使用多層負載平衡
  • 故障時自動切換流量至其他區域
  • Anycast IP 讓用戶連接最近的入口點

最佳實踐

  1. 多層架構：L4 + L7 組合使用，兼顧效能與功能
  2. 健康檢查配置：設定合理的檢查間隔與失敗閾值
  3. 連線池管理：後端連線復用減少建立開銷
  4. 監控指標：追蹤各後端的回應時間、錯誤率、連線數
