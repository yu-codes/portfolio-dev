單體 vs 微服務

「應該用單體還是微服務？」是後端架構中最常被討論的問題之一。事實上，這不是非黑即白的選擇，而是一個需要考量團隊規模、業務複雜度、營運成熟度等多重因素的權衡決策。本文深入分析兩種架構的優劣與適用場景。

核心概念

單體架構（Monolith）

單體架構將所有功能模組打包為一個部署單元：

  • 模組化單體（Modular Monolith）：內部有清晰的模組邊界，但仍為單一部署
  • 分層單體：傳統的三層或多層架構
  • 共享資料庫：所有模組使用同一個資料庫
  • 程序內通信：模組間透過方法呼叫互動，無網路開銷

微服務架構（Microservices）

將系統拆分為多個獨立部署的服務：

  • 服務自治：各服務獨立開發、測試、部署
  • 資料隔離：每個服務擁有自己的資料儲存
  • 網路通信：服務間透過 API 或訊息互動
  • 技術多樣性：各服務可選擇最適合的技術棧

中間地帶

  • Modular Monolith：單體但內部模組邊界清晰
  • Mini-services：比微服務大、比單體小的服務粒度
  • Self-contained Systems：獨立的全端系統，包含 UI 與後端
  • Monolith-first：先從單體開始，需要時再拆分

關鍵比較

面向單體架構微服務架構
開發速度（初期）快慢（需建立基礎設施）
開發速度（後期）慢（互相干擾）快（獨立開發）
部署整體部署獨立部署
擴展性整體擴展精準擴展
技術棧統一多樣
資料一致性ACID 交易最終一致性
除錯簡單（單一行程）複雜（分散式追蹤）
營運成本低高
團隊規模適合小團隊適合大組織
延遲低（程序內）較高（網路呼叫）

決策矩陣

條件建議
團隊 < 10 人單體或 Modular Monolith
新創 MVP單體（快速驗證）
業務邊界清晰可考慮微服務
需要獨立擴展微服務
有成熟 DevOps微服務可行
高可用要求微服務（故障隔離）
頻繁發布需求微服務（獨立部署）

應用場景

場景一：新創公司 MVP

建議：單體架構

  • 快速迭代驗證商業模式
  • 團隊小，不需要複雜的服務治理
  • 資源有限，無法承擔微服務的運維成本
  • 業務邊界尚不明確，過早拆分風險高

場景二：成長期產品

建議：Modular Monolith → 漸進式拆分

  • 業務邊界逐漸清晰
  • 部分模組需要獨立擴展
  • 使用 Strangler Fig Pattern 逐步遷移
  • 先拆出變更最頻繁或負載最高的模組

場景三：大規模企業系統

建議：微服務架構

  • 多團隊平行開發需要獨立部署
  • 不同模組有截然不同的擴展需求
  • 組織有成熟的 DevOps 與平台工程能力
  • 業務複雜度高，需要技術多樣性

常見反模式

  1. Distributed Monolith：拆成微服務但耦合度跟單體一樣高
  2. 過早微服務：業務邊界不明確時就拆分，導致頻繁重構
  3. 共享資料庫：微服務共用資料庫，失去資料隔離的優勢
  4. Nano-services：拆分過細，一個功能需要呼叫十幾個服務
  5. 忽略運維需求：沒有對應的基礎設施就導入微服務

Strangler Fig Pattern

漸進式從單體遷移到微服務的策略：

  1. 在新服務中實作新功能
  2. 將邊界明確的模組逐一抽出
  3. 透過反向代理將流量導向新服務
  4. 逐步縮小單體範圍
  5. 最終完全淘汰或保留精簡的核心
