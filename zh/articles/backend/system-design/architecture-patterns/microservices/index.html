微服務架構（Microservices Architecture）是一種將應用程式拆分為多個小型、獨立部署的服務的架構風格。每個服務圍繞特定業務能力建構，擁有獨立的資料儲存與部署生命週期。本文深入探討微服務的設計原則、架構模式與實務挑戰。

核心概念

微服務特徵

  • 單一職責：每個服務只負責一個明確的業務能力
  • 獨立部署：服務可以獨立開發、測試、部署與擴展
  • 去中心化治理：各團隊可選擇最適合的技術棧
  • 去中心化資料管理：每個服務擁有自己的資料庫
  • 故障隔離：一個服務的失敗不會直接導致整個系統崩潰
  • 可觀測性：需要完善的監控、日誌與分散式追蹤

服務拆分原則

  • 業務能力分解：依據組織的業務功能邊界拆分
  • 子域分解（DDD）：按照限界上下文（Bounded Context）拆分
  • 動詞/名詞分析：識別核心動作或資源作為服務邊界
  • 團隊組織（Conway’s Law）：服務邊界對齊團隊邊界

服務間通信

  • 同步通信：REST、gRPC、GraphQL
  • 非同步通信：訊息佇列、事件串流
  • 服務發現：DNS、Consul、Eureka
  • API Gateway：統一入口、路由、認證、限流

關鍵基礎設施

  • 容器化：Docker 封裝服務、Kubernetes 編排部署
  • CI/CD：每個服務獨立的建構與部署流水線
  • 可觀測性：分散式追蹤（Jaeger）、指標（Prometheus）、日誌（ELK）
  • 服務網格：處理服務間通信的基礎設施層

應用場景

場景一：電商平台

  • 用戶服務、商品服務、訂單服務、支付服務各自獨立
  • 各服務可依據負載獨立擴展（如大促時擴展訂單服務）
  • 不同服務使用最適合的資料庫（用戶用 PostgreSQL、商品用 MongoDB）
  • API Gateway 處理前端請求的路由與聚合

場景二：SaaS 多租戶平台

  • 租戶管理、計費、核心功能分離
  • 核心功能可按租戶需求客製化
  • 各服務獨立擴展應對不同租戶的負載
  • 新功能可漸進式上線（Feature Flag + 金絲雀部署）

場景三：資料密集型應用

  • 資料擷取、處理、儲存、查詢各為獨立服務
  • 擷取層使用 Kafka 處理高吞吐
  • 處理層按需水平擴展
  • 查詢服務使用專用的讀取資料庫

導入挑戰

  1. 分散式交易：跨服務的交易一致性需使用 Saga 模式
  2. 資料一致性：最終一致性的複雜度與除錯難度
  3. 網路問題：延遲、超時、分區容錯需仔細處理
  4. 運維複雜度：數十上百個服務的部署、監控、除錯
  5. 測試困難：整合測試需模擬多服務互動
