快取策略模式

快取策略決定了資料如何在快取與資料來源之間流動。不同的讀寫模式適用於不同的業務場景，選擇正確的快取策略是系統效能與資料一致性之間的關鍵權衡。

核心概念

Cache-Aside（旁路快取）

最常用的快取模式，應用程式同時管理快取與資料庫：

讀取流程：

  1. 應用程式先查詢快取
  2. 快取命中（Cache Hit）→ 直接返回
  3. 快取未命中（Cache Miss）→ 查詢資料庫
  4. 將結果寫入快取，設定 TTL
  5. 返回結果給呼叫者

寫入流程：

  1. 更新資料庫
  2. 刪除（Invalidate）快取中的對應項目
  3. 下次讀取時重新載入

Read-Through / Write-Through

快取作為資料存取的唯一介面：

  • Read-Through：快取未命中時，由快取層自動從資料庫載入
  • Write-Through：寫入操作同步寫入快取與資料庫
  • 優點：應用程式邏輯簡化，資料一致性較高
  • 缺點：寫入延遲增加，快取層需要資料庫存取能力

Write-Behind（Write-Back）

寫入時僅更新快取，非同步批次寫回資料庫：

  • 寫入快取後立即返回成功
  • 背景程式定期或按條件將變更批次寫入資料庫
  • 優點：寫入效能極佳，可合併多次寫入
  • 缺點：快取故障可能遺失資料，資料一致性較弱

Refresh-Ahead（預先刷新）

在快取即將過期前主動刷新：

  • 監控即將過期的項目
  • 在過期前觸發非同步刷新
  • 避免過期後的突發 Cache Miss（Cache Stampede）
  • 適用於可預測存取模式的資料

多層快取架構

請求 → L1 快取（本地記憶體）
         │ Miss
         ↓
      L2 快取（Redis）
         │ Miss
         ↓
      資料庫
         │
         ↓
      回填 L2 → 回填 L1

多層快取的考量：

  • L1（進程內）：無網路延遲，容量小，各實例獨立
  • L2（分散式）：共享，容量大，有網路延遲
  • 一致性挑戰：L1 失效需要廣播通知機制

應用場景

場景一：電商商品詳情頁

Cache-Aside + 多層快取：

  • L1：CDN 快取靜態內容
  • L2：Redis 快取商品資訊（TTL 5 分鐘）
  • 商品更新時刪除快取，搭配延遲雙刪策略
  • 價格與庫存使用較短 TTL 或即時查詢

場景二：社群媒體動態牆

Write-Behind 模式：

  • 發佈動態時寫入快取立即返回
  • 非同步 Worker 將資料寫入資料庫
  • 動態牆讀取直接從快取獲取
  • 容忍短暫的資料不一致

場景三：設定中心

Read-Through + Refresh-Ahead：

  • 設定資料讀取極頻繁但很少變更
  • 快取層自動從設定源載入
  • 設定變更時主動推送失效通知
  • 各服務本地快取 + 共享快取雙層保護

場景四：熱門內容排行榜

Write-Through + TTL：

  • 瀏覽計數寫入同步更新快取與資料庫
  • 排行榜結果設定短 TTL（如 1 分鐘）
  • 避免即時計算排名的高開銷
  • 允許分鐘級的排名延遲
