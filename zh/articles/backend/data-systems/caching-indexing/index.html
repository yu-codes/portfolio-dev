快取與索引

快取策略與失效模式

快取與索引是提升資料系統效能的兩大核心技術。快取將頻繁存取的資料暫存在高速記憶體中，避免重複的昂貴計算或磁碟 I/O；索引在資料上建立額外的查找結構，讓查詢能快速定位目標記錄。兩者共同作用，能將系統的回應時間從秒級降低到毫秒級，是大規模系統設計中不可或缺的效能利器。

核心概念

快取的基本原理基於時間局部性（Temporal Locality）和空間局部性（Spatial Locality）：最近被存取的資料很可能在短時間內再次被存取，而鄰近的資料也可能被一起存取。快取命中率（Hit Rate）是衡量快取效果的關鍵指標，通常需要達到 95% 以上才能顯著改善效能。

快取層級在現代系統中呈多層架構：CPU L1/L2/L3 快取 → 應用程式記憶體快取 → 分散式快取（Redis/Memcached）→ CDN 邊緣快取 → 瀏覽器快取。每一層都以較小的延遲服務請求，未命中時才往下一層查詢。

索引的基本原理是以額外的儲存空間換取查詢速度。資料庫索引本質上是一個從鍵到資料位置的映射結構，讓查詢不需要遍歷全部資料就能找到目標。常見的索引結構包括 B+Tree（適合範圍查詢）、Hash Index（適合等值查詢）、倒排索引（適合全文搜尋）。

快取一致性是快取系統面臨的最大挑戰。當原始資料被修改時，快取中的副本變得過時（Stale），需要透過失效策略（Invalidation）或更新策略（Update）來維持一致性。在分散式環境中，這個問題更加複雜，因為多個節點可能持有相同資料的不同版本。

架構總覽

快取模式（Caching Patterns） 定義了應用程式與快取之間的互動方式：

  • Cache-Aside（旁路快取）：應用程式先查快取，未命中時從資料庫讀取並回填快取。最常見的模式，適合讀多寫少的場景。
  • Read-Through（讀穿透）：快取層自動從資料庫載入未命中的資料，對應用透明。
  • Write-Through（寫穿透）：寫入操作同時更新快取和資料庫，保證一致性但增加寫入延遲。
  • Write-Behind（寫回）：寫入先更新快取，稍後異步寫入資料庫，提升寫入效能但有資料遺失風險。
  • Write-Around（寫繞過）：寫入直接到資料庫，只有讀取時才載入快取，適合寫入後短期內不會讀取的場景。

快取淘汰策略（Eviction Policies） 決定快取空間不足時淘汰哪些資料：

  • LRU（Least Recently Used）：淘汰最久未被存取的項目，適合大多數工作負載。
  • LFU（Least Frequently Used）：淘汰存取頻率最低的項目，適合存取模式穩定的場景。
  • TTL（Time-To-Live）：設定固定的過期時間，簡單有效但可能導致快取同時大量過期（Cache Stampede）。

索引類型與應用：

  • B+Tree 索引：資料庫中最普遍的索引結構，支援等值查詢、範圍查詢、排序操作。葉子節點形成有序鏈結串列，適合順序掃描。
  • Hash 索引：O(1) 等值查詢但不支援範圍查詢。適合精確匹配場景如快取鍵查詢。
  • 複合索引（Composite Index）：多欄位組合索引，遵循最左前綴原則，能覆蓋多條件查詢。

應用場景

高流量 Web 應用：使用 Redis 快取使用者 Session、頁面片段、API 回應。透過 Cache-Aside 模式處理讀取請求，配合 TTL 和事件驅動失效機制維持一致性。CDN 快取靜態資源減輕源站壓力。

電商庫存系統：庫存數量使用 Redis 原子操作保證扣減的正確性；商品詳情頁使用多級快取（本地快取 → Redis → 資料庫）；快取預熱（Warm-up）在大促前載入熱門商品資料。

搜尋引擎：倒排索引將文件中的詞彙映射到文件列表；TF-IDF 或 BM25 算分結果可快取以加速重複查詢；前綴樹（Trie）支援自動補全功能。

即時排行榜：Redis Sorted Set 提供 O(log n) 的分數更新和 O(log n + m) 的排名區間查詢，天然適合排行榜場景。配合 Pipeline 批量操作和定時持久化策略。

快取失效模式處理：Cache Stampede（快取雪崩）使用互斥鎖或機率性提前重新整理；Cache Penetration（快取穿透）使用 Bloom Filter 攔截不存在的 Key；Hot Key（熱點 Key）使用本地快取 + 分片策略分散壓力。
