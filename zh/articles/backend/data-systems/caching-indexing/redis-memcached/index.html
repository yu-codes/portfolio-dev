Redis 與 Memcached

Redis 與 Memcached 是兩大主流記憶體快取系統，它們透過將熱點資料儲存在記憶體中，大幅降低資料庫負載並縮短回應時間。理解兩者的差異與適用場景，是建構高效能系統的基礎。

核心概念

Redis 核心特性

Redis 不僅是快取，更是功能豐富的記憶體資料結構伺服器：

  • 豐富的資料結構：String、Hash、List、Set、Sorted Set、Stream、HyperLogLog
  • 持久化機制：RDB 快照 + AOF 日誌
  • 原子操作：單執行緒模型保證操作原子性
  • Pub/Sub：發布訂閱訊息模式
  • Lua 腳本：伺服器端執行複雜邏輯
  • 叢集模式：Redis Cluster 支援水平擴展

Memcached 核心特性

Memcached 專注於簡單高效的鍵值快取：

  • 簡單協定：GET/SET/DELETE 等基本操作
  • 多執行緒：利用多核心平行處理請求
  • Slab 分配器：預分配記憶體減少碎片
  • 一致性雜湊：客戶端分片策略
  • LRU 淘汰：記憶體不足時淘汰最近最少使用的項目
  • 無持久化：純記憶體操作，重啟後資料遺失

記憶體管理

Redis 記憶體策略：

  • noeviction：記憶體滿時拒絕寫入
  • allkeys-lru：從所有鍵中淘汰最近最少使用
  • volatile-lru：僅從設有過期時間的鍵中淘汰
  • allkeys-random：隨機淘汰
  • volatile-ttl：淘汰 TTL 最短的鍵
  • allkeys-lfu：淘汰最不常用的鍵

Memcached 記憶體管理：

  • Slab Class：依大小分級的記憶體區塊
  • 每個 Slab Class 有獨立的 LRU 鏈
  • 預設 Item 大小上限 1MB

Redis 持久化

  • RDB（Redis Database）：定時產生記憶體快照
    • 優點：檔案緊湊、恢復快
    • 缺點：可能遺失最後一次快照後的資料
  • AOF（Append Only File）：記錄每個寫入命令
    • 優點：資料安全性高
    • 缺點：檔案大、恢復慢
  • 混合模式：RDB + AOF 結合兩者優勢

應用場景

場景一：會話管理（Session Store）

使用 Redis 儲存使用者會話：

  • Hash 結構儲存會話資料
  • 設定 TTL 實現自動過期
  • Cluster 模式支援大量使用者
  • 持久化防止重啟遺失

場景二：排行榜系統

使用 Redis Sorted Set：

  • ZADD 更新分數
  • ZRANK 查詢排名 O(log N)
  • ZRANGEBYSCORE 範圍查詢
  • ZUNIONSTORE 合併多個排行榜

場景三：分散式鎖

使用 Redis 實現分散式鎖：

  • SET key value NX EX timeout 獲取鎖
  • Lua 腳本確保解鎖的原子性
  • RedLock 演算法提供多節點可靠性
  • 注意鎖續期與 fencing token

場景四：速率限制（Rate Limiting）

使用 Redis 實現 API 限流：

  • 固定窗口：INCR + EXPIRE
  • 滑動窗口：Sorted Set + 時間戳
  • 令牌桶：Lua 腳本實現
  • 支援多層級（IP/使用者/API）限流
