分散式資料庫

分散式資料庫結合了關聯式資料庫的 ACID 保證與 NoSQL 的水平擴展能力，被稱為 NewSQL。它們解決了傳統單機資料庫的擴展瓶頸，同時保持了 SQL 相容性與強一致性。

核心概念

分散式共識

確保多個節點間資料一致的演算法：

  • Raft：較易理解的共識演算法，CockroachDB、TiDB 採用
  • Paxos：經典共識演算法，Google Spanner 採用
  • Multi-Paxos：Paxos 的優化版，減少通訊輪次
  • EPaxos：無 Leader 的 Paxos 變體，降低延遲

分散式交易

跨多個節點的交易實現：

  • 2PC（Two-Phase Commit）：協調者驅動的準備-提交兩階段
  • Percolator 模型：Google 提出的樂觀交易模型（TiDB 採用）
  • Serializable Snapshot Isolation（SSI）：CockroachDB 採用的隔離模型
  • TrueTime：Google Spanner 的全域時間戳方案（依賴原子鐘）

資料分佈

  • Range-based Sharding：依主鍵範圍分割資料（TiDB Region）
  • Hash-based Sharding：依雜湊值分散資料
  • 自動分裂與合併：根據資料量自動調整分片
  • 資料放置策略：依地理位置或負載決定副本位置

全域一致性

  • 線性化（Linearizability）：操作彷彿在單一時間點發生
  • 外部一致性（External Consistency）：事件的先後順序在全域可見
  • 全域快照讀：在特定時間點讀取全域一致的快照
  • 時間戳排序：使用全域時間戳決定交易順序

應用場景

場景一：全球化電商平台

使用 CockroachDB 實現多區域部署：

  • 各區域設定資料放置策略，使用者資料就近存取
  • 全域交易保證庫存一致性
  • 自動容錯處理區域故障
  • PostgreSQL 相容降低遷移成本

場景二：金融交易系統

使用 TiDB 替換 MySQL 分庫分表：

  • 無需手動分片，自動水平擴展
  • ACID 交易保證資金安全
  • 與 MySQL 協定相容，應用無需修改
  • TiFlash 列存引擎支援即時分析（HTAP）

場景三：SaaS 多租戶平台

使用分散式資料庫管理多租戶資料：

  • 自動資料均衡避免熱點
  • 依租戶設定資料放置策略（合規需求）
  • 線性擴展支撐租戶增長
  • 強一致性簡化應用開發

場景四：從單體到分散式遷移

逐步遷移策略：

  • Phase 1：分散式資料庫作為從庫，同步資料
  • Phase 2：讀流量逐步切換
  • Phase 3：寫流量切換，原資料庫作為備份
  • Phase 4：完全切換，退役舊系統
