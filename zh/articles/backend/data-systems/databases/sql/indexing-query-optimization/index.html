索引與查詢優化

索引是資料庫效能調校的核心手段，正確的索引策略可將查詢時間從秒級降至毫秒級。理解索引的內部結構與查詢優化器的運作方式，是後端工程師必備的技能。

核心概念

索引類型

  • B-Tree 索引：最通用的索引類型，支援等值查詢、範圍查詢、排序
  • Hash 索引：僅支援等值查詢，查找速度 O(1)
  • GiST 索引：支援地理空間、全文搜尋等複雜資料類型
  • GIN 索引：適用於包含多值的欄位（陣列、JSON、全文）
  • BRIN 索引：適用於物理排序與邏輯排序相關的大表
  • Bitmap 索引：適用於低基數欄位，常見於資料倉儲

B-Tree 索引結構

B-Tree 的核心特性：

  • 平衡樹：所有葉節點在同一層級，查詢穩定 O(log N)
  • 有序儲存：支援範圍掃描與排序
  • 多路搜尋：每個節點包含多個鍵值，減少磁碟 I/O
  • 頁面分裂：插入時若節點已滿則分裂

複合索引（Composite Index）

複合索引的設計原則：

  • 最左前綴原則：查詢必須包含索引的最左欄位才能使用索引
  • 欄位順序：高選擇性欄位放前面、等值條件欄位放前面
  • 覆蓋索引（Covering Index）：索引包含查詢所需的所有欄位，避免回表

查詢優化器（Query Optimizer）

查詢優化器的決策過程：

  • 統計資訊：表大小、欄位基數、值分佈直方圖
  • 成本估算：估算不同執行計畫的 I/O 與 CPU 成本
  • 執行計畫選擇：選擇估算成本最低的計畫
  • JOIN 策略：Nested Loop / Hash Join / Merge Join

EXPLAIN 分析重點

EXPLAIN ANALYZE SELECT * FROM orders
WHERE customer_id = 123 AND status = 'active'
ORDER BY created_at DESC LIMIT 10;

關注指標：

  • Seq Scan vs Index Scan：是否使用索引
  • Rows：預估行數 vs 實際行數
  • Cost：啟動成本與總成本
  • Buffers：記憶體命中率

掃描方式比較

掃描方式說明適用場景
Seq Scan全表掃描小表或選擇率 > 10-20%
Index Scan索引掃描+回表選擇率低
Index Only Scan僅索引掃描覆蓋索引
Bitmap Index Scan點陣圖掃描中等選擇率

應用場景

場景一：高並發查詢優化

電商商品列表查詢優化：

  • 建立 (category_id, is_active, sort_order) 複合索引
  • 使用覆蓋索引包含列表頁所需欄位
  • 分頁使用 Keyset Pagination 取代 OFFSET
  • 熱門查詢結果快取至 Redis

場景二：慢查詢診斷

系統性慢查詢排查流程：

  • 開啟慢查詢日誌（slow_query_log）
  • 使用 EXPLAIN ANALYZE 分析執行計畫
  • 檢查統計資訊是否過期（ANALYZE 更新）
  • 確認索引是否被正確使用

場景三：寫入密集型優化

高寫入場景的索引策略：

  • 減少不必要的索引以降低寫入開銷
  • 使用部分索引（Partial Index）縮小索引範圍
  • 批次寫入時暫時停用索引，寫入後重建
  • 考慮使用 BRIN 索引取代 B-Tree

場景四：全文搜尋優化

實現高效全文搜尋：

  • PostgreSQL：使用 GIN 索引 + tsvector
  • MySQL：使用 FULLTEXT 索引
  • 複雜需求考慮 Elasticsearch 外部索引
  • 搭配前綴索引處理自動完成功能
