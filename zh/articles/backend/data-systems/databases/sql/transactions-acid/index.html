交易與 ACID 特性

資料庫交易（Transaction）是確保資料一致性的核心機制。ACID 特性為交易提供了嚴格的正確性保證，而不同的隔離等級則在一致性與效能之間提供了彈性選擇。

核心概念

ACID 特性

  • 原子性（Atomicity）：交易中的所有操作要嘛全部成功，要嘛全部回滾，不存在部分完成的狀態
  • 一致性（Consistency）：交易前後資料庫必須從一個合法狀態轉移到另一個合法狀態
  • 隔離性（Isolation）：並發交易之間互不干擾，如同序列執行
  • 持久性（Durability）：一旦交易提交，資料變更即使系統崩潰也不會遺失

隔離等級（Isolation Levels）

SQL 標準定義的四種隔離等級：

  • Read Uncommitted：可讀取未提交的資料（髒讀）
  • Read Committed：僅讀取已提交的資料，但可能不可重複讀
  • Repeatable Read：同一交易內多次讀取結果一致，但可能有幻讀
  • Serializable：最嚴格，完全序列化執行

並發問題

不同隔離等級下可能發生的問題：

  • 髒讀（Dirty Read）：讀到其他交易未提交的修改
  • 不可重複讀（Non-repeatable Read）：同筆資料兩次讀取結果不同
  • 幻讀（Phantom Read）：同一查詢條件兩次執行得到不同行數
  • 寫入偏斜（Write Skew）：兩個交易各自讀取後做出衝突決策
  • 遺失更新（Lost Update）：兩個交易同時修改同一資料，一方被覆寫

並發控制機制

  • 悲觀鎖定（Pessimistic Locking）：存取資料前先取得鎖
    • 共享鎖（S Lock）：允許多個讀取者
    • 排他鎖（X Lock）：完全獨占
  • 樂觀鎖定（Optimistic Locking）：提交時檢查衝突
    • 版本號碼（Version Number）
    • 時間戳記（Timestamp）
  • MVCC（Multi-Version Concurrency Control）：維護資料的多個版本，讀取不阻塞寫入

WAL（Write-Ahead Logging）

確保持久性的關鍵機制：

  1. 修改前先將變更寫入日誌（WAL）
  2. 日誌寫入磁碟後才確認提交
  3. 崩潰恢復時透過日誌重播（Redo）或回滾（Undo）
  4. Checkpoint 定期將記憶體中的髒頁寫回磁碟

應用場景

場景一：銀行轉帳

經典的交易應用：

  • 扣款與入帳必須在同一交易中完成（原子性）
  • 使用 Serializable 或悲觀鎖避免餘額不一致
  • 記錄交易日誌以供審計與對帳

場景二：庫存扣減

電商搶購場景：

  • 使用 SELECT FOR UPDATE 鎖定庫存記錄
  • 或使用樂觀鎖 WHERE version = expected_version
  • 高並發時考慮預扣庫存+非同步確認模式

場景三：分散式交易

跨服務的交易一致性：

  • 2PC（Two-Phase Commit）：強一致但效能差
  • Saga Pattern：透過補償操作實現最終一致
  • Outbox Pattern：可靠事件發布

場景四：批次處理中的交易

大量資料處理：

  • 分批提交避免長交易佔用資源
  • 設定合理的 lock_timeout 避免死鎖
  • 使用 Savepoint 實現部分回滾
