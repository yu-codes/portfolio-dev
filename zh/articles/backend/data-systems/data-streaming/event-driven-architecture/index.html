事件驅動架構（Event-Driven Architecture, EDA）是一種以事件為核心的軟體設計範式，系統中的元件透過產生、偵測與回應事件來互動。相較於傳統的請求-回應模式，EDA 提供了更高的解耦性、擴展性與即時響應能力，是現代分散式系統的重要架構風格。

核心概念

事件的定義

事件（Event）代表系統中已經發生的一個有意義的狀態變更或事實：

  • 事件名稱：描述發生了什麼，如 UserRegistered、OrderShipped
  • 事件資料：包含事件的詳細資訊（payload）
  • 元資料：時間戳、來源、相關 ID 等
  • 不可變性：事件是已發生的事實，不可修改

EDA 核心元件

  • Event Producer（事件生產者）：偵測到狀態變更時發布事件
  • Event Channel（事件通道）：傳輸事件的中介（如 Message Broker）
  • Event Consumer（事件消費者）：訂閱並處理感興趣的事件
  • Event Router（事件路由器）：根據規則將事件導向正確的消費者

事件類型

  1. Domain Event（領域事件）：業務邏輯中的重要事件
  2. Integration Event（整合事件）：跨服務邊界的事件
  3. Notification Event（通知事件）：僅通知某事發生，不含完整資料
  4. Event-Carried State Transfer：事件包含完整狀態，消費者無需回查

通信模式

  • Event Notification：輕量通知，消費者需自行查詢詳情
  • Event-Carried State Transfer：事件攜帶完整資料
  • Event Sourcing：以事件序列作為系統的事實來源
  • CQRS：讀寫模型分離，搭配事件同步

事件處理模式

  • Simple Event Processing：單一事件觸發單一動作
  • Complex Event Processing（CEP）：多個事件組合偵測模式
  • Event Stream Processing：持續處理事件串流

關鍵比較

特性請求-回應事件驅動
耦合度高（直接呼叫）低（透過事件）
擴展性受限高（獨立擴展）
即時性同步非同步（近即時）
除錯難度低（線性流程）高（分散追蹤）
容錯性連鎖失敗隔離失敗
一致性強一致最終一致

編排 vs 編舞

面向Orchestration（編排）Choreography（編舞）
控制方式中央協調器去中心化
耦合度中（依賴協調器）低
可見性高（流程集中）低（分散各處）
複雜度流程複雜時較易管理簡單場景適用
單點故障協調器是潛在瓶頸無單點

應用場景

場景一：電商訂單流程

  • 用戶下單 → OrderCreated 事件發布
  • 庫存服務消費事件進行扣庫
  • 支付服務消費事件發起扣款
  • 通知服務消費事件發送確認信
  • 各服務獨立運作，互不影響

場景二：即時推薦系統

  • 用戶行為事件（瀏覽、點擊、購買）即時串流
  • CEP 引擎偵測行為模式
  • 即時更新用戶興趣模型
  • 推薦結果即時反映最新偏好

場景三：IoT 設備監控

  • 設備持續發送遙測事件
  • 串流處理計算即時指標
  • 異常偵測觸發警報事件
  • 自動化回應（如關閉設備、通知工程師）

設計注意事項

  1. 事件順序：需考慮亂序情況的處理策略
  2. 冪等消費：確保重複消費不會產生副作用
  3. 事件 Schema 演進：向後相容的版本化策略
  4. 分散式追蹤：需引入 Correlation ID 追蹤完整流程
  5. 補償機制：失敗場景需設計反向事件或 Saga
