Kafka 與串流處理

Apache Kafka 是當今最流行的分散式事件串流平台，被廣泛應用於即時資料管線、串流處理與事件驅動架構。Kafka 結合高吞吐量、低延遲與持久化儲存，成為大規模資料處理的基礎設施。本文深入探討 Kafka 的核心架構與串流處理實務。

核心概念

Kafka 基本元件

  • Topic：訊息的邏輯分類，類似資料庫中的表
  • Partition：Topic 的物理分片，是平行處理的基本單位
  • Offset：每筆訊息在 Partition 中的唯一序號
  • Producer：將訊息發布至 Topic 的客戶端
  • Consumer：從 Topic 訂閱並消費訊息的客戶端
  • Consumer Group：一組消費者共同消費 Topic，每個 Partition 只被群組中一個消費者處理
  • Broker：Kafka 叢集中的伺服器節點

Kafka 的設計哲學

  1. 持久化日誌：訊息以追加日誌形式儲存，可配置保留時間
  2. 分散式設計：水平擴展能力，透過分區實現平行處理
  3. 零拷貝傳輸：利用 OS 的 sendfile 系統呼叫提升傳輸效能
  4. 批次處理：將多筆訊息批次寫入/讀取，減少 I/O 次數
  5. 拉取模型（Pull-based）：消費者主動拉取訊息，控制消費速度

串流處理概念

  • Stream：無界的、持續產生的資料序列
  • Window：將無界串流切分為有限的時間或數量區段
  • Watermark：用於處理亂序事件的進度追蹤機制
  • State Store：串流處理中維護的本地狀態儲存

應用場景

場景一：即時數據管線

  • 將各系統的資料變更事件匯入 Kafka
  • 使用 Kafka Connect 連接各種資料源與目標
  • Kafka Streams 進行即時轉換與清洗
  • 輸出至資料湖或資料倉儲供分析使用

場景二：事件驅動微服務

  • 各微服務透過 Kafka Topic 通信
  • Outbox Pattern 確保資料庫與事件的一致性
  • Consumer Group 實現負載均衡消費
  • 利用 Kafka 的保留機制實現事件回溯

場景三：即時監控與警報

  • 系統指標與日誌串流進入 Kafka
  • 串流處理計算即時聚合指標（每分鐘錯誤率、P99 延遲）
  • 觸發閾值時發送警報
  • 同時寫入時序資料庫供歷史分析

生產環境注意事項

  1. 分區數量規劃：分區數決定最大平行度，後期增加需資料再平衡
  2. 消費者落後監控：Consumer Lag 是最關鍵的營運指標
  3. 冪等 Producer：啟用 enable.idempotence=true 避免重複寫入
  4. 合理設定 retention：平衡儲存成本與回溯需求
