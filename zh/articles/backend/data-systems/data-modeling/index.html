資料建模

資料模型設計

資料建模是將真實世界的業務概念轉化為資料庫可存儲結構的過程，是系統設計中最關鍵的前期決策之一。好的資料模型能讓應用程式開發事半功倍，查詢邏輯清晰且高效；差的資料模型則會導致業務邏輯複雜化、效能瓶頸和維護噩夢。資料建模不僅是技術問題，更是對業務領域的深入理解與抽象表達。

核心概念

概念模型（Conceptual Model） 是最高層次的抽象，使用實體關係圖（ERD）描述業務中的核心實體、屬性與關係，不涉及任何技術實作細節。這一階段的重點是與業務領域專家溝通，確保模型準確反映業務規則。

邏輯模型（Logical Model） 在概念模型基礎上加入更多細節：定義主鍵、外鍵、欄位類型、約束條件，並應用正規化規則消除冗餘。邏輯模型仍然與具體的資料庫技術無關。

實體模型（Physical Model） 是針對特定資料庫系統的具體實作，包含索引設計、分區策略、儲存引擎選擇、資料類型映射等效能相關的決策。

正規化（Normalization） 是減少資料冗餘和更新異常的系統化方法：

  • 第一正規式（1NF）：所有欄位值為原子值，不可再分割
  • 第二正規式（2NF）：非鍵屬性完全依賴於主鍵（消除部分依賴）
  • 第三正規式（3NF）：非鍵屬性不傳遞依賴於主鍵（消除傳遞依賴）
  • BCNF：每個決定因子都是候選鍵

反正規化（Denormalization） 是為了查詢效能有意引入冗餘的策略。常見手法包括預計算欄位（如訂單總額）、冗餘複製（如在訂單中儲存商品名稱快照）、合併表格（減少 JOIN）。反正規化需要在寫入複雜度和讀取效能之間取捨。

架構總覽

關聯式建模（Relational Modeling） 遵循正規化理論，將資料組織為相互關聯的表格：

  • 實體對應表格、屬性對應欄位、實例對應記錄
  • 一對一關係透過共享主鍵或外鍵實現
  • 一對多關係透過在「多」端加入外鍵實現
  • 多對多關係透過中間關聯表（Junction Table）實現
  • 繼承關係可透過單表繼承、類別表繼承或具體表繼承實現

NoSQL 建模的思維方式與關聯式截然不同——以查詢模式驅動設計（Query-Driven Design），而非以資料關係驅動：

  • 文件模型：將相關資料嵌套在同一文件中（Embedding），減少 JOIN 操作。適合一對少（One-to-Few）關係。當嵌套文件過大或需要獨立存取時，改用引用（Referencing）。
  • 鍵值模型：設計有意義的鍵結構（如 user:{id}:orders），透過鍵的命名空間組織資料。
  • 寬欄模型：以查詢的 WHERE 條件設計分區鍵和排序鍵，確保查詢能在單一分區內完成。

領域驅動設計（DDD） 與資料建模的結合：

  • 聚合根（Aggregate Root）對應資料儲存的事務邊界
  • 值物件（Value Object）可以內嵌或序列化存儲
  • 限界上下文（Bounded Context）指導微服務間的資料邊界

關鍵比較

面向關聯式建模文件建模圖建模
設計驅動力資料關係與正規化查詢模式與存取路徑實體關係與遍歷路徑
Schema 管理嚴格（Migration）彈性（Schema-on-Read）半結構化
關係處理JOIN 操作嵌套或引用直接邊遍歷
更新一致性事務保證文件內原子性依實作而定
適合場景結構穩定、複雜查詢結構多變、讀取密集高度互連資料

正規化程度的取捨：

正規化等級優點缺點適用場景
高度正規化（3NF+）無冗餘、更新簡單多表 JOIN、查詢慢OLTP 核心系統
適度正規化（2NF）平衡冗餘與效能部分更新異常一般業務系統
反正規化查詢極快、無 JOIN冗餘大、更新複雜OLAP、讀取密集
完全反正規化單次讀取即完整寫入複雜、空間浪費快取、搜尋索引

應用場景

電商系統建模：商品具有多變屬性（電子產品 vs 服飾），適合使用 EAV（Entity-Attribute-Value）模式或 PostgreSQL JSONB 欄位處理。訂單系統需要嚴格的正規化和事務保證。商品搜尋則將資料反正規化到 Elasticsearch。

社交網路建模：使用者基本資料用關聯式表儲存；社交關係（關注/好友）用圖模型或鄰接表表示；動態貼文用文件模型儲存（因為每種貼文類型的欄位不同）；時間線使用反正規化的物化視圖或快取。

內容管理系統（CMS）：多態內容（文章、影片、相簿）適合文件模型的彈性 Schema；分類標籤體系使用關聯式的多對多關係；版本歷史使用事件溯源模式記錄每次變更。

物聯網資料建模：感測器元資料使用關聯式表管理；時序量測資料使用時序資料庫（窄表或寬表設計）；設備拓撲關係使用圖模型表達。分區策略通常以時間為主鍵配合設備 ID。

微服務資料建模：每個服務擁有獨立的資料存儲（Database per Service），透過事件發布實現跨服務資料同步。Saga Pattern 處理分散式事務；CQRS 分離讀寫模型以最佳化各自的資料結構。
