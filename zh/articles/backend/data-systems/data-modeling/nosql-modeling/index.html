NoSQL 資料建模

NoSQL 資料建模打破了關聯式資料庫的正規化框架，改以查詢模式驅動設計。不同類型的 NoSQL 資料庫各有其最佳建模策略，理解這些差異是選擇正確方案的關鍵。

核心概念

查詢驅動設計（Query-Driven Design）

與關聯式建模「先設計結構再考慮查詢」不同，NoSQL 建模的核心原則是：

  • 先定義查詢模式：明確系統需要回答哪些問題
  • 反正規化為常態：為加速讀取而容許資料重複
  • 嵌入 vs 參照：根據存取模式決定資料結構
  • 最終一致性：接受短暫的資料不一致以換取可用性

文件模型（Document Model）

文件資料庫（如 MongoDB）的建模考量：

  • 嵌入式文件（Embedding）：將相關資料嵌入同一文件
    • 適用於一對少、經常一起讀取的資料
    • 單一文件操作具有原子性
  • 參照（Referencing）：透過 ID 參照其他文件
    • 適用於一對多、多對多關係
    • 資料獨立更新時較有利

鍵值模型（Key-Value Model）

鍵值資料庫（如 Redis、DynamoDB）的設計要點：

  • 存取模式：所有查詢透過主鍵或排序鍵
  • 複合鍵設計：將多個屬性編碼至鍵中（如 USER#123#ORDER#456）
  • 單表設計（Single Table Design）：DynamoDB 中將多種實體放入同一表
  • GSI 超載：透過全域次要索引支援多種查詢模式

寬列模型（Wide-Column Model）

寬列資料庫（如 Cassandra、HBase）的建模原則：

  • Partition Key：決定資料分佈到哪個節點
  • Clustering Key：決定分區內的排序方式
  • 資料建模即查詢建模：每個查詢對應一張表
  • 避免大分區：控制分區大小避免熱點

嵌入 vs 參照決策

考量因素偏好嵌入偏好參照
關係基數一對少一對多/多對多
讀取模式總是一起讀取經常獨立存取
更新頻率不常更新頻繁更新
文件大小嵌入後仍小可能超出限制
原子性需求需要原子操作可接受分別更新
資料重複可接受需要單一事實來源

應用場景

場景一：社群媒體動態牆

使用文件資料庫建模用戶貼文：

  • 將貼文內容、按讚數、前 N 則留言嵌入同一文件
  • 完整留言列表使用參照，支援分頁查詢
  • 使用者資料獨立存放，貼文中嵌入精簡版用戶資訊

場景二：IoT 時序資料

使用寬列資料庫儲存感測器資料：

  • Partition Key：device_id + 日期（控制分區大小）
  • Clustering Key：timestamp（時間排序）
  • 自動過期：設定 TTL 清除舊資料
  • 預計算聚合：定期彙總至摘要表

場景三：電商產品目錄

使用文件資料庫管理異質產品資訊：

  • 不同品類的產品有不同屬性（無固定 Schema）
  • 巢狀文件儲存規格、變體、圖片集
  • 以商品 ID 為主鍵，支援分類與搜尋索引

場景四：遊戲排行榜

使用 Redis Sorted Set：

  • 即時排名查詢 O(log N)
  • 範圍查詢取得 Top N
  • 支援分數更新與排名變動通知
