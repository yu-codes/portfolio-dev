節流控制

API 節流與速率限制的設計策略，涵蓋演算法選擇、分散式限流架構與實作最佳實踐

節流控制（Throttling）是保護後端系統免於過載的關鍵機制。透過限制單位時間內允許的請求數量，節流控制確保系統在高壓力下仍能穩定運作，同時公平地分配資源給所有使用者。

核心概念

速率限制 vs 節流 vs 背壓

機制說明作用位置行為
速率限制（Rate Limiting）限制請求速率上限API Gateway/應用層超限即拒絕
節流（Throttling）控制處理速度應用內部可排隊等待
背壓（Backpressure）上游感知下游壓力服務間動態調節發送速率

速率限制演算法

演算法原理優點缺點
固定窗口固定時間窗口內計數簡單邊界突增問題
滑動窗口日誌記錄每個請求的時間戳精確記憶體消耗大
滑動窗口計數器加權計算前後窗口平衡精確與效能略有近似
令牌桶固定速率產生令牌允許突發流量需維護令牌狀態
漏桶固定速率消費請求輸出平滑不允許突發

固定窗口的邊界問題

窗口1: [0s ─────── 60s]    窗口2: [60s ─────── 120s]
限制: 100 req/min

問題情境:
   第 55-60 秒: 100 個請求 ✓
   第 60-65 秒: 100 個請求 ✓
   → 10 秒內實際有 200 個請求通過！

滑動窗口解決此問題。

關鍵比較

令牌桶 vs 漏桶

維度令牌桶（Token Bucket）漏桶（Leaky Bucket）
突發處理允許（消耗累積的令牌）不允許（固定速率輸出）
平滑度允許短暫突發完全平滑
參數桶大小 + 填充速率桶大小 + 漏出速率
適用場景API 限流（允許偶爾突發）流量整形（需固定速率）
實作複雜度低低
典型應用AWS API Gateway網路 QoS

本地限流 vs 分散式限流

維度本地限流分散式限流（Redis）
精確度低（每個節點獨立計數）高（全域統一計數）
延遲極低（記憶體操作）稍高（網路往返）
可用性高（不依賴外部）依賴 Redis 可用性
一致性最終一致接近強一致
適用場景粗略保護精確限額控制

超限回應策略

策略HTTP 狀態碼行為適用場景
硬拒絕429立即拒絕嚴格限流
排隊等待200（延遲回應）等待令牌可用平滑處理
降級回應200返回快取/簡化結果維持可用性
重試提示429 + Retry-After告知何時可重試API 友好

應用場景

API 速率限制設計

為 SaaS API 設計多層級的限流方案：

層級免費方案專業方案企業方案
每分鐘請求606006,000
每日請求1,00050,000無限
並行請求550500
突發容忍無2x 短期5x 短期

回應標頭設計：

  • X-RateLimit-Limit：限額
  • X-RateLimit-Remaining：剩餘額度
  • X-RateLimit-Reset：重置時間
  • Retry-After：建議重試時間

登入接口的特殊限流

防止暴力破解的漸進式限流：

失敗次數等待時間額外動作
1-3 次無正常
4-5 次5 秒顯示 CAPTCHA
6-10 次30 秒通知帳號擁有者
> 10 次15 分鐘暫時鎖定帳號

微服務的限流協調

在微服務架構中，各服務需要協調限流策略：

  • 上游限流：API Gateway 統一限流
  • 下游保護：每個服務設定自身的容量上限
  • 熔斷機制：當下游服務超載時，上游主動停止請求
  • 優先權佇列：VIP 請求優先處理，普通請求可排隊
