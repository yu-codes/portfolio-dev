XSS、CSRF 與注入攻擊

深入解析 XSS、CSRF 與各類注入攻擊的原理、分類與全面防禦策略

跨站腳本（XSS）、跨站請求偽造（CSRF）與注入攻擊是 Web 應用中最常見且危害最大的安全威脅。理解這些攻擊的原理與防禦方法，是每位後端工程師的必備知識。

核心概念

XSS（Cross-Site Scripting）

XSS 攻擊讓攻擊者將惡意腳本注入到其他使用者瀏覽的頁面中：

XSS 類型儲存位置觸發方式危害程度
儲存型（Stored）伺服器資料庫受害者瀏覽含惡意腳本的頁面高（影響所有訪客）
反射型（Reflected）URL 參數受害者點擊惡意連結中（需誘導點擊）
DOM 型用戶端 DOMJavaScript 直接操作 DOM中（不經伺服器）

XSS 攻擊可能造成：

  • 竊取 Cookie 和 Session Token
  • 鍵盤記錄（Keylogging）
  • 網頁內容篡改（Defacement）
  • 導向釣魚網站

CSRF（Cross-Site Request Forgery）

CSRF 利用使用者已認證的身份，在使用者不知情的情況下執行非預期操作：

攻擊條件：

  1. 使用者已登入目標網站（有有效 Session）
  2. 目標網站僅依賴 Cookie 進行身份驗證
  3. 攻擊者能預測請求的所有參數

SQL 注入

SQL 注入的危害等級：

等級能力範例
資料讀取存取任意資料表UNION SELECT
資料修改竄改或刪除資料; DROP TABLE
認證繞過以任意身份登入' OR 1=1 --
系統控制執行系統指令xp_cmdshell（MSSQL）
檔案存取讀寫伺服器檔案LOAD_FILE()（MySQL）

架構總覽

防禦層次設計

輸入層 ─── 輸入驗證（白名單）、類型檢查
  ↓
處理層 ─── 參數化查詢、安全 API 呼叫
  ↓
輸出層 ─── 輸出編碼（Context-aware）、CSP
  ↓
傳輸層 ─── CSRF Token、SameSite Cookie

輸出編碼的上下文

不同的輸出位置需要不同的編碼方式：

輸出位置編碼方式範例
HTML 內容HTML Entity 編碼< → &lt;
HTML 屬性屬性編碼" → &quot;
JavaScriptJavaScript 編碼' → \x27
URL 參數URL 編碼空格 → %20
CSSCSS 編碼特殊字元 → \XXXXXX

防禦方式效果實作難度適用場景

CSRF 防禦方案比較

方案原理優點缺點
Synchronizer Token表單中嵌入隨機 Token成熟、可靠需伺服器保存狀態
Double Submit CookieCookie 值與表單值比對無狀態需防止子域名注入

應用場景

RESTful API 的 CSRF 防禦

對於前後端分離的 SPA 應用：

  1. Token-Based 認證：使用 JWT 存於 localStorage（天然免疫 CSRF）
  2. 如果使用 Cookie：
    • 設定 SameSite=Strict 或 SameSite=Lax
    • 實施 Double Submit Cookie 模式
    • 驗證 Origin 標頭

富文本編輯器的 XSS 防禦

允許使用者輸入 HTML 的場景（如部落格系統）：

  1. 白名單過濾：只允許安全的 HTML 標籤和屬性
  2. 使用專業淨化庫：如 DOMPurify、Bleach
  3. CSP 限制：禁止 inline script 和 eval
  4. 沙箱渲染：在 iframe sandbox 中渲染使用者內容

防禦注入的程式碼範例原則

安全的查詢構建原則：

  • 永遠使用參數化查詢，絕不拼接使用者輸入
  • 動態表名/欄位名使用白名單驗證
  • 限制查詢回傳的資料量
  • 使用唯讀帳號執行查詢操作
