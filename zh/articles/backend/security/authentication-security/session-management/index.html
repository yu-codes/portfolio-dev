Session 管理

Web 應用 Session 管理的安全設計，涵蓋 Session 生命週期、安全配置、分散式 Session 與常見攻擊防禦

Session 管理是 Web 應用認證系統的核心機制。它維持使用者的登入狀態，讓無狀態的 HTTP 協議能夠識別連續的請求來自同一使用者。Session 管理的安全性直接關係到使用者帳號安全和應用整體安全。

核心概念

Session 的生命週期

階段操作安全要求
建立使用者認證成功後產生 Session使用安全隨機數、足夠長度
維持每次請求攜帶 Session ID安全傳輸、防竊取
更新權限變更時重新產生防止 Session Fixation
銷毀使用者登出或逾時徹底清除、無法重用

Session ID 的安全要求

要求說明建議值
長度足夠長以抵抗暴力猜測≥ 128 bits
隨機性使用密碼學安全的亂數產生器CSPRNG
唯一性不同使用者不應產生相同 ID碰撞機率極低
不可預測性無法從已知 ID 推算其他 ID無序列模式

Cookie 安全屬性

屬性作用建議設定
Secure僅透過 HTTPS 傳送必須啟用
HttpOnlyJavaScript 無法存取必須啟用
SameSite限制跨站傳送Strict 或 Lax
Domain限制 Cookie 的有效域最小範圍
Path限制 Cookie 的有效路徑最小範圍
Max-Age/ExpiresCookie 的有效期限合理的短期限

分散式 Session 方案

在多伺服器環境中，Session 需要共享：

方案原理優點缺點
Sticky Session負載均衡固定使用者到同一伺服器簡單負載不均、單點故障
Session 複製所有節點同步 Session 資料高可用網路開銷大
集中式儲存Redis/DB 統一存放一致性好依賴外部服務
Token-Based狀態存於 Token 中無狀態、易擴展Token 較大、無法主動撤銷

關鍵比較

Session vs Token（JWT）

維度Server-Side SessionJWT Token
狀態存放伺服器用戶端
可擴展性需共享 Session Store天然無狀態
撤銷能力立即撤銷需黑名單機制
大小Cookie 只有 ID（小）Token 包含資料（較大）
安全性Session 劫持風險Token 洩漏風險
適用場景傳統 Web 應用SPA、微服務、行動端

Session 攻擊與防禦

攻擊方式原理防禦方法
Session 劫持竊取他人 Session IDHTTPS、HttpOnly、短期限
Session 固定強制使用者使用攻擊者已知的 ID登入後重新產生 Session ID
Session 預測猜測或推算有效的 Session ID安全隨機數、足夠長度
CSRF利用已認證的 Session 發送請求CSRF Token、SameSite
XSS 竊取透過 XSS 讀取 CookieHttpOnly、CSP

應用場景

安全的 Session 實作檢查清單

  • 使用框架內建的 Session 管理，不自行實作
  • Session ID 使用密碼學安全隨機數產生
  • 認證成功後重新產生 Session ID
  • 設定合理的閒置逾時（如 30 分鐘）
  • 設定絕對逾時（如 8 小時）
  • 登出時在伺服器端銷毀 Session
  • Cookie 設定 Secure + HttpOnly + SameSite
  • 敏感操作前要求重新認證

多裝置 Session 管理

現代應用需要支援使用者在多個裝置上同時登入：

策略說明適用場景
單一 Session新登入使舊 Session 失效高安全需求（如銀行）
多重 Session允許同時登入多個裝置一般應用
裝置管理使用者可查看和撤銷各裝置的 Session推薦做法
異常偵測偵測異常登入並要求驗證進階安全

Session 監控與告警

應監控的 Session 相關指標：

  • 同時活躍 Session 數量（異常增長可能是攻擊）
  • Session 建立/銷毀速率
  • 同一帳號的並行 Session 數
  • Session 使用的異常模式（如地理位置突變）
  • Session Store 的回應時間與錯誤率
