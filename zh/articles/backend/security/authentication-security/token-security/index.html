Token 安全性

Token-Based 認證已成為現代 Web 應用和 API 的主流認證方式。JWT（JSON Web Token）是其中最廣泛使用的格式，它將使用者身份資訊編碼在 Token 中，實現無狀態認證。然而，Token 的不當使用也會帶來嚴重的安全風險。

核心概念

JWT 結構

JWT 由三部分組成，以 . 分隔：

部分內容編碼方式
Header演算法類型、Token 類型Base64URL
Payload使用者資訊（Claims）Base64URL
Signature簽名驗證HMAC 或 RSA/ECDSA

常見 Claims

Claim名稱說明
issIssuerToken 發行者
subSubjectToken 主題（通常是使用者 ID）
audAudienceToken 接收方
expExpiration過期時間
nbfNot Before生效時間
iatIssued At發行時間
jtiJWT IDToken 唯一識別符

Token 類型與用途

Token 類型有效期用途儲存位置
Access Token短（5-30 分鐘）API 存取授權記憶體 / Cookie
Refresh Token長（7-30 天）換取新的 Access TokenHttpOnly Cookie
ID Token短身份資訊傳遞（OIDC）用戶端
API Key長期服務對服務認證環境變數

簽名演算法選擇

演算法類型安全性適用場景
HS256對稱中單一服務，簡單場景
HS384/512對稱中-高需更強 HMAC
RS256非對稱高多服務驗證、公開金鑰
ES256非對稱高效能敏感場景
EdDSA非對稱最高新系統推薦

關鍵比較

JWT 常見安全漏洞

漏洞攻擊方式防禦方法
alg: none將演算法設為 none 繞過簽名明確指定允許的演算法
金鑰混淆用公鑰作為 HMAC 密鑰嚴格區分演算法類型
弱密鑰暴力破解 HMAC 密鑰使用 ≥ 256 bits 的隨機密鑰
Token 注入修改 Payload 中的權限驗證簽名完整性
無過期時間Token 永久有效必須設定 exp
過長有效期Token 洩漏後長時間可用短期 Access + Refresh 模式
敏感資料外洩Payload 中含敏感資訊Payload 僅放識別符

Token 儲存位置比較

儲存位置XSS 風險CSRF 風險適用場景
localStorage高（JS 可存取）無不推薦
sessionStorage高（JS 可存取）無臨時性使用
HttpOnly Cookie低需防禦推薦
記憶體變數低（頁面刷新丟失）無SPA 搭配 Refresh Token
Web Worker低無進階安全需求

Access Token 有效期策略

有效期安全性使用者體驗適用場景
1-5 分鐘極高需頻繁刷新金融系統
15-30 分鐘高平衡一般應用
1-2 小時中較少中斷內部系統
> 24 小時低極佳不推薦

應用場景

Token 撤銷策略

JWT 的無狀態特性使得撤銷較為困難，常見方案：

方案機制優點缺點
短期 Token依賴過期機制簡單無法立即撤銷
黑名單記錄被撤銷的 Token精確撤銷需要儲存狀態
Token 版本使用者資料中記錄版本號可批量撤銷需查詢資料庫
事件驅動發布撤銷事件到所有服務即時性好複雜度高

OAuth 2.0 Token 安全

OAuth 2.0 流程中的 Token 安全考量：

  • Authorization Code + PKCE：防止 Code 被攔截
  • Token Binding：將 Token 綁定到特定用戶端
  • DPoP（Demonstration of Proof-of-Possession）：證明 Token 持有者
  • Token Introspection：資源伺服器向授權伺服器驗證 Token

微服務間的 Token 傳遞

  • Token 轉發：直接傳遞使用者的 Access Token
  • Token 交換：用使用者 Token 換取服務專用 Token
  • 服務帳號 Token：服務間使用獨立的認證
  • mTLS：透過雙向 TLS 進行服務認證
