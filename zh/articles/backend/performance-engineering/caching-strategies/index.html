快取策略

全面解析後端快取策略，包括快取模式、失效策略、多級快取架構與常見陷阱

快取（Cache）是提升系統效能最直接有效的手段之一。透過將熱點資料存放在存取速度更快的儲存媒介中，可以大幅減少對後端資料庫或外部服務的存取次數，降低延遲並提升吞吐量。然而，快取也帶來了資料一致性的挑戰。

核心概念

快取的基本原則

  • 時間局部性（Temporal Locality）：最近被存取的資料，短期內很可能再次被存取
  • 空間局部性（Spatial Locality）：被存取資料附近的資料，也可能被存取
  • 80/20 法則：通常 20% 的資料承載了 80% 的存取流量

快取命中率

快取命中率是衡量快取效果的核心指標：

命中率效果評估建議行動
> 95%優秀維持現狀
80-95%良好觀察是否可優化
60-80%一般檢查快取策略與容量
< 60%較差重新設計快取方案

快取讀寫模式

模式讀取流程寫入流程一致性適用場景
Cache-Aside先查快取，未命中查DB後填入更新DB，刪除快取最終一致通用場景
Read-Through快取層代理DB讀取-最終一致讀多寫少
Write-Through-同步寫入快取和DB強一致一致性要求高
Write-Behind-先寫快取，非同步寫DB弱一致寫入密集
Refresh-Ahead預測性地提前刷新快取-較好可預測存取模式

架構總覽

多級快取架構

請求 → L1 (本地記憶體) → L2 (分散式快取) → L3 (資料庫)
       命中率:高           命中率:中          命中率:100%
       延遲:<1ms          延遲:1-5ms        延遲:10-100ms
       容量:小            容量:大            容量:最大

快取拓撲

  • 本地快取：Caffeine、Guava Cache，存在於應用進程內
  • 分散式快取：Redis、Memcached，獨立部署的快取服務
  • CDN 快取：CloudFront、Cloudflare，邊緣節點快取
  • 資料庫快取：MySQL Query Cache、PostgreSQL Buffer Pool

關鍵比較

Redis vs Memcached

維度RedisMemcached
資料結構豐富（String、Hash、List、Set、ZSet）僅 Key-Value
持久化支援（RDB、AOF）不支援
叢集模式Redis Cluster用戶端分片
記憶體效率中等較高（slab 分配）
功能性Pub/Sub、Lua、Stream純快取
適用場景複雜資料操作、需持久化簡單快取、極致效能

快取失效策略比較

策略機制優點缺點
TTL（Time-To-Live）設定固定過期時間簡單、可預測可能讀到舊資料
LRU（Least Recently Used）淘汰最久未使用的適應存取模式實作複雜度中等
LFU（Least Frequently Used）淘汰最少使用的保留熱點資料新資料難以上位
主動失效資料變更時刪除快取一致性好需維護失效邏輯
版本號Key 中包含版本號無需刪除操作Key 空間浪費

應用場景

快取穿透、擊穿、雪崩

問題描述解決方案
快取穿透查詢不存在的資料，每次都打到DB布隆過濾器、空值快取
快取擊穿熱點Key過期，大量請求同時打到DB互斥鎖、永不過期+背景刷新
快取雪崩大量Key同時過期，DB瞬間承受高壓過期時間加隨機偏移、多級降級

資料一致性保障

雙寫一致性的常見方案：

  1. 先更新DB，再刪除快取（推薦）

    • 配合消息佇列重試確保快取被刪除
    • 設定較短 TTL 作為兜底

  2. 延遲雙刪

    • 先刪快取 → 更新DB → 延遲一段時間再刪快取
    • 解決並行讀寫導致的不一致

  3. 訂閱 Binlog

    • 透過 Canal 等工具監聽DB變更
    • 自動觸發快取更新，解耦業務程式碼

快取預熱策略

系統啟動或新快取節點加入時的預熱方法：

  • 定時任務預熱：離峰時段預先載入熱點資料
  • 流量回放預熱：使用歷史流量模式填充快取
  • 漸進式預熱：逐步將流量導向新節點，讓快取自然填充
