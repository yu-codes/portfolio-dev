效能分析與除錯

效能分析（Profiling）是識別系統瓶頸的科學方法。與猜測式優化不同，Profiling 透過實際數據揭示程式的執行熱點，引導工程師將優化資源集中在真正重要的地方。正如 Donald Knuth 所言：「過早的優化是萬惡之源」，而 Profiling 正是幫助我們在對的時機做對的優化。

核心概念

Profiling 的類型

類型測量目標常用工具開銷
CPU ProfilingCPU 時間分布perf、async-profiler、pprof低-中
Memory Profiling記憶體分配與洩漏jmap、Valgrind、heaptrack中
I/O Profiling磁碟與網路 I/Ostrace、BPF低
Lock Profiling鎖競爭與等待JFR、async-profiler低
Allocation Profiling物件分配頻率JFR、dotMemory中

取樣 vs 儀器化

方式原理優點缺點
取樣（Sampling）定時中斷，記錄當前堆疊低開銷、適合生產環境可能遺漏短暫呼叫
儀器化（Instrumentation）在每個方法入口/出口插入計時精確、不遺漏高開銷、影響效能

在生產環境中，取樣式 Profiling 是首選，因為其開銷通常低於 5%。

效能分析的方法論

  1. 建立基準線：在優化前量測當前效能
  2. 識別瓶頸：使用 Profiling 工具找出熱點
  3. 形成假設：分析瓶頸原因
  4. 實施優化：針對性改善
  5. 驗證結果：對比優化前後的數據

架構總覽

火焰圖（Flame Graph）

火焰圖是視覺化 CPU Profiling 結果的最佳工具：

┌──────────────────────────────────────────────────┐
│                    main()                          │ ← 根節點
├──────────────────────┬───────────────────────────┤
│    handleRequest()    │      processQueue()       │ ← 寬度代表時間佔比
├──────────┬───────────┤───────────┬───────────────┤
│ parseJSON│ queryDB() │ serialize │  sendResponse  │
├──────────┤───────┬───┤           │               │
│          │ index │sort│           │               │ ← 越深代表呼叫越深
└──────────┴───────┴───┴───────────┴───────────────┘
  • X 軸：函式佔用的 CPU 時間比例（越寬佔越多時間）
  • Y 軸：呼叫堆疊深度
  • 顏色：通常用於區分不同類型（使用者空間/核心空間）

生產環境 vs 開發環境 Profiling

維度生產環境開發環境

安全考量注意敏感資料無限制

應用場景

記憶體洩漏診斷

Java 記憶體洩漏的診斷流程：

  1. 觀察症狀：Old Gen 持續增長不回收
  2. 取得 Heap Dump：jmap -dump:format=b,file=heap.hprof <pid>
  3. 分析支配樹：用 MAT 找出佔用最多記憶體的物件
  4. 追蹤 GC Root：確認哪些參考阻止了垃圾回收
  5. 定位程式碼：找出建立但未釋放物件的程式碼路徑

GC 調優

GC 類型特點適用場景
G1 GC平衡延遲與吞吐量通用伺服器應用
ZGC極低停頓（< 1ms）低延遲要求
Shenandoah併發壓縮大堆記憶體
Parallel GC高吞吐量批次處理

持續性效能監控

將 Profiling 整合到 CI/CD 流程中：

  • 效能基準測試：每次提交自動執行效能測試
  • 效能回歸偵測：與基準線比較，超過閾值則失敗
  • 持續 Profiling：Pyroscope、Grafana Phlare 等工具提供生產環境的持續分析
