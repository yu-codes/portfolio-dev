延遲優化

深入探討後端系統延遲優化的核心策略，包括網路延遲、應用層延遲與資料庫延遲的分析與改善方法

延遲（Latency）是衡量系統回應速度的關鍵指標，直接影響使用者體驗與系統吞吐量。在現代分散式架構中，一次請求可能穿越多個服務節點，每個節點都會貢獻延遲。理解延遲的來源並系統性地進行優化，是後端工程師的核心能力之一。

核心概念

延遲的組成

一次完整的請求延遲可拆解為以下幾個階段：

延遲來源說明典型範圍
DNS 解析域名轉換為 IP 位址1-100ms
TCP 握手建立連線的三次握手1-50ms
TLS 協商加密連線建立10-100ms
網路傳輸資料封包在網路中傳遞1-200ms
應用處理業務邏輯執行時間1-1000ms
資料庫查詢資料存取時間1-500ms
序列化/反序列化資料格式轉換0.1-10ms

P50、P95、P99 的意義

延遲通常以百分位數（Percentile）表示：

  • P50（中位數）：50% 的請求在此時間內完成，代表典型體驗
  • P95：95% 的請求在此時間內完成，代表大多數使用者的體驗
  • P99：99% 的請求在此時間內完成，代表尾端延遲（Tail Latency）

尾端延遲特別重要，因為在高流量系統中，即使只有 1% 的請求變慢，也意味著大量使用者受到影響。

架構總覽

延遲優化的分層策略

用戶端 → CDN → 負載均衡器 → 應用伺服器 → 快取層 → 資料庫
  ↓         ↓         ↓            ↓           ↓        ↓
DNS優化   邊緣快取   連線池     非同步處理    多級快取   索引優化

每一層都有對應的優化手段，關鍵在於找出瓶頸所在，而非盲目優化所有環節。

網路層優化

  • 連線重用（Connection Pooling）：避免反覆建立 TCP/TLS 連線
  • HTTP/2 多工：單一連線上並行處理多個請求
  • 就近部署：將服務部署在靠近使用者的區域
  • 壓縮傳輸：使用 gzip/brotli 減少傳輸資料量

應用層優化

  • 非同步處理：將非關鍵路徑的操作移至背景執行
  • 批次處理：合併多次小請求為單次大請求
  • 預計算：在請求到達前預先準備好結果
  • 懶載入：延遲載入非必要資源

關鍵比較

同步 vs 非同步處理

維度同步處理非同步處理
延遲感知使用者等待完整處理完成快速回應，背景完成
實作複雜度低中到高
錯誤處理直接回傳錯誤需要補償機制
適用場景即時查詢、交易確認郵件發送、報表產生
資源利用佔用執行緒等待釋放資源處理其他請求

快取位置比較

快取位置延遲改善一致性風險維護成本
瀏覽器快取極大（無需網路往返）高低
CDN 快取大（就近回應）中中
應用記憶體快取中（省去資料庫查詢）中中
分散式快取（Redis）中低高

應用場景

電商系統的延遲優化

電商平台的商品頁面通常需要組合多個資料來源：

  1. 商品基本資訊：從快取讀取（< 5ms）
  2. 即時價格：從定價服務查詢（< 20ms）
  3. 庫存狀態：從庫存服務查詢（< 15ms）
  4. 推薦商品：從推薦引擎取得（< 50ms）

優化策略：

  • 並行請求獨立的資料來源，而非串行呼叫
  • 對變動頻率低的資料使用較長的快取時間
  • 設定合理的逾時（Timeout），避免某個慢服務拖累整體回應

API Gateway 的延遲預算

為每個下游服務設定延遲預算（Latency Budget），例如：

  • 總延遲預算：200ms
  • 認證服務：30ms
  • 業務邏輯：100ms
  • 資料庫查詢：50ms
  • 序列化回應：20ms

當某個環節超出預算，觸發告警並啟動降級策略。
