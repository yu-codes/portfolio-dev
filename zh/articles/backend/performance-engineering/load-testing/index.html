負載測試

負載測試的方法論與實踐，包括測試類型、工具選擇、腳本設計與結果分析

負載測試（Load Testing）是在正式上線前驗證系統效能的關鍵手段。透過模擬真實或超出預期的流量，我們可以發現系統的效能瓶頸、確定最大承載能力，並驗證在壓力下系統是否能優雅降級而非崩潰。

核心概念

測試類型分類

測試類型目的負載模式持續時間
基準測試（Baseline）建立效能基準線低負載短
負載測試（Load）驗證目標負載下的效能預期負載中
壓力測試（Stress）找出系統極限逐步增加至崩潰中
峰值測試（Spike）驗證突發流量的應對瞬間大量增加短
浸泡測試（Soak）發現長時間運行問題中等負載長（數小時）
容量測試（Capacity）規劃未來擴展不同負載等級中

關鍵指標

負載測試需要關注的核心指標：

指標說明健康閾值（參考）
回應時間（P50/P95/P99）請求處理延遲依 SLA 定義
吞吐量（RPS）每秒處理請求數≥ 預期流量 × 2
錯誤率失敗請求比例< 0.1%
並行使用者數同時在線數≥ 預期峰值
CPU 使用率處理器負載< 70%
記憶體使用率記憶體佔用< 80%
連線數資料庫/服務連線< 連線池上限的 80%

負載模型設計

真實的負載模型應考慮：

  • 使用者行為模式：不同 API 的呼叫比例
  • 思考時間（Think Time）：使用者操作間的停頓
  • 資料分布：熱點資料 vs 長尾資料
  • 時間分布：尖峰時段 vs 離峰時段

測試環境要求

  • 環境隔離：測試環境應與生產環境隔離
  • 環境對等：硬體規格、網路拓撲應盡量與生產一致
  • 資料準備：測試資料量應與生產環境相當
  • 監控完備：需能觀察系統各層的指標

開源 vs 商業方案

維度開源方案商業方案（如 LoadRunner）
成本免費高（授權費）
協議支援HTTP/gRPC 為主幾乎所有協議
分散式管理需自建開箱即用
報表功能基本（需整合 Grafana）豐富
學習曲線中等較高
客製化高度靈活受限於框架

應用場景

電商大促壓測方案

針對雙 11 等大促活動的壓測策略：

  1. 流量預估：基於歷史數據預估峰值（通常為日常 10-50 倍）
  2. 場景設計：
    • 商品瀏覽：60% 流量
    • 加入購物車：20% 流量
    • 下單支付：15% 流量
    • 其他操作：5% 流量
  3. 漸進式加壓：
    • 第一輪：預期峰值的 50%
    • 第二輪：預期峰值的 100%
    • 第三輪：預期峰值的 150%（驗證彈性）
  4. 降級驗證：確認限流、熔斷等保護機制正常運作

CI/CD 整合效能門檻

將負載測試整合到持續交付流程中：

代碼提交 → 單元測試 → 整合測試 → 部署到測試環境 → 負載測試 → 效能門檻檢查 → 部署到生產

效能門檻定義範例：

指標通過條件失敗動作
P95 延遲< 200ms阻擋部署
錯誤率< 0.5%阻擋部署
P99 延遲< 500ms告警
吞吐量下降< 10%告警

測試結果分析

分析負載測試結果時的重點：

  • 拐點分析：找出效能開始劣化的負載等級
  • 資源相關性：將效能下降與資源使用率關聯
  • 瓶頸定位：確定是 CPU、記憶體、I/O 還是網路受限
  • 趨勢比較：與歷史測試結果對比，發現效能回歸
