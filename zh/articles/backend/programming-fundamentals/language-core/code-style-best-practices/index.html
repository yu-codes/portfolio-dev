良好的程式碼風格不僅是美學問題，更直接影響程式碼的可讀性、可維護性與團隊協作效率。本文從命名慣例、設計原則到各語言的最佳實踐，系統性地探討如何寫出高品質的程式碼。

核心概念

程式碼品質的維度

維度描述衡量方式
可讀性他人理解程式碼的難易度程式碼審查時間
可維護性修改程式碼的成本修改所需變動的檔案數
可測試性撰寫測試的難易度測試覆蓋率
一致性風格是否統一Linter 警告數量
簡潔性是否沒有冗餘程式碼行數/功能比

架構總覽

SOLID 原則

原則全稱核心思想違反的徵兆
S單一職責一個類別只有一個改變的理由類別過大、職責混雜
O開放封閉對擴展開放，對修改封閉頻繁修改既有程式碼
LLiskov 替換子類別可替代父類別型別檢查 (instanceof)
I介面隔離客戶端不依賴不需要的介面空實作方法
D依賴反轉依賴抽象而非具體直接 new 依賴物件

函式設計原則

原則描述建議
單一抽象層次函式內同一抽象層次高層邏輯不混底層細節
參數數量越少越好不超過 3 個，多則用物件
函式長度保持簡短20-30 行以內
無副作用純函式優先明確標示有副作用的函式
命令查詢分離做事或回傳，不要同時修改狀態的不回傳值

程式碼氣味 (Code Smells)

氣味描述重構手法
過長函式函式超過 30 行提取函式
過大類別類別職責過多提取類別
重複程式碼相似邏輯出現多處提取共用方法
過多參數函式參數過多引入參數物件
特性忌妒過度存取他類資料搬移方法
散彈式修改一個改變需修改多處內聯/搬移
