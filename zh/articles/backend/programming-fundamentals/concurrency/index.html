並行與非同步程式設計

並行程式設計是現代軟體工程中最具挑戰性也最重要的領域之一。從網頁伺服器同時處理數千個連線，到資料管線並行處理海量資料，幾乎所有後端系統都依賴並行來達到效能與可用性要求。本文深入探討並行的核心概念、程式設計模型及實踐策略。

核心概念

並行 vs. 平行

這兩個概念經常被混淆，但有本質區別：

並行 (Concurrency)：多個任務在邏輯上同時進行，可能在單核心上交錯執行。關注的是程式的結構——如何將工作拆分為可獨立推進的單元。

平行 (Parallelism)：多個任務在物理上同時執行於多個處理器核心。關注的是執行——如何利用硬體資源加速計算。

Rob Pike 的經典描述：「Concurrency is about dealing with lots of things at once. Parallelism is about doing lots of things at once.」

執行緒與程序

程序 (Process)：擁有獨立的記憶體空間，透過 IPC（管道、Socket、共享記憶體）通訊。隔離性好但切換開銷大。

執行緒 (Thread)：共享程序的記憶體空間，切換開銷較小，但需要同步機制防止資料競爭。

輕量級執行緒 / 協程：由語言執行時期管理的使用者態執行單元。Go 的 goroutine（初始 2KB 棧）、Kotlin 的 coroutine、Java 21 的 Virtual Thread 都屬於此類。可以高效地建立數十萬甚至百萬個。

同步原語

當多個執行流需要存取共享狀態時，需要同步機制：

  • 互斥鎖 (Mutex)：確保同一時間只有一個執行緒存取臨界區
  • 讀寫鎖 (RWLock)：允許多個讀取者或一個寫入者
  • 信號量 (Semaphore)：控制同時存取資源的執行緒數量
  • 條件變數 (Condition Variable)：讓執行緒等待特定條件成立
  • 原子操作 (Atomic)：透過硬體指令（CAS）實現無鎖同步

非同步程式設計模式

非同步編程避免執行緒在等待 I/O 時閒置浪費：

  • Callback（回呼）：最基礎但容易形成回呼地獄
  • Promise/Future：代表未來會完成的值，支援鏈式組合
  • async/await：以同步風格撰寫非同步程式碼，編譯器轉換為狀態機
  • Reactive Streams：處理非同步資料流，內建背壓（Backpressure）機制

架構總覽

並行程式設計的主要模型可以從共享狀態與訊息傳遞兩個維度來理解：

共享記憶體模型 (Shared Memory)：執行緒透過共享變數通訊，使用鎖或原子操作保護。程式設計直覺，但容易出錯——死鎖、活鎖、資料競爭都是常見問題。

訊息傳遞模型 (Message Passing)：執行單元之間不共享狀態，透過發送和接收訊息通訊。Go 的 Channel、Erlang/Elixir 的 Actor 模型都屬於此類。CSP (Communicating Sequential Processes) 是其理論基礎。

Actor 模型：每個 Actor 擁有私有狀態，只能透過非同步訊息與其他 Actor 互動。Erlang/OTP、Akka、Microsoft Orleans 都實作了 Actor 模型。天然適合分散式系統。

事件循環模型 (Event Loop)：單執行緒配合非阻塞 I/O 和事件佇列。Node.js、Python asyncio、Nginx 都採用此模型。適合 I/O 密集型工作負載，但不適合 CPU 密集型計算。

Fork-Join 模型：將任務遞迴地分割為子任務並行執行，最後合併結果。Java 的 ForkJoinPool、Rust 的 rayon 都實作了此模型。適合資料平行處理。

應用場景

場景一：高並發 Web 伺服器

一個 Web 伺服器需要同時處理數千個客戶端連線。使用 Go 的做法是為每個連線啟動一個 goroutine；使用 Node.js 則利用事件循環非阻塞地處理所有連線；使用 Java 可以選擇 Virtual Thread 或 Netty 的事件驅動模型。選擇取決於工作負載是 I/O 密集還是 CPU 密集。

場景二：避免死鎖

經典的哲學家用餐問題展示了死鎖的形成條件。實務中避免死鎖的策略包括：統一鎖的取得順序、使用 try-lock 加上逾時、偏好無鎖資料結構，或者從設計上避免持有多把鎖。Go 的 race detector 和 Rust 的型別系統可以在早期發現潛在的競爭條件。

場景三：生產者-消費者模式

日誌收集系統中，多個生產者執行緒寫入日誌，消費者執行緒批次處理並寫入儲存。使用有界佇列（Bounded Queue）作為緩衝，配合背壓機制——當佇列滿時生產者降速，防止記憶體溢出。Go 的 buffered channel 天然支援此模式。

場景四：並行資料處理管線

ETL 管線需要從多個資料源讀取、轉換並載入資料。設計為多階段管線，每個階段並行處理：讀取階段並行存取多個來源、轉換階段並行處理各記錄、載入階段批次寫入目標。使用 Fan-out/Fan-in 模式控制並行度。
