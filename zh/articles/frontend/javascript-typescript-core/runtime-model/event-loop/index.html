事件迴圈

深入剖析 JavaScript 事件迴圈機制，理解瀏覽器與 Node.js 的執行排程原理

事件迴圈（Event Loop）是 JavaScript 執行時期的核心調度機制。它讓單執行緒的 JavaScript 能夠處理非同步操作，協調呼叫堆疊、任務佇列與渲染之間的關係，是理解 JavaScript 非同步行為的關鍵。

核心概念

事件迴圈的組成元件

元件功能特性
呼叫堆疊（Call Stack）追蹤目前執行的函式LIFO，同步執行
任務佇列（Task Queue）存放巨任務回呼FIFO，每次迴圈取一個
微任務佇列（Microtask Queue）存放微任務回呼FIFO，清空後才進行下一步
Web APIs/Host APIs處理非同步操作瀏覽器/Node.js 提供
渲染管線處理畫面更新約 60fps（16.67ms）

事件迴圈的執行順序

每一輪事件迴圈（tick）的執行順序：

1. 執行呼叫堆疊中的同步程式碼
2. 堆疊清空後，清空所有微任務佇列
3. 檢查是否需要渲染更新
   3a. 執行 requestAnimationFrame 回呼
   3b. 執行佈局與繪製
4. 從任務佇列取出一個巨任務執行
5. 回到步驟 2

瀏覽器 vs Node.js 事件迴圈

面向瀏覽器Node.js
實作各瀏覽器引擎libuv
渲染步驟有（與動畫協調）無
階段數較簡化6 個明確階段
I/O 處理有限的 Web APIs完整的系統 I/O
計時器精度受限（最小 4ms）較高精度

架構總覽

瀏覽器事件迴圈模型

┌─────────────────────────────────────────┐
│           呼叫堆疊（Call Stack）           │
└──────────────────┬──────────────────────┘
                   │ 堆疊清空
                   ▼
┌─────────────────────────────────────────┐
│         微任務佇列（全部清空）             │
│  Promise.then, queueMicrotask,          │
│  MutationObserver                        │
└──────────────────┬──────────────────────┘
                   │ 微任務清空
                   ▼
┌─────────────────────────────────────────┐
│         渲染更新（如需要）                 │
│  rAF → Style → Layout → Paint           │
└──────────────────┬──────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│      任務佇列（取一個巨任務）             │
│  setTimeout, setInterval, I/O,          │
│  UI events, MessageChannel              │
└─────────────────────────────────────────┘

Node.js 事件迴圈階段

階段處理內容對應 API
Timers到期的計時器回呼setTimeout, setInterval
Pending callbacks系統級回呼TCP 錯誤等
Idle/Prepare內部使用—
PollI/O 事件fs, net 等
Check立即回呼setImmediate
Close關閉事件socket.on(‘close’)

每個階段之間都會清空微任務佇列與 process.nextTick 佇列。

關鍵比較

各種排程 API 的執行時機

API類型執行時機精度
queueMicrotask()微任務當前任務後立即最高
Promise.then()微任務當前任務後立即最高
requestAnimationFrame渲染回呼下次渲染前~16.67ms
setTimeout(fn, 0)巨任務下一輪事件迴圈≥4ms
setImmediateCheck 階段當前 Poll 後Node.js 限定
process.nextTick微任務級當前操作後立即Node.js 限定
requestIdleCallback閒置回呼瀏覽器閒置時不保證

來源觸發方式典型用途

網路XHR, fetch 完成資料載入

應用場景

避免長時間阻塞主執行緒

長時間同步運算會阻塞事件迴圈，導致：

  • UI 無法回應使用者操作
  • 動畫卡頓
  • 計時器不精確

解決策略：

策略方式適用場景
任務分割用 setTimeout 拆分大量 DOM 操作
Web Worker獨立執行緒CPU 密集運算
requestIdleCallback閒置時執行非關鍵計算
Scheduler API優先級排程React 等框架

精確的排程控制

理解事件迴圈有助於控制程式碼執行順序：

  • 使用微任務保證在渲染前執行
  • 使用 rAF 確保動畫流暢
  • 使用 setTimeout 讓出主執行緒

效能監控

工具用途指標
Performance API測量任務耗時Long Tasks > 50ms
PerformanceObserver監聽效能事件FID, INP
Chrome DevTools視覺化時間軸Main Thread 活動

常見誤解

誤解事實
setTimeout(fn, 0) 立即執行最少延遲 4ms，且需等待當前巨任務與微任務
Promise 是非同步的Promise 建構函式是同步的，只有 .then 是非同步
事件迴圈是 JavaScript 的一部分是執行環境（瀏覽器/Node.js）提供的
