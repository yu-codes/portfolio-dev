TypeScript 型別系統

全面解析 TypeScript 結構化型別系統的核心原理、型別推斷與進階型別操作

TypeScript 的型別系統是一套結構化（Structural）型別系統，它在 JavaScript 的動態特性上增加了靜態型別檢查層。這套型別系統的設計目標是在不犧牲 JavaScript 彈性的前提下，提供編譯期的型別安全保障。

核心概念

型別系統分類

分類維度TypeScriptJavaPython (mypy)
結構化 vs 名義化結構化名義化結構化（Protocol）
靜態 vs 動態靜態（漸進式）靜態動態 + 靜態標註
強型別 vs 弱型別強（編譯期）強強（執行期）
型別推斷完善有限有限
型別擦除是（編譯後無型別）部分保留標註不影響執行

基本型別分類

類別型別說明
原始型別string, number, boolean, bigint, symbol不可變值
特殊型別null, undefined, void, never特殊意義
物件型別object, {}, 介面, 類別結構化定義
頂端型別unknown, any所有型別的父型別
底端型別never所有型別的子型別
字面量型別"hello", 42, true精確的值

結構化型別（Duck Typing）

TypeScript 基於結構相容性判斷型別：

規則說明範例
屬性相容來源必須包含目標的所有屬性{a, b, c} 可賦值給 {a, b}
函式相容參數逆變、回傳值協變少參數的函式可賦值給多參數型別
多餘屬性檢查字面量物件不允許多餘屬性直接賦值時會報錯

窄化方式適用型別範例

自訂型別守衛任意邏輯function isString(x): x is string
控制流分析條件/迴圈/例外自動推斷

型別推斷策略

情境推斷規則範例
變數初始化從右側值推斷let x = 5 → number
函式回傳從所有 return 推斷自動聯合回傳型別
最佳共同型別陣列元素的聯合[1, "a"] → (number | string)[]
上下文型別從使用位置推斷回呼參數型別
泛型推斷從實際參數推斷型別參數identity(42) → T = number

關鍵比較

any vs unknown vs never

特性anyunknownnever
意義放棄型別檢查未知型別（安全）不可能發生的值
可賦值給任何型別只有 unknown/any任何型別
可接受任何值任何值無值可賦值
操作前需要無需檢查必須窄化不需要（到不了）
使用時機遷移、第三方安全的動態值永不回傳的函式

interface vs type

面向interfacetype
宣告合併支援（Declaration Merging）不支援
extends使用 extends使用 &（交集）
運算型別不支援支援（聯合、映射、條件）
效能較好（快取）每次重新計算
遞歸自然支援需要間接引用
適用場景物件形狀、API 契約聯合型別、複雜型別運算

協變 vs 逆變

位置變異性說明範例
函式回傳值協變（Covariant）子型別可替代父型別() => Dog 可賦值給 () => Animal
函式參數逆變（Contravariant）父型別可替代子型別(a: Animal) => void 可賦值給 (d: Dog) => void
陣列協變（不安全）TypeScript 的妥協Dog[] 可賦值給 Animal[]
readonly 屬性協變安全的{readonly x: Dog} → {readonly x: Animal}

應用場景

判別式聯合（Discriminated Union）

適合建模狀態機或 API 回應：

type Result<T> =
  | { status: 'success'; data: T }
  | { status: 'error'; error: Error }
  | { status: 'loading' };

品牌型別（Branded Types）

在結構化型別系統中模擬名義型別：

type USD = number & { __brand: 'USD' };
type EUR = number & { __brand: 'EUR' };
// USD 和 EUR 不可互相賦值

設計原則

原則建議
儘量窄使用字面量型別而非寬泛型別
善用推斷不需要為每個變數標註型別
優先 unknown避免使用 any
組合優於繼承使用交集型別組合功能
