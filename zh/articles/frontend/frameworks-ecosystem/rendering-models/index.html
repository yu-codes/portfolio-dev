渲染模型

前端渲染模型比較

前端渲染模型定義了框架如何將資料轉化為使用者介面。從響應式系統的設計哲學、模板語法的選擇，到元件生命週期的管理，這些底層機制決定了框架的效能特性、開發體驗與適用場景。

核心概念

響應式模型（Reactivity Model） 是現代前端框架的基石。當應用狀態變更時，框架需要高效地偵測變化並更新對應的 DOM 節點。不同框架對此問題的解法截然不同：

  • Virtual DOM Diffing：React 採用的方式。每次狀態變更觸發元件重新渲染，產生新的虛擬 DOM 樹，再透過 diff 演算法找出最小變更集合。
  • Fine-grained Reactivity：Vue 3 與 Solid.js 採用的方式。透過 Proxy 或 Signal 精確追蹤每個響應式值的依賴關係，只更新直接受影響的 DOM 節點。
  • Compile-time Reactivity：Svelte 的方式。在編譯階段分析狀態依賴，產生精確的命令式更新程式碼，完全避免執行時的 diff 開銷。

模板引擎（Template Engine） 決定了開發者描述 UI 結構的語法。JSX 將 HTML 嵌入 JavaScript，提供完整的程式語言表達力；而模板語法（如 Vue 的 SFC、Angular 的 Template）則將結構與邏輯分離，更易於靜態分析與編譯優化。

元件生命週期（Component Lifecycle） 定義了元件從建立到銷毀的完整過程。理解生命週期鉤子的觸發時機，對正確處理副作用、資源管理、效能優化至關重要。

架構總覽

渲染模型的架構可從三個維度來理解：

更新粒度維度：

  • 元件級更新（React）：以元件為最小更新單位，透過 memo、useMemo 等 API 避免不必要的重渲染
  • 節點級更新（Vue、Solid）：精確追蹤到每個 DOM 節點，無需開發者手動優化
  • 無執行時更新（Svelte）：編譯器在建構時決定最佳更新策略

排程維度：

  • 同步更新：變更立即反映到 DOM，簡單但可能造成阻塞
  • 批次更新：將多次狀態變更合併為一次 DOM 操作
  • 並發更新（React Concurrent）：可中斷的渲染，優先處理高優先級互動

執行環境維度：

  • 客戶端渲染：完全在瀏覽器端執行
  • 伺服器端渲染：在伺服器產生 HTML，客戶端接管（Hydration）
  • 串流渲染：伺服器分塊傳送 HTML，漸進式顯示內容

Virtual DOM 的取捨：React 的 VDOM 方案在大多數場景下表現良好，且提供了強大的抽象能力（如 React Native、React Three Fiber）。但在高頻更新的場景（如動畫、大型列表），diff 開銷可能成為瓶頸。

Fine-grained Reactivity 的優勢：Vue 3 與 Solid.js 的精確追蹤方案在更新效能上有天然優勢，尤其在大型應用中更為明顯。但這要求開發者理解「響應式值」的概念與解構限制。

編譯時方案的權衡：Svelte 將大量工作移至編譯階段，產出的程式碼極為精簡。然而，這也意味著某些動態模式較難表達，且偵錯時需要理解編譯產物。

應用場景

高互動 SPA：React 或 Vue 3 是首選。豐富的生態系、成熟的狀態管理方案，以及社群支援讓複雜互動邏輯的實現更為順暢。

效能敏感應用：當每一毫秒的渲染延遲都至關重要時（如即時協作編輯器、大型資料表格），Solid.js 或 Svelte 的精確更新機制能帶來顯著優勢。

嵌入式 Widget：需要嵌入第三方頁面的輕量元件，Svelte 的超小執行時使其成為理想選擇，不會對宿主頁面造成明顯的效能影響。

漸進式遷移：Vue 3 的漸進式設計哲學允許團隊逐步從 Options API 遷移至 Composition API，降低大型專案重構的風險。
