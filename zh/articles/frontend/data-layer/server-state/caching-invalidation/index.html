快取（Caching）是前端效能最佳化的核心策略之一，但正確的快取失效（Cache Invalidation）卻是電腦科學中公認最困難的問題之一。在前端資料層中，如何在效能與資料新鮮度之間取得平衡，是架構設計的關鍵決策。

核心概念

快取的基本原理

概念說明
快取命中（Hit）請求的資料存在於快取中
快取未命中（Miss）快取中沒有請求的資料
快取過期（Stale）資料存在但可能已不是最新
快取失效（Invalidation）主動標記快取為過期
快取淘汰（Eviction）移除快取中的資料
快取穿透（Pass-through）繞過快取直接請求

層級位置控制方生命週期

快取一致性模型

模型一致性保證效能適用場景
強一致性每次讀取都是最新低（無快取）金融交易
最終一致性延遲後達到一致高社交媒體內容
Stale-While-Revalidate先顯示舊值，背景更新高（體驗好）大多數 Web 應用
樂觀一致性假設操作會成功最高（即時回饋）使用者互動操作

快取失效策略

策略機制優點缺點
TTL（Time To Live）設定過期時間簡單可靠時間內可能過期
事件驅動失效Mutation 後主動失效精確需要知道影響範圍
輪詢（Polling）定時重新擷取簡單浪費資源
推播（Push）WebSocket/SSE 通知即時基礎設施複雜
版本比較ETag/版本號精確需要伺服器支援
條件請求If-None-Match節省頻寬仍需網路往返

React Query 快取生命週期

查詢首次使用
    ↓ fetch
資料存入快取（fresh）
    ↓ staleTime 過期
標記為 stale
    ↓ 觸發條件（focus/mount/interval）
背景重新擷取
    ↓ 成功
更新快取資料（fresh）
    ...
所有觀察者卸載
    ↓ gcTime 過期
從快取中移除
參數意義預設值
staleTime資料保持新鮮的時間0（立即過期）
gcTime非活躍查詢的快取保留時間5 分鐘
refetchInterval自動輪詢間隔false（關閉）
refetchOnWindowFocus視窗聚焦時重新擷取true

關鍵比較

TTL vs 手動失效 vs 推播

面向TTL手動失效推播
實現複雜度低中高
即時性低（等待過期）中（操作觸發）高（即時）
資源消耗低低中（維持連線）
適用變化不頻繁的資料CRUD 操作即時協作
精確度低高最高
基礎設施需求無無WebSocket/SSE 伺服器

Normalize Cache vs Document Cache

面向正規化快取（Apollo）文件快取（React Query）
儲存方式以 ID 扁平化儲存以查詢結果整體儲存
自動更新同 ID 的資料自動同步需要手動失效
複雜度高低
記憶體較低（去重）較高（可能重複）
適用GraphQLREST API
部分更新自動需要 setQueryData

樂觀更新流程

步驟動作失敗回滾
1取消進行中的重新擷取—
2儲存目前快取快照回滾的依據
3樂觀地更新快取—
4送出 Mutation—
5a成功：確認/失效相關查詢—
5b失敗：用快照回滾context.previousData
