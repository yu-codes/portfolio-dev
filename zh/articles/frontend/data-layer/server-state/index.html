伺服器狀態（Server State）是指源自遠端伺服器、被前端應用暫時借用的資料。與客戶端狀態不同，伺服器狀態具有「非同步」、「共享」、「可能過期」三大特性。傳統上，開發者常將伺服器狀態塞進全域狀態管理工具（如 Redux），但這導致了大量樣板程式碼與不必要的複雜度。現代的伺服器狀態管理函式庫徹底改變了這個局面。

核心概念

Stale-While-Revalidate（SWR）模式：這是伺服器狀態管理的核心策略。當使用者請求資料時，先回傳快取中的舊資料（stale），同時在背景重新取得最新資料（revalidate）。一旦新資料到達，自動更新 UI。這個模式源自 HTTP 的 Cache-Control: stale-while-revalidate 標頭。

查詢鍵（Query Key）：每一筆伺服器狀態都透過唯一的查詢鍵來識別。查詢鍵通常是一個陣列，包含端點路徑與參數，例如 ['users', { page: 1 }]。相同查詢鍵的請求會自動去重（deduplication），避免重複發送。

快取生命週期：伺服器狀態在快取中經歷以下階段：fresh（新鮮）→ stale（過期）→ inactive（未被任何元件訂閱）→ garbage collected（被回收）。開發者可設定 staleTime（資料多久後視為過期）與 gcTime（未使用的快取保留多久）。

背景重新取得（Background Refetch）：現代函式庫會在特定時機自動重新取得資料：視窗重新獲得焦點時、網路重新連線時、指定間隔輪詢時，或手動觸發時。這確保使用者總是看到相對最新的資料。

樂觀更新（Optimistic Updates）：在送出變更請求前先更新 UI，若請求失敗則回滾至先前狀態。這需要搭配 onMutate（樂觀更新快取）、onError（回滾）、onSettled（無論成敗都重新取得）等回呼函式。

架構總覽

伺服器狀態管理架構
├── 查詢層 (Query Layer)
│   ├── useQuery / useSWR（宣告式資料取得）
│   ├── Query Cache（記憶體快取池）
│   ├── Query Observer（訂閱與通知）
│   └── Query Deduplication（請求去重）
├── 變更層 (Mutation Layer)
│   ├── useMutation（命令式資料修改）
│   ├── Optimistic Updates（樂觀更新）
│   ├── Retry Logic（失敗重試）
│   └── Cache Invalidation（快取失效）
├── 同步策略
│   ├── Window Focus Refetch
│   ├── Network Reconnect Refetch
│   ├── Polling（定時輪詢）
│   └── WebSocket Integration
└── 進階功能
    ├── Infinite Queries（無限滾動）
    ├── Paginated Queries（分頁）
    ├── Prefetching（預取）
    └── Suspense Integration

這些函式庫的核心理念是將「資料取得」視為一個獨立的關注點，從 UI 元件中抽離。元件只需宣告「我需要什麼資料」，而非命令式地處理 loading/error/success 狀態。

何時選擇 React Query：需要精細的快取控制、複雜的依賴查詢、或跨框架使用。

何時選擇 SWR：專案較小、偏好極簡 API、或已使用 Next.js。

何時選擇 RTK Query：已大量使用 Redux、或需要與 Redux 狀態整合。

應用場景

社群平台的動態牆：使用無限查詢（Infinite Query）實現無限滾動，搭配 staleTime: 60000 讓資料在一分鐘內視為新鮮。當使用者發布新貼文時，透過 queryClient.invalidateQueries(['posts']) 觸發重新取得。

即時庫存管理系統：商品庫存需要高度即時性，使用短間隔輪詢（如每 10 秒）或 WebSocket 推送。搭配樂觀更新讓扣庫存操作即時反映在 UI，同時設定 retry: 3 確保網路不穩時能自動重試。

多步驟表單精靈：表單的選項列表（如國家、城市）來自伺服器，可設定 staleTime: Infinity 長時間快取。依賴查詢（Dependent Queries）確保選擇國家後才載入對應的城市列表。

儀表板資料聚合：同時取得多個 API 的資料，利用 useQueries 平行發送請求。設定 placeholderData 在載入時顯示骨架畫面，搭配 select 在快取層進行資料轉換，避免不必要的重新渲染。
