資料變更模式

資料變更（Mutation）是前端應用中除了讀取之外的另一半——建立、更新、刪除資料的操作。與查詢不同，Mutation 會改變伺服器狀態，因此需要考慮樂觀更新、錯誤回滾、快取同步等複雜場景。

核心概念

Mutation 的生命週期

階段說明UI 狀態
閒置（Idle）未執行任何操作正常
執行中（Pending）請求已發送，等待回應載入指示
成功（Success）伺服器確認操作完成成功回饋
失敗（Error）操作失敗錯誤訊息

Mutation 與 Query 的區別

面向Query（查詢）Mutation（變更）
HTTP 方法GETPOST, PUT, PATCH, DELETE
冪等性是（重複執行結果相同）不一定
快取可快取不可快取（但影響快取）
副作用無改變伺服器狀態
自動重試安全需要謹慎（非冪等）
觸發自動（組件掛載）手動（使用者操作）

變更策略分類

策略說明使用者體驗複雜度
等待確認（Pessimistic）等伺服器回應才更新 UI延遲感低
樂觀更新（Optimistic）立即更新 UI，失敗再回滾即時感高
背景變更使用者無感知的同步無感中
排隊變更離線時排入佇列離線友好高

架構總覽

樂觀更新流程

使用者操作
    │
    ├── 1. 取消進行中的相關查詢（避免覆蓋）
    │
    ├── 2. 儲存目前快取快照（回滾用）
    │
    ├── 3. 樂觀更新快取（UI 立即反映）
    │
    ├── 4. 發送 Mutation 到伺服器
    │
    ├── 5a. 成功
    │   └── 失效相關查詢 / 用回應更新快取
    │
    └── 5b. 失敗
        ├── 用快照回滾快取
        └── 顯示錯誤訊息

React Query useMutation 架構

選項觸發時機用途
mutationFn呼叫 mutate 時執行實際的 API 呼叫
onMutate發送前樂觀更新、儲存快照
onSuccess成功後失效快取、通知使用者
onError失敗後回滾、錯誤處理
onSettled無論成敗最終清理、重新擷取

CRUD 對應的 Mutation 模式

操作HTTP 方法快取影響UI 回饋
CreatePOST新增項目到列表成功提示 + 導覽
UpdatePUT/PATCH更新特定項目即時 UI 更新
DeleteDELETE移除項目即時移除 + 確認
BatchPOST (batch)多項目更新進度指示
UploadPOST (multipart)新增資源上傳進度

關鍵比較

樂觀更新 vs 悲觀更新

面向樂觀更新悲觀更新
使用者體驗即時回饋有延遲感
實現複雜度高（需要回滾邏輯）低
資料一致性可能暫時不一致始終一致
錯誤處理需要回滾 UI簡單顯示錯誤
適用場景高頻操作、低失敗率關鍵操作、高失敗率
複雜操作較難（多步驟回滾）簡單
衝突處理需要考慮伺服器處理

各工具的 Mutation 支援

工具API樂觀更新重試離線支援
React QueryuseMutation完整支援可設定有
SWRuseSWRMutation手動實作——
RTK Querymutation endpoint手動 / 自動—有限
Apollo ClientuseMutation完整支援—有
tRPCmutation procedure完整支援可設定—

快取更新策略比較

策略方法優點缺點
失效重擷取invalidateQueries保證最新額外網路請求
手動更新setQueryData無需額外請求需要知道快取結構
回應更新用 mutation 回應更新伺服器確認的資料只更新一個查詢
樂觀 + 失效onMutate + onSettled最佳體驗最複雜

應用場景

表單提交流程

步驟動作處理
1使用者填寫表單客戶端驗證
2點擊提交禁用按鈕、顯示載入
3發送 MutationAPI 呼叫
4a成功顯示成功、導覽/關閉
4b驗證錯誤 (422)標記欄位錯誤
4c伺服器錯誤 (5xx)顯示通用錯誤、允許重試

刪除操作的 UX 模式

模式說明適用
確認對話框刪除前要求確認重要資源
樂觀刪除 + Undo立即移除，提供撤銷Gmail 風格
軟刪除標記為已刪除，可恢復資料庫策略
批次刪除選取多項一次刪除列表管理

衝突處理

策略說明適用
Last Write Wins最後寫入覆蓋簡單場景
Optimistic Locking版本號/ETag 比對編輯表單
Merge合併變更協作編輯
Conflict Resolution UI讓使用者決定重要衝突

批次操作

情境策略注意事項
全選刪除單一批次 API一個請求處理所有
逐一操作並行多個 Mutation需要進度追蹤
有序操作串行 Mutation前一個成功才執行下一個
部分成功記錄成功/失敗項需要報告與重試

離線 Mutation 排隊

步驟線上行為離線行為
觸發 Mutation直接發送存入佇列
回應處理正常處理暫存結果
網路恢復—依序發送佇列中的 Mutation
衝突處理—版本比對 / 使用者確認

最佳實踐

實踐說明
冪等設計使用 idempotency key 防止重複提交
載入狀態禁用按鈕 + 載入指示防止雙重提交
錯誤粒度區分欄位錯誤 vs 通用錯誤
成功回饋Toast/導覽/內聯訊息
型別安全定義 Request/Response 型別
