單向資料流（Unidirectional Data Flow）是現代前端應用的核心架構原則。它規定資料在應用中只沿一個方向流動，使狀態變化可預測、可追蹤，大幅降低了複雜應用的認知負擔。

核心概念

單向資料流的定義

State → View → Action → State → View → ...
環節角色規則
State應用的唯一真相來源不可直接修改
View根據 State 渲染 UI純粹的狀態映射
Action描述使用者意圖唯一改變 State 的途徑

單向 vs 雙向資料流

面向單向資料流雙向資料流
資料流向State → View（單方向）State ↔ View（雙向同步）
可預測性高低（多來源修改）
除錯容易追蹤變更來源難以判斷誰觸發了變更
程式碼量較多（需明確的 Action）較少（自動同步）
適用規模中大型應用小型表單/原型
代表React/Redux, FluxAngular (ngModel), Vue v-model

Flux 架構的四個角色

角色職責單向流動
View顯示資料、發送 ActionView → Action
Action描述事件Action → Dispatcher
Dispatcher分發到所有 StoreDispatcher → Store
Store持有狀態、處理邏輯Store → View

各架構的單向資料流實現

架構流程特色
FluxView → Action → Dispatcher → Store → View多 Store
ReduxView → Action → Reducer → Store → View單一 Store
ElmView → Msg → Update → Model → View純函數式
MVI (Model-View-Intent)Intent → Model → View → Intent響應式流
MVVM + 單向綁定ViewModel → View, View → Command → ViewModel命令模式

單向資料流的規則

規則說明違反時的問題
狀態不可變不直接修改 state變更追蹤失效
透過 Action 變更所有變更都透過 dispatch繞過時無法追蹤
Reducer 是純函式相同輸入永遠相同輸出不可預測的行為
單一 Store（Redux）所有狀態集中管理分散時難以維護
View 是狀態的函式UI = f(state)隱式狀態導致不一致

Flux vs Redux vs MobX 的資料流

面向FluxReduxMobX
Store 數量多個單一多個
更新機制Store 自行處理外部 Reducer內部可變
是否嚴格單向是是較寬鬆
副作用處理Store 中中介軟體action/reaction
可追蹤性中高（時間旅行）中

單向資料流的優缺點

優點缺點
可預測的狀態變化程式碼量增加（模板程式碼）
容易除錯（時間旅行）簡單操作也需要完整流程
資料流向清晰初學者學習曲線
容易測試表單等場景較繁瑣
利於團隊協作過度設計的風險
易於實現 undo/redo—

設計建議

建議說明
Action 應描述「什麼發生了」而非「要做什麼」
一個事件可觸發多個 state 變更不要限制 1:1
Selector 做為讀取界面解耦 View 與 Store 結構
避免在 View 中計算衍生值放在 Selector/Computed
批次更新避免一個操作多次觸發渲染
