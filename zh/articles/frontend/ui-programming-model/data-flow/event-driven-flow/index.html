事件驅動流

事件驅動流（Event-Driven Flow）是一種以事件為核心的程式設計範式。在前端中，從 DOM 事件到組件通訊、從狀態管理到資料串流，事件驅動的思維無處不在。理解這種資料流模式有助於設計鬆耦合、高擴展性的應用架構。

核心概念

事件驅動的基本元素

元素說明範例
事件（Event）系統中發生的事情click, dataLoaded, userLoggedIn
發布者（Publisher）產生事件的來源Button, API 層, WebSocket
訂閱者（Subscriber）回應事件的處理器事件處理函式, Observer
事件通道（Channel）傳遞事件的媒介EventTarget, Subject, EventBus

事件驅動模式分類

模式耦合度通訊方式適用場景
直接回呼高1:1簡單組件間通訊
觀察者模式中1:N狀態變更通知
事件匯流排低N:N跨模組通訊
響應式串流低N:N（可組合）複雜非同步流程
CQRS/Event Sourcing極低解耦讀寫複雜領域邏輯

推（Push） vs 拉（Pull）模型

面向Push 模型Pull 模型
資料流向生產者主動推送消費者主動請求
觸發者資料來源資料消費者
即時性即時通知需要輪詢
資源使用有事件才處理持續檢查
前端範例Event Listener, Observablefetch, useQuery refetch
背壓處理需要（消費者可能來不及處理）不需要（消費者控制節奏）

架構總覽

瀏覽器事件模型

事件流三階段：
捕獲 (Capture) → 目標 (Target) → 冒泡 (Bubble)

document
├── body (捕獲: ▼)
│   ├── div (捕獲: ▼)
│   │   └── button [目標: ●]
│   │       (冒泡: ▲)
│   └── (冒泡: ▲)
└── (冒泡: ▲)
事件傳播控制API效果
停止冒泡stopPropagation()不再向上傳播
停止立即傳播stopImmediatePropagation()同一元素的其他監聽也不執行
阻止預設preventDefault()取消瀏覽器預設行為
事件委派在父元素上監聽利用冒泡減少監聽器數量

響應式串流模型（RxJS）

Observable（可觀察序列）
    │
    ▼ pipe（操作符管道）
    │
    ├── map()      ← 轉換
    ├── filter()   ← 過濾
    ├── debounce() ← 節流
    ├── merge()    ← 合併
    └── ...
    │
    ▼
Observer（觀察者）
    ├── next(value)   ← 接收值
    ├── error(err)    ← 錯誤處理
    └── complete()    ← 完成通知

關鍵比較

事件驅動 vs 命令式呼叫

面向事件驅動命令式呼叫
耦合鬆耦合（不知道誰在聽）緊耦合（明確呼叫對象）
擴展性高（新增訂閱者不修改發布者）低（需修改呼叫方）
可追蹤性較難（間接呼叫）容易（直接呼叫鏈）
除錯需要事件追蹤工具IDE 跳轉即可
測試需要模擬事件直接呼叫斷言
適用跨模組、插件系統模組內、明確依賴

場景不使用委派使用委派

節流與防抖

技術行為適用
Debounce等待停止觸發後執行搜尋輸入、視窗 resize
Throttle固定間隔最多執行一次捲動事件、滑鼠移動
requestAnimationFrame每幀最多一次動畫相關計算

常見陷阱

陷阱問題解決方案
記憶體洩漏未取消訂閱組件卸載時 unsubscribe
事件風暴過多事件觸發debounce/throttle
隱式依賴事件讓依賴關係不透明文件化事件契約
執行順序多個訂閱者順序不確定避免依賴順序
錯誤處理訂閱者拋出錯誤影響其他獨立的 try/catch
