可複用 UI 模式

系統化歸納前端可複用 UI 設計模式，從原子組件到複合佈局的實踐指南

可複用 UI 模式是前端工程中提升開發效率與一致性的關鍵。透過系統化的組件抽象與設計模式，團隊能建構出可維護、可擴展的使用者介面，同時確保產品的視覺與互動一致性。

核心概念

組件抽象層級

層級名稱特性範例
L1原子（Atoms）最小單位，無依賴Button, Icon, Badge
L2分子（Molecules）原子的組合SearchBar, FormField
L3有機體（Organisms）分子的組合，有商業邏輯Header, ProductCard
L4模板（Templates）頁面結構，無資料DashboardLayout
L5頁面（Pages）模板 + 真實資料HomePage

可複用性維度

維度說明設計策略
視覺重用相同的視覺呈現Design Token + 主題
行為重用相同的互動邏輯Headless 組件/Hook
結構重用相同的 DOM 結構佈局組件/Slot
邏輯重用相同的商業邏輯Custom Hook/Composable
狀態重用相同的狀態管理Context/Store

設計系統核心概念

概念說明實現
Design Token最小的設計決策CSS 變數、主題物件
變體（Variant）組件的不同外觀Props 控制（primary/secondary）
尺寸（Size）組件的大小規格sm/md/lg/xl
組合（Composition）組件的可組合性Slot/Children
可及性（Accessibility）無障礙支援ARIA 屬性、鍵盤導覽

架構總覽

常見 UI 模式分類

類別模式說明
資料展示List, Table, Card, Tree呈現資料集合
資料輸入Form, Input, Select, DatePicker收集使用者輸入
回饋Toast, Alert, Modal, Skeleton系統回饋
導覽Tabs, Breadcrumb, Pagination頁面導覽
佈局Grid, Stack, Flex, Container空間排列
揭露Accordion, Tooltip, Popover漸進揭露資訊

組件 API 設計模式

模式說明適用場景
配置式（Config）單一 prop 物件控制所有行為表格 columns 設定
宣告式（Declarative）用 JSX/模板組合子組件Compound Components
命令式（Imperative）透過 ref 呼叫方法Dialog.open(), Form.reset()
受控/非受控外部/內部管理狀態表單元素
多態（Polymorphic）as prop 改變底層元素<Button as="a">

Polymorphic Component 模式

讓組件可以渲染為不同的 HTML 元素或其他組件：

優點缺點
語意正確（a, button, div）型別推斷複雜
減少組件數量過度使用會模糊職責
繼承底層元素的 props需要良好的泛型設計

關鍵比較

組件庫架構策略

策略代表優點缺點
全功能組件庫Ant Design, MUI開箱即用體積大、客製化受限
Headless UIRadix, Headless UI完全控制樣式需要自行設計
CSS 框架 + 組件Tailwind + shadcn/ui輕量、可複製維護責任在專案
原子 CSS + 組合Tailwind + 自訂最大彈性開發初期較慢

應用場景

建構佈局系統

佈局組件是最基礎的可複用模式：

組件功能CSS 基礎
<Stack>垂直/水平堆疊Flexbox column/row
<Grid>網格佈局CSS Grid
<Container>最大寬度容器max-width + margin auto
<Spacer>彈性間距flex-grow
<Divider>分隔線border/hr
<AspectRatio>固定比例容器padding-top hack / aspect-ratio

設計決策指南

問題建議
何時提取組件？出現第三次重複時
組件介面多寬？先窄後寬，需要時再增加
放在哪一層？依據使用範圍決定（全域/功能/頁面）
多少 props 太多？超過 7 個考慮拆分或使用 Compound
配置 vs 組合？變化少用配置，變化大用組合
