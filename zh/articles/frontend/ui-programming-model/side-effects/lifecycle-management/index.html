生命週期管理

組件的生命週期是指組件從建立到銷毀的整個過程。理解生命週期的各個階段，是正確管理副作用、訂閱、資源申請與釋放的基礎。不同的前端框架有不同的生命週期模型，但核心概念是相通的。

核心概念

生命週期三大階段

階段說明典型操作
掛載（Mount）組件首次渲染到 DOM初始化、訂閱、API 呼叫
更新（Update）Props/State 變更觸發重渲染回應資料變化
卸載（Unmount）組件從 DOM 移除清理、取消訂閱、釋放資源

各框架生命週期對照

階段React (Hooks)Vue 3 (Composition)AngularSvelte
初始化函式體本身setup()constructor模組頂層
掛載前—onBeforeMountngOnInit—
掛載後useEffect(() => {}, [])onMountedngAfterViewInitonMount
更新前—onBeforeUpdate—beforeUpdate
更新後useEffect (依賴變化)onUpdatedngAfterViewCheckedafterUpdate
卸載useEffect 清理函式onUnmountedngOnDestroyonDestroy

React useEffect 的心智模型

依賴陣列執行時機等價傳統生命週期
[]掛載時執行一次componentDidMount
[a, b]a 或 b 變化時componentDidUpdate（有條件）
無（省略）每次渲染後componentDidMount + componentDidUpdate
清理函式下次 effect 前/卸載時componentWillUnmount

架構總覽

React 生命週期流程

渲染階段（純粹，可中斷）
├── 函式元件執行
├── 計算 JSX
└── 對比 Virtual DOM

提交階段（有副作用，同步）
├── 更新 DOM
├── 執行 useLayoutEffect（同步）
└── 瀏覽器繪製
    └── 執行 useEffect（非同步）

Angular 生命週期鉤子順序

順序鉤子觸發條件
1ngOnChangesInput 屬性變更
2ngOnInit初始化（僅一次）
3ngDoCheck每次變更偵測
4ngAfterContentInit投射內容初始化
5ngAfterContentChecked投射內容檢查後
6ngAfterViewInit視圖初始化
7ngAfterViewChecked視圖檢查後
8ngOnDestroy銷毀前

關鍵比較

Class Component vs Hooks 生命週期

面向Class ComponentHooks
思考方式按階段思考按副作用思考
組織同一階段的不同邏輯集中相關邏輯集中
程式碼重用HOC/Render Props自訂 Hook
清理時機componentWillUnmounteffect 清理函式
避免重複shouldComponentUpdateuseMemo/useCallback
學習曲線直觀的生命週期圖需要理解閉包和依賴

useEffect vs useLayoutEffect

面向useEffectuseLayoutEffect
執行時機瀏覽器繪製後（非同步）DOM 更新後、繪製前（同步）
阻塞渲染否是
用途資料擷取、訂閱、日誌DOM 測量、防止閃爍
SSR安全伺服器端會警告
效能較好（不阻塞）可能造成延遲

生命週期 vs 響應式 Effect

面向傳統生命週期響應式 Effect（Solid/Vue）
觸發組件階段（mount/update）依賴變更
粒度組件級表達式級
自動追蹤否（手動指定依賴）是（自動收集）
清理明確的 unmount 階段自動（依賴失效時）
心智模型時間線響應關係圖

應用場景

常見的生命週期操作

操作掛載時更新時卸載時
事件監聽addEventListener—removeEventListener
定時器setInterval—clearInterval
WebSocketconnect—disconnect

避免常見錯誤

錯誤問題正確做法
useEffect 無依賴陣列每次渲染都執行加上 [] 或正確依賴
忘記清理記憶體洩漏回傳清理函式
在清理中使用過期狀態讀到舊值使用 ref 保存最新值
useEffect 中的 async清理函式被覆蓋內部定義 async 函式
依賴遺漏過期的閉包值遵循 exhaustive-deps

效能模式

模式說明實現
延遲初始化首次需要時才初始化useMemo(() => init, [])
條件性副作用只在特定條件下執行if 判斷在 effect 內
去抖動效果頻繁變化時延遲執行debounce + cleanup
並行效果多個獨立副作用多個 useEffect
依賴穩定化避免不必要的重執行useRef 儲存穩定值

設計建議

建議原因
每個 Effect 只做一件事易於理解和測試
將副作用提取為自訂 Hook重用和隔離
優先使用事件處理器不是所有操作都需要 Effect
減少 Effect 依賴減少不必要的重執行
在 Effect 內宣告函式避免依賴外部不穩定函式
