狀態正規化

狀態正規化（State Normalization）是將巢狀的資料結構轉換為扁平化、以 ID 為索引的格式。這是源自關聯式資料庫的概念，應用於前端狀態管理中能有效避免資料重複、簡化更新邏輯並提升效能。

核心概念

為何需要正規化

問題非正規化時的表現正規化後的解決
資料重複同一實體在多處重複存在唯一儲存，ID 引用
更新一致性需要找到所有副本逐一更新單一位置更新即可
深層巢狀多層展開運算子更新扁平結構直接存取
效能深層比較、大量複製淺比較、局部更新
記憶體大量重複資料最小化儲存

正規化的基本結構

組件說明範例
entities以 ID 為鍵的實體映射{ '1': { id: '1', name: 'Alice' } }
ids有序的 ID 列表['1', '2', '3']
關聯以 ID 引用替代巢狀{ authorId: '1' } 而非 { author: {...} }

正規化轉換範例

非正規化（巢狀）：

posts: [
  { id: 1, title: 'A', author: { id: 10, name: 'Yu' }, comments: [...] },
  { id: 2, title: 'B', author: { id: 10, name: 'Yu' }, comments: [...] }
]

正規化（扁平）：

entities:
  posts: { 1: { id: 1, title: 'A', authorId: 10, commentIds: [100, 101] } }
  users: { 10: { id: 10, name: 'Yu' } }
  comments: { 100: { id: 100, text: '...' } }
ids:
  posts: [1, 2]

架構總覽

EntityAdapter 模式

功能API說明
新增addOne / addMany新增實體
更新updateOne / updateMany部分更新
刪除removeOne / removeMany移除實體
設定setAll / setOne覆蓋設定
取代upsertOne / upsertMany新增或更新

關聯類型處理

關聯類型儲存方式查詢方式
一對一外鍵欄位entities[foreignId]
一對多ID 陣列 或 反向查找ids.filter() 或 索引
多對多中間表/連接實體交叉查詢
自引用parentId 欄位遞歸查找

Selector 層級設計

State
├── selectAllPosts        ← 基本 selector
├── selectPostById(id)    ← 參數化 selector
├── selectPostsByAuthor   ← 過濾 selector
├── selectPostWithAuthor  ← 組合 selector（join）
└── selectPostsSorted     ← 衍生 selector
Selector 類型功能記憶化
基本選取從 store 取出原始片段不需要
過濾選取根據條件篩選需要（避免每次新陣列）
組合選取合併多個實體需要
衍生選取計算/轉換資料需要

關鍵比較

正規化 vs 非正規化

面向正規化非正規化
資料重複無可能大量重複
更新複雜度O(1) 直接定址O(n) 需尋找所有副本
讀取複雜度需要 join/組合直接使用
記憶體使用最小化可能較大
初始設定需要正規化邏輯直接儲存 API 回傳
適用規模中大型應用小型應用

各框架/工具的正規化支援

工具方式特色
Redux Toolkit (createEntityAdapter)EntityAdapter標準化的 CRUD + selector
normalizrSchema 定義 + normalize自動正規化 API 回應
Apollo Client自動正規化（by __typename + id）GraphQL 自動快取
React Query不正規化（各查詢獨立）依靠失效策略
Pinia手動實作彈性但需自行設計

正規化粒度選擇

粒度說明適用場景
完全正規化所有實體獨立、所有關聯用 ID資料高度關聯的 CRUD 應用
部分正規化主要實體正規化，小物件內嵌多數應用的平衡點
不正規化巢狀結構直接儲存資料很少重複、只讀為主

應用場景

CRUD 應用

正規化狀態非常適合典型的 CRUD 操作：

操作正規化的好處
Create只需 entities[newId] = newEntity，推入 ids
ReadO(1) 直接 entities[id]
Update只需更新一處
Delete刪除 entities 條目，從 ids 移除
Listids 陣列保持順序

列表 + 詳情模式

商品列表頁 → 需要 ids + 基本資訊
商品詳情頁 → 需要完整資訊

正規化策略：
entities.products: { [id]: { ...完整資料 } }
ids.productList: [排序後的 id 列表]

樂觀更新

步驟動作正規化的好處
1用戶操作—
2立即更新 entities直接定址更新
3送出 API 請求—
4a成功：確認無需額外操作
4b失敗：回滾只需還原該 entity

常見陷阱

陷阱問題解決方案
過度正規化簡單資料也拆分只對會重複的實體正規化
缺少 selector組件直接存取 store 結構使用 selector 層解耦
ID 不穩定臨時 ID 與伺服器 ID 衝突使用 UUID 或映射表
遺孤資料刪除父實體未清理關聯級聯刪除或定期清理
selector 效能每次回傳新參照使用 createSelector 記憶化
