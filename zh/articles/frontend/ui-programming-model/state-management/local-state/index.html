區域狀態（Local State）是指僅存在於單一組件內部的狀態，不與其他組件共享。它是前端狀態管理的基礎單元，正確使用區域狀態能大幅簡化應用架構，減少不必要的全域狀態汙染。

核心概念

區域狀態的定義

特性說明
範圍僅在宣告組件及其子組件可見
生命週期隨組件掛載而建立，卸載而銷毀
所有權組件自身完全控制
更新觸發只重新渲染擁有狀態的組件樹
獨立性多個實例各自維護獨立狀態

何時使用區域狀態

適合區域狀態不適合區域狀態
UI 互動狀態（展開/收合）需跨組件共享的資料
表單輸入值使用者認證狀態
動畫狀態全域主題/語系
臨時的篩選/排序URL 需要反映的狀態
載入/錯誤狀態快取的伺服器資料
元件的視覺模式（編輯/預覽）多個不相關組件需要的資料

狀態的設計原則

原則說明範例
最小化只儲存必要資料不存衍生值
正規化避免重複不存可計算的值
就近原則狀態放在最近的共同祖先減少 prop drilling
單一真相來源每份資料只有一個權威來源避免同步問題
衍生而非儲存能算出來的就算computed/useMemo

useState vs useReducer 選擇

面向useStateuseReducer
適用簡單值（boolean, string, number）複雜狀態物件、多欄位聯動
更新邏輯分散在各處集中在 reducer
可測試性一般高（純函式）
除錯較難追蹤action 可記錄
學習曲線低中
效能多個 useState 各自觸發一次 dispatch 更新多欄位

狀態機模式

用有限狀態機管理複雜的狀態轉換：

idle → loading → success
                → error → idle（重試）
傳統方式狀態機方式
多個布林值（isLoading, isError）單一狀態值（idle/loading/success/error）
可能出現不合法組合只有合法的狀態轉換
需要大量條件判斷明確的狀態圖
難以維護自我文件化

關鍵比較

區域狀態 vs 提升狀態 vs 全域狀態

面向區域狀態提升狀態全域狀態
範圍單一組件共同祖先子樹整個應用
複雜度低中高
效能影響最小中等（祖先重渲染）需要最佳化
維護成本低中高
重用性高（組件獨立）中低（耦合 store）
適用場景UI 狀態兄弟組件共享全域共享
