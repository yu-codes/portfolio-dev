Hydration 機制

深入解析前端 Hydration 的運作原理、效能瓶頸與現代優化策略，涵蓋 Selective、Progressive、Resumable Hydration

Hydration（水合）是伺服器渲染後在客戶端「復活」靜態 HTML 的過程——將事件處理器綁定至 DOM 節點、恢復元件狀態、啟動響應式系統。傳統全量 Hydration 是現代 SSR 應用的主要效能瓶頸，促使框架社群發展出 Selective、Progressive 與 Resumable 等創新模式。

核心概念

Hydration 的本質

階段動作目的
SSR HTML 到達瀏覽器顯示靜態內容快速 FCP
JS Bundle 載入下載框架 + 應用程式碼準備互動
元件樹重建在客戶端重新執行元件邏輯建立虛擬 DOM
DOM 匹配虛擬 DOM 與實際 DOM 比對驗證一致性
事件綁定附加事件監聽器啟用互動
狀態恢復注入伺服器序列化的狀態資料同步

Hydration 問題

問題描述影響
執行成本需重新執行所有元件延遲 TTI
阻塞性傳統 Hydration 為同步凍結主執行緒
重複工作伺服器做過的事客戶端再做浪費 CPU
Bundle 大小即使不互動也需載入邏輯增加下載量
Mismatch客戶端/伺服器不一致渲染錯誤

Hydration Mismatch 常見原因

原因範例解決方式
時間相關Date.now()固定時間或 useEffect
瀏覽器 APIwindow.innerWidth延遲至客戶端
隨機值Math.random()使用固定 seed
第三方腳本注入 DOM 內容suppressHydrationWarning
HTML 規範<p> 內嵌 <div>修正巢狀結構

Selective Hydration（React 18）

特性說明
Suspense 邊界每個 Suspense 為獨立水合單元
優先權排程使用者互動區域優先水合
串流搭配HTML 串流中漸進水合
事件重放水合前的點擊事件會被記錄並重放
非阻塞水合過程可被中斷

Resumability（Qwik）

面向HydrationResumability
客戶端啟動重新執行全部元件無需重新執行
事件監聽JS 執行後才綁定全域監聽器 + 延遲載入
初始 JS整個 Bundle極小 bootstrap
狀態恢復重建 + 注入HTML 中序列化
延遲載入元件/路由級互動/事件級
TTI依 Bundle 大小近乎即時

Islands Hydration（Astro）

特性傳統 SPAIslands
水合範圍整個頁面僅互動元件
靜態區域也需 JS 處理純 HTML 不載入 JS
獨立性全頁面耦合各島嶼獨立
載入時機一次性client:load/idle/visible
框架混用不可能各島嶼可不同框架

應用場景

大型內容網站優化

新聞網站頁面中，90% 內容為靜態文字與圖片，僅評論區、分享按鈕需要互動。使用 Islands 模式，靜態區域零 JS 開銷，互動區域獨立水合，TTI 可從數秒降至毫秒級。

電商產品頁

產品頁包含靜態描述、圖片輪播（互動）、購買按鈕（互動）、評論區（延遲互動）。透過 Progressive Hydration，優先水合購買按鈕（業務關鍵），圖片輪播可見時水合，評論區捲動至視口時載入。

高互動儀表板

對於完全互動的 SaaS 儀表板，Selective Hydration 是務實選擇。以 Suspense 邊界劃分面板，使用者首先可操作頂部導航，各資料面板隨資料載入漸進水合，減少首次互動等待時間。
