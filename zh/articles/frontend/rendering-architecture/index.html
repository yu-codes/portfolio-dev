渲染架構

CSR、SSR、SSG 與混合渲染

渲染架構（Rendering Architecture）是前端工程中最具策略性的架構決策之一。它決定了 HTML 內容在何處、何時被生成，直接影響首屏效能、SEO 表現、互動性與基礎設施成本。從傳統的伺服器端渲染到現代的混合式策略，前端渲染架構經歷了多次典範轉移，理解每種模式的取捨是做出正確架構決策的基礎。

核心概念

現代前端渲染架構主要包含五種模式：

Client-Side Rendering（CSR）：瀏覽器下載一個空的 HTML 殼與 JavaScript Bundle，由 JavaScript 在客戶端動態生成 DOM。典型代表是 Create React App。優點是互動性強、部署簡單（純靜態檔案）；缺點是首屏白屏時間長、SEO 不友善。

Server-Side Rendering（SSR）：每次請求時伺服器執行 JavaScript 生成完整 HTML，回傳給瀏覽器後再進行 Hydration（注水）使頁面具備互動性。優點是首屏快、SEO 友善；缺點是伺服器負載高、TTFB 可能較慢。

Static Site Generation（SSG）：在建構時期預先生成所有頁面的靜態 HTML 檔案。優點是極快的回應速度（可直接從 CDN 服務）、安全性高；缺點是內容更新需重新建構、不適合高度動態內容。

Incremental Static Regeneration（ISR）：結合 SSG 與 SSR 的優點，在背景按需重新生成過期的靜態頁面。使用者首次請求取得快取頁面，同時觸發背景重建。Next.js 率先實現此模式。

Streaming SSR：伺服器以串流方式逐步傳送 HTML，瀏覽器可在接收過程中就開始渲染。搭配 React 18 的 Suspense 與 renderToPipeableStream，能大幅改善 TTFB 與 FCP。

架構總覽

各渲染模式的執行流程比較：

CSR:    伺服器 → 空 HTML + JS → 瀏覽器執行 JS → 渲染 DOM
SSR:    伺服器執行 JS → 完整 HTML → 瀏覽器渲染 → Hydration
SSG:    建構時生成 HTML → CDN 快取 → 瀏覽器渲染 → Hydration
ISR:    CDN 快取 HTML → 瀏覽器渲染 → 背景重新驗證/重建
Stream: 伺服器串流 HTML → 瀏覽器漸進渲染 → 選擇性 Hydration

現代框架普遍支持 混合渲染——同一應用中不同頁面可採用不同策略。例如 Next.js App Router 允許在同一專案中混合 Server Components（RSC）、SSR、SSG，根據每個頁面的特性選擇最佳策略。

React Server Components（RSC） 是最新的渲染創新。它在伺服器端執行元件邏輯但不傳送 JavaScript 至客戶端，大幅減少 Bundle Size。客戶端元件僅用於需要互動的部分，實現精確的伺服器/客戶端分界。

選擇建議：

  • 部落格/文件站：SSG（Hugo、Astro）
  • 電商/新聞：ISR 或 Streaming SSR
  • 管理後台/Dashboard：CSR
  • 社群平台：SSR + CSR 混合
  • 行銷著陸頁：SSG + 局部 CSR

應用場景

場景一：電商商品頁。商品詳情使用 ISR，設定 60 秒 revalidate。絕大多數請求命中快取享受 CDN 速度，庫存等即時資訊透過客戶端 API 補充。結合 Streaming SSR 讓 Above-the-fold 內容優先呈現。

場景二：SaaS Dashboard。使用者登入後的個人化 Dashboard 使用 CSR，因為內容完全動態且無 SEO 需求。搭配 React Query 做客戶端資料快取與背景更新。

場景三：技術文件站。使用 SSG（如 Astro）在建構時產生所有頁面。搜尋功能使用客戶端 JavaScript（如 Algolia DocSearch）。部署至 CDN 後全球回應時間 < 50ms。

場景四：社群媒體動態牆。首屏使用 Streaming SSR 快速呈現前幾篇貼文，後續內容透過客戶端 Infinite Scroll 載入。React Server Components 用於非互動的貼文內容渲染，減少客戶端 JS 體積。
