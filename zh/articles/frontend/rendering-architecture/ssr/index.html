SSR 伺服器渲染

深入解析伺服器端渲染（SSR）的運作原理、架構模式、效能優化與現代框架的實作差異

伺服器端渲染（Server-Side Rendering, SSR）在伺服器上執行前端框架程式碼，產生完整 HTML 後傳送至瀏覽器。相較於 CSR，SSR 顯著改善首次內容顯示速度與 SEO 表現，但也引入了伺服器運算成本與 Hydration 複雜度。現代框架的 SSR 實作已遠超傳統的字串拼接，結合串流、選擇性水合等技術大幅提升體驗。

核心概念

SSR 運作流程

階段位置動作
1. 請求到達Server路由匹配、資料取得
2. 渲染 HTMLServer元件渲染為 HTML 字串/串流
3. 傳送回應NetworkHTML + CSS 傳至瀏覽器
4. 首次繪製Client瀏覽器解析 HTML 即顯示
5. JS 載入Client下載並解析 JS Bundle
6. HydrationClient綁定事件、啟動互動

SSR 效能指標

指標SSR 表現對比 CSR
TTFB較慢（需伺服器渲染）較快（靜態回應）
FCP快（HTML 即有內容）慢（需等 JS）
LCP快（關鍵內容已在 HTML）慢（需 API 回應）
TTI中（需等 Hydration）中（JS 執行後）
INP好（Hydration 後流暢）好

SSR 模式分類

模式說明框架支援
傳統 SSR每次請求完整渲染全部
Streaming SSR漸進式傳送 HTMLReact 18+, Vue
Selective Hydration優先水合互動區域React 18+
Progressive Hydration延遲水合非關鍵部分手動實作
Partial Hydration僅水合互動元件Astro, Qwik
Resumability無需重新執行元件Qwik

策略時機優勢劣勢

應用場景

SEO 關鍵型應用

電商產品頁、新聞媒體、內容平台等需要搜尋引擎良好索引的應用，SSR 確保爬蟲收到完整 HTML 內容。搭配結構化資料（Schema.org）與 Open Graph 標記，最大化搜尋引擎與社群媒體的展示效果。

首次載入敏感型應用

對於行動裝置使用者或低網速環境，SSR 的快速 FCP 至關重要。使用者在 JavaScript 載入前就能閱讀內容，配合 Streaming SSR 和 Selective Hydration，可在保持快速顯示的同時漸進啟用互動功能。

動態個人化內容

需要根據使用者身份、地理位置或 A/B 測試群組動態生成的頁面，SSR 比 SSG 更適合。伺服器可在渲染時存取 Cookie、Header 等請求資訊，產生個人化內容。搭配 Edge SSR 可減少延遲。
