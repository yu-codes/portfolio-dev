CSR 客戶端渲染

全面解析客戶端渲染（CSR）的運作機制、效能特徵、適用場景與現代優化策略

客戶端渲染（Client-Side Rendering, CSR）是現代單頁應用（SPA）的基礎渲染模式。伺服器僅提供最小 HTML 殼與 JavaScript Bundle，所有頁面內容由瀏覽器下載、解析並執行 JS 後動態生成。CSR 帶來了流暢的使用者互動體驗，但也面臨首次載入效能與 SEO 的挑戰。

核心概念

CSR 運作流程

階段動作耗時因素
1. HTML 下載極小 HTML（空殼）網路 RTT
2. JS Bundle 下載整個應用 JavaScriptBundle 大小 + 頻寬
3. JS 解析執行瀏覽器引擎處理CPU 效能
4. 資料取得API 請求伺服器回應 + 網路
5. DOM 渲染產生並插入 DOM頁面複雜度
6. 可互動事件綁定完成JS 執行完成

CSR 的效能指標影響

Web VitalCSR 表現原因
FCP (First Contentful Paint)差需等 JS 執行才有內容
LCP (Largest Contentful Paint)差主要內容依賴 API 回應
FID/INP (Interaction)中JS 執行完後互動流暢
CLS (Cumulative Layout Shift)中動態載入可能位移
TTFB (Time to First Byte)好靜態檔案快速回應

CSR 優劣分析

優勢劣勢
互動體驗流暢（SPA 導航）首次載入慢
伺服器負載低SEO 不友善
前後端完全分離JavaScript 強制依賴
部署簡單（CDN 靜態託管）低階裝置效能差
離線支援容易（Service Worker）首次白屏時間長

應用場景

Progressive Web App

CSR 搭配 Service Worker 可建構完整的離線體驗。Service Worker 快取 App Shell 和靜態資源，後續訪問即使無網路也能顯示 UI，資料層透過快取策略（Cache First / Network First）提供離線可用的內容。

嵌入式微前端

作為宿主應用的一部分嵌入時，CSR 的獨立性是優勢。每個微前端 Bundle 獨立載入和執行，無需伺服器端配合。Module Federation 或 Web Components 封裝後，可在任何宿主環境中執行。

即時通訊與協作

聊天應用、協作編輯器等高互動場景，CSR 的持久 WebSocket 連接和客戶端狀態管理是必要的。伺服器渲染對這類應用幾無意義，因為內容完全由即時互動產生。
