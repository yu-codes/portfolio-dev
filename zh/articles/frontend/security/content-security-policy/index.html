Content Security Policy（CSP）是一種額外的安全層，透過 HTTP 標頭或 meta 標籤告訴瀏覽器哪些資源可以被載入和執行。CSP 是防禦 XSS 攻擊最強大的瀏覽器原生機制，它能有效阻止未經授權的腳本執行，即使攻擊者成功注入了惡意程式碼。

核心概念

CSP 指令（Directives）

CSP 使用指令來控制不同類型資源的載入政策：

  • default-src：所有資源的預設策略
  • script-src：JavaScript 的載入來源
  • style-src：CSS 樣式表的載入來源
  • img-src：圖片的載入來源
  • connect-src：AJAX、WebSocket 等連線的目標
  • font-src：字型檔案的載入來源
  • frame-src：iframe 的載入來源
  • media-src：音訊和影片的載入來源
  • object-src：插件的載入來源

來源值（Source Values）

指令的來源值定義允許的資源來源：

來源值說明
'self'同源
'none'禁止所有來源
'unsafe-inline'允許行內腳本/樣式（不建議）
'unsafe-eval'允許 eval()（不建議）
'nonce-{random}'允許帶有指定 nonce 的元素
'strict-dynamic'信任已授權腳本載入的子腳本
https:允許所有 HTTPS 來源
data:允許 data: URI
specific-domain.com允許特定域名

Nonce 機制

Nonce 是每次請求生成的隨機值，用於標記合法的行內腳本：

<!-- 伺服器每次生成新的 nonce -->
<script nonce="abc123">
  // 只有帶正確 nonce 的腳本才會執行
  console.log('Legitimate script');
</script>

CSP 標頭中需要包含對應的 nonce：

Content-Security-Policy: script-src 'nonce-abc123'

Hash 機制

透過內容的 SHA 雜湊值授權特定的行內腳本：

Content-Security-Policy: script-src 'sha256-base64EncodedHash'

適合內容固定不變的行內腳本，不需要動態生成 nonce。

Report 機制

CSP 提供違規報告機制：

  • report-uri：指定接收違規報告的端點（已棄用）
  • report-to：使用 Reporting API 發送報告
  • Content-Security-Policy-Report-Only：僅報告不阻擋，適合測試

架構總覽

CSP 部署流程

┌─────────────────────────────────────────────┐
│  Phase 1: 監控（Report-Only 模式）           │
│  ├─ 部署 Report-Only 標頭                   │
│  ├─ 收集違規報告                             │
│  └─ 分析合法 vs 惡意來源                     │
├─────────────────────────────────────────────┤
│  Phase 2: 調整                               │
│  ├─ 根據報告調整策略                         │
│  ├─ 新增合法來源到白名單                     │
│  └─ 移除不必要的寬鬆規則                     │
├─────────────────────────────────────────────┤
│  Phase 3: 強制執行                           │
│  ├─ 切換到強制模式                           │
│  ├─ 持續監控違規報告                         │
│  └─ 定期審查和更新策略                       │
└─────────────────────────────────────────────┘

完整安全標頭架構

CSP 通常與其他安全標頭搭配使用：

Content-Security-Policy: ...
Strict-Transport-Security: max-age=31536000; includeSubDomains
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()

關鍵比較

CSP 層級安全性實作難度適用場景
基礎（白名單域名）中低快速導入
嚴格（nonce-based）高中現代應用
最嚴格（hash + no inline）最高高高安全需求

CSP 策略比較

策略優點缺點
白名單策略易理解可能被繞過（JSONP 端點）
Nonce 策略安全性高需伺服器端渲染支援
strict-dynamic支援動態載入舊瀏覽器不支援
Hash 策略無需動態生成腳本變更需更新 hash

Report-Only vs 強制模式

模式行為用途
Report-Only僅報告，不阻擋測試和部署準備
Enforce阻擋違規資源正式防護
同時使用強制現有 + 測試新規則漸進式收緊策略

應用場景

新專案導入 CSP

從零開始建立 CSP 策略：

  • 使用 Report-Only 模式收集基線資料
  • 識別所有合法的第三方資源
  • 逐步收緊策略直到達到安全標準
  • 自動化 nonce 生成整合到 CI/CD

現有專案升級 CSP

為已有的專案加入 CSP：

  • 審計所有行內腳本和樣式
  • 將行內腳本外部化或加入 nonce
  • 清點第三方腳本並建立白名單
  • 使用 Report-Only 模式驗證不會破壞功能

微前端架構

多團隊共同維護的微前端應用：

  • 定義基礎 CSP 策略供所有微應用遵循
  • 各微應用聲明需要的額外來源
  • 動態合併 CSP 策略
  • 使用 strict-dynamic 支援動態載入模組

第三方腳本管理

管控廣告、分析等第三方腳本：

  • 透過 CSP 限制第三方腳本的行為
  • 使用 connect-src 控制資料外傳目標
  • 配合 Subresource Integrity（SRI）驗證完整性
  • 監控第三方腳本的 CSP 違規報告
