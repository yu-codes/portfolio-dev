在前端應用中安全地儲存敏感資料是一項重要但經常被忽視的挑戰。從存取令牌到使用者個人資料，不當的儲存方式可能導致資料洩漏、身份冒用等嚴重安全問題。本文探討前端安全儲存的最佳實踐，幫助開發者在便利性和安全性之間取得平衡。

核心概念

前端儲存的威脅模型

前端儲存面臨的主要威脅：

  • XSS 攻擊：惡意腳本存取 LocalStorage 或非 HttpOnly 的 Cookie
  • CSRF 攻擊：利用自動附帶的 Cookie 發送偽造請求
  • 中間人攻擊：未加密傳輸導致資料被攔截
  • 物理存取：裝置遺失或被盜時的資料暴露
  • 第三方腳本：不受信任的第三方程式碼存取儲存空間

Token 儲存策略

不同 Token 類型的儲存建議：

Access Token：

  • 最佳：僅存在記憶體中（JavaScript 變數）
  • 次佳：HttpOnly Cookie（需 CSRF 防護）
  • 避免：LocalStorage（XSS 可竊取）

Refresh Token：

  • 最佳：HttpOnly + Secure + SameSite Cookie
  • 搭配 Token Rotation 機制
  • 伺服器端追蹤已發放的 Refresh Token

CSRF Token：

  • 存在 meta 標籤或非 HttpOnly Cookie
  • 每次請求透過自訂標頭或表單欄位發送

加密儲存

在客戶端進行加密儲存的考量：

  • Web Crypto API：瀏覽器原生的密碼學 API
  • 金鑰管理：加密金鑰本身的儲存是最大挑戰
  • 對稱加密：AES-GCM 適合客戶端資料加密
  • 金鑰衍生：使用 PBKDF2 從密碼衍生加密金鑰

安全清除

確保敏感資料在不需要時被徹底清除：

  • 登出時清除所有儲存的敏感資料
  • Token 過期時立即從記憶體中移除
  • 使用 crypto.getRandomValues() 覆寫敏感變數
  • 注意 JavaScript 垃圾回收可能保留記憶體副本

Token 生命週期管理

登入成功
  │
  ▼
伺服器發放 Access Token + Refresh Token
  │
  ├─ Access Token → 記憶體變數（短效 15min）
  │
  └─ Refresh Token → HttpOnly Cookie（長效 7d）
  
Access Token 過期
  │
  ▼
使用 Refresh Token 靜默更新
  │
  ▼
舊 Refresh Token 作廢，發放新的一組 Token

登出
  │
  ▼
清除記憶體中的 Access Token
清除 Cookie 中的 Refresh Token
伺服器端撤銷所有 Token

BFF（Backend for Frontend）模式

┌──────────┐      ┌──────────┐      ┌──────────┐
│ Browser  │─────→│   BFF    │─────→│  API     │
│          │Cookie│          │Token │  Server  │
│(無 Token)│←─────│(管理Token)│←─────│          │
└──────────┘      └──────────┘      └──────────┘

BFF 將 Token 管理責任從前端移至後端，前端僅使用 HttpOnly Cookie 與 BFF 通訊。

關鍵比較

儲存方式XSS 安全CSRF 安全持久性適用資料
記憶體變數較安全安全頁面關閉即消失Access Token
HttpOnly Cookie安全需防護可設定Refresh Token
LocalStorage不安全安全永久非敏感設定
SessionStorage不安全安全分頁關閉臨時非敏感資料
Encrypted Storage較安全安全依底層中等敏感資料

應用場景

SPA 認證架構

單頁應用的安全認證實作：

  • Access Token 存在記憶體中的閉包變數
  • Refresh Token 存在 HttpOnly Secure Cookie
  • 實作 Token 靜默更新機制（Silent Refresh）
  • 多分頁透過 BroadcastChannel 同步登入狀態

離線優先應用

需要在離線時存取敏感資料的應用：

  • 使用 Web Crypto API 加密後存入 IndexedDB
  • 金鑰從使用者密碼衍生（PBKDF2）
  • 上線後與伺服器同步並更新加密金鑰
  • 設定資料過期機制防止長期暴露

多租戶 SaaS 應用

切換組織/帳號的場景：

  • 每個租戶的 Token 獨立管理
  • 安全地切換 Context 不洩漏其他租戶資料
  • 統一的 Token 撤銷機制

金融和醫療應用

高合規要求的應用：

  • 最小化客戶端儲存的敏感資料
  • 實作閒置逾時自動登出
  • 敏感操作要求重新驗證
  • 完整的稽核日誌記錄
