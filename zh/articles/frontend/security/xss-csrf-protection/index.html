XSS 與 CSRF 防護

XSS（Cross-Site Scripting）和 CSRF（Cross-Site Request Forgery）是 Web 應用中最常見的兩大安全威脅。XSS 允許攻擊者在受害者的瀏覽器中執行惡意腳本，而 CSRF 則利用已認證使用者的身份發送未經授權的請求。深入理解這兩種攻擊的原理和防禦方式，是每位前端工程師的必備技能。

核心概念

XSS 攻擊類型

反射型 XSS（Reflected XSS）： 惡意腳本透過 URL 參數注入，伺服器將其反射回頁面中：

  • 攻擊者製作包含惡意程式碼的 URL
  • 受害者點擊後，伺服器回傳包含惡意腳本的頁面
  • 瀏覽器執行惡意腳本

儲存型 XSS（Stored XSS）： 惡意腳本被永久儲存在伺服器（如資料庫）中：

  • 攻擊者將惡意程式碼提交到論壇/留言板
  • 其他使用者瀏覽該頁面時自動執行惡意腳本
  • 影響範圍更廣，危害更大

DOM-Based XSS： 惡意腳本在客戶端 DOM 操作中被注入：

  • 不經過伺服器，純粹在前端發生
  • 常見於 innerHTML、document.write()、eval() 等操作
  • 伺服器端掃描難以偵測

CSRF 攻擊原理

CSRF 利用瀏覽器自動附帶 Cookies 的行為：

  1. 使用者登入目標網站（例如銀行）
  2. 使用者造訪惡意網站
  3. 惡意網站自動發送偽造請求到目標網站
  4. 瀏覽器自動附帶目標網站的 Cookie
  5. 伺服器認為是合法請求並執行

XSS 防禦策略

  • 輸出編碼（Output Encoding）：根據上下文對資料進行編碼
  • 輸入驗證（Input Validation）：在伺服器端驗證和清理使用者輸入
  • Content Security Policy：限制腳本載入來源
  • HttpOnly Cookie：防止 JavaScript 存取敏感 Cookie
  • 使用安全的 API：偏好 textContent 而非 innerHTML

CSRF 防禦策略

  • SameSite Cookie：限制跨站請求攜帶 Cookie
  • CSRF Token：在表單中加入不可預測的 Token
  • Double Submit Cookie：將 Token 同時放在 Cookie 和請求中
  • 驗證 Origin/Referer：檢查請求來源
  • 自訂請求標頭：AJAX 請求加入自訂標頭

完整防禦架構

在現代 Web 應用中，XSS 和 CSRF 防禦應融入整體安全架構：

  • WAF 層：攔截已知攻擊模式
  • 應用層：輸入驗證、輸出編碼、Token 驗證
  • 框架層：利用 React/Vue 的自動轉義
  • 瀏覽器層：CSP、SameSite Cookie

應用場景

使用者生成內容（UGC）平台

處理使用者提交的內容（如論壇、部落格）：

  • 使用 DOMPurify 等函式庫消毒 HTML 內容
  • 實施白名單策略，僅允許安全的 HTML 標籤
  • 對動態插入的內容進行嚴格的 CSP 控制

電商和金融應用

高安全需求的場景：

  • 所有表單操作加入 CSRF Token
  • 敏感操作要求二次驗證
  • 嚴格的 CSP 策略防止腳本注入
  • 交易前驗證 Referer 和 Origin

單頁應用（SPA）

前後端分離架構的防護：

  • API 請求自動加入 CSRF Token 或自訂標頭
  • 利用框架的自動轉義機制
  • 避免使用 eval() 和 innerHTML
  • 嚴格控制第三方腳本

第三方整合

嵌入外部內容或提供 Widget：

  • 使用 <iframe> 搭配 sandbox 屬性隔離第三方內容
  • 透過 postMessage 安全地跨域通訊
  • 限制第三方腳本的存取權限
