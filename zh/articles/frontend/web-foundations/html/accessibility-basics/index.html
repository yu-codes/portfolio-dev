網頁無障礙（Web Accessibility，簡稱 a11y）確保所有使用者——包括視覺、聽覺、運動或認知障礙者——都能平等地存取與使用網頁內容。本文介紹 WCAG 標準、ARIA 屬性與實務開發中的無障礙最佳實踐。

核心概念

WCAG 四大原則（POUR）

Web Content Accessibility Guidelines（WCAG）定義了四個核心原則：

  1. 可感知（Perceivable）：資訊必須以使用者能察覺的方式呈現

    • 為圖片提供替代文字
    • 影片提供字幕與描述
    • 內容可在不同呈現方式下保持意義

  2. 可操作（Operable）：介面組件必須可被操作

    • 所有功能可透過鍵盤操作
    • 給予足夠的操作時間
    • 避免引發癲癇的內容

  3. 可理解（Understandable）：資訊與操作方式必須可被理解

    • 文字可讀且可理解
    • 頁面行為可預測
    • 幫助使用者避免與修正錯誤

  4. 穩健（Robust）：內容必須與各種使用者代理相容

    • 與輔助技術相容
    • 使用標準化標記

WCAG 符合等級

  • Level A：最基本的無障礙要求
  • Level AA：大多數法規要求的等級（推薦目標）
  • Level AAA：最高等級，非所有內容都能達成

輔助技術類型

  • 螢幕閱讀器：NVDA、JAWS、VoiceOver
  • 螢幕放大鏡：ZoomText
  • 語音輸入：Dragon NaturallySpeaking
  • 替代輸入裝置：開關控制、眼動追蹤

架構總覽

ARIA 屬性系統

ARIA（Accessible Rich Internet Applications）補充了原生 HTML 語義不足之處：

<!-- ARIA 角色（Roles） -->
<div role="navigation" aria-label="主選單">
  <ul role="menubar">
    <li role="menuitem"><a href="/">首頁</a></li>
    <li role="menuitem" aria-haspopup="true" aria-expanded="false">
      <a href="/products">產品</a>
      <ul role="menu" aria-hidden="true">
        <li role="menuitem"><a href="/products/a">產品 A</a></li>
      </ul>
    </li>
  </ul>
</div>

<!-- ARIA 狀態與屬性 -->
<button aria-pressed="false" aria-label="切換深色模式">
  🌙
</button>

<!-- 即時區域（Live Regions） -->
<div role="alert" aria-live="assertive">
  <!-- 動態錯誤訊息會自動被朗讀 -->
</div>

<div aria-live="polite" aria-atomic="true">
  <!-- 非緊急的狀態更新 -->
  <p>已載入 10 筆新結果</p>
</div>

焦點管理架構

// 模態對話框的焦點捕獲
class FocusTrap {
  constructor(element) {
    this.element = element;
    this.focusableElements = element.querySelectorAll(
      'a[href], button, input, select, textarea, [tabindex]:not([tabindex="-1"])'
    );
    this.firstElement = this.focusableElements[0];
    this.lastElement = this.focusableElements[this.focusableElements.length - 1];
  }

  activate() {
    this.firstElement.focus();
    this.element.addEventListener('keydown', this.handleKeyDown.bind(this));
  }

  handleKeyDown(e) {
    if (e.key === 'Tab') {
      if (e.shiftKey && document.activeElement === this.firstElement) {
        e.preventDefault();
        this.lastElement.focus();
      } else if (!e.shiftKey && document.activeElement === this.lastElement) {
        e.preventDefault();
        this.firstElement.focus();
      }
    }
    if (e.key === 'Escape') {
      this.deactivate();
    }
  }

  deactivate() {
    this.element.addEventListener('keydown', this.handleKeyDown);
  }
}

關鍵比較

原生語義 vs ARIA

情境建議做法避免做法
按鈕<button>點擊</button><div role="button">點擊</div>
連結<a href="/page">連結</a><span role="link" onclick>連結</span>
標題<h2>標題</h2><div role="heading" aria-level="2">
核取方塊<input type="checkbox"><div role="checkbox">

第一規則：能用原生 HTML 就不用 ARIA。

aria-hidden vs display: none vs visibility: hidden

方式視覺可見螢幕閱讀器可見佔據空間
aria-hidden="true"✅❌✅
display: none❌❌❌
visibility: hidden❌❌✅
.sr-only class❌✅❌

aria-live 的 politeness 等級

  • off：不主動通知（預設）
  • polite：等待使用者空閒時通知
  • assertive：立即中斷目前朗讀來通知

應用場景

場景一：自訂下拉選單

原生 <select> 樣式受限時，自訂元件必須完整實現鍵盤操作與 ARIA：

<div role="listbox" aria-label="選擇國家" tabindex="0"
     aria-activedescendant="option-tw">
  <div role="option" id="option-tw" aria-selected="true">台灣</div>
  <div role="option" id="option-jp" aria-selected="false">日本</div>
  <div role="option" id="option-us" aria-selected="false">美國</div>
</div>

場景二：通知與 Toast 訊息

使用 aria-live="polite" 讓螢幕閱讀器在適當時機朗讀新出現的通知。

場景三：SPA 路由切換

單頁應用切換頁面時不會觸發瀏覽器的頁面載入事件，需手動管理焦點與通知：

router.afterEach((to) => {
  document.title = to.meta.title;
  const main = document.getElementById('main-content');
  main.focus();
  announceRouteChange(to.meta.title);
});

場景四：圖表與資料視覺化

為圖表提供替代的表格形式或文字摘要，確保非視覺使用者也能獲取資訊。
