CORS 跨域資源共享

CORS（Cross-Origin Resource Sharing，跨域資源共享）是瀏覽器實作的安全機制，用於控制不同來源之間的資源存取。對於前端開發者而言，理解 CORS 的運作原理是解決跨域問題、設計 API 架構與確保應用安全性的必備知識。

核心概念

同源政策（Same-Origin Policy）

同源政策是瀏覽器最基本的安全機制，規定只有相同來源的資源才能互相存取。所謂「同源」必須滿足三個條件完全一致：

  • 協定（Protocol）：http 與 https 不同源
  • 主機（Host）：example.com 與 api.example.com 不同源
  • 埠號（Port）：localhost:3000 與 localhost:8080 不同源
https://example.com:443/path
  │         │        │
  協定      主機     埠號

CORS 的角色

CORS 是一種「放寬」同源政策的機制，讓伺服器透過 HTTP 標頭明確聲明哪些跨域請求是被允許的。重要的是，CORS 是由瀏覽器強制執行的——伺服器端的 API 不受此限制，這就是為什麼使用 Postman 或 curl 不會遇到 CORS 問題。

請求分類

CORS 將跨域請求分為兩類：

簡單請求（Simple Request）：

  • 方法：GET、HEAD、POST
  • 標頭：僅限 Accept、Accept-Language、Content-Language、Content-Type（值限於 application/x-www-form-urlencoded、multipart/form-data、text/plain）
  • 不使用 ReadableStream

預檢請求（Preflight Request）：

  • 使用 PUT、DELETE、PATCH 等方法
  • 包含自定義標頭（如 Authorization、X-Custom-Header）
  • Content-Type 為 application/json
  • 瀏覽器會先發送 OPTIONS 請求確認伺服器允許

架構總覽

簡單請求流程

瀏覽器                                    伺服器
  │                                         │
  ├─── GET /api/data ──────────────────────►│
  │    Origin: https://app.example.com      │
  │                                         │
  │◄── 200 OK ─────────────────────────────┤
  │    Access-Control-Allow-Origin: *       │
  │    Content-Type: application/json       │
  │                                         │

預檢請求流程

瀏覽器                                    伺服器
  │                                         │
  ├─── OPTIONS /api/data ──────────────────►│  ← 預檢請求
  │    Origin: https://app.example.com      │
  │    Access-Control-Request-Method: PUT   │
  │    Access-Control-Request-Headers:      │
  │      Authorization, Content-Type        │
  │                                         │
  │◄── 204 No Content ─────────────────────┤  ← 預檢回應
  │    Access-Control-Allow-Origin:         │
  │      https://app.example.com            │
  │    Access-Control-Allow-Methods:        │
  │      GET, POST, PUT, DELETE             │
  │    Access-Control-Allow-Headers:        │
  │      Authorization, Content-Type        │
  │    Access-Control-Max-Age: 86400        │
  │                                         │
  ├─── PUT /api/data ──────────────────────►│  ← 實際請求
  │    Origin: https://app.example.com      │
  │    Authorization: Bearer token          │
  │    Content-Type: application/json       │
  │                                         │
  │◄── 200 OK ─────────────────────────────┤  ← 實際回應
  │    Access-Control-Allow-Origin:         │
  │      https://app.example.com            │

關鍵 CORS 標頭

回應標頭（伺服器設定）：

標頭用途範例值
Access-Control-Allow-Origin允許的來源https://app.example.com 或 *
Access-Control-Allow-Methods允許的方法GET, POST, PUT, DELETE
Access-Control-Allow-Headers允許的標頭Authorization, Content-Type
Access-Control-Allow-Credentials是否允許攜帶憑證true
Access-Control-Expose-Headers前端可存取的回應標頭X-Total-Count
Access-Control-Max-Age預檢快取時間（秒）86400

請求標頭（瀏覽器自動設定）：

標頭用途
Origin請求來源
Access-Control-Request-Method預檢時聲明實際方法
Access-Control-Request-Headers預檢時聲明自定義標頭

關鍵比較

CORS 解決方案比較

方案適用環境優點缺點
伺服器設定 CORS 標頭生產環境標準做法、安全需後端配合
代理伺服器（Proxy）開發/生產前端獨立處理增加架構複雜度
JSONP舊系統相容不受 CORS 限制僅支援 GET、有安全風險
開發伺服器代理開發環境快速解決僅限開發
postMessageiframe 通訊靈活限定場景

credentials 模式比較

模式行為Allow-Origin 限制
same-origin同源才帶 Cookie無限制
include跨域也帶 Cookie不可使用 *
omit不帶 Cookie無限制

常見錯誤與解法

錯誤訊息原因解法
No ‘Access-Control-Allow-Origin’伺服器未設定 CORS 標頭新增對應標頭
Credential is not supported if Allow-Origin is ‘*’使用 credentials 時不可用 *指定具體來源
Method not allowed預檢未允許該方法新增 Allow-Methods
Request header not allowed預檢未允許該標頭新增 Allow-Headers

應用場景

場景一：前後端分離架構

// 前端 fetch 請求
fetch('https://api.example.com/users', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${token}`
  },
  credentials: 'include',
  body: JSON.stringify({ name: 'Yu' })
});

// Express 後端 CORS 設定
const cors = require('cors');
app.use(cors({
  origin: 'https://app.example.com',
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
  allowedHeaders: ['Content-Type', 'Authorization'],
  credentials: true,
  maxAge: 86400
}));

場景二：Vite 開發伺服器代理

// vite.config.js
export default defineConfig({
  server: {
    proxy: {
      '/api': {
        target: 'https://api.example.com',
        changeOrigin: true,
        rewrite: (path) => path.replace(/^\/api/, '')
      }
    }
  }
});

場景三：多來源白名單

// 動態設定 Allow-Origin
const allowedOrigins = [
  'https://app.example.com',
  'https://admin.example.com'
];

app.use((req, res, next) => {
  const origin = req.headers.origin;
  if (allowedOrigins.includes(origin)) {
    res.setHeader('Access-Control-Allow-Origin', origin);
    res.setHeader('Vary', 'Origin');
  }
  next();
});

場景四：預檢快取最佳化

透過 Access-Control-Max-Age 減少預檢請求次數，提升效能：

Access-Control-Max-Age: 86400  // 快取 24 小時

不同瀏覽器有最大值限制：Chrome 為 7200 秒，Firefox 為 86400 秒。

場景五：安全性考量

  • 避免使用 Access-Control-Allow-Origin: * 搭配 credentials
  • 驗證 Origin 標頭值，防止 Origin 偽造
  • 限制 Expose-Headers 僅暴露必要的標頭
  • 定期檢視 CORS 設定，移除不再需要的來源
