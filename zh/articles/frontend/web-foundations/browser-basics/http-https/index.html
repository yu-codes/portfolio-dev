HTTP（HyperText Transfer Protocol）與 HTTPS（HTTP Secure）是現代網路通訊的基石。理解兩者的差異與運作機制，對於前端開發者在效能優化、安全性設計與除錯能力上都至關重要。本文將深入剖析 HTTP 與 HTTPS 的核心原理、協定演進、加密機制以及實務應用場景。

核心概念

HTTP 協定基礎

HTTP 是一種無狀態（stateless）的應用層協定，建立在 TCP/IP 之上，負責客戶端與伺服器之間的資料傳輸。其核心特性包括：

  • 請求-回應模型（Request-Response）：客戶端發送請求，伺服器返回回應
  • 無狀態性：每次請求都是獨立的，伺服器不會記住先前的互動
  • 文字型協定：HTTP/1.x 使用純文字格式傳輸標頭資訊
  • 可擴展性：透過 Header 機制支援各種功能擴展

HTTP 請求由方法（GET、POST、PUT、DELETE 等）、URL、標頭與主體組成。回應則包含狀態碼（如 200、301、404、500）、標頭與回應主體。

HTTPS 與 TLS 加密

HTTPS 本質上是在 HTTP 與 TCP 之間加入 TLS（Transport Layer Security）層，提供三大安全保障：

  • 加密（Encryption）：使用對稱加密保護傳輸中的資料不被竊聽
  • 身份驗證（Authentication）：透過數位憑證確認伺服器身份
  • 完整性（Integrity）：使用 MAC（Message Authentication Code）確保資料未被篡改

TLS 握手過程（以 TLS 1.3 為例）：

  1. Client Hello：客戶端發送支援的加密套件與金鑰共享參數
  2. Server Hello：伺服器選擇加密套件並回傳憑證
  3. 金鑰交換：使用 ECDHE 等演算法建立共享密鑰
  4. 完成握手：雙方確認加密通道建立

HTTP 版本演進

版本主要特性連線方式
HTTP/1.0每次請求建立新連線短連線
HTTP/1.1持久連線、管線化Keep-Alive
HTTP/2多路復用、標頭壓縮、伺服器推送單一 TCP 連線
HTTP/3基於 QUIC/UDP、零 RTTUDP 連線

架構總覽

HTTP 通訊架構

客戶端 (Browser)
    │
    ├── DNS 解析 → 取得伺服器 IP
    │
    ├── TCP 三次握手 → 建立連線
    │
    ├── [TLS 握手] → 建立加密通道（僅 HTTPS）
    │
    ├── HTTP 請求 → 發送資料
    │
    ├── HTTP 回應 ← 接收資料
    │
    └── 連線管理 → Keep-Alive 或關閉

憑證信任鏈

HTTPS 的信任機制依賴 PKI（Public Key Infrastructure）：

  • 根憑證機構（Root CA）：預裝在作業系統與瀏覽器中
  • 中繼憑證機構（Intermediate CA）：由根 CA 簽發
  • 伺服器憑證（Server Certificate）：由中繼 CA 簽發給特定網域

瀏覽器驗證憑證時，會沿著信任鏈逐層向上驗證，直到找到受信任的根 CA。

加密套件組合

典型的 TLS 1.3 加密套件：

  • 金鑰交換：X25519 或 P-256（ECDHE）
  • 對稱加密：AES-256-GCM 或 ChaCha20-Poly1305
  • 雜湊函數：SHA-256 或 SHA-384

應用場景

場景一：前端開發中的混合內容問題

當 HTTPS 頁面載入 HTTP 資源時，瀏覽器會封鎖或警告：

<!-- 錯誤：混合內容 -->
<img src="http://example.com/image.png">

<!-- 正確：使用相對協定或 HTTPS -->
<img src="https://example.com/image.png">
<img src="//example.com/image.png">

場景二：本地開發 HTTPS 環境

使用 mkcert 建立本地開發憑證：

# 安裝 mkcert
brew install mkcert
mkcert -install

# 產生本地憑證
mkcert localhost 127.0.0.1 ::1

場景三：HTTP/2 效能優化

HTTP/2 的多路復用消除了 HTTP/1.1 的隊頭阻塞：

  • 不再需要域名分散（domain sharding）
  • 不再需要合併 CSS/JS 檔案
  • 伺服器推送可預載關鍵資源

場景四：HSTS 強制安全連線

透過 HTTP Strict Transport Security 標頭強制使用 HTTPS：

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

場景五：API 通訊安全

前端與後端 API 通訊時的安全考量：

  • 所有 API 端點必須使用 HTTPS
  • 實作 Certificate Pinning 防止中間人攻擊
  • 使用 CORS 限制跨域存取
