工程模式

前端設計模式與架構模式

前端工程模式涵蓋了從程式碼組織、專案架構到團隊協作的一系列最佳實踐。隨著前端應用的規模與複雜度急劇成長，單一應用、單一團隊的開發模式已無法滿足企業需求。Monorepo、微前端、設計系統等工程模式的出現，正是為了解決大型前端專案的可擴展性與可維護性問題。

核心概念

關注點分離（Separation of Concerns）：工程模式的根基。不僅在程式碼層面將邏輯、樣式、結構分離，更在架構層面將不同業務域、不同團隊的關注點隔離，使得各部分能獨立開發、測試與部署。

抽象層級（Levels of Abstraction）：好的工程模式會建立清晰的抽象層級。底層是設計令牌（Design Tokens），往上是基礎元件、複合元件，再到頁面模板與完整應用。每一層只依賴其下層，形成穩定的依賴方向。

約定優於配置（Convention over Configuration）：大型團隊需要一致的程式碼風格、檔案命名、目錄結構。透過建立並執行約定（搭配 ESLint、Prettier、commitlint 等工具），降低團隊溝通成本與認知負擔。

漸進式採用（Incremental Adoption）：優秀的工程模式允許漸進式導入。不需要一次性重構整個專案，而是能在現有架構上逐步引入新模式，這對於有大量歷史程式碼的專案至關重要。

應用場景

大型電商平台：首頁、商品頁、購物車、結帳流程分別由不同團隊負責。採用微前端架構，各團隊可使用不同技術棧（首頁用 Next.js 追求 SEO，購物車用 React SPA 追求互動性），透過 Module Federation 共享設計系統元件。

企業內部工具平台：多個內部應用（HR 系統、CRM、專案管理）使用 Monorepo 管理，共享基礎 UI 元件庫與工具函式。Nx 的受影響分析確保只建構和測試有變更的部分，大幅縮短 CI 時間。

跨產品設計一致性：公司旗下多個產品需要統一的視覺風格。建立設計系統，定義 Design Tokens（色彩、字型、間距），發布為獨立的 npm 套件。設計師在 Figma 中使用相同的 Token，確保設計稿與程式碼的一致性。

遺留系統漸進式現代化：舊的 jQuery 或 AngularJS 應用無法一次性重寫。使用 Single-SPA 將新功能以 React 微前端的形式嵌入舊應用，逐步取代舊元件，最終完成遷移。

開源元件庫：如 Radix UI、shadcn/ui 等專案使用 Monorepo 管理數十個獨立元件套件。每個元件可獨立安裝、獨立版本發布，但共享建構工具與測試基礎設施。
