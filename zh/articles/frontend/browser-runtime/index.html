瀏覽器執行環境

DOM、渲染引擎與瀏覽器 API

瀏覽器是前端程式碼的執行平台，理解瀏覽器的內部運作機制對於寫出高效能的 Web 應用至關重要。從 HTML 解析到像素繪製，從 JavaScript 執行到網路請求，瀏覽器內部包含多個精密協作的子系統。本文將深入探討瀏覽器執行環境的核心架構，幫助開發者理解「為什麼」某些做法能提升效能。

核心概念

瀏覽器的多程序架構

現代瀏覽器（如 Chrome）採用多程序架構，每個程序負責不同職責：

  • 瀏覽器程序（Browser Process）：管理地址列、書籤、網路請求、檔案存取
  • 渲染程序（Renderer Process）：每個頁籤獨立一個，負責 HTML/CSS/JS 處理
  • GPU 程序（GPU Process）：處理圖層合成與硬體加速繪製
  • 網路程序（Network Process）：統一處理所有網路請求
  • 插件程序（Plugin Process）：隔離執行第三方插件

此架構的優勢在於：一個頁籤崩潰不會影響其他頁籤，同時提供更好的安全隔離（站點隔離，Site Isolation）。

事件迴圈（Event Loop）

JavaScript 是單執行緒語言，透過事件迴圈機制實現非同步操作：

  1. 呼叫堆疊（Call Stack）：同步程式碼依序執行
  2. 微任務佇列（Microtask Queue）：Promise.then、MutationObserver，在每個任務結束後立即清空
  3. 巨集任務佇列（Macrotask Queue）：setTimeout、setInterval、I/O、UI 渲染
  4. 執行順序：同步程式碼 → 清空微任務 → 取一個巨集任務 → 重複

理解事件迴圈是避免 UI 卡頓與正確處理非同步邏輯的基礎。

關鍵渲染路徑（Critical Rendering Path）

瀏覽器將 HTML/CSS 轉換為像素的過程：

  1. HTML 解析 → DOM Tree（文件物件模型）
  2. CSS 解析 → CSSOM Tree（CSS 物件模型）
  3. 合併 → Render Tree（僅包含可見元素）
  4. 佈局（Layout） → 計算每個元素的幾何位置
  5. 繪製（Paint） → 將元素轉換為圖層上的像素
  6. 合成（Composite） → GPU 將多個圖層合併為最終畫面

應用場景

場景一：效能最佳化 — 避免 Layout Thrashing

當 JavaScript 交替讀取與修改 DOM 幾何屬性時，瀏覽器被迫多次同步計算佈局（Forced Synchronous Layout）。解法是批次讀取所有需要的值，再批次寫入修改，或使用 requestAnimationFrame 將寫入推遲到下一幀。

場景二：長任務拆分

當主執行緒被 JavaScript 長任務（>50ms）阻塞時，UI 會失去回應。可透過 scheduler.yield()（新 API）或 setTimeout(0) 將長任務拆分為多個小塊，讓瀏覽器有機會處理使用者互動與渲染。

場景三：離線優先應用

透過 Service Worker 攔截網路請求，實現 Cache First 或 Stale While Revalidate 策略。使用者在離線時仍可存取已快取的內容，恢復連線後自動同步資料。

場景四：複雜動畫效能

將動畫限制在 transform 與 opacity 屬性上，這兩者可由 GPU 合成器執行緒獨立處理，不會觸發主執行緒的 Layout/Paint。使用 will-change 提示瀏覽器提前建立合成圖層。
