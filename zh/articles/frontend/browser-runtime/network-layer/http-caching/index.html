HTTP 快取機制是網頁效能優化的基石，透過在客戶端或中間代理伺服器儲存回應副本，避免重複的網路請求，大幅降低頁面載入時間和伺服器負載。理解快取的運作原理對於前端工程師至關重要。

核心概念

快取類型

HTTP 快取依據儲存位置分為多種類型：

  • 瀏覽器快取（Private Cache）：儲存在使用者瀏覽器中，僅供該使用者使用
  • 代理快取（Shared Cache）：CDN 或反向代理伺服器上的共享快取
  • 閘道快取：位於伺服器前方的快取層

強快取（Strong Cache）

強快取直接使用本地副本，不與伺服器溝通：

  • Expires：指定絕對過期時間（HTTP/1.0），容易因時鐘不同步產生問題
  • Cache-Control: max-age：指定相對過期秒數（HTTP/1.1），優先於 Expires

當資源未過期時，瀏覽器直接從快取讀取，HTTP 狀態碼顯示為 200 (from cache)。

協商快取（Conditional Cache）

資源過期後，瀏覽器向伺服器驗證資源是否變更：

  • Last-Modified / If-Modified-Since：基於最後修改時間比對
  • ETag / If-None-Match：基於內容雜湊值比對，精確度更高

若資源未變更，伺服器回應 304 Not Modified，瀏覽器繼續使用快取副本。

Cache-Control 指令

Cache-Control 標頭提供精細的快取控制：

指令說明
public回應可被任何快取儲存
private僅允許瀏覽器快取
no-cache每次使用前必須驗證
no-store完全禁止快取
max-age=N快取有效期（秒）
s-maxage=N共享快取有效期
must-revalidate過期後必須重新驗證
stale-while-revalidate=N允許在背景驗證時提供過期內容

Memory Cache vs Disk Cache

  • Memory Cache：存在記憶體中，讀取速度極快，關閉分頁即消失
  • Disk Cache：存在磁碟中，持久保存，容量較大但速度略慢

瀏覽器會根據資源大小和使用頻率自動決定儲存位置。

資源類型建議策略
HTML 頁面no-cache 或短 max-age
CSS/JS（含 hash）max-age=31536000, immutable
圖片（含 hash）max-age=31536000, immutable
API 回應no-store 或短 max-age
字型檔案max-age=31536000

應用場景

靜態資源長效快取

搭配檔案名稱雜湊（Content Hash）實現永久快取：

  • 建置工具產出帶 hash 的檔名（如 app.a1b2c3.js）
  • 設定超長 max-age（一年）和 immutable
  • 內容變更時檔名自動更新，自然失效舊快取

SPA 應用快取策略

單頁應用的典型快取配置：

  • index.html：no-cache（確保每次取得最新版本）
  • JS/CSS chunk：max-age=31536000（搭配 content hash）
  • 靜態圖片：max-age=86400（一天）

CDN 快取層

CDN 作為共享快取的最佳實踐：

  • 使用 s-maxage 獨立控制 CDN 快取時間
  • 搭配 stale-while-revalidate 提供零延遲體驗
  • 透過 API 主動清除特定路徑的 CDN 快取

Service Worker 快取

使用 Service Worker 實現離線快取和精細快取策略：

  • Cache First：優先使用快取，適合靜態資源
  • Network First：優先使用網路，適合 API 請求
  • Stale While Revalidate：先回快取再背景更新
