佈局、繪製與合成

瀏覽器渲染管線的三大核心階段——佈局（Layout）、繪製（Paint）與合成（Composite）——決定了網頁內容如何從抽象的樣式規則轉化為螢幕上的實際像素。深入理解這三個階段的運作機制，是實現流暢 60fps 動畫與高效能介面的基礎。

核心概念

佈局（Layout / Reflow）

佈局階段負責計算每個可見元素在頁面中的精確幾何資訊：位置（x, y 座標）、尺寸（寬度、高度）、以及與其他元素的空間關係。

觸發佈局的操作：

  • 修改元素的幾何屬性（width、height、margin、padding）
  • 新增或移除 DOM 元素
  • 改變字型大小或文字內容
  • 讀取特定佈局屬性（offsetWidth、scrollTop 等）
  • 視窗大小改變（resize）

佈局的範圍：

  • 全域佈局（Global Layout）：影響整個文件的佈局重新計算，如視窗 resize
  • 增量佈局（Incremental Layout）：僅重新計算受影響的子樹，如修改特定元素的寬度

繪製（Paint / Rasterization）

繪製階段將佈局結果轉換為實際的像素資訊。瀏覽器遍歷 Render Tree，為每個節點執行繪製指令，生成像素陣列。

繪製順序（Stacking Order）：

  1. 背景顏色（background-color）
  2. 背景圖片（background-image）
  3. 邊框（border）
  4. 子元素
  5. 輪廓（outline）

繪製層（Paint Layers）： 瀏覽器會將頁面分割為多個繪製層，各層獨立繪製。建立獨立層的條件包含：

  • position: fixed 或 position: sticky
  • opacity 小於 1
  • CSS 濾鏡（filter）
  • will-change 屬性
  • <video>、<canvas> 元素
  • 具有 3D 變換的元素

合成（Composite）

合成階段將所有繪製層按照正確的堆疊順序（z-index、stacking context）組合成最終畫面。此階段由 GPU 處理，效能極高。

合成器執行緒（Compositor Thread）： 合成操作在獨立的合成器執行緒上執行，不會被主執行緒的 JavaScript 阻塞。這意味著純合成屬性的動畫（如 transform、opacity）即使在主執行緒繁忙時仍能保持流暢。

架構總覽

         主執行緒 (Main Thread)
┌─────────────────────────────────────┐
│  Style → Layout → Paint (Record)    │
└──────────────────┬──────────────────┘
                   │ Paint Records
                   ▼
    合成器執行緒 (Compositor Thread)
┌─────────────────────────────────────┐
│  Tiling → Rasterization → Composite │
└──────────────────┬──────────────────┘
                   │ GPU Commands
                   ▼
         GPU Process → Display

分塊處理（Tiling）

合成器會將繪製層分割為小塊（Tiles），優先光柵化視窗內的區塊，並利用多個光柵化執行緒（Raster Threads）平行處理。

GPU 加速

以下屬性的變更僅觸發合成，不需要重新佈局或繪製：

  • transform（位移、旋轉、縮放）
  • opacity（透明度）
  • filter（部分濾鏡效果）

這些操作直接在 GPU 上執行，效能成本極低，是實現高效能動畫的首選方式。

關鍵比較

階段觸發條件效能成本執行緒可優化方式
Layout幾何屬性變更高主執行緒避免強制同步佈局
Paint視覺屬性變更中主執行緒減少繪製區域
Compositetransform/opacity低合成器執行緒使用 will-change 提升層

will-change vs transform: translateZ(0)

方法機制適用時機注意事項
will-change明確告知瀏覽器即將變更動態添加，事後移除過度使用浪費記憶體
translateZ(0)強制建立合成層靜態提升hack 手法，語義不明確

應用場景

場景一：高效能 CSS 動畫

利用僅觸發合成的屬性實現 60fps 動畫：

/* 推薦：僅觸發 Composite */
.animate-good {
  transition: transform 0.3s ease, opacity 0.3s ease;
}
.animate-good:hover {
  transform: scale(1.05);
  opacity: 0.9;
}

/* 避免：觸發 Layout + Paint + Composite */
.animate-bad {
  transition: width 0.3s ease, margin 0.3s ease;
}
.animate-bad:hover {
  width: 110%;
  margin-left: -5%;
}

場景二：滾動效能優化

長列表或複雜頁面的滾動效能：

/* 將固定元素提升為獨立層 */
.fixed-header {
  position: fixed;
  will-change: transform;
}

/* 使用 contain 限制佈局影響範圍 */
.card-item {
  contain: layout style paint;
}

/* 延遲渲染離屏內容 */
.offscreen-section {
  content-visibility: auto;
  contain-intrinsic-size: 0 500px;
}

場景三：避免強制同步佈局

// 錯誤：讀寫交錯導致強制同步佈局
elements.forEach(el => {
  const width = el.offsetWidth; // 強制佈局
  el.style.width = width * 2 + 'px'; // 標記佈局無效
});

// 正確：批量讀取後批量寫入
const widths = elements.map(el => el.offsetWidth);
elements.forEach((el, i) => {
  el.style.width = widths[i] * 2 + 'px';
});

場景四：層管理策略

/* 動畫開始前提升層 */
.card {
  transition: transform 0.3s;
}
.card:hover {
  will-change: transform;
}
.card.animating {
  transform: translateY(-10px);
}
