關鍵渲染路徑

關鍵渲染路徑（Critical Rendering Path, CRP）是瀏覽器從接收 HTML、CSS 和 JavaScript 到最終像素渲染到螢幕上的完整流程。理解 CRP 是前端效能優化的基石，直接影響使用者感知的首次內容繪製（FCP）與最大內容繪製（LCP）等核心指標。

核心概念

關鍵渲染路徑由以下幾個關鍵步驟組成：

1. 建構 DOM 樹

瀏覽器接收到 HTML 位元組後，會經過字元解碼、Token 化、節點建構，最終組裝成完整的 DOM（Document Object Model）樹。DOM 樹是文件的結構化表示，每個 HTML 元素對應一個節點。

2. 建構 CSSOM 樹

與 DOM 平行，瀏覽器解析所有 CSS 樣式表（包含外部樣式表、內嵌樣式和行內樣式），建構 CSSOM（CSS Object Model）樹。CSSOM 反映了樣式的層疊與繼承關係。

3. 合併為 Render Tree

瀏覽器將 DOM 與 CSSOM 結合，生成 Render Tree。此樹僅包含可見元素（display: none 的元素會被排除），每個節點包含其計算後的樣式資訊。

4. 佈局（Layout）

根據 Render Tree，瀏覽器計算每個元素在視窗中的確切位置與尺寸。此步驟也稱為 Reflow，涉及盒模型計算、浮動定位等幾何運算。

5. 繪製（Paint）

將佈局結果轉換為實際像素，包含文字渲染、顏色填充、圖片繪製、邊框與陰影等視覺效果。

6. 合成（Composite）

將多個繪製層（Layers）按照正確順序合成最終畫面，利用 GPU 加速完成最終輸出。

架構總覽

HTML Bytes → Characters → Tokens → Nodes → DOM
                                              ↘
                                               Render Tree → Layout → Paint → Composite → Display
                                              ↗
CSS Bytes  → Characters → Tokens → Nodes → CSSOM

渲染阻塞資源

  • CSS 是渲染阻塞資源：瀏覽器必須先完成 CSSOM 建構才能進行渲染，因為任何 CSS 規則都可能影響元素的最終樣式。
  • JavaScript 是解析阻塞資源：預設情況下，<script> 標籤會阻塞 DOM 解析，因為 JavaScript 可能透過 document.write() 修改 DOM 結構。
  • 字型載入：Web Fonts 可能導致文字渲染延遲，產生 FOIT（Flash of Invisible Text）或 FOUT（Flash of Unstyled Text）。

關鍵資源數量

CRP 的效能取決於三個維度：

  1. 關鍵資源數量：阻塞首次渲染的資源總數
  2. 關鍵路徑長度：取得所有關鍵資源所需的往返次數（RTT）
  3. 關鍵位元組數：所有關鍵資源的總大小

async vs defer 比較

  • async：腳本下載與 HTML 解析並行，下載完成立即執行（可能中斷解析）。適合獨立的第三方腳本如分析工具。
  • defer：腳本下載與 HTML 解析並行，但延遲到 DOM 解析完成後依序執行。適合依賴 DOM 的應用邏輯。

Preload vs Prefetch

  • preload：告知瀏覽器當前頁面即將需要的資源，應以高優先序載入。
  • prefetch：暗示瀏覽器未來導航可能需要的資源，以低優先序在閒置時載入。

應用場景

場景一：電商首頁效能優化

電商網站首頁通常包含大量圖片、樣式和互動腳本。透過 CRP 優化：

<!-- 內聯關鍵 CSS -->
<style>
  .hero { /* 首屏樣式 */ }
  .nav  { /* 導航樣式 */ }
</style>

<!-- 延遲載入非關鍵 CSS -->
<link rel="preload" href="/css/below-fold.css" as="style" onload="this.onload=null;this.rel='stylesheet'">

<!-- 預載入關鍵字型 -->
<link rel="preload" href="/fonts/main.woff2" as="font" type="font/woff2" crossorigin>

<!-- 延遲執行非關鍵 JS -->
<script src="/js/analytics.js" async></script>
<script src="/js/app.js" defer></script>

場景二：SPA 應用首次載入

單頁應用（SPA）的 JavaScript bundle 通常較大，可透過以下方式優化 CRP：

  1. 程式碼分割（Code Splitting）：僅載入首屏所需的程式碼
  2. SSR/SSG：伺服器端渲染或靜態生成首屏 HTML
  3. 骨架屏（Skeleton Screen）：在內容載入前提供視覺回饋
  4. Service Worker 快取：離線優先策略減少網路依賴

場景三：媒體密集型頁面

新聞或部落格網站包含大量圖片與嵌入媒體：

  • 使用 loading="lazy" 延遲載入非首屏圖片
  • 透過 <picture> 元素提供響應式圖片
  • 影片使用 poster 屬性避免自動載入
  • 使用 content-visibility: auto 延遲渲染離屏內容
