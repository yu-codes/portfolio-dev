瀏覽器儲存

客戶端資料儲存

瀏覽器儲存是前端開發中不可或缺的一環，它讓 Web 應用程式能夠在客戶端持久化資料，減少對伺服器的依賴，同時提升使用者體驗。從最早期的 Cookies 到現代的 IndexedDB，瀏覽器提供了多種儲存機制，每種都有其適用場景與限制。理解這些儲存方案的差異，是建構高效能、離線友善應用的基礎。

核心概念

瀏覽器儲存的核心在於「同源政策」（Same-Origin Policy）。每個儲存機制都受到同源限制，意即只有相同協定、網域與埠號的頁面才能存取相同的儲存空間。這是瀏覽器安全模型的基石。

儲存機制可依據幾個維度來分類：

  • 生命週期：資料是永久保存（persistent）還是隨會話結束而清除（session-scoped）
  • 容量上限：從 4KB（Cookies）到數百 MB（IndexedDB）不等
  • 存取方式：同步（localStorage）或非同步（IndexedDB）
  • 是否隨 HTTP 請求自動傳送：僅 Cookies 具備此特性

此外，現代瀏覽器引入了 Storage API 來管理配額。當儲存空間不足時，瀏覽器會依據 LRU（Least Recently Used）策略清除資料，但標記為 persistent 的資料例外。

架構總覽

瀏覽器儲存可分為以下層級：

瀏覽器儲存
├── Cookie Storage（隨請求發送）
│   ├── Session Cookies
│   └── Persistent Cookies
├── Web Storage API
│   ├── localStorage（永久、同步）
│   └── sessionStorage（會話、同步）
├── IndexedDB（永久、非同步、結構化）
├── Cache API（Service Worker 快取）
└── File System Access API（實驗性）

localStorage 與 sessionStorage 共用相同的 Web Storage API，差別僅在於生命週期。localStorage 的資料在瀏覽器關閉後依然存在；sessionStorage 則在分頁關閉後清除。兩者皆提供簡單的鍵值對介面，容量通常為 5–10 MB。

Cookies 是最古老的儲存機制，設計上用於伺服器端狀態管理。每次 HTTP 請求都會自動帶上對應的 Cookies，因此適合身份驗證令牌等場景，但不適合大量資料儲存。

IndexedDB 是一個完整的客戶端資料庫，支援索引、事務、遊標等進階功能。它採用非同步 API，適合儲存大量結構化資料，如離線應用的資料快取。

效能考量：localStorage 的同步特性意味著大量讀寫會阻塞主執行緒。對於需要頻繁存取或大量資料的場景，IndexedDB 的非同步模型更為合適。

安全性考量：所有客戶端儲存都容易受到 XSS 攻擊。敏感資料（如 JWT）應搭配 HttpOnly Cookies 使用，因為 JavaScript 無法存取標記為 HttpOnly 的 Cookie。同時，Cookies 應設定 Secure 與 SameSite 屬性以防範 CSRF 攻擊。

應用場景

使用者偏好設定：主題切換、語言選擇等輕量設定適合存放於 localStorage，因為這些資料需要跨會話保存且不涉及敏感資訊。

購物車資料：電商網站的購物車可使用 localStorage 或 IndexedDB 保存，確保使用者重新開啟瀏覽器時購物車內容不會消失。

身份驗證令牌：Access Token 與 Refresh Token 最安全的做法是使用 HttpOnly + Secure + SameSite Cookies，避免 XSS 竊取。

離線資料同步：PWA（Progressive Web App）常使用 IndexedDB 搭配 Service Worker 實現離線功能。使用者在離線時的操作會先寫入 IndexedDB，待網路恢復後再同步至伺服器。

表單草稿自動儲存：使用 sessionStorage 暫存表單填寫進度，避免使用者意外關閉分頁後遺失資料。若需跨分頁保存，則改用 localStorage。

快取 API 回應：搭配 Cache API 或 IndexedDB 快取已取得的 API 資料，減少重複請求，提升頁面載入速度。
