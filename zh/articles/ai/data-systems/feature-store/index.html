特徵存儲與服務系統

特徵倉儲（Feature Store）是機器學習基礎設施的核心組件，它為 ML 團隊提供了一個集中化的特徵管理平台——統一特徵的定義、計算、儲存與服務。隨著 ML 系統從實驗階段邁向大規模生產部署，Feature Store 已成為 MLOps 技術棧中不可或缺的一環。本文將深入探討 Feature Store 的設計理念、架構模式與工程實踐。

核心概念

什麼是特徵（Feature）？

在機器學習語境中，特徵是模型輸入的原子單位。例如：

  • user_avg_purchase_amount_30d：使用者近 30 天平均消費金額
  • item_click_rate_7d：商品近 7 天點擊率
  • user_last_login_minutes_ago：使用者距上次登入的分鐘數

特徵的計算可能涉及複雜的聚合邏輯、多表關聯與時間視窗運算。

Feature Store 解決什麼問題？

特徵重複計算：不同團隊為相同概念的特徵各自實作，造成邏輯不一致與運算資源浪費。Feature Store 提供單一真相來源（Single Source of Truth）。

訓練-服務偏斜（Training-Serving Skew）：訓練時用 Spark/SQL 計算特徵，服務時用 Python 重新實作，兩者邏輯微妙不同導致模型效果下降。Feature Store 確保訓練與服務使用相同的特徵定義。

時間旅行需求（Point-in-Time Correctness）：訓練時需要還原歷史某一時刻的特徵快照，避免資料洩漏（Data Leakage）。Feature Store 支援 Point-in-Time Join。

特徵發現與複用：組織內特徵資產缺乏可見性。Feature Store 提供特徵目錄（Feature Catalog），支援搜尋、文件化與治理。

核心抽象

  • Feature（特徵）：單一特徵的定義，包含名稱、類型、描述與計算邏輯
  • Feature View（特徵視圖）：一組相關特徵的集合，定義了資料來源與轉換邏輯
  • Entity（實體）：特徵的主鍵，如 user_id、item_id
  • Feature Service：面向特定模型的特徵集合，定義了模型需要哪些特徵

架構總覽

Feature Store 的雙軌架構是其最核心的設計模式：

                    ┌─────────────────┐
                    │  Feature 定義    │
                    │  (Registry)     │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
    ┌─────────▼──────┐      │    ┌─────────▼──────┐
    │  離線儲存       │      │    │  線上儲存       │
    │  (Offline)     │      │    │  (Online)      │
    │  Parquet/Hive  │      │    │  Redis/DynamoDB│
    └─────────┬──────┘      │    └─────────┬──────┘
              │              │              │
    ┌─────────▼──────┐      │    ┌─────────▼──────┐
    │  批次計算       │      │    │  低延遲服務     │
    │  訓練資料產出   │      │    │  線上推論       │
    └────────────────┘      │    └────────────────┘
                             │
                    ┌────────▼────────┐
                    │  物化管線        │
                    │  (Materialization)│
                    └─────────────────┘

離線儲存（Offline Store）：存放歷史特徵資料，支援大規模批次讀取。通常使用 Data Lake 格式（Parquet on S3）或數據倉儲（BigQuery、Snowflake）。服務於模型訓練與批次推論。

線上儲存（Online Store）：存放最新特徵值，支援低延遲單筆查詢。通常使用 Redis、DynamoDB 或 Bigtable。服務於即時推論，延遲要求通常在個位數毫秒。

物化管線（Materialization Pipeline）：將離線計算的特徵同步到線上儲存，或將串流計算結果同時寫入離線與線上儲存。

特徵註冊中心（Registry）：特徵的元資料目錄，記錄每個特徵的定義、擁有者、上游依賴與下游消費者。

應用場景

即時推薦系統

電商推薦需要在使用者瀏覽商品的毫秒內計算特徵並返回推薦結果。Feature Store 的線上儲存確保特徵查詢延遲在可接受範圍內，離線儲存則支援推薦模型的持續訓練。

詐欺偵測

金融交易的詐欺偵測需要結合即時特徵（最近 5 分鐘交易次數）與歷史特徵（帳戶 90 天平均交易金額）。Feature Store 統一管理不同時間粒度的特徵，確保線上推論的正確性。

搜尋排序

搜尋引擎的排序模型依賴大量特徵：文件品質分數、使用者偏好、即時點擊率等。Feature Store 管理數千個特徵的生命週期，支援快速迭代與 A/B 測試。

多模型共享特徵

當組織內有數十個模型共用相同的使用者特徵時，Feature Store 避免了重複計算，確保所有模型看到一致的特徵值。
