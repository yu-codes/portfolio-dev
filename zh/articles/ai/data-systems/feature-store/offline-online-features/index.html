離線與線上特徵

深入解析 Feature Store 的離線與線上特徵系統，涵蓋架構設計、一致性保證與效能最佳化策略

Feature Store 是現代 MLOps 架構中的核心元件，負責管理機器學習模型所需的特徵。其中，離線特徵與線上特徵的分離與統一是架構設計的最大挑戰——既要保證訓練與推論使用相同的特徵邏輯，又要滿足截然不同的效能需求。

核心概念

離線特徵（Offline Features）

離線特徵用於模型訓練與批次推論，強調資料的完整性與歷史性。通常存儲在資料湖或資料倉儲中，以批次方式計算。

線上特徵（Online Features）

線上特徵用於即時推論，強調低延遲與高可用性。通常存儲在高速鍵值存儲中，以毫秒級回應特徵查詢。

特徵分類

特徵類型說明更新頻率存儲位置
靜態特徵很少變化的實體屬性天/週離線 + 線上
批次特徵定期計算的聚合指標小時/天離線 + 線上
近即時特徵分鐘級更新的指標分鐘線上
即時特徵請求時計算的特徵即時僅線上計算
歷史特徵帶時間戳的時序特徵N/A僅離線

訓練推論偏差（Training-Serving Skew）

當訓練使用的特徵與推論時使用的特徵存在差異時，就會產生訓練推論偏差。這是 ML 系統中最隱蔽且最有害的 Bug 之一。Feature Store 的核心價值就是消除這種偏差。

架構總覽

離線特徵系統

計算引擎：使用 Spark、Flink 或 SQL 引擎進行批次特徵計算。

儲存層：Parquet 檔案、Delta Lake 或 Hive 表，支援高效的大規模掃描。

時間旅行（Point-in-Time）：確保訓練資料不會使用未來資訊（避免資料洩漏）。

回填能力：新定義的特徵可以回算歷史值。

線上特徵系統

特徵存儲：Redis、DynamoDB 或 Bigtable 等低延遲鍵值存儲。

特徵服務 API：接收實體 ID，返回最新的特徵向量。

快取策略：多層快取降低延遲並減少後端壓力。

即時計算：使用串流處理引擎計算近即時特徵。

同步機制

離線計算的特徵需要同步到線上存儲：

同步方式延遲一致性複雜度
批次同步小時級最終一致低
微批次同步分鐘級最終一致中
串流同步秒級最終一致高
雙寫毫秒級強一致極高

離線 vs 線上特徵

面向離線特徵線上特徵
主要用途模型訓練、批次推論即時推論
延遲要求秒至分鐘可接受< 10ms
吞吐量高（大量資料掃描）高（大量併發請求）
資料量TB 至 PB 級GB 至 TB 級（最新值）
存儲類型列式存儲（Parquet）鍵值存儲（Redis）
計算方式批次（Spark/SQL）預計算 + 即時查詢
成本存儲便宜、計算按需存儲昂貴、持續運行
歷史性完整歷史僅最新值

策略說明正確性效能

應用場景

場景一：即時推薦系統

推薦系統需要結合用戶歷史行為特徵（離線計算的偏好向量）與即時行為特徵（最近 5 分鐘的點擊序列）。離線特徵每日更新同步到線上存儲，即時特徵由串流管線計算並直接寫入。

場景二：信用風控評分

風控模型需要申請人的靜態資料（年齡、職業）、批次計算的信用特徵（近 6 個月還款記錄統計）以及即時查詢的外部資料。Feature Store 統一管理這三類特徵的存取，確保訓練與推論的一致性。

場景三：動態定價

電商動態定價需要商品的歷史銷售特徵、競品價格特徵（每小時爬取更新）以及當前庫存特徵。不同時效性的特徵由不同管線計算，最終統一通過 Feature Store 服務給定價模型。

設計考量

考量決策點建議
一致性保證離線線上是否完全一致使用同一份特徵定義
延遲預算可接受的 P99 延遲根據業務 SLA 決定
成本效益哪些特徵值得上線上僅即時推論需要的特徵
回填策略新特徵如何生成歷史值設計回填管線
特徵共享跨團隊特徵重用建立特徵目錄與權限
