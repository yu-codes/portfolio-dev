資料版本控制與追蹤

在機器學習與資料工程領域，資料的可重現性（Reproducibility）是確保實驗結果可信賴的基礎。資料版本控制與追蹤系統使團隊能夠精確回溯任何時間點的資料狀態，理解資料的來龍去脈，並在問題發生時快速定位根因。

核心概念

資料版本控制（Data Versioning）

資料版本控制記錄資料隨時間變化的歷史。不同於程式碼版本控制，資料版本控制需要處理大型檔案、二進位格式與持續增長的資料集，這使得傳統的 Git 模式不適用。

資料血緣（Data Lineage）

資料血緣追蹤資料從源頭到最終消費點的完整路徑，包括所經過的每一次轉換、誰執行了操作、以及何時發生。

資料目錄（Data Catalog）

資料目錄是組織內所有資料資產的集中式索引，提供搜尋、發現與治理功能。

概念核心問題產出
版本控制資料在何時變成什麼樣子？版本歷史、差異比對
血緣追蹤資料從哪裡來、到哪裡去？血緣圖、影響分析
資料目錄組織裡有哪些資料？可搜尋的資產清單
元資料管理資料的屬性與含義是什麼？Schema、統計、語意

架構總覽

資料版本控制架構

儲存層：實際存放資料版本的位置。可以是物件儲存（S3、GCS）、分散式檔案系統（HDFS）或資料湖（Delta Lake、Iceberg）。

索引層：維護版本的元資料，包括版本號、時間戳、雜湊值、變更摘要。這一層通常輕量化，可以用 Git 管理。

存取層：提供版本切換、差異比對、時間旅行查詢等功能的介面。

血緣追蹤架構

收集器（Collector）：從各種執行環境中捕捉血緣事件。可以是 SQL 解析器、Spark 監聽器或 API 攔截器。

血緣儲存（Lineage Store）：以圖形結構存儲資料實體與轉換之間的關係。常用圖資料庫或專用後端。

視覺化介面：呈現血緣關係圖，支援影響分析與根因追蹤。

應用場景

場景一：ML 實驗可重現性

資料科學家在進行模型實驗時，需要精確記錄使用了哪個版本的訓練資料。透過 DVC 配合 Git，可以用一個 commit 同時記錄程式碼版本與資料版本，任何時候都能還原完整的實驗環境。

場景二：法規合規追蹤

金融或醫療產業需要證明報表數據的來源可追溯。資料血緣系統自動記錄每一筆資料的轉換路徑，當監管單位詢問某個數字如何計算時，可以完整呈現從原始資料到最終報表的每一步。

場景三：資料問題影響分析

當某個上游資料表被發現有錯誤時，透過血緣追蹤系統可以快速找出所有受影響的下游表格、報表與模型。這使得修復範圍評估從數小時縮短為數分鐘。

場景四：資料回溯與修正

業務邏輯變更需要重新計算歷史資料。使用 Delta Lake 的 Time Travel 功能，可以查詢任意歷史時間點的資料狀態，配合 Backfill 管線重新計算受影響的衍生資料。
