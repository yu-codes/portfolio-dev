資料異常偵測是資料品質保障的重要手段，透過自動化的方法識別偏離正常模式的資料點或資料區段。在資料管線、系統監控與業務分析中，及時發現異常可以防止錯誤擴散、辨識潛在問題並保護下游系統。

核心概念

異常的定義與分類

異常（Anomaly）是指不符合預期模式的資料觀測值。根據性質不同，異常可分為：

異常類型說明範例
點異常（Point Anomaly）單一資料點明顯偏離某筆交易金額異常高
上下文異常（Contextual Anomaly）在特定上下文中異常夏天的暖氣銷售峰值
群體異常（Collective Anomaly）一組資料點共同構成異常連續多天的系統異常行為
趨勢異常（Trend Anomaly）整體趨勢發生偏移資料量逐日遞減
缺失異常（Missing Anomaly）預期有的資料未出現某感測器停止回報

異常偵測的挑戰

  • 正常模式會演化：業務變化導致「正常」的定義隨時間改變
  • 標註稀缺：異常本質上稀少，難以收集足夠的標註資料
  • 高維度困難：維度越高，資料越稀疏，異常越難定義
  • 即時性要求：許多場景要求在秒級內判定異常
  • 誤報成本：過多的假警報會降低團隊的信任度

架構總覽

異常偵測系統架構

資料收集層：持續收集需要監控的指標與特徵。

特徵工程層：計算滑動窗口統計量、季節性分解、趨勢指標等衍生特徵。

偵測引擎層：執行異常偵測演算法，支援多模型集成。

評分與排序層：為偵測到的異常計算嚴重度分數並排列優先順序。

告警與視覺化層：向相關人員發送告警，並提供互動式儀表板。

偵測流程

  1. 資料進入 → 2. 特徵計算 → 3. 模型推論 → 4. 異常評分 → 5. 門檻判斷 → 6. 告警發送

應用場景

場景一：資料管線健康監控

監控 ETL 管線的關鍵指標：每日資料量、空值比例、欄位值域分佈。使用基於歷史趨勢的自適應門檻，當指標偏離歷史同期模式時觸發告警。結合星期效應與節假日修正避免誤報。

場景二：即時交易詐欺偵測

金融交易系統需要在毫秒內判斷交易是否異常。使用特徵向量（交易金額、時間、地點、裝置等）配合 Isolation Forest 或 Autoencoder 進行即時評分，高風險交易進入人工審核流程。

場景三：ML 模型輸入監控

監控生產環境中模型的輸入特徵分佈是否漂移（Data Drift）。使用 KS 檢定或 PSI（Population Stability Index）比較當前分佈與訓練分佈，當漂移超過門檻時觸發模型重新訓練。

場景四：基礎設施效能異常

監控伺服器的 CPU、記憶體、磁碟與網路指標。使用多變量分析同時考慮指標間的相關性，避免個別指標的正常波動觸發假警報。
