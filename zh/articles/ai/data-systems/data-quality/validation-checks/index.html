資料品質驗證

資料品質驗證是資料管線中的防護網，確保流經系統的資料符合預期的規格與業務規則。在「垃圾進、垃圾出」的鐵律下，一套完善的驗證機制可以在問題擴散前將其攔截，避免下游分析與模型產出錯誤結果。

核心概念

資料品質驗證的層次

資料品質驗證可以在不同粒度進行：

驗證層次檢查對象範例
Schema 驗證結構與型別欄位是否存在、型別是否正確
值域驗證個別值的合理性年齡是否在 0-150 之間
統計驗證整體分佈特徵均值是否偏移超過門檻
關聯驗證欄位間的邏輯關係開始日期是否早於結束日期
業務規則驗證領域特定規則訂單金額是否與商品價格一致
跨表驗證跨資料集一致性外鍵參照完整性

驗證時機

  • 入口驗證（Ingestion Validation）：資料進入系統時的第一道防線
  • 轉換中驗證（In-pipeline Validation）：每個轉換步驟後的中間檢查
  • 出口驗證（Output Validation）：資料交付給消費者前的最終檢查
  • 持續監控（Continuous Monitoring）：已存儲資料的定期健康檢查

品質指標定義

指標計算方式告警門檻範例
完整率非空值數 / 總記錄數< 95% 告警
唯一率唯一值數 / 總記錄數< 99% 告警（主鍵）
有效率通過規則數 / 總記錄數< 98% 告警
新鮮度當前時間 - 最新資料時間> 2 小時告警
資料量本次記錄數 / 歷史平均偏差 > 30% 告警

架構總覽

驗證系統架構

規則引擎（Rule Engine）：存儲與管理驗證規則。支援規則的版本控制、啟停與參數化配置。

執行引擎（Execution Engine）：在資料上執行驗證規則。需要與資料處理框架整合（如 Spark、BigQuery）。

結果存儲（Result Store）：記錄每次驗證的結果，包括通過率、失敗記錄樣本與趨勢資料。

告警系統（Alerting System）：當驗證失敗時，根據嚴重程度觸發不同等級的告警。

修復機制（Remediation）：定義驗證失敗後的處理流程——阻止管線繼續、隔離問題資料或人工介入。

驗證策略比較

策略原理優點缺點
規則式驗證人工定義明確規則可解釋、可控制維護成本高
統計式驗證基於歷史分佈判斷自動適應變化可能誤報
ML 式驗證模型學習正常模式能捕捉複雜異常黑箱難解釋
Schema 合約預定義資料格式規範預防式、強制性靈活度低

失敗處理模式

模式說明適用場景
阻斷（Block）驗證失敗則停止管線關鍵資料、法規要求
隔離（Quarantine）將問題資料移至隔離區部分資料可用的場景
標記（Flag）標記問題但繼續處理非關鍵路徑
修復（Fix）自動套用修正規則已知模式的常見問題
降級（Degrade）使用替代資料源高可用性要求

應用場景

場景一：資料合約（Data Contracts）

資料生產者與消費者之間建立正式合約，明確定義資料的格式、品質標準與 SLA。當上游資料不符合合約時，自動阻止其流入下游系統，並通知相關團隊。

場景二：ML 訓練資料品質關卡

在模型訓練管線中設置品質關卡：訓練資料必須通過完整性、分佈穩定性與標籤一致性檢查後，才允許啟動模型訓練。防止因資料品質下降導致模型效能退化。

場景三：即時串流資料驗證

在串流管線中嵌入輕量級驗證邏輯，即時檢查每筆資料的格式與值域。異常資料被路由到死信佇列（Dead Letter Queue），正常資料繼續處理，確保低延遲的同時維持品質。

場景四：週期性資料健康報告

定期對資料倉儲中的核心表進行全面健康掃描，生成品質報告。追蹤品質趨勢，識別逐漸惡化的指標，在問題爆發前預警。

實施建議

階段重點
起步期對關鍵表建立基本 Schema 與非空檢查
成長期加入統計驗證與跨表一致性檢查
成熟期實施資料合約、自動化修復與全面監控
優化期ML 驅動的異常偵測、預測性品質管理
