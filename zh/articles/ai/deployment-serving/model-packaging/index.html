模型封裝與容器化是將訓練好的機器學習模型從實驗環境安全遷移至生產環境的關鍵步驟。透過標準化的封裝格式與容器化技術，消除環境差異帶來的部署問題，實現「一次封裝，處處運行」。

核心概念

模型序列化（Model Serialization）

將記憶體中的模型物件轉換為可持久化儲存的格式：

  • 框架原生格式：PyTorch (.pt/.pth)、TensorFlow SavedModel、Keras (.h5)
  • 跨框架格式：ONNX（Open Neural Network Exchange）
  • 優化格式：TensorRT Engine、OpenVINO IR、Core ML

模型封裝（Model Packaging）

在序列化模型的基礎上，將所有執行依賴打包為自包含的部署單元：

  • 模型權重與架構
  • 推論程式碼（inference code）
  • 前處理/後處理邏輯
  • 依賴套件清單
  • 環境配置

容器化（Containerization）

使用 Docker 等容器技術，將模型服務及其完整執行環境封裝為可移植的容器映像：

  • 作業系統層（Base Image）
  • 系統依賴（CUDA、cuDNN）
  • Python 環境與套件
  • 模型檔案與服務程式碼
  • 服務框架（Flask、FastAPI、gRPC）

模型簽章（Model Signature）

定義模型的輸入輸出規格：

  • 輸入張量的形狀、資料型別
  • 輸出張量的形狀、資料型別
  • 可選的輸入約束（數值範圍、允許值）

ONNX 轉換管線

ONNX 作為跨框架的模型交換標準：

  1. 框架模型匯出為 ONNX 計算圖
  2. ONNX 圖優化（運算子融合、常數摺疊）
  3. ONNX Runtime 推論驗證
  4. 可選：進一步轉換為平台特定格式

Docker 多階段建置

# Stage 1: 建置階段
FROM python:3.10 AS builder
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Stage 2: 運行階段
FROM python:3.10-slim
COPY --from=builder /usr/local/lib/python3.10/site-packages /usr/local/lib/python3.10/site-packages
COPY model/ /app/model/
COPY src/ /app/src/
EXPOSE 8080
CMD ["python", "/app/src/serve.py"]

安全強化

容器化模型的安全考量：

  • 使用非 root 使用者執行
  • 最小化基礎映像（distroless / alpine）
  • 定期掃描漏洞（Trivy、Snyk）
  • 簽署映像（cosign）

應用場景

跨框架模型部署

團隊中使用 PyTorch 訓練的模型需要部署到僅支援 TensorFlow Serving 的基礎設施。透過 ONNX 作為中間格式，實現無縫的跨框架遷移。

GPU 推論服務容器化

將需要 GPU 加速的深度學習模型封裝為包含 CUDA 驅動的容器，透過 Kubernetes 的 GPU 資源排程實現彈性部署。

邊緣裝置部署封裝

將模型轉換為 TFLite 或 Core ML 格式，搭配輕量化容器或原生 SDK 部署到行動裝置或 IoT 設備。

模型市集與共享

將模型封裝為標準化格式（如 MLflow Model 或 BentoML Bento），上傳到組織內部的模型市集，供其他團隊直接取用部署。
