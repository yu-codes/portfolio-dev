推論擴展策略（Scaling Inference）是處理生產環境中流量波動與成長需求的關鍵能力。從單一 GPU 到全球分散式叢集，推論系統需要在成本、延遲與吞吐量之間動態調整。

核心概念

水平擴展 vs 垂直擴展

  • 垂直擴展（Scale Up）：升級單一節點的硬體（更強的 GPU、更多記憶體）
  • 水平擴展（Scale Out）：增加服務節點數量，透過負載均衡分散流量

在推論場景中，水平擴展是主流策略，因為：

  • 硬體升級有物理上限
  • 水平擴展提供更好的容錯能力
  • 可根據需求精細調整資源

自動擴縮（Autoscaling）

根據即時負載動態調整推論服務的副本數量：

  • 反應式擴縮：基於當前指標（CPU/GPU 使用率、請求佇列長度）
  • 預測式擴縮：基於歷史模式預測未來負載
  • 排程式擴縮：根據已知的流量高峰時段預設資源

關鍵指標：

  • 請求延遲（P50、P95、P99）
  • GPU/CPU 使用率
  • 佇列深度
  • 每秒請求數（RPS）

批次處理策略

將多個請求合併處理以提升 GPU 利用率：

  • 靜態批次：固定 batch size，等待湊滿
  • 動態批次：在延遲預算內盡量合併
  • 連續批次（Continuous Batching）：LLM 場景中在 token 級別動態管理批次
  • 優先級批次：高優先級請求不等待批次

多區域部署

將推論服務部署到全球多個區域：

  • 降低使用者到服務的網路延遲
  • 提供地理級別的容災能力
  • 符合資料在地性法規要求
  • 全球流量的智慧路由

Kubernetes 推論擴展

在 K8s 環境中擴展推論服務：

  • HPA（Horizontal Pod Autoscaler）：基於 CPU/GPU 指標自動擴縮 Pod 數量
  • VPA（Vertical Pod Autoscaler）：自動調整 Pod 的資源請求
  • KEDA：基於事件驅動的擴縮（如訊息佇列長度）
  • Knative：Scale-to-zero 支援，降低閒置成本

GPU 資源管理

高效利用昂貴的 GPU 資源：

  • GPU 共享：多個模型共用一張 GPU（MIG、MPS、Time-slicing）
  • GPU 池化：虛擬化 GPU 資源，按需分配
  • 搶佔式調度：低優先級任務可被高優先級搶佔
  • Spot/Preemptible 實例：使用低成本的搶佔式雲端實例

快取策略

減少冗餘計算，提升有效吞吐量：

  • 請求去重：相同請求只計算一次
  • 結果快取：常見輸入直接回傳快取結果
  • Embedding 快取：快取高頻實體的向量表示
  • Prefix 快取：LLM 場景中快取共同前綴的 KV-Cache

關鍵比較

擴展策略回應速度成本效率適用場景複雜度
反應式 HPA中（分鐘級）中一般波動低

應用場景

電商大促流量突增

雙十一或黑色星期五期間，推薦系統流量可達平時的 10-50 倍。系統需要：預測式提前擴容、動態批次大小調整、降級策略（如返回熱門推薦替代個性化推薦）。

LLM 推論服務擴展

大語言模型推論面臨獨特的擴展挑戰：單次推論佔用大量 GPU 記憶體、生成時間長且不確定。需要連續批次處理（Continuous Batching）、張量平行（Tensor Parallelism）跨多 GPU，以及 KV-Cache 管理。

全球化推論服務

為全球使用者提供一致的低延遲體驗。透過 CDN 層的邊緣快取、全球多區域部署、以及基於地理位置的智慧路由，將推論延遲控制在可接受範圍。

批次離線推論

定期為數百萬使用者產生推薦候選、內容索引或風險評估。使用 Spark/Ray 進行大規模平行推論，搭配 Spot 實例降低成本，容忍部分實例中斷的韌性。
