ML 的 CI/CD

ML 的 CI/CD（持續整合與持續交付）將軟體工程的最佳實踐延伸至機器學習領域，確保模型從開發到部署的每個環節都經過自動化驗證，大幅降低人為疏失並加速模型交付。

核心概念

CI/CD 在 ML 的特殊性

傳統軟體的 CI/CD 主要處理程式碼變更，而 ML 的 CI/CD 需要額外應對三重變更維度：

  • 程式碼變更：模型架構、特徵工程、訓練腳本的修改
  • 資料變更：訓練資料的新增、修正或分佈改變
  • 模型變更：超參數調整、模型版本迭代

持續整合（CI for ML）

ML 的持續整合涵蓋：

  • 程式碼品質檢查（linting、type checking）
  • 單元測試（資料轉換邏輯、特徵計算）
  • 資料驗證測試（schema check、分佈檢查）
  • 模型訓練冒煙測試（小規模資料快速驗證）
  • 模型品質門檻測試（accuracy、latency threshold）

持續交付（CD for ML）

ML 的持續交付確保通過驗證的模型能安全部署：

  • 模型打包與容器化
  • 預發布環境（Staging）驗證
  • 漸進式部署（Canary/Blue-Green）
  • 自動回滾機制

持續訓練（CT）

ML 特有的概念，指在新資料到來時自動觸發模型重新訓練：

  • 資料管線監控
  • 自動重訓觸發
  • 模型品質把關
  • 模型註冊與版本管理

測試金字塔

ML 專案的測試金字塔從底層到頂層：

  1. 單元測試：特徵函數、資料轉換、工具函數
  2. 整合測試：管線端到端、資料流正確性
  3. 模型測試：準確率、公平性、魯棒性
  4. 系統測試：服務延遲、吞吐量、資源使用

品質門檻（Quality Gate）

自動化決策是否放行模型進入下一階段：

  • 模型指標閾值（如 F1 > 0.85）
  • 效能指標（如 P99 延遲 < 50ms）
  • 公平性指標（如各群體準確率差異 < 5%）
  • 安全性檢查（如對抗樣本魯棒性）

應用場景

特徵工程的自動化驗證

當資料工程師修改特徵計算邏輯時，CI 管線自動執行特徵品質測試，驗證新特徵的分佈、缺失率與下游模型相容性，避免有問題的特徵進入生產環境。

模型效能回歸測試

每次模型程式碼變更後，自動在標準測試集上評估模型效能，與基準版本比較，若效能下降超過容許範圍則阻擋合併請求。

多環境一致性驗證

確保模型在開發、測試與生產環境中行為一致。透過容器化與環境快照技術，消除「在我機器上可以跑」的問題。

合規性自動化檢查

在金融或醫療等受監管領域，CI/CD 管線能自動執行公平性審計、可解釋性報告生成與模型文件產出，滿足法規要求。
