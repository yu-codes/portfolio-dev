稽核日誌（Audit Logging）是雲端合規治理的基石，它記錄了「誰在什麼時候對什麼資源做了什麼操作」。完善的稽核日誌系統不僅滿足法規要求，更是安全事件調查、異常偵測與存取控制驗證的關鍵數據來源。在多雲環境中，建立統一的稽核日誌策略更顯得格外重要。

核心概念

稽核日誌的定義與目的

稽核日誌與一般的應用程式日誌不同，它專注於記錄具有合規與安全意義的事件。核心特性包括：

  • 不可竄改性（Immutability）：日誌一旦寫入就不能被修改或刪除，確保證據的完整性。
  • 完整性（Completeness）：所有受管控的操作都必須被記錄，不能有遺漏。
  • 時效性（Timeliness）：事件發生後應立即記錄，時間戳必須準確。
  • 可搜尋性（Searchability）：能夠快速查詢特定時間範圍、使用者或資源的操作記錄。

稽核日誌的五個 W

每筆稽核日誌記錄都應回答五個問題：

  1. Who：誰執行了操作？（使用者身份、IP 位址、來源）
  2. What：執行了什麼操作？（API 呼叫、操作類型）
  3. When：何時執行的？（精確時間戳、時區）
  4. Where：在哪裡執行？（資源位置、帳戶、區域）
  5. Whether：操作是否成功？（結果狀態、錯誤碼）

事件類型分類

  • 管理事件（Management Events）：對雲端資源的建立、修改、刪除操作。如建立 EC2 實例、修改 IAM 政策。
  • 資料事件（Data Events）：對資料的讀寫操作。如 S3 物件存取、DynamoDB 讀寫。
  • 網路事件（Network Events）：網路層的活動記錄。如 VPC Flow Logs、DNS 查詢。
  • 登入事件（Sign-in Events）：使用者認證相關的事件。如登入成功/失敗、MFA 驗證。

核心組件

1. 日誌收集層

各雲端供應商的原生稽核服務：

  • AWS CloudTrail：記錄所有 AWS API 呼叫，支援管理事件與資料事件。可設定組織級（Organization Trail）收集所有帳戶的日誌。
  • Azure Monitor Activity Log：記錄 Azure 訂閱層級的操作，整合 Azure AD 登入日誌。
  • GCP Cloud Audit Logs：自動記錄管理活動，可選擇性啟用資料存取日誌。

2. 日誌傳輸與儲存

  • 即時串流：透過 CloudWatch Logs、Event Hub 或 Pub/Sub 即時傳輸日誌。
  • 批次匯出：定期將日誌匯出到集中式儲存（S3、GCS）。
  • 長期保存：使用低成本儲存（S3 Glacier、Archive Storage）保存法規要求的保留期限。

3. 日誌分析與監控

  • SIEM 整合：將日誌匯入 Splunk、Elastic SIEM 或 Microsoft Sentinel 進行關聯分析。
  • 即時告警：設定規則偵測可疑操作（如 Root 帳戶登入、安全群組修改）。
  • 合規儀表板：視覺化呈現合規狀態與異常趨勢。

日誌保護機制

  • 寫入鎖定（WORM）：使用 S3 Object Lock 或 Azure Immutable Blob Storage 防止日誌被刪除。
  • 加密保護：日誌在傳輸中與靜態時都使用加密保護。
  • 存取控制：嚴格限制誰可以讀取與管理稽核日誌，遵循最小權限原則。
  • 完整性驗證：使用雜湊鏈或數位簽章驗證日誌未被竄改（如 CloudTrail Log File Integrity Validation）。

應用場景

場景一：安全事件調查

資安團隊收到告警，發現一個 IAM 使用者在非工作時間從異常 IP 位址建立了多個 EC2 實例。透過 CloudTrail 日誌，團隊可以追蹤該使用者的完整操作軌跡：何時取得憑證、從哪裡登入、執行了哪些 API 呼叫、影響了哪些資源。這些日誌是判斷是否為安全事件的關鍵證據。

場景二：合規稽核準備

企業即將接受 SOC 2 Type II 稽核。稽核人員需要證明：所有特權操作都有記錄、存取控制變更有審批流程、異常登入會觸發告警。透過集中式稽核日誌系統，企業可以快速產出所需的報告，證明控制措施持續有效運作。

場景三：變更追蹤與回溯

生產環境出現異常效能下降。透過稽核日誌，運維團隊可以查看過去數小時內的所有基礎設施變更：是否有人修改了安全群組規則、調整了 Auto Scaling 配置、或更新了應用部署。快速定位問題根因並還原變更。

場景四：跨雲統一合規視圖

跨國企業在 AWS、Azure 與 GCP 上運行不同的工作負載。使用 SIEM 平台整合三朵雲的稽核日誌，建立統一的合規儀表板。當任何雲端發生高風險操作時，統一的告警系統會立即通知資安團隊，無論事件發生在哪個平台。

AWS 服務實作

AWS CloudTrail 概述

AWS CloudTrail 提供整個帳號內所有操作的稽核紀錄與事件歷史，自動預設啟用全區域記錄。記錄來源涵蓋：使用者控制台操作、AWS CLI 指令、SDK 呼叫，以及其他 AWS 服務之間的 API 呼叫。預設提供 90 天的事件歷史（免費），如需長期保留則需建立 Trail 並輸出至 S3。

事件類型

類型說明預設
Management EventsAPI 呼叫、登入等管理控制平面事件啟用（免費）
Data EventsS3 物件、Lambda 呼叫等資料平面事件停用（需額外付費）
Insights Events異常 API 活動偵測停用（需額外付費）

Management Events（管理事件）

控制平面操作，預設收錄且免費：

  • 建立、修改、刪除 EC2 實例
  • 建立 VPC 或修改安全群組
  • 登入 AWS Management Console
  • 建立或修改 IAM 政策

Data Events（資料事件）

資料平面操作，需額外付費啟用：

  • S3 物件層級操作（GetObject、PutObject、DeleteObject）
  • Lambda 函數呼叫（Invoke）
  • DynamoDB 表操作（GetItem、PutItem）

輸出目的地

目的地用途
S3長期儲存、合規保留、成本最低
CloudWatch Logs即時監控、設定 Metric Filter 告警

Trail 類型與配置

類型範圍
單一區域僅記錄該區域的事件
所有區域記錄所有區域事件（推薦做法）
組織 Trail整個 AWS Organizations 內所有成員帳號的事件

全域服務事件（Global Service Events）

IAM、STS、CloudFront 等全域服務的事件：

  • 只記錄在 us-east-1 區域
  • 需在 Trail 配置中明確啟用 Include global service events

日誌檔案完整性驗證

Log File Validation

  • 使用 SHA-256 雜湊演算法對每個日誌檔案生成雜湊值
  • CloudTrail 每小時自動產生一個摘要檔案（Digest File），包含前一小時所有日誌的雜湊
  • 可使用 AWS CLI 或 SDK 驗證日誌是否自交付後被竄改、修改或刪除

S3 保護措施

措施說明
Bucket Policy限制對 CloudTrail S3 Bucket 的存取
SSE-S3 / SSE-KMS靜態加密日誌檔案
版本控制（Versioning）保留歷史版本，防止意外覆蓋
MFA Delete刪除物件時要求 MFA 驗證
S3 Object Lock設定 WORM（Write Once Read Many）保護，使日誌不可刪除

CloudTrail Insights

功能說明
基準線建立分析歷史 API 活動，建立正常行為模式
異常偵測自動偵測偏離基準線的異常 API 呼叫頻率
Insights 事件產生獨立的 Insights 事件並記錄至 Trail 輸出目的地

常見偵測場景：IAM 政策大量建立、資源異常終止、API 呼叫量突然暴增。

CloudTrail Lake

  • 不可變的集中式事件儲存（Immutable Event Store）
  • 使用 SQL 語法直接查詢事件資料，無需將日誌匯出至 S3 再分析
  • 最長保留 7 年
  • 支援跨組織（Organizations）查詢，統一分析多帳號事件

與 CloudWatch 整合：即時告警

透過以下流程實現 CloudTrail 事件的即時監控：

CloudTrail → CloudWatch Logs → Metric Filter → CloudWatch Alarm → SNS 通知

範例：Root 帳號登入告警

  1. 在 CloudTrail Trail 配置中，將事件傳送至 CloudWatch Logs
  2. 建立 Metric Filter，過濾條件為：
    • { $.userIdentity.type = "Root" && $.eventType != "AwsServiceEvent" }
  3. 建立 CloudWatch Alarm，當該指標 ≥ 1 時觸發
  4. 設定 SNS 主題通知資安團隊

CloudTrail vs CloudWatch

項目CloudTrailCloudWatch
用途稽核 API 活動（誰做了什麼）監控效能指標與日誌
資料類型事件（API 呼叫記錄）指標（數值）、日誌（文字）
預設保留90 天（免費）依設定（需付費儲存）
主要適用合規、安全調查營運監控、效能優化

常見使用場景

1. 安全調查：追蹤誰刪除了資源

CloudTrail → 搜尋 TerminateInstances API 呼叫
           → 找到執行者（IAM User / Role）、操作時間、來源 IP
           → 結合 IAM 日誌確認是否為授權操作

2. 合規稽核：提供稽核證明

  • 所有 AWS API 操作都有完整記錄
  • 啟用 Log File Integrity Validation 確保日誌不可竄改
  • 使用 S3 Object Lock 實現長期不可變保留，符合 SOC 2、ISO 27001、PCI DSS 等法規要求

3. 即時安全告警

CloudTrail → CloudWatch Logs → Metric Filter → Alarm → SNS

常見告警規則：Root 帳號登入、IAM 政策大幅修改、安全群組開放 0.0.0.0/0、MFA 停用等高風險操作。

4. 問題除錯：追蹤資源異常變更

生產環境資源被誤刪或異常修改時，透過 CloudTrail 查詢 API 呼叫序列，找出操作者、時間點與完整操作內容，協助快速還原與預防再次發生。

最佳實踐

  1. 啟用所有區域 Trail：確保沒有遺漏任何區域的事件
  2. 輸出至 S3 長期保留：滿足合規要求的稽核記錄保留期限
  3. 整合 CloudWatch Logs：實現關鍵事件的即時監控與告警
  4. 啟用 Log File Integrity Validation：確保日誌完整性，提供法庭級證據
  5. S3 Bucket 多重保護：結合 Bucket Policy、版本控制、MFA Delete 與 Object Lock
  6. 考慮啟用 CloudTrail Insights：自動偵測異常 API 活動，無需手動建立規則
  7. 使用 CloudTrail Lake 進行進階分析：透過 SQL 查詢快速回應安全調查需求
