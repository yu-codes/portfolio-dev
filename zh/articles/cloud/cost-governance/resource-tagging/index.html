資源標籤（Resource Tagging）是雲端治理的基礎設施，透過為每個雲端資源附加結構化的元資料（metadata），實現成本歸屬、存取控制、自動化管理與合規性追蹤。沒有良好的標籤策略，組織將面臨「誰在用什麼、花了多少錢」的黑箱問題。

核心概念

標籤（Tag）的本質：標籤是一組鍵值對（Key-Value Pair），附加在雲端資源上，用於分類、搜尋和管理。例如 Environment: Production、Team: Platform、CostCenter: CC-1234。

標籤分類體系：

  • 技術標籤：Environment、Application、Version、Component
  • 業務標籤：CostCenter、Project、BusinessUnit、Owner
  • 安全標籤：DataClassification、Compliance、Encryption
  • 營運標籤：ManagedBy、BackupPolicy、MaintenanceWindow、AutoShutdown

標籤治理原則：

  • 一致性：統一命名慣例（如 CamelCase 或 kebab-case）
  • 必要性：定義必填標籤（Mandatory Tags）與選填標籤
  • 可驗證性：透過自動化工具驗證標籤的正確性
  • 不可變性：某些標籤在建立後不應被修改（如 CreatedBy）
  • 有限性：控制標籤值的範圍（使用允許清單而非自由文字）

標籤繼承與傳播：在階層式資源架構中（如 Azure Management Group → Subscription → Resource Group → Resource），上層標籤應自動傳播至下層，減少手動維護成本。

未標籤資源管理：建立流程處理未標籤資源，包括自動識別、通知擁有者、限期標籤、以及最終處置策略。

標籤策略定義了規範，預防控制（如 AWS SCP、Azure Policy）在資源建立時強制執行；偵測控制定期掃描識別違規；矯正控制則自動修復或通知相關人員。

選型建議：原生策略工具適合單一雲端的基本需求；IaC 工具（Terraform）適合在部署前預防；Cloud Custodian 適合需要跨雲統一治理與複雜修復邏輯的場景。

應用場景

場景一：強制標籤策略實施 透過 Azure Policy 設定 Deny 效果，當資源缺少 Environment、Owner、CostCenter 三個必填標籤時，拒絕資源建立。同時透過 Audit 效果追蹤建議標籤的覆蓋率，目標達到 95% 以上。

場景二：自動化成本歸屬 利用標籤將每月雲端帳單自動分攤至各部門。定義 CostCenter 標籤對應到財務系統的成本中心代碼，透過計費報表的標籤篩選功能，自動產生各部門的費用明細與趨勢報告。

場景三：環境生命週期管理 根據 Environment 標籤自動化管理資源生命週期。開發環境（Environment: Dev）在每日下班後自動關機，週末完全停止；臨時環境（Environment: Sandbox）設定 TTL（Time-to-Live），超過 7 天未使用自動刪除。

場景四：合規性與安全分類 透過 DataClassification: Confidential 標籤識別存放敏感資料的資源，自動套用加密政策、限制跨區域複製、並啟用進階的存取日誌記錄。安全團隊可基於此標籤快速盤點敏感資料的分佈。

場景五：未標籤資源治理流程 每週自動掃描所有資源，識別缺少必填標籤的項目。第一週發送提醒通知給最可能的擁有者（基於建立者資訊推斷）；第二週升級通知至團隊主管；第四週若仍未處理，自動加入隔離群組並限制網路存取。

成本分配標籤啟用步驟

  1. 在 AWS Billing Console 啟用所需標籤
  2. 等待約 24 小時讓標籤生效
  3. 至 Cost Explorer 依標籤篩選與分析成本

AWS Organizations 標籤政策（Tag Policy）

  • 強制執行標籤命名標準
  • 定義允許的標籤鍵與標籤值
  • 可套用至特定 OU 或帳號

Tag Policy 範例（JSON）：

{
  "tags": {
    "Environment": {
      "tag_key": {
        "@@assign": "Environment"
      },
      "tag_value": {
        "@@assign": ["dev", "staging", "prod"]
      }
    }
  }
}

類型說明

預算動作（Budget Actions）

  • 發送 SNS 通知或 Email 警示
  • 自動套用 IAM 政策以限制用量
  • 套用 Service Control Policy（SCP）強制管控

成本異常偵測（Cost Anomaly Detection）

  • 使用機器學習自動建立消費基準線
  • 即時偵測異常消費行為並發送警示
  • 監控範圍可設定為帳號、服務或特定標籤
  • 設定步驟：
    1. 選擇監控範圍（帳號 / 服務 / 標籤）
    2. 設定警示閾值（絕對金額或百分比）
    3. 配置通知目標（SNS / Email）

AWS Glue 與成本分析整合

  • 使用 AWS Glue Data Catalog 掃描 CUR（Cost and Usage Report）資料
  • 建立資料目錄後，透過 Amazon Athena 以 SQL 查詢費用明細
  • 可整合 Amazon QuickSight 建立視覺化成本報表儀表板

多帳號成本管理（AWS Organizations 合併帳單）

  • 所有成員帳號費用合併至單一付款帳號
  • 批量使用可獲得量折優惠
  • Reserved Instances 與 Savings Plans 可在組織內共用

成本分配方法

方法說明
標籤（Tags）按專案或部門標籤分攤成本
帳號（Account）以帳號為單位分配費用
組織單位（OU）依 Organizations OU 層級分配
