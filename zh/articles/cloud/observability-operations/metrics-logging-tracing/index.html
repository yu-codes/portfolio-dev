指標、日誌與追蹤

深入解析可觀測性三大支柱：指標（Metrics）、日誌（Logs）與追蹤（Traces）的概念、工具與整合策略

指標（Metrics）、日誌（Logs）與追蹤（Traces）是可觀測性的三大支柱，為運維團隊提供了理解系統行為的完整視角。三者各有其獨特的價值：指標告訴你「系統怎麼了」，日誌告訴你「發生了什麼」，追蹤告訴你「請求去了哪裡」。整合三者才能建構完整的可觀測性體系。

核心概念

三大支柱的特性

面向指標（Metrics）日誌（Logs）追蹤（Traces）
資料型態數值時間序列文字事件結構化 Span
回答問題多少？多快？多常？發生了什麼？請求如何流動？
基數低（預聚合）高（每事件）中（每請求）
存儲成本低高中
查詢速度快（索引好）中（需搜尋）中（按 Trace ID）
適合告警、趨勢除錯、審計效能分析、依賴
取樣不適用可選常需取樣

指標類型

類型說明範例用途
Counter只增不減的計數請求總數、錯誤數計算速率
Gauge可升可降的值CPU 使用率、記憶體當前狀態
Histogram數值的分佈延遲分位數效能分析
Summary客戶端計算的分位數P99 延遲快速概覽

結構化日誌

面向非結構化結構化（JSON）
格式自由文字Key-Value/JSON
搜尋全文搜尋欄位查詢
解析需正則表達式原生解析
自動化困難容易
範例“Error in user 123”{“level”:“error”,“user_id”:123}

關聯性（Correlation）

Trace ID: abc-123
├── Span 1: API Gateway (logs: traceId=abc-123)
├── Span 2: Auth Service (metrics: request_count{traceId=abc-123})
├── Span 3: Order Service (logs: traceId=abc-123)
└── Span 4: Database Query (metrics: query_duration{traceId=abc-123})

透過 Trace ID 將三大支柱的資料關聯，實現從告警到根因的快速定位。

應用場景

黃金信號（Golden Signals）

信號指標告警閾值
延遲（Latency）P50、P95、P99 回應時間P99 > SLO
流量（Traffic）QPS、RPS異常偏離基線
錯誤（Errors）錯誤率、5xx 比例> SLO 錯誤預算
飽和度（Saturation）CPU、記憶體、連線池使用率> 80%

RED 方法（微服務）

指標定義計算方式
Rate每秒請求數Counter 的速率
Errors失敗請求比例失敗數 / 總數
Duration請求延遲分佈Histogram 分位數

USE 方法（基礎設施）

指標定義資源範例
Utilization資源忙碌程度CPU 使用率
Saturation排隊等待程度執行緒佇列長度
Errors錯誤事件數磁碟 I/O 錯誤

日誌分級標準

等級用途何時使用告警
FATAL系統即將崩潰不可恢復的錯誤立即
ERROR操作失敗可恢復但需關注高優先
WARN潛在問題非預期但不影響功能監控
INFO業務事件重要的狀態轉換無
DEBUG除錯資訊開發除錯用無
TRACE詳細追蹤極詳細的執行路徑無
