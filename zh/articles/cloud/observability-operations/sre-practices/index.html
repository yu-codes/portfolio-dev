SRE 實踐

站點可靠性工程

站點可靠性工程（Site Reliability Engineering, SRE）是 Google 提出的一套方法論，透過軟體工程的方式解決維運問題。SRE 團隊負責確保服務的可用性、延遲、效能與容量達到業務需求，同時維持合理的變更速率。本文將深入介紹 SRE 的核心實踐，包括 SLO 與錯誤預算、混沌測試以及值班管理。

核心概念

服務層級指標（SLI） 是對服務行為的定量量測，反映使用者體驗的關鍵維度。典型的 SLI 包括：可用性（成功請求比例）、延遲（請求處理時間分佈）、吞吐量（單位時間處理的請求數）與正確性（回傳正確結果的比例）。好的 SLI 應該直接映射使用者感知的服務品質。

服務層級目標（SLO） 是團隊對 SLI 設定的目標值，例如「99.9% 的請求在 200ms 內完成」。SLO 是團隊與利害關係人之間的內部協議，決定了投入可靠性改善與功能開發之間的平衡點。設定 SLO 時需考慮使用者期望、技術限制與業務成本。

錯誤預算（Error Budget） 是 SLO 允許的不可靠額度。若 SLO 設為 99.9%，則每月有 0.1%（約 43 分鐘）的錯誤預算。當預算充足時，團隊可以加速功能發布；當預算即將耗盡時，應暫停變更並專注於可靠性改善。這個機制讓開發與維運團隊有了共同的量化決策基礎。

消除辛勞（Toil Elimination） 是 SRE 的重要原則。辛勞指的是手動、重複、可自動化且隨服務規模線性增長的維運工作。SRE 團隊應將至少 50% 的時間投入工程工作，減少辛勞佔比，以提升團隊的可擴展性。

事後分析（Postmortem） 是在事件發生後進行的無責任檢討，目的是理解根因、記錄時間線並產出具體的改善行動項。好的事後分析文化強調學習而非追究責任，鼓勵團隊誠實報告問題。

架構總覽

SRE 實踐的系統架構通常包含以下組件：

SLO 監測平台 — 持續計算各服務的 SLI 值並與 SLO 目標比較。使用滑動視窗（通常 28 天或 30 天）計算錯誤預算剩餘量。當預算消耗超過預期速率時觸發警報。Prometheus + Grafana 或 Google Cloud SLO Monitoring 都是常見方案。

變更管理閘道 — 整合 CI/CD 流水線，在錯誤預算不足時自動阻止非關鍵部署。支援例外審批機制以處理緊急修復。

混沌工程平台 — 在生產或預生產環境中注入故障（如網路延遲、服務中斷、資源耗盡），驗證系統的韌性假設。Chaos Monkey、Litmus 與 Gremlin 是常見工具。

值班管理系統 — 排定值班輪值、定義升級路徑、管理警報路由。整合通訊工具（Slack、PagerDuty）確保事件能即時通知到正確的人員。

自動化修復引擎 — 針對已知故障模式定義自動修復程序（Runbook Automation），減少人為介入的需求。例如自動重啟失敗的 Pod、自動擴展資源或自動切換流量。

應用場景

錯誤預算驅動的發布節奏 — 電商平台將結帳服務 SLO 設為 99.95%。開發團隊每週檢視錯誤預算：預算充足時每日部署；預算低於 50% 時降為每週部署並增加測試覆蓋；預算低於 20% 時凍結功能發布，全力修復可靠性問題。

混沌測試驗證韌性 — 金融服務公司定期在預生產環境執行混沌實驗：隨機終止微服務實例、注入網路分割、模擬第三方 API 超時。每次實驗前制定假設（如「單一 Pod 失敗不會影響使用者」），實驗後驗證並修復發現的弱點。

值班輪替最佳化 — SRE 團隊分析過去 6 個月的值班資料，發現 70% 的深夜警報為非緊急事件。通過改善警報閾值、增加自動修復覆蓋率，將深夜 Pager 頻率降低 60%，顯著改善團隊的工作生活平衡。

事後分析文化建設 — 每次 P1 事件後 48 小時內完成事後分析文件，包含詳細時間線、根因分析、影響範圍與行動項。每月舉辦「事件回顧會」分享學習，將通用教訓轉化為自動化防護。
