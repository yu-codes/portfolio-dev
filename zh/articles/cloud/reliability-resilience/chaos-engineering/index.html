混沌工程

混沌工程（Chaos Engineering）是一種透過主動注入故障來驗證系統韌性的工程實踐。其核心理念是「在生產環境中進行受控實驗，以發現系統在面對異常狀況時的弱點」。Netflix 於 2010 年代初期率先提出此概念，並開發了 Chaos Monkey 工具，成為業界標竿。

核心概念

混沌工程的基礎建立在以下幾個關鍵原則之上：

穩態假說（Steady State Hypothesis）：在實驗開始前，先定義系統的「正常行為」指標，例如請求成功率、延遲百分位數、錯誤率等。實驗的目的是驗證在注入故障後，系統是否仍能維持穩態。

故障注入類型：

  • 基礎設施層級：VM 關閉、磁碟滿載、網路分割（network partition）、DNS 解析失敗
  • 應用程式層級：服務延遲注入、例外拋出、記憶體洩漏模擬
  • 依賴層級：資料庫連線中斷、第三方 API 超時、訊息佇列堆積

爆炸半徑控制（Blast Radius）：每次實驗應限制影響範圍，從小規模開始逐步擴大。可透過流量百分比、特定區域或特定用戶群來控制。

自動化回滾機制：實驗必須具備緊急停止按鈕（kill switch），當指標超過預設閾值時自動終止實驗並恢復正常狀態。

持續驗證：混沌實驗不應是一次性活動，而是融入 CI/CD 管線中持續執行，確保系統韌性隨時間維持。

控制平面負責定義實驗參數、排程執行時間、監控安全護欄並產生報告。故障注入代理部署於目標環境中，接收控制平面指令並執行具體的故障注入動作。觀測性平台則持續收集指標，用於判斷穩態假說是否成立。

選型建議：若已深度使用 Kubernetes，Litmus Chaos 提供最佳整合；若需要快速開始且預算充足，Gremlin 的 SaaS 模式最為便捷；若環境純粹在 AWS 上，FIS 則提供最緊密的原生整合。

應用場景

場景一：微服務韌性驗證 在包含數十個微服務的電商平台中，注入下游支付服務的延遲（從正常 50ms 增加至 5000ms），驗證上游服務的斷路器（Circuit Breaker）是否正確觸發，以及降級策略（Fallback）是否如預期運作。此實驗可確保在支付服務異常時，用戶仍能瀏覽商品與加入購物車。

場景二：多區域容錯切換 模擬整個可用區（Availability Zone）失效，驗證流量是否自動路由至健康區域，以及資料庫副本是否正確提升為主節點。此實驗可驗證 RPO/RTO 是否符合 SLA 承諾。

場景三：資源耗盡測試 逐步消耗 Pod 的 CPU 和記憶體資源，觀察 Horizontal Pod Autoscaler（HPA）的擴展行為是否符合預期，以及在資源完全耗盡前是否有足夠的告警通知 SRE 團隊。

場景四：狀態資料損毀 在分散式快取（如 Redis Cluster）中模擬部分節點資料損毀，驗證應用程式的快取穿透處理邏輯、資料一致性恢復機制是否正確運作。

場景五：GameDay 演練 定期舉辦全組織的混沌實驗日，結合多種故障場景同時注入，訓練 On-Call 工程師的事件回應能力，並記錄改善項目追蹤至完成。
