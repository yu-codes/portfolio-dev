備份與復原是資料保護的最後一道防線，確保在任何數據遺失場景下都能恢復關鍵資料。在雲端環境中，備份策略需要涵蓋從虛擬機快照到資料庫邏輯備份、從物件儲存版本控制到跨區域複製等多個層面，形成全方位的資料保護體系。

核心概念

備份類型

完整備份（Full Backup）：

  • 備份所有資料的完整副本
  • 恢復速度最快，直接使用最新完整備份
  • 佔用空間最大，備份時間最長
  • 適合作為基線備份

增量備份（Incremental Backup）：

  • 僅備份自上次備份（含增量）以來變更的資料
  • 佔用空間最小，備份速度最快
  • 恢復需要完整備份 + 所有後續增量備份
  • 任一增量損毀會影響後續恢復

差異備份（Differential Backup）：

  • 備份自上次完整備份以來所有變更的資料
  • 空間和速度介於完整和增量之間
  • 恢復僅需完整備份 + 最新差異備份
  • 隨時間推移差異備份會逐漸增大

備份策略模式

3-2-1 規則：

  • 保留 3 份資料副本
  • 存放在 2 種不同的儲存媒體上
  • 其中 1 份存放在異地

GFS（Grandfather-Father-Son）輪替：

  • 每日備份保留 7 天
  • 每週備份保留 4 週
  • 每月備份保留 12 個月
  • 每年備份長期保留

快照（Snapshot）

快照是特定時間點的資料狀態記錄：

  • 卷快照：磁碟層級的時間點副本（如 EBS Snapshot）
  • 資料庫快照：資料庫一致性狀態的備份
  • 檔案系統快照：使用 Copy-on-Write 技術的即時快照
  • 應用一致性快照：確保應用層面資料一致的快照

復原點類型

  • 時間點恢復（Point-in-Time Recovery, PITR）：恢復到指定的任意時間點
  • 快照恢復：恢復到特定快照的時間點
  • 邏輯恢復：使用邏輯備份（如 SQL dump）重建資料
  • 檔案級恢復：僅恢復特定檔案而非整個系統

  └─ 檔案系統：凍結 I/O
  │
  ▼

  ├─ 快照建立

  └─ 檔案複製

  ├─ 驗證備份完整性（checksum）
  ├─ 加密備份資料

  └─ 更新備份目錄
  │
  ▼

應用場景

資料庫備份策略

關聯式資料庫的備份實踐：

  • 自動化每日完整備份 + 持續的 WAL/Binlog 歸檔
  • 啟用 PITR 實現任意時間點恢復
  • 跨區域複製備份檔案
  • 定期測試恢復流程確保備份可用

Kubernetes 狀態備份

容器化環境的備份挑戰：

  • 使用 Velero 備份 Kubernetes 資源和 PV
  • etcd 快照保護叢集狀態
  • Persistent Volume 的跨 AZ 快照
  • Helm Release 和 ConfigMap 的版本控制

物件儲存保護

大量非結構化資料的保護：

  • 啟用物件版本控制防止意外覆寫
  • 設定 Object Lock 防止惡意刪除
  • Cross-Region Replication 實現異地備份
  • 生命週期規則自動轉移舊版本到低成本儲存

勒索軟體防護

針對勒索軟體的備份強化：

  • 不可變備份（Immutable Backup）防止備份被加密
  • Air-gapped 備份隔離惡意軟體
  • 多層備份策略確保至少一份乾淨備份
  • 定期的備份完整性驗證和恢復測試

AWS 服務實作

RDS Multi-AZ

運作方式

  • 在不同可用區（AZ）建立同步備援實例
  • 發生故障時自動將 CNAME 指向備援實例
  • 故障轉移時間通常為 1–2 分鐘

RDS Multi-AZ 特性

特性說明
複製方式同步複製（Synchronous）
備援用途僅用於故障轉移，不處理讀取請求
額外費用需支付備援實例費用
適用引擎所有 RDS 引擎均支援

Aurora 高可用性

Aurora 儲存架構

  • 資料自動複製至 3 個可用區
  • 每個 AZ 儲存 2 份副本，共 6 份副本
  • 可容忍最多 2 份副本遺失（仍維持寫入能力）
  • 可容忍最多 3 份副本遺失（仍維持讀取能力）

Aurora 故障轉移

  • 存在 Aurora Replica 時：將 Replica 提升為主節點（通常 < 30 秒）
  • 無 Replica 時：建立全新主節點（時間較長）

項目說明

CLI 預設值1 天
備份類型增量備份（Incremental）

RDS 手動快照

  • 手動建立，不自動過期，需手動刪除
  • 支援跨區域複製（Cross-Region Copy）
  • 可分享給其他 AWS 帳號

Aurora 備份

  • 自動連續備份至 S3（無需額外設定）
  • 保留期 1–35 天
  • 支援時間點還原（Point-in-Time Recovery, PITR）

AWS Backup 集中備份服務

AWS Backup 可集中管理並自動備份下列資源：

  • Amazon EC2
  • Amazon EBS
  • Amazon EFS
  • Amazon RDS
  • Amazon DynamoDB
  • Amazon FSx（Lustre / Windows）
  • AWS Storage Gateway

典型使用場景：

  • 備份 RDS 並保留 90 天（RDS 原生自動備份最多僅支援 35 天）
  • 跨區域備份（Cross-Region Backup）
  • 集中管理多帳號備份策略與合規報告

策略RTORPO成本

Aurora Global Database

  • RPO：1 秒（次要區域的資料延遲上限）
  • RTO：1 分鐘（跨區域故障切換時間）
  • 適用於需要最低 RPO/RTO 的跨區域災難復原場景

串流與大數據分析服務

Amazon Kinesis

全託管即時資料串流服務，包含四個子服務：

類型說明
Kinesis Data Streams接收並儲存即時資料流，保留期 24 小時至 365 天；生產者：Kinesis Agent / SDK / KPL；消費者：Analytics / Firehose / KCL
Kinesis Data Firehose將資料流直接導入 S3、Redshift 或 OpenSearch，可搭配 Lambda 進行即時轉換
Kinesis Data Analytics使用 SQL 或 Apache Flink 即時分析資料流
Kinesis Video Streams接收並處理即時影音串流

Amazon Redshift

  • 以**欄式儲存（Columnar Storage）**為基礎的 OLAP 分析型資料庫
  • 支援 **MPP（Massively Parallel Processing）**平行查詢架構
  • 適合大規模資料分析與資料倉儲（Data Warehouse）場景
  • 可整合 Amazon QuickSight、Tableau 等 BI 工具
  • Redshift Spectrum：不需載入資料，直接對 S3 上的資料執行 SQL 查詢

Amazon EMR

  • 全託管大數據平台（Elastic MapReduce）
  • 支援 Hadoop、Spark、Hive、Flink、Presto 等主流開源大數據框架
  • 適用於大規模資料轉換（ETL）與分析工作負載
