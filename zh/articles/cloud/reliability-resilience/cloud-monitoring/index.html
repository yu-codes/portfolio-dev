雲端監控

雲端監控的核心概念、指標體系與 AWS CloudWatch 完整實作指南

雲端監控（Cloud Monitoring）是維持現代雲端系統穩定運行的基礎。透過持續收集各類指標（Metrics）、日誌（Logs）與追蹤資料（Traces），運維團隊得以即時掌握系統健康狀態、提前發現潛在問題，並在故障發生時快速定位根因。完善的監控體系不僅降低了平均故障修復時間（MTTR），更讓工程師能從被動救火轉為主動預防。

在雲端原生環境中，服務數量與複雜度呈指數成長，傳統的主機層監控已遠遠不夠。現代可觀測性（Observability）實踐要求從三個維度——指標、日誌、追蹤——構建完整的監控鏈路，並透過自動化告警與智慧分析，讓系統異常在影響終端使用者之前就被偵測與處理。

各大雲端供應商均提供原生監控服務，AWS 的 CloudWatch 是整合度最高的選項之一，支援從基礎設施指標到應用程式效能監控（APM）的完整鏈路；而 Prometheus + Grafana 組合在開源社群中廣受歡迎，適合多雲或混合雲場景；Datadog 則以易用性和豐富的整合取勝。理解各工具的定位與取捨，是建立高效監控體系的關鍵。

核心概念

可觀測性的三大支柱

1. 指標（Metrics）

指標是對系統狀態的數值型測量，通常以時間序列的形式儲存。常見指標類型：

  • 系統指標：CPU 使用率、記憶體使用量、磁碟 I/O、網路流量
  • 應用指標：請求延遲（P50/P95/P99）、吞吐量（RPS）、錯誤率
  • 業務指標：訂單量、支付成功率、使用者活躍數

指標的解析度（Resolution）決定了監控精細程度：標準解析度通常為每分鐘一個資料點，高解析度可達每秒一個資料點（適用於需要快速響應的告警）。

2. 日誌（Logs）

日誌是系統在特定時刻產生的事件記錄，提供上下文豐富的診斷資訊。有效的日誌管理包括：

  • 結構化日誌：使用 JSON 格式，便於機器解析與搜尋
  • 集中式收集：將分散的服務日誌匯聚到統一平台
  • 保留策略：依合規要求設定不同的保留期限
  • 日誌等級：DEBUG / INFO / WARN / ERROR / FATAL，生產環境通常設定為 INFO 或以上

3. 追蹤（Traces）

分散式追蹤紀錄一個請求在多個服務之間的完整旅程，幫助識別效能瓶頸與錯誤根因：

  • Trace ID：唯一標識一個完整請求鏈路
  • Span：請求鏈路中的單個操作單元
  • OpenTelemetry：雲原生追蹤的業界標準，支援多語言 SDK

儀表板與視覺化

儀表板（Dashboard）將指標與日誌以視覺化方式呈現，是監控體系的「指揮中心」：

  • Golden Signals：延遲（Latency）、流量（Traffic）、錯誤（Errors）、飽和度（Saturation）
  • SLI / SLO：服務等級指標與目標，量化可靠性承諾
  • 告警規則：基於閾值或異常偵測觸發通知，避免告警疲勞

核心組件職責

資料收集：CloudWatch Agent 安裝於 EC2，收集作業系統層級指標（記憶體、磁碟使用量等 EC2 預設不提供的指標）；各 AWS 服務自動發布指標至 CloudWatch，無需額外配置。

資料儲存：CloudWatch Metrics 使用時間序列資料庫儲存指標，標準保留期為 15 個月；CloudWatch Logs 以 Log Group 和 Log Stream 組織日誌，保留期限從 1 天到永久可自訂。

告警與自動化：CloudWatch Alarms 監控指標是否超過閾值，並觸發 SNS 通知、Auto Scaling 動作或 EC2 恢復操作；EventBridge 則負責排程任務與事件驅動的自動化工作流。

AWS CloudWatch 完整實作

概述

AWS CloudWatch 是 AWS 的核心資料收集與監控服務，整合了指標（Metrics）、**日誌（Logs）與事件（Events）**三大功能，涵蓋 EC2、ALB、S3、Lambda、DynamoDB、RDS 等所有主要 AWS 服務。

資料流架構：

AWS 服務/應用程式 → CloudWatch Metrics/Logs → Alarms/Dashboard/Insights

指標（Metrics）

解析度

類型間隔費用
標準每分鐘免費
高解析度最多 1 秒付費

EC2 預設指標

支援不支援（需安裝 CloudWatch Agent）
CPU 使用率記憶體使用量
網路 I/O磁碟使用量（百分比）
磁碟 I/O（IOPS）自訂應用程式指標

自訂指標

使用 PutMetricData API 發布自訂指標：

  • 可設定維度（Dimensions）進行多維度分析
  • 支援高解析度（最短 1 秒間隔）
  • 常見應用：記憶體使用率、應用程式效能指標、業務指標

Metric Math

  • 對多個指標進行數學運算，建立複合指標
  • 可用於告警和儀表板
  • 例如：計算請求錯誤率 = 錯誤請求數 / 總請求數 × 100

CloudWatch Logs

架構層級

層級說明
Log Group日誌的邏輯容器，通常對應一個服務或應用程式
Log StreamLog Group 內的日誌序列，通常對應一個實例
Log Event單筆日誌記錄（時間戳記 + 訊息）

主要功能

  • 收集並集中管理應用程式日誌
  • 即時日誌搜尋與過濾
  • 日誌保留設定（1 天至永久）
  • 匯出至 S3（批次，非即時）
  • 使用 KMS 加密日誌

CloudWatch Logs Agent / Unified Agent

CloudWatch Logs Agent 安裝於 EC2 實例，負責：

  • 收集自訂指標（記憶體、磁碟使用量等）
  • 收集並傳輸應用程式日誌至 CloudWatch Logs
  • Unified CloudWatch Agent 同時支援 EC2 和本地伺服器

寫入 CloudWatch Logs 所需權限

任何服務若要寫入 CloudWatch Logs，需具備以下 IAM 權限：

  • logs:CreateLogGroup
  • logs:CreateLogStream
  • logs:PutLogEvents

Subscriptions（訂閱過濾器）

CloudWatch Logs → Subscription Filter → Lambda / Kinesis Data Streams / Kinesis Firehose
  • 支援即時日誌處理
  • 可跨帳號傳遞日誌
  • 常見用途：即時日誌轉發至 Elasticsearch 或第三方 SIEM

Logs Insights

  • 互動式日誌查詢工具
  • 類 SQL 查詢語法，支援 filter、stats、sort、limit 等命令
  • 視覺化查詢結果（時間序列圖、長條圖）

CloudWatch Alarms

告警狀態

狀態說明
OK指標在定義的閾值內
ALARM指標在定義的閾值外
INSUFFICIENT_DATA資料不足以判斷狀態

告警配置參數

設定說明
Period指標評估的時間長度（例如：60 秒）
Evaluation Periods判斷告警前的評估次數（例如：評估最近 3 個週期）
Datapoints to Alarm觸發告警所需的違規資料點數量（例如：3 個中的 2 個）

告警動作類型

  • SNS 通知：發送 Email、SMS 或觸發 HTTP endpoint
  • Auto Scaling 動作：Scale Out 或 Scale In
  • EC2 動作：停止（Stop）、終止（Terminate）、重啟（Reboot）、恢復（Recover）

EC2 自動復原

CloudWatch Alarm 可根據系統狀態檢查失敗，自動觸發 EC2 復原（Recovery）：

  • 保留實例 ID、彈性 IP、私有 IP
  • 保留 EBS 磁碟資料
  • 復原後實例重新上線，通常無需人工介入

CloudWatch Dashboard

  • 視覺化展示指標圖表與告警狀態
  • 支援跨帳號、跨區域數據整合在同一儀表板
  • 可自訂時間範圍、自動重新整理間隔
  • 可將 Logs Insights 查詢結果嵌入儀表板

CloudWatch Events / EventBridge

主要用途

  • 排程任務（類似 cron 或 rate 表達式）
  • 回應 AWS 服務狀態變更事件
  • 觸發 Lambda、SNS、SQS、Step Functions 等目標

EventBridge 詳解

EventBridge 是 CloudWatch Events 的進化版，新增：

  • 預設事件匯流排 + 自訂事件匯流排
  • 可接收第三方 SaaS 應用程式事件
事件來源
來源範例
AWS 服務EC2 狀態變更、S3 物件建立
排程Cron 表達式或 rate 表達式
第三方SaaS 應用程式（Zendesk、Datadog 等）
自訂您的應用程式透過 PutEvents API
事件模式範例
{
  "source": ["aws.ec2"],
  "detail-type": ["EC2 Instance State-change Notification"],
  "detail": {
    "state": ["stopped"]
  }
}

CloudWatch 進階功能

Contributor Insights

  • 識別對系統負載貢獻最高的資料來源（高流量 IP、熱點資源等）
  • 找出效能問題與流量異常的根本原因
  • 支援自訂規則，分析 CloudWatch Logs 或 DynamoDB 資料

CloudWatch Synthetics

  • 建立 Canary 腳本，主動監控 API 端點與 Web 頁面可用性
  • 模擬真實使用者行為（點選流程、表單提交等）
  • 在問題影響真實使用者前主動發現，並觸發告警

CloudWatch Internet Monitor

  • 監控網際網路連線效能（從 AWS 到終端使用者）
  • 提供影響應用程式可用性的網路問題洞察
  • 識別網路擁塞、路由問題，協助判斷是否為 ISP 或 CDN 問題

CloudWatch Application Signals

  • 應用程式效能監控（APM），整合 AWS X-Ray 追蹤資料
  • 自動化儀表板，展示服務地圖與依賴關係
  • 服務等級目標（SLO）追蹤，量化可靠性承諾

AWS Config（相關服務）

概述

  • 記錄 AWS 資源的配置變更歷史
  • 評估資源是否符合合規規則
  • 不阻止變更，只記錄、評估並警示

核心概念

概念說明
Configuration Item資源在特定時間點的配置快照
Configuration History資源的完整歷史配置變更紀錄
Configuration Stream配置變更的即時串流

Config Rules

類型說明
AWS Managed RulesAWS 預定義的規則（如 restricted-ssh）
Custom Rules使用 Lambda 函數自訂評估邏輯

範例規則：

  • restricted-ssh：確保安全群組不允許 0.0.0.0/0:22 SSH 存取
  • encrypted-volumes：確保所有 EBS 磁碟已啟用加密
  • required-tags：確保資源具備必要的標籤（如 Environment、Owner）

Config Rules 支援觸發自動修復，透過 SSM Automation Documents 還原不合規配置。

最佳實踐

  1. 安裝 Unified CloudWatch Agent：收集 EC2 記憶體、磁碟等預設不支援的指標
  2. 設定合適的警示閾值：利用 Datapoints to Alarm 避免因瞬間抖動觸發誤報
  3. 使用 Dashboard 集中監控：建立跨服務的統一視圖，包含 Golden Signals
  4. 配置日誌保留策略：依業務需求設定保留期，避免無限期累積增加成本
  5. 使用 Logs Insights 主動分析：定期查詢錯誤模式，而非等待告警才行動
  6. 部署 Synthetics Canary：主動探測關鍵端點，在使用者感知前發現問題
  7. 啟用 Contributor Insights：快速識別異常流量來源或 Hot Key 問題

應用場景

場景一：EC2 高記憶體用量告警

生產環境的 EC2 實例時常因記憶體洩漏導致服務降級。安裝 Unified CloudWatch Agent 後，設定記憶體使用率指標，當超過 85% 達 5 分鐘以上時觸發 CloudWatch Alarm，透過 SNS 通知值班工程師，並同步觸發 Systems Manager Automation 嘗試重啟相關應用程式服務。

場景二：Lambda 錯誤率監控

微服務架構中，多個 Lambda 函數組成關鍵業務鏈路。使用 CloudWatch 自動收集的 Lambda 指標（Errors、Duration、Throttles），設定錯誤率 > 1% 的複合告警，搭配 Logs Insights 快速查詢錯誤堆疊，縮短從告警到定位根因的時間。

場景三：API Gateway 效能儀表板

電商平台在促銷期間流量暴增。利用 CloudWatch Dashboard 整合 API Gateway 的延遲（P99）、4xx/5xx 錯誤率、Auto Scaling 實例數，以及 RDS 連線數等指標，構建即時監控大屏。當 P99 延遲超過 500ms 時自動觸發 Scale Out 策略。

場景四：Synthetics 主動監控

金融服務的交易介面要求 99.9% 可用性。部署 CloudWatch Synthetics Canary，每分鐘模擬登入、查詢帳戶餘額、發起轉帳等完整使用者旅程。一旦任何步驟失敗或超時，立即觸發 PagerDuty 告警，確保服務中斷在 SLA 響應時間內得到處理。

場景五：成本異常偵測

透過 CloudWatch Metric Math 計算每小時 AWS 費用趨勢，結合 EventBridge 定時觸發 Lambda 查詢 Cost Explorer API，當費用突然超過滾動平均的 150% 時，發送 SNS 告警至財務與技術負責人，防止因配置錯誤或安全事件導致的費用暴增。
