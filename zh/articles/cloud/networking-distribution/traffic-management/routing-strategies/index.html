路由策略（Routing Strategies）定義了流量如何在分散式系統中被導向不同的服務版本或實例。正確的路由策略是實現安全部署、流量控制與系統彈性的關鍵。

核心概念

路由決策維度

維度說明範例
權重（Weight）按百分比分配90% v1, 10% v2
標頭（Header）按 HTTP 標頭X-Version: canary
Cookie按使用者 Cookiebeta_user=true
地理（Geo）按來源地理位置台灣 → AP Region
使用者屬性按用戶特徵付費用戶 → 新版
路徑（Path）按 URL 路徑/v2/* → 新服務

架構總覽

藍綠部署（Blue-Green Deployment）

階段Blue（當前版）Green（新版）流量
部署前100% 服務中不存在全部→Blue
部署中100% 服務中部署並驗證全部→Blue
切換待命驗證通過全部→Green
觀察保留（回滾用）100% 服務中全部→Green
完成銷毀正式環境全部→Green

金絲雀發布（Canary Release）

階段舊版流量新版流量觀察指標
起始100%0%—
第一階段95%5%錯誤率、延遲
第二階段80%20%商業指標
第三階段50%50%全面監控

A/B 測試路由

面向金絲雀A/B 測試
目的驗證穩定性驗證業務效果
分流依據隨機百分比用戶特徵
時間短（確認後全量）長（收集統計顯著性）
指標技術指標（錯誤、延遲）商業指標（轉換率）
用戶體驗同功能不同版本可能不同功能

流量鏡像（Shadow Traffic）

面向說明
原理複製生產流量到新版本
回應新版本回應被丟棄
風險對用戶零影響
用途效能測試、正確性驗證
注意避免重複寫入操作
工具Istio Mirror、GoReplay

應用場景

微服務漸進式發布

  1. 部署新版本到 Kubernetes
  2. Istio VirtualService 設定 5% 流量到新版
  3. 監控錯誤率、P99 延遲、商業指標
  4. 自動或手動逐步增加比例
  5. 100% 流量切換完成，移除舊版

多租戶 SaaS 路由

  • 企業客戶 → 獨立叢集（隔離）
  • 付費用戶 → 優先路由（低延遲路徑）
  • Beta 用戶 → 新功能版本
  • 免費用戶 → 標準路由

全球流量管理

  • 地理路由：用戶就近存取
  • 故障轉移：Region 故障自動切換
  • 合規路由：特定資料不出特定區域
  • 成本路由：閒時流量導向便宜 Region
