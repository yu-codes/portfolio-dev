斷路器模式

深入解析斷路器模式（Circuit Breaker Pattern）的設計原理、狀態機制與在分散式系統中防止級聯故障的應用

斷路器模式（Circuit Breaker Pattern）源自電力系統的概念，用於防止分散式系統中的級聯故障。當某個下游服務持續故障時，斷路器會「跳閘」中斷請求，避免無意義的等待與資源浪費，同時給予故障服務恢復的時間。

核心概念

級聯故障問題

在微服務架構中，一個服務的故障可能導致連鎖反應：

階段事件影響
1服務 C 故障回應超時
2服務 B 等待服務 C執行緒堆積
3服務 B 資源耗盡服務 B 故障
4服務 A 等待服務 B同樣堆積
5全系統雪崩所有服務不可用

斷路器的價值：在步驟 2 就切斷無效等待，快速失敗，保護上游服務。

斷路器狀態機

狀態說明行為
Closed（閉合）正常運作請求正常通過，監控失敗率
Open（斷開）故障保護立即拒絕所有請求（快速失敗）
Half-Open（半開）探測恢復允許少量請求通過測試

狀態轉換規則：

從到觸發條件
ClosedOpen失敗率超過閾值（如 50%）
OpenHalf-Open等待超時後（如 30 秒）
Half-OpenClosed探測請求成功
Half-OpenOpen探測請求失敗

關鍵參數

參數說明典型值影響
Failure Threshold觸發開啟的失敗率50%靈敏度
Request Volume計算失敗率的最少請求數20避免誤判
Sleep WindowOpen 狀態持續時間30-60 秒恢復等待
Half-Open Max Requests探測請求數量5-10驗證充分性
Success Threshold恢復所需成功率60-100%恢復信心
Sliding Window統計窗口大小10 秒/100 請求統計精度

關鍵比較

斷路器 vs 重試 vs 逾時

面向斷路器重試逾時
目的防止級聯故障克服暫時故障限制等待時間
粒度服務/端點級單次請求單次請求
決策依據歷史失敗統計當前失敗時間
保護對象呼叫方 + 被呼叫方呼叫方呼叫方
生效時機持續故障後每次失敗每次請求

斷路器 vs Bulkhead

面向斷路器Bulkhead（隔板）
目的切斷故障鏈隔離資源
機制失敗率統計資源配額
保護方式快速失敗限制並發
觸發反應式（故障後）預防式（配額滿）
範圍每個下游服務每個資源池
搭配常與 Bulkhead 一起使用常與斷路器一起使用

什麼場景不適合斷路器

場景原因替代方案
單一關鍵依賴無法降級重試 + 佇列
一致性要求極高不能快速失敗同步等待 + 逾時
批次處理不需即時回應重試佇列
極低流量統計不準確健康檢查
資料庫直連連線池已有控制連線池設定

降級策略（Fallback）

策略說明適用場景
快取回應回傳上次成功結果產品列表、設定
預設值回傳預設資料推薦系統
備用服務呼叫替代服務多供應商
佇列延遲存入佇列稍後處理非即時操作
優雅降級部分功能不可用複合頁面
錯誤回應明確告知用戶通用

應用場景

電商微服務

  • 商品服務 → 庫存服務（斷路器）
  • 庫存不可用時 → Fallback 顯示「請洽客服」
  • 支付服務 → 第三方金流（斷路器）
  • 金流不可用 → 佇列延遲處理

API 聚合層

  • BFF（Backend for Frontend）呼叫多個微服務
  • 每個下游服務獨立斷路器
  • 部分資料不可用時組合可用資料回傳
  • 降低整體故障影響面

Service Mesh 自動管理

  • Istio DestinationRule 設定 Outlier Detection
  • 無需修改應用程式碼
  • Envoy Sidecar 自動執行
  • 搭配 Retry Policy 與 Timeout
