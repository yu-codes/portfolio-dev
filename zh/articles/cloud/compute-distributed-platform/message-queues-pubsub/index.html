訊息佇列與 Pub/Sub

深入解析訊息佇列與發布訂閱模式的核心概念、架構設計與各雲平台服務比較

訊息佇列（Message Queue）與發布訂閱（Pub/Sub）是分散式系統中實現非同步通訊的兩大核心模式。它們讓系統元件之間解耦合，提高可靠性與擴展性。

核心概念

佇列模型 vs Pub/Sub 模型

面向佇列（Queue）發布訂閱（Pub/Sub）
消費模式一對一（競爭消費者）一對多（廣播）
訊息命運被一個消費者處理後刪除所有訂閱者都收到
用途工作分配、任務佇列事件通知、廣播
負載分散天然支援每個訂閱者獨立
範例SQS、RabbitMQ QueueSNS、Pub/Sub Topic

訊息傳遞保證

等級說明複雜度適用
At-most-once最多一次（可能遺失）最低日誌、指標
At-least-once至少一次（可能重複）中大多數場景
Exactly-once恰好一次最高金融交易

訊息順序保證

等級說明效能影響範例
無順序任意順序處理最高吞吐獨立任務
分區順序同一分區內有序中等Kafka Partition
FIFO嚴格先進先出較低吞吐SQS FIFO
全域順序所有訊息全序最低極少見

應用場景

非同步工作佇列

  • Web 請求快速回應
  • 耗時工作推入佇列
  • Worker 群組競爭消費
  • Auto Scaling 依佇列深度擴展

事件扇出（Fan-out）

  • 訂單事件 → SNS Topic
  • SNS → SQS（庫存服務）
  • SNS → SQS（物流服務）
  • SNS → SQS（通知服務）
  • SNS → Lambda（即時分析）

系統解耦合

  • 服務 A 不直接呼叫服務 B
  • 透過佇列非同步通訊
  • 服務 B 暫時不可用 → 訊息等待
  • 恢復後自動處理積壓

AWS 服務實作

AWS SQS

概述

  • 完全託管佇列服務
  • 訊息保留時間：預設 4 天，最長 14 天
  • 單筆訊息最大 256 KB
  • 可無限量訊息排隊
  • 公共服務，在 AWS 公共區域端點存取
  • 完全託管、高可用，用於分離應用程式元件

Standard SQS 特性

特性說明
吞吐量無限制
發送/接收延遲<10ms
送達保證至少一次（可能重複）
訊息順序盡力維持（可能無序）

消費者行為

  • 消費者（EC2 或 Lambda）以批次方式 poll（最多 10 筆）
  • 處理完後必須手動刪除訊息，否則會再次被處理
  • 需在 SQS 存取政策中允許 Producer 和 Consumer 存取佇列

訊息可見性逾時（Message Visibility Timeout）

  • 訊息被取出處理期間會變為不可見
  • 避免被其他消費者重複處理
  • 預設 30 秒，可調整 0 秒 ~ 12 小時
  • 處理時間超過 Visibility Timeout 時，訊息會再次可見

死信佇列（DLQ）

  • 將多次處理失敗的訊息轉入另一個 SQS 中保存
  • 設定 maxReceiveCount：訊息被接收次數超過此值後移至 DLQ
  • 適用於錯誤分析和重新處理
  • 可將 DLQ 訊息移回原始佇列（修復問題後重新處理）
  • DLQ 類型必須與來源佇列相同（Standard 對 Standard）
  • 建議設定足夠長的保留期限

訊息延遲

類型設定位置最大延遲
Delivery DelayQueue 層級15 分鐘
DelaySeconds訊息層級15 分鐘
  • 預設延遲為 0 秒

長輪詢（Long Polling）

  • 設定 ReceiveMessageWaitTimeSeconds > 0
  • 當無訊息時等待訊息出現再回傳
  • 可節省費用和減少空請求
  • 建議值：1–20 秒

FIFO 佇列

特性StandardFIFO
訊息順序盡力維持嚴格保證
送達保證至少一次僅一次
吞吐量無限制300 msg/s（非批次）或 3,000 msg/s（批次）
佇列名稱任意必須以 .fifo 結尾
  • 無法從 Standard 轉為 FIFO
FIFO 關鍵概念
概念說明
Message Group ID同一 Group 內訊息保證順序
Message Deduplication ID防止重複訊息（5 分鐘內）

Standard vs FIFO 詳解

Standard 佇列
特性說明
順序盡力維持，但不保證
送達至少一次（可能重複）
效能近乎無限擴展
適用大多數使用案例
FIFO 佇列
特性說明
順序嚴格保證
送達僅一次
效能3,000 msg/s（批次）、300 msg/s（非批次）
命名必須以 .fifo 結尾
  • Message Group ID：同一群組內訊息按順序處理
  • Message Deduplication ID：5 分鐘內去重

SQS Extended Client Library

  • SQS 訊息最大 256 KB；某些場景需要更大的訊息
  • 使用 SQS Extended Client Library 解決此限制
  • 實際資料存儲在 S3，SQS 訊息包含 S3 物件的參照
Producer → S3（大型資料）→ SQS（參照）→ Consumer → 從 S3 讀取

SQS Temporary Queue Client

  • 實作 SQS 版 Request-Response 模式
  • 適用於需要回應的非同步通訊

與 Lambda 整合

  • Lambda 輪詢 SQS（事件來源映射）
  • 批次處理訊息，自動刪除成功處理的訊息
設定說明
Batch Size每次調用的訊息數（1–10）
Batch Window等待收集訊息的時間
並發與佇列類型相關
  • 失敗的訊息保持可見，可配置 DLQ 處理失敗訊息

與 Auto Scaling 整合

使用 CloudWatch 自訂指標觸發 Auto Scaling：

指標 = SQS 佇列長度 / EC2 數量

當指標超過閾值時，增加消費者數量。

與其他服務整合

  • Lambda：作為觸發器自動處理訊息
  • SNS：Fan-out 架構
  • EventBridge：事件驅動架構
  • S3：物件事件通知
  • EC2 Auto Scaling：根據佇列深度擴展

安全性

加密類型說明
傳輸中HTTPS 端點
靜態SSE-SQS（預設）或 SSE-KMS
  • IAM Policy：控制誰可以執行 SQS 操作
  • SQS Access Policy：資源政策，控制跨帳號存取

重要參數彙整

概念說明
訊息大小最大 256 KB
保留期限1 分鐘 – 14 天（預設 4 天）
可見性超時0 秒 – 12 小時（預設 30 秒）
加密SSE-SQS 或 SSE-KMS

SQS 訊息處理流程

Producer → SQS Queue → Consumer（Poll）→ 處理 → 刪除訊息

最佳實踐

  1. 使用長輪詢：減少空請求和成本
  2. 配置 DLQ：處理失敗訊息
  3. 適當設定 Visibility Timeout：大於預期處理時間
  4. 使用批次操作：提高效率
  5. FIFO 用於順序需求：嚴格順序場景
  6. 監控佇列深度：設定警示

AWS SNS

概述

  • 發布/訂閱（PubSub）模型
  • Publisher 發送訊息到 SNS 主題
  • 所有 Subscriber 同步接收訊息
  • 支援最多 10 萬個 Topic
  • 每個 Topic 最多 1,250 萬個訂閱者
  • 公共 AWS 服務，高度可用、持久、安全

訊息大小

  • 最大 256 KB
  • 超過時可使用 Extended Client Library（儲存於 S3）

訂閱端點類型

端點類型說明
Kinesis Firehose串流至資料湖
SQS佇列處理
HTTP/HTTPSWebhook
Lambda無伺服器處理
Email郵件通知
SMS簡訊通知
行動推播App 推播

訂閱過濾政策（Subscription Filter Policy）

  • 用 JSON 設定條件
  • 僅讓特定訊息送到特定訂閱者
  • 減少不必要的訊息傳遞
過濾政策範例
{
  "eventType": ["order_placed"],
  "store": ["store-1", "store-2"]
}
支援的過濾條件
條件類型說明
精確匹配"key": ["value"]
前綴匹配"key": [{"prefix": "val"}]
數值範圍"key": [{"numeric": [">", 0, "<=", 100]}]
存在檢查"key": [{"exists": true}]
  • 每個 Subscriber 可設定過濾政策，只接收符合條件的訊息

Fan-out 模式

概念

SNS 主題對應多個 SQS 訂閱者，達成訊息分發。

架構
Producer → SNS Topic → SQS Queue 1 → Consumer 1
                    → SQS Queue 2 → Consumer 2
                    → SQS Queue 3 → Consumer 3
使用案例

訂單事件依狀態發送給多個服務處理：

訂單建立 → SNS Topic → SQS Queue 1 → Consumer 1（庫存）
                    → SQS Queue 2 → Consumer 2（帳單）
                    → SQS Queue 3 → Consumer 3（物流）
                    → Lambda → Email 通知
優勢
優勢說明
解耦Producer 不需知道 Consumer
可靠性SQS 提供訊息持久化
擴展性輕鬆新增 Subscriber
獨立處理每個 Consumer 獨立失敗/重試

FIFO Topic

  • 支援嚴格訊息順序
  • 訊息去重複
  • 訊息分組
  • 可與 FIFO SQS 佇列搭配使用

訊息屬性

屬性說明
MessageId訊息唯一識別碼
Subject訊息主題（Email 用）
MessageAttributes自訂屬性（用於過濾）
Timestamp發布時間

最佳實踐

  1. 使用過濾政策：減少不必要的訊息
  2. 搭配 SQS 實現 Fan-out：可靠的訊息分發
  3. 啟用加密：保護敏感訊息
  4. 監控指標：追蹤發布和傳遞狀態
  5. 使用 DLQ：處理傳遞失敗
