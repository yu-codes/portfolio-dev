串流處理（Stream Processing）是對持續產生的資料流進行即時分析與轉換的技術。與批次處理不同，串流處理強調低延遲與連續性，適合需要即時洞察的場景。

核心概念

批次 vs 串流處理

面向批次處理串流處理
資料處理方式累積後一次處理逐筆或小批次即時處理
延遲分鐘到小時毫秒到秒
觸發方式排程或手動持續自動
資料範圍有界資料集無界資料串流
狀態管理重新計算增量更新
典型框架MapReduce、SparkFlink、Kafka Streams
適用場景報表、ETL即時監控、即時推薦

串流處理語義

語義說明實作難度適用
At-most-once最多處理一次最低容忍遺失
At-least-once至少處理一次中大多數場景
Exactly-once恰好處理一次最高金融、計費

時間語義

時間類型說明用途
Event Time事件實際發生時間精確分析
Processing Time系統處理時間簡單場景
Ingestion Time進入系統時間折衷方案

視窗（Window）類型

視窗說明適用場景
Tumbling固定大小、不重疊每分鐘統計
Sliding固定大小、可重疊移動平均
Session活動間隔為界使用者行為分析
Global無時間界限自訂觸發

串流處理架構模式

Lambda Architecture（批次 + 串流）：

層級功能工具範例
Batch Layer完整歷史處理Spark、BigQuery
Speed Layer即時增量處理Flink、Kinesis
Serving Layer合併結果查詢資料庫、快取

Kappa Architecture（純串流）：

特點說明
統一處理所有資料都視為串流
簡化架構消除批次層
重播透過 Kafka 重播歷史
適用事件溯源、純串流場景

應用場景

即時異常偵測

  • 資料源：IoT 感測器/應用日誌
  • 串流處理：滑動視窗統計、規則引擎
  • 輸出：即時告警（PagerDuty、Slack）
  • 延遲需求：< 5 秒

即時推薦引擎

  • 使用者行為事件 → Kafka
  • Flink 即時特徵計算
  • 更新使用者特徵向量
  • 推薦服務查詢最新特徵

即時 ETL 管線

  • 來源：CDC（資料庫變更捕獲）
  • 處理：資料清洗、轉換、彙總
  • 目標：資料倉儲即時更新
  • 取代每日批次 ETL
