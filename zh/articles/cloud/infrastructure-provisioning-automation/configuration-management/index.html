組態管理（Configuration Management）是確保基礎設施狀態一致性的核心實踐。它透過程式化的方式定義、部署與維護系統配置，確保所有環境中的伺服器、應用程式與服務都處於期望的狀態。隨著容器化的普及，組態管理的角色正在演變，但其核心原則仍然是基礎設施自動化的基石。

核心概念

組態管理的核心原則

原則說明實踐
冪等性多次執行結果相同只有偏離期望狀態時才修改
可重複性任何時間執行結果一致版本控制所有配置
宣告式描述期望狀態而非步驟系統自行決定如何達成
可追溯性所有變更可追蹤Git 歷史、審計日誌
自癒能力自動修正偏移定期收斂（Convergence）

推送 vs 拉取模型

面向推送模型（Push）拉取模型（Pull）
代表工具AnsiblePuppet、Chef
架構無 Agent需要 Agent
觸發方式中央執行Agent 定期輪詢
即時性按需執行輪詢間隔
擴展性受限於控制節點Agent 分散負載
維護成本低（無 Agent）高（需管理 Agent）

架構總覽

組態管理架構演進

Phase 1: 手動管理
  管理員 → SSH → 逐一配置伺服器

Phase 2: 腳本自動化
  Shell Scripts → 批次執行 → 缺乏冪等性

Phase 3: 組態管理工具
  Ansible/Puppet → 宣告式配置 → 冪等性保證

Phase 4: 不可變基礎設施
  Packer → 映像建構 → 替換而非修改

Phase 5: 容器化
  Dockerfile → 容器映像 → 環境即程式碼

組態管理 vs IaC 工具

面向組態管理（Ansible）IaC（Terraform）
主要用途伺服器內部配置雲端資源佈建
抽象層次OS、軟體、配置雲端服務、網路
狀態追蹤每次執行檢查State file
執行頻率持續或定期變更時執行
典型任務安裝套件、修改配置建立 VPC、EC2
互補性Terraform 佈建 → Ansible 配置

組態偏移（Configuration Drift）管理

策略做法適用場景
定期收斂Agent 定期執行修正Puppet/Chef 環境
按需掃描定期執行 dry-run 檢查Ansible 環境
不可變替換偏移時替換整個實例容器/不可變基礎設施
監控告警偵測偏移並告警所有環境
GitOps 收斂Git 為唯一事實來源K8s 環境

現代化組態管理實踐

  1. IaC + CM 協作：Terraform 佈建基礎設施，Ansible 配置應用
  2. 不可變優先：優先使用容器/映像替換，減少 in-place 變更
  3. GitOps 整合：所有配置存儲在 Git，自動化同步
  4. 策略即程式碼：用 InSpec/OPA 驗證配置合規性
  5. 密鑰整合：與 Vault/Secrets Manager 整合，安全管理密鑰

挑戰解決方案
