API 安全

API 是現代雲端應用的核心介面，也是最常見的攻擊目標。OWASP API Security Top 10 列出的威脅每年造成大量的資料洩露與服務中斷。全面的 API 安全策略必須涵蓋認證、授權、輸入驗證、限流、加密與監控等多個面向。

核心概念

API 安全的威脅面

威脅類別說明嚴重度

授權缺陷BOLA、BFLA高

資源耗盡缺乏限流導致服務不可用中

架構總覽

API 安全架構

客戶端
    ↓
[API Gateway]
├── 認證（JWT 驗證、OAuth Token 驗證）
├── 限流（速率限制、配額管理）
├── WAF（注入偵測、異常請求過濾）
├── 請求驗證（Schema 驗證、大小限制）
└── 日誌記錄（所有請求審計）
    ↓
[後端服務]
├── 授權（RBAC/ABAC 檢查）
├── 業務邏輯驗證
├── 資料過濾（僅返回必要欄位）
└── 內部 API 間 mTLS

API 認證流程

Client → Authorization Server (IdP)
    ↓ 取得 Access Token
Client → API Gateway
    ↓ 附帶 Bearer Token
Gateway: 驗證 Token 簽名 + 過期 + 範圍
    ↓ 通過驗證
Backend Service: 檢查細粒度權限
    ↓ 授權通過
返回結果（僅必要欄位）

OAuth 2.0 最佳實踐

實踐說明
短期 Access Token有效期 5-60 分鐘
Refresh Token 輪換每次使用後作廢舊 Token
Token Scope 最小化僅請求必要的 Scope
PKCE公開客戶端強制使用

Audience 驗證驗證 Token 的目標受眾

API 安全檢查清單

  1. 認證：所有端點強制認證（除公開 API）
  2. 授權：每個操作檢查物件級權限
  3. 輸入驗證：嚴格的 Schema 驗證
  4. 限流：每個端點設定合理的速率限制
  5. 加密：強制 HTTPS、敏感欄位加密
  6. 日誌：記錄所有 API 存取（不含敏感資料）
  7. 版本管理：明確的版本策略與棄用流程
  8. 文件：完整的 API 安全文件

常見漏洞修復

漏洞原因修復方式
BOLA未驗證物件所有權每次存取驗證擁有者
Mass Assignment接受所有輸入欄位白名單允許的欄位
資料過曝返回整個物件明確指定返回欄位
SSRF未驗證 URL 輸入URL 白名單、禁止內部 IP
Injection未清理輸入參數化查詢、輸入驗證
