密鑰輪換（Secrets Rotation）是降低密鑰洩露風險的關鍵安全實踐。即使密鑰遭到未偵測的洩露，定期輪換也能限制攻擊者的利用時間窗口。自動化密鑰輪換消除了手動操作的風險與負擔，是現代雲端安全基礎設施的必備能力。

核心概念

為什麼需要密鑰輪換

原因說明影響
限制洩露影響縮短洩露後的可利用時間降低安全風險
合規要求PCI DSS、SOC 2 等要求定期輪換滿足審計
人員異動知曉密鑰的人員離開防止前員工存取
加密降級長期使用的金鑰面臨更多破解嘗試維持加密強度
事件回應安全事件後強制更換阻斷攻擊

輪換策略類型

策略說明停機時間複雜度
單密鑰輪換直接替換舊密鑰有（短暫）低
雙密鑰輪換新舊並行過渡期無中
N 密鑰輪換多版本同時有效無高
動態密鑰每次使用產生新密鑰無最高

架構總覽

雙密鑰輪換流程

Phase 1: 建立新密鑰
  [舊密鑰: Active] [新密鑰: Created]

Phase 2: 雙寫期
  [舊密鑰: Active] [新密鑰: Active]
  ↓ 目標服務接受新舊兩個密鑰

Phase 3: 切換期
  [舊密鑰: Active] [新密鑰: Primary]
  ↓ 所有客戶端切換到新密鑰

Phase 4: 停用舊密鑰
  [舊密鑰: Deprecated] [新密鑰: Active]
  ↓ 確認無流量使用舊密鑰

Phase 5: 銷毀
  [舊密鑰: Destroyed] [新密鑰: Active]

動態密鑰 vs 靜態輪換

面向動態密鑰（Vault）靜態輪換（SM）
密鑰生命週期分鐘至小時天至月
洩露風險窗口極小輪換週期
架構複雜度高中
依賴性需 Vault 可用僅啟動時
效能影響頻繁請求 Token幾乎無
適用場景高安全要求一般場景

應用場景

資料庫密碼輪換

步驟操作驗證
1在 DB 中建立新密碼新密碼可連線
2更新 Secret Manager 的值API 返回成功
3等待應用程式讀取新密碼連線計數轉移
4刪除舊密碼舊密碼無法連線
5記錄輪換結果審計日誌

多區域輪換挑戰

主區域輪換 → 複製到其他區域 → 等待傳播 → 所有區域驗證
     ↓
注意事項：
├── 複製延遲可能導致暫時不一致
├── 需要全域健康檢查確認
├── 回滾需要全域操作
└── 考慮時區差異與維護窗口

輪換失敗處理

失敗場景偵測方式自動回應
新密鑰無法連線目標驗證步驟失敗回滾到舊密鑰
Secret Manager 更新失敗API 錯誤重試 + 告警
應用未讀取新密鑰連線使用舊密鑰延長過渡期
跨區域複製失敗複製狀態檢查暫停刪除舊版本
輪換超時超時告警人工介入
