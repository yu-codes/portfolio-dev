資料遷移策略（Data Migration Strategy）是企業進行雲端轉型或多雲部署時的關鍵決策。選擇正確的遷移策略不僅影響專案的時程與成本，更直接決定了遷移後系統的效能、可維護性與未來的擴展性。本文深入探討各種遷移策略的適用場景、技術實作與最佳實踐。

核心概念

遷移的六種策略（6R）

AWS 提出的 6R 遷移策略框架已成為業界標準：

  1. Rehost（Lift-and-Shift）：將現有系統原封不動地搬遷到雲端。不修改應用程式架構，僅改變運行環境。最快速但無法充分利用雲端優勢。

  2. Replatform（Lift-Tinker-and-Shift）：在遷移過程中進行少量優化。例如將資料庫遷移到雲端託管服務（RDS），但不重寫應用程式邏輯。

  3. Refactor/Re-architect：重新設計應用程式以充分利用雲原生功能。通常涉及微服務化、容器化或無伺服器架構轉換。

  4. Repurchase：放棄自建系統，改用 SaaS 產品。例如將自建 CRM 遷移到 Salesforce。

  5. Retire：識別並淘汰不再需要的系統，減少遷移範圍與成本。

  6. Retain：暫時保留不適合遷移的系統，待未來條件成熟後再處理。

資料遷移的關鍵挑戰

  • 資料一致性：確保遷移過程中資料的完整性與一致性，特別是在持續運行的系統中。
  • 停機時間：業務連續性要求最小化服務中斷時間。
  • 資料量：TB 到 PB 級的資料遷移需要特殊的傳輸策略。
  • 格式轉換：不同平台之間的資料格式與結構差異需要轉換處理。
  • 驗證確認：遷移完成後必須驗證資料的正確性與完整性。

架構總覽

遷移流程架構

典型的資料遷移流程包含四個階段：

第一階段：評估與規劃

  • 盤點現有資料資產（資料庫、檔案系統、物件儲存）
  • 分析資料相依性與資料流
  • 評估各工作負載的遷移策略
  • 建立遷移優先順序與時程表

第二階段：設計與準備

  • 設計目標架構與資料模型
  • 建立遷移管線（ETL/ELT Pipeline）
  • 準備網路連線（專線或 VPN）
  • 建立驗證機制與回滾計畫

第三階段：執行遷移

  • 初始資料載入（Full Load）
  • 增量資料同步（Change Data Capture, CDC）
  • 應用程式切換（Cutover）
  • 即時監控與問題處理

第四階段：驗證與優化

  • 資料完整性驗證（行數、校驗和、業務規則）
  • 效能基準測試
  • 清理源端資料與資源
  • 文件化與知識轉移

資料同步模式

  1. 一次性遷移（Big Bang）：在維護窗口內完成所有資料遷移。簡單但需要較長停機時間。

  2. 漸進式遷移（Trickle Migration）：持續同步增量變更，最終在極短的切換窗口完成遷移。複雜但停機時間最短。

  3. 平行運行（Parallel Run）：新舊系統同時運行一段時間，驗證新系統正確性後再切換。最安全但成本最高。

應用場景

場景一：資料庫遷移至雲端託管服務

企業將地端的 Oracle 資料庫遷移到 AWS RDS for PostgreSQL。使用 AWS Schema Conversion Tool（SCT）轉換資料庫結構，再透過 AWS Database Migration Service（DMS）進行持續的資料複製。在驗證資料一致性後，於維護窗口完成應用程式切換。

場景二：跨雲資料倉儲遷移

將 AWS Redshift 中的資料倉儲遷移到 GCP BigQuery。由於兩者的資料模型與查詢語法有差異，需要進行 Schema 調整與 SQL 轉換。使用中間物件儲存（S3/GCS）作為資料暫存區，透過批次匯出與匯入完成遷移。

場景三：零停機微服務遷移

電商平台需要將訂單服務從單體架構遷移到微服務。採用 Strangler Fig Pattern，先建立新的微服務，透過 API Gateway 逐步將流量切換到新服務。使用 CDC 同步舊資料庫的變更到新服務的資料庫，確保遷移過程中業務不中斷。

場景四：大規模非結構化資料遷移

媒體公司需要遷移 500TB 的影音檔案從地端到雲端物件儲存。由於網路頻寬限制，使用實體傳輸設備（AWS Snowball Edge）進行初始載入，後續的增量檔案透過專線持續同步。

AWS 服務實作

AWS Snow Family（離線大量資料遷移）

適用於網路頻寬受限時，將本地大型資料遷移至 Amazon S3。

成員儲存空間記憶體傳輸方式支援 DataSync資料量等級
Snowcone8 TB4 GB線上與離線是GB～TB
Snowball Edge Storage Optimized80 TB80 GB離線否PB
Snowball Edge Compute Optimized42 TB208 GB離線否PB
Snowmobile100 PBN/A離線否EB

注意事項：

  • 需安裝 AWS OpsHub 軟體管理 Snow 裝置並從本機傳輸資料
  • Snow Family 無法直接遷移資料至 S3 Glacier；須先遷入 S3，再透過生命週期規則轉入 Glacier
  • 如需直接傳輸至 Glacier，請使用 AWS DataSync

AWS Storage Gateway（混合雲儲存整合）

連接本地端應用程式與雲端儲存資源的混合雲服務，長期持續運行。

Gateway 類型通訊協定後端儲存用途說明
File GatewayNFS & SMBS3（可轉 S3-IA / One Zone-IA）將本地檔案儲存為 S3 物件，支援本地快取與 AD 驗證
FSx File GatewaySMB & NTFSAmazon FSx整合 Microsoft AD 的 Windows/Lustre 檔案伺服器
Volume GatewayiSCSIS3（快照可轉 EBS）將本地磁碟區備份至 S3
Tape GatewayiSCSI VTLS3（可轉 Glacier）模擬傳統磁帶備份，取代實體磁帶庫

Volume Gateway 模式

模式說明
Cached Volume（快取模式）主要資料存於 S3，常用資料快取在本地以降低存取延遲
Stored Volume（儲存模式）主要資料存於本地，異步（Async）備份至 S3

AWS DataSync（大規模資料遷移）

主要功能

  • 用於大規模資料遷移與週期性資料同步
  • 支援來源：本地 NFS、SMB 伺服器
  • 支援目標：Amazon S3、Amazon EFS、Amazon FSx
  • 可直接將冷資料歸檔至 S3 Glacier 或 S3 Glacier Deep Archive
  • 支援所有 S3 儲存類別

與 Direct Connect 整合

  • 搭配 AWS Direct Connect 透過私有網路傳輸資料
  • 確保安全傳輸至 VPC 中的 AWS 儲存服務，避免走公網

DataSync vs Storage Gateway 比較

項目AWS DataSyncAWS Storage Gateway
主要用途一次性或週期性大量資料遷移持續的混合雲儲存整合
部署架構遷移完成後可移除長期持續運行
支援目標S3、EFS、FSxS3、FSx、EBS

AWS Database Migration Service（DMS）

核心功能

  • 支援不中斷運作下的資料庫遷移（來源資料庫在遷移過程中仍可持續使用）
  • 需在 EC2 實例上執行 DMS 複製引擎

遷移類型

類型說明範例
同構遷移（Homogeneous）相同資料庫引擎之間遷移PostgreSQL → PostgreSQL
異構遷移（Heterogeneous）不同資料庫引擎之間遷移Oracle → Aurora

常見遷移路徑

  • SQL Server → MySQL
  • Oracle → Amazon Aurora
  • Teradata → Amazon Redshift
  • MongoDB → Amazon DynamoDB
  • Amazon S3 → Amazon DynamoDB

AWS Schema Conversion Tool（SCT）

  • 進行異構遷移前需先使用 SCT
  • 在來源端分析並自動轉換資料庫 Schema 結構
  • 生成供 DMS 使用的轉換後結構定義

AWS Application Migration Service（MGN）

主要功能

  • 遷移虛擬機器至 AWS（支援 VMware vSphere、Hyper-V、Microsoft Azure）
  • 使用連續區塊層級複製（Continuous Block-Level Replication）
  • 可在分鐘內完成切換（Cutover）
  • 建議新案使用（取代舊版 SMS）

舊版 Server Migration Service（SMS）

  • 使用快照為主的複製方式
  • 切換時間約為數小時
  • 已不建議新案使用，請改用 MGN

6R 遷移策略（AWS 標準框架）

策略說明實施複雜度
Rehost（Lift & Shift）直接將系統搬遷至 AWS，不修改架構低
Replatform（Lift-Tinker-and-Shift）遷移時進行少量調整（如換用 RDS 代替自建 DB）低–中
Repurchase改用 SaaS 產品（如改用 Salesforce 取代自建 CRM）低
Refactor / Re-architect重新設計為雲原生架構（微服務、容器化、Serverless）高
Retire淘汰不再需要的系統，縮小遷移範圍—
Retain暫時保留不適合立即遷移的系統，待條件成熟再處理—

AWS 遷移三階段流程

第一階段：評估（Assess）

  • 盤點現有資源與應用程式相依性
  • 評估各工作負載的遷移適合性（6R 選型）
  • 規劃遷移優先順序與時程表

第二階段：遷移（Migrate）

  • 執行資料與應用程式遷移
  • 驗證資料完整性與應用功能
  • 功能測試與回歸測試

第三階段：優化（Optimize）

  • 調整資源規格（Right Sizing）
  • 實施成本優化措施（RI、Savings Plans）
  • 建立監控與告警機制
