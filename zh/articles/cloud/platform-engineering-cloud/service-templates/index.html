服務模板（Service Templates）是平台工程中將最佳實踐程式碼化的關鍵機制。透過預先定義好的模板，開發團隊能在數分鐘內建立一個符合組織標準的新服務，包含完整的程式碼結構、CI/CD 管線、基礎設施配置與監控設定。

核心概念

服務模板的組成

一個完整的服務模板通常包含以下組件：

組件說明範例
程式碼腳手架應用程式碼的初始結構main.go、handler/、model/
建構配置容器化與建構設定Dockerfile、Makefile
CI/CD 管線持續整合與部署定義.github/workflows/、Jenkinsfile
基礎設施定義IaC 模板Terraform modules、Helm chart
監控配置可觀測性設定Grafana dashboard、alert rules
文件模板標準化文件結構README、ADR、API docs
測試框架測試設定與範例unit test、integration test

模板 vs 腳手架 vs 樣板

概念用途生命週期可更新性
模板 (Template)初始建立 + 持續同步持續演進可推送更新
腳手架 (Scaffold)僅初始建立建立後獨立不追蹤上游
樣板 (Boilerplate)複製貼上一次性手動更新
原型 (Archetype)範例參考參考用N/A

# template-parameters.yaml

  - name: service_name
    type: string
    required: true
    validation: "^[a-z][a-z0-9-]{2,30}$"
  - name: language
    type: enum
    options: [go, python, java, typescript]
  - name: database
    type: enum
    options: [postgresql, mysql, mongodb, none]
  - name: messaging
    type: enum
    options: [kafka, rabbitmq, sqs, none]
  - name: tier
    type: enum
    options: [critical, standard, experimental]
    description: "影響 SLO 與資源配額"

關鍵比較

模板引擎比較

工具語言特色適用場景更新機制
Backstage TemplatesTypeScriptUI 整合、步驟式已用 Backstage無（scaffold）
CookiecutterPython/Jinja2簡單、廣泛使用通用無
YeomanJavaScript互動式、生態豐富Node.js 生態無
CopierPython/Jinja2支援更新推送需持續同步✓
ProjenTypeScript程式化管理需強管理✓

模板管理策略比較

策略優勢劣勢適用場景
單一倉庫統一管理、原子更新可能過大模板數量少
多倉庫獨立演進、權限隔離版本協調難大型組織
Monorepo + Package共享程式碼、統一工具需良好工具鏈中型組織

模板粒度比較

粒度範例優勢劣勢
全棧模板完整微服務 + 基礎設施一步到位靈活性低
組合式模板基礎模板 + 可選插件彈性高組合複雜度
層疊模板基礎層 + 框架層 + 業務層重用性高維護多層
片段模板單一功能片段最大靈活性組裝負擔

模板版本管理策略

策略適用場景對現有服務的影響
語義化版本所有模板明確的相容性承諾
主動推送更新安全修復、合規要求自動建立 PR
被動通知功能改進通知團隊可用更新
強制遷移廢棄舊版本設定遷移截止日期

模板治理

維護服務模板的最佳實踐：

  1. 擁有者明確：每個模板有明確的維護團隊
  2. 定期審查：季度審查模板是否仍符合最佳實踐
  3. 使用者回饋：建立回饋渠道，收集改進建議
  4. 自動化測試：模板本身有 CI 確保渲染正確
  5. 遷移工具：提供從舊版本升級到新版本的自動化工具
  6. 廢棄策略：明確的模板生命週期與廢棄流程
