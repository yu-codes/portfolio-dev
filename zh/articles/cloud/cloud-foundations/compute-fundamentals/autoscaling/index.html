自動擴展（Auto Scaling）是雲端運算的核心價值主張之一，讓應用程式能根據即時需求自動增減運算資源。正確的擴展策略不僅能確保效能，更能顯著優化成本。

核心概念

擴展方向

方向說明優點限制
水平擴展（Scale Out）增加實例數量理論上無上限、高可用需無狀態設計
垂直擴展（Scale Up）增加單一實例規格簡單、不需改架構有上限、需停機
混合擴展先垂直再水平兼顧兩者優勢策略較複雜

擴展觸發方式

類型說明反應速度適用場景
反應式（Reactive）指標超過閾值後觸發中（有延遲）一般工作負載
排程式（Scheduled）按時間表預設快（提前準備）可預測流量模式
預測式（Predictive）ML 預測未來需求最快規律性波動
手動（Manual）人工調整視反應時間已知事件

擴展策略類型

策略運作方式優點缺點
目標追蹤（Target Tracking）維持指標在目標值簡單直觀控制粒度低
步進擴展（Step Scaling）依閾值區間增減精確控制設定複雜

預測式擴展（Predictive）ML 預測需求提前準備需歷史資料

應用場景

電商流量高峰

  • 基線：10 個實例處理日常流量
  • 預測式：促銷活動前 2 小時開始擴展
  • 反應式：CPU > 70% 增加 2 個實例
  • 排程式：每日 9:00-22:00 最少 15 個實例
  • 縮減：流量恢復後逐步縮減（避免驟降）

API 服務彈性

  • Target Tracking：維持平均延遲 < 200ms
  • 最少 3 個實例（跨 AZ 高可用）
  • 最多 50 個實例（成本上限）
  • 健康檢查間隔 30 秒
  • 不健康實例自動替換

批次處理管線

  • 以佇列深度作為擴展指標
  • 工作完成後自動縮減至零
  • 使用 Spot 實例降低成本
  • 搭配 SQS/Pub/Sub 解耦合

AWS 服務實作

ASG（Auto Scaling Group）概述

  • 根據 CPU、網路或自訂指標或排程，自動調整 EC2 數量
  • 設定最小 / 最大 / 期望容量，確保應用彈性擴縮容
  • 使用 Launch Template（新版）或 Launch Configuration（舊版）建立實例

Launch Template vs Launch Configuration

Spot/On-Demand支援組合不支援

Scaling Policy（擴縮策略）

策略類型說明範例
Target Tracking維持指標在目標值平均 CPU 利用率在 40%
Simple Scaling單一條件觸發CPU >80% 加 2 台
Step Scaling依條件階梯調整CPU 60-80% 加 1 台，>80% 加 3 台
Scheduled Action排定時間調整每週五 5pm 加到 10 台

Target Tracking 常用指標：

  • ASGAverageCPUUtilization
  • ASGAverageNetworkIn
  • ASGAverageNetworkOut
  • ALBRequestCountPerTarget

Step Scaling 詳解

範圍動作
40-60% CPU+1 實例
60-80% CPU+2 實例
>80% CPU+4 實例
  • 多個步驟，更精細控制
  • 建議用於詳細控制（Simple Scaling 建議用 Step Scaling 取代）

Termination Policy（終止策略）

預設流程：

  1. 找出 EC2 最多的 AZ
  2. 刪除該 AZ 中使用最舊啟動模板的實例
  3. 若並列，刪除最接近下個計費時段的
策略說明
Default預設策略
OldestInstance最舊的實例
NewestInstance最新的實例
OldestLaunchConfiguration最舊啟動配置
OldestLaunchTemplate最舊啟動模板
ClosestToNextInstanceHour最接近下個計費時段
AllocationStrategy根據配置策略

Cooldown / Grace Period

設定說明
Cooldown調整後冷卻期，避免過度擴縮
Health Check Grace Period進入服務後的健康檢查緩衝期（預設 300 秒）

Lifecycle Hooks

可在擴增或縮減時加入自定邏輯：

擴增時（Launching）：

  • 安裝軟體
  • 配置應用程式
  • 註冊到服務

縮減時（Terminating）：

  • 備份資料
  • 上傳 log
  • 取消註冊

Lifecycle Hook 狀態：

狀態說明
Pending:Wait等待 Hook 完成
Pending:Proceed繼續啟動
Terminating:Wait等待 Hook 完成
Terminating:Proceed繼續終止

Lifecycle Hook 架構：

Pending → Pending:Wait → Pending:Proceed → InService
                ↓
         自訂動作（SNS/SQS/Lambda）

整合服務：

  • EventBridge：觸發 Lambda
  • SNS：發送通知
  • SQS：佇列處理

擴展 Hook 使用案例：

動作說明
安裝軟體安裝應用程式
提取配置從 S3 或 Parameter Store
註冊服務向服務發現註冊
準備資料預熱快取

終止 Hook 使用案例：

動作說明
備份上傳日誌到 S3
取消註冊從服務發現移除
通知發送通知
清理刪除暫存資料

類型說明

與 ELB 整合

架構：

流量 → ELB → Target Group ← ASG 自動註冊/取消註冊
  • ASG 可自動將新實例註冊到 Target Group
  • 不健康實例會被終止並替換
  • 支援多個 Target Group

CloudWatch 擴縮監控指標

指標說明
GroupMinSize最小容量
GroupMaxSize最大容量
GroupDesiredCapacity期望容量
GroupInServiceInstances服務中實例
GroupPendingInstances啟動中實例
GroupTerminatingInstances終止中實例

進階功能

Instance Refresh：

  • 滾動更新實例
  • 指定更新的百分比
  • 自動替換舊實例

Warm Pools：

  • 預熱實例池，減少擴展時間
  • 實例處於停止或運行狀態

Predictive Scaling：

  • 使用 ML 預測需求
  • 提前擴展
  • 適合可預測的流量模式

Scheduled Scaling 使用案例

使用案例說明
營業時間9am 擴展，6pm 縮減
週末減少容量
促銷提前擴展

ASG 最佳實踐

  1. 使用 Target Tracking：最簡單且有效的擴縮策略
  2. 設定適當的 Cooldown：避免不必要的擴縮
  3. 配置 Health Check Grace Period：給應用足夠啟動時間
  4. 使用 Lifecycle Hooks：確保安全的擴縮流程
  5. 跨多 AZ 部署：提高可用性
  6. 使用 Launch Template：更靈活的實例配置
