可觀測性

前端監控與追蹤

前端可觀測性（Frontend Observability）是現代 Web 應用品質保障的核心支柱。它不僅僅是監控系統是否正常運作，更是深入理解使用者在真實環境中的體驗、快速定位問題根因、並持續驅動產品改善的系統性方法論。隨著前端應用日趨複雜，單純依靠後端日誌已無法覆蓋使用者端的完整體驗鏈路，前端可觀測性因此成為工程團隊不可或缺的能力。

核心概念

前端可觀測性建立在三大支柱之上：日誌（Logs）、指標（Metrics） 與 追蹤（Traces）。

日誌 記錄離散事件，例如 JavaScript 錯誤、API 請求失敗、使用者操作等。前端日誌通常透過 console 收集或自訂 Logger 上報到集中式日誌平台。

指標 是可聚合的數值資料，例如頁面載入時間、首次內容繪製（FCP）、最大內容繪製（LCP）、累計版面位移（CLS）等。這些量化數據能幫助團隊追蹤效能趨勢與回歸。

追蹤 則描述一個請求或操作的完整生命週期。在前端場景中，分散式追蹤可以將使用者的一次點擊串聯到前端渲染、API 呼叫、後端處理等多個環節，實現端到端的問題診斷。

除了三大支柱，前端可觀測性還涵蓋 Real User Monitoring（RUM） 與 Synthetic Monitoring（合成監控） 兩大策略。RUM 捕捉真實使用者的瀏覽器資料，反映實際體驗；合成監控則透過自動化腳本模擬使用者行為，主動偵測異常。

架構總覽

一個完整的前端可觀測性架構通常包含以下層次：

  1. 資料採集層：透過 SDK 或 Agent 在瀏覽器中收集效能指標、錯誤、使用者行為等原始資料。常見工具包括 Sentry SDK、Datadog RUM、OpenTelemetry Browser SDK。
  2. 資料傳輸層：將收集的資料透過 Beacon API、Fetch 或 WebSocket 上報至後端收集器。需考量批次傳送、取樣策略與網路效率。
  3. 資料儲存與處理層：後端接收資料後進行清洗、聚合、索引。時序資料庫（如 InfluxDB、Prometheus）適合指標，Elasticsearch 適合日誌搜尋。
  4. 視覺化與告警層：透過 Grafana、Kibana 或廠商自有儀表板呈現資料，並設定告警規則在異常時即時通知。

資料採集時需特別注意 取樣策略。全量上報會產生巨大流量成本，合理的取樣率（如 10%–50%）配合動態取樣（對錯誤請求 100% 取樣）能在成本與可見性之間取得平衡。

關鍵比較

面向RUM（真實使用者監控）Synthetic（合成監控）
資料來源真實使用者瀏覽器自動化腳本模擬
覆蓋範圍所有使用者環境組合預定義場景與地理位置
問題發現被動，需有流量才可偵測主動，可在無流量時偵測
效能基線反映真實分布（P50/P95/P99）提供穩定基線用於回歸偵測
成本模型與流量成正比與測試頻率成正比

在工具選擇上，Sentry 專精於錯誤追蹤與 Source Map 還原；Datadog RUM 提供完整的使用者旅程重播；OpenTelemetry 則是開源標準，避免廠商綁定。小型團隊可從 Sentry 免費方案起步，大型企業則傾向 Datadog 或自建 OpenTelemetry Pipeline。

應用場景

場景一：線上錯誤快速定位。當使用者回報白屏問題時，透過錯誤追蹤平台可立即查看堆疊追蹤、受影響瀏覽器版本、發生頻率，搭配 Source Map 還原至原始碼位置，大幅縮短排查時間。

場景二：效能回歸偵測。每次部署後透過 RUM 指標觀察 LCP、FID 是否惡化。若 P95 載入時間從 2.5 秒升至 4 秒，告警觸發後可立即回滾或修復。

場景三：使用者行為分析。追蹤使用者在頁面上的點擊熱圖、捲動深度、功能使用率，為產品決策提供數據支持。結合 A/B 測試框架可量化功能上線的效果。

場景四：跨團隊協作診斷。當使用者反映「操作很慢」時，前端追蹤可串聯到後端 Span，確認瓶頸是在前端渲染、網路傳輸還是後端 API，避免跨團隊推諉。
