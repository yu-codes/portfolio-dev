前端監控

錯誤追蹤與效能監控

前端監控（Frontend Monitoring）是確保 Web 應用在真實使用者環境中穩定運作的關鍵實踐。它涵蓋效能監控（Performance Monitoring）與使用者分析（User Analytics）兩大面向，幫助團隊從技術指標與商業指標雙重維度理解應用品質。透過系統化的監控策略，團隊能在問題影響大量使用者之前主動發現並修復異常。

核心概念

前端監控的核心在於回答三個問題：應用是否正常？ 使用者體驗如何？ 哪裡需要改善？

效能監控 聚焦於量化載入速度與互動流暢度。關鍵指標包括：

  • First Contentful Paint（FCP）：首次有意義內容繪製的時間
  • Largest Contentful Paint（LCP）：最大內容元素繪製完成的時間，Google 建議控制在 2.5 秒以內
  • First Input Delay（FID）/ Interaction to Next Paint（INP）：使用者首次互動到瀏覽器回應的延遲
  • Cumulative Layout Shift（CLS）：頁面元素非預期位移的累計分數
  • Time to First Byte（TTFB）：伺服器回應首位元組的時間

使用者分析 則關注使用者行為模式：頁面瀏覽路徑、功能使用頻率、轉換率漏斗、跳出率等。這些數據直接驅動產品決策與 UX 優化。

錯誤監控 捕捉 JavaScript 運行時錯誤、未處理的 Promise 拒絕、資源載入失敗等異常。每個錯誤都需記錄堆疊追蹤、使用者環境（瀏覽器、OS、裝置）、重現路徑等上下文資訊。

架構總覽

前端監控系統的典型架構包含四個階段：

採集階段：在瀏覽器中注入輕量級 SDK，透過 Performance API、PerformanceObserver、Error Event 等原生介面收集資料。SDK 設計需考量對頁面效能的影響，避免阻塞主執行緒。

傳輸階段：使用 navigator.sendBeacon() 在頁面卸載時可靠傳送資料，或透過批次 Fetch 請求定期上報。傳輸層需實作重試機制、資料壓縮與取樣邏輯。

處理階段：後端收集器接收原始資料後，進行去重、聚合、關聯。Source Map 服務將混淆後的堆疊追蹤還原為可讀的原始碼位置。

呈現階段：儀表板展示即時指標趨勢、錯誤排行、使用者旅程重播。告警引擎根據預設閾值或異常偵測演算法觸發通知。

瀏覽器 SDK → Beacon/Fetch → 收集器 → 處理管線 → 時序DB/ES
                                                    ↓
                                              儀表板 + 告警

效能監控 vs 使用者分析的取捨：效能監控偏技術視角，告訴工程師「哪裡慢」；使用者分析偏商業視角，告訴產品經理「使用者在做什麼」。兩者應互補而非替代。

應用場景

場景一：Core Web Vitals 持續追蹤。透過 RUM 收集 CWV 指標，在 Grafana 中建立各頁面的 P75 趨勢圖。當某頁面 LCP 超過 2.5 秒時觸發 Slack 告警，工程師可立即定位是圖片過大、第三方腳本阻塞還是伺服器回應慢。

場景二：Session Replay 重現 Bug。使用者回報「按鈕點了沒反應」，透過 Session Replay 回放使用者操作錄影，發現是特定瀏覽器版本的 CSS 相容性問題導致按鈕被遮蔽。

場景三：轉換率漏斗分析。追蹤使用者從首頁 → 商品頁 → 加入購物車 → 結帳的完成率。發現結帳頁跳出率異常高後，進一步檢查該頁面的效能指標與錯誤日誌，定位到付款 SDK 載入超時。

場景四：部署品質關卡。將效能預算（Performance Budget）整合至 CI/CD，每次部署後自動比對 Bundle Size、LCP 等指標。若超出預算則阻止部署或標記為需關注。
