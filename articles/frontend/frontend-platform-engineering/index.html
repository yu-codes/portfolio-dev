前端平台工程關注的是如何為開發團隊建構高效的基礎建設：自動化的 CI/CD 流程、Monorepo 管理策略、以及全方位的開發者體驗（Developer Experience）優化。這些看似「非功能性」的投資，實際上直接決定了團隊的交付速度與程式碼品質。

核心概念

CI/CD for Frontend 不只是「跑測試然後部署」那麼簡單。前端的建構流程涉及程式碼轉譯、資產打包、環境變數注入、Preview 環境建立、效能預算檢查等多個步驟。一條成熟的前端 CI/CD Pipeline 能在每次 PR 時自動執行 lint、type check、unit test、e2e test、bundle size 分析、Lighthouse 評分，並產生可預覽的 Preview URL。

Monorepo 管理 解決的是多套件、多應用在同一程式碼庫中如何高效共存的問題。工具如 Turborepo、Nx、Lerna 提供了任務排程、快取機制、受影響範圍偵測等能力，讓建構時間從線性增長變為近乎常數。

開發者體驗（Developer Experience, DX） 涵蓋了從環境設定、程式碼撰寫、除錯、到部署的整個開發循環。好的 DX 投資包括：快速的 HMR（Hot Module Replacement）、完善的型別提示、自動化的程式碼格式化、一致的錯誤訊息、以及清晰的文件。

架構總覽

前端平台工程的完整架構包含以下系統：

版本控制與分支策略：

  • Trunk-based Development：所有開發者直接向主分支提交，搭配 Feature Flags 控制功能發布
  • GitHub Flow：短命的 Feature Branch + PR Review + 自動部署
  • Release Branch：適合需要維護多版本的產品

CI Pipeline 設計：

PR Created → Install → Lint + Type Check → Unit Test → Build → 
Bundle Analysis → E2E Test → Preview Deploy → Visual Regression

關鍵優化策略：

  • 平行化：獨立步驟同時執行（lint 與 test 平行）
  • 快取：利用 Turborepo Remote Cache 或 GitHub Actions Cache 避免重複建構
  • 增量式：只建構與測試受影響的套件（affected packages）

Monorepo 結構：

monorepo/
├── apps/
│   ├── web/          # 主應用
│   ├── admin/        # 管理後台
│   └── mobile/       # React Native
├── packages/
│   ├── ui/           # 共用元件庫
│   ├── utils/        # 工具函式
│   ├── config/       # 共用配置
│   └── types/        # 共用型別
├── tooling/
│   ├── eslint/       # ESLint 配置
│   ├── typescript/   # TSConfig 配置
│   └── tailwind/     # Tailwind 配置
└── turbo.json

應用場景

大型企業 Design System：Monorepo 中的 packages/ui 套件包含所有共用元件。搭配 Storybook 作為元件文件與視覺化測試工具，Chromatic 做視覺回歸測試。每次 PR 自動產生 Storybook Preview，設計師可直接在瀏覽器中驗證。

多產品線共享核心：當組織有多個前端應用需要共享業務邏輯（如認證模組、支付模組）時，Monorepo 讓共用套件的修改能立即被所有消費端驗證，避免跨 repo 的版本不一致問題。

效能監控與品質門檻：在 CI 中整合 Lighthouse CI，設定效能預算（如 LCP < 2.5s、Bundle Size < 200KB）。任何超過門檻的 PR 自動被擋下，確保效能不會隨功能迭代而退化。

開發環境標準化：透過 Dev Container、dotfiles、共用的 VSCode Settings，確保團隊成員的開發環境完全一致。新成員 onboarding 從「花兩天設環境」變為「clone 後一個指令啟動」。

Feature Flags 管理：平台工程團隊維護 Feature Flag 基礎建設，讓產品團隊能安全地進行漸進式發布、A/B 測試、緊急回滾，而不需要重新部署程式碼。
