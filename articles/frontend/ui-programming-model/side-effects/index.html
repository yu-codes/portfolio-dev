副作用管理

副作用管理是前端開發中最容易出錯、也最難以調試的領域。所謂「副作用」是指任何超出組件純渲染邏輯的操作——API 呼叫、DOM 操作、事件監聽、計時器、WebSocket 連線等。這些操作與外部世界互動，具有不可預測性，若未正確管理將導致記憶體洩漏、競態條件（Race Condition）與不一致的 UI 狀態。

核心概念

純函式 vs 副作用：理想的組件是純函式——給定相同的 props，永遠回傳相同的 UI。然而現實應用不可能完全沒有副作用。現代框架的設計哲學是：將副作用隔離到特定的區域（Effect Hooks、Lifecycle Methods），讓組件的核心渲染邏輯保持純粹。

生命週期鉤子（Lifecycle Hooks）：框架提供的鉤子函式讓開發者在組件的特定時刻執行副作用：

  • Mount（掛載）：組件首次渲染後，適合啟動資料獲取、建立訂閱
  • Update（更新）：依賴的資料變更時，適合同步外部系統、重新獲取資料
  • Unmount（卸載）：組件從 DOM 移除前，必須清理所有訂閱、計時器與事件監聽

Effect 清理（Cleanup）：每個副作用都應有對應的清理邏輯。React 的 useEffect 透過回傳清理函式實現；Vue 的 onUnmounted 提供卸載時的清理時機。忘記清理是前端記憶體洩漏的首要原因。

依賴追蹤：Effect 的重新執行取決於依賴陣列（React）或響應式系統的自動追蹤（Vue）。正確宣告依賴是避免「過時閉包」（Stale Closure）問題的關鍵。

應用場景

即時資料訂閱：股票報價、聊天訊息等即時資料透過 WebSocket 或 Server-Sent Events 推送。副作用管理需處理連線建立、斷線重連、訊息緩衝與組件卸載時的連線關閉。使用 RxJS 的 Observables 或自訂 Hook 可將複雜的訂閱邏輯封裝為可重用的模組。

無限滾動與分頁：監聽 IntersectionObserver 來偵測列表底部，觸發下一頁資料載入。副作用需管理 Observer 的建立與銷毀、防抖（Debounce）機制、以及載入狀態的正確轉換。若未清理 Observer，頁面切換後仍會持續觸發回呼。

動畫與過渡效果：CSS Transition 之外的複雜動畫（如 GSAP、Framer Motion）需要在 JavaScript 層面管理動畫實例的建立與銷毀。組件卸載時必須調用 animation.kill() 或 timeline.revert()，否則會產生 DOM 節點無法回收的問題。

表單自動儲存：在使用者編輯時定期將表單內容存入 localStorage 或發送到後端。需要管理 debounced save 函式的計時器、避免組件卸載後仍觸發儲存、以及處理並發請求的取消邏輯。

第三方 SDK 整合：地圖（Google Maps）、圖表（ECharts）、編輯器（Monaco）等第三方 SDK 通常需要手動管理初始化與銷毀。這些 SDK 會建立全域事件監聽、Canvas 繪圖上下文與內部快取，必須在組件卸載時完整清理，否則會累積記憶體直到頁面崩潰。
