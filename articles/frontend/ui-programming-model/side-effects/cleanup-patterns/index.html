清理模式（Cleanup Patterns）是前端副作用管理中至關重要的一環。每當組件建立訂閱、啟動計時器或發起異步操作時，都必須在適當時機進行清理，否則會導致記憶體洩漏、過期回呼和不可預期的行為。

核心概念

為什麼需要清理

不清理的後果說明影響
記憶體洩漏訂閱、監聽器持續占用記憶體應用越來越慢
過期更新已卸載組件仍嘗試更新狀態React 警告/錯誤
資源浪費計時器、連線持續運行效能降低、電量消耗
競態條件過期的異步回應覆蓋新資料顯示錯誤的資料
事件洩漏事件處理器累積同一事件觸發多次

架構總覽

React 清理流程

首次渲染：
  Effect 執行 → 建立訂閱/監聽

依賴變更：
  1. 執行上一次的清理函式
  2. 執行新的 Effect
  3. 建立新的訂閱/監聯

組件卸載：
  執行最後一次的清理函式
情境清理時機說明
依賴不變組件卸載時只清理一次
依賴變化每次依賴變化前 + 卸載重建訂閱
無依賴陣列每次渲染前 + 卸載頻繁清理重建

清理的時序問題

時間軸：
t1: Effect A 執行（依賴 = [x:1]）
t2: x 變為 2
t3: Effect A 清理（舊的）
t4: Effect A 執行（依賴 = [x:2]）
t5: 組件卸載
t6: Effect A 清理（最後的）

關鍵比較

AbortController vs 手動旗標

面向AbortController手動旗標 (isCancelled)
標準Web API 標準自訂
支援fetch, addEventListener任何異步操作
傳播signal 可傳遞給子操作需手動檢查
中斷機制拋出 AbortError需要 if 判斷
複雜度低低
網路請求取消真正取消 TCP 連線只忽略回應

各種清理模式比較

模式優點缺點適用場景
回傳清理函式React 標準、直覺閉包可能捕獲過期值React Effect
onCleanup 回呼明確的 API需要框架支援Vue/Solid
Disposable 物件統一介面、可組合需要額外封裝複雜資源

using 宣告自動清理（TC39 提案）瀏覽器支援有限未來標準

同步 vs 異步清理

面向同步清理異步清理
保證一定執行可能被中斷
時機立即可能延遲
範例removeEventListener關閉 DB 連線
框架支援所有框架有限（React 不支援 async cleanup）
錯誤處理try/catch需要特別處理

Disposable 模式

統一管理多個需要清理的資源：

class DisposableGroup {
  disposables = [];
  add(disposable) { this.disposables.push(disposable); }
  dispose() { this.disposables.forEach(d => d.dispose()); }
}
優勢說明
統一介面所有資源用 .dispose() 釋放
組合多個 disposable 組合為一個
防遺忘一次性清理所有資源
可測試驗證 dispose 被呼叫

常見錯誤與修正

錯誤問題修正
在清理中使用過期 state閉包捕獲了舊值使用 ref 保存最新值
async cleanupReact 不支援 async 清理同步設定 flag，清理中不 await
條件性 Hook違反 Hook 規則在 Effect 內條件判斷
重複監聽每次渲染都加新監聽正確設定依賴陣列
忘記 unsubscribeObservable 訂閱洩漏統一在清理中取消
