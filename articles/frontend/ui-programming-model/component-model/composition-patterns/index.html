組合模式（Composition Patterns）是前端 UI 架構中最核心的設計理念。相較於繼承，組合提供了更彈性、更可維護的方式來建構複雜的使用者介面。現代前端框架幾乎都以組合作為首要的程式碼重用策略。

核心概念

組合 vs 繼承

面向組合（Composition）繼承（Inheritance）
關係has-a（擁有）is-a（是一種）
耦合度低高
彈性執行期組合編譯期確定
重用粒度功能片段整個類別
現代框架優先採用幾乎不用
測試性容易隔離測試需要整個繼承鏈

組合的基本形式

形式說明框架實現
子組件（Children）透過插槽接收子內容React children, Vue slot
高階組件（HOC）函式接收組件回傳增強組件React HOC
Render Props透過函式 prop 控制渲染React render props
Hooks/Composables提取可重用的狀態邏輯React Hooks, Vue Composables
Mixins物件混入（已不推薦）Vue 2 Options API
裝飾器類別/方法增強Angular Decorators

組件組合層級

頁面（Page）
├── 版面（Layout）
│   ├── 容器組件（Container）
│   │   ├── 展示組件（Presentational）
│   │   └── 展示組件
│   └── 容器組件
└── 共用元素
    ├── 原子組件（Atoms）
    ├── 分子組件（Molecules）
    └── 有機體組件（Organisms）

架構總覽

常見組合模式

模式原理適用場景複雜度
Container/Presenter分離邏輯與展示資料驅動的 UI低
Compound Components隱式共享狀態的組件組下拉選單、Tab、表單中
Render Props透過函式控制渲染跨關注點邏輯共享中
Hook Composition函式組合狀態邏輯任何狀態邏輯重用低-中
Provider PatternContext 提供共享狀態主題、認證、國際化中
Slot Pattern內容投射彈性的佈局組件低
Controller Pattern分離控制邏輯複雜互動邏輯高

Compound Components 模式

讓多個組件隱式共享狀態，對外提供統一的 API：

<Select>
  <Select.Trigger>選擇...</Select.Trigger>
  <Select.Options>
    <Select.Option value="a">選項 A</Select.Option>
    <Select.Option value="b">選項 B</Select.Option>
  </Select.Options>
</Select>
優點缺點
API 直覺清晰實作較複雜
使用者有完全的排版控制需要 Context 或 cloneElement
組件關係明確僅限直接/間接子組件

Headless UI 模式

將行為邏輯與視覺呈現完全分離：

層級責任範例
邏輯層（Headless）狀態、鍵盤互動、ARIAuseCombobox, useDialog
呈現層（Styled）樣式、動畫、佈局自訂 CSS/Tailwind
整合層組合邏輯與呈現專案中的組件

關鍵比較

各組合模式的取捨

模式重用性可讀性型別安全效能
Children/Slots高高高好
Hooks/Composables最高高高好
HOC高中（包裝層多）中中
Render Props高中高中
Mixins中低（隱式依賴）低好
Compound Components中高中-高好

React Hooks vs Vue Composables vs Angular Services

面向React HooksVue ComposablesAngular Services
語法函式 + 規則函式類別 + DI
響應性useState/useReducerref/reactiveBehaviorSubject
生命週期useEffectonMounted 等ngOnInit 等
依賴注入Contextprovide/injectDI Container
狀態共享自訂 Hook + Context模組級 composableSingleton Service
組合方式呼叫其他 Hook呼叫其他 Composable注入其他 Service

跨關注點重用

關注點推薦模式原因
資料擷取Custom Hook/Composable最乾淨的邏輯重用
權限控制HOC 或 Provider宣告式包裝
動畫Render Props 或 Hook需要細粒度控制
表單驗證Hook + Context狀態邏輯複雜
主題切換Provider Pattern全域狀態
錯誤邊界HOC（React 限定）生命週期需求

反模式

反模式問題正確做法
Prop Drilling多層傳遞相同 propContext/Provider
巨型組件單一組件過度複雜拆分 + 組合
過度抽象過早/不必要的抽象等待重複出現再提取
隱式耦合組件間隱式依賴明確的 Props/API
