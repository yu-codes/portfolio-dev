全域狀態（Global State）是指整個應用中多個不相關組件需要共享存取的狀態。正確的全域狀態管理是大型前端應用可維護性的關鍵，但過度使用全域狀態也是常見的架構陷阱。

核心概念

全域狀態的特徵

特徵說明
跨組件共享不相關的組件樹都能存取
持久生命週期通常跟隨應用整個生命週期
單一真相來源任何修改都反映到所有訂閱者
需要同步狀態變更需要通知所有消費者

何時使用全域狀態

適合全域不適合全域
使用者認證/Session表單臨時輸入值
全域 UI 設定（主題、語言）元件展開/收合
購物車/收藏動畫狀態
全域通知/Toast單一頁面的篩選條件
權限/角色資料載入狀態（改用 Server State）
應用級快取可從 Props 推導的資料

狀態管理演進

世代代表模式年代
第一代Flux單向資料流 + Dispatcher2014
第二代Redux單一 Store + Reducer2015
第三代MobX可觀察物件 + 自動追蹤2016
第四代Zustand/Jotai/Recoil原子化/輕量2020+
第五代Signal (Angular/Solid)細粒度響應式2023+

架構總覽

主要架構模式

模式核心概念代表工具
Flux/Redux單一 Store + Action + ReducerRedux, Redux Toolkit
Proxy/Observable可觀察物件 + 自動追蹤MobX, Valtio
Atomic獨立的原子狀態單元Jotai, Recoil
輕量 Store簡化的 Pub/SubZustand, Pinia
Signal細粒度響應式原語Angular Signals, Solid
Context框架內建的依賴注入React Context, Vue provide/inject

Redux 架構

View (UI)
  │ dispatch(action)
  ▼
Action → Middleware → Reducer → Store → 通知訂閱者
                                  │
                                  ▼
                            Selector → View 更新
組件職責特性
Store儲存整個應用狀態單一、不可變
Action描述發生了什麼純物件 { type, payload }
Reducer計算新狀態純函式 (state, action) => newState
Selector衍生/選取狀態可記憶化
Middleware處理副作用Redux Thunk/Saga

Atomic 架構（Jotai/Recoil）

atom('count', 0)     ← 獨立的原子狀態
atom('doubled', get => get(count) * 2)  ← 衍生原子
                         │
Component A ─── 訂閱 count
Component B ─── 訂閱 doubled
Component C ─── 訂閱 count（獨立訂閱）

Context vs 專用狀態庫

面向React Context專用狀態庫（Zustand 等）
效能Context 變化會重渲染所有消費者精確的訂閱機制
適用規模小型應用/低頻更新中大型應用/高頻更新
額外依賴無需要安裝
開發者工具無專用工具通常有 DevTools
使用複雜度Provider 巢狀通常更簡潔
中介軟體/插件無豐富

不可變 vs 可變狀態更新

面向不可變（Redux/Zustand）可變（MobX/Valtio）
更新方式回傳新物件直接修改
變更偵測參照比較Proxy 攔截
除錯時間旅行、快照較難追蹤歷史
心智模型函數式物件導向
效能需注意不必要的新參照自動精確追蹤
程式碼量較多（展開運算子）較少（直覺修改）

狀態切分策略

策略方式好處
按領域切分user store, cart store, ui store職責清晰
按功能切分auth slice, notification slice模組化
按更新頻率高頻獨立、低頻合併效能最佳化
按生命週期session store, persistent store持久化策略清晰

狀態持久化

策略儲存位置適用資料
localStorage瀏覽器使用者偏好、主題
sessionStorage瀏覽器 Tab暫時性的表單草稿
IndexedDB瀏覽器大量結構化資料
Cookie瀏覽器/伺服器認證 Token
URL 參數網址可分享的狀態
伺服器後端需要跨裝置同步的資料

效能最佳化

技巧說明
精確訂閱只訂閱組件需要的狀態片段
Selector 記憶化使用 reselect 或 useMemo
狀態正規化避免深層巢狀
延遲初始化不立即載入所有狀態
避免頻繁更新debounce 高頻狀態變更
