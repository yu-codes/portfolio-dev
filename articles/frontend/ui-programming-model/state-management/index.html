狀態管理

前端狀態管理策略

狀態管理是前端開發中最核心也最具挑戰性的議題。應用程式的「狀態」代表 UI 在某個時間點的完整快照——使用者是否登入、購物車裡有哪些商品、表單目前填到哪裡、Modal 是否打開。隨著應用規模增長，狀態散落在數十甚至數百個組件中，如何有效地組織、更新與同步這些狀態，決定了程式碼的可維護性與團隊的開發效率。

核心概念

狀態的分類：

  • Local State（區域狀態）：只影響單一組件的狀態，如 input 的值、dropdown 是否展開、loading 指示器。使用框架內建的 useState（React）或 ref()（Vue）即可管理。
  • Shared State（共享狀態）：多個組件需要讀取或修改的狀態，如目前登入使用者、主題設定。需要透過 Context/Provide-Inject 或全域 Store 來共享。
  • Server State（伺服器狀態）：來自後端 API 的資料，具有快取、失效、重新獲取等獨特需求。React Query、SWR 等專用庫能大幅簡化管理。
  • URL State（路由狀態）：儲存在 URL 中的狀態（路徑、查詢參數），是使用者可以分享與書籤的狀態。

狀態提升（Lifting State Up）：當兩個兄弟組件需要共享狀態時，將狀態提升到它們的共同父組件中。這是最基礎的狀態共享策略，但在深層組件樹中會導致過多的 Props Drilling。

單一資料來源（Single Source of Truth）：每份資料應該只有一個「正式」的存放位置。衍生資料（Derived Data）應透過計算取得，而非另外存儲一份。違反此原則是狀態不一致 bug 的主要來源。

不可變更新（Immutable Updates）：狀態更新時建立新的物件/陣列參照，而非直接修改原物件。這讓框架能高效偵測變更（淺比較），也讓時間旅行調試與 undo/redo 成為可能。

信號說明

應用場景

使用者認證狀態：登入狀態是典型的全域共享狀態——Navbar 需要顯示使用者名稱、路由守衛需要判斷權限、API 攔截器需要附加 Token。適合使用全域 Store + 持久化中介軟體（localStorage/sessionStorage），確保頁面重新載入後不會遺失登入狀態。

多步驟表單 (Wizard)：每個步驟是獨立的組件，但表單資料需跨步驟保持。使用 useReducer 或 State Machine 管理步驟轉換邏輯（Next、Back、Skip），確保不合法的步驟轉換不會發生。State Machine 在此場景特別適合，因為它能明確定義每個狀態下允許的轉換。

即時通訊（Chat）：訊息列表、未讀計數、對話清單都是需要即時更新的共享狀態。Server State 工具（React Query）處理 API 層快取與失效，WebSocket 訂閱負責推送即時更新，兩者結合形成完整的資料同步策略。

購物車與結帳：購物車是經典的跨頁面全域狀態。商品可以從列表頁、詳情頁、推薦區加入，在購物車頁修改數量，最終在結帳頁使用。Redux/Pinia 搭配 localStorage 持久化是常見方案，同時需要與後端購物車 API 同步。

複雜 UI 互動：如拖拉排序、多選操作、右鍵選單等涉及多個 UI 元素協調的場景。State Machine 能清晰建模各種交互狀態（idle → hovering → dragging → dropping），避免出現「正在拖拉同時又觸發了右鍵選單」這類不合法的狀態組合。
