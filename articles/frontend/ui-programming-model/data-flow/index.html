資料流

資料流動模式

資料流（Data Flow）描述的是資料在應用程式的組件樹中如何傳遞、轉換與更新。選擇正確的資料流模式直接影響應用的可預測性、可調試性與開發效率。從早期 AngularJS 的雙向綁定引發的「變更偵測風暴」，到 React/Flux 確立的單向資料流哲學，前端社群在資料流設計上經歷了深刻的典範轉移。

核心概念

資料流方向性：資料流最根本的分類依據是方向性。單向資料流（Unidirectional）意味著資料只能沿著固定路徑傳遞——從父到子透過 Props，從子到父透過 Events。雙向資料流（Bidirectional）允許子組件直接修改父組件的資料，簡化了表單類互動但增加了追蹤難度。

不可變性（Immutability）：單向資料流通常搭配不可變資料結構。每次狀態變更產生新的物件參照，讓框架能透過淺比較（Shallow Comparison）快速判斷是否需要重新渲染，同時讓時間旅行調試（Time-travel Debugging）成為可能。

事件驅動更新：無論哪種資料流模式，狀態更新都是由事件觸發。使用者互動、API 回應、計時器——這些事件產生 Action，Action 經過處理後產生新狀態，新狀態再驅動 UI 更新。這個循環構成了所有前端應用的核心運作機制。

衍生狀態（Derived State）：並非所有 UI 需要的資料都應存為獨立狀態。部分資料可以從既有狀態計算而來（如篩選後的清單、合計金額）。正確識別衍生狀態能大幅降低狀態同步的負擔。

應用場景

電商購物車：購物車是單向資料流的經典場景。使用者的「加入購物車」操作觸發 Action，Reducer 計算新的購物車狀態（商品清單、數量、總價），View 根據新狀態重新渲染。因為狀態不可變，能輕鬆實現「復原」功能與操作記錄。

即時表單編輯：表單場景中雙向綁定大幅減少 boilerplate。v-model 在 Vue 中將 input 的值與組件狀態雙向同步，開發者只需關注驗證邏輯而非手動綁定 value + onChange。對於複雜表單，可搭配 Computed Properties 實現衍生驗證狀態。

協作編輯器：多人同時編輯文件需要 CRDT 或 OT 演算法處理衝突。資料流模式必須支援「來自本地的變更」與「來自遠端的變更」兩種來源，並確保最終一致性。這類場景通常需要獨立的同步層，不適合簡單的單向或雙向模式。

儀表板（Dashboard）：多個 Widget 共享資料來源（如同一份 API 回應），但各自有不同的視覺化邏輯。Flux 架構中的 Store 作為單一資料來源（Single Source of Truth），各 Widget 訂閱所需的 slice，避免重複請求與資料不一致。

遊戲 UI：遊戲狀態更新頻率極高（每幀 60 次），傳統的 Immutable 模式因大量物件建立會造成 GC 壓力。Signal-based 或 Mutable + Proxy 的資料流模式在此類場景更具效能優勢，能精確追蹤變更範圍、最小化重繪面積。
