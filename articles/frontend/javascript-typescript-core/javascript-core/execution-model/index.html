JavaScript 執行模型

深入解析 JavaScript 執行上下文、呼叫堆疊與執行階段的運作機制

JavaScript 是一門單執行緒語言，其執行模型決定了程式碼如何被解析、編譯與執行。理解執行模型是掌握 JavaScript 行為的基石，從變數提升到非同步處理，所有行為都源於這套精密的執行機制。

核心概念

執行上下文（Execution Context）

執行上下文是 JavaScript 引擎用來追蹤程式碼執行狀態的抽象概念。每當程式碼執行時，都會建立對應的執行上下文。

類型建立時機特性
全域執行上下文程式載入時整個程式只有一個，建立全域物件（window/global）
函式執行上下文函式被呼叫時每次呼叫都會建立新的上下文
Eval 執行上下文eval() 執行時不建議使用，有安全疑慮

每個執行上下文包含三個核心組件：

  • 變數環境（Variable Environment）：儲存 var 宣告與函式宣告
  • 詞彙環境（Lexical Environment）：儲存 let/const 宣告與區塊作用域
  • this 綁定（This Binding）：決定 this 的指向

呼叫堆疊（Call Stack）

呼叫堆疊是一種 LIFO（後進先出）的資料結構，用於管理執行上下文的生命週期。

| func_c() |  ← 堆疊頂部（目前執行中）
| func_b() |
| func_a() |
| 全域上下文 |  ← 堆疊底部

當函式被呼叫時，新的執行上下文被推入堆疊；當函式執行完畢後，該上下文從堆疊中彈出。

執行階段

JavaScript 引擎處理程式碼分為兩個主要階段：

階段動作影響
建立階段（Creation Phase）掃描宣告、建立作用域鏈、決定 this產生變數提升（Hoisting）
執行階段（Execution Phase）逐行執行程式碼、賦值運算實際運算與副作用發生

架構總覽

引擎處理流程

原始碼 → 詞法分析 → 語法分析 → AST → 位元碼 → 機器碼
         (Lexer)    (Parser)         (Interpreter) (JIT Compiler)

現代 JavaScript 引擎（如 V8）採用即時編譯（JIT）策略：

  1. Ignition（直譯器）：將 AST 轉換為位元碼快速執行
  2. TurboFan（最佳化編譯器）：將熱點程式碼編譯為高效機器碼
  3. 去最佳化（Deoptimization）：當假設失效時退回直譯器

變數提升機制

宣告方式提升行為初始值暫時性死區（TDZ）
var提升至函式頂部undefined無
let提升至區塊頂部未初始化有
const提升至區塊頂部未初始化有
function提升至函式頂部完整函式無

應用場景

避免堆疊溢出

遞迴呼叫若沒有正確的終止條件，會導致呼叫堆疊超出限制。解決方案包括：

  • 尾呼叫最佳化（TCO）：將遞迴放在函式最後一步（僅部分引擎支援）
  • 蹦床函式（Trampoline）：將遞迴轉換為迴圈
  • 迭代重寫：使用迴圈取代遞迴

效能最佳化考量

理解執行模型有助於撰寫對引擎友善的程式碼：

  • 保持函式參數型別一致，有利於 JIT 最佳化
  • 避免在熱路徑中使用 try/catch（舊版引擎無法最佳化）
  • 避免改變物件的「形狀」（Shape），影響隱藏類別（Hidden Class）

除錯技巧

  • 利用 DevTools 的 Call Stack 面板觀察執行上下文
  • 使用 debugger 語句設定中斷點
  • 透過 Performance 面板分析執行時間分佈
