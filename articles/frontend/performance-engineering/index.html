前端效能工程（Performance Engineering）是一門系統性的學科，旨在透過科學的方法論與工程實踐，確保 Web 應用在各種網路環境與裝置條件下都能提供快速、流暢的使用者體驗。它不只是「讓頁面載入快一點」的零散優化，而是從架構設計、建構流程、執行時期到監控回饋的全生命週期效能管理。

核心概念

前端效能工程圍繞四個核心維度展開：

載入效能（Loading Performance）：衡量使用者從發起請求到看見可互動頁面所需時間。關鍵策略包括減少關鍵路徑資源、Code Splitting、預載入（Preload/Prefetch）、圖片最佳化與 CDN 分發。

執行時期效能（Runtime Performance）：確保頁面在互動過程中維持 60fps 的流暢度。需避免 Long Task 阻塞主執行緒、減少不必要的 DOM 操作、善用 requestAnimationFrame 與 Web Worker。

網路效能（Network Performance）：最小化網路往返次數與傳輸量。HTTP/2 多工、資源壓縮（Brotli/Gzip）、Service Worker 快取策略、以及 103 Early Hints 都是關鍵手段。

感知效能（Perceived Performance）：即使實際載入時間不變，透過骨架屏（Skeleton Screen）、漸進式載入、樂觀更新（Optimistic Update）等手法，讓使用者「感覺」更快。

效能預算（Performance Budget）是效能工程的管理工具。團隊設定明確的量化目標（如 Bundle < 200KB、LCP < 2.5s），並在 CI/CD 中自動化檢查，防止效能退化。

架構總覽

效能工程的實踐貫穿開發生命週期：

設計階段 → 建構階段 → 部署階段 → 運行階段 → 監控階段
   │          │          │          │          │
效能預算   Tree Shaking  CDN/Edge   Runtime    RUM/CWV
架構決策   Code Split    壓縮策略   優化       告警回饋

設計階段：選擇合適的渲染策略（SSR/SSG/ISR）、定義效能預算、評估第三方依賴的體積成本。

建構階段：透過 Tree Shaking 移除未使用程式碼、Code Splitting 實現按需載入、圖片轉換為 WebP/AVIF 格式、CSS/JS 壓縮與最小化。

部署階段：配置 CDN 快取策略、啟用 Brotli 壓縮、設定適當的 Cache-Control 標頭、使用 Immutable Asset 模式確保長期快取。

運行階段：Lazy Loading 圖片與元件、虛擬化長列表、Debounce/Throttle 事件處理、Web Worker 卸載計算密集任務。

監控階段：透過 Core Web Vitals 持續追蹤效能趨勢，設定告警閾值，形成改善閉環。

關鍵比較

優化策略影響指標實施複雜度效益

Tree ShakingBundle Size低（建構工具內建）中

虛擬化列表Runtime FPS中高（長列表場景）

Web WorkerINP, Long Task高中（CPU 密集場景）

在方法論上，RAIL 模型（Response、Animation、Idle、Load）提供了以使用者為中心的效能框架：回應要在 100ms 內、動畫要維持 60fps、利用閒置時間預處理、載入要在 5 秒內完成。

應用場景

場景一：電商首頁極速載入。首屏使用 SSG + Edge Cache 實現近乎即時載入；折疊下方內容使用 Intersection Observer 觸發 Lazy Load；產品圖片使用 <picture> 元素搭配 WebP/AVIF 格式與響應式尺寸。

場景二：大型表單應用流暢互動。數千列資料的管理後台使用虛擬化列表（如 react-virtualized）僅渲染可視區域；表單驗證使用 Debounce 避免每次按鍵都觸發；複雜計算移至 Web Worker 避免阻塞 UI。

場景三：漸進式載入的 SPA。路由級 Code Splitting 確保首次載入僅包含當前頁面程式碼；<link rel="prefetch"> 預載入使用者可能訪問的下一頁；Service Worker 快取靜態資源實現離線可用。

場景四：第三方腳本管理。廣告、分析、聊天 Widget 等第三方腳本使用 async/defer 或動態插入避免阻塞首屏；設定效能預算監控第三方腳本體積；使用 Partytown 將第三方腳本移至 Web Worker。
