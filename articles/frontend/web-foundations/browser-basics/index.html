瀏覽器基礎

網路協定與請求生命週期

瀏覽器是 Web 應用的執行環境，理解它的內部運作機制是前端效能最佳化與問題診斷的基礎。從使用者在網址列輸入 URL 的那一刻起，瀏覽器便啟動一連串複雜的流程——DNS 解析、TCP 連線、TLS 握手、HTTP 請求、回應解析、渲染繪製。每一個環節都有最佳化的空間，也都可能成為效能瓶頸的來源。

核心概念

HTTP/HTTPS 協定：HTTP（HyperText Transfer Protocol）是 Web 的通訊基礎。HTTP/1.1 引入持久連線與管線化（Pipelining）；HTTP/2 帶來多路復用（Multiplexing）、標頭壓縮（HPACK）與伺服器推送（Server Push）；HTTP/3 基於 QUIC 協定，消除了 TCP 的隊頭阻塞（Head-of-Line Blocking）。HTTPS 在 HTTP 之上加入 TLS 加密層，確保傳輸過程中資料不被竊聽或竄改。

CORS（跨來源資源共享）：瀏覽器的同源政策（Same-Origin Policy）限制跨網域的資源存取。CORS 是一套 HTTP 標頭機制，讓伺服器明確宣告允許哪些來源（Origin）存取其資源。簡單請求直接發送；需要預檢的請求（如 PUT、DELETE 或自訂標頭）會先發送 OPTIONS 預檢請求（Preflight）確認伺服器是否允許。

URL 與請求生命週期：一個完整的 URL 包含 scheme、host、port、path、query、fragment。當瀏覽器解析 URL 後，依序進行 DNS 查詢（將域名解析為 IP）、建立 TCP 連線（三次握手）、TLS 協商（密鑰交換）、發送 HTTP 請求、接收回應、解析內容。

快取機制：瀏覽器的多層快取架構是效能的關鍵——Memory Cache（最快，Tab 關閉即消失）→ Disk Cache（持久化，依 Cache-Control 決定）→ Service Worker Cache（可程式化控制）→ Push Cache（HTTP/2 伺服器推送的暫存）→ Network（最慢，從伺服器獲取）。

協定版本連線模型多路復用標頭壓縮傳輸層

HTTP/2單一連線多路復用是（串流）HPACKTCP + TLS
HTTP/3QUIC 連線是（獨立串流）QPACKUDP + QUIC

標頭/機制用途行為

應用場景

CDN 快取策略設計：靜態資源（JS/CSS/Images）使用內容雜湊命名（app.a1b2c3.js）搭配 Cache-Control: max-age=31536000, immutable，實現永久快取。HTML 入口頁面使用 Cache-Control: no-cache，確保使用者總是獲取最新版本。Service Worker 提供離線降級方案，在網路不可用時返回快取的 App Shell。

API 跨域整合：前後端分離架構中，前端 https://app.example.com 需存取 https://api.example.com 的 API。後端需正確配置 CORS 標頭——Access-Control-Allow-Origin 指定允許的來源、Access-Control-Allow-Methods 列出允許的方法、Access-Control-Max-Age 設定預檢結果快取時間。避免使用 * 萬用字元搭配 credentials 以防安全風險。

首屏載入效能最佳化：理解請求生命週期才能有效縮短首屏時間。DNS Prefetch（<link rel="dns-prefetch">）提前解析第三方域名；Preconnect 提前建立 TCP + TLS 連線；Preload 提前載入關鍵資源（字型、Hero 圖片）；HTTP/2 Server Push 在 HTML 回應前就推送 CSS/JS。每一步都對應生命週期中的特定階段。

安全防護：HTTPS 防止中間人攻擊，HSTS（HTTP Strict Transport Security）強制瀏覽器使用 HTTPS。CSP（Content Security Policy）限制頁面可載入的資源來源，防止 XSS 攻擊注入惡意腳本。SRI（Subresource Integrity）驗證 CDN 資源的完整性，防止供應鏈攻擊。這些安全機制都建立在 HTTP 標頭之上。

效能監控與診斷：瀏覽器的 Performance API（Navigation Timing、Resource Timing）提供精確的時間測量——DNS 查詢花了多久、TLS 握手延遲多少、TTFB（Time to First Byte）是否正常。搭配 PerformanceObserver 可以即時蒐集 Web Vitals 數據，回報到監控平台進行告警與分析。
