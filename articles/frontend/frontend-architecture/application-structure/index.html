前端應用程式結構決定了程式碼的組織方式、團隊協作效率與長期可維護性。從小型專案的扁平結構到大型應用的領域驅動設計，選擇合適的結構模式是架構決策的第一步。好的結構應讓開發者直覺地找到程式碼、理解依賴關係並安全地進行修改。

核心概念

結構設計原則

原則說明實踐方式
就近原則相關程式碼放一起元件 + 樣式 + 測試同目錄
明確邊界模組間依賴可控barrel exports / index.ts
可發現性容易找到目標程式碼有意義的命名與目錄
最小知識元件只知道需要的封裝內部實作
漸進揭露複雜度隨深度增加頂層簡潔、深層詳細

模式適用規模特點

領域驅動大型domains/user/, domains/product/

架構總覽

按類型 vs 按功能

面向按類型按功能
結構components/, hooks/, utils/features/auth/, features/dashboard/
優勢簡單直覺高內聚低耦合
劣勢功能分散多處需定義功能邊界
適合<20 元件20-100+ 功能模組
修改影響需跨多目錄集中在單一目錄
新手入門容易需理解領域

Feature-Sliced Design (FSD)

層用途包含
app/應用初始化providers, routing, styles
processes/跨頁流程auth flow, checkout
pages/路由頁面組合 features/entities
widgets/獨立 UI 區塊Header, Sidebar
features/使用者互動AddToCart, LoginForm
entities/業務實體User, Product, Order
shared/共用基礎UI kit, API, config

Monorepo 結構

目錄用途工具
apps/web主要 Web 應用Next.js / Vite
apps/admin管理後台同上
packages/ui共用 UI 元件設計系統
packages/utils共用工具純函式
packages/config共用配置ESLint, TS
packages/types共用型別TypeScript

barrel export 策略

策略做法優點缺點
完整 barrel每目錄 index.ts 匯出所有匯入路徑短Tree-shaking 問題
選擇性 barrel僅匯出公開 API明確邊界需維護
無 barrel直接匯入檔案路徑無額外開銷路徑較長
混合共用模組用 barrel平衡需約定

應用場景

新專案起步

小型專案以按類型分組開始，隨功能增長逐步演進為按功能分組。避免過早設計複雜架構，但預留演進空間。建立基本的 shared/ 共用層與明確的匯入規則，為未來重構奠定基礎。

大型團隊協作

多團隊開發時，按功能/領域劃分讓各團隊擁有獨立的程式碼空間，減少合併衝突。搭配 CODEOWNERS 設定各功能目錄的負責人，結合 ESLint 規則限制跨模組匯入。

遺留專案重構

漸進式遷移策略：建立新的 features/ 結構，新功能直接在此開發；舊程式碼保留在 legacy/ 中，逐步搬遷。設定匯入限制規則，防止新模組依賴舊結構，確保遷移方向不可逆。
