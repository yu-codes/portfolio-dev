WebAssembly（WASM）是一種低階的二進位指令格式，設計為在瀏覽器中高效執行。它不是要取代 JavaScript，而是作為補充，讓效能關鍵的程式碼能以接近原生的速度執行，同時保持 Web 平台的安全性與可攜性。

核心概念

WebAssembly 設計目標

目標說明
高效能接近原生執行速度
安全性沙箱執行，記憶體隔離
可攜性跨平台、跨瀏覽器
開放標準W3C 標準
與 JS 共存互相呼叫，共享資源
語言無關C/C++/Rust/Go 等皆可編譯至 WASM

核心架構

組件功能說明
Module編譯後的程式碼類似 .dll/.so，可快取
Instance模組的執行實例包含記憶體、表格、全域
Memory線性記憶體ArrayBuffer，可成長
Table函式參照表間接呼叫用
Global全域變數可匯入/匯出

型別系統

數值型別說明對應 JS
i3232 位元整數Number
i6464 位元整數BigInt
f3232 位元浮點數Number
f6464 位元浮點數Number
v128128 位元 SIMD—
funcref函式參照Function
externref外部參照任意 JS 值

    │
    ▼ 編譯器（Emscripten/wasm-pack/TinyGo）
    │
.wasm 二進位檔（+ 膠水 JS）
    │
    ▼ 瀏覽器載入
    │
┌──────────────────────────┐
│  驗證 → 編譯 → 實例化     │
│  (Validate → Compile →   │

└──────────────────────────┘
    │
    ▼
  執行（接近原生速度）

JavaScript 與 WASM 互操作

方向方式說明
JS → WASM呼叫匯出函式instance.exports.add(1, 2)
WASM → JS匯入函式在 importObject 中提供
共享記憶體Memory 物件透過 ArrayBuffer 讀寫
字串傳遞手動編碼/解碼記憶體中的 UTF-8 位元組
複雜物件序列化JSON 或自訂格式

載入方式比較

方式流程效能適用
WebAssembly.instantiate(buffer)下載 → 實例化基本小模組
WebAssembly.instantiateStreaming(fetch)串流編譯最佳推薦方式
WebAssembly.compile + cache預編譯 + IndexedDB 快取重複載入快大模組

關鍵比較

WASM vs JavaScript 效能特性

面向WebAssemblyJavaScript
解析速度快（二進位格式）慢（文字解析）
編譯AOT/串流編譯JIT（需要暖機）
執行速度穩定、接近原生熱點程式碼接近原生，冷路徑較慢
記憶體管理手動/語言 GC自動 GC
型別靜態強型別動態型別（JIT 推測）
啟動時間較快（預編譯）較慢（解析+編譯）
除錯較困難DevTools 完整支援
體積二進位壓縮小文字較大（但 gzip 後也小）

各語言編譯至 WASM 的比較

語言工具鏈輸出大小GC適用場景
C/C++Emscripten中等無（手動）移植既有程式碼
Rustwasm-pack/wasm-bindgen小無（所有權）新專案首選
GoTinyGo較大有（內建 GC）Go 開發者
AssemblyScriptasc小有（選用）TypeScript 開發者
C#Blazor/.NET大有（.NET GC）.NET 生態
Zig內建極小無極致最佳化

WASM 適用 vs 不適用的場景

適用不適用
CPU 密集運算（編碼/解碼）DOM 操作
影像/音訊/視訊處理簡單的 CRUD 應用
遊戲引擎輕量的 UI 互動
加密/雜湊運算主要是 I/O bound 的任務
科學計算/模擬小型的腳本邏輯
移植既有 C/C++ 庫需要頻繁 JS 互調的場景
