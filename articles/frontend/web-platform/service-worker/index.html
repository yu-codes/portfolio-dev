Service Worker 是瀏覽器在背景執行的腳本，獨立於網頁主執行緒。它作為網路代理（Network Proxy），能夠攔截和處理網路請求、管理快取，是建構 Progressive Web App（PWA）和離線應用的核心技術。

核心概念

Service Worker 特性

特性說明影響
獨立執行緒不在主執行緒上執行無法直接存取 DOM
事件驅動閒置時被終止，事件觸發時喚醒不可依賴全域狀態
HTTPS 限定僅在安全連線下可用localhost 除外（開發用）
可程式化代理攔截所有同源網路請求完全控制回應策略
非同步設計所有 API 皆為 Promise不可使用同步 API（如 localStorage）

生命週期

階段事件說明
註冊—瀏覽器下載並解析 SW 腳本
安裝install預快取資源的最佳時機
等待—等待舊版 SW 控制的頁面全部關閉
啟動activate清理舊快取的最佳時機
運行fetch, push, sync攔截請求、推送通知、背景同步
終止—閒置一段時間後被瀏覽器終止

作用域（Scope）

註冊路徑預設作用域控制範圍
/sw.js/整個網站
/app/sw.js/app/僅 /app/ 底下
/sw.js + { scope: '/blog/' }/blog/明確指定

架構總覽

快取策略

策略說明適用場景離線支援
Cache First優先使用快取，未命中才走網路靜態資源（CSS/JS/圖片）是
Network First優先走網路，失敗才用快取API 資料、動態內容是（降級）
Stale While Revalidate先回傳快取，同時更新頻繁更新但可容忍舊資料是
Network Only只走網路不可快取的操作（POST）否
Cache Only只用快取預快取的靜態資源是

快取策略選擇指南

資源類型推薦策略原因
App Shell（HTML 框架）Cache First + 背景更新快速載入基本結構
CSS/JS BundleCache First（版本化）不可變的靜態資源
API 資料Network First / SWR需要最新資料
用戶頭像Stale While Revalidate偶爾更新
分析/追蹤Network Only必須即時
字型檔案Cache First很少變更

更新機制比較

面向Service Worker 更新傳統快取（HTTP Cache）
控制權開發者完全控制依賴 Cache-Control 標頭
更新檢測逐位元組比對 SW 檔案Expires/ETag 標頭
原子性整個 SW 一次更新各檔案獨立
使用者介入可實作提示更新無法提示
離線存取完整的離線存取控制非常有限

應用場景

離線優先架構

設計離線優先的應用：

  1. App Shell 架構：快取基本 UI 框架
  2. 動態內容快取：API 回應存入 Cache Storage
  3. 離線佇列：使用 Background Sync 排隊操作
  4. 衝突解決：重新上線時處理資料衝突
