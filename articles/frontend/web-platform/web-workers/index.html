Web Workers 讓 JavaScript 能夠在背景執行緒中執行程式碼，不阻塞主執行緒的使用者介面操作。這是瀏覽器中實現真正平行運算的機制，適用於 CPU 密集的任務處理。

核心概念

Worker 類型

類型特性生命週期通訊範圍
Dedicated Worker專屬單一頁面隨頁面關閉建立它的頁面
Shared Worker多頁面共享所有連線頁面關閉時同源的所有頁面
Service Worker獨立於頁面瀏覽器管理作用域內的所有頁面

Dedicated Worker 基本特性

特性說明
獨立執行緒不與主執行緒共享記憶體（預設）
無 DOM 存取無法操作 document、window
獨立全域self 代替 window，有 WorkerGlobalScope
非同步通訊透過 postMessage 交換資料
可用 APIfetch、IndexedDB、WebSocket、定時器等
不可用 APIDOM、localStorage、alert、confirm

資料傳輸方式

方式機制效能適用場景
結構化複製（Structured Clone）深拷貝中（複製成本）一般資料傳遞
可轉移物件（Transferable）所有權轉移高（零拷貝）ArrayBuffer、ImageBitmap
SharedArrayBuffer共享記憶體最高需要共享大量資料

關鍵比較

Web Worker vs 主執行緒非同步

面向Web Worker主執行緒非同步（async/await）
執行方式真正的平行（多執行緒）併發（單執行緒切換）
CPU 密集不阻塞 UI仍會阻塞 UI
資料交換需要序列化直接共享記憶體
DOM 存取無法可以
複雜度較高較低
啟動成本有（建立執行緒）無
適用場景重計算、大資料處理I/O 等待、輕量操作

Dedicated vs Shared vs Service Worker

面向DedicatedSharedService
頁面關聯一對一多對一與頁面解耦
通訊postMessageMessagePortpostMessage/Clients
存活時間頁面存在時有連線時瀏覽器管理
網路攔截否否是
離線功能否否是
瀏覽器支援優秀中等優秀

Transferable vs SharedArrayBuffer

面向TransferableSharedArrayBuffer
機制所有權轉移共享記憶體
原始資料存取轉移後原持有者無法存取雙方皆可存取
效能O(1) 轉移無需轉移
同步問題無（所有權獨佔）需要 Atomics 同步
安全要求一般需要 COOP/COEP 標頭
使用複雜度低高

應用場景

適合 Worker 的任務

任務類型範例效益
影像處理濾鏡、縮放、格式轉換不阻塞 UI 互動
資料解析大型 JSON/CSV 解析避免長時間阻塞
加密運算雜湊計算、加解密CPU 密集不凍結畫面
搜尋/排序大資料集的複雜查詢即時回應使用者輸入
物理模擬遊戲物理引擎穩定的幀率
編譯/轉換WASM 編譯、程式碼轉換背景處理

Comlink 簡化通訊

Comlink 將 Worker 通訊包裝為類似 RPC 的介面：

傳統方式Comlink 方式
手動 postMessage直接呼叫函式
手動處理事件await 回傳值
手動序列化自動處理
難以維護清晰的 API 邊界

效能考量

考量建議
Worker 數量不超過 navigator.hardwareConcurrency
啟動成本使用 Worker Pool 復用
通訊開銷批次傳輸、使用 Transferable
記憶體每個 Worker 有獨立的記憶體空間
Module Worker使用 type: 'module' 支援 ESM

最佳實踐

  • 避免為簡單操作建立 Worker（啟動成本 > 計算成本）
  • 使用 Worker Pool 管理多個任務
  • 大型資料傳輸使用 Transferable Objects
  • 為 Worker 設定逾時機制防止假死
  • 使用 terminate() 釋放不再需要的 Worker
