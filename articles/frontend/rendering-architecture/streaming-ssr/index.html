Streaming SSR

深入解析 Streaming SSR 的運作機制、效能優勢與實作模式，探索 React 18 串流渲染與 Suspense 的整合應用

Streaming SSR（串流式伺服器渲染）將 HTML 回應拆分為多個區塊（chunks），伺服器一邊渲染一邊傳送，瀏覽器收到第一個區塊即開始解析與顯示。這消除了傳統 SSR 需等待整個頁面渲染完成才回應的瓶頸，顯著改善 TTFB 與 FCP。

核心概念

傳統 SSR vs Streaming SSR

面向傳統 SSRStreaming SSR
回應方式整個 HTML 一次送出分區塊漸進送出
TTFB等最慢的資料立即開始
FCP等整頁完成第一個區塊到達
記憶體使用需暫存完整 HTML串流處理
慢查詢影響阻塞整個回應僅延遲對應區塊
HTTP 協定Content-LengthTransfer-Encoding: chunked

架構總覽

串流渲染流程

步驟動作使用者看到
1伺服器開始渲染，送出 <head> 和頁面骨架空白 → 骨架
2非同步資料的 Suspense fallback 串流送出Loading 狀態
3第一批資料就緒，對應 HTML 區塊送出部分內容顯示
4剩餘資料陸續就緒並送出漸進填充
5所有區塊完成，串流結束完整頁面
6JS 載入，Selective Hydration 啟動可互動

Out-of-order Streaming

傳統串流必須按 HTML 順序送出，但 React 18 支援亂序串流：

機制說明
佔位模板Suspense fallback 作為佔位
隱藏內容完成的內容以 <template> 傳送
客戶端腳本小型 inline script 執行 DOM 替換
無需全頁 JS替換邏輯極小（~1KB）

應用場景

資料密集型頁面

電商產品頁需載入產品資訊、評論、推薦商品等多個資料來源。串流 SSR 讓產品基本資訊先到達使用者眼前（最快的 API），評論和推薦區塊隨後漸進出現。使用者無需等待所有資料就能開始閱讀。

儀表板首頁

管理儀表板包含多個獨立面板，各需不同 API 回應。串流渲染每個面板為獨立 Suspense 區塊，最快的面板先顯示，慢的面板顯示 Skeleton 等待。使用者立即看到部分有效資訊。

搜尋結果頁

搜尋引擎結果頁的頂部結果最為重要。串流 SSR 可先送出前幾筆結果，使用者開始瀏覽的同時，後續結果持續串流到達，實現「即搜即看」的體驗。
