前端安全

XSS、CSRF 防護與安全儲存

Table of Contents

前端安全（Frontend Security）是 Web 應用安全防線的第一道關卡。隨著前端應用承載越來越多的業務邏輯與敏感資料，攻擊面也隨之擴大。從經典的 XSS、CSRF 攻擊到現代的供應鏈攻擊、第三方腳本劫持，前端工程師必須具備系統性的安全思維，在開發流程的每個環節融入防禦措施。

核心概念

前端安全的威脅模型主要涵蓋以下攻擊向量：

Cross-Site Scripting（XSS）：攻擊者將惡意腳本注入到受信任的網頁中，在其他使用者的瀏覽器中執行。分為三種類型：

  • Stored XSS：惡意腳本被永久儲存在伺服器（如評論區），每次載入頁面都會執行
  • Reflected XSS：惡意腳本透過 URL 參數反射回頁面
  • DOM-based XSS：完全在客戶端發生，JavaScript 將不可信的資料寫入 DOM

Cross-Site Request Forgery（CSRF）：攻擊者誘使已登入使用者的瀏覽器發送偽造請求，利用瀏覽器自動攜帶 Cookie 的特性執行未授權操作。

Content Security Policy（CSP）：透過 HTTP 標頭限制頁面可載入資源的來源，有效防止 XSS 攻擊的執行階段。即使攻擊者成功注入腳本，CSP 可阻止其執行。

安全儲存：前端如何安全地處理 Token、使用者資料等敏感資訊。涉及 Cookie 屬性設定（HttpOnly、Secure、SameSite）、LocalStorage/SessionStorage 的風險評估、以及加密策略。

供應鏈安全：第三方 npm 套件與 CDN 腳本可能被植入惡意程式碼。Subresource Integrity（SRI）與 Lock File 是主要防禦手段。

架構總覽

多層防禦（Defense in Depth）是前端安全的基本原則：

第一層：輸入驗證與消毒（Input Validation & Sanitization）
第二層：輸出編碼（Output Encoding）
第三層：Content Security Policy（CSP）
第四層：安全標頭（Security Headers）
第五層：安全儲存與傳輸（Secure Storage & Transport）
第六層：供應鏈控制（Supply Chain Security）

輸入驗證 在資料進入系統時過濾惡意內容。使用白名單而非黑名單，僅允許預期格式的資料通過。

輸出編碼 在資料渲染至頁面時進行適當轉義。HTML Context 使用 HTML Entity 編碼、JavaScript Context 使用 JS 編碼、URL Context 使用 URL 編碼。現代框架（React、Vue）預設對插值進行 HTML 轉義，但使用 dangerouslySetInnerHTML 或 v-html 時仍需謹慎。

安全標頭 提供瀏覽器級別的防護：

  • Strict-Transport-Security：強制 HTTPS
  • X-Content-Type-Options: nosniff：防止 MIME 嗅探
  • X-Frame-Options：防止 Clickjacking
  • Referrer-Policy：控制 Referer 資訊洩漏

關鍵比較

防禦措施對抗威脅實施位置複雜度
輸出編碼XSS前端/後端低
CSPXSS 執行HTTP 標頭高
SameSite CookieCSRF瀏覽器/後端低
CSRF TokenCSRF前後端協作中
SRI供應鏈攻擊HTML 標籤低
HttpOnly CookieXSS 竊取 Token後端設定低

Token 儲存策略比較：

  • HttpOnly Cookie：安全性最高（JS 無法存取），但需處理 CSRF
  • LocalStorage：方便存取，但 XSS 可竊取
  • Memory（變數）：最安全但頁面重載會遺失
  • 推薦：Access Token 存 Memory，Refresh Token 存 HttpOnly Cookie

應用場景

場景一：富文本編輯器安全。使用者提交的富文本內容需經過 HTML Sanitizer（如 DOMPurify）過濾，僅保留白名單標籤與屬性。搭配 CSP 禁止 inline script，即使過濾遺漏也能阻止腳本執行。

場景二：OAuth Token 管理。Access Token 使用短效期（15 分鐘）存放在 JavaScript 變數中；Refresh Token 設定為 HttpOnly、Secure、SameSite=Strict 的 Cookie。Token 刷新透過專用 endpoint，並實作 Token Rotation 防止重放攻擊。

場景三：CSP 漸進式部署。先以 Content-Security-Policy-Report-Only 模式收集違規報告，了解現有腳本的載入模式。逐步收緊策略，從 script-src 'self' 開始，必要時使用 Nonce 或 Hash 允許特定 inline script。

場景四：第三方腳本隔離。廣告與分析腳本使用 <iframe sandbox> 隔離，限制其存取主頁面 DOM 與 Cookie。或使用 Partytown 將第三方腳本移至 Web Worker，完全阻斷其對主執行緒的存取。
