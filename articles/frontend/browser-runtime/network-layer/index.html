網路層

網路層（Network Layer）是瀏覽器執行環境中負責處理所有資料傳輸的核心子系統。從頁面載入的第一個 HTML 文件，到動態取得的 API 資料、即時通訊的 WebSocket 連線，乃至靜態資源的快取管理，網路層貫穿了整個前端應用的生命週期。深入理解瀏覽器網路層的運作機制，是打造高效能、低延遲 Web 應用程式的基石。

現代前端開發已不再只是簡單地發送 HTTP 請求與接收回應。隨著 Web 應用複雜度的提升，開發者需要掌握多種網路通訊方式、理解瀏覽器的快取策略、管理並行連線數限制，以及處理各種網路異常情境。本文將全面梳理瀏覽器網路層的核心組成，為深入各子主題建立完整的知識框架。

核心概念

瀏覽器網路請求的生命週期

當一個網路請求從 JavaScript 發出時，它會經歷以下關鍵階段：

  1. DNS 解析：將網域名稱轉換為 IP 位址，瀏覽器會先查詢本地快取、作業系統快取，再向 DNS 伺服器查詢
  2. TCP 連線建立：透過三次握手建立可靠的傳輸連線（HTTPS 還需額外的 TLS 握手）
  3. HTTP 請求發送：組裝請求標頭與請求體，透過已建立的連線發送
  4. 伺服器處理：等待伺服器處理請求並回傳回應
  5. 回應接收：逐步接收回應標頭與回應體，觸發相應事件
  6. 連線管理：根據 Connection 標頭決定是否保持連線（HTTP/1.1 Keep-Alive）

瀏覽器並行連線限制

瀏覽器對同一網域的並行連線數有嚴格限制（通常為 6 個，HTTP/2 除外）。這個限制影響了資源載入策略：

  • HTTP/1.1：每個連線同時只能處理一個請求，因此並行數直接決定了同時載入的資源數量
  • HTTP/2：透過多路復用（Multiplexing），一個連線可同時處理多個請求，大幅提升效率
  • HTTP/3：基於 QUIC 協議，消除了隊頭阻塞問題，進一步優化傳輸效能

同源政策與跨域請求

瀏覽器的同源政策（Same-Origin Policy）是網路安全的基礎防線。不同源的請求需要透過 CORS（Cross-Origin Resource Sharing）機制來授權：

// 簡單請求：GET/POST/HEAD，且符合特定條件時不需要預檢
fetch('https://api.example.com/data', {
  method: 'GET',
  headers: { 'Accept': 'application/json' }
});

// 預檢請求：瀏覽器會先發送 OPTIONS 請求確認伺服器允許
fetch('https://api.example.com/data', {
  method: 'PUT',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ key: 'value' })
});

架構總覽

瀏覽器網路層的架構可分為四大核心模組：

1. Fetch API — 現代非同步請求介面

Fetch API 是當前推薦的網路請求方式，基於 Promise 設計，提供簡潔而強大的 API。它支援串流處理（Streaming）、請求中止（AbortController）、以及與 Service Worker 的深度整合。Fetch 的設計哲學是低階且可組合的，開發者可以在此基礎上建構更高階的抽象。

2. XMLHttpRequest — 傳統 AJAX 機制

XMLHttpRequest（XHR）是 AJAX 技術的基礎，雖然已被 Fetch 取代作為首選，但在某些場景仍有其不可替代性，例如上傳進度監控（upload.onprogress）和同步請求的需求。許多舊有的函式庫和框架仍基於 XHR 實作。

3. WebSocket — 全雙工即時通訊

WebSocket 提供了持久化的雙向通訊管道，適合需要即時更新的場景：聊天室、即時通知、協作編輯、股票行情推送等。與 HTTP 的請求-回應模式不同，WebSocket 允許伺服器主動推送資料。

4. HTTP 快取 — 效能優化的第一防線

瀏覽器的 HTTP 快取機制透過 Cache-Control、ETag、Last-Modified 等標頭，避免重複下載未變更的資源。合理的快取策略可以大幅減少網路請求，顯著提升頁面載入速度。

選擇指南

  • 一般 API 請求：優先使用 Fetch API，語法簡潔、支援現代特性
  • 需要上傳進度：使用 XHR 的 upload.onprogress 事件
  • 即時雙向通訊：使用 WebSocket 建立持久連線
  • 需支援舊瀏覽器：XHR 相容性最佳，或使用 polyfill

應用場景

單頁應用（SPA）資料載入

現代 SPA 大量依賴網路層來動態取得資料。常見模式包括：

  • 懶載入（Lazy Loading）：路由切換時才載入對應的資料與元件
  • 預取（Prefetching）：預測使用者行為，提前載入可能需要的資料
  • 背景同步：利用 Service Worker 在背景進行資料同步
  • 樂觀更新（Optimistic Update）：先更新 UI，再等待伺服器確認

即時應用

  • 聊天應用：WebSocket 配合訊息佇列實現即時訊息推送
  • 協作工具：透過 WebSocket 同步多人編輯的操作轉換（OT/CRDT）
  • 即時儀表板：Server-Sent Events 或 WebSocket 推送即時指標數據

離線優先（Offline-First）架構

結合 Service Worker 與 Cache API，實現：

  • 離線可用：快取關鍵資源，斷線時仍可使用基本功能
  • 背景同步：恢復連線後自動同步離線期間的操作
  • Stale-While-Revalidate：先回傳快取資料，同時背景更新

效能優化策略

  • 請求合併（Request Batching）：將多個小請求合併為一次批量請求
  • 請求去重（Deduplication）：相同請求在短時間內只發送一次
  • 優先級排程：重要請求優先發送，非關鍵請求延遲處理
  • 連線預熱：使用 dns-prefetch、preconnect 提前建立連線
