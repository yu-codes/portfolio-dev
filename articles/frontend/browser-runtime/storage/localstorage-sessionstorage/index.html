LocalStorage 與 SessionStorage

Web Storage API 提供了 LocalStorage 和 SessionStorage 兩種客戶端儲存機制，讓網頁應用能夠在瀏覽器中以鍵值對的方式儲存資料。相較於傳統的 Cookies，Web Storage 提供更大的儲存空間和更簡潔的 API，是現代前端開發中不可或缺的工具。

核心概念

Web Storage API

Web Storage 定義了兩個全域物件：

  • window.localStorage：資料持久保存，除非手動清除
  • window.sessionStorage：資料僅在當前瀏覽器分頁的工作階段中有效

兩者共享相同的 API 介面：

// 儲存資料
storage.setItem('key', 'value');

// 讀取資料
const value = storage.getItem('key');

// 移除資料
storage.removeItem('key');

// 清除所有資料
storage.clear();

// 取得項目數量
const count = storage.length;

// 根據索引取得鍵名
const key = storage.key(index);

儲存限制

Web Storage 具有以下限制：

  • 容量限制：每個來源（origin）約 5-10 MB
  • 資料格式：僅支援字串，物件需經過 JSON 序列化
  • 同步操作：所有操作都是同步的，大量資料操作可能阻塞主執行緒
  • 同源政策：嚴格遵循同源政策，不同來源無法互相存取

生命週期差異

LocalStorage 和 SessionStorage 的根本差異在於資料的生命週期：

  • LocalStorage：資料永久保存在瀏覽器中，關閉瀏覽器後仍存在，只有透過程式碼、清除瀏覽資料或超出容量才會被移除
  • SessionStorage：資料僅存活於建立它的瀏覽器分頁中，關閉分頁即消失。注意：重新整理頁面不會清除 SessionStorage

Storage 事件

當 Storage 發生變更時，同源的其他視窗會收到 storage 事件：

window.addEventListener('storage', (event) => {
  console.log(event.key);        // 被修改的鍵
  console.log(event.oldValue);   // 舊值
  console.log(event.newValue);   // 新值
  console.log(event.storageArea); // localStorage 或 sessionStorage
});

注意：觸發事件的頁面本身不會收到事件，僅其他同源頁面會收到。

典型應用架構

在現代前端框架中，通常會建立一個 Storage 抽象層：

  • Storage Service：封裝原生 API，加入序列化/反序列化
  • Expiry Manager：為 LocalStorage 實現過期機制
  • Encryption Layer：對敏感資料進行加密儲存
  • Fallback Handler：當 Storage 不可用時的降級方案

LocalStorage vs SessionStorage 使用時機

場景推薦方案原因
使用者偏好設定LocalStorage需要持久保存
購物車（已登入）LocalStorage跨分頁需要同步
表單暫存SessionStorage僅需單次工作階段
一次性認證 TokenSessionStorage安全性考量

應用場景

使用者偏好儲存

儲存使用者的個人化設定：

  • 深色/淺色主題偏好
  • 語言選擇
  • 表格欄位排序方式
  • 側邊欄展開/收合狀態

表單資料暫存

防止使用者意外遺失填寫中的表單：

  • 定期將表單內容存入 SessionStorage
  • 頁面重新整理後自動恢復
  • 成功提交後清除暫存資料

快取 API 回應

減少不必要的網路請求：

  • 將不常變更的 API 回應存入 LocalStorage
  • 搭配時間戳記實現簡易的過期機制
  • 作為離線狀態的降級資料來源

跨分頁通訊

利用 Storage 事件實現分頁間通訊：

  • 登入/登出狀態同步
  • 購物車數量即時更新
  • 共享的通知已讀狀態

應用狀態持久化

搭配狀態管理工具實現狀態持久化：

  • Redux/Vuex 狀態存入 LocalStorage
  • 頁面重載後自動還原狀態
  • 選擇性持久化部分狀態
