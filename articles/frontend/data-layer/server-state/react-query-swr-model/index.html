React Query（TanStack Query）和 SWR 代表了前端資料擷取的典範轉移。它們將「伺服器狀態」與「客戶端狀態」明確區分，提供了自動快取、背景更新、樂觀更新等能力，大幅簡化了傳統的資料擷取邏輯。

核心概念

Server State vs Client State

面向Server StateClient State
來源遠端伺服器客戶端本地
所有權伺服器客戶端
同步需求需要（可能過期）不需要
共享範圍多個客戶端共享單一客戶端
範例使用者資料、文章列表UI 狀態、表單輸入
適合的管理工具React Query / SWRuseState / Zustand

核心設計理念

概念說明
Stale-While-Revalidate先顯示快取（可能過期），同時背景重新驗證
Query Key唯一識別查詢的鍵，決定快取與失效
Query Function實際擷取資料的非同步函式
快取時間資料在快取中保留的時間
過期時間資料被視為過期的時間
自動重新擷取視窗聚焦、網路恢復時自動更新

SWR 的 Stale-While-Revalidate 流程

請求發起
├── 有快取？
│   ├── 是 → 立即回傳快取資料（可能 stale）
│   │        同時背景重新驗證
│   │        ├── 驗證成功 → 更新快取 → 通知 UI
│   │        └── 驗證失敗 → 保留舊快取 / 顯示錯誤
│   └── 否 → 顯示 loading → 擷取 → 存入快取 → 顯示資料

Query Key 設計

結構範例用途
靜態['users']固定的資源列表
參數化['users', userId]特定資源
巢狀['users', userId, 'posts']關聯資源
篩選['users', { role: 'admin' }]帶條件的查詢
分頁['users', { page: 1, size: 20 }]分頁資料

React Query vs Redux (RTK Query)

面向React QueryRTK Query
定位獨立的 server state 工具Redux 生態的一部分
與 Redux 共存獨立運作整合在 Redux Store
API 定義各組件分散集中的 API slice
快取策略Query KeyTag 系統
程式碼生成—OpenAPI 自動生成
學習曲線低中（需了解 Redux）
適用不使用 Redux 的專案已使用 Redux 的專案

傳統擷取 vs React Query/SWR

面向傳統（useEffect + useState）React Query/SWR
快取手動管理自動
去重手動（AbortController）自動（相同 key 合併）
重試手動實作內建（可設定次數）
過期更新手動 refetch自動 stale-while-revalidate
樂觀更新複雜的手動邏輯一行設定
載入/錯誤狀態多個 useState內建狀態
記憶體管理手動清理自動垃圾回收
Window Focus 更新手動監聽自動

快取策略設定

選項意義建議值
staleTime資料被視為新鮮的時間依業務需求（0~5分鐘）
gcTime (cacheTime)非活躍快取保留時間5~30 分鐘
refetchOnWindowFocus視窗聚焦時重新擷取true（大多數情況）
refetchOnReconnect網路恢復時重新擷取true
retry失敗重試次數3（預設）
retryDelay重試間隔指數退避

失效策略

策略方法觸發時機
精確失效invalidateQueries(['users', id])更新特定資源後
前綴失效invalidateQueries(['users'])任何使用者變更後
全域失效invalidateQueries()重新登入後
手動更新setQueryData(['users', id], newData)樂觀更新
Tag-basedRTK Query tagsAPI 回應指示
