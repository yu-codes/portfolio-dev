非同步取得邊界

非同步取得邊界（Async Fetch Boundary）是指在元件樹中定義非同步資料載入的處理範圍。React 的 Suspense 和 Error Boundary 是這個概念的具體實現，它們讓開發者能以宣告式的方式處理載入狀態和錯誤狀態。

核心概念

邊界的概念

邊界類型處理的狀態機制顯示
Suspense Boundary載入中（pending）捕獲 Promise throwfallback UI
Error Boundary錯誤（rejected）捕獲 render 拋出的錯誤fallback UI
資料邊界資料存在/不存在條件渲染空狀態 UI

Suspense 的工作原理

步驟說明
1子組件在渲染時發現資料未就緒
2子組件「拋出」一個 Promise
3最近的 Suspense 邊界捕獲 Promise
4顯示 fallback（Loading UI）
5Promise 解決後重新嘗試渲染子組件
6子組件成功渲染，替換 fallback

三態模型

任何非同步操作都有三種狀態：

狀態UI 呈現處理方式
LoadingSkeleton / SpinnerSuspense fallback
Success正常內容正常渲染
Error錯誤訊息 / 重試按鈕Error Boundary

架構總覽

邊界的巢狀架構

<ErrorBoundary fallback={<ErrorPage />}>
  <Suspense fallback={<PageSkeleton />}>
    <Layout>
      <ErrorBoundary fallback={<SectionError />}>
        <Suspense fallback={<CardSkeleton />}>
          <DataCard />
        </Suspense>
      </ErrorBoundary>
      <ErrorBoundary fallback={<SectionError />}>
        <Suspense fallback={<ListSkeleton />}>
          <DataList />
        </Suspense>
      </ErrorBoundary>
    </Layout>
  </Suspense>
</ErrorBoundary>

粒度選擇

粒度說明使用者體驗複雜度
頁面級整頁一個 Suspense整頁 loading低
區塊級各區段獨立漸進式載入中
組件級每個組件獨立最細粒度高
混合結合不同層級最佳體驗中-高

面向Suspense傳統 (isLoading 判斷)

面向Error Boundarytry/catch

Waterfall vs Parallel Fetch

模式行為效能實現
瀑布（Waterfall）A 完成 → B 開始 → C 開始最慢（延遲累加）組件內各自 fetch
平行（Parallel）A, B, C 同時開始最快外層統一 fetch
Render-as-you-fetch載入與渲染同步進行最佳體驗路由預載 + Suspense

應用場景

Suspense 與 SSR Streaming

模式說明效果
傳統 SSR等待所有資料就緒才送出 HTMLTTFB 高
Streaming SSR先送出 shell，資料就緒再串流補上TTFB 低
Selective Hydration優先 hydrate 使用者互動的區域TTI 低
Progressive Enhancement靜態內容先顯示，動態部分懶載入最佳 FCP

錯誤恢復策略

策略實現適用
重試Error Boundary + reset + key暫時性錯誤
降級顯示部分內容/舊快取非關鍵功能
報告錯誤上報服務所有錯誤
導向跳轉至錯誤頁面致命錯誤
重新載入window.location.reload最後手段

載入狀態最佳實踐

原則說明
避免閃爍短暫載入不顯示 spinner（設延遲）
漸進揭露先顯示框架再填入資料
保持佈局穩定Skeleton 與實際內容尺寸一致
狀態轉換動畫useTransition 避免突兀切換
內容優先優先載入可見區域的資料

設計邊界的決策

決策點考量

fallback 設計Skeleton vs Spinner vs 空白
錯誤粒度局部錯誤 vs 全頁錯誤

預載時機路由切換前 vs 組件掛載時
