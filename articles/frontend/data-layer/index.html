資料層

前端資料管理與 API 整合

前端資料層（Data Layer）是現代 Web 應用架構中最關鍵的抽象層之一。它負責管理應用程式中所有資料的取得、快取、同步與更新，將 UI 元件與底層資料來源解耦。良好的資料層設計能夠大幅降低程式碼複雜度，提升應用的可維護性與效能表現。

核心概念

資料層的設計圍繞著幾個核心問題：

資料來源分類：前端應用的資料可分為「伺服器狀態」與「客戶端狀態」兩大類。伺服器狀態是來自後端 API 的資料，具有共享、非同步、可能過期等特性；客戶端狀態則是純粹存在於瀏覽器中的資料，如 UI 狀態、表單輸入、路由參數等。

資料流方向：資料在應用中的流動方式決定了架構的可預測性。單向資料流（Unidirectional Data Flow）是目前主流的設計理念，資料從 store 流向 view，view 透過 action 觸發狀態更新，形成清晰的循環。

快取策略：資料層需要決定何時使用快取、何時重新取得資料。策略包括 stale-while-revalidate、cache-first、network-first 等，不同場景需要不同的快取失效機制。

樂觀更新：為了提升使用者體驗，資料層可在伺服器回應前先行更新 UI，若請求失敗再回滾。這需要精心設計的錯誤處理與回滾機制。

架構總覽

前端資料層架構
├── 伺服器狀態管理
│   ├── React Query / TanStack Query
│   ├── SWR
│   ├── Apollo Client (GraphQL)
│   └── RTK Query
├── 客戶端狀態管理
│   ├── 元件層級狀態 (useState, ref)
│   ├── 全域狀態 (Redux, Zustand, Pinia)
│   └── URL 狀態 (Router params, search params)
├── 資料取得模式
│   ├── REST API
│   ├── GraphQL
│   ├── tRPC (端對端類型安全)
│   └── WebSocket (即時資料)
└── 快取層
    ├── 記憶體快取
    ├── 持久化快取 (localStorage, IndexedDB)
    └── HTTP 快取 (Cache-Control, ETag)

現代前端框架的趨勢是將伺服器狀態與客戶端狀態分開管理。伺服器狀態由專門的資料取得函式庫（如 React Query、SWR）處理，它們內建快取、重試、背景更新等機制；客戶端狀態則由輕量的狀態管理工具（如 Zustand、Jotai）處理。

應用場景

電商產品列表：產品資料屬於伺服器狀態，適合使用 React Query 管理。透過分頁、無限滾動搭配 prefetch 機制，使用者在瀏覽時幾乎感受不到載入延遲。快取失效策略可設為 5 分鐘，確保價格資訊的即時性。

即時聊天應用：訊息列表需要結合 HTTP 取得歷史訊息與 WebSocket 接收即時訊息。資料層需要將兩者合併為統一的訊息流，同時處理離線佇列與訊息重傳。

儀表板與分析頁面：大量圖表需要來自不同 API 的資料。資料層可利用平行請求與選擇性快取，確保頁面快速載入。背景輪詢機制可讓資料保持最新，而不影響使用者操作。

表單密集型應用：CRM 或後台管理系統中，表單資料需要在客戶端狀態（使用者輸入）與伺服器狀態（預設值、選項列表）間協調。樂觀更新可讓儲存操作感覺即時，提升使用者信心。

多人協作工具：類似 Google Docs 的應用需要處理衝突解決（Conflict Resolution）。CRDT（Conflict-free Replicated Data Types）或 OT（Operational Transformation）等演算法在資料層中實現，確保多人同時編輯時的一致性。
