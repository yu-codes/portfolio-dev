雜湊表是最常用的資料結構之一，提供 O(1) 平均時間的查找、插入與刪除操作。索引則是資料庫系統效能的核心。本文深入探討雜湊函式設計、碰撞處理、以及資料庫索引的各種策略。

核心概念

雜湊表的基本原理

組件功能要求
雜湊函式將鍵映射到索引均勻分布、快速計算
桶陣列 (Bucket Array)儲存資料的容器大小影響碰撞率
碰撞處理處理多鍵映射到同一位置不影響正確性
負載因子 (Load Factor)已用/總容量通常 0.7-0.75 觸發擴容

雜湊函式的特性

特性描述重要性
確定性相同輸入永遠產生相同輸出必要
均勻分佈輸出均勻分散在範圍中減少碰撞
效率計算速度快不成為瓶頸
雪崩效應輸入微小變化導致輸出大變化密碼學需要

常見雜湊函式

函式用途特點
除法取餘通用簡單，質數表較好
乘法法通用不依賴表大小
MurmurHash3通用雜湊快速、分佈好
xxHash高速雜湊極快
SipHash安全雜湊表防 Hash DoS
SHA-256密碼學安全但慢
CRC32校驗偵測位元錯誤

架構總覽

碰撞處理策略

策略方法優點缺點
鏈結法 (Chaining)每個桶是鏈結串列實作簡單、刪除容易額外記憶體、快取不友善
開放定址 (Open Addressing)在表內尋找下一個位置快取友善、無額外記憶體刪除複雜、聚集問題
完美雜湊無碰撞O(1) 保證僅適用靜態集合
Cuckoo Hashing兩個表、踢出O(1) 最壞查找插入可能失敗
Robin Hood Hashing開放定址變體減少方差實作複雜

開放定址的探測方式

探測方式公式優點缺點
線性探測h(k) + i快取友善初級聚集
二次探測h(k) + i²減少聚集次級聚集
雙重雜湊h₁(k) + i·h₂(k)分佈最好計算較慢

資料庫索引類型

索引類型實作適用查詢範例
B+ Tree 索引多路平衡樹等值、範圍、排序MySQL InnoDB
Hash 索引雜湊表等值查詢Memory 引擎
全文索引倒排索引文字搜尋Elasticsearch
空間索引R-Tree地理查詢PostGIS
點陣圖索引位元陣列低基數列Oracle
GIN/GiST通用倒排陣列、JSONBPostgreSQL

特性B+ Tree 索引Hash 索引
等值查詢O(log n)O(1)
範圍查詢✅ 支援❌ 不支援
排序✅ 支援❌ 不支援
前綴匹配✅ 支援❌ 不支援
磁碟友善✅一般
空間使用中等較高（需預留空間）

索引策略比較

策略描述優點缺點
單列索引一個欄位簡單複合查詢效率低
複合索引多個欄位滿足多條件查詢順序很重要
覆蓋索引包含所有查詢欄位避免回表空間較大
部分索引只索引部分行節省空間適用範圍有限
函數索引對表達式建索引優化函數查詢需精確匹配

雜湊的實際應用

應用使用方式具體場景
快取 (Cache)鍵值存取Redis, Memcached

索引設計最佳實踐

實踐說明
最左前綴原則複合索引查詢需從左邊開始匹配
高選擇性欄位優先基數高的欄位放前面
避免過度索引索引會降低寫入效能
定期分析使用 EXPLAIN 檢查索引使用
覆蓋索引避免回表查詢
避免索引失效不對索引欄位做函數運算

一致性雜湊 (Consistent Hashing)

特性傳統雜湊取模一致性雜湊
節點增減影響幾乎所有鍵重新映射只影響相鄰節點
負載平衡取決於模數虛擬節點保證
實際應用簡單場景分散式快取、資料庫
代表系統-Cassandra, DynamoDB
