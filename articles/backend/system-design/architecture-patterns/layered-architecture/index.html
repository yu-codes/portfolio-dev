分層架構（Layered Architecture）是最經典也最廣泛使用的軟體架構模式之一。透過將系統職責劃分為多個水平層級，每層只依賴其下層，實現關注點分離與模組化。本文將深入探討分層架構的演進，從傳統三層式到 Clean Architecture 與六角架構。

核心概念

傳統分層架構

傳統分層架構將應用程式分為三到四層：

  • Presentation Layer（展示層）：處理用戶介面與 HTTP 請求/回應
  • Business Logic Layer（業務邏輯層）：核心業務規則與流程
  • Data Access Layer（資料存取層）：與資料庫的互動邏輯
  • Infrastructure Layer（基礎設施層）：外部服務、檔案系統等

依賴規則

  • 由上而下：上層可以依賴下層，但下層不可依賴上層
  • 跨層禁止：通常不允許跳層呼叫（Strict Layering）
  • 鬆散分層：有時允許跳層存取以提升效能（Relaxed Layering）

Clean Architecture

Robert C. Martin 提出的 Clean Architecture 強調依賴反轉：

  • Entities（實體）：最核心的業務規則，不依賴任何外部框架
  • Use Cases（使用案例）：應用程式特定的業務邏輯
  • Interface Adapters（介面轉接）：轉換資料格式（Controller、Presenter、Gateway）
  • Frameworks & Drivers（框架與驅動）：最外層，如 Web 框架、資料庫驅動

六角架構（Hexagonal Architecture）

又稱 Ports and Adapters，由 Alistair Cockburn 提出：

  • Port（端口）：定義應用核心與外部世界的互動介面
  • Adapter（轉接器）：Port 的具體實作，連接外部系統
  • Driving Adapter（驅動端）：外部呼叫應用（如 REST Controller）
  • Driven Adapter（被驅動端）：應用呼叫外部（如 DB Repository）

應用場景

場景一：企業 CRUD 應用

  • 傳統分層架構最為適用
  • Controller → Service → Repository 三層分明
  • 共享資料庫，交易管理簡單
  • 快速開發、易於理解與維護

場景二：複雜領域邏輯系統

  • 使用 Clean Architecture 或六角架構
  • 領域模型獨立於框架，易於測試
  • Use Case 層明確定義業務流程
  • 基礎設施可替換（如更換資料庫不影響核心邏輯）

場景三：需要多種入口的系統

  • 六角架構的 Driving Port 天然支援多入口
  • 同一業務邏輯可透過 REST、gRPC、CLI、Message Consumer 觸發
  • 每個入口是一個 Adapter，核心邏輯不變

設計原則

  1. 依賴反轉（DIP）：高層模組不依賴低層模組，都依賴抽象
  2. 單一職責：每層只負責特定類型的職責
  3. 介面隔離：定義精確的 Port 介面，避免胖介面
  4. 不要過度分層：簡單系統不需要複雜架構，YAGNI 原則

常見陷阱

  1. 貧血模型：Domain 層只有 getter/setter 沒有行為
  2. Service 層過胖：所有邏輯堆在 Service，Entity 變成資料容器
  3. 層級穿透：為了方便跨層呼叫，破壞架構一致性
  4. 過度抽象：每個類都有對應的介面，增加不必要的複雜度
