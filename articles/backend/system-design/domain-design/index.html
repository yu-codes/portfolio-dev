DDD 與系統拆分

領域驅動設計（Domain-Driven Design, DDD）是一種以業務領域為核心的軟體設計方法論，強調技術實作應緊密反映業務概念。當系統複雜度超越單純的 CRUD 操作時，DDD 提供了一套系統性的思維框架來管理複雜性、劃分系統邊界，並確保軟體模型與業務語言保持一致。本文將深入探討 DDD 的戰略設計與戰術設計，以及如何運用這些概念進行有效的系統拆分。

核心概念

DDD 的核心理念是讓軟體設計與業務領域緊密對齊。這需要開發團隊與領域專家持續協作，建立共同的語言與模型。

統一語言（Ubiquitous Language） 是 DDD 最重要的實踐之一。團隊中所有人——無論是開發者、產品經理或領域專家——都使用相同的術語描述業務概念。這些術語直接反映在程式碼中的類別名稱、方法名稱與變數命名。

限界上下文（Bounded Context） 定義了模型的適用邊界。同一個業務術語在不同的上下文中可能有完全不同的含義。例如「帳戶」在銀行系統中可能是存款帳戶，在身分認證系統中則是使用者帳號。明確界定上下文邊界避免了語意混淆。

領域模型（Domain Model） 是對業務概念的抽象表達，包含：

  • 實體（Entity）：具有唯一標識的物件，其生命週期中身分不變
  • 值物件（Value Object）：以屬性值定義的不可變物件，無獨立標識
  • 聚合（Aggregate）：一組相關物件的集合，以聚合根為入口點，維護內部一致性
  • 領域事件（Domain Event）：領域中發生的重要業務事件
  • 領域服務（Domain Service）：不屬於任何實體或值物件的業務邏輯

上下文映射（Context Mapping） 描述不同限界上下文之間的關係模式：

  • 合作關係（Partnership）：兩個團隊共同演進
  • 客戶-供應商（Customer-Supplier）：上游供應商決定介面
  • 防腐層（Anti-Corruption Layer）：隔離外部模型的影響
  • 開放主機服務（Open Host Service）：提供標準化的存取協定

架構總覽

DDD 的設計分為兩個層次：戰略設計關注系統的宏觀結構與邊界劃分；戰術設計關注單一限界上下文內部的模型實作。

戰略設計流程：

  1. 透過事件風暴（Event Storming）識別領域事件
  2. 依據業務能力與資料擁有權劃分限界上下文
  3. 定義上下文之間的關係與整合模式
  4. 確定每個上下文的核心域、支撐域與通用域分類

分層架構在 DDD 中通常採用：

  • 使用者介面層：處理外部請求
  • 應用層：編排用例流程，不含業務邏輯
  • 領域層：核心業務邏輯所在
  • 基礎設施層：技術實作細節（資料庫、訊息佇列等）

CQRS 與事件溯源：複雜領域常搭配 CQRS（命令查詢職責分離）將讀寫模型分離，並以事件溯源（Event Sourcing）記錄所有狀態變更的歷史，支援審計追蹤與時間旅行式除錯。

應用場景

電商訂單系統：訂單是天然的聚合根，包含訂單項目、配送資訊、支付狀態等。訂單的狀態轉換（建立→支付→出貨→完成）可建模為狀態機，每次轉換產生領域事件通知其他上下文。

保險理賠系統：理賠流程涉及複雜的業務規則（核保條件、理賠計算、防詐偵測），適合使用 DDD 將規則封裝在領域層。不同險種的理賠邏輯可透過策略模式實現多型。

物流調度平台：運送、路線、車輛、司機各自形成限界上下文。運送上下文關注包裹從出發到送達的流程，路線上下文負責最佳路徑計算，透過領域事件協調。

多租戶 SaaS 平台：租戶管理、訂閱計費、功能權限可拆分為獨立的限界上下文。防腐層確保計費模型的變更不會影響核心業務邏輯。
