系統設計

架構模式與分散式系統

系統設計涵蓋架構模式選擇、可擴展性設計、分散式系統原理、服務基礎設施、領域設計與工程權衡，為構建大規模可靠系統提供完整的知識框架。

核心概念

系統設計（System Design）是將業務需求轉化為技術架構的過程。它不僅涉及選擇正確的技術元件，更關鍵的是理解不同設計決策之間的權衡關係（tradeoffs），並在效能、可用性、一致性、成本和開發效率之間找到最適合當前場景的平衡點。

架構模式（Architecture Patterns） 提供了經過驗證的系統結構範本。從單體架構（Monolith）到微服務（Microservices），從事件驅動架構（EDA）到 CQRS（Command Query Responsibility Segregation），每種模式都有其適用的組織規模和業務複雜度。選擇架構模式不應追求時髦，而是匹配團隊能力和業務實際需求。

可擴展性設計（Scalability Design） 確保系統能隨負載增長而擴展。水平擴展（Scale Out）透過增加節點分散負載，垂直擴展（Scale Up）透過升級硬體提升單節點能力。負載均衡、快取策略、資料庫分片（Sharding）和 CDN 是常用的擴展手段。設計時需考慮系統的擴展瓶頸在哪——是運算、儲存、網路還是 I/O。

分散式系統（Distributed Systems） 帶來了本地系統不存在的挑戰：網路分區、時鐘偏移、拜占庭故障和最終一致性。CAP 定理指出分散式系統無法同時滿足一致性（Consistency）、可用性（Availability）和分區容忍性（Partition Tolerance）。理解這些基本限制是做出正確設計決策的前提。

服務基礎設施（Service Infrastructure） 涵蓋了支撐微服務運行的平台能力：服務發現、配置管理、可觀測性（Observability）、CI/CD 管線和容器編排。這些基礎設施的品質直接影響了開發團隊的生產力和系統的可維護性。

領域設計（Domain Design） 基於 DDD（Domain-Driven Design）思想，將業務領域的知識反映在程式碼結構中。限界上下文（Bounded Context）定義了微服務的邊界，聚合（Aggregate）保證了領域物件的一致性，領域事件（Domain Events）實現了上下文間的解耦通訊。

系統設計的藝術在於理解沒有銀彈——每個選擇都伴隨著代價。最好的架構是能夠隨業務演進而平滑演化的架構。

單體架構小-中低低產品初期
模組化單體中低中成長期
微服務大高高成熟期/大團隊
Serverless變動大中高事件驅動/不穩定負載
一致性模式一致性等級延遲可用性適用場景
強一致性（線性化）最高高較低金融交易
因果一致性高中中社交互動
最終一致性低低最高計數器、動態
Read Your Writes中中中使用者個人資料

單體 vs 微服務：初創公司應從單體或模組化單體開始，避免過早引入分散式系統的複雜性。當團隊規模超過 50 人或部署頻率需求超過每日多次時，再考慮拆分微服務。康威定律指出架構最終會反映組織結構。

同步 vs 非同步通訊：同步 HTTP/gRPC 呼叫簡單直觀但產生耦合；非同步事件驅動解耦了服務但增加了最終一致性的複雜度。核心交易路徑通常使用同步，旁路流程使用非同步。

應用場景

  1. 電商平台設計：商品目錄使用讀寫分離和快取加速查詢；訂單服務使用 Saga 模式處理跨服務交易；庫存扣減使用樂觀鎖保證一致性；推薦系統使用事件驅動非同步計算。

  2. 社交媒體動態流：Fan-out on write vs Fan-out on read 的經典權衡。大 V 使用 Pull 模式（讀時扇出），普通用戶使用 Push 模式（寫時扇出）。時間軸儲存在 Redis 中，配合分頁和快取策略。

  3. 即時通訊系統：WebSocket 長連線管理、訊息佇列保證送達、讀取狀態同步、離線訊息儲存。需要設計連線管理服務、訊息路由層和持久化層的分離。

  4. 串流影音平台：影片轉碼使用任務佇列分散處理；CDN 邊緣快取減少源站壓力；自適應位元率（ABR）根據網路品質動態調整。分散式儲存系統支撐海量內容。

  5. 支付系統：強一致性需求使用分散式交易，冪等設計防止重複扣款，事件溯源提供完整的交易軌跡。對帳系統定期比對發現不一致並觸發修復。
