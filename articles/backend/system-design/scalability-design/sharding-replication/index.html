當資料量成長到單一資料庫無法承受時，分片（Sharding）與複製（Replication）是兩大核心擴展策略。分片透過將資料分散到多個節點來提升寫入吞吐量與儲存容量；複製則透過多份副本來提升讀取效能與可用性。本文深入探討這兩種策略的原理、設計與實務挑戰。

核心概念

分片（Sharding / Partitioning）

分片是將大型資料集水平切割為多個較小的子集，分散儲存在不同節點上：

  • Shard Key（分片鍵）：決定資料分配到哪個分片的欄位
  • Shard（分片）：資料的一個子集，存放在特定節點
  • Router（路由器）：根據分片鍵將請求導向正確的分片
  • Rebalancing（再平衡）：節點增減時重新分配資料

分片策略

  • Range Sharding（範圍分片）：依據鍵值範圍分配（如日期、ID 區間）
  • Hash Sharding（雜湊分片）：對分片鍵做雜湊決定分配
  • Directory Sharding（目錄分片）：使用查找表明確指定分配
  • Geo Sharding（地理分片）：依據地理位置分配

複製（Replication）

複製是維護多份資料副本以提升可用性與讀取效能：

  • Primary-Replica（主從複製）：一個主節點處理寫入，多個副本處理讀取
  • Multi-Primary（多主複製）：多個節點都可接受寫入
  • 同步複製：寫入確認需等待副本回應，強一致但延遲高
  • 非同步複製：寫入確認不等副本，低延遲但可能遺失

CAP 定理

在網路分區（P）發生時，系統只能選擇一致性（C）或可用性（A）：

  • CP 系統：犧牲可用性保證一致性（如 ZooKeeper）
  • AP 系統：犧牲一致性保證可用性（如 Cassandra）
  • 日常運作：無分區時可同時保有 C 和 A

應用場景

場景一：社群平台用戶資料分片

  • 使用 User ID Hash 作為分片鍵
  • 同一用戶的所有資料位於同一分片（避免跨分片查詢）
  • 每個分片配置 2 個 Replica 確保高可用
  • 使用一致性雜湊減少新增節點時的資料遷移量

場景二：多區域部署的複製策略

  • 主節點在主要區域，各區域有本地唯讀副本
  • 讀取就近路由到本地副本降低延遲
  • 寫入統一路由到主節點確保一致性
  • 主節點故障時自動提升副本為新主節點

場景三：時序資料的分片設計

  • 使用時間範圍分片（每月一個分片）
  • 最新分片處理寫入，歷史分片主要讀取
  • 歷史分片可遷移到較低成本的儲存
  • 定期歸檔並刪除過期分片

設計挑戰

  1. 跨分片查詢：JOIN 操作需要查詢多個分片並合併結果
  2. 分散式交易：跨分片的 ACID 交易極為困難
  3. 熱點分片：某些分片負載遠超其他，需重新分配
  4. 再平衡代價：新增或移除節點時的資料遷移成本
  5. 複製延遲：非同步複製可能讀到過期資料

最佳實踐

  1. 慎選分片鍵：選擇高基數（cardinality）且存取均勻的欄位
  2. 避免跨分片操作：設計 schema 使常見查詢限制在單一分片
  3. 監控分片大小：確保各分片資料量均衡
  4. 規劃再平衡策略：制定新增節點時的資料遷移計畫
