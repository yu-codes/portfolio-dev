可擴展性

系統擴展設計

可擴展性（Scalability）是系統在負載增長時，透過增加資源來維持或提升效能的能力。在現代網際網路應用中，從數百用戶擴展到數百萬用戶是常見的挑戰。理解擴展策略的原理與權衡，能幫助工程師在系統設計階段就為未來的成長做好準備。本文將探討水平與垂直擴展的策略、負載均衡技術，以及資料分片與複製的實踐方法。

核心概念

擴展性設計的核心在於識別系統瓶頸並選擇合適的擴展策略：

垂直擴展（Vertical Scaling / Scale Up）：透過升級單一節點的硬體規格（CPU、記憶體、磁碟）來提升效能。優點是不需修改程式碼，缺點是存在硬體上限且成本呈指數增長。

水平擴展（Horizontal Scaling / Scale Out）：透過增加更多節點來分攤負載。優點是理論上無上限，缺點是需要處理分散式系統的複雜性（資料一致性、服務發現、負載均衡）。

負載均衡（Load Balancing） 是水平擴展的基石，負責將請求分配到多個後端節點。常見演算法：

  • 輪詢（Round Robin）：依序分配，簡單公平
  • 加權輪詢（Weighted Round Robin）：考慮節點能力差異
  • 最少連接（Least Connections）：分配給當前連接數最少的節點
  • 一致性雜湊（Consistent Hashing）：相同請求固定路由到相同節點，利於快取命中

資料分片（Sharding） 將資料集依據分片鍵分散到多個資料庫節點：

  • 範圍分片：按時間或 ID 範圍劃分，支援範圍查詢
  • 雜湊分片：均勻分布，避免熱點但犧牲範圍查詢
  • 目錄分片：維護分片映射表，靈活但增加一層間接性

資料複製（Replication） 在多個節點間維護資料副本，提供讀取擴展與高可用性：

  • 同步複製：強一致但延遲高
  • 非同步複製：低延遲但可能資料遺失
  • 半同步複製：折衷方案，至少一個副本同步確認

架構總覽

可擴展系統的典型架構由多層元件組成：

接入層：CDN 快取靜態資源、DNS 實現地理級負載均衡、反向代理（如 Nginx）處理 SSL 終止與請求路由。

應用層：無狀態的應用伺服器叢集，配合自動擴展（Auto Scaling）根據 CPU 使用率或請求佇列長度動態調整實例數量。

快取層：多級快取策略減少資料庫壓力。本地快取（如 Caffeine）處理熱點資料、分散式快取（如 Redis Cluster）提供共享的資料存取層。Cache-Aside、Write-Through、Write-Behind 是常見的快取模式。

資料層：讀寫分離是最基本的資料擴展策略。主節點處理寫入，多個從節點分擔讀取流量。當單一分片無法承載時，進一步進行水平分片。

訊息層：非同步處理透過訊息佇列（如 Kafka、RabbitMQ）削峰填谷，將同步請求轉為非同步任務。生產者不需等待消費者完成即可返回。

應用場景

社群媒體動態牆：讀取頻率遠高於寫入，適合使用 Fan-out-on-write 模式——發文時預先計算並寫入追蹤者的時間線快取，讀取時直接從快取取得。大 V 帳號的追蹤者過多時改用 Fan-out-on-read。

即時排行榜：Redis 的 Sorted Set 天然支援排行榜操作，透過 Redis Cluster 分散負載。若需全域排行，可使用 merge-sort 方式定期合併各分片結果。

檔案儲存服務：大型檔案使用物件儲存（如 S3），透過一致性雜湊決定儲存節點。小型元資料存入分散式資料庫，搭配 CDN 加速全球存取。

電商秒殺系統：多層防護——前端限流（按鈕防抖 + 驗證碼）、接入層限流（令牌桶 + 滑動視窗）、應用層庫存預扣（Redis 原子操作）、資料層最終扣減（樂觀鎖更新）。
