水平擴展（Horizontal Scaling）是透過增加更多機器來提升系統處理能力的策略，相對於垂直擴展（增加單機規格），水平擴展能突破單機硬體上限，實現近乎無限的擴展能力。本文深入探討水平擴展的設計原則、挑戰與實務模式。

核心概念

垂直 vs 水平擴展

  • 垂直擴展（Scale Up）：升級單機 CPU、記憶體、磁碟，簡單但有上限
  • 水平擴展（Scale Out）：增加更多節點，理論上無上限但設計複雜

水平擴展的前提條件

  1. 無狀態設計（Stateless）：應用程式不在本地儲存會話狀態
  2. 外部化狀態：Session、快取存放於外部共享儲存（Redis、Memcached）
  3. 冪等操作：重複請求不產生副作用
  4. 無共享架構（Shared-Nothing）：節點間無直接共享記憶體或磁碟

狀態管理策略

  • Session 外部化：將 Session 存放至 Redis 等分散式快取
  • Token-based Auth：使用 JWT 等無狀態認證，無需伺服器端 Session
  • 分散式快取：多節點共享快取層（Redis Cluster、Memcached）
  • Sticky Session：退而求其次，將同一用戶固定到同一節點（不推薦）

Auto Scaling

自動擴縮容是水平擴展的關鍵能力：

  • 基於指標擴展：CPU、記憶體、請求佇列長度
  • 預測式擴展：根據歷史模式預測性增加容量
  • 排程式擴展：根據已知的流量模式定時調整
  • 冷卻時間：避免頻繁擴縮導致的震盪

應用場景

場景一：Web API 服務擴展

  • 應用層完全無狀態，Session 存入 Redis
  • 負載平衡器使用 Least Connections 演算法
  • 基於 CPU 使用率設定 Auto Scaling（目標 70%）
  • 最小 3 個實例確保高可用，最大 20 個應對峰值

場景二：電商大促活動

  • 提前根據預估流量手動擴容基礎容量
  • 活動期間啟用積極的 Auto Scaling 策略
  • 使用預熱（Warm Pool）避免冷啟動延遲
  • 活動結束後逐步縮容，設定較長冷卻時間

場景三：即時通訊系統

  • WebSocket 連線需要考慮 Sticky Session 或 Pub/Sub 廣播
  • 使用 Redis Pub/Sub 在不同節點間同步訊息
  • 依據連線數而非 CPU 作為擴展指標
  • 連線遷移需要客戶端重連機制

設計原則

  1. Design for Failure：假設任何節點隨時可能消失
  2. Graceful Degradation：超載時降級而非崩潰
  3. 快速啟動：新節點必須能快速加入叢集
  4. 無共享：節點間不依賴共享本地狀態
  5. 監控先行：擴展策略基於可靠的監控數據

常見陷阱

  1. 本地快取過度依賴：節點重啟後快取冷啟動影響效能
  2. 資料庫成為瓶頸：應用層水平擴展但資料庫沒有跟上
  3. 不均勻的負載分配：某些節點承受更多流量
  4. 過度擴展的成本失控：Auto Scaling 未設定上限
