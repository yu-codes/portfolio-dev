在分散式系統中，一致性模型定義了資料在多個節點之間如何保持同步的規則與保證。選擇正確的一致性模型直接影響系統的效能、可用性與使用者體驗。

核心概念

一致性模型描述的是分散式系統中，多個副本對同一份資料的讀寫操作所提供的保證程度。不同的一致性等級在正確性與效能之間提供不同的權衡。

強一致性（Strong Consistency）

強一致性保證所有節點在任意時刻看到的資料完全一致。一旦寫入操作完成，後續所有的讀取操作都會返回最新值。線性化（Linearizability）是最強的一致性保證，它要求所有操作看起來像是在單一時間點原子性地執行。

最終一致性（Eventual Consistency）

最終一致性允許系統在短時間內存在不一致的狀態，但保證在沒有新的寫入操作時，所有副本最終會收斂到相同的狀態。這是許多大規模分散式系統（如 Amazon DynamoDB）採用的模型。

因果一致性（Causal Consistency）

因果一致性保證有因果關係的操作在所有節點上以相同的順序被觀察到。沒有因果關係的並發操作可以被不同節點以不同順序觀察。

順序一致性（Sequential Consistency）

順序一致性要求所有操作的結果等同於某種全域序列順序，且每個處理器的操作在該序列中保持程式順序。

在實際架構中，一致性模型的實現通常依賴以下機制：

  • 共識協議：如 Paxos、Raft，用於達成強一致性
  • 向量時鐘：用於追蹤因果關係
  • 反熵協議：如 Gossip Protocol，用於最終一致性
  • Quorum 機制：通過 R + W > N 調整一致性等級

關鍵比較

一致性模型延遲可用性正確性保證典型場景
Linearizability高低最強分散式鎖、選主
Sequential中高中強共享記憶體系統
Causal中中高中等協作編輯、社交網路
Eventual低高最終收斂DNS、CDN、購物車

實現複雜度比較

強一致性需要跨節點協調，通常需要至少 2 次網路往返（如兩階段提交）。最終一致性只需要本地寫入加上背景同步，延遲可降至個位數毫秒。

衝突解決策略

  • Last-Writer-Wins (LWW)：簡單但可能丟失資料
  • CRDT（Conflict-free Replicated Data Types）：無衝突合併，適用於特定資料結構
  • 應用層解決：由業務邏輯決定如何合併衝突

應用場景

金融交易系統

銀行轉帳、股票交易等場景需要強一致性保證。帳戶餘額不能出現「超賣」情況，因此通常採用 Linearizability 配合兩階段提交或 Saga 模式。

社交媒體動態

用戶發文後，追蹤者不需要立即看到更新。採用最終一致性可以大幅提升系統吞吐量，延遲通常在秒級別。

協作文件編輯

如 Google Docs 採用因果一致性，確保編輯操作的因果順序被保留，同時允許並發編輯在不同用戶端以不同順序呈現後再收斂。

電商購物車

Amazon 的經典案例：採用最終一致性加上 CRDT 的合併策略。寧可多加商品也不丟失用戶的選擇，在結帳時再做最終確認。

分散式快取

快取系統通常採用最終一致性，搭配 TTL（Time-to-Live）策略。Cache-Aside 模式允許短暫的不一致窗口，以換取極高的讀取效能。
