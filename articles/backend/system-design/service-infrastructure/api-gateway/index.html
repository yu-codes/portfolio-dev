API Gateway 是微服務架構中的核心基礎設施元件，作為所有客戶端請求的統一入口點。它負責路由、認證、限流、監控等橫切關注點，讓後端服務專注於業務邏輯。

核心概念

什麼是 API Gateway

API Gateway 是位於客戶端和後端服務之間的反向代理，提供單一入口點來管理所有 API 請求。它是微服務架構中實現關注點分離的關鍵元件。

核心功能

  • 請求路由（Request Routing）：根據路徑、標頭或內容將請求導向正確的後端服務
  • 負載均衡（Load Balancing）：將請求分發到多個服務實例
  • 認證與授權（Authentication & Authorization）：集中處理 JWT 驗證、OAuth2、API Key
  • 限流（Rate Limiting）：保護後端服務免受過量請求
  • 請求/回應轉換（Transform）：修改請求或回應的格式與內容
  • 快取（Caching）：減少後端服務的重複計算
  • 監控與日誌（Observability）：統一的指標收集與存取日誌

Gateway 模式

  1. Edge Gateway：面向外部客戶端的單一入口
  2. Internal Gateway：服務間通訊的內部閘道
  3. BFF（Backend for Frontend）：為特定前端量身定做的閘道層

請求處理流程

  1. 客戶端發送請求到 Gateway
  2. Gateway 執行認證（驗證 Token）
  3. 檢查限流規則（是否超過配額）
  4. 請求轉換（添加標頭、修改路徑）
  5. 路由到目標後端服務
  6. 接收回應並轉換格式
  7. 返回回應給客戶端

API Gateway vs Service Mesh

  • API Gateway：處理南北向流量（客戶端到服務），關注外部 API 管理
  • Service Mesh：處理東西向流量（服務到服務），關注內部通訊治理
  • 兩者互補而非互斥，大型系統通常同時使用

限流演算法比較

  • Fixed Window：固定時間視窗計數，簡單但有邊界突刺問題
  • Sliding Window：滑動視窗，更精確但記憶體開銷大
  • Token Bucket：允許突發流量，適合大多數場景
  • Leaky Bucket：平滑流量，適合需要穩定速率的場景

應用場景

多客戶端 BFF 模式

不同的客戶端（Web、Mobile、IoT）需要不同的 API 格式。通過 BFF Gateway 為每種客戶端提供量身定做的 API，Mobile 端返回精簡資料減少頻寬，Web 端返回完整資料提升體驗。

漸進式系統遷移

從單體架構遷移到微服務時，API Gateway 作為「絞殺者模式（Strangler Fig）」的關鍵元件。新功能路由到微服務，舊功能繼續路由到單體，逐步完成遷移而不中斷服務。

API 版本管理

通過 Gateway 實現 API 版本控制（如 /v1/users、/v2/users），不同版本路由到不同服務實例。可以實現灰度發布、A/B 測試，以及舊版本的優雅退役。

多租戶 SaaS

為不同租戶設定不同的限流策略、認證方式和存取權限。企業級租戶可以有更高的 QPS 配額和專屬的服務實例，而免費用戶共享資源池。

API 貨幣化

追蹤每個 API Key 的呼叫次數和資料量，實現基於使用量的計費。結合限流和配額管理，提供不同等級的 API 存取方案。
