服務基礎設施是支撐微服務架構正常運作的底層平台元件，包含服務發現、API 閘道、服務網格等關鍵組成。當系統從單體演進為數十甚至數百個微服務時，如何有效地管理服務間的通訊、路由、安全與可觀測性成為核心挑戰。本文將深入探討微服務基礎設施的各個面向，幫助工程師建構穩健的服務平台。

核心概念

微服務基礎設施解決的是「服務如何找到彼此、如何安全通訊、如何統一管理」的問題。

服務發現（Service Discovery） 讓服務能動態地找到其他服務的網路位置。在容器化環境中，服務實例的 IP 地址會頻繁變動，靜態配置不再可行。

  • 客戶端發現：客戶端查詢服務註冊中心（如 Consul、Eureka），自行選擇可用實例
  • 伺服端發現：透過負載均衡器（如 AWS ELB、Kubernetes Service）代理請求
  • DNS 發現：利用 DNS SRV 記錄實現服務定位，簡單但 TTL 快取可能導致延遲

API 閘道（API Gateway） 是系統的統一入口點，承擔多種跨切面（Cross-Cutting）功能：

  • 請求路由：依據路徑、Header 或內容將請求路由到對應服務
  • 認證授權：集中處理 JWT 驗證、OAuth 流程、API Key 檢查
  • 流量控制：實施速率限制（Rate Limiting）、配額管理與熔斷保護
  • 協定轉換：將外部的 REST 請求轉換為內部的 gRPC 呼叫
  • 請求聚合：合併多個後端服務的回應為單一客戶端回應（BFF 模式）

服務網格（Service Mesh） 將服務間通訊的關注點從應用程式碼抽離到基礎設施層。透過 Sidecar Proxy（如 Envoy）攔截所有進出流量，統一實施：

  • mTLS 加密：服務間的雙向 TLS 認證
  • 流量管理：金絲雀部署、A/B 測試、故障注入
  • 可觀測性：自動收集延遲、錯誤率、吞吐量指標
  • 韌性模式：重試、超時、斷路器

架構總覽

現代微服務基礎設施的典型分層架構：

外部流量層：CDN → 全域負載均衡 → WAF（Web 應用防火牆）→ API 閘道。外部請求經過多層過濾後才抵達內部服務。

API 閘道層：可採用單一閘道或 BFF（Backend for Frontend）模式。BFF 模式為不同客戶端（Web、Mobile、IoT）提供專屬的閘道，避免單一閘道過於臃腫。常見實作包括 Kong、Envoy Gateway、AWS API Gateway。

服務通訊層：

  • 同步通訊：REST（簡單易懂）、gRPC（高效能、強型別）、GraphQL（靈活查詢）
  • 非同步通訊：事件匯流排（Kafka）、訊息佇列（RabbitMQ）、發布-訂閱模式

服務網格層：控制平面（如 Istio 的 istiod）管理配置與憑證；資料平面（Envoy Sidecar）處理實際流量。零信任網路模型假設內部網路也不可信。

觀測性層：

  • 追蹤（Tracing）：OpenTelemetry + Jaeger/Zipkin
  • 指標（Metrics）：Prometheus + Grafana
  • 日誌（Logging）：ELK Stack 或 Loki
  • 三者關聯可實現從告警到根因的快速定位

應用場景

多租戶 SaaS 平台：API 閘道根據租戶 ID 實施不同的速率限制與配額。服務網格確保租戶資料隔離，透過命名空間與網路策略限制跨租戶存取。

金融科技支付系統：服務網格的 mTLS 確保支付資料在傳輸中加密。API 閘道實施嚴格的認證流程（OAuth 2.0 + PKCE）。分散式追蹤幫助定位支付失敗的根因。

遊戲後端：API 閘道處理玩家登入與匹配請求，gRPC 用於遊戲伺服器間的低延遲通訊。服務發現支援動態房間分配與自動擴展。

邊緣計算 IoT：閘道在邊緣節點預處理設備資料，減少雲端頻寬消耗。服務網格的流量管理支援漸進式韌體更新。
