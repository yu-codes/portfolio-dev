一致性與可用性的權衡是分散式系統設計中最根本的決策之一。理解這個權衡的本質，以及在不同業務場景下如何做出正確的選擇，是後端工程師必備的系統設計能力。

核心概念

權衡的本質

在分散式系統中，當網路分區發生時（這是不可避免的），系統必須在以下兩者之間選擇：

  • 一致性優先（CP）：拒絕可能不一致的請求，犧牲部分可用性
  • 可用性優先（AP）：接受所有請求但允許回應可能不是最新值

這不是全有全無的選擇，而是一個光譜。現代系統通常在不同的操作和資料類型上做出不同的選擇。

一致性的層次

  • 線性化（Linearizability）：最強保證，如同單一副本
  • 順序一致性：全域操作順序一致
  • 因果一致性：因果相關操作有序
  • 讀己之寫：用戶能看到自己的最新寫入
  • 最終一致性：最弱保證，最終收斂

可用性的定義

  • 高可用（High Availability）：99.9%~99.999% 的時間回應請求
  • 永遠可用（Always Available）：任何非故障節點都能回應
  • 降級可用：核心功能可用，部分功能暫時不可用

Harvest vs Yield

Eric Brewer 提出的替代框架：

  • Harvest：回應中包含多少完整的資料（資料完整度）
  • Yield：成功回應的請求比例（可用性）

系統可以選擇犧牲部分 Harvest（返回不完整資料）來保證 Yield（持續可用）。

實現一致性的機制

  • Paxos/Raft 共識：多數派同意才完成寫入
  • 兩階段提交（2PC）：跨節點原子操作
  • 同步複製：寫入確認前等待副本
  • 分散式鎖：互斥存取共享資源

實現可用性的機制

  • 多主複製：任何節點都可寫入
  • 最終一致性 + CRDT：無衝突合併
  • 讀修復（Read Repair）：讀取時修復不一致
  • 提示遞交（Hinted Handoff）：暫存失敗節點的寫入

應用場景

銀行轉帳（一致性優先）

帳戶餘額必須精確，不能出現雙重支付或超額提款。系統選擇一致性：在網路異常時拒絕交易，用戶看到「系統繁忙請稍後重試」，而非執行可能錯誤的交易。使用 Serializable 隔離等級和同步複製。

社交媒體按讚數（可用性優先）

按讚數短暫不精確完全可以接受，但如果用戶按讚後系統報錯會嚴重影響體驗。選擇可用性：本地計數器即時回應，背景非同步同步。顯示「約 1.2 萬」而非精確數字。

電商庫存（混合策略）

熱門商品的最後幾件庫存需要強一致（避免超賣），但大量庫存時可以使用最終一致性（允許短暫的數量不精確）。系統根據庫存水位動態切換一致性策略。

全球分散式系統

用戶寫入偏好設定（語言、主題）採用可用性優先，各區域本地寫入後背景同步。但購買記錄和帳戶安全設定使用一致性優先，集中到用戶歸屬區域處理。

搜尋索引

搜尋索引天然適合最終一致性。新建立的商品可能需要數秒才會出現在搜尋結果中，這對用戶體驗影響極小，但帶來的效能提升卻是巨大的。
