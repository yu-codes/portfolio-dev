資料串流

訊息佇列與事件驅動

資料串流（Data Streaming）是現代分散式系統中處理即時資料流動的核心技術。從使用者行為追蹤、IoT 感測器數據收集、到微服務間的非同步通訊，串流架構讓系統能夠以近乎即時的方式處理持續湧入的資料，而非依賴傳統的批次處理。本篇將深入探討訊息佇列、串流處理引擎與事件驅動架構的設計原則與實踐。

核心概念

資料串流的三大支柱為：訊息佇列（Message Queues）、串流處理（Stream Processing）、以及事件驅動架構（Event-Driven Architecture）。

訊息佇列提供了生產者與消費者之間的解耦機制。生產者將訊息發送至佇列，消費者依照自己的速率消費訊息。這種模式天然支援流量削峰（Peak Shaving）與負載均衡。常見實作包括 RabbitMQ、Amazon SQS 與 Redis Streams。

串流處理則更進一步，支援對無界資料流（Unbounded Data Stream）進行即時計算。Apache Kafka Streams、Apache Flink 與 Apache Spark Streaming 是主流的串流處理框架，支援視窗聚合（Windowed Aggregation）、狀態計算與複雜事件處理（CEP）。

事件驅動架構（EDA） 是一種以事件為中心的系統設計範式。系統中的各個元件透過發佈（Publish）與訂閱（Subscribe）事件進行溝通，而非直接的同步呼叫。這使得系統具備更好的鬆耦合性、可擴展性與容錯能力。

架構總覽

典型的資料串流架構可分為以下階段：

  1. 資料攝取（Ingestion）：透過 Producer API、Change Data Capture（CDC）或 HTTP Webhook 將資料注入串流平台
  2. 訊息代理（Broker）：Kafka、Pulsar 或 RabbitMQ 負責訊息的持久化、分區與路由
  3. 串流處理（Processing）：消費者群組或串流處理引擎進行即時轉換、過濾與聚合
  4. 資料匯出（Sink）：將處理結果寫入資料庫、搜尋引擎、資料倉儲或觸發下游服務

以 Kafka 為核心的架構中，Topic 是訊息的邏輯分類，Partition 提供並行處理能力，Consumer Group 實現水平擴展。每條訊息透過 Offset 精確定位，支援重放（Replay）與精確一次語義（Exactly-Once Semantics）。

應用場景

即時推薦系統：電商平台捕獲使用者的瀏覽、點擊與購買事件，透過串流處理即時更新使用者畫像與推薦模型，實現秒級個性化推薦。

微服務事件匯流排：在微服務架構中，以 Kafka 作為事件匯流排（Event Bus），各服務透過發佈/訂閱事件進行非同步協作。例如訂單服務發佈「訂單已建立」事件，庫存服務與通知服務各自消費處理。

IoT 資料管線：大量感測器持續產生數據，透過 MQTT → Kafka → Flink 的管線進行即時異常偵測與聚合統計，再將結果寫入時序資料庫供儀表板展示。

日誌聚合與分析：將分散式系統中各節點的日誌統一收集至 Kafka，再透過串流處理進行即時告警規則匹配，同時批次匯入 Elasticsearch 支援全文搜尋。

金融市場數據：股票行情、匯率變動等高頻資料需要低延遲處理，串流架構能在毫秒級內完成風控規則計算與異常交易偵測。
