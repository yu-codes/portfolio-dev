快取失效模式

「電腦科學中只有兩件困難的事：快取失效與命名。」快取失效是分散式系統中最具挑戰性的問題之一。選擇正確的失效策略，攸關系統能否在效能與資料一致性之間取得平衡。

核心概念

基於時間的失效（TTL-based）

最簡單也最常用的失效機制：

  • 固定 TTL：設定固定的存活時間，到期自動失效
  • 滑動 TTL：每次存取重新計時
  • 動態 TTL：依據資料變更頻率動態調整
  • 階梯式 TTL：熱門資料較長 TTL，冷門資料較短

TTL 設定考量：

  • 太短：快取命中率低，資料庫壓力大
  • 太長：資料過期風險高，使用者看到舊資料
  • 合理值：依業務容忍的最大過期時間決定

基於事件的失效（Event-based）

資料變更時主動觸發失效：

  • 主動失效（Active Invalidation）：寫入操作直接刪除相關快取
  • 訊息驅動：透過 Message Queue 廣播失效事件
  • CDC（Change Data Capture）：監聽資料庫變更日誌觸發失效
  • Webhook 通知：外部系統通知快取失效

版本化失效（Version-based）

透過版本號判斷快取是否過期：

  • ETag/If-None-Match：HTTP 條件式請求
  • 版本計數器：資料每次更新遞增版本號
  • 內容雜湊：依內容計算雜湊值判斷是否變更
  • 時間戳比較：比較快取時間與最後修改時間

快取失效的常見問題

  • Cache Stampede（快取驚群）：熱點資料同時過期，大量請求湧入資料庫
  • Cache Penetration（快取穿透）：查詢不存在的資料，每次都打到資料庫
  • Cache Breakdown（快取擊穿）：單一熱點 key 過期，瞬間大量請求
  • Cache Avalanche（快取雪崩）：大量快取同時過期

關鍵比較

失效策略比較

策略一致性複雜度延遲適用場景
固定 TTL最弱最低TTL 週期內容忍過期的資料
事件驅動刪除強中秒級即時性要求高
CDC 驅動強高秒~分鐘解耦合場景
版本化強中請求時判斷HTTP 快取
手動失效視操作低立即緊急修正

防護機制比較

問題防護方案原理複雜度
Cache Stampede互斥鎖（Mutex）僅一個請求回源中
Cache Stampede邏輯過期資料不真正過期中
Cache Penetration布隆過濾器攔截不存在的查詢中
Cache Penetration空值快取快取空結果低
Cache Breakdown永不過期+背景刷新避免過期瞬間中
Cache Avalanche隨機化 TTL打散過期時間低
Cache Avalanche多層快取降級保護高

失效傳播方式比較

方式即時性可靠性擴展性適用規模
直接刪除最高高低單機/少量節點
Pub/Sub 廣播高中中中等規模
Message Queue中高高大規模
CDC Pipeline中高高大規模/解耦
輪詢比較低高低簡單場景

應用場景

場景一：電商商品價格更新

事件驅動 + 延遲雙刪策略：

  • 商品服務更新價格後發送失效事件
  • 立即刪除快取（第一次刪除）
  • 延遲 500ms 再次刪除（處理主從延遲）
  • 搭配短 TTL 作為兜底保護

場景二：使用者權限變更

CDC + 多層失效：

  • 權限變更寫入資料庫
  • Debezium 捕獲 Binlog 變更
  • 透過 Kafka 發送至各服務
  • 各服務刪除本地快取 + Redis 快取
  • 強制下次請求重新驗證權限

場景三：內容發布系統

版本化 + 事件驅動：

  • 內容每次發布產生新版本號
  • CDN 使用版本號作為快取鍵的一部分
  • 發布時通知 CDN 清除舊版本
  • 使用者端透過版本號獲取最新內容

場景四：熱點資料保護

互斥鎖 + 邏輯過期：

  • 熱點商品快取設定邏輯過期時間（非 Redis TTL）
  • 讀取時檢查邏輯過期時間
  • 若已過期：獲取互斥鎖，單一執行緒回源更新
  • 其他執行緒返回舊資料（降級策略）
  • 更新完成後釋放鎖，後續請求獲得新資料
