關聯式資料建模

關聯式資料建模是資料庫設計的基石，透過實體關係模型（ER Model）將真實世界的業務需求轉化為結構化的資料表設計。掌握正規化理論與反正規化策略，是打造高效能資料庫的關鍵。

核心概念

實體關係模型（ER Model）

ER 模型的三大組成元素：

  • 實體（Entity）：代表真實世界的物件或概念，如客戶、訂單、產品
  • 屬性（Attribute）：描述實體的特徵，如客戶姓名、訂單日期
  • 關係（Relationship）：實體之間的關聯，如客戶「下」訂單

關係類型

  • 一對一（1:1）：一個員工對應一個員工檔案
  • 一對多（1:N）：一個客戶對應多筆訂單
  • 多對多（M:N）：多個學生選修多門課程，需要中間表

正規化（Normalization）

正規化的目的是消除資料冗餘與更新異常：

  • 第一正規形式（1NF）：消除重複群組，確保每欄位為原子值
  • 第二正規形式（2NF）：消除部分相依，非主鍵屬性完全相依於主鍵
  • 第三正規形式（3NF）：消除遞移相依，非主鍵屬性不相依於其他非主鍵
  • BCNF：每個決定因素都是候選鍵
  • 第四正規形式（4NF）：消除多值相依

主鍵設計策略

  • 自然鍵（Natural Key）：使用業務欄位如身分證號碼
  • 代理鍵（Surrogate Key）：系統產生的無意義識別碼（如自增 ID、UUID）
  • 複合鍵（Composite Key）：多個欄位組合形成主鍵

完整性約束

  • 實體完整性：主鍵不可為 NULL
  • 參考完整性：外鍵必須參照存在的主鍵值
  • 域完整性：欄位值必須在合法範圍內
  • 業務完整性：符合特定業務規則的約束

應用場景

場景一：電商系統資料建模

電商核心資料模型設計：

  • 客戶（Customer）→ 訂單（Order）→ 訂單項目（OrderItem）→ 產品（Product）
  • 使用代理鍵作為主鍵，保留自然鍵作為唯一約束
  • 訂單金額同時儲存於訂單表（反正規化），避免即時計算

場景二：多租戶 SaaS 系統

多租戶資料隔離策略：

  • 共享資料庫+租戶欄位：透過 tenant_id 區分資料
  • 共享資料庫+獨立 Schema：每個租戶獨立 Schema
  • 獨立資料庫：每個租戶獨立資料庫實例

場景三：版本化資料模型

需要追蹤資料歷史變更時：

  • 使用 Temporal Table 記錄有效期間
  • SCD Type 2：新增記錄標記生效/失效時間
  • Event Sourcing：儲存所有變更事件

場景四：階層式資料結構

組織架構或分類樹的建模方式：

  • 鄰接表（Adjacency List）：parent_id 自我參照
  • 巢狀集合（Nested Set）：左右值標記
  • 路徑列舉（Path Enumeration）：儲存完整路徑
  • Closure Table：儲存所有祖先-後代關係
