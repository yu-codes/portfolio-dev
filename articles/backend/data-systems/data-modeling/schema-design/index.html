Schema 設計模式

Schema 設計模式是在不同業務情境下，針對資料結構的最佳實踐與常見解決方案。從多型態繼承到版本化演進，優秀的 Schema 設計直接影響系統的可維護性與擴展性。

核心概念

Schema 演進策略

隨著業務發展，Schema 不可避免地需要變更：

  • 向前相容（Forward Compatibility）：舊程式碼可處理新 Schema 資料
  • 向後相容（Backward Compatibility）：新程式碼可處理舊 Schema 資料
  • Schema Registry：集中管理 Schema 版本與相容性驗證
  • 遷移腳本（Migration）：版本化的 Schema 變更管理

繼承模式（Inheritance Patterns）

在關聯式資料庫中表示物件繼承：

  • 單表繼承（Single Table Inheritance）：所有子類型放入同一表，使用 type 欄位區分
  • 類別表繼承（Class Table Inheritance）：每個類別一張表，透過 JOIN 組合
  • 具體表繼承（Concrete Table Inheritance）：每個具體類別獨立一張表

多租戶模式（Multi-tenancy Patterns）

  • 共享表+租戶欄位：所有租戶共用表，以 tenant_id 區隔
  • Schema 隔離：每個租戶獨立 Schema（PostgreSQL Schema）
  • 資料庫隔離：每個租戶獨立資料庫
  • 混合模式：依租戶等級採用不同隔離策略

時態資料模式（Temporal Patterns）

追蹤資料隨時間的變化：

  • Slowly Changing Dimension Type 1：直接覆寫，不保留歷史
  • SCD Type 2：新增記錄，標記有效期間
  • SCD Type 3：增加欄位存放前一版值
  • Bi-temporal：同時追蹤有效時間與交易時間
  • Event Sourcing：儲存所有狀態變更事件

應用場景

場景一：電商產品 Schema

多品類產品的 Schema 設計：

  • 共用屬性（名稱、價格、庫存）放入基礎表
  • 品類特有屬性使用 EAV（Entity-Attribute-Value）或 JSON 欄位
  • 產品變體（尺寸、顏色）使用獨立的 variant 表
  • 價格歷史使用 SCD Type 2 追蹤

場景二：SaaS 權限系統

RBAC 權限系統的 Schema 設計：

  • Users → UserRoles → Roles → RolePermissions → Permissions
  • 支援角色繼承：角色可包含子角色
  • 資源層級權限：使用 closure table 表示資源階層
  • 時效性權限：加入有效期間欄位

場景三：內容管理系統

CMS 的靈活 Schema 設計：

  • 內容類型定義表：動態定義內容結構
  • 欄位值表：EAV 模式儲存各類型欄位
  • 版本控制：每次編輯產生新版本
  • 發布工作流：草稿→審核→發布狀態機

場景四：金融交易系統

金融系統的嚴謹 Schema 設計：

  • 雙重記帳（Double Entry）：每筆交易有借方與貸方
  • 不可變記錄：僅新增，不修改或刪除
  • 精確數值：使用 DECIMAL 而非 FLOAT
  • 完整審計追蹤：who/what/when/where
