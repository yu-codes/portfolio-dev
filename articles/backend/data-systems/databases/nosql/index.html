NoSQL 資料庫生態

NoSQL 資料庫打破了關聯式資料庫一統天下的局面，為不同的工作負載提供了更適切的解決方案。從文件資料庫到圖形資料庫，每種類型都有其獨特的優勢與最佳應用場景。

核心概念

CAP 定理

分散式系統中的三個保證最多只能同時滿足兩個：

  • 一致性（Consistency）：所有節點在同一時間看到相同資料
  • 可用性（Availability）：每個請求都能得到回應
  • 分區容錯（Partition Tolerance）：網路分區時系統仍能運作

由於網路分區不可避免，實際選擇在 CP（一致性優先）與 AP（可用性優先）之間。

NoSQL 四大類型

  • 文件資料庫（Document）：以 JSON/BSON 文件為單位儲存，如 MongoDB、CouchDB
  • 鍵值資料庫（Key-Value）：最簡單的 key-value 對應，如 Redis、DynamoDB
  • 寬列資料庫（Wide-Column）：列族模型，如 Cassandra、HBase
  • 圖形資料庫（Graph）：以節點與邊表示關係，如 Neo4j、Amazon Neptune

一致性模型

  • 強一致性（Strong Consistency）：讀取一定能看到最新寫入
  • 最終一致性（Eventual Consistency）：經過一段時間後所有副本趨於一致
  • 因果一致性（Causal Consistency）：有因果關係的操作保證順序
  • 讀自己寫（Read Your Writes）：使用者一定能讀到自己的寫入

分片策略

  • 範圍分片（Range Sharding）：依鍵值範圍分配，支援範圍查詢
  • 雜湊分片（Hash Sharding）：依雜湊值分配，分佈均勻
  • 複合分片：結合範圍與雜湊策略
  • 地理分片：依地理位置分配資料

主要 NoSQL 資料庫架構特點

  • MongoDB：分片叢集 + 副本集，支援靈活查詢
  • Cassandra：無主架構（Masterless），線性擴展
  • DynamoDB：全託管，自動分區，預設最終一致
  • Redis：記憶體為主，支援持久化，單執行緒模型

應用場景

場景一：內容管理平台

使用 MongoDB 管理多樣化內容：

  • 靈活的文件結構適應不同內容類型
  • 豐富的查詢與聚合管道
  • 全文搜尋索引支援內容檢索
  • Change Streams 實現即時通知

場景二：IoT 資料平台

使用 Cassandra 處理大量時序資料：

  • 線性擴展應對持續增長的資料量
  • 寫入優化的 LSM-Tree 引擎
  • TTL 自動過期清理舊資料
  • 多資料中心複製確保可用性

場景三：Serverless 應用後端

使用 DynamoDB 作為 Serverless 資料層：

  • 與 AWS Lambda 無縫整合
  • 按需計費，無需容量規劃
  • DynamoDB Streams 觸發事件處理
  • 全域表實現多區域部署

場景四：即時推薦系統

使用 Redis 支撐即時推薦：

  • Sorted Set 維護推薦分數排名
  • HyperLogLog 計算唯一訪客數
  • Pub/Sub 傳播即時更新
  • 搭配 RedisSearch 進行向量搜尋
