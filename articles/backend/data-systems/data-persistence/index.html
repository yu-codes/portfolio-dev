資料持久化

備份、遷移與事件溯源

資料持久化（Data Persistence）是後端系統中最核心的關注點之一。無論是使用者產生的交易記錄、系統日誌、還是業務狀態快照，如何確保資料在各種故障場景下不遺失、可恢復、並能安全地進行版本演進，決定了整個系統的可靠度與可維護性。本篇文章將系統性地介紹資料持久化的核心概念、架構模式，以及在現代分散式系統中的實踐策略。

核心概念

資料持久化涵蓋三大面向：備份與復原（Backup & Recovery）、資料遷移策略（Migration Strategies）、以及事件溯源與 CQRS（Event Sourcing & CQRS）。

備份與復原的目標是在硬體故障、人為誤操作或災難事件發生時，能夠將系統回復到已知的正確狀態。常見策略包括全量備份（Full Backup）、增量備份（Incremental Backup）與差異備份（Differential Backup）。在雲端環境中，還可利用快照（Snapshot）機制實現近乎即時的資料保護。

資料遷移策略關注的是資料庫結構演進時，如何在不中斷服務的前提下完成 Schema 變更。這包括版本化遷移腳本（Versioned Migration Scripts）、藍綠資料庫切換、以及雙寫（Dual Write）等模式。

事件溯源（Event Sourcing） 則是一種根本性的思維轉變：不再儲存「當前狀態」，而是記錄所有導致狀態變化的事件序列。搭配 CQRS（Command Query Responsibility Segregation），將讀寫路徑分離，可獲得極高的可審計性與擴展彈性。

架構總覽

一個完善的資料持久化架構通常包含以下層次：

  1. 應用層：負責產生寫入請求與事件
  2. 持久化層：包含主資料庫（Primary Store）與事件日誌（Event Log）
  3. 備份層：排程備份、快照管理與異地複寫
  4. 復原層：災難復原流程、RTO/RPO 管理

在事件溯源架構中，Event Store 作為唯一的真實資料來源（Single Source of Truth），所有狀態變更都以不可變事件的形式追加寫入。投影（Projection）機制負責將事件流物化為可查詢的讀取模型。

對於需要跨區域高可用的系統，常見做法是採用主從複寫搭配異步備份，並透過 WAL（Write-Ahead Log）確保寫入操作的原子性。

應用場景

金融交易系統：每筆交易都必須完整記錄且不可篡改，事件溯源天然適合此類需求。搭配 CQRS 可將即時查詢與歷史審計分離，各自優化。

電商平台：訂單狀態機的每次轉換（建立 → 付款 → 出貨 → 完成）都可建模為事件，支援退款追溯與爭議處理。

SaaS 多租戶系統：資料遷移策略在此特別重要。每次 Schema 升級都必須考慮向後相容性，並支援租戶級別的灰度升級。

災難復原（DR）：透過跨區域備份與自動化復原腳本，確保在主資料中心失效時能在 RPO/RTO 目標內恢復服務。常見做法包括 Pilot Light、Warm Standby 與 Multi-Site Active-Active。

協作編輯系統：如 Google Docs 類應用，利用事件溯源記錄每次編輯操作，支援撤銷、重做與衝突解決。
