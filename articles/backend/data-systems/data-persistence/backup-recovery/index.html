備份與復原是資料庫管理中最基礎也最關鍵的運維能力。無論是人為誤操作、硬體故障，還是自然災害，完善的備份策略能確保業務資料的安全性與可恢復性。本文將深入探討備份與復原的核心概念、架構設計與實務應用。

核心概念

備份類型

備份可依照範圍與方式分為以下幾類：

  • 完整備份（Full Backup）：將整個資料庫完整複製一份，恢復時最簡單但佔用空間最大
  • 增量備份（Incremental Backup）：僅備份自上次備份以來變更的資料，節省空間但恢復較複雜
  • 差異備份（Differential Backup）：備份自上次完整備份以來所有變更的資料，為完整與增量的折中方案
  • 日誌備份（Transaction Log Backup）：備份交易日誌，可實現時間點還原（Point-in-Time Recovery）

RPO 與 RTO

  • RPO（Recovery Point Objective）：可容忍的最大資料遺失量，決定備份頻率
  • RTO（Recovery Time Objective）：系統從故障到恢復正常運作的最大允許時間，決定恢復策略的複雜度

備份一致性

資料庫備份必須確保一致性，避免備份到「半完成」的交易狀態：

  • 冷備份（Cold Backup）：停機後備份，一致性最高但影響可用性
  • 熱備份（Hot Backup）：不停機備份，需要資料庫引擎支援一致性快照
  • 邏輯備份（Logical Backup）：匯出 SQL 語句或結構化資料，如 mysqldump、pg_dump
  • 物理備份（Physical Backup）：直接複製資料檔案，如使用 xtrabackup、LVM snapshot

架構總覽

備份排程策略

常見的排程組合模式：

  1. 每日完整 + 每小時增量：適合中小型資料庫
  2. 每週完整 + 每日差異 + 每小時日誌：適合大型生產環境
  3. 連續歸檔（Continuous Archiving）：如 PostgreSQL WAL archiving，實現接近零 RPO

儲存與保留策略

  • 3-2-1 原則：至少三份副本、兩種不同媒體、一份異地存放
  • 保留週期：依合規需求設定，通常為 7 天短期 + 30 天中期 + 1 年長期
  • 加密與存取控制：備份資料同樣需要加密保護與嚴格的存取權限管理

關鍵比較

特性完整備份增量備份差異備份日誌備份
備份速度慢快中等最快
恢復速度最快慢（需逐步還原）中等視範圍而定
儲存空間大小中等小
複雜度低高中等高
RPO取決於頻率可做到較小中等最小

物理 vs 邏輯備份

比較面向物理備份邏輯備份
速度快（直接複製檔案）慢（需解析資料）
跨版本相容通常不支援支援
部分還原困難容易（可選擇表）
一致性保證需快照支援交易內一致

應用場景

場景一：電商平台的備份策略

  • 每日凌晨進行完整備份至 AWS S3
  • 每 15 分鐘進行 WAL 歸檔以實現 PITR
  • 跨區域複製至災備站點
  • 每月進行一次恢復演練驗證備份可用性

場景二：金融系統的災難復原

  • RPO < 1 分鐘、RTO < 15 分鐘
  • 使用同步複製（Synchronous Replication）確保零資料遺失
  • 主備自動切換（Automatic Failover）搭配健康檢查
  • 定期進行災難復原演練（DR Drill）

場景三：多租戶 SaaS 的備份隔離

  • 租戶層級的備份與還原能力
  • 使用邏輯備份實現單一租戶資料匯出
  • 自助式還原功能減少運維負擔
  • 備份資料依租戶加密，確保資料隔離

常見陷阱

  1. 未驗證備份可用性：備份存在但無法恢復是最常見的問題
  2. 忽略備份加密：備份檔案被竊取等同資料外洩
  3. RPO/RTO 目標不明確：缺乏明確指標導致備份策略不當
  4. 未考慮相依服務：僅備份資料庫卻忽略設定檔、密鑰等
