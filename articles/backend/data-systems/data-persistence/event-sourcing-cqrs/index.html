Event Sourcing 與 CQRS 是兩個經常搭配使用的架構模式，它們從根本上改變了我們對資料儲存與查詢的思維方式。Event Sourcing 將狀態變更記錄為不可變事件序列，而 CQRS 將讀寫操作分離為獨立模型，兩者結合能構建出高擴展性與高審計性的系統。

核心概念

Event Sourcing

Event Sourcing 的核心理念是：不直接儲存當前狀態，而是儲存導致狀態變更的所有事件。

  • 事件（Event）：已發生的不可變事實，如 OrderCreated、PaymentReceived
  • 事件流（Event Stream）：按時間排序的事件序列，代表一個聚合的完整歷史
  • 快照（Snapshot）：定期儲存的聚合狀態，加速重建過程
  • 投影（Projection）：從事件流衍生出的讀取模型

CQRS（Command Query Responsibility Segregation）

CQRS 將系統的讀取與寫入操作分離為兩個獨立的模型：

  • Command（命令）：改變系統狀態的操作，不回傳資料
  • Query（查詢）：讀取系統狀態的操作，不改變狀態
  • Write Model：針對寫入優化的資料模型
  • Read Model：針對查詢優化的資料模型，可有多個針對不同查詢場景

事件的特性

  1. 不可變性：事件一旦記錄就不可修改或刪除
  2. 時序性：事件有明確的發生順序
  3. 因果性：事件之間可能存在因果關係
  4. 完整性：事件序列包含系統狀態演變的完整歷史

Event Store 設計

Event Store 是 Event Sourcing 的核心儲存元件：

  • 追加寫入（Append-Only）：只新增不修改
  • 樂觀並發控制：透過版本號避免衝突
  • 訂閱機制：支援即時推送新事件給投影處理器
  • 分片策略：依聚合 ID 分片確保事件順序

關鍵比較

特性傳統 CRUDEvent Sourcing
儲存內容當前狀態事件歷史
歷史追溯需額外審計表天然支援
查詢效能直接查詢需建立投影
儲存空間較小較大（可壓縮）
除錯能力需日誌可重播事件
複雜度低高

CQRS vs 傳統架構

面向單一模型CQRS
讀寫模型共用分離
擴展性受限讀寫獨立擴展
一致性強一致最終一致
複雜度低中到高
適用場景CRUD 為主讀寫比例懸殊

應用場景

場景一：金融交易系統

  • 每筆交易記錄為不可變事件
  • 帳戶餘額透過事件重播計算
  • 完整的審計軌跡滿足合規要求
  • 可回溯任意時間點的帳戶狀態

場景二：電商訂單系統

  • 訂單生命週期以事件序列表達：OrderPlaced → PaymentConfirmed → Shipped → Delivered
  • 不同投影支援不同查詢需求（客戶訂單列表、庫存統計、財務報表）
  • 支援複雜的補償邏輯（取消、退款）

場景三：協作編輯平台

  • 每個編輯操作記錄為事件
  • 支援無限撤銷/重做
  • 版本歷史天然可用
  • 支援多人衝突解決

導入注意事項

  1. 最終一致性：讀取模型可能有短暫延遲，需向使用者說明
  2. 事件版本化：事件結構變更需考慮向後相容
  3. 事件風暴（Event Storming）：建議先以工作坊形式發掘領域事件
  4. 避免過度使用：簡單 CRUD 場景不需要引入此模式
