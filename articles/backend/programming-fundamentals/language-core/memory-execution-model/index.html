記憶體與執行模型是理解程式語言底層行為的關鍵。無論使用哪種語言，程式碼最終都需要在記憶體中配置資料並在 CPU 上執行。本文深入探討記憶體配置策略、值語意與參考語意的差異，以及不同語言的執行模型。

核心概念

記憶體區域劃分

程式執行時的記憶體空間通常被劃分為以下區域：

記憶體區域用途生命週期配置速度
程式碼段 (Text)儲存機器指令程式整個生命週期N/A
資料段 (Data)全域變數、靜態變數程式整個生命週期編譯期決定
堆疊 (Stack)區域變數、函式呼叫函式呼叫期間極快 (O(1))
堆積 (Heap)動態配置的物件手動或 GC 管理較慢

堆疊 vs 堆積

堆疊與堆積是最重要的兩個記憶體區域：

特性堆疊 (Stack)堆積 (Heap)
配置方式自動（LIFO）手動或 GC
配置速度極快（移動指標）較慢（搜尋空閒塊）
大小限制有限（通常 1-8 MB）受系統記憶體限制
碎片化無可能有
存取速度快（快取友善）較慢（間接存取）
執行緒安全天然安全（每執行緒一份）需要同步機制

值語意與參考語意

語意描述語言範例影響
值語意變數持有資料本身的副本Go (struct), Rust, C複製開銷，但安全
參考語意變數持有對資料的參考Java (物件), Python共享狀態，潛在副作用
混合語意基本型別值語意，物件參考語意Java, C#需注意差異

架構總覽

函式呼叫與堆疊框架

每次函式呼叫都會在堆疊上建立一個框架（Stack Frame）：

高位址
┌─────────────────────┐
│    呼叫者的框架       │
├─────────────────────┤
│    回傳地址           │
│    前一個框架指標      │
│    區域變數 n         │
│    區域變數 ...       │
│    區域變數 1         │
│    參數 n            │
│    參數 ...          │
│    參數 1            │
├─────────────────────┤  ← 堆疊指標 (SP)
│    (可用空間)         │
└─────────────────────┘
低位址

各語言的記憶體模型

語言堆疊配置堆積配置特殊機制
C區域變數malloc/free完全手動管理
C++區域變數、RAIInew/delete, smart ptrRAII + 智慧指標
Java基本型別所有物件GC (G1, ZGC)
Go逃逸分析決定逃逸分析決定GC + 逃逸分析
Rust預設Box, Vec 等所有權 + 借用
Python(CPython 內部)所有物件引用計數 + GC

逃逸分析 (Escape Analysis)

Go 語言使用逃逸分析決定變數配置在堆疊還是堆積：

情況配置位置原因
變數不離開函式堆疊安全，效能更好
變數被回傳堆積函式結束後仍需存在
變數被閉包捕獲堆積生命週期超出函式
大型變數堆積避免堆疊溢出
介面型別堆積（通常）動態分發需要

關鍵比較

參數傳遞機制

機制描述語言效果
傳值 (Pass by Value)複製整個值C, Go (基本型別)修改不影響原始值
傳參考 (Pass by Reference)傳遞變數的別名C++ (&), Rust (&mut)可直接修改原始值
傳物件參考 (Pass by Object Reference)複製參考/指標Java, Python可修改物件內容，不可重新指派
移動語意 (Move)轉移所有權Rust原始變數不再可用

記憶體對齊與快取效率

概念影響最佳化策略
快取行 (Cache Line)64 bytes 為單位載入資料局部性
偽共享 (False Sharing)多核心競爭同一快取行填充對齊
記憶體對齊CPU 存取效率結構體欄位排序
指標追蹤 (Pointer Chasing)快取未命中使用連續記憶體

不同語言的物件佈局

語言物件佈局額外開銷
C struct欄位連續排列僅對齊填充
C++ classvtable 指標 + 欄位8 bytes (虛擬函式)
Java object物件頭 + 欄位12-16 bytes
Python objectPyObject 頭 + 屬性字典數十 bytes
Go struct欄位連續排列僅對齊填充

效能敏感場景的記憶體策略

場景建議策略原因
高頻交易預配置記憶體池避免配置延遲
遊戲引擎ECS + 連續記憶體快取友善

現代語言的記憶體安全保證

保證等級語言機制

大部分安全Go, JavaGC + 邊界檢查
