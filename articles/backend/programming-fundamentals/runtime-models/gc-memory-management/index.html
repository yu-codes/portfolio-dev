垃圾回收與記憶體管理

垃圾回收（Garbage Collection, GC）是現代程式語言自動管理記憶體的核心機制。理解 GC 的工作原理對於寫出高效能的後端系統至關重要。本文深入探討各種 GC 演算法的設計哲學、效能特性與調優策略。

核心概念

記憶體管理的三種策略

策略代表語言優點缺點
手動管理C, C++精確控制、無暫停容易出錯
引用計數Swift, CPython, Rust (Arc)即時釋放循環引用、計數開銷
追蹤式 GCJava, Go, C#自動、安全STW 暫停
所有權系統Rust編譯期決定、零成本學習曲線陡峭

GC 演算法分類

演算法原理STW 暫停記憶體碎片吞吐量
標記-清除 (Mark-Sweep)標記存活物件，清除未標記長有高
標記-整理 (Mark-Compact)標記後移動物件消除碎片長無中
複製收集 (Copying)將存活物件複製到新區域中無高（浪費空間）
分代收集 (Generational)按物件年齡分區處理短（Minor GC）依實作高
並發收集 (Concurrent)與應用程式同時執行極短依實作中高
增量收集 (Incremental)分多次小步驟完成多次極短依實作中

分代假說 (Generational Hypothesis)

大多數物件的壽命很短（嬰兒死亡率高）：

代特性GC 頻率演算法
年輕代 (Young Gen)新建立的物件高頻 (Minor GC)複製收集
老年代 (Old Gen)存活多次 GC 的物件低頻 (Major GC)標記-整理
永久代/元空間類別資訊等極少標記-清除

Go GC 設計

Go 使用並發三色標記清除演算法：

顏色含義狀態
白色未訪問可能是垃圾
灰色已發現但未掃描子節點待處理
黑色已完全掃描確認存活

Go GC 特性：

特性描述
並發執行大部分工作與應用同時進行
寫入屏障確保並發標記的正確性
非分代不區分年輕/老年代
STW 暫停< 1ms (通常 < 100μs)
GOGC 參數控制 GC 觸發閾值

Python 記憶體管理

CPython 使用引用計數 + 循環偵測：

機制功能觸發條件
引用計數即時釋放無引用物件引用數降為 0
循環偵測器偵測循環引用配置計數達閾值
世代收集分三代管理各代有不同閾值
記憶體池小物件快速配置小於 512 bytes

GC 暫停時間比較

GC最大暫停時間堆積大小影響適合場景
Go GC< 1ms幾乎無關低延遲服務
G1 (Java)可設定目標 (如 200ms)有關通用
ZGC (Java)< 1ms無關大堆積低延遲

吞吐量 vs 延遲

GC 類型吞吐量延遲記憶體開銷
高吞吐 (Parallel)高高（長暫停）低
低延遲 (ZGC/Go)中高極低中高
引用計數中穩定中（計數開銷）
手動管理最高無 GC 暫停最低

GC 調優場景

場景問題調優方向
低延遲服務GC 暫停影響 P99使用低延遲 GC (ZGC/Go)
批次處理吞吐量不足使用 Parallel GC，增大堆積
記憶體受限OOM 頻繁減小堆積，調整 GC 頻率
大物件多Full GC 頻繁調整大物件閾值
短期物件多Minor GC 頻繁增大 Young Gen

減少 GC 壓力的策略

策略方法語言範例

避免裝箱避免基本型別轉物件Java generics
