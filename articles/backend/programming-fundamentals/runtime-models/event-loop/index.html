事件迴圈機制

事件迴圈（Event Loop）是單執行緒非同步程式設計的核心機制。它透過不斷地輪詢事件並分發到對應的處理器，實現了在單一執行緒上高效處理大量並發 I/O 操作的能力。本文深入解析事件迴圈的運作原理與各平台的實作差異。

核心概念

事件迴圈的基本模型

事件迴圈的核心是一個無限迴圈，持續做以下事情：

階段動作說明
1. 輪詢 (Poll)檢查是否有就緒的事件使用 OS 提供的多工器
2. 分發 (Dispatch)將事件交給對應回呼呼叫註冊的處理函式
3. 執行 (Execute)執行回呼函式同步執行直到完成
4. 回到步驟 1繼續下一輪如此反覆

事件驅動 vs 多執行緒

特性事件驅動多執行緒
並發模型協作式搶佔式
記憶體使用低（無執行緒堆疊）高（每執行緒 MB 級）
CPU 利用單核多核
程式複雜度回呼/Promise鎖/同步
適合場景I/O 密集CPU 密集
延遲可預測性取決於回呼執行時間取決於排程

架構總覽

Node.js 事件迴圈（libuv）

Node.js 的事件迴圈由 libuv 驅動，包含多個階段：

階段處理內容範例
TimerssetTimeout, setInterval 到期的回呼計時器觸發
Pending Callbacks系統操作的回呼（如 TCP 錯誤）延遲的系統回呼
Idle, Prepare內部使用libuv 內部
Poll新的 I/O 事件、執行 I/O 回呼網路/檔案 I/O
ChecksetImmediate 回呼立即執行
Close Callbacksclose 事件回呼socket.on(‘close’)

微任務佇列（在每個階段之間）：

佇列優先級內容
process.nextTick最高nextTick 回呼
Promise microtask高Promise.then/catch/finally
階段回呼依階段各階段的巨任務

事件迴圈的效能特性

場景事件迴圈表現原因
10K 連接、輕量操作優秀低記憶體、無切換開銷
CPU 密集計算差阻塞迴圈
長時間回呼差其他事件延遲
短時間大量 I/O優秀批次處理事件
混合 I/O + 計算需要 worker隔離計算到其他執行緒

常見陷阱

陷阱原因解決方案
回呼中的同步阻塞阻塞整個事件迴圈移到 Worker Thread
微任務飢餓持續產生微任務限制巢狀深度
未處理的 rejectionPromise 錯誤被忽略全域錯誤處理
Timer 精確度不保證精確觸發不依賴精確計時
記憶體洩漏閉包持有大物件注意引用生命週期

效能最佳化策略

策略方法效果
避免阻塞所有 I/O 使用非同步 API維持回應性
拆分計算大計算拆分為小塊避免長時間阻塞
Worker 隔離CPU 密集任務移到 Worker不影響主迴圈
批次處理合併多個小操作減少系統呼叫
