I/O 模型決定了程式如何與外部設備（磁碟、網路）交互。選擇正確的 I/O 模型是構建高效能網路伺服器的基礎。本文系統性地比較五種經典 I/O 模型，並探討現代作業系統提供的先進 I/O 機制。

核心概念

五種經典 I/O 模型

根據 Stevens 的《Unix Network Programming》，I/O 操作分為兩個階段：

  1. 等待資料就緒
  2. 將資料從核心複製到使用者空間
模型等待階段複製階段特點
阻塞 I/O阻塞阻塞最簡單
非阻塞 I/O非阻塞（輪詢）阻塞需要輪詢
I/O 多工阻塞在 select/poll阻塞可同時監控多 FD
信號驅動 I/O非阻塞（信號通知）阻塞較少使用
非同步 I/O (AIO)非阻塞非阻塞完全非阻塞

阻塞 vs 非阻塞 vs 非同步

特性阻塞 (Blocking)非阻塞 (Non-blocking)非同步 (Async)
呼叫行為等待完成立即回傳立即回傳
完成通知函式回傳即完成需輪詢或多工回呼/事件通知
資料複製核心→使用者（同步）核心→使用者（同步）核心→使用者（非同步）
程式複雜度低中高
效能低（一個連接一執行緒）中高

同步 vs 非同步（POSIX 定義）

類型定義包含的模型
同步 I/O資料複製階段會阻塞呼叫者阻塞、非阻塞、多工、信號驅動
非同步 I/O整個過程完全不阻塞僅 POSIX AIO / io_uring

epoll 的兩種觸發模式

模式觸發條件特點使用方式
水平觸發 (LT)只要就緒就通知簡單、不會遺漏預設模式
邊緣觸發 (ET)狀態變化時才通知高效、但需一次讀完搭配非阻塞

比較：

特性LT (水平觸發)ET (邊緣觸發)
通知頻率多次（直到處理）一次（狀態變化）
系統呼叫較多較少
程式複雜度低高（需迴圈讀取）
效能較低較高
使用者預設、較安全Nginx、高效能伺服器

io_uring：新一代 I/O

特性epollio_uring
模型就緒通知完成通知
系統呼叫每次操作一次批次提交
資料複製使用者→核心→設備可零拷貝
支援操作網路 I/O網路 + 檔案 + 其他
效能高更高 (30-80% 提升)
複雜度中高

應用場景

根據場景選擇 I/O 模型

場景推薦模型理由
少量連接阻塞 I/O + 多執行緒簡單直觀
中等連接 (< 1000)poll/select跨平台
高併發連接epoll/kqueue高效能
檔案 I/Oio_uring / 執行緒池檔案 I/O 本質上阻塞
超高效能io_uring批次 + 零拷貝
跨平台libuv封裝平台差異

C10K/C10M 問題

問題瓶頸解決方案
C10K (萬連接)select/poll 效能epoll/kqueue
C100K (十萬連接)記憶體、排程開銷事件驅動 + 協程
C1M (百萬連接)系統參數限制核心調優 + io_uring
C10M (千萬連接)核心開銷Kernel bypass (DPDK)
