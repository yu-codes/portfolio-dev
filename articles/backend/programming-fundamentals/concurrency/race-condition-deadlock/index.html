競態條件（Race Condition）與死結（Deadlock）是並發程式設計中最棘手的兩類問題。它們難以重現、難以除錯，且往往在高負載環境下才會暴露。本文深入分析這兩類問題的本質、偵測方法與系統性的預防策略。

核心概念

競態條件 (Race Condition)

競態條件發生在多個執行緒同時存取共享資料，且至少有一個是寫入操作時：

類型描述範例
Read-Modify-Write讀取、修改、寫回之間被插入counter++ 非原子操作
Check-Then-Act檢查條件後到行動之間狀態改變if (file.exists()) file.read()
Time-of-Check-Time-of-UseTOCTOU 漏洞權限檢查後資源被替換

死結的四個必要條件 (Coffman Conditions)

死結必須同時滿足以下四個條件才會發生：

條件描述打破方式
互斥 (Mutual Exclusion)資源同一時間只能被一方使用使用無鎖資料結構
持有並等待 (Hold and Wait)持有資源的同時等待其他資源一次取得所有資源
不可搶佔 (No Preemption)資源不能被強制釋放支援超時放棄
循環等待 (Circular Wait)形成環形等待鏈固定取鎖順序

同步原語

原語功能適用場景效能
Mutex互斥存取保護共享資料中
RWLock讀寫分離鎖讀多寫少讀高寫中
Semaphore限制並發數量資源池中
Condition Variable等待特定條件生產者-消費者中
Atomic無鎖原子操作計數器、旗標高
Channel通訊同步任務傳遞中
Barrier等待所有方到達階段同步低

架構總覽

競態條件的分析模型

執行緒 A                    執行緒 B
────────                   ────────
1. 讀取 counter = 5
                           2. 讀取 counter = 5
3. counter + 1 = 6
                           4. counter + 1 = 6
5. 寫入 counter = 6
                           6. 寫入 counter = 6

結果: counter = 6 (預期 7)

死結的等待圖

執行緒持有等待
Thread ALock 1Lock 2
Thread BLock 2Lock 1

形成循環等待：A → Lock2 → B → Lock1 → A

活鎖 vs 死結 vs 飢餓

問題描述特徵解決方式
死結 (Deadlock)互相等待，完全停止CPU 使用率低打破四條件之一
活鎖 (Livelock)持續嘗試但無進展CPU 使用率高引入隨機退避
飢餓 (Starvation)某執行緒永遠得不到資源部分執行緒無進展公平排程策略

關鍵比較

鎖策略比較

策略優點缺點適用場景
粗粒度鎖簡單安全並發度低低競爭場景
細粒度鎖高並發複雜、易死結高效能需求
無鎖 (Lock-free)最高效能極難正確實作極致效能
樂觀鎖 (CAS)讀多寫少高效高競爭時自旋浪費衝突少
悲觀鎖保證一致性可能造成等待衝突頻繁

各語言的並發安全機制

語言競態條件防護死結防護工具
Rust編譯期保證 (所有權 + Send/Sync)無內建保護Miri, ThreadSanitizer
Go-race 旗標無內建保護go run -race
Javasynchronized + volatileJVM 可偵測jstack, VisualVM
C++std::mutex + atomic無內建保護ThreadSanitizer
PythonGIL (部分保護)仍可能死結threading 模組

樂觀並發 vs 悲觀並發

特性樂觀 (Optimistic)悲觀 (Pessimistic)
假設衝突很少發生衝突經常發生
機制CAS / 版本號加鎖
讀取效能高中
寫入衝突重試等待
適用場景讀多寫少寫入頻繁
資料庫對應MVCCSELECT FOR UPDATE

應用場景

常見競態條件場景與解法

場景問題解決方案
單例模式雙重建立Double-check locking / sync.Once
計數器更新遺失Atomic 操作
快取失效快取擊穿分散式鎖 / singleflight
帳戶轉帳餘額不一致交易 + 行級鎖
設定重載讀到部分更新Copy-on-Write
連接池超額分配Semaphore

死結預防策略

策略方法代價
固定加鎖順序所有執行緒按相同順序取鎖需要全域規劃
超時機制tryLock with timeout可能活鎖
一次取得同時取得所有需要的鎖降低並發度
無鎖設計使用 CAS / 不可變資料實作複雜
層級鎖按層級取鎖限制彈性

偵測與除錯工具

工具用途語言
ThreadSanitizer (TSan)競態條件偵測C/C++, Go
Helgrind鎖使用問題C/C++ (Valgrind)
go run -race競態條件偵測Go
jstack執行緒狀態轉儲Java
Miri未定義行為偵測Rust
py-spy執行緒分析Python

最佳實踐

實踐描述
最小化共享狀態減少需要同步的資料
偏好不可變資料不可變資料天然執行緒安全
使用高層抽象Channel, Actor 而非原始鎖
保持鎖的粒度小鎖住的區域越小越好
避免鎖中做 I/O減少持鎖時間
文件化鎖的順序防止未來維護者引入死結
