執行緒與行程模型

深入比較執行緒與行程的差異、各種並發模型的設計哲學，以及在不同場景下的最佳選擇

執行緒與行程是作業系統提供的兩種核心並發機制。理解它們的差異、開銷與適用場景，是設計高效能後端系統的基礎。本文系統性地比較各種並發模型，並探討現代語言如何在此基礎上建構更高層的抽象。

核心概念

行程 vs 執行緒 vs 協程

特性行程 (Process)執行緒 (Thread)協程 (Coroutine)
記憶體空間獨立共享共享
建立開銷高 (~MB)中 (~MB stack)低 (~KB)
切換開銷高（需切換頁表）中（核心排程）極低（使用者空間）
通訊方式IPC (管道、Socket)共享記憶體直接存取
隔離性強（崩潰不影響他人）弱（可能影響整個行程）弱
排程方式OS 核心OS 核心使用者空間/執行時
數量限制數百至數千數千數十萬至數百萬

執行緒的類型

類型描述比例範例
核心執行緒 (Kernel Thread)由 OS 管理與排程1:1Linux pthread
使用者執行緒 (User Thread)由使用者空間管理N:1Green Thread
混合模型多個使用者執行緒對應核心執行緒M:NGo goroutine
虛擬執行緒由 VM 管理的輕量執行緒M:NJava Virtual Thread

行程間通訊 (IPC) 機制

機制速度複雜度適用場景
管道 (Pipe)快低父子行程通訊
具名管道 (Named Pipe)快低不相關行程通訊
共享記憶體最快高（需同步）大量資料交換
訊息佇列中中解耦合通訊
Socket較慢中跨機器通訊
信號 (Signal)快高簡單通知

關鍵比較

1:1 vs M:N 執行緒模型

特性1:1 模型M:N 模型
實作語言Rust, C, Java (傳統)Go, Erlang, Java (Loom)
排程器OS 核心語言執行時
切換成本~1-10μs~100ns-1μs
可建立數量數千數十萬至百萬
系統呼叫直接可能需要特殊處理
除錯難度中等較高
利用多核自然需要排程器支援

共享記憶體 vs 訊息傳遞

特性共享記憶體訊息傳遞
效能高（直接存取）較低（序列化開銷）
安全性低（需手動同步）高（天然隔離）
複雜度高（鎖管理）中（訊息設計）
擴展性受限（單機）好（可分散式）
除錯困難（競態條件）較易（追蹤訊息流）
代表Java synchronizedGo channel, Erlang

多行程 vs 多執行緒架構

架構優點缺點適用場景
多行程 (fork)隔離性強、穩定記憶體開銷大Nginx worker, Chrome
多執行緒共享資料方便一個崩潰全毀資料庫連接池
多行程+多執行緒兼顧兩者架構複雜Apache
單執行緒+事件驅動簡單、高效無法利用多核Node.js, Redis
執行緒池避免頻繁建立需調參Web 伺服器

應用場景

選擇並發模型的決策樹

條件推薦模型原因
I/O 密集 + 高連接數非同步/協程低開銷，高並發
CPU 密集 + 需要並行多執行緒/多行程利用多核
需要強隔離多行程崩潰不擴散
分散式系統Actor/訊息傳遞天然適合網路
低延遲要求共享記憶體 + 無鎖最小開銷
需要簡單性CSP (Go)語言層級支援

實際系統的並發架構

系統架構並發策略
Nginx多行程 + 事件驅動Worker 行程 + epoll
Redis單執行緒 + 事件驅動避免鎖開銷
PostgreSQL多行程每連接一個行程
MySQL多執行緒執行緒池
Node.js單執行緒 + WorkerEvent Loop + Worker Threads
Go HTTP ServerGoroutine per requestM:N 排程
