非同步程式設計是現代後端開發的核心技能。在 I/O 密集型的網路服務中，非同步模型能大幅提升系統吞吐量。本文深入探討非同步程式設計的演進、各語言的實作方式，以及正確使用非同步的關鍵原則。

核心概念

同步 vs 非同步 vs 並行

概念描述比喻
同步 (Synchronous)等待操作完成才繼續排隊等候一個窗口
非同步 (Asynchronous)發出請求後做其他事點餐後等叫號
並行 (Parallel)多個操作同時執行多個窗口同時服務
並發 (Concurrent)多個任務交替執行一個人同時處理多件事

非同步程式設計的演進

階段模式代表問題
第一代回呼 (Callback)Node.js (早期)回呼地獄
第二代Promise/FutureES6, Java鏈式但仍複雜
第三代async/awaitES2017, Python 3.5+語法糖，接近同步
第四代結構化並發Kotlin, Swift生命週期管理

非同步的本質：非阻塞 I/O

非同步程式設計的核心是避免執行緒在等待 I/O 時被阻塞：

操作類型同步方式非同步方式效率差異
網路請求執行緒等待回應註冊回呼，繼續處理100-1000x
檔案讀取執行緒等待磁碟通知機制10-100x
資料庫查詢連接佔用等待連接池 + 非同步10-50x
定時任務sleep 阻塞計時器回呼∞

async/await 的運作機制

async/await 在編譯時被轉換為狀態機：

async function fetchData() {     編譯器轉換為
    let a = await fetch(url1);   ──────────→  狀態機
    let b = await fetch(url2);               State 0: 發起 fetch(url1)
    return a + b;                            State 1: 收到 a，發起 fetch(url2)
}                                            State 2: 收到 b，回傳 a + b
狀態動作下一狀態
Initial開始執行到第一個 awaitSuspended_1
Suspended_1收到第一個結果Suspended_2
Suspended_2收到第二個結果Completed
Completed回傳結果-

關鍵比較

Promise/Future vs async/await

特性Promise/Futureasync/await
可讀性鏈式呼叫，較複雜接近同步風格
錯誤處理.catch() 鏈try/catch
除錯堆疊追蹤不完整較好的堆疊追蹤
組合Promise.all, Promise.race可搭配使用
學習曲線中等較低

非同步模型比較

模型優點缺點適用場景
回呼簡單直接回呼地獄、錯誤處理困難簡單的單次操作
Promise可組合、鏈式仍然較冗長需要組合多個操作
async/await可讀性最佳可能隱藏並行機會通用非同步場景
Goroutine極低開銷、自然併發不同思維模型高併發服務
Actor隔離性強、容錯複雜度較高分散式系統

錯誤處理策略

語言同步錯誤非同步錯誤注意事項
JavaScripttry/catchtry/catch + .catch()未處理的 rejection
Pythontry/excepttry/except in async忘記 await 導致靜默失敗
RustResult<T, E>Result<T, E> (相同)編譯期強制處理
Goerror 回傳值error 回傳值 (相同)Channel 傳遞錯誤

應用場景

何時使用非同步

場景是否適合非同步原因
HTTP API 伺服器✅ 非常適合大量 I/O 等待
檔案批次處理✅ 適合I/O 等待
CPU 密集計算❌ 不適合需要真正的並行
即時通訊✅ 非常適合大量連接等待
影像處理❌ 不適合CPU 密集型
資料庫操作✅ 適合網路 I/O 等待
