GitOps 是一種以 Git 儲存庫作為系統期望狀態唯一真實來源的運維方法論。透過將所有配置宣告式地儲存在 Git 中，並使用自動化工具持續將實際狀態與期望狀態同步，GitOps 帶來了更高的可審計性、一致性與恢復能力。

核心概念

GitOps 的四大原則

原則說明實踐
宣告式系統狀態以宣告方式定義YAML/HCL 描述期望狀態
版本化所有變更都在 Git 中追蹤Git commit = 變更記錄
自動拉取系統自動同步期望狀態Controller 持續比對並修復
持續調和自動修正偏離偵測到差異即自動修復

Push vs Pull 模型

維度Push（傳統 CI/CD）Pull（GitOps）
觸發方式CI 工具主動推送變更Controller 拉取期望狀態
權限模型CI 需要目標環境的權限Controller 在目標環境內運行
安全性憑證暴露在 CI 環境憑證不離開目標環境
偏離修復需人工或排程修復自動持續調和
審計追蹤CI 日誌Git 歷史

GitOps 的工作流程

開發者 → Pull Request → Code Review → Merge
                                        │
                                        ▼
                               Git Repository
                             (期望狀態：宣告式配置)
                                        │
                                        ▼
                              GitOps Controller
                          (偵測變更、比對實際狀態)
                                        │
                                        ▼
                              Kubernetes Cluster
                           (實際狀態趨向期望狀態)

架構總覽

GitOps 儲存庫結構

結構模式說明優點缺點
單一 Repo應用程式碼和配置在同一 Repo簡單耦合度高
雙 Repo應用和配置分開管理關注點分離管理兩個 Repo
多環境目錄用目錄區分環境直觀配置重複
Overlay 模式Base + 環境覆蓋DRY、維護性好學習曲線

建議的目錄結構：

infrastructure-repo/
├── base/               ← 共用的基礎配置
│   ├── deployment.yaml
│   ├── service.yaml
│   └── kustomization.yaml
├── overlays/
│   ├── dev/            ← 開發環境覆蓋
│   ├── staging/        ← 預備環境覆蓋
│   └── production/     ← 生產環境覆蓋
└── apps/
    ├── app-a/
    └── app-b/

漸進式交付與 GitOps

結合 GitOps 實現安全的發布：

  1. 變更配置中的映像版本 → Git commit
  2. ArgoCD 偵測到變更 → 開始同步
  3. Argo Rollouts 執行金絲雀策略 → 逐步切換流量
  4. 監控指標通過 → 自動推進
  5. 指標異常 → 自動回滾到上一個 commit

GitOps 的挑戰與解決

挑戰說明解決方式
配置漂移有人直接修改叢集啟用自動修復、RBAC 限制
Secret 管理敏感資訊不能存 GitSealed Secrets/External Secrets
環境差異不同環境的配置管理Kustomize overlays
大型 Repo同步速度慢分割 Repo、路徑過濾
回滾衝突revert 可能有衝突保持配置簡潔、減少合併
