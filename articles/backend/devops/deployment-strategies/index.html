部署策略

部署策略與版本發布

部署策略（Deployment Strategies）決定了新版本如何從構建產物轉變為正在服務使用者的生產系統。一個良好的部署策略必須在發布速度、風險控制與使用者體驗之間取得平衡。本篇將詳細介紹主流的部署模式——藍綠部署、金絲雀發布與滾動更新，分析各自的優缺點與適用場景。

核心概念

部署策略的三大模式為：藍綠部署（Blue-Green Deployment）、金絲雀發布（Canary Release）、以及滾動更新（Rolling Update）。

藍綠部署維護兩套完全相同的生產環境——藍色（當前版本）與綠色（新版本）。部署時先將新版本部署至綠色環境並完成驗證，然後透過負載均衡器或 DNS 一次性切換所有流量至綠色環境。若出現問題，可立即切回藍色環境實現秒級回滾。

金絲雀發布是一種漸進式發布策略，將新版本先部署給一小部分使用者（如 5%），觀察關鍵指標（錯誤率、延遲、業務指標）一段時間，確認正常後逐步擴大流量比例（5% → 25% → 50% → 100%）。如果偵測到異常，立即將流量全部回退至舊版本。

滾動更新是 Kubernetes 的預設部署策略。它逐個（或逐批）替換舊版本的 Pod 為新版本，同時確保始終有一定數量的 Pod 在提供服務。透過 maxSurge 和 maxUnavailable 參數控制更新的並行度與可用性保證。

架構總覽

各部署策略的流程可概括如下：

藍綠部署流程：

  1. 綠色環境部署新版本
  2. 在綠色環境執行冒煙測試與驗證
  3. 切換負載均衡器指向綠色環境
  4. 監控一段時間確認穩定
  5. 舊藍色環境保留作為回滾備援，或釋放資源

金絲雀發布流程：

  1. 部署新版本至少量實例（Canary Pod）
  2. 路由小比例流量至 Canary
  3. 收集並比對 Canary 與 Baseline 的指標
  4. 自動化分析判定（透過 Flagger、Argo Rollouts 等工具）
  5. 通過則擴大比例，失敗則自動回滾

滾動更新流程：

  1. 建立新版本 Pod
  2. 等待新 Pod 通過 Readiness Probe
  3. 將舊版本 Pod 從 Service 中移除
  4. 終止舊版本 Pod
  5. 重複直到所有 Pod 更新完成

進階的部署架構還包含：Feature Flag（功能旗標）允許在不重新部署的情況下控制功能可見性；A/B 測試則在部署層面支援多版本並行，根據使用者屬性路由至不同版本。

應用場景

電商平台大促前發布：在雙十一或黑色星期五前的版本發布，採用藍綠部署確保能在秒級回滾。綠色環境先以模擬流量進行壓力測試，確認效能達標後再切換。

SaaS 產品漸進式發布：B2B SaaS 產品使用金絲雀發布，先對內部員工帳號開放（Dogfooding），再擴展至友善客戶（Beta Users），最後全量發布。每一階段都搭配自動化指標監控。

微服務頻繁迭代：團隊每日多次部署的微服務採用 Kubernetes 滾動更新作為基線策略，搭配 Readiness/Liveness Probe 確保服務可用性。對於核心服務則升級為 Argo Rollouts 金絲雀模式。

資料庫相關變更：涉及 Schema Migration 的部署需特別注意向後相容性。常見做法是先部署相容新舊 Schema 的應用版本，再執行遷移，最後部署僅相容新 Schema 的版本（Expand-Contract Pattern）。

跨區域部署：全球化服務的部署通常採用區域級金絲雀策略——先在流量最小的區域部署新版本，逐步擴展至其他區域，確保任何區域的問題都能在影響大量使用者前被發現。
