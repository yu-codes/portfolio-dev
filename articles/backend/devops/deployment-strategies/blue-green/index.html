藍綠部署（Blue-Green Deployment）是一種透過維護兩套完全相同的生產環境來實現零停機發布的部署策略。在任何時刻，只有一個環境（藍或綠）接收生產流量，另一個用於準備新版本。部署新版本時，透過流量切換實現瞬間生效。

核心概念

藍綠部署的運作原理

階段藍環境（當前）綠環境（新版）流量指向
初始狀態運行 v1.0閒置→ 藍
部署新版繼續服務部署 v1.1→ 藍
測試新版繼續服務內部驗證→ 藍
切換流量待命接管服務→ 綠
確認穩定可隨時回滾正常服務→ 綠
清理/準備升級為下次使用持續服務→ 綠

核心優勢

優勢說明
零停機流量切換是瞬間的
快速回滾切回舊環境即可（秒級）
完整測試新版可在生產級環境完整驗證
風險隔離新版問題不影響線上使用者
可預測發布過程標準化、可重複

流量切換機制

機制切換速度實作方式
DNS 切換分鐘級（TTL）修改 DNS A 記錄
負載均衡切換秒級修改後端伺服器群組
路由規則秒級Nginx/Envoy 路由更新
Kubernetes Service秒級修改 selector
CDN 切換分鐘級修改 origin

資料庫的挑戰

藍綠部署最大的挑戰在於資料庫的處理：

方案策略優點缺點
共享資料庫兩套環境共用同一資料庫簡單、無資料同步問題Schema 變更受限
雙寫切換前短暫期間雙寫資料一致實作複雜
向下兼容遷移Schema 變更需向下兼容安全限制遷移類型
獨立資料庫各自擁有資料庫完全隔離資料同步困難

關鍵比較

藍綠 vs 金絲雀 vs 滾動更新

維度藍綠部署金絲雀發布滾動更新
資源需求2x（雙倍環境）1x + 少量新版1x + 漸進替換
切換速度瞬間（全量切換）漸進式漸進式
回滾速度極快（切回）快（停止放量）中（重新部署）
驗證方式切換前完整驗證小流量驗證逐步驗證
風險全量切換風險低（只影響小部分）中
成本高（雙倍基礎設施）低最低
複雜度中高低

不同場景的適用性

場景推薦策略原因
需要瞬間切換藍綠全量即時切換
資源有限滾動更新不需額外資源
高風險變更金絲雀小流量驗證
資料庫遷移藍綠 + 兼容遷移可整體驗證
需要快速回滾藍綠秒級回滾

應用場景

雲端環境的藍綠部署

在 AWS/GCP 等雲端平台的實作方式：

平台實作方式切換機制
AWS兩個 Auto Scaling GroupALB Target Group 切換
GCP兩個 Instance GroupURL Map 切換
K8s兩個 DeploymentService selector 切換
Serverless兩個版本/別名Alias 路由切換

藍綠部署檢查清單

切換前的驗證項目：

檢查項方法通過標準
功能測試自動化 E2E100% 通過
效能驗證負載測試符合 SLA
資料遷移驗證查詢資料完整
配置正確比對配置與預期一致
連通性健康檢查所有依賴可達
監控就緒儀表板確認指標正常收集
