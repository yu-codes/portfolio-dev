滾動更新（Rolling Update）是最常用的零停機部署策略，透過逐步將舊版本的實例替換為新版本，確保系統在整個更新過程中始終保持可用。Kubernetes 的 Deployment 資源預設即採用滾動更新策略。

核心概念

滾動更新的運作方式

初始狀態: [v1] [v1] [v1] [v1] [v1]  ← 5 個舊版本 Pod

第一步:   [v1] [v1] [v1] [v1] [v2]  ← 啟動 1 個新版，關閉 1 個舊版
第二步:   [v1] [v1] [v1] [v2] [v2]  ← 繼續替換
第三步:   [v1] [v1] [v2] [v2] [v2]
第四步:   [v1] [v2] [v2] [v2] [v2]
完成:     [v2] [v2] [v2] [v2] [v2]  ← 全部更新完成

關鍵參數

參數說明預設值影響
maxSurge超出期望副本數的最大值25%更多=更快，但需更多資源
maxUnavailable不可用的最大副本數25%更多=更快，但容量降低
minReadySeconds新 Pod 就緒後等待時間0越長越安全，但更慢
progressDeadlineSeconds更新進度超時600超時則標記為失敗

參數組合效果

maxSurgemaxUnavailable行為特徵適用場景
25%25%平衡速度與資源通用（預設）
100%0%先擴後縮（類似藍綠）不容忍任何容量下降
0%25%先縮後擴資源受限
10最保守，一次替換一個高風險變更
50%50%快速更新低風險、需快速完成

架構總覽

滾動更新與健康檢查的協作

新 Pod 啟動
     │
     ▼
[Startup Probe]    ← 等待應用完成初始化
     │ 通過
     ▼
[Readiness Probe]  ← 確認可接收流量
     │ 通過
     ▼
加入 Service 端點  ← 開始接收流量
     │
     ▼
[minReadySeconds]  ← 額外等待確認穩定
     │ 通過
     ▼
繼續替換下一個 Pod

優雅關閉流程

步驟動作目的
1Pod 被標記為 Terminating開始關閉流程
2從 Service Endpoints 移除不再接收新請求
3發送 SIGTERM通知應用準備關閉
4執行 preStop hook自定義清理邏輯
5等待 terminationGracePeriodSeconds給時間處理剩餘請求
6發送 SIGKILL強制結束

關鍵比較

滾動更新 vs 其他策略的取捨

維度滾動更新藍綠金絲雀
額外資源少量（maxSurge）2x少量
更新速度漸進瞬間漸進
回滾速度需要時間（重新部署）瞬間快速
版本共存更新期間新舊共存切換前後不共存長期共存
適用複雜度低中高
K8s 原生支援是（預設）需額外工具需額外工具

常見問題與對策

問題症狀原因對策
更新卡住Pod 一直處於 Pending資源不足調整資源限制或節點
頻繁重啟CrashLoopBackOff應用啟動失敗檢查配置和依賴
流量失敗更新期間錯誤率高Readiness 配置不當調整探測參數
回滾失敗舊版也無法啟動環境已變（如 DB 遷移）向下兼容遷移
超時progressDeadlineExceeded更新速度太慢調整並行度

版本兼容性要求

滾動更新期間新舊版本同時運行，因此：

兼容面向要求範例
API 兼容新版 API 向下兼容新增欄位可選，不刪除欄位
資料格式新舊版能讀寫同一資料JSON 新增欄位需有預設值
資料庫Schema 向下兼容先加欄位，後刪欄位（分開部署）
訊息格式消費者能處理新舊格式版本標記在訊息中

應用場景

Kubernetes 滾動更新最佳實踐

配置建議：

實踐原因
設定合理的 Readiness Probe確保新 Pod 真正就緒才接流量
設定 minReadySeconds (10-30s)避免 Pod 啟動後立即崩潰
配置 PodDisruptionBudget確保最小可用數量
使用 preStop hook優雅處理進行中的請求
設定資源 requests/limits確保調度成功
合理的 terminationGracePeriod給足時間完成清理

零停機的資料庫遷移

在滾動更新中安全地進行資料庫變更：

步驟動作版本相容
1新增欄位（允許 NULL）v1 忽略新欄位，v2 可使用
2部署 v2（寫入新欄位）v1 仍可正常運作
3回填舊資料背景任務
4設定 NOT NULL（如需要）v1 已完全下線
5移除舊欄位（如需要）確認不再使用

大規模叢集的滾動更新策略

對於數百個 Pod 的服務：

策略適用情況參數建議
分批更新需要觀察每批效果maxSurge:10%, maxUnavailable:0
快速更新低風險變更maxSurge:50%, maxUnavailable:25%
分區域更新多區域部署每個區域獨立 Deployment
時間窗口業務低峰期配合 CronJob 或手動觸發
