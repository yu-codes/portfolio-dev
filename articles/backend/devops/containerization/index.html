容器化

Docker 與 Kubernetes

容器化（Containerization）是現代應用程式部署的標準方式，它將應用程式及其所有相依性打包為輕量、可攜帶的標準化單元。Docker 讓容器化變得普及，Kubernetes 則解決了大規模容器叢集的編排問題。本篇將探討容器技術的核心原理、編排平台的架構設計，以及生產環境中的最佳實踐。

核心概念

容器化技術的三大核心為：Docker 容器化、Kubernetes 編排、以及容器編排進階模式。

Docker 利用 Linux Kernel 的 Namespace 與 Cgroup 技術實現程序級別的隔離，相較於虛擬機器（VM），容器共享宿主機的 Kernel，啟動時間從分鐘級降至秒級，資源開銷也大幅減少。Dockerfile 定義了映像的建構步驟，每一層（Layer）都被快取，實現增量建構。

Kubernetes（K8s） 是容器編排的事實標準。它提供了聲明式的 API，讓使用者描述「期望狀態」，系統自動調和（Reconcile）至該狀態。核心抽象包括 Pod（最小部署單元）、Service（服務發現與負載均衡）、Deployment（無狀態應用管理）與 StatefulSet（有狀態應用管理）。

容器編排進階模式涵蓋 Service Mesh（如 Istio）、自動擴縮（HPA/VPA/KEDA）、以及多叢集管理（Federation/Fleet）。這些模式解決了微服務間的通訊治理、資源效率優化與跨地域部署等進階需求。

架構總覽

Kubernetes 叢集的架構分為控制平面與資料平面：

控制平面（Control Plane）：

  • API Server：所有操作的入口，處理 RESTful API 請求
  • etcd：分散式鍵值儲存，保存叢集的所有狀態
  • Scheduler：決定 Pod 應部署至哪個節點
  • Controller Manager：執行各類控制迴圈（Deployment、ReplicaSet 等）

資料平面（Data Plane）：

  • kubelet：每個節點上的代理，管理本機 Pod 生命週期
  • kube-proxy：處理網路規則與服務路由
  • Container Runtime：實際執行容器（containerd、CRI-O）

容器網路模型（CNI）確保每個 Pod 獲得唯一 IP，Pod 間可直接通訊。常見 CNI 實作有 Calico、Cilium 與 Flannel。

儲存方面，Persistent Volume（PV）與 Persistent Volume Claim（PVC）抽象了底層儲存實作，支援動態配置（Dynamic Provisioning）。

應用場景

微服務部署：每個微服務打包為獨立容器，透過 Kubernetes Deployment 管理副本數量，Service 提供穩定的存取端點，Ingress 處理外部流量路由。

開發環境標準化：使用 Docker Compose 定義本地開發環境，確保所有開發者使用相同的資料庫版本、中介軟體配置與網路拓撲，消除「在我機器上可以跑」的問題。

批次運算任務：Kubernetes Job 與 CronJob 管理批次任務的執行，自動處理重試、並行度控制與失敗通知。搭配 KEDA 可根據佇列深度動態擴縮 Worker。

藍綠部署與金絲雀發布：Kubernetes 的 Service 與 Deployment 天然支援流量切換，搭配 Istio 的流量管理能力，可實現基於百分比、Header 或使用者屬性的精細化流量路由。

多租戶平台：利用 Namespace、ResourceQuota 與 NetworkPolicy 實現租戶間的資源隔離與網路隔離，搭配 RBAC 控制存取權限，構建安全的多租戶 PaaS 平台。
