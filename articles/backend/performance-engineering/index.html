效能工程是一門系統性的工程實踐，涵蓋從設計階段的效能建模，到開發階段的程式碼優化，再到營運階段的持續調校。與事後的「效能調校」不同，效能工程強調在整個軟體生命週期中主動管理效能。本文探討效能工程的方法論、核心技術和實踐策略。

核心概念

延遲與吞吐量

效能工程的兩大核心指標：

延遲 (Latency)：完成單一操作所需的時間。關鍵認知是延遲不是單一數值，而是一個分佈。P50（中位數）反映典型體驗；P99 反映長尾使用者的體驗。很多系統在 P50 看起來良好，但 P99 存在嚴重問題。

吞吐量 (Throughput)：系統在單位時間內能處理的操作數量。吞吐量與延遲之間存在根本性的取捨——當系統接近飽和時，排隊效應會導致延遲急劇上升。

理解 Little’s Law 是效能分析的基礎：L = λ × W（系統中的平均請求數 = 到達速率 × 平均停留時間）。

效能剖析 (Profiling)

效能剖析是識別效能瓶頸的核心工具。主要類型包括：

  • CPU Profiling：識別消耗最多 CPU 時間的程式碼路徑。火焰圖（Flame Graph）是最直觀的視覺化方式。
  • Memory Profiling：追蹤記憶體分配模式、洩漏和 GC 壓力。
  • I/O Profiling：分析磁碟讀寫和網路呼叫的等待時間。
  • Lock Profiling：識別競爭激烈的同步點。

資源管理

有效的資源管理包括：

  • 連線池 (Connection Pooling)：重用資料庫、HTTP 等連線，避免重複建立的開銷
  • 記憶體管理：物件池、零拷貝 (Zero-Copy)、堆外記憶體
  • 執行緒模型：事件驅動 vs. 執行緒池，根據工作負載特性選擇
  • 快取策略：多層級快取（L1/L2/分散式）、失效策略、預熱機制

架構總覽

效能工程的實踐框架涵蓋以下階段：

效能需求定義：在設計階段明確效能目標。例如「API P99 延遲 < 200ms」「系統支撐 10,000 RPS」。沒有明確目標的效能優化容易陷入過早優化的陷阱。

效能建模：透過理論分析（排隊論模型、Amdahl’s Law）預測系統在不同負載下的行為，在實作前識別架構層級的瓶頸。

負載測試：模擬真實流量模式驗證系統效能。關鍵測試類型包括：

  • 基準測試（Benchmark）：固定負載下的效能基準線
  • 負載測試（Load Test）：逐步增加負載直到目標量
  • 壓力測試（Stress Test）：超出預期負載，觀察系統退化行為
  • 浸泡測試（Soak Test）：長時間運行，檢測記憶體洩漏等慢性問題

持續效能監控：在生產環境中持續追蹤關鍵效能指標，及早發現退化。效能迴歸應該像功能缺陷一樣被追蹤和修復。

優化迭代：基於數據識別瓶頸、實施優化、驗證效果的循環過程。

應用場景

場景一：API 延遲優化

某 API 的 P99 延遲從 150ms 退化到 800ms。透過 CPU Profiling 發現序列化操作佔比異常高；進一步分析發現是新增欄位導致 JSON 序列化量暴增。解決方案包括：啟用響應壓縮、實作欄位篩選（只返回客戶端需要的欄位）、以及考慮遷移至 Protocol Buffers。

場景二：資料庫效能調校

電商系統在促銷期間資料庫 CPU 飆升至 95%。慢查詢日誌顯示多個未使用索引的全表掃描。短期對策是加入適當索引和查詢改寫；中期策略包括讀寫分離和查詢快取；長期規劃則考慮資料分片（Sharding）。

場景三：負載測試揭露隱患

上線前的負載測試顯示系統在 5,000 RPS 時回應時間急劇上升。原因是連線池大小設定不當——預設的 10 個資料庫連線在高並發下成為嚴重瓶頸。調整連線池大小並加入連線排隊超時後，系統能穩定處理 15,000 RPS。

場景四：GC 調校

Java 服務出現週期性的延遲尖峰。GC 日誌分析顯示 Full GC 每小時發生一次，每次暫停 2-3 秒。解決方案包括：調整堆大小分配比例、切換至低暫停 GC 算法（ZGC/Shenandoah）、以及減少物件分配率（透過物件池和避免不必要的裝箱）。
