服務模板（Service Templates）是平台工程中將最佳實踐轉化為可重用起始點的關鍵機制。透過精心設計的模板，組織能夠確保每個新服務從第一天起就遵循安全、效能和可維護性的標準，同時大幅縮短從零到可部署的時間。

核心概念

服務模板的價值

面向沒有模板有模板
新服務建立時間數天到數週數分鐘到數小時
一致性各團隊實踐差異大統一的結構和標準
安全基線容易遺漏安全配置內建安全最佳實踐
維護性各服務結構不同統一結構易於跨團隊協作
合規性需額外檢查內建合規要求

模板包含的內容

類別具體內容目的
程式碼結構目錄結構、入口點、配置統一專案組織
CI/CD 配置管線定義、建置腳本開箱即用的部署能力
容器化Dockerfile、compose標準化的容器映像
測試測試框架、範例測試測試文化的起點
監控健康檢查、指標暴露可觀測性就緒
文件README、ADR 模板文件習慣
安全依賴掃描、lint 規則安全護欄

模板的粒度設計

粒度說明範例靈活性
完整服務端到端可運行的服務REST API 服務模板低
元件可組合的功能模組認證模組、日誌模組高
配置片段特定工具的配置CI 管線片段最高
基礎設施IaC 資源模板RDS + VPC 模板中

模板工具比較

工具整合平台模板格式動作能力適用
Backstage TemplatesBackstageYAML + Nunjucks豐富（Git、CI/CD）完整 IDP
Cookiecutter獨立Jinja2僅生成檔案簡單腳手架
Yeoman獨立JavaScript互動式前端專案
copier獨立Jinja2生成+更新需模板更新
Projen獨立TypeScript持續管理維護專案配置

模板管理模式

模式更新方式優點缺點
生成即分離生成後完全獨立簡單、自由無法統一更新
持續同步模板更新自動推播統一維護可能衝突
抽象層共用程式庫封裝更新透明耦合度高
PR 通知模板更新時發 PR受控更新需人工合併

模板 vs 程式庫 vs 框架

維度模板共用程式庫內部框架
使用時機專案建立時開發過程中持續依賴
更新方式生成後獨立版本升級框架升級
耦合度低（一次性）中高
標準化程度起始結構一致特定功能一致全面一致
靈活性高中低

典型的服務模板設計

REST API 服務模板應包含：

組件內容可配置項
應用框架Express/Spring Boot/Gin語言選擇
API 規範OpenAPI 3.0 定義服務名稱
資料存取ORM + 遷移工具資料庫類型

CI/CD標準管線配置部署目標
測試測試框架 + 範例覆蓋率要求

模板的版本管理策略

策略適用場景實作
SemVer 標記模板有明確版本Git tag
分支策略不同穩定度main/beta/experimental
變更日誌追蹤模板改動CHANGELOG.md
相容性矩陣模板與平台的相容文件化
