內部開發者平台與開發體驗

平台工程是近年來最受矚目的工程實踐之一，它致力於建構內部開發者平台（Internal Developer Platform, IDP），讓應用開發者能夠以自助服務的方式完成基礎設施操作，而無需深入了解底層複雜性。本文探討平台工程的核心理念、架構設計及實施策略。

核心概念

開發者體驗 (Developer Experience, DevEx)

開發者體驗是平台工程的最終目標。好的開發者體驗意味著：

  • 認知負載最小化：開發者不需要了解 Kubernetes YAML 的細節就能部署服務
  • 黃金路徑 (Golden Path)：為常見任務提供預設且最佳的方式，但不強制限制
  • 快速回饋循環：從程式碼變更到看到結果的時間越短越好
  • 自助服務：開發者能自行完成環境建立、服務部署、監控設定等操作
  • 文件即平台：好的平台本身就是文件，操作直覺到不需要額外說明

衡量 DevEx 的框架包括 SPACE 指標（Satisfaction、Performance、Activity、Communication、Efficiency）和 DORA 指標（部署頻率、變更前置時間、失敗率、恢復時間）。

內部開發者平台 (IDP)

IDP 不是單一產品，而是一組工具、流程和自助服務能力的組合。典型的 IDP 包含：

  • 服務目錄 (Service Catalog)：所有服務的中央註冊表，包含擁有者、文件、依賴關係
  • 基礎設施即程式碼 (IaC) 模板：預設的最佳實踐模板，開發者填入參數即可
  • CI/CD 管線模板：標準化的建構與部署流程
  • 環境管理：一鍵建立開發、測試、預生產環境
  • 可觀測性整合：自動配置監控、日誌和追蹤

平台即產品

平台工程的核心理念是將內部平台視為產品——開發者是客戶。這意味著：

  • 需要產品經理思維：理解使用者需求、排定優先序
  • 持續蒐集回饋並迭代改善
  • 量測採用率和滿意度
  • 提供卓越的文件和支援
  • 避免「強制使用」——好的平台是開發者主動選擇使用的

架構總覽

現代 IDP 的參考架構通常包含五個層次：

開發者介面層 (Developer Interface)：開發者與平台互動的入口。包括開發者入口網站（如 Backstage）、CLI 工具、IDE 外掛和 ChatOps 整合。

協作層 (Orchestration Layer)：將開發者的高階意圖轉換為底層基礎設施操作。平台編排器（如 Crossplane、Humanitec）負責解析宣告式配置並協調各資源供應器。

整合層 (Integration Layer)：連接各種基礎設施服務和工具鏈。包括 CI/CD 系統、容器編排、雲端服務、監控系統、秘密管理等。

資源層 (Resource Layer)：實際的基礎設施資源——計算、儲存、網路、資料庫、訊息佇列等。可以是雲端託管服務或自建叢集。

安全與合規層 (Security & Compliance)：貫穿所有層的安全控制——身份認證、存取控制、策略引擎（如 OPA）、審計日誌和合規檢查。

應用場景

場景一：新服務建立自助化

開發者透過開發者入口選擇「建立新服務」模板，填入服務名稱、程式語言、需要的資料庫類型等參數。平台自動執行：建立 Git Repository、配置 CI/CD Pipeline、部署開發環境、設定監控和告警、註冊至服務目錄。整個過程在 5 分鐘內完成。

場景二：環境一致性

過去開發、測試和生產環境常常存在配置差異，導致「在我的機器上沒問題」的窘境。透過 IDP 的環境模板，所有環境從同一份宣告式配置產生，確保一致性。開發者可以一鍵複製一套完整環境進行實驗。

場景三：安全左移

平台內建安全策略，在 CI/CD 階段自動進行容器映像掃描、依賴漏洞檢查、基礎設施配置合規性驗證。違反策略的變更無法通過 Pipeline，將安全檢查從事後轉為事前。

場景四：平台團隊的運作模式

平台團隊以雙披薩團隊規模（6-8 人）起步，使用產品管理方法運作。每季度進行使用者研究，識別開發者最大的痛點；使用 OKR 追蹤平台採用率和開發者滿意度；定期舉辦 Platform Days 展示新功能並蒐集回饋。
