監控系統

指標、儀表板與告警

監控是維持系統穩定性的核心實踐，它讓團隊能夠即時了解系統健康狀態、預測潛在問題，並在事故發生時快速回應。本文探討現代監控系統的設計理念、指標收集策略以及告警最佳實踐。

核心概念

指標收集 (Metrics Collection)

指標是對系統行為的數值化量測，具有時間序列的特性。與日誌不同，指標是高度壓縮的數據形式，適合長期保存與趨勢分析。核心指標類型包括：

  • 計數器 (Counter)：單調遞增的累計值，如 HTTP 請求總數、錯誤次數
  • 量規 (Gauge)：可升可降的瞬時值，如 CPU 使用率、記憶體佔用、佇列深度
  • 直方圖 (Histogram)：將觀測值分入多個桶中，用於計算分位數（P50、P95、P99）
  • 摘要 (Summary)：在客戶端計算分位數，適合不需要跨實例聚合的場景

黃金信號 (Golden Signals)

Google SRE 提出的四個黃金信號是監控任何系統的起點：

  1. 延遲 (Latency)：服務處理請求所需的時間
  2. 流量 (Traffic)：系統承受的請求量
  3. 錯誤率 (Errors)：失敗請求的比例
  4. 飽和度 (Saturation)：系統資源的使用程度

USE 方法

針對基礎設施資源（CPU、記憶體、磁碟、網路），Brendan Gregg 提出的 USE 方法提供了系統性的監控框架：

  • Utilization（使用率）：資源忙碌的時間比例
  • Saturation（飽和度）：排隊等待的工作量
  • Errors（錯誤）：錯誤事件的數量

RED 方法

針對微服務的監控，RED 方法聚焦於請求層面：

  • Rate（速率）：每秒處理的請求數
  • Errors（錯誤）：每秒失敗的請求數
  • Duration（持續時間）：請求的延遲分佈

架構總覽

現代監控系統的架構通常包含以下元件：

資料採集：透過 Pull（Prometheus 主動抓取）或 Push（應用主動推送至 Gateway）模式收集指標。Pull 模式更適合服務發現動態變化的環境，Push 模式適合短生命週期的任務。

時間序列資料庫 (TSDB)：專門為時間序列數據優化的儲存引擎。Prometheus 內建 TSDB 適合中小規模；Thanos 或 Cortex 提供水平擴展與長期儲存；VictoriaMetrics 以高壓縮比著稱。

查詢引擎：PromQL 是事實上的標準查詢語言，支援向量運算、聚合函數和子查詢。學會撰寫高效的 PromQL 是 SRE 的基本功。

視覺化與儀表板：Grafana 是最普及的開源視覺化工具，支援多種資料來源和豐富的面板類型。好的儀表板應該分層設計：總覽層顯示黃金信號，詳情層提供下鑽能力。

告警引擎：基於規則評估指標，在異常時觸發通知。Alertmanager 負責告警的分組、靜音、路由和降噪。

應用場景

場景一：容量規劃

透過長期趨勢分析（如過去 30 天的 CPU 使用率增長曲線），預測何時需要擴容。結合業務指標（用戶數增長率）可以更準確地預估資源需求。

場景二：告警策略設計

好的告警策略應該基於症狀而非原因。例如，直接監控「API 回應時間 P99 > 500ms 持續 5 分鐘」比監控「CPU > 80%」更能反映使用者體驗的惡化。多視窗告警（短視窗偵測突發、長視窗偵測慢性退化）可以減少誤報。

場景三：SLO 監控

透過定義 SLI（如成功率 = 成功請求數 / 總請求數）並設定 SLO 目標（如 99.9%），監控系統可以即時追蹤錯誤預算的消耗速率，協助團隊在可靠性與開發速度之間做出數據驅動的決策。

場景四：事故回應中的角色

在事故處理流程中，監控儀表板是第一個被查看的工具。良好設計的儀表板能讓值班人員在 60 秒內判斷問題的影響範圍和嚴重程度，大幅縮短平均偵測時間 (MTTD)。
