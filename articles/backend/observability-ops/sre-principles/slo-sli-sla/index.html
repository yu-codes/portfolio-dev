SLO（Service Level Objective）、SLI（Service Level Indicator）和 SLA（Service Level Agreement）構成了服務可靠性管理的核心框架。它們定義了「什麼是足夠可靠」，讓工程團隊能在可靠性與開發速度之間做出數據驅動的決策。

核心概念

SLI、SLO、SLA 的關係

概念定義受眾範例
SLI衡量服務品質的具體指標工程團隊成功請求比例
SLOSLI 的目標值工程+產品團隊成功率 ≥ 99.9%
SLA違反 SLO 時的商業後果客戶+法務未達 99.9% 賠償 10%

三者的層次關係：

SLA（商業承諾）
  └── 基於 SLO（內部目標，通常比 SLA 更嚴格）
        └── 衡量 SLI（實際度量指標）

常見 SLI 類型

SLI 類別衡量方式適用服務
可用性成功請求 / 總請求所有服務
延遲請求延遲 < 閾值的比例API 服務
品質未降級回應的比例有降級機制的服務
新鮮度資料更新及時的比例資料管線
正確性產生正確結果的比例計算/資料服務
吞吐量處理速率 ≥ 預期的時間比例批次處理
耐久性資料未丟失的比例儲存服務

錯誤預算（Error Budget）

錯誤預算 = 1 - SLO 目標

SLO 目標錯誤預算/月可容忍停機時間/月
99%1%7.3 小時
99.9%0.1%43.8 分鐘
99.95%0.05%21.9 分鐘
99.99%0.01%4.4 分鐘
99.999%0.001%26.3 秒

架構總覽

SLO 驅動的決策框架

錯誤預算充足
  → 加速功能開發
  → 嘗試風險較高的變更
  → 減少可靠性投資

錯誤預算耗盡
  → 暫停功能開發
  → 專注於可靠性改善
  → 減少部署頻率
  → 增加測試覆蓋

SLI 的量測架構

量測位置優點缺點範例
用戶端最接近使用者體驗數據收集困難RUM、行動 SDK
負載均衡器覆蓋所有流量不含應用細節ALB 日誌
應用層細粒度、可客製自身故障時無法量測Prometheus 指標
合成監控主動偵測不代表真實使用者Synthetic checks

關鍵比較

SLO 目標的選擇

考量因素選高目標（99.99%）選低目標（99.9%）
業務影響核心收入路徑非關鍵功能
使用者預期即時性要求高容許偶爾不可用
工程成本極高（冗餘、複雜度）中等
變更速度慢（錯誤預算少）快
適用支付系統、核心 API內部工具、報表系統

SLI 公式設計

SLI 類型公式時間窗口
可用性good_events / total_events × 100%滾動 28 天
延遲requests_under_threshold / total_requests × 100%滾動 28 天
品質non_degraded_responses / total_responses × 100%滾動 28 天

時間窗口的選擇

窗口優點缺點適用
滾動 7 天反應快波動大快速迭代服務
滾動 28 天平衡穩定和反應-推薦預設
日曆月對齊商業報告月初刷新預算SLA 報告
滾動 90 天極穩定反應太慢長期趨勢

層級服務SLISLO

錯誤預算策略範本

當錯誤預算耗盡時的升級動作：

預算剩餘狀態建議動作
> 50%健康正常開發節奏
25-50%警告增加變更審查嚴格度
10-25%危險暫停非關鍵變更
< 10%凍結停止所有非可靠性相關的變更
耗盡危機全力投入可靠性改善
