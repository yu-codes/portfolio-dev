密碼雜湊是保護使用者憑證的最後一道防線。即使資料庫被入侵，正確的密碼雜湊策略也能確保攻擊者無法輕易還原使用者的原始密碼。選擇合適的雜湊演算法和正確的實作方式，是每個處理使用者認證的系統必須重視的安全議題。

核心概念

為什麼不能明文儲存密碼

資料外洩事件頻繁發生，明文密碼的風險：

  • 資料庫被入侵時所有帳號即時暴露
  • 內部人員可直接查看使用者密碼
  • 使用者常在多個網站重複使用密碼，一處洩漏全面受害
  • 違反 GDPR 等隱私法規的安全要求

密碼雜湊的演進

時代方法安全性問題
遠古明文儲存無任何外洩即全部暴露
早期MD5/SHA-1極低彩虹表可秒破
中期加鹽 SHA-256低-中速度太快，可暴力破解
現代bcrypt/scrypt高專為密碼設計的慢雜湊
最新Argon2最高抗 GPU/ASIC，記憶體硬化

密碼雜湊的關鍵特性

一個安全的密碼雜湊方案必須具備：

特性說明目的
單向性無法從雜湊值反推原始密碼基本安全保證
加鹽（Salt）每個密碼使用唯一的隨機鹽值防止彩虹表攻擊
計算成本可調可控制雜湊的計算時間隨硬體進步提升強度
記憶體硬化需要大量記憶體抗 GPU/ASIC 平行攻擊
確定性相同輸入+相同鹽 = 相同輸出驗證時重現結果

架構總覽

密碼驗證流程

註冊流程：
密碼 → 產生隨機鹽 → 雜湊(密碼 + 鹽, 成本因子) → 儲存(雜湊值 + 鹽 + 參數)

驗證流程：
輸入密碼 → 讀取儲存的鹽和參數 → 雜湊(輸入 + 鹽, 相同參數) → 比較雜湊值
                                                                    ↓
                                                          相同 → 認證成功
                                                          不同 → 認證失敗

攻擊方式原理防禦手段

方面正確做法錯誤做法

應用場景

密碼雜湊升級策略

當需要從舊演算法遷移到新演算法時：

  1. 透明升級：使用者下次登入時，用新演算法重新雜湊
  2. 雙層雜湊：Argon2(舊的bcrypt雜湊值)，無需使用者操作
  3. 強制重設：通知使用者重設密碼（體驗較差）
策略使用者影響遷移速度安全性
透明升級無感知漸進式未登入的帳號仍用舊演算法
雙層雜湊無感知即時全面提升
強制重設需使用者操作取決於使用者最高

Pepper 的使用

Pepper 是與鹽值互補的額外保護：

  • Salt：與雜湊值一起儲存在資料庫，防彩虹表
  • Pepper：儲存在應用程式碼或 HSM 中，不在資料庫

即使資料庫被完整取得，沒有 Pepper 也無法離線破解。
