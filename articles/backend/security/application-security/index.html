核心概念

應用安全（Application Security）聚焦於程式碼層級的漏洞防護，是整個安全體系中最靠近業務邏輯的一層。多數資安事件的根因都源自應用層的弱點——不安全的輸入處理、錯誤的授權邏輯或敏感資料的不當暴露。

OWASP Top 10 是全球公認的 Web 應用安全風險清單，每隔數年更新一次。2021 年版本涵蓋了 Broken Access Control、Cryptographic Failures、Injection 等十大風險類別。理解這些風險並建立對應的防護機制，是每位後端開發者的基本功。

輸入驗證（Input Validation） 是應用安全的第一道防線。所有來自使用者的輸入——無論是 HTTP 請求參數、表單資料、檔案上傳還是 API 請求體——都必須被視為不可信資料，經過嚴格的驗證和清理（sanitization）後才能使用。驗證策略應採用白名單模式，明確定義合法輸入的格式，拒絕所有不符合規範的資料。

輸出編碼（Output Encoding） 與輸入驗證互補。根據資料的輸出上下文（HTML、JavaScript、SQL、URL），對特殊字元進行適當的編碼轉換，防止注入攻擊。例如在 HTML 上下文中，< 應被編碼為 &lt;，在 SQL 上下文中則使用參數化查詢。

安全標頭（Security Headers） 提供瀏覽器層級的防護。Content-Security-Policy（CSP）限制可載入的資源來源；X-Frame-Options 防止點擊劫持；Strict-Transport-Security 強制 HTTPS 連線。這些標頭構成了一層額外的防禦網。

│   ├── 請求大小限制
│   ├── Content-Type 檢查
│   ├── 參數型別與範圍驗證

│   └── 編碼正規化（Unicode、URL decode）
├── 邏輯層防護
│   ├── 存取控制（RBAC/ABAC）
│   ├── 業務邏輯漏洞防護
│   ├── 資源存取授權檢查
│   ├── 競態條件（Race Condition）防護
│   └── 批次操作限制
├── 輸出層防護
│   ├── 上下文感知的輸出編碼
│   ├── 敏感資料遮蔽
│   ├── 錯誤訊息控制
│   ├── 安全回應標頭
│   └── CORS 策略配置
└── 監控與偵測
    ├── 安全日誌記錄
    ├── 異常行為偵測
    ├── WAF 規則管理
    └── 漏洞掃描整合

防護策略需要在多層之間協調運作。僅靠前端驗證不足以保證安全，後端必須獨立進行完整的驗證；僅靠 WAF 也不能替代程式碼層級的安全編碼實踐。

靜態分析 vs 動態測試：SAST（靜態應用安全測試）能在開發期發現程式碼中的潛在漏洞，如 SQL 拼接、硬編碼密碼等。DAST（動態應用安全測試）則在運行時模擬攻擊，發現配置錯誤和邏輯漏洞。兩者互補，應同時在 CI/CD 管線中整合。

白名單 vs 黑名單：輸入驗證應優先使用白名單策略（明確允許什麼），而非黑名單（試圖阻擋所有惡意輸入）。黑名單容易被繞過，因為攻擊者總能找到未被覆蓋的變形手法。

應用場景

  1. RESTful API 安全：對每個端點實施嚴格的輸入驗證 schema（如 JSON Schema 或 Zod），配合 RBAC 確保使用者只能存取授權的資源。使用 API 版本控制避免破壞性變更暴露舊版本漏洞。

  2. 檔案上傳功能：驗證檔案類型（不僅是副檔名，還要檢查 magic bytes）、限制檔案大小、重新命名上傳檔案、儲存在獨立的非公開目錄，並對圖片進行重新編碼以移除可能嵌入的惡意內容。

  3. 使用者生成內容（UGC）：論壇、留言板等需要顯示使用者輸入的場景，必須嚴格實施 HTML 清理（如使用 DOMPurify）和 CSP 策略，防止儲存型 XSS 攻擊。

  4. 第三方整合：Webhook 接收端需驗證請求簽名，OAuth 回調需驗證 state 參數防止 CSRF，外部 API 回應需要驗證和清理後才能使用。

  5. 管理後台：通常擁有更高的權限和更敏感的操作，應實施更嚴格的存取控制、操作日誌和二次確認機制，並限制可存取的網路範圍。
