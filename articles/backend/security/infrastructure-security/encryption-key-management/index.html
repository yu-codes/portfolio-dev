加密與金鑰管理

加密是保護資料機密性的基礎技術，而金鑰管理則是加密系統安全性的核心。即使使用最強的加密演算法，若金鑰管理不當，整個加密體系也形同虛設。本文探討加密技術的選擇與金鑰管理的最佳實踐。

架構總覽

信封加密（Envelope Encryption）

┌─────────────────────────────────────────────────┐
│  KMS（金鑰管理服務）                             │
│  ┌─────────────┐                                │
│  │ Master Key  │ ← 受硬體保護，永不匯出          │
│  └──────┬──────┘                                │
│         │ 加密/解密                              │
│  ┌──────▼──────┐                                │
│  │  Data Key   │ ← 用於加密實際資料              │
│  │ (Encrypted) │                                │
│  └─────────────┘                                │
└─────────────────────────────────────────────────┘

加密資料 = Encrypt(明文, Data Key)
儲存：加密資料 + 加密後的 Data Key

信封加密的優勢：

  • Master Key 不離開 KMS，降低暴露風險
  • 每份資料可使用不同 Data Key
  • 金鑰輪換時只需重新加密 Data Key

欄位層級加密 vs 資料庫層級加密

維度欄位層級資料庫層級（TDE）
加密粒度特定欄位整個資料庫/表空間
應用層可見密文（需解密）透明（自動解密）
效能影響僅加密欄位有開銷全面但較小的開銷
保護對象防止 DBA 查看敏感欄位防止磁碟被竊
實作複雜度高（需修改應用程式）低（資料庫配置）
索引影響加密欄位無法直接索引無影響

密碼學常見錯誤

錯誤後果正確做法
自行實作加密演算法幾乎必定有漏洞使用經過驗證的程式庫
金鑰寫死在程式碼中版本控制暴露使用環境變數或 KMS
使用 ECB 模式模式可辨識使用 GCM 或 CTR
重複使用 Nonce/IV破壞加密安全性每次加密使用唯一值
忽略 MAC 驗證可能被篡改先驗證再解密
使用已棄用演算法已有已知攻擊使用現代演算法
