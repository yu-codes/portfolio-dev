流量防護

速率限制與防護策略

深入解析限流演算法與 DDoS 防護策略，探討如何透過流量管理確保服務在異常負載下仍維持可用性與穩定性。

核心概念

限流保護（Rate Limiting & Protection）是系統穩定性的最後防線。無論是惡意的 DDoS 攻擊、API 濫用，還是突發的流量洪峰，都可能耗盡系統資源導致服務中斷。有效的流量防護策略能在維持合法使用者體驗的同時，阻擋或緩解異常流量。

速率限制（Rate Limiting） 對單位時間內的請求數量設定上限。根據限制維度的不同，可以基於 IP 位址、使用者身份、API 金鑰或自定義屬性進行限速。限流不僅防止濫用，還能保護後端服務免受過載，確保資源公平分配。

限流演算法 是實現速率限制的核心。不同演算法在精確度、記憶體使用和實作複雜度之間各有取捨：

  • 固定視窗（Fixed Window）：將時間分為固定區間，每個區間獨立計數。實作簡單但有邊界突波問題（視窗交界處可能出現兩倍流量）。
  • 滑動視窗（Sliding Window）：結合固定視窗的效率和滑動日誌的精確度，使用加權計算消除邊界效應。
  • 令牌桶（Token Bucket）：以固定速率向桶中添加令牌，每次請求消耗一個令牌。允許短暫的突波流量（桶中有餘額時），同時維持長期平均速率。
  • 漏桶（Leaky Bucket）：以固定速率處理請求，多餘請求排隊等待。產出速率平穩但不允許突波，適合需要穩定下游流量的場景。

DDoS 防護 需要多層次的策略。網路層（L3/L4）攻擊如 SYN Flood 需要在網路邊緣處理；應用層（L7）攻擊如 HTTP Flood 則需要更智慧的分析來區分合法流量和攻擊流量。CDN 和雲端 DDoS 防護服務（如 Cloudflare、AWS Shield）提供大規模的吸收能力。

有效的流量防護需要在多個層次協調運作。邊緣層過濾明顯的惡意流量，閘道層實施精細的配額控制，服務層提供最後的自我保護機制。

集中式 vs 分散式限流：單節點部署使用本地計數器即可；分散式系統需要共享狀態（如 Redis）來維持全域一致的計數。分散式限流存在些微的不精確性（網路延遲導致），但透過滑動視窗等演算法可將誤差控制在可接受範圍。

硬限流 vs 自適應限流：固定閾值簡單但不能適應流量變化。自適應限流根據系統當前負載（CPU、延遲、錯誤率）動態調整閾值，在系統健康時允許更多流量，在壓力增加時主動收緊。Netflix 的 Concurrency Limits 是自適應限流的代表實作。

應用場景

  1. 公共 API 平台：實施分層配額（免費 100 req/min、付費 10K req/min），使用 API Key 或 OAuth Token 識別呼叫者。配合 429 Too Many Requests 回應和 Retry-After 標頭，幫助客戶端實施退避策略。

  2. 電商大促活動：預期流量洪峰時，前置排隊系統（Virtual Waiting Room）控制進入速率，配合動態擴容和非核心服務降級，確保核心交易流程的可用性。

  3. 登入防暴力破解：對單一帳號的登入嘗試實施嚴格限速（如 5 次/分鐘），失敗累積後觸發帳號鎖定或 CAPTCHA 挑戰。同時對單一 IP 的全域登入請求也設限，防止分散式暴力破解。

  4. 微服務間保護：上游服務對下游的呼叫量需要限制，防止某個服務的異常重試風暴壓垮下游。熔斷器在偵測到下游故障時快速失敗，避免級聯故障。

  5. 即時通訊系統：訊息發送頻率限制防止垃圾訊息氾濫，同時需要區分正常的群組聊天（合法的高頻率）和自動化的垃圾發送。行為分析模型可以更智慧地判斷。
