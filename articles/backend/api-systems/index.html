API 系統

API 設計、協定與治理

API（Application Programming Interface）是現代軟體系統之間溝通的橋樑。無論是前後端分離架構中的 HTTP API、微服務之間的 gRPC 呼叫，還是第三方平台的開放介面，API 系統的設計品質直接決定了系統的可維護性、可擴展性與開發者體驗。本章節將全面介紹 API 系統的核心面向，從通訊協定選擇到 API 設計原則、文件標準、認證授權機制以及契約治理策略。

核心概念

API 系統的核心目標是提供穩定、一致、易用且安全的服務介面。一個設計良好的 API 應具備以下特質：

  • 可預測性（Predictability）：遵循一致的命名慣例與回應格式，讓使用者能直覺理解
  • 向後相容性（Backward Compatibility）：新版本不應破壞既有客戶端的功能
  • 自描述性（Self-Descriptive）：透過良好的文件與錯誤訊息，降低整合成本
  • 安全性（Security）：實施適當的認證、授權與速率限制機制
  • 效能（Performance）：提供分頁、篩選、快取等機制以優化回應時間

API 的生命週期管理同樣重要，從設計、開發、測試、發布到棄用，每個階段都需要明確的流程與工具支持。API-First 設計理念主張在撰寫任何實作程式碼之前，先定義好 API 規格，這能有效促進前後端協作並提早發現設計問題。

架構總覽

一個完整的 API 系統架構通常包含以下組件：

客戶端 → API Gateway → 負載均衡 → 應用服務 → 資料層
                ↓
        認證/授權服務
                ↓
        速率限制/節流
                ↓
        日誌/監控/追蹤

API Gateway 扮演統一入口的角色，負責路由轉發、協定轉換、認證驗證、速率限制與回應快取。常見的 API Gateway 包括 Kong、AWS API Gateway、Nginx 與 Envoy。

服務網格（Service Mesh） 在微服務架構中提供服務間通訊的基礎設施層，處理負載均衡、故障恢復、指標收集與安全策略，代表性工具如 Istio 與 Linkerd。

API 管理平台 則提供完整的 API 生命週期管理能力，包括開發者入口、API 金鑰管理、使用量分析與計費整合。

選擇通訊協定時，需要綜合考量以下因素：目標使用者（內部 vs 外部）、效能需求、開發團隊技術棧、既有基礎設施以及未來擴展計畫。許多大型系統會同時採用多種協定——對外使用 REST 或 GraphQL，內部服務間使用 gRPC，即時功能使用 WebSocket。

應用場景

API 系統在不同場景中有不同的設計重點：

開放平台 API：如 Stripe、Twilio 等，需要極高的文件品質、穩定的版本策略、完善的 SDK 與沙箱環境。設計原則著重於開發者體驗（DX）。

微服務內部 API：強調效能與型別安全，通常採用 gRPC 搭配 Protocol Buffers。需要服務發現與熔斷機制配合。

BFF（Backend for Frontend）：為特定前端裝置（Web、Mobile、IoT）量身定制的 API 層，聚合多個後端服務的資料。

事件驅動 API：透過 Webhook、Server-Sent Events 或 WebSocket 提供即時通知能力，適用於訊息推送、狀態更新等場景。

資料 API：提供資料查詢與分析能力，如 GraphQL 允許客戶端精確指定所需欄位，避免過度取得資料（Over-fetching）。

子主題導覽

本 API 系統系列涵蓋五大核心主題：

  • 通訊協定（Protocols）：深入比較 REST、gRPC、GraphQL 與 WebSocket 的特性、優劣與適用場景
  • API 設計（API Design）：RESTful 設計原則、GraphQL Schema 設計、版本控制策略
  • API 文件（API Documentation）：OpenAPI/Swagger 規格、文件自動產生工具與最佳實踐
  • 認證與授權（Authentication & Authorization）：JWT、OAuth 2.0/OIDC、Session 管理機制
  • 契約治理（Contract Governance）：Schema 驗證、向後相容性檢查、API 生命週期管理
