向後相容性（Backward Compatibility）是 API 設計中最重要的原則之一。它確保新版本的 API 不會破壞既有客戶端的正常運作。本文深入探討向後相容性的判斷準則、常見破壞場景與系統性的維護策略。

核心概念

相容性的定義

方向定義保護對象範例
向後相容 (Backward)新版本可處理舊客戶端的請求舊客戶端新 API 接受舊格式
向前相容 (Forward)舊版本可處理新客戶端的請求新客戶端舊 API 忽略未知欄位
雙向相容 (Full)新舊版本互相相容所有人完全無感知升級

向後相容的層次

層次描述檢查方式
語法相容請求/回應格式不變Schema 比對
語意相容行為邏輯不變整合測試
效能相容回應時間不顯著變化效能測試
操作相容部署流程不變運維確認

常見破壞性變更

類別變更影響嚴重度
移除端點DELETE /users/deactivate404 錯誤嚴重
移除欄位回應少了 phone 欄位客戶端解析失敗嚴重
新增必填參數原本選填變必填400 錯誤嚴重
改變型別age: string → number解析錯誤嚴重
改變語意status 含義改變邏輯錯誤嚴重
改變預設值page_size 從 20 改 10行為差異中等
改變排序列表排序邏輯改變顯示差異輕微
改變錯誤格式錯誤回應結構改變錯誤處理失效中等

架構總覽

安全的 API 變更清單

變更類型安全性條件
新增端點✅ 安全不影響既有端點
新增選填欄位（回應）✅ 安全客戶端忽略未知欄位
新增選填參數（請求）✅ 安全有合理預設值
放寬驗證✅ 安全接受更多輸入
新增列舉值⚠️ 可能安全客戶端需處理未知值
加嚴驗證❌ 不安全舊請求可能被拒絕
移除端點❌ 不安全棄用流程
改變行為❌ 不安全需要版本化

相容性保證策略

策略方法適用場景
寬容輸入 (Tolerant Reader)忽略不認識的欄位回應處理
保守輸出 (Conservative Writer)只輸出已承諾的欄位回應生成
預設值填充新欄位有合理預設新增欄位
欄位棄用標記而非移除逐步移除
多態回應支援多種格式格式遷移

棄用流程

階段時間動作通知方式
宣布棄用T+0標記欄位/端點為 deprecated文件 + Changelog
回應標頭T+0加入 Deprecation 標頭HTTP Header
監控使用T+0~T+N追蹤仍在使用的客戶端日誌分析
通知客戶T+N/2主動聯繫使用者Email/通知
回傳警告T+N-M回應中加入警告Warning Header
移除T+N停止支援410 Gone

應用場景

各場景的相容性策略

場景策略棄用期理由
公開 API (外部)嚴格向後相容12-24 月無法控制客戶端
行動 App API支援 N-2 版本6-12 月App 更新有延遲
微服務 (內部)短期相容2-4 週可協調部署
前後端分離同步部署相容即時可同時部署
第三方整合長期相容12+ 月合約約束

維護相容性的最佳實踐

實踐說明
自動化相容性測試CI 中比較新舊 Schema
消費者測試用舊客戶端測新 API
變更日誌記錄所有 API 變更
棄用標記明確標記棄用的欄位/端點
使用量監控追蹤每個端點/欄位的使用情況
漸進式發布金絲雀部署驗證相容性
文件化承諾明確 SLA 中的相容性保證

應對破壞性變更的方案

方案描述適用情況
新端點提供新的 URL行為大改
欄位別名新舊欄位同時存在重新命名
版本分支URL 或標頭版本大版本升級
轉接層 (Adapter)舊格式轉新格式過渡期間
功能旗標依客戶端切換行為漸進遷移
協商格式Accept 標頭選擇格式多格式支援

相容性 SLA 範本

指標承諾
破壞性變更通知至少提前 90 天
棄用端點支援期至少 12 個月
安全修復不改變行為
錯誤修復不改變正常情況的行為
新功能純新增，不影響既有
效能變更不顯著降低 (< 10%)
