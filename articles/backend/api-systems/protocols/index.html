通訊協定

HTTP、gRPC 與即時通訊協定

通訊協定是分散式系統中服務之間交換資訊的基礎規範。選擇合適的通訊協定直接影響系統的效能、可靠性、開發效率與維護成本。從傳統的 HTTP/REST 到高效能的 gRPC、彈性查詢的 GraphQL，再到即時雙向的 WebSocket，每種協定都有其獨特的優勢與適用場景。本章節將全面分析各主流通訊協定的技術特性與工程實踐。

核心概念

通訊協定的選擇需要考量以下幾個核心維度：

通訊模式（Communication Pattern）：

  • 請求-回應（Request-Response）：客戶端發送請求、等待回應，最基本的同步模式
  • 串流（Streaming）：單向或雙向的持續資料流，適合即時資料傳輸
  • 發布-訂閱（Pub/Sub）：解耦生產者與消費者，支援一對多廣播
  • 事件驅動（Event-Driven）：基於事件觸發的非同步通訊

序列化格式（Serialization Format）：

  • JSON：人類可讀、廣泛支援，但體積較大且解析較慢
  • Protocol Buffers：二進制格式、體積小、解析快，但需要 Schema 定義
  • MessagePack：類 JSON 的二進制格式，兼顧效能與彈性
  • Avro：Schema 內嵌、支援 Schema Evolution，常用於大數據場景

傳輸層協定：

  • HTTP/1.1：文字協定、每次請求需建立連線（或 Keep-Alive 重用）
  • HTTP/2：二進制框架、多路復用、Header 壓縮、伺服器推送
  • HTTP/3（QUIC）：基於 UDP、消除隊頭阻塞、連線遷移支援
  • TCP/WebSocket：全雙工連線、低延遲、適合即時場景

架構總覽

在典型的後端系統中，不同層次會使用不同的通訊協定：

外部客戶端（瀏覽器/App）
    ↓ REST/GraphQL over HTTPS
API Gateway
    ↓ gRPC（內部通訊）
微服務叢集
    ↓ 事件（Kafka/NATS）
非同步處理服務
    ↓ WebSocket
即時通知服務 → 客戶端

北向流量（North-South）：客戶端到服務的流量，通常使用 REST 或 GraphQL over HTTPS，考量瀏覽器相容性與防火牆穿越。

東西向流量（East-West）：服務到服務的內部流量，可選擇 gRPC 以獲得更高效能與型別安全，或使用訊息佇列實現非同步解耦。

API Gateway 的協定轉換能力讓外部使用 REST 的請求在內部透明轉換為 gRPC 呼叫，結合兩者的優勢。

連線管理也是重要考量：HTTP/2 的多路復用能在單一 TCP 連線上並行處理多個請求，大幅減少連線建立的延遲。gRPC 基於 HTTP/2 天然繼承了這一優勢。

應用場景

高效能微服務通訊：使用 gRPC + Protocol Buffers，服務間呼叫延遲可降至 1-5ms。搭配服務網格（Istio）自動處理 mTLS、負載均衡與重試策略。適合對延遲敏感的計算密集型服務鏈。

公開 RESTful API：對外暴露標準 REST 介面，利用 HTTP 快取機制減少重複請求。搭配 CDN、ETag 與條件請求實現高效的資源分發。適合開放平台與第三方整合。

即時協作應用：WebSocket 建立持久雙向連線，伺服器可主動推送更新。搭配 Redis Pub/Sub 實現跨伺服器的訊息廣播。適合線上文件協作、遊戲、股票行情等場景。

彈性前端資料層：GraphQL 讓前端精確指定所需資料，避免 Over-fetching 與 Under-fetching。搭配 DataLoader 解決 N+1 問題，Apollo Federation 實現跨服務 Schema 組合。

IoT 裝置通訊：MQTT 協定專為低頻寬、高延遲環境設計，支援 QoS 等級與離線訊息保留。搭配 MQTT Broker（如 EMQX）管理大量裝置連線。

子主題導覽

本通訊協定系列深入探討四大主流協定：

  • REST：HTTP 方法語義、狀態碼使用、HATEOAS 實踐、快取策略與效能優化
  • gRPC：Protocol Buffers 語法、四種串流模式、攔截器、負載均衡與錯誤處理
  • GraphQL：Schema 設計、Resolver 模式、Subscription、Federation 與效能調校
  • WebSocket：連線生命週期、心跳機制、重連策略、擴展性與安全考量
