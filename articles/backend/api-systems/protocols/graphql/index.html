GraphQL 是 Facebook 於 2015 年開源的 API 查詢語言。它讓客戶端能精確指定所需的資料，解決了 REST API 中過度取得（Over-fetching）與不足取得（Under-fetching）的問題。本文深入探討 GraphQL 的設計理念、實作考量與效能最佳化。

核心概念

GraphQL vs REST 核心差異

特性RESTGraphQL
端點多個（每資源一個）單一端點
資料形狀伺服器決定客戶端決定
型別系統選填（OpenAPI）內建強型別
版本控制URL 或標頭版本欄位棄用（無版本）
過度取得常見不存在
不足取得需多次請求一次查詢
快取HTTP 快取需應用層快取
檔案上傳原生支援需額外處理

GraphQL 操作類型

操作用途HTTP 對應範例
Query讀取資料GET查詢使用者資料
Mutation修改資料POST/PUT/DELETE建立訂單
Subscription即時更新WebSocket接收新訊息

Schema 設計的核心元素

元素描述範例
Object Type物件型別定義type User { id: ID!, name: String }
Query Type查詢進入點type Query { user(id: ID!): User }
Mutation Type修改進入點type Mutation { createUser(…): User }
Input Type輸入參數型別input CreateUserInput { name: String! }
Enum列舉型別enum Role { ADMIN, USER }
Interface介面定義interface Node { id: ID! }
Union聯合型別union SearchResult = User | Post
Scalar標量型別scalar DateTime

架構總覽

GraphQL 執行流程

步驟動作說明
1. 解析 (Parse)將查詢字串轉為 AST語法檢查
2. 驗證 (Validate)對照 Schema 驗證型別檢查
3. 執行 (Execute)逐欄位呼叫 Resolver資料取得
4. 回應 (Response)組裝結果符合查詢形狀

Resolver 架構

層級Resolver 職責範例
根 Resolver查詢入口Query.user → 從 DB 取使用者
型別 Resolver型別欄位解析User.orders → 取使用者的訂單
標量 Resolver值轉換DateTime → ISO 格式化
預設 Resolver直接取屬性User.name → user.name

N+1 問題與解決方案

方案原理優點缺點
DataLoader批次 + 快取自動合併查詢需手動設定
Join MonsterSQL 查詢合併單一 SQL僅適用關聯式 DB
預載入 (Eager Loading)預先載入關聯減少查詢次數可能過度取得
Lookahead查看查詢樹決定載入精確實作複雜

效能最佳化策略

策略方法效果
查詢複雜度限制計算查詢深度/寬度防止惡意查詢
持久化查詢預先註冊查詢，僅傳 ID減少傳輸、安全
回應快取依查詢 + 變數快取減少重複計算
@defer / @stream分段回傳結果改善首次回應時間
DataLoader批次載入減少資料庫查詢
查詢白名單僅允許預定義查詢安全 + 效能

安全性考量

威脅描述防護
深度攻擊極深巢狀查詢消耗資源最大深度限制
寬度攻擊單層大量欄位複雜度計算
批次攻擊大量並行查詢請求限流
自省洩漏Schema 暴露所有型別生產環境停用自省
注入攻擊惡意查詢參數輸入驗證

應用場景

何時選擇 GraphQL

場景推薦度理由
多平台客戶端 (Web + Mobile)✅ 非常適合各平台取不同欄位
複雜關聯資料✅ 適合一次查詢取得所有關聯
快速迭代的前端✅ 適合無需等後端改 API
簡單 CRUD❌ 過度設計REST 即可
檔案上傳⚠️ 需額外處理非原生支援
高快取需求⚠️ 需額外設計HTTP 快取不適用
微服務聚合✅ 適合Federation/Stitching

GraphQL Federation（聯邦架構）

模式描述適用場景
Schema Stitching手動合併多個 Schema簡單場景
Apollo Federation宣告式 Schema 組合大型微服務
統一閘道單一 GraphQL 閘道統一入口

從 REST 遷移到 GraphQL

階段方式風險
包裝層GraphQL 呼叫 REST低（遞增）
混合部分用 GraphQL低
全面遷移替換所有 REST高
BFF 模式GraphQL 作為 BFF中

最佳實踐

實踐說明
Schema 先行設計先定義 Schema 再實作
使用 Input Type複雜參數用 Input Type 包裝
錯誤處理統一的錯誤格式，區分使用者錯誤與系統錯誤
分頁使用 Relay Cursor 規範
命名規範Query 用名詞，Mutation 用動詞
棄用策略使用 @deprecated 標註
