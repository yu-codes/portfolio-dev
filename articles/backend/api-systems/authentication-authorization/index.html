認證與授權

身份驗證與權限管理

認證（Authentication）與授權（Authorization）是 API 安全性的兩大基石。認證回答「你是誰？」的問題，授權回答「你能做什麼？」的問題。在現代分散式系統中，這兩個機制的正確實作攸關整個系統的資料安全與合規性。本章節將深入探討各種認證授權機制的原理、適用場景與實作考量。

核心概念

認證（Authentication, AuthN） 是驗證使用者或系統身份的過程。常見的認證因子包括：

  • 知識因子（Something you know）：密碼、PIN 碼、安全問題
  • 持有因子（Something you have）：手機 OTP、硬體安全金鑰、智慧卡
  • 固有因子（Something you are）：指紋、臉部辨識、虹膜掃描

多因素認證（MFA）結合兩個以上的因子，大幅提升安全性。

授權（Authorization, AuthZ） 則決定已認證的使用者可以存取哪些資源、執行哪些操作。常見的授權模型有：

  • RBAC（Role-Based Access Control）：基於角色的權限控制，如 Admin、Editor、Viewer
  • ABAC（Attribute-Based Access Control）：基於屬性的細粒度權限控制
  • ReBAC（Relationship-Based Access Control）：基於資源關係的權限控制（如 Google Zanzibar）
  • PBAC（Policy-Based Access Control）：使用策略語言（如 OPA/Rego）定義權限規則

在 API 系統中，認證通常透過以下方式實現：

Client → [帶上 Credentials] → API Gateway → [驗證身份] → 後端服務
                                    ↓
                            Token 驗證服務
                            (JWT 解碼 / Session 查詢 / OAuth 驗證)

架構總覽

現代認證授權系統的架構趨勢是集中化身份管理與去中心化決策執行：

身份提供者（Identity Provider, IdP）：集中管理使用者身份、認證流程與 Token 簽發。代表性解決方案包括 Keycloak、Auth0、AWS Cognito、Azure AD。

Token 服務：負責簽發、驗證與刷新存取 Token。JWT（JSON Web Token）是最常見的 Token 格式，支援無狀態驗證。

策略決策點（Policy Decision Point, PDP）：集中化的授權決策引擎，如 Open Policy Agent（OPA）。

策略執行點（Policy Enforcement Point, PEP）：分散在各服務中的權限檢查邏輯，攔截請求並詢問 PDP。

典型的 OAuth 2.0 + JWT 架構：

使用者 → 登入介面 → IdP（認證 + 發 Token）
                         ↓
         Access Token（JWT, 短效期）
         Refresh Token（加密, 長效期）
                         ↓
使用者 → [Bearer Token] → API Gateway → 驗證 JWT 簽章 → 後端服務

應用場景

SaaS 多租戶系統：每個租戶有獨立的使用者、角色與權限設定。使用 OAuth 2.0 搭配自訂 Claims（tenant_id）實現租戶隔離，結合 RBAC 管理租戶內部權限。

微服務間認證：Service-to-Service 通訊使用 mTLS（雙向 TLS）或服務帳戶 Token。Service Mesh 可自動處理憑證輪換與信任驗證。

第三方應用授權：使用 OAuth 2.0 Authorization Code Flow + PKCE，讓使用者安全地授予第三方應用有限的存取權限，而無需分享密碼。

零信任架構（Zero Trust）：不再依賴網路邊界作為信任基礎，每次存取都需要驗證身份、檢查裝置狀態與評估風險等級。

API Gateway 集中認證：將認證邏輯從各微服務抽離到 API Gateway 層統一處理，後端服務只需處理已驗證的請求與授權邏輯。
