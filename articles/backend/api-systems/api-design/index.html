API 設計

RESTful 與 API 設計原則

API 設計是後端工程中最關鍵的技能之一。一個設計精良的 API 能降低團隊溝通成本、減少整合錯誤、提升開發效率，並為系統的長期演進奠定堅實基礎。相反地，糟糕的 API 設計會導致文件與實作不一致、客戶端頻繁中斷、版本管理混亂等問題。本章節將深入探討 API 設計的核心原則、常見模式與最佳實踐。

核心概念

優秀的 API 設計遵循幾個基本原則：

一致性（Consistency）：整個 API 應採用統一的命名規則、回應格式與錯誤處理模式。例如，資源命名統一使用複數名詞（/users、/orders），回應格式統一包含 data、meta、errors 欄位。

最小驚訝原則（Principle of Least Astonishment）：API 的行為應符合使用者的直覺預期。HTTP 方法語義要正確——GET 不應有副作用、POST 用於建立資源、PUT 用於完整更新、PATCH 用於部分更新、DELETE 用於刪除。

資源導向設計（Resource-Oriented Design）：將系統中的實體建模為資源，每個資源有唯一的 URI 標識。操作透過標準 HTTP 方法表達，而非自定義動詞。

HATEOAS（Hypermedia as the Engine of Application State）：回應中包含相關資源的連結，讓客戶端能動態探索 API 功能而非硬編碼 URL。

{
  "data": {
    "id": "123",
    "name": "John",
    "links": {
      "self": "/users/123",
      "orders": "/users/123/orders",
      "profile": "/users/123/profile"
    }
  }
}

冪等性（Idempotency）：GET、PUT、DELETE 操作應為冪等——重複執行相同請求應產生相同結果。這對於網路重試與故障恢復至關重要。

架構總覽

API 設計的決策層次可分為三個層面：

策略層：決定 API 風格（REST vs GraphQL vs RPC）、版本策略、認證機制、錯誤格式等全局決策。

戰術層：資源建模、端點結構、查詢參數設計、分頁策略、篩選與排序機制。

實作層：HTTP 狀態碼使用、Header 設計、Content Negotiation、Rate Limiting 回應格式。

在 URL 結構設計上，推薦的模式為：

/{version}/{resource}/{id}/{sub-resource}
GET /v1/users/123/orders?status=active&page=2&limit=20

分頁策略有多種選擇：Offset-based（簡單但大偏移量效能差）、Cursor-based（適合即時資料流）、Keyset-based（結合效能與一致性）。大多數場景建議採用 Cursor-based 分頁。

應用場景

公開 RESTful API：面向第三方開發者的 API 需要極高的一致性與向後相容性。採用 URL 版本控制、完善的錯誤回應、Rate Limiting Header 與 Webhook 通知機制。

GraphQL BFF：為 React/Vue 前端提供彈性查詢能力，透過 Schema 定義明確的資料契約。善用 Fragment、Directive 與 Subscription 提供豐富的前端互動體驗。

內部微服務 API：使用 Protocol Buffers 定義嚴格的介面契約，搭配 gRPC 實現高效能通訊。Schema 演進規則確保服務獨立部署。

版本遷移策略：使用 Sunset Header 通知棄用時程、提供遷移指南、設置相容層（Adapter Pattern）協助客戶端平滑升級。
