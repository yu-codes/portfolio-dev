冪等性（Idempotency）是構建可靠 API 的關鍵特性。在分散式系統中，網路不穩定導致的重試是常態，冪等設計確保重複請求不會產生非預期的副作用。本文深入探討冪等性的概念、各 HTTP 方法的冪等性分析，以及實作冪等 API 的各種模式。

核心概念

冪等性的定義

冪等操作是指執行一次與執行多次產生相同結果的操作：f(f(x)) = f(x)

特性描述範例
冪等多次執行結果同首次設定使用者名稱為 “Yu”
非冪等多次執行結果不同帳戶餘額 +100
安全不修改伺服器狀態讀取使用者資料

HTTP 方法的冪等性

方法冪等安全說明
GET✅✅讀取不修改狀態
HEAD✅✅同 GET 但無回應體
PUT✅❌全量替換，重複放置同結果
DELETE✅❌刪除已刪除的資源不變
PATCH❌*❌依實作而定
POST❌❌可能重複建立資源
OPTIONS✅✅查詢能力

*PATCH 可以被設計為冪等的

為何需要冪等性

場景問題冪等性解決方式
網路超時請求到達但回應遺失安全重試
負載均衡重試自動重新發送請求不產生重複效果
使用者重複點擊前端發送多次請求只處理一次
訊息佇列重複消費消息被投遞多次處理結果一致
分散式事務補償補償操作可能重複安全重試補償

架構總覽

實作冪等性的模式

模式方法適用場景複雜度
冪等鍵 (Idempotency Key)客戶端提供唯一鍵POST 請求中
資料庫唯一約束唯一索引防重複建立操作低
樂觀鎖/版本號條件更新更新操作中
狀態機狀態只能正向轉換訂單等流程中
Token 機制一次性 token表單提交中
去重表記錄已處理的請求通用中高

冪等鍵的實作流程

步驟動作說明
1客戶端生成 Idempotency-KeyUUID v4
2伺服器檢查 Key 是否已存在查詢去重表
3aKey 不存在 → 執行操作正常處理
3bKey 已存在 → 回傳先前結果不重複執行
4儲存結果與 Key 的映射供後續查詢
5設定 Key 過期時間避免永久佔用

各操作的冪等設計

操作非冪等設計冪等設計
建立訂單POST /orders (每次建新的)POST /orders + Idempotency-Key
扣款balance -= amount用交易 ID 去重
發送通知直接發送記錄 notification_id，已發不再發
更新庫存stock -= 1stock = absolute_value
修改狀態status = next_statusif status == expected then update

關鍵比較

PUT vs PATCH 的冪等性

特性PUTPATCH
語意全量替換部分更新
冪等性天然冪等取決於實作
冪等範例PUT /users/1 {name:“Yu”, age:30}PATCH /users/1 {name:“Yu”}
非冪等範例-PATCH /users/1 {age_increment: 1}
建議總是傳完整資源使用絕對值而非增量

冪等性儲存方案比較

方案效能持久性一致性適用場景
Redis (SET NX EX)高可配置最終一致高併發、可容忍遺失
資料庫唯一索引中高強一致金融級需求
記憶體快取極高無無保證測試/開發
分散式鎖 + 資料庫較低高強一致複雜操作

重試策略與冪等性

策略描述是否需要冪等
立即重試失敗後馬上重試✅ 必需
指數退避逐漸增加間隔✅ 必需
固定間隔固定時間後重試✅ 必需
不重試直接返回錯誤不需要
重試佇列放入佇列稍後處理✅ 必需
