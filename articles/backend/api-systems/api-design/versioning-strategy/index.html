API 版本控制是 API 生命週期管理中最具挑戰性的問題之一。好的版本策略既要允許 API 持續演化，又要保護現有客戶端不受破壞性變更的影響。本文深入比較各種版本控制策略，並提供實用的遷移指南。

核心概念

為何需要版本控制

原因描述範例
破壞性變更移除或修改現有行為刪除欄位、改變型別
需求演化新功能需要新的介面新增資源、改變結構
技術債清理修正早期設計錯誤重新設計資料模型
平台遷移架構重大變更微服務拆分
法規要求合規需要的改變資料格式變更

破壞性 vs 非破壞性變更

類型變更是否破壞
新增選填欄位回應新增欄位❌ 非破壞
新增選填參數請求新增可選參數❌ 非破壞
新增端點新的 URL❌ 非破壞
移除欄位回應移除欄位✅ 破壞
重新命名欄位改變欄位名稱✅ 破壞
改變型別string → number✅ 破壞
新增必填參數請求新增必要參數✅ 破壞
改變語意相同端點行為不同✅ 破壞
改變錯誤碼不同的錯誤回應✅ 可能破壞

版本控制的位置

位置方式範例
URL 路徑版本嵌入 URL/v1/users, /v2/users
查詢參數版本作為參數/users?version=2
請求標頭自定義標頭X-API-Version: 2
Accept 標頭內容協商Accept: application/vnd.api.v2+json
子域名不同域名v2.api.example.com

架構總覽

版本控制策略比較

策略優點缺點使用者
URL 路徑版本直覺、易快取、易文件化URL 膨脹、版本耦合Google, Stripe
標頭版本URL 乾淨、靈活不直覺、難測試GitHub (V3)
查詢參數簡單、選填容易忽略、快取複雜少見
內容協商RESTful、精確複雜、工具支援差GitHub (V3)
無版本 (演化式)最簡單需嚴格向後相容GraphQL

語意化版本 (SemVer) 在 API 中的應用

版本號含義API 對應
MAJOR (v1 → v2)破壞性變更不相容的 API 改變
MINOR (v1.1 → v1.2)新功能向後相容的新增
PATCH (v1.1.1 → v1.1.2)修復向後相容的修正

API 版本生命週期

階段狀態描述持續時間
Current活躍推薦使用的版本持續
Supported維護仍接受請求，接收修復6-12 月
Deprecated棄用仍可用但不推薦6-12 月
Retired停用不再接受請求-

關鍵比較

各大平台的版本策略

平台策略方式特點
Google CloudURL 路徑/v1/, /v2/主版本在 URL
StripeURL 路徑 + 日期版本/v1/ + Stripe-Version header細粒度日期版本
GitHub標頭Accept: application/vnd.github.v3媒體類型
TwilioURL 路徑 + 日期/2010-04-01/日期作為版本
GraphQL無版本演化式欄位棄用
gRPC套件版本package api.v1Proto 文件管理

URL 版本 vs 標頭版本

考量URL 版本標頭版本
瀏覽器直接測試✅ 可以❌ 需要工具
HTTP 快取✅ 簡單⚠️ 需 Vary 標頭
API 文件✅ 清晰⚠️ 較不直覺
客戶端實作✅ 簡單⚠️ 需設定標頭
RESTful 程度❌ 違反（資源不應隨版本變）✅ 較符合
路由管理⚠️ 多重路由✅ 中央處理

版本粒度比較

粒度範例適用場景管理成本
全局版本/v1/, /v2/簡單 API低
資源版本/users/v2, /orders/v1大型 API中
操作版本每操作獨立版本細粒度控制高
日期版本2024-01-01頻繁變更高
無版本永遠向後相容GraphQL依場景

應用場景

版本遷移策略

策略描述適用場景
並行運行新舊版本同時存在漸進遷移
轉接層舊版本轉譯為新版本呼叫減少維護成本
功能旗標同一程式碼根據版本分支小差異
大爆炸強制所有客戶端遷移內部 API
棄用通知逐步引導遷移公開 API

棄用 API 的最佳實踐

步驟動作說明
1發布棄用通知提前 6-12 月告知
2回應加入 Deprecation 標頭讓客戶端程式化偵測
3文件標記明確標示棄用端點
4監控使用量追蹤誰還在用
5聯繫客戶主動協助遷移
6回傳 410 Gone最終停用

版本控制的設計決策

決策考量因素
選擇版本位置團隊習慣、客戶端類型、快取需求
版本粒度API 大小、變更頻率
支援時長客戶端更新能力、合約要求
預設版本無版本時使用最新還是最舊
遷移工具是否提供遷移腳本或相容層
