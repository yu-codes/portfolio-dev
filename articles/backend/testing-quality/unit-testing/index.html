單元測試（Unit Testing）是軟體測試金字塔的基礎層，針對程式中最小的可測試單元進行隔離驗證。良好的單元測試不僅能捕捉回歸錯誤，更能作為程式行為的活文件，引導更好的程式設計。

核心概念

單元測試的特性（FIRST 原則）

原則英文說明實踐
快速Fast毫秒級完成不依賴 I/O、網路、資料庫
獨立Independent測試間不互相依賴每個測試可獨立執行
可重複Repeatable任何環境結果一致不依賴外部狀態
自驗證Self-Validating自動判斷通過/失敗使用斷言，非人工檢查
及時Timely與產品程式碼同步撰寫TDD 或開發後立即補充

測試結構（AAA 模式）

階段英文作用佔比
準備Arrange建立測試所需的前置條件最大
執行Act呼叫被測試的方法最小（通常一行）
驗證Assert確認結果符合預期中等

什麼該測、什麼不該測

應該測試不需測試
業務邏輯框架內建功能
邊界條件簡單的 getter/setter
錯誤處理路徑第三方程式庫內部
複雜的條件判斷配置/常數
公開 API 介面私有方法（間接測試）

架構總覽

測試金字塔

         △
        /  \
       / E2E \          ← 少量，慢，高成本
      /────────\
     / 整合測試  \       ← 中量，中速
    /──────────────\
   /    單元測試     \    ← 大量，快速，低成本
  /──────────────────\

理想比例：

  • 單元測試：70%
  • 整合測試：20%
  • 端對端測試：10%

測試替身（Test Double）類型

類型說明適用場景驗證方式
Dummy填充參數，不使用必須參數但不影響邏輯不驗證
Stub回傳預設值控制依賴的輸出驗證狀態
Mock預期特定呼叫驗證互動行為驗證互動
Spy記錄呼叫資訊事後檢查呼叫驗證呼叫次數/參數
Fake簡化實作需要有行為的替代驗證結果

Mock 策略比較

策略時機優點風險
不 Mock（真實依賴）依賴簡單且快速最真實可能變慢
Mock 外部服務HTTP、第三方 API隔離、穩定可能與實際行為不一致
Mock 資料庫Repository 層快速遺漏 SQL 問題
過度 Mock--測試失去意義
