身份與存取管理（Identity and Access Management, IAM）是雲端安全的基石，負責控制誰（身份）可以對什麼（資源）執行什麼操作（權限）。在雲端環境中，IAM 不僅管理人員存取，還管理服務、應用程式與自動化流程的身份。正確實施 IAM 是防止未授權存取與資料洩漏的第一道防線。本文將深入介紹角色與策略設計、聯邦身份與 SSO 以及最小權限實踐。

核心概念

身份（Identity） 是 IAM 系統中的主體，代表能夠發起存取請求的實體。身份類型包括：人員使用者（員工、承包商）、服務帳戶（應用程式、自動化腳本）、機器身份（EC2 Instance、K8s Pod）與外部身份（合作夥伴、客戶）。每種身份需要不同的管理策略與驗證機制。

角色（Role） 是一組權限的命名集合，代表特定的工作職能或用途。角色式存取控制（RBAC）將權限授予角色，再將角色指派給身份。相比直接授予權限，RBAC 大幅簡化了大規模環境的權限管理。常見角色如 Admin、Developer、ReadOnly、SecurityAuditor。

策略（Policy） 是定義存取控制規則的文件，通常以 JSON 格式表達。策略包含：效果（Allow/Deny）、操作（Actions，如 s3:GetObject）、資源（Resources，如特定 S3 Bucket）與條件（Conditions，如來源 IP、時間限制）。多個策略可以附加到同一身份，系統以最終一致的方式評估所有策略。

聯邦身份（Federation） 允許使用外部身份提供者（IdP）的身份存取雲端資源，無需在雲端建立重複帳戶。透過 SAML 2.0、OIDC 或 WS-Federation 協定實現跨域信任。員工使用企業 Active Directory 登入後即可存取 AWS、Azure 或 GCP 資源。

最小權限原則（Least Privilege） 要求身份只擁有完成當前工作所需的最少權限。實施步驟：從零權限開始 → 根據實際需求逐步授予 → 定期審查並移除未使用的權限。JIT（Just-In-Time）存取讓權限在需要時才啟動，使用後自動撤銷。

條件式存取（Conditional Access） 根據請求的上下文動態決定是否授權。條件因素包括：請求者的 IP 位址、裝置合規狀態、存取時間、MFA 驗證狀態、風險分數等。高風險操作可要求額外的驗證步驟。

架構總覽

企業級 IAM 架構的核心組件：

身份提供者（Identity Provider） — 集中管理所有身份的權威來源。企業通常使用 Azure AD（Entra ID）、Okta 或 Ping Identity 作為主要 IdP。IdP 負責身份的生命週期管理（建立、修改、停用、刪除）、驗證（密碼、MFA、無密碼）與屬性管理。

存取管理引擎 — 各雲端供應商的 IAM 服務（AWS IAM、GCP IAM、Azure RBAC）處理授權決策。接收存取請求後，引擎收集適用的所有策略，評估後做出允許或拒絕的決定。明確 Deny 優先於 Allow。

權限邊界（Permission Boundary） — 設定身份可擁有的最大權限範圍。即使附加了 Admin 策略，如果權限邊界只允許特定服務，最終有效權限仍受限於邊界。這是實現委派管理（Delegated Administration）的關鍵機制。

服務控制策略（SCP/OCP） — 在組織層級設定帳戶的最大權限。AWS 的 Service Control Policies 或 GCP 的 Organization Policies 可以限制整個帳戶或 OU 的可用服務與操作，防止任何使用者（包括 Root）違反組織策略。

存取審計與分析 — CloudTrail（AWS）、Activity Log（Azure）、Audit Log（GCP）記錄所有 API 呼叫。IAM Access Analyzer 識別向外部共享的資源；Access Advisor 顯示最後使用特定權限的時間，幫助識別可移除的閒置權限。

應用場景

多帳戶 IAM 架構 — 企業使用 AWS Organizations 管理 50+ 帳戶。透過 AWS SSO（IAM Identity Center）整合企業 Azure AD，員工使用公司帳號即可存取各帳戶。SCP 限制所有帳戶只能在核准的區域部署資源。Permission Sets 根據職能定義標準角色，統一跨帳戶管理。

服務對服務安全通訊 — 微服務架構中，每個服務使用 K8s Service Account 綁定雲端 IAM Role（透過 Workload Identity Federation）。服務間通訊使用 mTLS 雙向驗證。每個服務的 IAM Role 只授予其實際需要存取的資源權限，遵循最小權限原則。

特權存取管理 — 生產環境的管理存取使用 JIT 模式：工程師提出存取請求 → 自動或人工審批 → 臨時授予權限（最長 4 小時）→ 到期自動撤銷。所有特權操作完整錄影與稽核。使用 Break-glass 程序處理緊急情況。

權限治理自動化 — 定期（每月）執行權限審查：IAM Access Analyzer 標記過度授權；未使用超過 90 天的權限自動標記為候選移除；Service Account 金鑰超過 90 天未輪替則自動停用；離職員工帳戶在 HR 系統通知後 24 小時內自動停用。

AWS 服務實作

IAM 核心元件

IAM 是 全球性服務（全區域共用），免費使用。預設建立的 Root 帳號 擁有完整權限，但不建議日常使用。

元件說明
User（使用者）對應實際人員，應使用自己的帳號密碼登入 AWS 控制台
Group（群組）可包含多個使用者，不能巢狀其他群組
Role（角色）授權給受信任實體（如 EC2、Lambda、其他帳號、Web Identity、SAML 等）以暫時性存取 AWS 資源
Policy（政策）JSON 文件，用來允許或拒絕特定使用者/群組/角色對 AWS 資源的操作權限

Policy 結構

欄位說明
Version政策語言版本，目前為 2012-10-17
Statement政策內容的集合
Sid（可選）標記該條政策
EffectAllow 或 Deny
Principal指定授權對象（使用者、群組、角色）
Action允許執行的操作（如 s3:GetObject）
Resource指定作用的資源 ARN
Condition（可選）設定條件，例如 IAM 身份、來源 IP 等

Policy 範例

{
    "Version":"2012-10-17",
    "Statement":[{
        "Sid": "Deny-Barclay-S3-Access",
        "Effect":"Deny",
        "Principal": { "AWS": ["arn:aws:iam:123456789012:barclay"] },
        "Action": [ "s3:GetObject", "s3:PutObject", "s3:List*" ],
        "Resource": ["arn:aws:s3:::mybucket/*"]
    },{
        "Effect": "Allow",
        "Action": "iam:CreateServiceLinkedRole",
        "Resource": "*",
        "Condition": {
            "StringLike": {
                "iam:AWSServiceName": [
                    "rds.amazonaws.com",
                    "rds.application-autoscaling.amazonaws.com"
                ]
            }
        }
    }]
}

優先順序

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "FullAccess",
            "Effect": "Allow",
            "Action": ["s3:*"],
            "Resource": ["*"]
        },
        {
            "Sid": "DenyCatBucket",
            "Effect": "Deny",
            "Action": ["s3:*"],
            "Resource": ["arn:aws:s3:::catgifs", "arn:aws:s3:::catgifs/*"]
        }
    ]
}
  1. 明確拒絕（Explicit DENY）：最高優先，永遠生效
  2. 明確允許（Explicit ALLOW）：次優先
  3. 預設拒絕（Default DENY）：未指定時預設拒絕

IAM 進階功能

  • Resource-based Policy：S3、SNS、SQS 可針對資源設政策
  • IAM Permission Boundary：定義使用者/角色的最大權限邊界
  • IAM Roles Anywhere：讓 AWS 外部工作負載使用 IAM 角色，透過 X.509 憑證驗證身份
  • 屬性式存取控制（ABAC）：基於標籤屬性授予權限，簡化大規模管理
  • IAM Access Analyzer：識別與外部實體共享的資源，分析政策發現過度權限

IAM 評估順序

  1. 明確拒絕
  2. SCPs（Service Control Policies）
  3. 資源政策
  4. 權限邊界
  5. 使用者/角色政策

Amazon Resource Name（ARN）

arn:aws:partition:service:region:account-id:resource-id
arn:aws:partition:service:region:account-id:resource-type
arn:aws:partition:service:region:account-id:resource-type:resource-id

# 範例
arn:aws:s3:::catgifs          # Bucket 本身
arn:aws:s3:::catgifs/*        # Bucket 內的物件

注意：前者是管理 bucket 的權限，後者是管理物件的權限，兩者不重疊。

IAM Users 與 Groups 限制

  • 每個帳號最多 5000 個 IAM Users
  • 每個使用者最多加入 10 個群組
  • 最多 300 個群組
  • 不支援巢狀群組，Resource Policy 無法直接授權給群組
  • 大規模應用應使用 IAM Roles 與 Identity Federation

IAM Roles 深入解析

角色是被 假設（Assume） 的身份，有兩種 Policy：

  • Trust Policy：定義誰可以假設此角色
  • Permissions Policy：定義角色擁有的權限

STS（Security Token Service）

  • 使用 sts:AssumeRole API 產生臨時安全憑證
  • 憑證過期後需重新假設角色

角色使用場景

場景說明
AWS 服務Lambda Execution Role、EC2 Instance Role 等
緊急存取Break Glass 場景，臨時獲取寫入權限
大型組織超過 5000 身份時使用 Identity Federation
Web 應用Web Identity Federation，使用社交帳號登入
跨帳號存取Cross-Account Access

Service-linked Roles 與 PassRole

  • Service-linked Roles：直接連結到特定 AWS 服務，由服務預先定義，無法在使用中時刪除
  • PassRole：授權使用者將角色傳遞給 AWS 服務，防止使用者透過服務獲取超出自身的權限

Inline Policy vs Managed Policy

類型說明適用場景
Inline Policy直接寫入使用者/群組/角色特殊例外情況
Managed Policy可重複使用，低管理開銷預設建議使用

非 IAM 使用者存取 AWS（Federation）

非 IAM 使用者需先經由聯邦身份驗證，透過 STS AssumeRole API 取得臨時憑證：

聯邦身份驗證方式說明
SAML 2.0與 AD / ADFS 整合，使用 AssumeRoleWithSAML
自訂身份代理當 IdP 不支援 SAML 時，使用 AssumeRole 或 GetFederationToken
Web Identity Federation使用 Amazon、Facebook、Google、OIDC IdP，使用 AssumeRoleWithWebIdentity
AWS CognitoAmazon 官方推薦身分驗證服務
AWS SSO單一登入服務，不需呼叫 STS API

AWS Directory Service

類型說明
AWS Managed Microsoft AD完整 AD 功能，適合需要與內部 AD 信任關聯的場景
AD Connector代理連線至內部 AD，不儲存資料
Simple AD基本 AD 功能，無法與內部 AD 連線

Amazon Cognito

Cognito User Pool（用戶池）

  • 提供 App 使用者的身份驗證功能，回傳 JWT token
  • 支援社群登入（Google/Facebook/Apple）與 OIDC/SAML
  • JWT 可用於換取 AWS 臨時憑證（存取資源）或存取 API Gateway

Cognito Identity Pool（身份池）

  • 用來授權 AWS 資源使用權限
  • 支援 guest user、透過 federated identity 提供 S3/其他資源臨時權限

AWS Organizations 與 SCP

AWS Organizations 架構：

  • 管理帳號（Management Account）是唯一一個，可邀請或建立成員帳號
  • Organization Root 包含帳號或組織單位（OU）
  • 統一帳單（Consolidated Billing）：整合預留和批量折扣

Service Control Policies（SCP）：

  • JSON 格式政策，可附加到整個組織、OU 或帳號
  • 向下繼承到所有下層 OU 和帳號
  • 管理帳號不受 SCP 影響
  • SCP 是帳號權限邊界，限制帳號（包括 root user）可以執行的操作，SCP 不授予任何權限
架構說明
拒絕清單（預設）FullAWSAccess 預設開啟，明確拒絕特定服務
允許清單移除 FullAWSAccess，只允許特定服務

AWS Control Tower

  • 快速設定多帳號環境，整合 Organizations、IAM Identity Center、CloudFormation、Config
  • Landing Zone：Well-Architected 多帳號環境
  • Guard Rails：偵測/強制規則與標準（預防性透過 SCP，偵測性透過 Config Rules）
  • Account Factory：自動化帳號建立與套用 Guard Rails

SSL/TLS 憑證管理

若使用第三方 CA 的 SSL/TLS 憑證，應匯入到 AWS Certificate Manager (ACM) 或 IAM 憑證儲存區。

AWS IAM 最佳實踐

  1. 遵循最小權限原則（Least Privilege）：只給需要的權限
  2. 啟用 MFA：為 Root 帳號和重要使用者啟用多因素認證
  3. 定期輪替 Access Key：避免憑證外洩風險
  4. 使用 IAM Roles 而非 IAM Users：適用於服務對服務存取
  5. 使用 Permission Boundary：實現委派管理
  6. 啟用 IAM Access Analyzer：識別意外的存取權限
