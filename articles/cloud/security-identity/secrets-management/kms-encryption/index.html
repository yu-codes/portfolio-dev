KMS（Key Management Service）是雲端平台提供的託管式金鑰管理服務，用於建立、管理與控制加密金鑰的使用。KMS 是雲端資料保護的核心基礎設施，支撐著靜態加密、傳輸加密、數位簽章等多種安全能力。理解 KMS 的工作原理對於設計安全的雲端架構至關重要。

核心概念

加密層次結構

層次金鑰類型存儲位置管理者
根金鑰HSM 主金鑰硬體安全模組雲端平台
客戶主金鑰（CMK）加密 DEK 的金鑰KMS 服務客戶
資料加密金鑰（DEK）加密實際資料與資料一起存儲（加密態）服務

信封加密（Envelope Encryption）

信封加密是 KMS 的核心模式：

資料 → 使用 DEK 加密 → 加密後的資料
DEK → 使用 CMK 加密 → 加密後的 DEK
[加密後的資料] + [加密後的 DEK] → 一起存儲

解密時：
加密後的 DEK → KMS 使用 CMK 解密 → 明文 DEK
加密後的資料 → 使用 DEK 解密 → 明文資料

為什麼使用信封加密

原因說明
效能大量資料用本地 DEK 加密（快速），僅小量 DEK 需 KMS 操作
安全CMK 永遠不離開 KMS/HSM
成本減少 KMS API 呼叫次數
彈性可為每個物件使用不同的 DEK

金鑰管理最佳實踐

  1. 金鑰分離：不同用途使用不同金鑰
  2. 自動輪換：啟用所有金鑰的自動輪換
  3. 存取最小化：嚴格控制 kms:Decrypt 權限
  4. 監控使用：監控金鑰使用模式，偵測異常
  5. 刪除保護：設定金鑰刪除等待期（7-30 天）
  6. 災難恢復：確保金鑰的跨區域可用性

AWS 服務實作

AWS Key Management Service (KMS)

  • 集中式金鑰管理服務，可產生、儲存與輪替金鑰（CMK）
  • 支援 對稱加密金鑰（加/解密用同一把）與 非對稱金鑰（公開/私密鑰）
  • 支援 自動金鑰輪替（每年一次）
  • KMS 每次呼叫最多加密 4KB 資料，更大的資料使用信封加密（Envelope Encryption）
  • KMS 金鑰永不離開 KMS，受到 FIPS 140-2（L2） 保護

KMS 金鑰類型

類型說明
AWS 管理金鑰（AWS Managed Keys）AWS 建立、管理與使用，每年自動輪替
客戶管理金鑰（CMK）客戶自行建立與管理，可設定自動輪替
AWS Owned KeysAWS 服務內部使用
對稱金鑰AES-256 加密
非對稱金鑰RSA 或 ECC 金鑰對
HMAC 金鑰用於訊息驗證碼
自訂金鑰存放區使用 CloudHSM 作為金鑰存放區

KMS 金鑰組成

  • Key ID：唯一識別碼
  • 建立日期
  • Key Policy：資源政策（類似 S3 Bucket Policy）
  • 描述和狀態

KMS 使用場景

Data Encryption Keys（DEKs）與信封加密

步驟說明
1使用 GenerateDataKey 產生 DEK
2DEK 用於加密資料
3加密後的 DEK 與資料一起儲存
4解密時先解密 DEK，再用 DEK 解密資料
KMS Key → 產生 DEK（明文 + 加密版本）
               ↓
           明文 DEK 加密資料 → 丟棄明文 DEK
               ↓
           儲存：加密的 DEK + 加密的資料

Key Policy 與授權

每個 KMS 金鑰都有一個 Key Policy，預設允許帳號 Root 使用者完全存取：

{
  "Sid": "Enable IAM User Permissions",
  "Effect": "Allow",
  "Principal": {"AWS": "arn:aws:iam::ACCOUNT-ID:root"},
  "Action": "kms:*",
  "Resource": "*"
}
授權模式說明
Key Policy明確允許主體
Key Policy + IAM預設模式，兩者結合
Grants程式化授權，可撤銷

金鑰輪替

自動輪替（僅適用於 Customer Managed Keys）：

  • AWS Managed Keys 每年自動輪替
  • 舊的金鑰保留用於解密，新的加密使用新金鑰

手動輪替：

  • 建立新的 KMS 金鑰，更新應用程式參照
  • 使用別名簡化切換

多區域金鑰

  • 相同的金鑰存在於多個區域，一個主要金鑰 + 多個副本金鑰
  • 可在任一區域加解密
  • 適用場景：全域應用程式、災難復原、全域 DynamoDB Global Table 加密

AWS CloudHSM

  • AWS 託管的 專用硬體安全模組 (Hardware Security Modules)
  • 單租戶 HSM，符合 FIPS 140-2 Level 3
  • 自行控制加密金鑰（不可匯出）
  • 支援 PKCS#11 / JCE / CNG API
  • 在 VPC 中運行，透過 ENI 存取，建議部署在多個 AZ

CloudHSM 使用場景

  • 需要完全控制金鑰的合規要求
  • FIPS 140-2 Level 3 合規需求
  • 需要高效能加密運算
  • 與 KMS 整合作為自訂金鑰存放區（KMS Custom Key Store）

加密最佳實踐

  1. 使用 KMS 進行大部分加密需求：簡單、成本效益高
  2. 敏感工作負載使用 CloudHSM：需要完全金鑰控制時
  3. 啟用金鑰輪替：定期更換金鑰降低風險
  4. 使用金鑰政策：控制誰可以使用哪些金鑰
  5. 監控金鑰使用：透過 CloudTrail 追蹤金鑰操作
  6. 金鑰分離：不同用途使用不同金鑰
  7. 設定刪除等待期：金鑰刪除等待 7-30 天
