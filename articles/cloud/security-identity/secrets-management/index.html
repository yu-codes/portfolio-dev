在現代雲端架構中，應用程式需要管理大量的機密資訊：資料庫密碼、API 金鑰、TLS 憑證、OAuth Token 等。若這些機密以明文形式儲存在程式碼或設定檔中，一旦洩漏將造成災難性後果。密鑰管理（Secrets Management）提供了系統化的方式來儲存、存取、輪換與審計這些敏感資訊，是雲端安全基礎建設不可或缺的一環。

核心概念

機密資訊的生命週期

每個機密都有其完整的生命週期，需要在各階段妥善管理：

  1. 產生（Generation）：使用加密安全的隨機數產生器（CSPRNG）建立強度足夠的密鑰
  2. 儲存（Storage）：以加密形式儲存於專用的保險庫系統中
  3. 分發（Distribution）：透過安全通道將機密傳送至授權的消費者
  4. 使用（Usage）：應用程式在執行時取得機密，並最小化在記憶體中的存活時間
  5. 輪換（Rotation）：定期更換機密，降低長期暴露的風險
  6. 撤銷（Revocation）：在可疑洩漏時立即使機密失效
  7. 銷毀（Destruction）：安全地清除不再需要的機密

信封加密（Envelope Encryption）

信封加密是雲端密鑰管理的核心技術模式：

  • 資料加密金鑰（DEK）：直接加密資料的對稱金鑰
  • 金鑰加密金鑰（KEK）：用來加密 DEK 的主金鑰，通常由 HSM 保護
  • 流程：資料以 DEK 加密 → DEK 以 KEK 加密 → 加密後的 DEK 與密文一同儲存

此設計的優勢在於：即使需要重新加密大量資料，只需用新 KEK 重新加密 DEK 即可，無需重新加密所有資料本身。

動態機密（Dynamic Secrets）

相較於靜態機密，動態機密在每次請求時即時產生，使用完畢後自動過期。例如 HashiCorp Vault 可以在應用程式需要時動態建立資料庫帳號，設定短暫的 TTL，過期後自動撤銷。

集中式 vs 分散式管理

  • 集中式：所有機密存放在單一 Vault，統一策略管理，適合中小規模或安全團隊集中管控的組織
  • 分散式：各團隊自行管理命名空間，搭配聯邦式策略，適合大型組織與多雲環境

應用場景

場景一：CI/CD Pipeline 機密注入

在 CI/CD 流程中，建置與部署階段需要存取各種機密（容器映像倉庫憑證、部署金鑰等）。最佳實踐是透過 Vault Agent 或 CSI Secret Store Driver 在執行時動態注入，避免機密出現在環境變數或建置日誌中。

場景二：微服務間的 mTLS 憑證管理

微服務架構中，每個服務需要自己的 TLS 憑證來實現相互認證。密鑰管理系統可搭配 PKI 引擎自動簽發短期憑證（如 24 小時有效期），並在到期前自動更新，無需人工介入。

場景三：資料庫憑證自動輪換

傳統做法是在設定檔中寫死資料庫密碼，一旦需要輪換就得重啟所有服務。透過動態機密，每個服務實例取得獨立的短期資料庫帳號，Vault 自動建立與撤銷，完全消除靜態密碼的風險。

場景四：多雲環境統一管理

當組織同時使用 AWS、Azure、GCP 時，每個雲端有各自的密鑰管理服務。透過 HashiCorp Vault 作為統一介面，提供一致的存取策略、審計日誌與輪換機制，降低多雲管理的複雜度。
