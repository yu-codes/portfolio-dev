安全群組（Security Groups）與防火牆（Firewalls）是雲端網路安全的第一道防線，透過控制入站和出站流量來保護雲端資源。正確的網路存取控制配置能有效縮小攻擊面，防止未授權的網路存取，是雲端安全架構的基礎構件。

規則數量較少較多（入出都要）

安全群組設計最佳實踐

  1. 最小開放原則：僅開放必要的端口與來源
  2. SG 引用優於 IP：使用安全群組 ID 引用而非 IP 位址
  3. 功能分離：不同功能使用不同安全群組
  4. 避免 0.0.0.0/0：除 ALB 外避免允許所有來源
  5. 描述完整：每條規則都加上清晰的描述
  6. 定期審查：使用自動化工具偵測過寬規則

AWS 服務實作

AWS Security Group（安全群組）

  • EC2 層級防火牆，只支援 Allow 規則
  • Stateful：回應流量自動允許
  • 可跨多實例與子網，支援共用與多重綁定
  • 預設拒絕所有入站流量，允許所有出站流量
特性說明
作用層級EC2 實例（ENI）
規則類型僅 Allow
狀態Stateful
評估方式所有規則同時評估

AWS NACL（網路 ACL）

  • 子網層級防火牆，支援 Allow/Deny 規則
  • Stateless：入站和出站流量需分別設定規則
  • 具規則號碼順序，數字越小優先權越高
特性說明
作用層級子網
規則類型Allow 和 Deny
狀態Stateless
評估方式按規則號碼順序評估

常見使用場景（AWS）

Security Group 使用場景

  1. Web 伺服器：允許 80/443 端口入站
  2. 資料庫伺服器：僅允許應用伺服器 SG 存取
  3. SSH 存取：限制特定 IP 存取 22 端口

NACL 使用場景

  1. 阻擋特定 IP：使用 Deny 規則阻擋惡意 IP
  2. 子網隔離：控制不同子網之間的流量
  3. 額外安全層：作為 Security Group 之外的額外防護

AWS 安全群組設計最佳實踐

  1. 分層防護：同時使用 Security Group 和 NACL
  2. 最小權限：只開放必要的端口和 IP 範圍
  3. 使用 Security Group 引用：允許特定 SG 的流量而非 IP
  4. 定期審核：檢查並清理不再使用的規則
  5. 防止 0.0.0.0/0：除 ALB 外避免允許所有來源
  6. 描述完整：每條規則都加上清晰的描述
