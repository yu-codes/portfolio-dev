遙測標準

OpenTelemetry 與上下文傳播

遙測標準（Telemetry Standards）定義了如何產生、收集、傳輸與儲存可觀測性資料的統一規範。OpenTelemetry（OTel）作為 CNCF 的旗艦項目，已成為業界事實上的遙測標準，它統一了 OpenTracing 與 OpenCensus 兩個前身項目，提供廠商中立的 API、SDK 與工具鏈。本文將深入介紹 OpenTelemetry 的架構、協定以及分散式追蹤在雲端環境中的應用。

核心概念

OpenTelemetry 訊號模型 — OTel 定義了三種核心訊號類型：Traces（追蹤）記錄請求的完整路徑；Metrics（指標）量化系統的數值狀態；Logs（日誌）記錄離散事件。每種訊號都有獨立的 API 與 SDK，但共享統一的上下文傳播機制與資源（Resource）模型。

OTLP 協定 — OpenTelemetry Protocol 是 OTel 原生的資料傳輸協定，支援 gRPC 與 HTTP/protobuf 兩種傳輸方式。OTLP 為三種訊號提供統一的序列化格式，相比專有協定具有更好的互通性與壓縮效率。

資源（Resource） 是產生遙測資料的實體描述，包含服務名稱、版本、部署環境、主機資訊等屬性。Resource 讓所有遙測資料都能關聯回其來源服務，是實現跨服務關聯分析的基礎。

上下文傳播（Context Propagation） 確保追蹤資訊能跨越進程邊界。W3C Trace Context 標準定義了 traceparent（包含 trace-id、parent-id、trace-flags）和 tracestate（廠商特定資訊）兩個 HTTP 標頭。Baggage 標準則允許傳遞任意鍵值對做為業務上下文。

自動插裝（Auto-instrumentation） 是 OTel 提供的零程式碼埋點方案。透過位元組碼修改（Java Agent）、猴子修補（Python）或編譯時注入等技術，自動為常見框架與函式庫產生遙測資料，大幅降低接入門檻。

Span 模型 — 追蹤的基本單元是 Span，每個 Span 記錄一個操作的名稱、開始/結束時間、屬性（Attributes）、事件（Events）與連結（Links）。Span 之間通過 parent-child 關係形成樹狀結構，一個完整的 Trace 就是一棵 Span 樹。

架構總覽

OpenTelemetry 的端對端架構由以下組件構成：

應用程式 SDK — 嵌入應用程式中的 OTel SDK，負責建立 TracerProvider、MeterProvider 與 LoggerProvider。SDK 管理 Span 的生命週期、指標的聚合以及日誌的批次處理。支援 Java、Python、Go、JavaScript、.NET、Rust 等主流語言。

OTel Collector — 獨立部署的資料管道組件，架構分為三層：Receivers 接收來自多種來源的資料（OTLP、Prometheus、Jaeger、Zipkin 等）；Processors 進行過濾、取樣、批次、屬性修改等處理；Exporters 將資料發送到最終儲存後端。Collector 可以 Agent 模式（Sidecar/DaemonSet）或 Gateway 模式部署。

資料管道 — 典型部署中，應用程式 SDK 將遙測資料透過 OTLP 發送到本地 Collector Agent，Agent 進行初步處理後轉發到中央 Collector Gateway，Gateway 執行全域取樣與路由後匯出到各後端儲存。

語義慣例（Semantic Conventions） — OTel 定義了標準化的屬性命名規範，如 http.request.method、db.system、messaging.system 等。遵循語義慣例讓不同服務產生的遙測資料具有一致的結構，便於跨服務查詢與分析。

應用場景

漸進式遷移 — 大型企業從 Datadog 專有 Agent 遷移到 OTel。第一階段：部署 OTel Collector 並設定 Datadog Exporter，保持現有後端不變。第二階段：逐步將應用程式 SDK 從 Datadog SDK 替換為 OTel SDK。第三階段：引入替代後端（如 Grafana Stack），實現多後端並行，最終完成遷移。

Kubernetes 環境整合 — 在 K8s 叢集中以 DaemonSet 方式部署 OTel Collector Agent，自動收集所有 Pod 的遙測資料。搭配 OTel Operator 實現自動插裝注入，新部署的服務無需修改程式碼即可產生追蹤資料。透過 Resource Detector 自動添加 Pod、Node、Namespace 等 K8s 元資料。

多雲遙測統一 — 跨 AWS、GCP、Azure 部署的微服務，統一使用 OTel SDK 產生遙測資料。各雲端區域的 Collector Gateway 將資料彙整到中央可觀測性平台，實現跨雲的端對端追蹤與統一儀表板。

成本控制 — 高流量服務每秒產生數萬個 Span。透過 Collector 的 Tail-based Sampler，只保留包含錯誤或高延遲的完整 Trace，將儲存成本降低 90% 的同時保留所有值得分析的異常路徑。
