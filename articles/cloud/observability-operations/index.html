可觀測性與維運

監控、日誌與 SRE 實踐

可觀測性（Observability）是現代雲端系統維運的核心能力，它讓工程團隊能夠從系統的外部輸出推斷內部狀態，快速定位問題根因並持續改善服務品質。與傳統監控不同，可觀測性不僅告訴你「發生了什麼」，更幫助你理解「為什麼發生」。本文將全面介紹可觀測性的三大支柱、維運實踐以及現代化遙測標準。

核心概念

可觀測性建立在三大支柱之上：

指標（Metrics） 是可聚合的數值型時間序列資料，用於描述系統在特定時間點的狀態。常見指標包括 CPU 使用率、請求延遲百分位數（P50/P95/P99）、錯誤率與吞吐量。指標的優勢在於儲存成本低且查詢速度快，適合用於警報觸發與趨勢分析。Prometheus 是目前最主流的指標收集與儲存解決方案，搭配 Grafana 進行視覺化呈現。

日誌（Logging） 是離散的事件記錄，包含時間戳、嚴重程度與結構化訊息。結構化日誌（Structured Logging）使用 JSON 格式輸出，使日誌資料可被機器解析與索引。集中式日誌管理平台如 Elasticsearch（ELK Stack）、Loki 或 CloudWatch Logs 讓團隊能夠跨服務搜尋與關聯日誌事件。

分散式追蹤（Distributed Tracing） 記錄請求在微服務架構中的完整路徑，每個操作以 Span 表示，多個 Span 組成一個 Trace。追蹤資料幫助團隊識別瓶頸服務、理解服務依賴關係並量化每個環節的延遲貢獻。Jaeger 與 Zipkin 是常見的分散式追蹤後端。

上下文傳播（Context Propagation） 是將追蹤識別碼（Trace ID）與其他中繼資料跨服務邊界傳遞的機制。W3C Trace Context 標準定義了 traceparent 和 tracestate HTTP 標頭，確保不同廠商的工具能夠互通。

架構總覽

現代可觀測性平台的典型架構包含以下層次：

資料產生層 — 應用程式透過 SDK 或自動插裝（Auto-instrumentation）產生遙測資料。OpenTelemetry 提供統一的 API 與 SDK，支援指標、日誌與追蹤三種訊號。

資料收集層 — OpenTelemetry Collector 作為中介代理，接收、處理並匯出遙測資料。它支援多種接收器（Receiver）、處理器（Processor）與匯出器（Exporter），可實現資料過濾、取樣與路由。

資料儲存層 — 不同訊號使用專門的儲存後端：時間序列資料庫（如 Prometheus、Mimir）儲存指標；日誌儲存（如 Loki、Elasticsearch）索引日誌事件；追蹤後端（如 Tempo、Jaeger）儲存 Span 資料。

視覺化與分析層 — Grafana 提供統一的儀表板，整合所有訊號來源。團隊可以從指標異常深入到相關日誌，再從日誌中的 Trace ID 跳轉至完整追蹤路徑，實現無縫的除錯體驗。

警報與事件管理層 — 基於指標閾值或異常偵測觸發警報，通過 PagerDuty、Opsgenie 等平台通知值班工程師，並整合事件管理（Incident Management）流程。

應用場景

微服務除錯 — 當使用者回報 API 延遲時，團隊從 P99 延遲指標發現異常，查詢對應時間段的追蹤資料定位到慢查詢服務，再透過該服務的日誌找到具體的資料庫查詢語句。整個過程從分鐘級縮短到秒級。

容量規劃 — 通過歷史指標趨勢分析預測資源需求，設定自動擴縮策略。結合業務指標（如訂單數）與基礎設施指標（如 CPU、記憶體）建立關聯模型。

SLO 管理 — 定義服務層級目標（如 99.9% 可用性），持續追蹤錯誤預算消耗。當預算接近耗盡時自動凍結變更部署，優先修復可靠性問題。

安全事件偵測 — 透過日誌分析偵測異常存取模式，如短時間內大量失敗登入、非常規時段的特權操作或未預期的跨區域資料存取。

成本優化 — 分析遙測資料量的成長趨勢，識別高基數（High Cardinality）指標與冗餘日誌，透過過濾與聚合降低儲存與查詢成本。
