雲端成本優化是在不犧牲效能與可靠性的前提下，最大化每一分雲端支出的商業價值。這不僅是「省錢」，更是一種持續的工程實踐，需要在架構設計、購買策略、資源管理與文化建設等多個面向同步推進。

核心概念

成本優化的三大槓桿：

1. 購買模式優化（Pricing Model）：

  • 隨需（On-Demand）：最高彈性，最高單價，適合不確定的工作負載
  • 預留實例（Reserved Instances / Savings Plans）：承諾 1-3 年使用量，獲得 30%-72% 折扣
  • Spot / Preemptible 實例：利用閒置容量，獲得 60%-90% 折扣，但可能被中斷
  • Enterprise Discount Program（EDP）：大型客戶的整體承諾折扣

2. 資源規模調整（Right-sizing）：

  • 垂直縮放：降低過度配置的實例規格（如 m5.xlarge → m5.large）
  • 世代升級：遷移至新一代實例獲得更好的性價比（如 m5 → m7）
  • 架構轉換：從 x86 遷移至 ARM（Graviton/Ampere），獲得 20-40% 成本節省

3. 使用模式優化（Usage Pattern）：

  • 排程啟停：非工作時間關閉非必要資源
  • 自動擴縮：根據實際負載動態調整資源數量
  • 資料生命週期：將冷資料移至低成本儲存層

成本優化成熟度模型：

  • Level 0：無管理（帳單到了才看）
  • Level 1：可見性（知道錢花在哪裡）
  • Level 2：基本優化（Right-sizing + 排程）
  • Level 3：進階優化（RI/Spot + 架構優化）
  • Level 4：持續優化（自動化 + 文化內建）

完整的成本優化循環從分析開始，產生具體建議後透過自動化或審批流程執行，最終追蹤實際節省金額以驗證效果。

優先序建議：先做低風險高回報的項目（排程、Right-sizing），再處理需要承諾的購買優化（RI/SP），最後考慮需要架構變更的深度優化（Spot、Serverless）。

應用場景

場景一：Spot Instance 混合架構 在 Kubernetes 叢集中實施 Spot 混合策略：Node Pool 由 30% On-Demand（運行關鍵服務）+ 70% Spot（運行可容錯服務）組成。搭配 Spot 中斷處理程式（interrupt handler），在收到 2 分鐘中斷警告時優雅地遷移 Pod 至 On-Demand 節點。

場景二：Reserved Instance 組合最佳化 分析過去 12 個月的使用量資料，識別穩定的基礎負載（P10 使用量）作為 3 年 All Upfront RI 的覆蓋目標；中層穩定負載（P10-P50）使用 1 年 Partial Upfront；波動部分保持 On-Demand。每季度重新評估並調整組合。

場景三：自動化 Right-sizing Pipeline 建立每月執行的 Right-sizing 分析管線：收集 CloudWatch/Azure Monitor 指標 → 識別 CPU/Memory 使用率持續低於 40% 的實例 → 產生降規建議 → 經團隊審核後透過 Terraform 自動變更 → 追蹤變更後的效能指標確認無負面影響。

場景四：資料儲存生命週期優化 實施 S3 Intelligent-Tiering 與自訂生命週期策略：熱資料（0-30天）存放 Standard；溫資料（30-90天）移至 Infrequent Access；冷資料（90-365天）移至 Glacier Instant Retrieval；歸檔資料（365天以上）移至 Glacier Deep Archive。預估節省 60% 儲存成本。

場景五：開發環境自動排程 實施智慧排程策略：工作日 9:00-19:00 自動啟動開發環境，19:00 後自動停止；若偵測到活躍的 SSH/RDP 連線則延後停止；週末完全關閉除非手動啟動。透過 Slack Bot 提供隨需啟動能力，保留開發者彈性。

核心優化策略

1. 右調整（Right Sizing）

  • 根據 CloudWatch 使用率監控選擇最適合的實例大小
  • 定期審查並縮減過度配置的資源
  • 使用 AWS Compute Optimizer 取得機器學習驅動的最佳化建議（支援 EC2、EBS、Lambda）

2. Reserved Instance 策略

  • 針對 1 年或 3 年穩定工作負載購買 RI，可節省最高 72%
  • Convertible RI 可轉換為不同規格，保留彈性
  • 透過 AWS Organizations 在多帳號間共享 RI 效益

3. Spot Instance 應用

  • 適用於容錯應用、批次處理、CI/CD 管線
  • 節省最高可達 90%，但須容忍隨時中斷的風險
  • 建議搭配 Spot 中斷處理機制（2 分鐘警告通知）

4. 自動擴縮（Auto Scaling）

  • 根據實際需求動態調整資源數量，避免靜態過度配置
  • 使用 Target Tracking 或 Step Scaling 策略
  • 搭配 CloudWatch 指標實現精準擴縮

S3 儲存類別成本優化

來源層級目標層級使用場景

Standard-IAS3 Glacier需要長期歸檔的資料
任意層級Intelligent-Tiering存取模式不確定的資料（自動分層）

S3 生命週期政策建議：

  • 自動轉換儲存類別以降低儲存成本
  • 自動刪除過期物件釋放空間
  • 定期清理未完成的 Multipart Upload（避免產生隱性費用）

EBS 成本優化

  • 刪除已停止 EC2 實例遺留的孤立 Volume
  • 依工作負載選擇適合的 Volume 類型（gp3 比 gp2 成本效益更高，且效能更佳）
  • 定期清理不再需要的 EBS Snapshot

資料傳輸成本管理

免費傳輸

  • 同一可用區（AZ）內的資料傳輸
  • 從網際網路傳入 AWS（Ingress）

收費傳輸

  • 跨可用區（Cross-AZ）傳輸
  • 跨區域（Cross-Region）傳輸
  • 傳出至網際網路（Egress）

優化方式

  • 使用 VPC Endpoint 讓流量走 AWS 內部網路，避免走公網出站費用
  • 將相互依賴的資源部署在同一可用區
  • 使用 Amazon CloudFront CDN 減少 EC2/S3 的出站流量費用

資料庫成本優化

Amazon RDS

  • 使用 Reserved Instance 可節省 RDS 費用最高 72%
  • 根據 CloudWatch 監控資料調整實例規格（Right Sizing）
  • 評估使用 Aurora Serverless 以按實際用量付費模式節省成本

Amazon DynamoDB

  • 依工作負載特性選擇 On-Demand 或 Provisioned 容量模式
  • 使用 DAX（DynamoDB Accelerator） 快取熱點讀取以降低 RCU 消耗
  • 配置適當的讀寫容量避免過度配置

AWS Organizations 合併帳單與成本管理

  • 所有成員帳號費用整合至單一付款帳號，自動獲得批量折扣
  • Reserved Instances 與 Savings Plans 可在組織內所有帳號共用
  • 可設定帳號層級預算，並按 OU 或標籤追蹤各部門的成本
