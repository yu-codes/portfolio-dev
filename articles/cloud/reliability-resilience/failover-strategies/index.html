故障轉移策略

故障轉移（Failover）是當主要系統發生故障時，自動或手動將流量和工作負載切換到備用系統的過程。有效的故障轉移策略是實現高可用性的關鍵，它決定了系統在面對故障時的恢復速度和資料完整性。從簡單的 DNS 切換到複雜的有狀態故障轉移，不同的策略適用於不同的業務需求和技術條件。

核心概念

故障偵測

故障轉移的前提是準確的故障偵測：

  • 健康檢查（Health Check）：定期探測服務端點的回應
  • 心跳偵測（Heartbeat）：節點間定期交換存活訊號
  • 合成監控（Synthetic Monitoring）：模擬使用者行為偵測功能異常
  • 被動監控：分析實際流量的錯誤率和延遲

故障偵測的挑戰

  • 腦裂（Split-Brain）：網路分區導致兩邊都認為對方故障
  • 誤判（False Positive）：暫時性問題被誤判為永久故障
  • 延遲偵測：平衡偵測速度和準確性的取捨
  • 級聯故障：一個組件的故障觸發其他組件的故障轉移

故障轉移模式

Active-Passive（主動-被動）：

  • 主節點處理所有流量
  • 被動節點待命接管
  • 適合有狀態服務

Active-Active（雙活/多活）：

  • 多個節點同時處理流量
  • 故障節點的流量自動分配到其他節點
  • 適合無狀態服務

N+1 冗餘：

  • N 個活躍節點 + 1 個備用節點
  • 備用節點可接管任一故障節點
  • 成本效益的平衡方案

故障轉移層級

  • 應用層：負載均衡器將流量從故障實例移除
  • 資料庫層：資料庫主從切換（Promotion）
  • 區域層：DNS 或全域負載均衡切換到備用區域
  • 服務層：Circuit Breaker 切換到降級服務

回切（Failback）

故障恢復後將服務切回原主系統：

  • 確認原主系統完全恢復
  • 資料同步追上（追平 replication lag）
  • 計劃性的流量切回
  • 驗證服務正常後完成回切

有狀態服務故障轉移

有狀態服務的故障轉移比無狀態服務複雜得多：

  • Session 轉移：使用外部 Session Store（如 Redis）
  • 連線轉移：WebSocket 重連機制
  • 交易處理：未完成交易的回滾或續傳
  • 佇列重分配：故障消費者的訊息重新分配

策略RTO資料風險成本複雜度自動化程度

複寫模式對故障轉移的影響

複寫模式資料損失風險效能影響適用場景
同步複寫零高（延遲增加）金融交易
半同步複寫極低中等關鍵業務
非同步複寫可能遺失數秒無一般應用
串聯複寫可能遺失更多無讀取擴展

自動 vs 手動故障轉移

面向自動故障轉移手動故障轉移
恢復速度快（秒到分鐘）慢（需人工介入）
誤判風險有（需仔細設計）低（人工判斷）
適用場景明確的故障模式複雜或模糊的故障
腦裂風險需機制防護人工協調
營運成本低（建設期高）高（需 On-Call）

應用場景

資料庫高可用

資料庫層的故障轉移策略：

  • 主從架構搭配自動 Promotion
  • Multi-AZ 部署的自動故障轉移（如 AWS RDS Multi-AZ）
  • Patroni/Orchestrator 等工具管理 PostgreSQL/MySQL 叢集
  • 故障轉移後的 DNS/連線字串自動更新

微服務架構

分散式系統的故障隔離和轉移：

  • Circuit Breaker 防止級聯故障
  • 服務網格（如 Istio）的自動重試和故障轉移
  • 降級策略：核心功能優先，非關鍵功能降級
  • 熔斷後的自動恢復偵測

Kubernetes 環境

容器化環境的故障轉移：

  • Pod 層級：liveness probe 失敗自動重啟
  • Node 層級：節點故障時 Pod 自動調度到其他節點
  • 叢集層級：多叢集部署搭配全域負載均衡
  • StatefulSet 的有序故障轉移

全域服務

跨區域的故障轉移策略：

  • Anycast DNS 實現最近區域路由
  • Global Accelerator / Front Door 的即時切換
  • 資料一致性和故障轉移速度的取捨決策
  • 定期 DR 演練驗證故障轉移流程
