速率限制（Rate Limiting）是控制客戶端在特定時間窗口內可發送請求數量的機制。它是保護 API 服務免受濫用、確保公平使用，並維護系統穩定性的關鍵防線。

核心概念

速率限制的目的

目的說明範例
防止濫用阻止惡意大量請求DDoS、爬蟲
公平使用確保所有用戶公平分享資源API 配額
成本控制防止意外的巨額費用雲端資源用量
系統保護避免後端過載突發流量
服務分級不同方案不同限制免費/付費方案

限制維度

維度說明範例
用戶級每個用戶的限制100 req/min per user
IP 級每個 IP 的限制1000 req/min per IP
API Key 級每個金鑰的限制依方案不同
端點級每個 API 端點POST 較嚴格
全域整體系統限制系統保護閾值

架構總覽

速率限制演算法

演算法說明優點缺點
Fixed Window固定時間窗口計數簡單、記憶體低邊界突發問題
Sliding Window Log記錄每個請求時間精確記憶體高
Sliding Window Counter加權計算平衡精確與效率近似值
Token Bucket令牌桶允許突發稍複雜
Leaky Bucket漏桶穩定輸出不允許突發

Fixed Window Counter：

  • 每分鐘重置一次計數
  • 問題：窗口邊界可能出現 2x 突發

Sliding Window：

  • 結合前一窗口與當前窗口的加權
  • 計算：前窗口剩餘比例 × 前窗口計數 + 當前計數

Token Bucket：

  • 桶容量 = 突發上限
  • 令牌填充速率 = 穩態速率
  • 允許短期突發但維持長期平均

Leaky Bucket：

  • 請求進入佇列
  • 以固定速率處理
  • 佇列滿則拒絕

速率限制 vs 其他保護機制

機制層級目的粒度
Rate Limiting應用層控制請求頻率用戶/IP/API
Throttling服務層控制處理速度全域/服務
Circuit Breaker服務間防止級聯故障服務對服務
Backpressure串流消費者控制生產速度管線
Admission Control系統層入口流量控制全域

DDoS 防護的第一道防線

  • 邊緣層（CDN/WAF）：IP 級限制

  • 應用層：業務邏輯級限制
  • 多層防護、逐層收緊
