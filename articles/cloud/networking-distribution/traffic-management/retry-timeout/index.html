重試與逾時

深入解析分散式系統中重試（Retry）與逾時（Timeout）的設計策略、最佳實踐與常見陷阱

在分散式系統中，暫時性故障（Transient Failures）是不可避免的。網路延遲、服務暫時過載、資料庫連線中斷都可能導致請求失敗。重試（Retry）與逾時（Timeout）是處理這些暫時性故障的兩大基本機制，但不當設計反而會加劇問題。

核心概念

暫時性故障分類

故障類型說明可重試範例
網路暫斷短暫連線中斷是TCP reset
服務過載暫時無法處理是HTTP 503
限流超出速率限制是（延遲後）HTTP 429
資源競爭鎖定衝突是DB deadlock
永久故障邏輯錯誤否HTTP 400/404
認證失敗權限問題否HTTP 401/403

逾時類型

逾時類型說明建議值考量
Connect Timeout建立連線的等待時間1-5 秒網路品質
Read Timeout等待回應的時間5-30 秒下游處理時間
Write Timeout傳送資料的時間5-30 秒資料大小
Idle Timeout連線閒置逾時60-300 秒連線池管理
Total Timeout整體請求逾時視業務包含所有重試

重試策略

策略說明適用場景
Immediate Retry立即重試極短暫故障
Fixed Delay固定間隔重試簡單場景
Exponential Backoff指數退避標準做法
Exponential + Jitter指數退避 + 隨機抖動分散式推薦
Linear Backoff線性增加間隔漸進恢復

架構總覽

指數退避演算法

重試次數基本等待（2^n 秒）加上 Jitter目的
第 1 次1 秒0.5-1.5 秒快速重試
第 2 次2 秒1-3 秒稍等恢復
第 3 次4 秒2-6 秒給更多時間
第 4 次8 秒4-12 秒避免繼續打
第 5 次16 秒8-24 秒最後嘗試

Jitter 的重要性：

無 Jitter有 Jitter
所有客戶端同時重試客戶端分散重試
造成「重試風暴」平滑化負載
放大故障幫助恢復

逾時設計原則

原則說明範例
層級逾時外層 > 內層API GW 30s > Service 10s > DB 5s
預算逾時總逾時分配給各步驟30s 分給 3 個呼叫各 8s
Deadline Propagation傳播剩餘時間gRPC deadline
快速失敗逾時後立即失敗不讓用戶等太久

關鍵比較

重試策略比較

策略恢復速度系統壓力適用複雜度
無重試—無非關鍵/冪等性未知最低
立即重試 ×1最快低網路抖動低
固定 1s ×3快中簡單場景低
指數退避 ×5中低標準做法中
指數 + Jitter中最低分散式系統中

重試 vs 其他恢復機制

機制目的層級觸發
Retry克服暫時故障請求單次失敗
Circuit Breaker停止無效嘗試服務連續失敗
Fallback降級替代業務故障後
Bulkhead隔離影響資源預防性
Timeout限制等待請求時間超限

重試反模式

反模式問題正確做法
無限重試永不放棄、浪費資源設定最大重試次數
無退避重試風暴加重負載指數退避 + Jitter
重試非冪等重複副作用確保冪等或使用冪等 Key
層層重試指數放大效應僅在最外層重試
重試不可重試的錯誤浪費時間區分可重試/不可重試
無逾時請求卡住每個呼叫設逾時
逾時過長資源佔用合理逾時 + 快速失敗

應用場景

API 閘道重試策略

  • 上游回應 5xx → 重試（最多 2 次）
  • 回應 429 → 延遲後重試（遵守 Retry-After）
  • 回應 4xx → 不重試（客戶端錯誤）
  • 逾時 → 重試一次後回 504
  • 總逾時 30 秒（含所有重試）

微服務間通訊

  • Service Mesh（Istio）配置自動重試
  • 重試預算：不超過原始請求的 20%
  • Deadline Propagation：傳播剩餘時間
  • 搭配 Circuit Breaker：連續失敗後停止重試

訊息佇列消費

  • 處理失敗 → 指數退避重新入列
  • 最大重試 5 次
  • 超過重試次數 → Dead Letter Queue
  • DLQ 定期人工或自動處理
