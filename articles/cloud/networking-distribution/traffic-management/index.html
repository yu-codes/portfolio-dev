流量管理

流量控制與路由策略

流量管理是雲端網路架構中控制請求如何被接收、路由與分配至後端服務的核心機制。透過負載均衡器、API Gateway 與 Ingress Controller 的組合，架構師能夠實現精細的流量控制策略，包括灰度發布、A/B 測試、限流熔斷與故障轉移。

核心概念

流量管理涉及以下核心概念與元件：

  • 負載均衡（Load Balancing）：將入站流量分散至多個後端實例。L4 負載均衡基於 TCP/UDP 連接資訊（IP、Port）進行分發；L7 負載均衡解析 HTTP 層級資訊（URL Path、Header、Cookie）實現內容路由。
  • API Gateway：作為所有 API 請求的統一入口，提供認證授權、限流（Rate Limiting）、請求轉換、API 版本管理與開發者入口網站。
  • Ingress Controller：Kubernetes 叢集中管理外部流量進入叢集的元件。根據 Ingress 資源定義的規則，將流量路由至對應的 Service。
  • 流量路由策略：基於權重、Header、Cookie 或地理位置將流量導向不同版本的服務，實現金絲雀部署與 A/B 測試。
  • 限流與熔斷（Rate Limiting & Circuit Breaking）：保護後端服務免受過載。限流控制請求速率，熔斷在下游故障時快速失敗避免級聯故障。
  • 健康檢查（Health Check）：週期性探測後端實例的健康狀態，自動將不健康的實例從負載均衡池中移除。

架構總覽

流量管理的典型架構分為南北向（North-South）與東西向（East-West）兩個維度：

南北向流量（外部到內部）：使用者請求 → DNS → CDN/WAF → Global LB → Regional LB → API Gateway → Ingress Controller → Pod/Container。

東西向流量（服務到服務）：Service A → Service Mesh Sidecar → mTLS → Service Mesh Sidecar → Service B。服務網格處理重試、超時、斷路器與可觀測性。

應用場景

金絲雀部署（Canary Deployment）：新版本上線時先將 5% 的流量導向新版本，監控錯誤率與延遲指標。指標正常則逐步增加至 25%、50%、100%。透過 Ingress Controller 的加權路由或服務網格的流量分割實現。

API 版本管理：API Gateway 根據 URL 路徑（/v1/、/v2/）或 Header（Accept-Version）將請求路由至不同版本的後端。舊版本可設定 Deprecation Header 通知客戶端遷移。

地理路由與合規：全球負載均衡器根據使用者的地理位置將請求路由至最近的區域。歐盟使用者的請求強制路由至歐洲區域以符合 GDPR。

多租戶限流：API Gateway 根據 API Key 或 OAuth Client ID 實施差異化限流。免費方案每分鐘 100 次、付費方案每分鐘 10,000 次，防止單一租戶耗盡共享資源。

故障轉移（Failover）：當主要區域的健康檢查失敗時，DNS 或全球負載均衡器自動將流量切換至備援區域。搭配 Connection Draining 確保現有連接優雅結束。
