AWS CloudFormation 是 AWS 原生的基礎設施即程式碼服務，透過 JSON 或 YAML 模板定義 AWS 資源的完整堆疊。作為 AWS 生態系統中最深度整合的 IaC 工具，CloudFormation 提供了與 AWS 服務的即時同步支援，並由 AWS 管理狀態與執行引擎，無需額外維護基礎設施。

核心概念

基本元素

概念說明用途
Template資源定義的 JSON/YAML 檔案描述所需的 AWS 資源
Stack模板的實例化部署管理一組相關資源
StackSet跨帳號/區域的堆疊集合多帳號統一部署
Change Set預覽變更的差異安全審查變更影響
Nested Stack模板中引用其他模板模組化與重用
Drift Detection偵測手動變更確保基礎設施一致性

Parameters:       # 輸入參數

CloudFormation 執行流程

模板上傳 → 驗證 → Change Set 建立 → 審核 → 執行
                                              ↓
                                    資源建立順序計算（DAG）
                                              ↓
                                    平行/序列資源建立
                                              ↓
                                    堆疊建立完成/回滾

堆疊生命週期

狀態說明可執行操作

CREATE_COMPLETE所有資源建立成功更新、刪除
UPDATE_IN_PROGRESS正在更新資源等待
UPDATE_COMPLETE更新成功更新、刪除
ROLLBACK_IN_PROGRESS正在回滾等待
DELETE_IN_PROGRESS正在刪除資源等待
CREATE_FAILED建立失敗刪除

應用場景

多帳號部署（StackSets）

組織根帳號
├── StackSet: 安全基線
│   ├── 帳號 A (us-east-1, eu-west-1)
│   ├── 帳號 B (us-east-1, ap-northeast-1)
│   └── 帳號 C (us-east-1)
├── StackSet: 監控配置
│   └── 所有帳號、所有區域
└── StackSet: 網路基礎設施
    └── 所有帳號、主要區域

資源更新策略

策略說明適用場景
Update原地更新資源可變屬性修改
Replacement建立新資源後刪除舊的不可變屬性修改
Delete then Create先刪後建名稱衝突場景
Retain刪除堆疊時保留資源資料保護

進階模式

自定義資源（Custom Resource）： 透過 Lambda 函數擴展 CloudFormation 的能力，管理非 AWS 資源或執行自定義邏輯。

宏（Macros）： 在模板處理時進行自定義轉換，實現模板級別的程式碼重用。

Guard Hooks： 在資源建立前執行策略驗證，確保合規性。

最佳實踐

  1. 使用 Change Set：永遠先預覽變更再執行
  2. 啟用終止保護：生產堆疊啟用 TerminationProtection
  3. 參數化模板：使用 Parameters 實現環境差異化
  4. 模組化設計：大型基礎設施拆分為多個嵌套堆疊
  5. 標籤策略：統一的資源標籤便於成本追蹤
  6. Drift 定期檢查：偵測未經 IaC 管理的手動變更

Intrinsic Functions（內建函數）

函數用途範例
!Ref參照參數或資源!Ref MyEC2
!GetAtt取得資源屬性!GetAtt MyEC2.PublicIp
!Sub字串替換!Sub "arn:aws:s3:::${BucketName}"
!Join連接字串!Join ["-", [a, b, c]]
!Select選取清單元素!Select [0, !Ref MyList]
!Split分割字串!Split [",", !Ref MyString]
!If條件判斷!If [CreateProd, m5.large, t3.micro]
!FindInMap從 Mappings 查值!FindInMap [RegionAMI, !Ref "AWS::Region", HVM64]
Fn::GetAZs取得區域的可用區清單!Select [0, !GetAZs '']

Mappings（靜態對照表）

Mappings:
  RegionAMI:
    us-east-1:
      HVM64: ami-12345678
    us-west-2:
      HVM64: ami-87654321

Resources:
  MyEC2:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: !FindInMap [RegionAMI, !Ref "AWS::Region", HVM64]

Outputs（輸出）與跨 Stack 參照

# 匯出
Outputs:
  VPCId:
    Description: VPC ID
    Value: !Ref MyVPC
    Export:
      Name: !Sub "${AWS::StackName}-VPCID"

# 匯入（其他 Stack 中）
SubnetId: !ImportValue MyStack-VPCID

AWS Elastic Beanstalk

  • 平台即服務（PaaS），簡化應用程式部署與管理
  • 自動處理容量配置、負載平衡、自動擴展與應用健康監控
  • 支援多種平台：Java (Tomcat)、PHP、Python、Node.js、Ruby、.NET、Docker
  • 支援 Web Tier（HTTP 服務）與 Worker Tier（處理背景任務）
  • 本身免費，僅對應資源（EC2、ELB、ASG）計費
  • 支援自訂 AMI、多版本佈署與版本回滾

AWS OpsWorks

  • 託管 Chef 與 Puppet 的設定管理服務（Configuration as Code）
  • 可用來自動佈署、配置與維護 EC2 環境
  • 支援 Stack 架構將應用拆分為多層（如資料庫層、應用伺服器層）

AWS ParallelCluster

  • 利用簡單設定檔即可快速部署與管理高效能運算（HPC）叢集
  • 完全控制底層資源，免費使用（資源另計）
  • 可結合 Elastic Fabric Adapter（EFA）取得低延遲網路

Terraform 深度解析

核心特性

  • 多雲支持：支援 AWS、Azure、Google Cloud、本地資源等多個雲平台
  • 開源免費：由 HashiCorp 開發並開源
  • 宣告式語言：使用 HCL（HashiCorp Configuration Language）
  • 業界標準：廣泛應用於企業和開源社群
  • 狀態管理：透過 State File 追蹤資源狀態
  • Plan 機制：執行前預覽變更

基本概念

概念說明
Provider定義與哪個雲平台互動（如 aws、azurerm）
Resource定義要建立的基礎設施元素
Variable輸入變數，增加可重用性
Output輸出值，供其他配置參照
Module可重用的 Terraform 程式碼集合
State File記錄實際部署的資源狀態
Backend儲存 State File 的位置（本地、S3、Terraform Cloud）

Terraform 生命週期

terraform init     # 初始化，下載 Provider
terraform plan     # 預覽變更（重要！）
terraform apply    # 執行變更
terraform destroy  # 刪除資源

基本語法範例

# Provider 配置
provider "aws" {
  region = "us-east-1"
}

# 定義變數
variable "instance_type" {
  type        = string
  default     = "t3.micro"
  description = "EC2 Instance Type"
}

# 建立資源
resource "aws_instance" "web_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = var.instance_type

  tags = {
    Name = "WebServer"
  }
}

# 輸出值
output "public_ip" {
  value       = aws_instance.web_server.public_ip
  description = "Public IP of the EC2 instance"
}

Terraform State Management（狀態管理）

State File 包含 Terraform 管理資源的現況，用於確定何時需要建立/更新/刪除資源，注意：包含密碼、金鑰等敏感資訊。

Backend說明用途
Local本地文件系統個人開發
S3AWS S3 Bucket團隊協作，支援遠程鎖定
Terraform CloudHashiCorp 管理企業級，含版本控制、權限管理
Consul分散式儲存高可用性

遠程 State 最佳實踐：

terraform {
  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "prod/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true              # 啟用加密
    dynamodb_table = "terraform-locks" # 狀態鎖定
  }
}

Terraform Module（模組化）

my-module/
├── main.tf          # 主要資源定義
├── variables.tf     # 輸入變數
├── outputs.tf       # 輸出值
└── terraform.tfvars # 變數賦值

module "vpc" {
  source   = "./modules/vpc"
  vpc_cidr = "10.0.0.0/16"
  name     = "my-vpc"
}

# 參照模組輸出
resource "aws_security_group" "app" {
  vpc_id = module.vpc.vpc_id
}

Terraform 工作流程最佳實踐

  1. Version Control：將 .tf 文件存放在 Git 中
  2. Remote State：使用 S3 + DynamoDB 管理狀態
  3. Pre-commit Hooks：自動格式化和驗證
  4. CI/CD 整合：GitHub Actions、GitLab CI 自動執行 plan/apply
  5. 環境隔離：使用 Workspace 或不同目錄管理開發/測試/生產環境
  6. 安全敏感資訊：使用 aws_secretsmanager 存放密鑰，不提交到 Git
