IaC

基礎設施即程式碼

基礎設施即程式碼（Infrastructure as Code, IaC）是以程式碼定義、佈建與管理雲端基礎設施的方法論。透過 IaC，團隊將基礎設施的期望狀態寫入版本控制的程式檔案中，工具自動將實際環境收斂至目標狀態。這消除了手動操作的不確定性，讓基礎設施的管理具備可重複性、可審計性與一致性。

核心概念

IaC 的核心概念圍繞著將基礎設施管理軟體工程化：

  • 宣告式（Declarative）：描述「期望的最終狀態」而非「執行步驟」。Terraform HCL 與 CloudFormation YAML 都屬宣告式。工具自動計算從當前狀態到目標狀態的差異並執行變更。
  • 命令式（Imperative）：明確定義執行的每一個步驟。Pulumi 使用通用程式語言（Python、TypeScript）即偏向命令式風格，但仍追蹤狀態。
  • 狀態檔案（State File）：記錄已部署資源與其屬性的檔案。Terraform 的 terraform.tfstate 是狀態管理的核心，用於偵測漂移與規劃變更。
  • Provider：IaC 工具與各雲端 API 溝通的外掛模組。AWS Provider、Azure Provider 等封裝了資源建立、讀取、更新與刪除（CRUD）的邏輯。
  • 模組（Module）：可重用的 IaC 程式碼單元，將一組相關資源封裝為可參數化的元件，促進 DRY 原則。
  • 計劃（Plan）：在實際套用變更之前，預覽將要執行的新增、修改與刪除操作，提供安全的變更審查機制。

架構總覽

IaC 的技術架構通常包含以下元件：

程式碼層：Terraform .tf 檔案、CloudFormation 模板或 Pulumi 程式碼。按環境與模組組織目錄結構。

狀態管理層：遠端後端儲存狀態檔案。Terraform 常用 S3 + DynamoDB（鎖定）、Terraform Cloud 或 Consul。CloudFormation 狀態由 AWS 完全託管。

執行層：CLI 工具或 CI/CD 管線執行 init → plan → apply 流程。支援並行操作與相依性圖解析。

Registry/模組庫：公開或私有的模組註冊表，提供經過驗證的可重用元件。Terraform Registry 擁有數千個社群模組。

應用場景

環境複製：使用 Terraform Workspace 或目錄結構，以相同模組產生開發、測試與生產三套完全一致的環境。變數檔案控制實例大小與數量的差異。

基礎設施模組目錄：平台團隊建立內部模組庫，封裝合規與安全最佳實踐（如強制加密、日誌啟用）。開發團隊引用模組即可獲得合規基礎設施。

多帳戶治理：使用 Terraform 搭配 AWS Organizations，自動化新帳戶的建立、基線安全配置（GuardDuty、CloudTrail）與網路互聯。

成本預估：terraform plan 搭配 Infracost 工具，在 PR 階段即可預覽基礎設施變更的月度成本影響，將成本審查左移至開發流程。

災難復原演練：定期在備援區域執行 terraform apply 驗證完整環境可成功佈建，確保 DR 計劃的有效性。
