無狀態 vs 有狀態服務

在雲端應用架構中，服務的狀態管理是最關鍵的設計決策之一。無狀態（Stateless）與有狀態（Stateful）服務各有其適用場景，正確的選擇直接影響系統的可擴展性、可用性與運維複雜度。理解兩者的本質差異，是設計雲原生應用的基礎能力。

核心概念

無狀態服務

無狀態服務不在本地保存任何請求間的狀態資訊。每個請求都是獨立的，包含處理該請求所需的全部資訊。服務實例之間完全等價，任何實例都能處理任何請求。

特徵：

  • 不依賴本地記憶體或磁碟存儲會話資料
  • 請求可被任意實例處理
  • 水平擴展簡單直接
  • 實例可隨時創建或銷毀

有狀態服務

有狀態服務在處理過程中維護某種形式的本地狀態。這些狀態可能是會話資訊、快取資料、或業務邏輯所需的持久化資料。

特徵：

  • 依賴本地存儲或記憶體中的狀態
  • 特定請求可能需要路由到特定實例
  • 擴展需要考慮狀態遷移
  • 實例失敗可能導致狀態丟失

架構總覽

無狀態架構模式

客戶端 → 負載均衡器 → [任意服務實例] → 外部狀態存儲
                                              ├── Redis (Session)
                                              ├── S3 (Files)
                                              └── RDS (Data)

無狀態架構將所有狀態外部化到專門的存儲服務中。服務本身僅負責業務邏輯的計算，狀態的讀取與寫入透過外部服務完成。

有狀態架構模式

客戶端 → 負載均衡器 (Sticky Session) → [指定服務實例]
                                              ├── 本地快取
                                              ├── 記憶體狀態
                                              └── 本地磁碟

有狀態架構透過會話親和性（Session Affinity）確保請求被路由到持有相關狀態的實例。

關鍵比較

維度無狀態服務有狀態服務
擴展性簡單，直接增減實例複雜，需處理狀態分片與遷移
可用性高，任何實例故障不影響服務需額外機制保障（主從複製）
延遲較高（每次存取外部存儲）較低（本地狀態存取）
一致性由外部存儲保證需處理分散式一致性
部署複雜度低高
故障恢復快速，新實例即可服務需狀態恢復或重建
資源效率可能較低（外部調用開銷）可能較高（本地存取）
測試難度低，無副作用高，需模擬狀態

狀態外部化策略比較

策略適用場景優勢劣勢
Redis/Memcached會話、快取低延遲、高吞吐記憶體成本

應用場景

適合無狀態的場景

Web API 服務： RESTful API 天然是無狀態的。每個 HTTP 請求包含完整的認證資訊（JWT Token）與業務參數，不依賴伺服器端的會話狀態。

函數計算： AWS Lambda、Google Cloud Functions 等 FaaS 平台設計上就是無狀態的。每次調用都是獨立的，函數實例可能在請求之間被銷毀。

資料處理管線： ETL 任務、批次處理等工作負載，每個處理單元獨立處理一批資料，不依賴其他單元的狀態。

適合有狀態的場景

資料庫服務： PostgreSQL、MongoDB 等資料庫天然是有狀態的，需要持久化存儲與一致性保證。

即時通訊： WebSocket 連線維持客戶端的會話狀態，需要將特定客戶端的訊息路由到持有其連線的實例。

機器學習推論： 載入大型模型到記憶體中，模型本身就是一種狀態。頻繁載入/卸載模型會造成嚴重的效能問題。

混合策略

現代架構通常採用混合策略：應用層保持無狀態以利於擴展，將狀態管理委託給專門的有狀態服務（託管資料庫、快取服務等）。

[無狀態應用層]          [有狀態資料層]
├── API Gateway         ├── Aurora (SQL)
├── Business Logic      ├── ElastiCache (Cache)
├── Auth Service        ├── DynamoDB (NoSQL)
└── Notification        └── S3 (Object Store)

設計決策框架

選擇無狀態或有狀態設計時，考慮以下問題：

  1. 擴展需求：是否需要快速水平擴展？→ 優先無狀態
  2. 延遲要求：是否對延遲極為敏感？→ 考慮有狀態本地快取
  3. 資料一致性：是否需要強一致性保證？→ 有狀態 + 共識協議
  4. 故障容忍：是否能接受短暫的狀態丟失？→ 無狀態 + 外部存儲
  5. 運維能力：團隊是否有管理有狀態服務的經驗？→ 影響方案選擇
