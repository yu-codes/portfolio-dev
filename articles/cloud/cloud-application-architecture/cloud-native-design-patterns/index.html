雲原生設計模式

系統性介紹雲原生應用的核心設計模式，涵蓋彈性、可擴展性與分散式系統模式

雲原生設計模式是針對分散式雲端環境所發展出的架構模式集合。這些模式解決了雲端環境中特有的挑戰：網路不可靠、服務可能隨時失敗、需要動態擴展。掌握這些模式是設計高可用、高彈性雲端應用的關鍵。

核心概念

雲原生設計模式可分為以下幾大類：

類別目標代表模式
彈性模式容錯與自癒Circuit Breaker、Retry、Bulkhead
擴展模式動態擴縮容CQRS、Event Sourcing、Sharding
通訊模式服務間互動API Gateway、Service Mesh、Saga
部署模式安全發佈Blue-Green、Canary、Feature Flag
資料模式分散式資料管理Database per Service、Event-Driven

架構總覽

彈性模式家族

請求 → [Retry] → [Circuit Breaker] → [Timeout] → [Bulkhead] → 目標服務
         ↓              ↓                              ↓
      指數退避       Fallback                       資源隔離

斷路器（Circuit Breaker）： 當下游服務失敗率超過閾值時，自動中斷請求，避免級聯失敗。三種狀態：Closed（正常）→ Open（中斷）→ Half-Open（試探）。

重試（Retry with Backoff）： 對暫時性故障自動重試，使用指數退避策略避免雪崩效應。

艙壁（Bulkhead）： 將系統資源分隔為獨立的池，一個部分的失敗不會耗盡整個系統的資源。

超時（Timeout）： 為每個外部呼叫設定合理的超時時間，避免請求無限等待。

擴展模式

CQRS（命令查詢職責分離）：

面向命令端查詢端
職責寫入操作讀取操作
模型領域模型讀取模型
存儲事務型資料庫讀取優化存儲
擴展獨立擴展獨立擴展
一致性強一致最終一致

Event Sourcing： 將狀態變更記錄為不可變的事件序列，而非僅存儲當前狀態。

API Gateway 模式

API Gateway 作為所有客戶端請求的統一入口，提供：

  • 請求路由：將請求分發到對應的後端服務
  • 協議轉換：REST ↔ gRPC、WebSocket 等
  • 認證授權：集中處理身份驗證
  • 限流熔斷：保護後端服務
  • 回應聚合：合併多個服務的回應

Sidecar 模式

將橫切關注點（日誌、監控、安全、通訊）從應用程式碼中抽離，部署為獨立的 Sidecar 容器：

Sidecar 職責實現工具對應用的影響
服務通訊Envoy/Istio透明代理
日誌收集Fluentd無侵入
密鑰注入Vault Agent自動更新
健康檢查自定義標準化

反壓（Backpressure）模式

當下游服務無法承受上游的請求量時，透過反壓機制通知上游降低發送速率：

  1. 佇列限制：設定佇列最大深度
  2. 流量整形：使用令牌桶或漏桶演算法
  3. 負載卸除：在過載時主動丟棄低優先級請求
  4. 自適應限流：根據系統負載動態調整限流閾值
