雲端儲存與資料平台是現代應用程式架構的基石。從簡單的檔案儲存到複雜的資料湖（Data Lake）與資料倉儲（Data Warehouse），雲端供應商提供了豐富的選項來滿足不同的效能、成本與合規需求。本文將系統性地介紹雲端儲存與資料平台的核心概念、架構設計，以及如何根據業務需求選擇合適的方案。

核心概念

儲存的三種基本類型

雲端儲存根據存取模式與資料結構，分為三種基本類型：

  1. 物件儲存（Object Storage）：以扁平命名空間儲存非結構化資料，適合圖片、影片、備份、日誌等。代表服務：AWS S3、Azure Blob Storage、GCS
  2. 區塊儲存（Block Storage）：提供低延遲的區塊級存取，作為虛擬機器的磁碟使用。代表服務：AWS EBS、Azure Managed Disks、GCP Persistent Disk
  3. 檔案儲存（File Storage）：提供共享檔案系統介面（NFS/SMB），支援多個實例同時掛載。代表服務：AWS EFS、Azure Files、GCP Filestore

資料溫度分層

根據存取頻率，資料可分為不同的溫度層級，直接影響儲存成本：

  • 熱資料（Hot）：頻繁存取，毫秒級延遲需求，成本最高
  • 溫資料（Warm）：偶爾存取，可接受稍高延遲，成本適中
  • 冷資料（Cool）：很少存取，通常用於合規保留，成本較低
  • 歸檔資料（Archive）：極少存取，恢復需數小時，成本最低

CAP 定理與一致性模型

分散式儲存系統面臨 CAP 定理的取捨：

  • 強一致性（Strong Consistency）：寫入後立即可讀取最新值，但延遲較高
  • 最終一致性（Eventual Consistency）：寫入後經過短暫延遲才能保證讀取最新值，但效能較佳
  • 因果一致性（Causal Consistency）：保證因果相關的操作順序一致

資料湖 vs 資料倉儲 vs 湖倉一體

  • 資料湖（Data Lake）：儲存原始資料（結構化與非結構化），Schema-on-Read，成本低但查詢需額外計算引擎
  • 資料倉儲（Data Warehouse）：儲存經清洗轉換的結構化資料，Schema-on-Write，查詢效能優異但成本較高
  • 湖倉一體（Lakehouse）：結合兩者優勢，在物件儲存上實現 ACID 事務與 Schema 管理（如 Delta Lake、Apache Iceberg）

應用場景

場景一：全球化應用的資料複寫

跨國應用需要在多個區域存放資料副本，以降低延遲並滿足資料主權要求。雲端儲存支援跨區域複寫（Cross-Region Replication），可設定選擇性複寫規則，僅同步特定前綴或標籤的物件。

場景二：成本最佳化的生命週期策略

透過智能分層策略（Intelligent Tiering），自動將不常存取的資料移至低成本儲存層。典型配置為：30 天後移至 Infrequent Access、90 天後移至 Glacier/Archive、365 天後依合規需求決定保留或刪除。

場景三：即時分析管線

建構從資料擷取到洞察的完整管線：串流資料透過 Kinesis/Kafka 寫入物件儲存（Parquet 格式），再由 Spark 或 Athena 執行 SQL 查詢，最終透過 BI 工具視覺化呈現。

場景四：災難復原（DR）策略

依據 RPO（Recovery Point Objective）與 RTO（Recovery Time Objective）需求，設計適當的備份與複寫策略。關鍵資料庫採用同步複寫（RPO=0），一般應用資料採用非同步複寫（RPO=分鐘級）。
