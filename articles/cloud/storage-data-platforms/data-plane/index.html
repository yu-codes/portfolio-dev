資料層

物件、區塊與檔案儲存

資料平面（Data Plane）是雲端儲存架構中直接處理資料讀寫操作的層級。它涵蓋了所有實際儲存與檢索資料的服務，包括物件儲存、區塊儲存與託管資料庫。理解資料平面的特性與取捨，是設計高效能、低成本雲端架構的關鍵。

核心概念

資料存取模式

不同的工作負載有截然不同的存取模式，選擇儲存方案時必須考量：

  • 循序存取（Sequential Access）：如日誌寫入、串流媒體讀取，適合物件儲存或 append-only 設計
  • 隨機存取（Random Access）：如資料庫 IOPS、虛擬機器磁碟操作，需要區塊儲存的低延遲
  • 大量小檔案：如機器學習訓練集中的圖片，需注意物件儲存的 LIST 操作效能
  • 少量大檔案：如影片檔案、資料庫備份，需要高吞吐量的串流傳輸

IOPS、吞吐量與延遲

這三個指標定義了儲存系統的效能特徵：

  • IOPS（I/O Operations Per Second）：每秒能處理的讀寫操作數，對資料庫工作負載至關重要
  • 吞吐量（Throughput）：單位時間傳輸的資料量（MB/s），對大量資料搬移至關重要
  • 延遲（Latency）：單次操作從請求到回應的時間，對即時應用至關重要

資料持久性與可用性

  • 持久性（Durability）：資料不遺失的機率，物件儲存通常提供 99.999999999%（11 個 9）
  • 可用性（Availability）：服務可正常存取的時間比例，通常 99.9%-99.99%
  • 兩者的差異：資料可能是持久的（不會遺失）但暫時不可用（服務中斷）

物件儲存效能最佳化

物件儲存的效能瓶頸通常在於：

  1. 前綴熱點（Prefix Hotspot）：大量請求集中在相同前綴，解法是使用隨機化前綴或 hash
  2. 小檔案問題：大量小物件的 LIST/GET 操作昂貴，可透過打包（Tar/Parquet）或使用 S3 Express One Zone 改善
  3. 多部分上傳（Multipart Upload）：大檔案應拆分為多個部分平行上傳，提升吞吐量

應用場景

場景一：資料湖建構

以物件儲存作為中央資料存放區，使用 Apache Parquet 或 ORC 格式儲存結構化資料，搭配 Delta Lake/Iceberg 提供 ACID 事務支援。計算引擎（Spark、Presto、Athena）直接查詢物件儲存中的資料，實現儲存與計算分離。

場景二：高效能資料庫部署

關鍵 OLTP 工作負載需要最低延遲的區塊儲存。使用 io2 Block Express（AWS）或 Ultra Disk（Azure）提供百萬級 IOPS，搭配多 AZ 部署確保高可用。資料庫的 WAL（Write-Ahead Log）寫入需要一致性的低延遲保證。

場景三：全球分散式應用

使用 Cosmos DB 或 DynamoDB Global Tables 實現多區域的資料同步。應用程式寫入最近的區域，資料自動複寫至其他區域。讀取操作可設定一致性等級：最終一致性提供最低延遲，強一致性確保資料正確。

場景四：媒體資產管理

影音平台需處理大量媒體檔案：原始檔案存放於物件儲存標準層，轉碼後的不同解析度版本依存取頻率自動分層，縮圖和元資料存放於 CDN 快取。透過事件通知觸發轉碼工作流程。
