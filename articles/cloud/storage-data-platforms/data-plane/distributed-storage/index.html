分散式儲存系統將資料分散儲存在多個節點上，提供高可用性、容錯能力與水平擴展能力。理解分散式儲存的理論基礎與實作策略，是設計可靠雲端資料平台的關鍵。

核心概念

CAP 定理

在分散式系統中，以下三個特性最多只能同時滿足兩個：

特性說明犧牲時的影響
Consistency（一致性）所有節點同時看到相同資料讀取可能不是最新
Availability（可用性）每次請求都能得到回應可能暫時無法服務
Partition Tolerance（分區容忍）網路分區時仍能運作—

CAP 組合：

組合犧牲代表系統適用場景
CP可用性Spanner、HBase、ZooKeeper金融、強一致需求
AP一致性DynamoDB、Cassandra、CouchDB高可用、容忍暫時不一致
CA分區容忍傳統單機 RDBMS不適合分散式

PACELC 定理

CAP 的延伸：在沒有分區（Partition）時，需要在延遲（Latency）和一致性（Consistency）之間取捨。

情況選擇系統範例
分區時選 A正常時選 LDynamoDB、Cassandra
分區時選 C正常時選 CSpanner、VoltDB
分區時選 A正常時選 C—

一致性等級

等級說明延遲適用
強一致（Linearizable）讀取永遠最新最高金融交易
順序一致操作有全域順序高分散式鎖
因果一致因果相關操作有序中社群動態
最終一致終將收斂最低非關鍵讀取
讀己所寫自己的寫入立即可見中用戶體驗

架構總覽

資料分片（Sharding）策略

策略說明優點缺點
範圍分片按 Key 範圍範圍查詢友善熱點問題
雜湊分片按 Key 雜湊均勻分散範圍查詢困難
目錄分片查找表對應靈活查找表瓶頸
地理分片按地理位置就近存取跨區查詢
時間分片按時間範圍時序查詢熱點集中

資料複寫策略

策略寫入延遲一致性可用性資料遺失風險
同步複寫（全部）最高強一致低無
同步複寫（多數）高強一致中極低
半同步中近一致中高極低
非同步最低最終一致最高有（視延遲）

共識演算法

演算法說明使用系統特點
Paxos經典共識Spanner、Chubby理論完善但複雜
Raft簡化共識etcd、CockroachDB易理解實作
ZabZooKeeper 專用ZooKeeper總序廣播
PBFT拜占庭容錯區塊鏈容忍惡意節點

分散式儲存系統比較

系統類型CAP一致性擴展
Cassandra寬列AP可調（Quorum）線性
HBase寬列CP強一致線性
MongoDB文件CP（預設）可調分片
CockroachDB關聯式CP強一致自動分片

Spanner關聯式CP外部一致性全球分散
DynamoDB鍵值AP可選強一致自動
Ceph物件/區塊/檔案CP強一致線性

關鍵比較

Quorum 讀寫機制

設定寫入節點（W）讀取節點（R）一致性可用性
W=N, R=1全部一個強一致寫可用低
W=1, R=N一個全部強一致讀可用低
W>N/2, R>N/2多數多數強一致平衡
W=1, R=1一個一個最終一致最高

衝突解決策略

策略說明適用範例
Last-Write-Wins最後寫入覆蓋簡單場景Cassandra
Vector Clock因果追蹤需偵測衝突DynamoDB
CRDT無衝突複寫資料型態自動合併Riak
Application Level應用層解決複雜業務邏輯購物車合併
Operational Transform操作轉換協作編輯Google Docs

模式一致性效能複雜度適用
2PC強一致低（阻塞）中傳統分散式 DB
3PC強一致中高理論（少用）
Saga最終一致高中微服務

事件溯源最終一致高中事件驅動

應用場景

全球分散式應用

  • Google Spanner：全球強一致
  • 使用 TrueTime API 實現外部一致性
  • 自動資料分片與複寫
  • 適合需要全球事務的金融系統

大規模時序資料

  • 時間分片 + 範圍查詢
  • 近期資料：高效能儲存
  • 歷史資料：自動降精度 + 歸檔
  • Bigtable/TimescaleDB/InfluxDB

多主寫入架構

  • 多區域同時寫入
  • 衝突偵測與解決（CRDT/Vector Clock）
  • 最終一致性模型
  • 適合用戶就近寫入的場景
