託管式資料庫（Managed Database / DBaaS）是雲端供應商提供的全託管資料庫服務，處理備份、修補、高可用與擴展等維運工作。開發者只需專注於 Schema 設計與查詢最佳化，無需管理底層基礎設施。

應用場景

微服務資料庫策略

  • Database per Service（每服務獨立 DB）
  • 服務間透過 API/事件同步
  • 混合使用 SQL + NoSQL
  • Saga 管理跨服務事務

讀寫分離架構

  • Primary：處理所有寫入
  • Read Replica（多個）：處理讀取
  • 應用層路由讀寫請求
  • 最終一致性可接受的場景

全球分散式應用

  • Aurora Global Database / Spanner / Cosmos DB
  • 每個區域有本地讀取
  • 寫入路由到主區域（或多寫入）
  • RPO 近零的災難復原

延伸閱讀

  • AWS RDS 最佳實踐
  • GCP Cloud SQL 效能調校指南
  • Azure Cosmos DB 設計模式
  • 從自建到託管資料庫的遷移策略

AWS 服務實作

DynamoDB（NoSQL）

核心概念

  • AWS 託管的 Serverless NoSQL 資料庫
  • 適合儲存 JSON 結構資料；毫秒等級延遲，適合會話資料與即時查詢
  • 無限自動水平擴展；每帳號可建立無限數量的表格

表格結構

概念說明
Table基本單位，是項目的集合
Item表格中的一筆資料（類似行），最大 400 KB
Attribute項目中的屬性（類似欄位）
Primary Key唯一識別項目的鍵

主鍵類型：

類型組成使用場景
Simple（Partition Key）PK唯一值即可識別項目
CompositePK + SK（Sort Key）需要組合鍵識別項目
  • Partition Key（PK）：決定資料儲存位置
  • Sort Key（SK）：可選，允許同一 PK 下有多個項目

容量模式

模式說明適用場景
Provisioned預先配置 RCU/WCU；每個表格有 300 秒突發池可預測的工作負載
On-demand按請求計費不可預測/未知的工作負載

容量單位計算：

操作類型容量單位
1 WCU每秒寫入 1 KB
1 RCU（強一致性）每秒讀取 4 KB
1 RCU（最終一致性）每秒讀取 8 KB

DAX（DynamoDB Accelerator）

  • 記憶體內快取服務，與 DynamoDB API 相容
  • 微秒等級延遲（比 DynamoDB 毫秒更快）
  • 加速查詢並降低熱點負載

DynamoDB Streams

  • 將資料異動事件串流（INSERT/UPDATE/DELETE）
  • 可觸發 Lambda 函數；保留 24 小時
  • 使用案例：新增使用者即寄送歡迎信；事件驅動架構

Global Table

  • 支援多區域同步讀寫（Active-Active 複製）
  • 需先啟用 Streams
  • 適用於全球應用場景

索引類型

類型說明
Local Secondary Index（LSI）與基礎表相同的 PK，不同 SK
Global Secondary Index（GSI）可使用不同的 PK 和 SK

讀取一致性

類型說明
Eventually Consistent最終一致性，延遲較低，消耗 0.5 RCU
Strongly Consistent強一致性，保證讀取最新資料，消耗 1 RCU

資料遷移

可透過 Amazon DMS 將以下來源遷移至 DynamoDB：

  • MongoDB、Oracle、MySQL、S3

Amazon Redshift（資料倉儲）

  • PB 級資料倉儲服務，使用 **MPP（大規模平行處理）**架構
  • 支援欄式儲存，優化分析查詢效能；與標準 SQL 相容
功能說明
Concurrency Scaling自動擴展處理並發查詢
Materialized Views預先計算複雜查詢結果
Redshift ML在 Redshift 中建立/部署 ML 模型
Redshift Serverless無需管理叢集即可運行資料倉儲，適合變動分析負載

NoSQL 架構概念

  • Key-Value 模型：無結構，無關聯性，非常快速，可高度擴展
  • Wide Column Store：動態欄位，Partition Key（必需）+ 其他鍵
  • Document 模型：每個 Item 都是 Document（JSON/XML），理想用於訂單、聯絡人、文件結構
  • NoSQL 傾向在一個表格中處理所有資料：讀取較便宜但寫入較貴

RDS（Relational Database Service）

核心概念

  • AWS 託管的關聯式資料庫服務（DBaaS）
  • 支援引擎：PostgreSQL、MySQL、MariaDB、Oracle、SQL Server、Amazon Aurora
  • 在 VPC 內的子網路中運行，使用 DB Subnet Group；無法存取作業系統或 SSH
  • 成本因素：引擎、類型、大小、Multi-AZ；預設使用 EBS：gp2（1GB-16TB）

Read Replica（讀取副本）

  • 支援最多 5 個 Direct Read Replica（每個 Replica 可再有最多 5 個）
  • 非同步複製，讀取為最終一致性；Read Replica 只能讀取，不能寫入
  • 可部署於不同區域；跨區複製需額外付費，跨 AZ 不需額外費用
  • 全球效能改善；可與 Multi-AZ 結合使用

Multi-AZ（高可用性）

  • 使用 Multi-AZ 部署，支援同步複製
  • 發生異常時自動將 CNAME 指向備援 DB 實例；60-120 秒的故障轉移時間
  • 備用實例在故障轉移前不能讀取
  • 備份從備用實例進行（減少主要實例負擔）

Multi-AZ 部署選項：

類型說明
Multi-AZ with Standby單一備援實例，同步複製
Multi-AZ DB Cluster1 Writer + 2 Readers，跨 3 AZ，寫入確認後才回傳，故障轉移約 35 秒（+交易日誌應用）

Multi-AZ vs Read Replica 比較：

類型同步方式用途
Multi-AZ同步高可用性
Read Replica非同步效能擴展

備份與還原

  • 自動備份：每天快照 + 每 5 分鐘交易日誌備份
  • 保留期：1-35 天（Console 預設 7 天，CLI 預設 1 天）；需要 90 天可搭配 AWS Backup
  • 備份到 S3
備份類型還原精度說明
自動備份任何 5 分鐘的時間點Point-in-Time Recovery
快照（手動）單一時間點手動觸發

還原時會建立新的 RDS 實例和新的端點位址

安全性

  • IAM Database Authentication：讓 IAM 使用者透過 IAM 權限登入資料庫（僅支援 MySQL 與 PostgreSQL）
  • SSL/TLS：傳輸中加密（可配置為強制 SSL）
  • 靜態加密：由 EBS 和 KMS 處理，資料離開主機後即被加密（含儲存、日誌、快照、副本）
  • AWS Secrets Manager 整合：自動輪替資料庫認證，安全儲存資料庫連線資訊

加密未加密的資料庫步驟：

  1. 建立 Snapshot → 2. 複製並加密此快照 → 3. 從加密快照還原為新資料庫

Enhanced Monitoring

  • 檢視每個資料庫程序的 CPU 與記憶體使用率
  • 提供更詳細的系統層級指標

Amazon Aurora

概述

  • AWS 自有、相容 MySQL 與 PostgreSQL 的高效能關聯式資料庫
  • 效能為 MySQL 的 5 倍，PostgreSQL 的 3 倍
  • 儲存自動擴展（最高 128TB）

Aurora Cluster 架構

元件說明
Primary DB instance主節點，支援讀寫
Aurora Replica最多 15 個，只支援讀取，支援自動故障轉移（通常 < 30 秒）

Aurora Endpoint 類型

Endpoint 類型說明
Cluster Endpoint連至主節點，可執行讀寫操作
Reader Endpoint讀取節點入口，支援讀取負載平衡
Custom Endpoint最多 5 個，自訂節點群組
Instance Endpoint直接連至指定實例

Aurora Global Database

  • 跨區域部署一個資料庫，實現低延遲全球讀取與災難復原
  • RPO 1 秒，RTO 1 分鐘

Aurora Serverless v2

  • 可自動擴展資源；適用於間歇性或難以預測的負載
  • 比 v1 更快擴展、更精細的容量調整
