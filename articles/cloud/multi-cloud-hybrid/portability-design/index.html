可攜性設計

可攜性設計（Portability Design）是多雲架構中降低供應商鎖定（Vendor Lock-in）風險的核心策略。透過適當的抽象層設計與標準化技術選型，企業可以保持在不同雲端供應商之間遷移工作負載的靈活性，同時避免為了可攜性而犧牲過多的雲端原生優勢。

核心概念

供應商鎖定的類型

供應商鎖定並非只有一種形式，理解不同類型的鎖定有助於制定針對性的應對策略：

  1. 技術鎖定：使用供應商專有的 API、SDK 或服務。例如 AWS Lambda 的事件格式、Azure Cosmos DB 的一致性模型。

  2. 資料鎖定：資料以供應商特有的格式儲存，或由於資料量過大而難以搬遷（Data Gravity）。

  3. 架構鎖定：系統架構深度依賴供應商的特定服務組合，重新設計成本極高。

  4. 合約鎖定：長期的承諾使用折扣（如 Reserved Instances、Committed Use Discounts）產生的經濟約束。

  5. 技能鎖定：團隊的專業知識集中於特定平台，切換成本高。

可攜性的光譜

可攜性不是全有或全無的選擇，而是一個光譜：

  • 完全可攜：使用純開源、標準化的技術棧，可在任何環境運行。但可能犧牲效能與開發效率。
  • 策略性可攜：關鍵元件保持可攜性，非關鍵元件允許使用雲端原生服務。平衡靈活性與效率。
  • 完全原生：深度使用雲端原生服務，最大化效率但最高鎖定風險。

設計原則

  1. 抽象原則：透過介面和抽象層隔離供應商特定的實作細節。
  2. 標準優先：優先選用開放標準和業界通用的技術。
  3. 資料可攜：確保資料可以用標準格式匯出並在其他平台匯入。
  4. 漸進式承諾：從低鎖定的服務開始，在確認需求穩定後再考慮更深度的整合。

關鍵可攜性技術

容器化（Containerization）

Docker 容器提供應用程式層級的可攜性。透過將應用程式及其相依性打包成容器映像，可以在任何支援容器的環境中運行。Kubernetes 作為容器編排標準，進一步提供了跨雲的工作負載管理能力。

Infrastructure as Code（IaC）

使用跨雲的 IaC 工具定義基礎設施：

  • Terraform：支援所有主要雲端供應商的統一 HCL 語法。透過 Provider 抽象底層差異。
  • Pulumi：使用通用程式語言（TypeScript、Python、Go）定義基礎設施，提供更強的抽象能力。
  • Crossplane：Kubernetes-native 的基礎設施管理，將雲端資源定義為 Kubernetes Custom Resources。

資料層可攜性

  • 使用開源資料庫（PostgreSQL、MongoDB）而非供應商專有資料庫
  • 使用標準資料格式（Parquet、Avro）儲存分析資料
  • 透過抽象的儲存介面（S3-compatible API）存取物件儲存

應用程式框架

  • Dapr：提供跨雲的分散式應用運行時，抽象狀態管理、事件發布、服務呼叫等能力
  • Knative：跨雲的無伺服器應用框架
  • OpenTelemetry：統一的可觀測性標準

應用場景

場景一：多雲 Kubernetes 平台

企業建立以 Kubernetes 為核心的應用平台，在 AWS EKS 與 GCP GKE 上運行相同的工作負載。使用 Helm Charts 定義應用部署、Crossplane 管理雲端資源、Istio 提供跨叢集的服務網格。當需要切換雲端時，只需調整底層的 Infrastructure Provider。

場景二：抽象儲存層設計

電商平台設計了統一的儲存抽象介面，支援 S3、GCS 與 Azure Blob Storage。應用程式碼只依賴抽象介面，具體的儲存實作透過依賴注入在部署時決定。這使得平台可以根據成本或效能需求在不同雲端之間分配儲存工作負載。

場景三：漸進式去鎖定

一家原本深度使用 AWS 專有服務的新創公司，在獲得企業客戶後需要支援多雲部署。他們採用漸進式策略：先將 Lambda 函數遷移到容器化服務、將 DynamoDB 替換為 MongoDB Atlas、將 SQS 替換為 Kafka。每次替換都透過功能開關（Feature Flag）逐步切換，降低風險。

場景四：資料平台可攜性

資料工程團隊使用 Apache Spark 作為核心處理引擎，資料以 Parquet 格式存放在物件儲存中，中繼資料使用 Apache Hive Metastore 管理。這個架構可以無縫在 AWS EMR、Azure HDInsight 或 GCP Dataproc 之間遷移，因為所有組件都基於開放標準。
