Amazon Web Services（AWS）是全球市佔率最高的公有雲平台，提供超過 200 種服務，涵蓋運算、儲存、資料庫、機器學習、物聯網等領域。理解 AWS 的核心服務架構，是建構現代雲端應用的基礎。

核心概念

區域與可用區域

AWS 的全球基礎設施由 Region（區域） 和 Availability Zone（可用區域，AZ） 組成。每個 Region 包含多個實體隔離的 AZ，確保高可用性與容錯能力。

概念說明範例
Region地理區域，包含多個 AZus-east-1、ap-northeast-1
Availability Zone獨立資料中心群組us-east-1a、us-east-1b
Edge LocationCDN 邊緣節點CloudFront 全球 400+ 節點
Local Zone低延遲本地擴展靠近使用者的運算資源

IAM 身份與存取管理

AWS Identity and Access Management（IAM）是所有服務的安全基礎，採用最小權限原則管理存取控制。

  • User：代表實際操作人員
  • Role：可被服務或使用者暫時承擔的身份
  • Policy：JSON 格式的權限定義文件
  • Group：使用者的邏輯分組

儲存服務

S3（Simple Storage Service） 是 AWS 最具代表性的物件儲存服務：

  • S3 Standard：頻繁存取，99.999999999%（11 個 9）耐久性
  • S3 Intelligent-Tiering：自動分層，適合存取模式不確定的資料
  • S3 Glacier：歸檔儲存，取回時間分鐘到小時不等
  • S3 Glacier Deep Archive：最低成本，取回時間 12-48 小時

網路服務

VPC（Virtual Private Cloud） 是 AWS 網路的基礎：

  • Subnet：公有子網（含 Internet Gateway）與私有子網
  • Security Group：狀態式防火牆，控制入站/出站流量
  • NACL：無狀態網路存取控制清單
  • NAT Gateway：讓私有子網存取外部網路

應用場景

三層架構 Web 應用

典型的 AWS 三層架構：

  1. 前端層：CloudFront + S3 靜態網站 + Route 53
  2. 應用層：ALB + EC2/ECS Auto Scaling Group
  3. 資料層：RDS Multi-AZ + ElastiCache + S3

事件驅動架構

利用 AWS 無伺服器服務組建：

  • API Gateway → Lambda → DynamoDB
  • S3 Event → Lambda → SQS → Lambda
  • EventBridge → Step Functions → 多個 Lambda

大數據分析管線

  • 資料收集：Kinesis Data Streams
  • 資料處理：EMR / Glue
  • 資料儲存：S3 Data Lake + Redshift
  • 資料查詢：Athena（S3 直接查詢）

AWS 服務實作

AWS 雲端的歷史與現況

歷史發展

  • 2002 年：AWS 最初在 Amazon 內部推出，Amazon 發現 IT 基礎設施可被「外部化」
  • 2004 年：首次公開服務，推出 Amazon SQS
  • 2006 年：服務大幅擴展，正式推出 SQS、S3、EC2
  • 之後：從美國擴展至全球（如歐洲）
  • 現今：大量知名企業使用 AWS（Dropbox、Netflix、Airbnb、NASA 等）

市場地位

  • 在 Gartner 魔力象限（Magic Quadrant）中長期為領導者
  • 2023 年營收：約 900 億美元
  • 2024 Q1 市占率：約 31%（微軟 Azure 約 25%，排名第二）
  • 連續 13 年為雲端市場領導者
  • 超過 100 萬名活躍用戶

常見應用類型

  • 企業 IT 系統上雲、備份與儲存、大數據分析
  • 網站託管、行動 / 社群應用後端、遊戲伺服器全面雲端化
  • 代表客戶：Netflix、McDonald’s、21st Century Fox、Activision

AWS 全球基礎架構詳解

AWS 為全球化服務，核心組成包括：

  • Regions（區域）：地理位置分散的實體區域
  • Availability Zones（可用區）：每個區域內的獨立資料中心群組
  • Data Centers（資料中心）：實體基礎設施
  • Edge Locations / Points of Presence（邊緣節點）：全球分布的邊緣位置

各 Region 透過 AWS 私有網路互相連接，可利用 AWS 基礎設施快速打造全球化應用。

如何選擇 Region

  • 法規與合規（Compliance）：資料需留在特定國家（如法國）
  • 延遲（Latency）：服務部署應接近主要使用者
  • 服務可用性：並非每個 Region 都有所有 AWS 服務
  • 價格（Pricing）：不同 Region 價格不同

可用區（AZ）詳解

  • AZ 是 AWS Region 中一個或多個獨立的資料中心群組，具備冗餘電力、網路與連接
  • 每個 Region 通常有 3 個 AZ（最少 3、最多 6）
    • 例如：Sydney（ap-southeast-2）有 ap-southeast-2a、ap-southeast-2b、ap-southeast-2c
  • AZ 之間實體隔離，設計為災害不會連鎖影響，透過高速、低延遲網路互連
  • 容錯部署建議：在 3 個 AZ 中各放 2 台（共 6 台），確保單一 AZ 故障時仍正常運作

Edge Locations / Points of Presence

  • AWS 在全球擁有 400+ 個節點，分布於 90 座城市、40 個國家
  • 主要用途：內容傳遞、降低使用者延遲
  • 常見於 CDN 與邊緣服務（如 CloudFront）

AWS 服務的作用範圍

類型服務範例
全球性服務（Global Services）IAM、Route 53、CloudFront、WAF
區域性服務（Region Scoped）EC2、Elastic Beanstalk、Lambda、Rekognition

AWS Organizations

  • 全域服務，用於集中管理多個 AWS 帳號與成本
  • 支援跨帳號使用量合併計費（Consolidated Billing）
  • 可依部門、環境分群 OU（Organization Unit）
  • 有一個主帳號（Master），其餘為成員帳號（Member）
  • 可針對 OU 或帳號套用 SCP（Service Control Policy）限制資源操作範圍
  • SCP 的 Deny 會蓋過 Allow，主帳號不受 SCP 限制
  • 可使用 RAM（Resource Access Manager）跨帳號共用資源（如 Transit Gateway、子網、Route 53 規則）
  • 合併組織：將成員帳號退出舊組織並刪除，再加入新組織
  • AWS Control Tower 整合 Organizations 提供 Landing Zone，快速建立帳號架構與最佳實踐設定

多環境帳號分離策略

環境用途特性
Development開發測試寬鬆權限、低成本資源
Staging預發布環境模擬生產環境
Production正式環境嚴格權限、高可用性

最佳實踐：

  1. 使用 Organizations 管理多帳號：分離環境、部門或專案
  2. 實施 SCP 限制：防止意外操作影響生產環境
  3. 跨 AZ 部署：確保高可用性
  4. 選擇適當 Region：考量延遲、合規、成本等因素
