無狀態設計（Stateless Design）是雲端原生架構的核心原則之一。在無狀態架構中，伺服器不保存客戶端的會話資訊，每個請求都包含處理所需的完整資訊。這使得應用程式能夠輕鬆地水平擴展、自動恢復，並大幅簡化維運。

核心概念

狀態的定義

狀態類型說明範例處理方式
會話狀態使用者登入資訊Session ID、Token外部化到 Redis
應用狀態運算中間結果購物車、表單進度儲存到外部系統
持久狀態業務資料訂單、使用者資料資料庫
快取狀態效能加速資料熱門商品、設定分散式快取
暫時狀態請求處理中的資料變數、暫存保留在記憶體（無需外部化）

無狀態 vs 有狀態

面向無狀態有狀態
擴展性水平擴展容易需 Sticky Session 或同步
故障復原任何實例可接管需狀態遷移
負載平衡Round Robin 即可需 Session Affinity
部署滾動更新無感需考慮狀態遷移
維運複雜度低高
效能可能需額外網路跳轉本地存取快

12-Factor App 相關原則

因素原則與無狀態的關係
VI. Processes無狀態、無共享直接相關
IV. Backing Services外部化附加資源狀態外部化
VIII. Concurrency透過 Process 擴展水平擴展前提
IX. Disposability快速啟停無狀態才能做到

模式說明適用場景

方案延遲持久性擴展性成本適用

無狀態化改造難度

系統類型改造難度關鍵挑戰策略

Web 應用中Session 外部化Redis + JWT

應用場景

微服務架構

  • 每個服務獨立無狀態
  • 服務間透過 API 或事件通訊
  • 共享狀態透過資料庫或快取
  • 可獨立擴展各服務

Kubernetes 部署

  • Pod 隨時可被終止或重新排程
  • 水平 Pod 自動擴展器（HPA）
  • 滾動更新零停機
  • 所有持久狀態使用 PVC 或外部服務

Serverless 函數

  • 天然無狀態（每次呼叫獨立）
  • 狀態存入 DynamoDB/Firestore
  • 跨呼叫的快取利用 /tmp（不可靠）
  • 外部服務作為狀態協調者
