架構原則

雲端架構原則是指導工程師在雲端環境中做出正確設計決策的一套方法論。這些原則跨越特定的雲端供應商，提供了評估、設計與優化雲端系統的通用框架。掌握這些原則能讓你在面對複雜的技術選型時，有系統地權衡取捨並做出合理的架構決策。

核心概念

架構即取捨（Architecture is About Trade-offs）：不存在完美的架構，每個設計決策都涉及取捨。高可用性可能增加成本；安全性強化可能降低開發速度；效能優化可能增加複雜度。架構原則幫助我們在這些取捨中找到平衡點。

設計原則的層次性：

  • 戰略層：業務目標如何映射到技術架構（例如：全球擴張需要多區域部署）。
  • 戰術層：特定問題域的解決模式（例如：如何設計容錯的微服務通訊）。
  • 實作層：具體技術的最佳實踐（例如：S3 Bucket 的存取控制配置）。

持續演進的思維：雲端架構不是一次設計就定型的產物。業務需求變化、技術演進、新服務推出都可能促使架構調整。定期的架構審查（Architecture Review）是維持系統健康的重要實踐。

量化驅動決策：好的架構原則可被量化衡量。例如 RTO（恢復時間目標）與 RPO（恢復點目標）量化了可靠性需求；P99 延遲量化了效能需求；每月預算上限量化了成本約束。

架構總覽

雲端架構原則可從三個主要框架來理解：

Well-Architected Framework

三大供應商都提供了自己的 Well-Architected 框架，核心支柱高度相似：

  1. 營運卓越（Operational Excellence）：以自動化營運、可觀測性與持續改進為目標。
  2. 安全性（Security）：在所有層面實施防禦，保護資料與系統。
  3. 可靠性（Reliability）：系統能從故障中恢復並滿足需求。
  4. 效能效率（Performance Efficiency）：高效利用運算資源滿足需求。
  5. 成本優化（Cost Optimization）：以最低成本交付業務價值。
  6. 永續性（Sustainability）：最小化環境影響（較新的支柱）。

共同責任模型

這是雲端安全的基石概念：

  • 供應商責任：實體安全、硬體維護、網路基礎設施、虛擬化層。
  • 客戶責任：因服務類型而異。IaaS 中客戶負責 OS 以上全部；PaaS 中客戶負責應用與資料；SaaS 中客戶負責資料與存取管理。

理解共同責任模型是避免安全盲區的關鍵——許多雲端安全事件源於客戶誤以為供應商會負責某一層的安全。

區域與可用區

全球基礎設施的地理組織方式：

  • 區域（Region）：地理上分離的部署位置，通常由 2-6 個可用區組成。
  • 可用區（Availability Zone）：區域內物理隔離的資料中心群，獨立電力、冷卻與網路。
  • 邊緣站點（Edge Location）：用於 CDN 與低延遲服務的分佈節點。

應用場景

  • 新系統架構設計：在設計初期使用 Well-Architected 支柱作為檢查清單，確保不遺漏關鍵面向。例如：是否考慮了故障恢復？是否有成本預算機制？
  • 既有系統健康檢查：利用 Well-Architected Review 工具（AWS 提供免費的 Well-Architected Tool）對生產系統進行全面評估，識別改善機會。
  • 安全合規審查：對照共同責任模型，確認每一層的安全控制措施是否到位。特別關注 IaaS 場景中常被忽略的 OS 修補與網路配置。
  • 全球部署規劃：根據使用者地理分佈、資料法規（如 GDPR 的資料居留要求）與延遲需求，選擇區域與設計跨區域架構。
  • 災難復原規劃：根據 RTO/RPO 要求，選擇合適的多可用區或多區域部署策略。
