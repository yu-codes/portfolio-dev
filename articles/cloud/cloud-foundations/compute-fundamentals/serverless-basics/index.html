Serverless（無伺服器）運算是雲端運算的一種執行模型，開發者無需管理伺服器基礎設施，由雲端供應商自動處理資源分配、擴展與運維。「無伺服器」並非沒有伺服器，而是將伺服器管理的責任完全轉移給雲端平台。

核心概念

Serverless 的兩大類型

類型全稱說明範例
FaaSFunction as a Service事件驅動函數執行Lambda、Cloud Functions
BaaSBackend as a Service全託管後端服務Firebase、Auth0、Stripe

FaaS 執行模型

特性說明
事件驅動由事件觸發執行（HTTP、佇列、排程）
短暫執行有最大執行時間限制
無狀態每次呼叫獨立，不保留狀態
自動擴展從零擴展到數千並行
按量計費僅在執行時收費

冷啟動（Cold Start）

冷啟動是 Serverless 最關鍵的效能議題：

階段說明耗時
容器建立分配運行環境100-500ms
Runtime 載入啟動語言執行環境50-200ms
程式碼載入下載並載入部署包50-500ms
初始化執行全域初始化程式碼視程式碼而定
總冷啟動—200ms-數秒

Serverless 架構模式

模式一：API 後端

Client → API Gateway → Lambda → DynamoDB

模式二：事件處理管線

S3 Upload → Lambda → 處理 → 結果存入 S3/DB

模式三：排程任務

CloudWatch Events → Lambda → 資料庫清理/報表產生

模式四：串流處理

Kinesis → Lambda → DynamoDB/S3

何時不該用 Serverless

場景原因替代方案
長時間運行超出執行時間限制容器、VM
高頻固定流量成本較容器高ECS/GKE
需要低延遲冷啟動影響常駐容器
大型單體應用不適合函數拆分App Service
GPU 運算FaaS 不支援 GPUVM、專用實例

應用場景

Web API 後端

  • API Gateway + Lambda/Cloud Functions
  • 按請求付費，閒置零成本
  • 自動擴展到數千並發
  • 搭配 DynamoDB/Firestore 全無伺服器

圖片/影片處理

  • S3 上傳觸發 Lambda
  • 產生縮圖、轉檔
  • 結果存入目標 Bucket
  • 突發負載自動擴展

資料管線 ETL

  • 定時觸發資料擷取
  • Step Functions 編排多步驟流程
  • 錯誤自動重試與 Dead Letter Queue
  • 按實際處理量付費
