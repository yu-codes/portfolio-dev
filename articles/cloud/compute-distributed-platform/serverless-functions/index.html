Serverless 函數（Function as a Service，FaaS）是雲端運算抽象的最高層級，開發者只需關注業務邏輯程式碼，所有基礎設施管理由平台負責。本文聚焦於 Serverless 函數在分散式運算平台中的進階應用與設計模式。

應用場景

API 微服務

  • 每個端點一個 Lambda 函數
  • API Gateway 路由與認證
  • DynamoDB 作為資料存儲
  • 冷啟動最佳化確保低延遲

即時資料處理

  • Kinesis/Pub/Sub 觸發函數
  • 每批次處理一組事件
  • 結果寫入資料庫或佇列
  • 自動擴展處理流量高峰

定時任務與自動化

  • CloudWatch Events 定時觸發
  • 資料庫備份、報表產生
  • 過期資源清理
  • 合規性檢查與通知

AWS 服務實作

AWS Lambda 概述

  • FaaS（無伺服器函數服務）
  • 支援 Node.js、Python、Java、C#、Go、Ruby 等語言
  • 秒級擴展：不需像 EC2 等待幾分鐘
  • 按量計費：僅根據請求次數與執行時間計費
  • 無需管理伺服器：專注於程式碼
  • 高可用性：內建跨 AZ 容錯

Lambda 調用模式

同步調用：

  • 客戶端等待結果
  • 成功或失敗直接返回
  • 例如：API Gateway、CLI

非同步調用：

  • 不等待結果
  • Lambda 自動重試（0、1、2 次）
  • 可配置 Dead Letter Queue
  • 例如：S3 事件、SNS

事件來源映射（Event Source Mapping）：

  • 用於串流服務（Kinesis、DynamoDB Streams）
  • 用於佇列服務（SQS）
  • Lambda 輪詢來源

重試行為：

觸發類型重試行為
同步呼叫不重試
非同步呼叫重試 2 次
串流（Kinesis/DynamoDB）直到資料過期

事件批次處理：

設定說明
Batch Size每次調用的記錄數
Batch Window最長等待時間
部分批次失敗僅重新處理失敗的項目

錯誤處理

Dead Letter Queue（DLQ）：

  • 捕捉處理失敗請求
  • 支援 SQS 或 SNS 作為 DLQ

Lambda Destinations

  • 非同步調用的結果路由
  • 成功和失敗可發送到不同目的地
目的地類型說明
SNS Topic通知其他服務
SQS Queue佇列處理
Lambda 函數鏈式處理
EventBridge事件路由

Lambda 與 VPC

  • Lambda 可部署在 VPC 中存取私有資源
  • 需要 ENI（Elastic Network Interface）和足夠的子網路 IP 地址
  • 需要 NAT Gateway 存取網際網路（如需要）

VPC 整合架構：

Lambda → ENI → VPC 子網路 → 私有資源
                    ↓
              NAT Gateway → 網際網路

Lambda Layers

  • 可重用的程式碼和相依性
  • 最多 5 個 Layer
  • 減少部署套件大小

版本與別名

版本（Version）：

  • 不可變的函數快照，包含程式碼和配置
  • $LATEST 指向最新未發布版本
  • 發布後無法修改

別名（Alias）：

  • 指向特定版本的指標
  • 支援加權流量分配（金絲雀部署）
  • 別名是可變的

部署策略：

策略說明
全量部署直接切換到新版本
金絲雀小流量測試新版本
線性逐步增加新版本流量

ARM64 (Graviton2) 支援：

  • 支援 arm64 架構，提供更好的性價比
  • 比 x86_64 便宜約 20%
  • 大多數執行時期都支援

容器映像支援：

  • 支援最大 10GB 的容器映像
  • 可使用自訂執行時期
  • 需要從 ECR 提取映像

執行環境生命週期

類型說明時間
冷啟動建立新執行環境100ms - 數秒
暖啟動重用現有環境毫秒級

生命週期階段：

INIT（冷啟動）→ INVOKE → SHUTDOWN

最佳化策略：

  1. 程式碼放在 Handler 外部：利用環境重用
  2. 資料庫連線重用：避免每次建立連線
  3. 使用 Provisioned Concurrency：預熱執行環境
  4. 減少部署包大小：加快初始化

Lambda 效能調校

記憶體與 CPU：

  • 記憶體範圍：128MB - 10,240MB
  • CPU 與記憶體成比例分配
  • 1,769MB = 1 個完整 vCPU
場景建議
CPU 密集型增加記憶體
I/O 密集型增加記憶體可能無幫助
高並發使用 Provisioned Concurrency
長時間執行考慮 Step Functions

計費模式

  • 請求：每次調用計費
  • 執行時間：以 1ms 為單位
  • 記憶體分配影響價格
  • 免費層：每月 100 萬次請求 + 40 萬 GB-秒

CloudWatch 監控整合

執行紀錄自動寫入 CloudWatch Logs，需要 IAM Role 具備：

  • logs:CreateLogGroup
  • logs:CreateLogStream
  • logs:PutLogEvents

CloudWatch 指標：

指標說明
Invocations呼叫次數
Duration執行時間
Errors錯誤次數
Throttles節流次數
ConcurrentExecutions並發執行數
IteratorAge串流處理延遲

Lambda 與其他 AWS 服務整合

API Gateway + Lambda（完全無伺服器架構）：

客戶端 → API Gateway → Lambda → DynamoDB

S3 + Lambda（事件處理）：

上傳檔案 → S3 事件 → Lambda → 處理（壓縮、縮圖等）

Step Functions + Lambda：

  • 編排多個 Lambda 函數
  • 狀態機工作流程
  • 錯誤處理和重試

IAM 安全設定

  • 執行角色（Execution Role）：Lambda 執行時假設，提供存取 AWS 服務的臨時憑證
  • 資源政策：控制誰可以呼叫 Lambda，類似 S3 Bucket 政策，可允許其他帳號或服務調用

Lambda 最佳實踐

  1. 設定適當記憶體：記憶體也影響 CPU 分配
  2. 使用環境變數：配置敏感資訊
  3. 實施 DLQ：處理失敗請求
  4. 使用 Layers：重用相依性
  5. 監控與告警：設定 CloudWatch 警示
  6. 優化冷啟動：使用 Provisioned Concurrency
  7. 考慮使用 ARM64：提高性價比
  8. 使用 Lambda SnapStart：Java 函數冷啟動優化
