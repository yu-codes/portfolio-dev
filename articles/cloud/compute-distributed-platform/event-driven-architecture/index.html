事件驅動架構

事件驅動架構（Event-Driven Architecture，EDA）是一種軟體設計模式，系統中的元件透過產生、偵測和消費事件來進行通訊。這種架構天然適合雲端分散式系統，提供鬆耦合、高擴展性與即時反應能力。

核心概念

事件的定義

事件特性說明範例
不可變事件一旦發生不可修改OrderPlaced 事件
有時間順序帶有時間戳記2025-01-01T10:30:00Z
自描述包含足夠上下文包含訂單完整資訊
過去式命名描述已發生的事實UserRegistered
單向生產者不期待回應Fire-and-forget

事件類型分類

類型說明用途範例
Domain Event業務領域事件業務流程OrderShipped
Integration Event跨服務事件系統整合InventoryUpdated
Notification Event通知型事件（薄）觸發查詢OrderChanged（僅 ID）
Event-Carried State Transfer攜帶完整狀態減少回查OrderWithDetails
Command Event命令型（請求動作）嚴格來說非事件ProcessPayment

EDA vs 傳統架構

面向請求-回應事件驅動
耦合度強耦合鬆耦合
通訊方式同步非同步

失敗影響級聯故障隔離

應用場景

電商訂單流程

  1. OrderService 發布 OrderPlaced 事件
  2. PaymentService 消費 → 處理支付 → 發布 PaymentCompleted
  3. InventoryService 消費 → 扣減庫存 → 發布 InventoryReserved
  4. ShippingService 消費 → 建立出貨 → 發布 ShipmentCreated
  5. NotificationService 消費 → 通知客戶

即時資料管線

  • 使用者行為事件 → EventBridge/Pub/Sub
  • 即時分析：串流處理（Kinesis/Dataflow）
  • 離線分析：事件存入 S3/GCS → BigQuery
  • 即時儀表板：WebSocket 推送

CQRS + Event Sourcing

  • 寫入側：接受命令 → 產生事件 → 存入 Event Store
  • 讀取側：消費事件 → 更新讀取模型（多種投影）
  • 獨立擴展讀寫側
  • 完整稽核軌跡

AWS 服務實作

概述

  • AWS 的事件匯流排服務
  • 前身為 CloudWatch Events
  • 用於建構事件驅動架構

核心概念

Event Bus（事件匯流排）
類型說明
DefaultAWS 服務事件的預設匯流排
Custom自訂應用程式事件
Partner第三方 SaaS 整合
事件來源
  • AWS 服務（EC2、S3、Lambda 等）
  • 自訂應用程式
  • SaaS 應用程式（Zendesk、Datadog 等）
規則（Rules）
  • 定義事件匹配模式
  • 指定目標處理

EventBridge 進階功能

EventBridge Pipes
  • 建立事件來源與目標之間的點對點整合
  • 支援可選的過濾和資料轉換
  • 無需編寫粘合程式碼
API Destinations
  • 將事件傳送到任何 HTTP API 端點
  • 支援 OAuth 和 API Key 認證
  • 內建重試和速率限制
EventBridge Archive
  • 長期儲存事件以供重播
  • 支援事件重播用於測試和復原
  • 可設定保留期限

概述
  • 無伺服器工作流程編排服務
  • 使用 Amazon States Language（JSON）定義工作流程
  • 自動處理重試和錯誤
工作流程類型
類型說明適用場景

Express Workflows高吞吐量、短期執行串流處理、IoT
功能

  • Distributed Map：大規模並行資料處理
  • Workflow Studio：視覺化工作流程設計
  • 與 CloudWatch 和 X-Ray 整合監控

最佳實踐

  1. 使用 EventBridge：新專案優先選擇
  2. 善用 Schema Registry：管理事件結構
  3. 實施事件過濾：減少不必要的處理
  4. 設定 DLQ：處理失敗事件
  5. 監控事件流：追蹤事件處理狀態
  6. 使用 EventBridge Pipes：簡化點對點整合
  7. 考慮 Step Functions：複雜工作流程編排

AWS 整合模式

概述

應用程式整合是現代架構的關鍵，AWS 提供多種服務來實現不同的整合模式。

同步 vs 非同步

同步整合
特性說明
通訊方式請求-回應
耦合程度緊耦合
等待呼叫者等待回應
適用場景即時查詢、API 呼叫
非同步整合
特性說明
通訊方式訊息傳遞
耦合程度鬆耦合
等待呼叫者不等待
適用場景任務處理、事件驅動

常見整合模式

1. Point-to-Point（點對點）
Producer → SQS Queue → Consumer
  • 使用 SQS
  • 一對一通訊
  • 訊息只被一個消費者處理
2. Publish-Subscribe（發布訂閱）
Publisher → SNS Topic → Subscriber 1
                      → Subscriber 2
                      → Subscriber 3
  • 使用 SNS
  • 一對多通訊
  • 所有訂閱者收到相同訊息
3. Fan-out（扇出）
Producer → SNS → SQS 1 → Consumer 1
              → SQS 2 → Consumer 2
              → SQS 3 → Consumer 3
  • SNS + SQS 組合
  • 可靠的訊息分發
  • 每個消費者獨立處理
4. Event-Driven（事件驅動）
Event Source → EventBridge → Lambda
                           → Step Functions
                           → SQS
  • 使用 EventBridge
  • 基於事件的鬆耦合架構
  • 支援複雜的事件路由

選擇適當的服務

需求推薦服務
簡單佇列SQS Standard
順序處理SQS FIFO
廣播通知SNS
事件路由EventBridge
複雜流程Step Functions
MQTT 協定Amazon MQ

解耦最佳實踐

使用佇列解耦
Web Tier → SQS → Processing Tier → SQS → Database Tier
  • 各層獨立擴展
  • 容錯能力提升
  • 尖峰流量緩衝
使用事件解耦
Order Service → SNS → Inventory Service
                    → Billing Service
                    → Notification Service
  • 服務獨立演進
  • 新增服務不影響現有服務

錯誤處理模式

Dead Letter Queue（DLQ）
  • 收集處理失敗的訊息
  • 支援 SQS、SNS、Lambda
  • 用於錯誤分析和重新處理
重試策略
策略說明
指數退避每次重試間隔增加
固定間隔固定時間重試
有限重試最大重試次數後放棄

流量控制

限流（Throttling）
  • API Gateway 限流
  • Lambda 並發限制
  • SQS 可見性超時
緩衝（Buffering）
  • SQS 作為緩衝層
  • Kinesis 處理串流資料

整合模式比較

模式優點缺點
同步 API簡單、即時耦合高、容錯低
非同步佇列解耦、容錯複雜度增加
事件驅動高度解耦除錯困難

最佳實踐

  1. 優先非同步：提高系統彈性
  2. 使用 DLQ：處理失敗訊息
  3. 實施冪等性：處理重複訊息
  4. 監控佇列深度：預防積壓
  5. 設計回壓機制：處理過載情況
  6. 記錄追蹤 ID：分散式追蹤
