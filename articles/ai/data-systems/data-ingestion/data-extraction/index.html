資料擷取與收集是整個資料系統的起點，決定了後續所有分析與模型訓練的品質上限。無論是建構機器學習管線還是即時分析系統，如何高效、可靠地從多元資料源中提取資訊，是每位資料工程師必須面對的核心挑戰。

核心概念

資料擷取（Data Extraction）指的是從各種來源系統中讀取原始資料的過程。這些來源可能包括關聯式資料庫、NoSQL 資料庫、API 端點、檔案系統、訊息佇列、IoT 裝置等。

資料收集（Data Collection）則是更廣義的概念，涵蓋了資料擷取、傳輸、暫存與初步驗證的完整流程。

擷取模式分類

擷取模式說明適用場景延遲
全量擷取每次擷取完整資料集小型資料集、初始載入高
增量擷取僅擷取變更的部分大型資料集、定期同步中
CDC（變更資料擷取）追蹤資料庫日誌變更即時同步需求低
事件驅動擷取由事件觸發擷取動作微服務架構極低

關鍵設計原則

  • 冪等性（Idempotency）：相同的擷取操作重複執行應產生相同結果
  • 容錯性（Fault Tolerance）：系統必須能處理網路中斷、來源不可用等異常
  • 可觀測性（Observability）：提供完整的監控指標與日誌追蹤
  • 背壓處理（Backpressure）：當下游處理速度跟不上時，應有適當的流量控制機制

架構總覽

典型的資料擷取架構可分為三層：

連接層（Connector Layer）

負責與各種資料源建立連接。每種資料源需要專屬的連接器，處理認證、協議轉換與資料格式解析。常見的連接器框架包括 Debezium（用於 CDC）、Airbyte 與 Fivetran。

傳輸層（Transport Layer）

負責將擷取的資料可靠地傳輸到目的地。這一層通常使用訊息佇列或串流平台作為緩衝區，例如 Apache Kafka、Amazon Kinesis 或 Google Pub/Sub。

落地層（Landing Layer）

資料的第一個持久化存儲點，通常是資料湖或暫存區域。這裡的資料保持原始格式，不進行轉換，確保資料的完整性與可追溯性。

關鍵比較

批次處理 vs 串流處理

比較維度批次處理串流處理
處理延遲分鐘至小時毫秒至秒
資料完整性容易保證需要額外機制
系統複雜度較低較高
資源使用峰值高，平時低持續穩定
錯誤處理重新執行整批需要精確的偏移管理
適用場景報表、模型訓練即時監控、推薦系統
代表技術Apache Spark、AirflowApache Kafka、Flink
成本模式按執行時間計費持續運行成本

Push vs Pull 模式

Pull 模式：擷取系統主動向來源查詢資料。優點是控制權在擷取端，可以管理擷取頻率；缺點是可能造成來源系統負擔。

Push 模式：來源系統主動推送資料到擷取端。優點是延遲低、來源系統可控制發送節奏；缺點是需要來源系統配合改造。

CDC 實作方式比較

方式原理優點缺點
時間戳式根據更新時間欄位判斷實作簡單無法偵測刪除操作
觸發器式資料庫觸發器記錄變更即時性高影響資料庫效能
日誌式解析資料庫交易日誌不影響來源效能實作複雜度高
快照比對式前後快照差異比對通用性強資源消耗大

應用場景

場景一：電商即時推薦系統

電商平台需要即時收集使用者的瀏覽行為、點擊事件與購買記錄。透過事件驅動架構，將使用者行為串流到 Kafka，再由下游的推薦引擎即時更新使用者特徵向量，實現毫秒級的個人化推薦。

場景二：金融交易監控

金融機構需要從多個交易系統中擷取交易資料，進行即時的反詐欺分析。這要求極低的延遲（通常 < 100ms）與極高的可靠性（零資料遺失）。典型架構採用 CDC 技術擷取核心系統變更，配合串流處理引擎進行即時風險評估。

場景三：機器學習訓練資料管線

ML 團隊需要定期從多個業務系統收集訓練資料。這類場景通常採用批次擷取模式，配合增量擷取策略，將資料匯入資料湖後再進行特徵工程。關鍵挑戰在於確保資料的一致性與完整性。

場景四：IoT 感測器資料收集

工業物聯網場景中，數以萬計的感測器持續產生時序資料。擷取系統必須處理高併發寫入、網路不穩定導致的資料重傳，以及邊緣裝置的資源限制。通常採用 MQTT 協議搭配邊緣計算進行前處理。
