資料收集與擷取方法

資料擷取（Data Ingestion）是整個資料系統的起點，負責從各種異質資料來源收集、傳輸並載入資料到目標儲存系統中。無論是建構機器學習管線、即時分析平台，還是資料倉儲，資料擷取的品質與效率直接決定了下游應用的可靠性。本文將深入探討資料擷取的核心概念、架構模式與最佳實踐。

核心概念

資料擷取涵蓋三個基本維度：來源多樣性、傳輸模式與載入策略。

來源多樣性

現代企業的資料來源極為多元，包括：

  • 結構化資料：關聯式資料庫（PostgreSQL、MySQL）、ERP 系統、CRM 平台
  • 半結構化資料：JSON API 回應、XML 文件、日誌檔案（Log Files）
  • 非結構化資料：圖片、影片、PDF 文件、社群媒體貼文
  • 串流資料：IoT 感測器、點擊流（Clickstream）、交易事件

每種來源都有其獨特的存取協定、資料格式與更新頻率，擷取系統必須能夠統一處理這些差異。

傳輸模式

資料擷取主要分為兩大模式：

批次擷取（Batch Ingestion）：以固定時間間隔（每小時、每天）批量拉取資料。適合對即時性要求不高的場景，如每日報表、歷史資料回填。優點是實作簡單、容錯性高；缺點是資料延遲較大。

串流擷取（Streaming Ingestion）：資料產生後立即傳輸，延遲通常在毫秒到秒級。適合即時推薦、詐欺偵測、即時監控等場景。常用技術包括 Apache Kafka、Amazon Kinesis、Google Pub/Sub。

載入策略

  • 全量載入（Full Load）：每次擷取完整資料集，簡單但資源消耗大
  • 增量載入（Incremental Load）：僅擷取變更部分，透過時間戳記或 CDC（Change Data Capture）追蹤變更
  • CDC 模式：監聽資料庫的事務日誌（WAL/Binlog），精確捕獲每一筆變更

架構總覽

典型的資料擷取架構包含以下層次：

資料來源 → 擷取層 → 緩衝層 → 儲存層 → 消費層

擷取層負責連接器管理、協定轉換與初步驗證。常見工具如 Apache NiFi、Airbyte、Fivetran 提供了豐富的預建連接器。

緩衝層通常採用訊息佇列（如 Kafka）作為中間緩衝，解耦上下游系統，提供背壓（Backpressure）控制與重試機制。

儲存層根據使用場景選擇目標系統：資料湖（Data Lake）適合原始資料保存，資料倉儲（Data Warehouse）適合分析查詢，特徵儲存（Feature Store）適合機器學習場景。

應用場景

機器學習資料管線

在 MLOps 場景中，資料擷取需要處理訓練資料的定期更新與特徵資料的即時同步。典型做法是：批次擷取歷史資料用於模型訓練，串流擷取即時資料用於線上推論。

即時數據平台

電商平台需要即時追蹤使用者行為、庫存變動與訂單狀態。透過 Kafka 串流擷取 + Flink 即時處理，可在秒級完成從事件產生到洞察產出的完整鏈路。

資料湖建構

企業資料湖需要從數十甚至數百個來源系統擷取資料。此時擷取層的可擴展性、連接器生態與 schema 演進（Schema Evolution）支援尤為重要。

合規與審計

金融與醫療領域要求完整的資料血統（Data Lineage）追蹤。擷取系統需記錄每筆資料的來源、時間戳記與轉換歷程，確保可審計性。
