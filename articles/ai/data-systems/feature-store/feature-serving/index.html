特徵服務系統是 Feature Store 面向即時推論的核心元件，負責在毫秒級時間內將正確的特徵值交付給 ML 模型。其設計必須同時滿足低延遲、高吞吐量與高可用性的嚴格要求，是連接離線特徵工程與線上模型推論的關鍵橋樑。

核心概念

特徵服務的職責

特徵服務系統接收實體標識符（如用戶 ID、商品 ID），查詢對應的特徵值，並以結構化格式返回給模型推論服務。

服務模式

模式說明延遲適用場景
預計算服務特徵預先計算並存儲極低（<5ms）大部分特徵
即時計算請求時動態計算中（10-100ms）依賴請求上下文的特徵
混合模式預計算 + 即時補充低（5-20ms）結合靜態與動態特徵
批次預取預測性預載特徵極低可預期的存取模式

一致性語意

  • 最終一致性（Eventually Consistent）：特徵更新有延遲但最終會一致，適合大部分推薦場景
  • 讀己之寫（Read-Your-Writes）：使用者的操作立即反映在特徵中，適合互動式場景
  • 強一致性（Strong Consistency）：所有讀取都看到最新值，適合金融場景但犧牲延遲

架構總覽

特徵服務系統分層

API 閘道層：接收特徵查詢請求，負責認證、限流與路由。

查詢路由層：根據特徵定義決定從哪個存儲後端讀取。不同特徵可能存儲在不同後端。

快取層：多級快取策略——本地快取（進程內）→ 分散式快取（Redis）→ 後端存儲。

存儲後端層：實際存放特徵值的系統。根據存取模式選擇不同引擎。

特徵計算層：處理即時特徵的計算邏輯，通常整合串流處理引擎。

後端延遲吞吐量適用特徵

關鍵比較

特徵服務架構模式

模式架構優點缺點
中心化服務獨立服務叢集統一管理、易於監控額外網路跳轉
嵌入式 SDK特徵邏輯嵌入推論服務最低延遲更新需重新部署

邊緣快取CDN/邊緣節點快取特徵地理分佈式低延遲一致性挑戰

效能最佳化策略

策略原理效果
批次查詢（Batching）合併多個實體的查詢減少 RTT，提升吞吐量
預取（Prefetching）預測性地提前載入特徵降低有效延遲
壓縮儲存特徵值的壓縮編碼減少網路傳輸量
連接池重用後端連接避免連接建立開銷
局部性優化將常一起查詢的特徵共置減少跨分區查詢
非同步填充快取未命中時非同步回填避免阻塞請求

應用場景

場景一：搜尋排序特徵服務

搜尋引擎的排序模型需要在 50ms 內對上百個候選結果評分。特徵服務需要在 10ms 內返回每個候選的特徵向量，支援批次查詢以減少延遲，並使用本地快取存儲高頻查詢的特徵。

場景二：即時風控決策

支付系統的風控模型需要在 100ms 內完成交易風險評估。特徵服務同時查詢用戶歷史特徵（預計算）與交易上下文特徵（即時計算），使用混合模式在延遲預算內完成特徵組裝。

場景三：廣告競價特徵

程序化廣告系統的 RTB（Real-Time Bidding）要求在 100ms 內完成競價決策。特徵服務需要處理每秒數十萬的查詢請求，使用多層快取與邊緣部署來保證延遲 SLA。

場景四：邊緣設備推論

自動駕駛或工業 IoT 場景中，推論在邊緣設備上執行。特徵服務採用嵌入式 RocksDB 存儲本地特徵，定期從雲端同步更新，確保離線時仍可正常推論。
