資料版本管理（Data Versioning）是實現機器學習可重現性（Reproducibility）的基石。當模型出現問題時，我們需要精確回溯到訓練時使用的資料版本；當資料管線演進時，我們需要追蹤每一次變更的影響。本文將深入探討資料版控的核心理念、工具生態與工程實踐。

核心概念

為什麼需要資料版控？

在軟體工程中，程式碼版控（Git）早已是標準實踐。然而資料領域的版控卻面臨獨特挑戰：

  • 體積龐大：訓練資料動輒數 GB 到 TB，無法像程式碼一樣存入 Git
  • 變更頻繁：資料可能每日甚至每小時更新
  • 格式多樣：CSV、Parquet、影像、音訊等各種格式
  • 關聯複雜：資料版本需要與程式碼版本、模型版本對齊

資料版控解決的核心問題是：在任意時間點，精確還原出當時使用的完整資料集。

版控策略

快照式（Snapshot-based）：每個版本保存完整的資料副本。簡單直觀但儲存成本高。適合小型資料集或變更比例大的場景。

差異式（Delta-based）：僅保存與上一版本的差異。儲存效率高但還原速度較慢。適合大型資料集的增量更新。

指標式（Pointer-based）：版本記錄僅包含資料的元資料與指標（如 hash 值、儲存路徑）。實際資料存於外部儲存（S3、GCS）。這是 DVC 等工具採用的主流策略。

語義版本化

借鑑軟體的語義版本規範（SemVer），資料版本也可以定義明確的語義：

  • Major 版本：Schema 破壞性變更（新增/移除欄位）
  • Minor 版本：新增資料但不影響既有結構
  • Patch 版本：修正資料錯誤、填補缺失值

架構總覽

資料版控系統的典型架構：

程式碼倉庫（Git） ←→ 版控元資料 ←→ 遠端儲存（S3/GCS/Azure Blob）
                         ↓
                    版本目錄（Catalog）
                         ↓
                    血統追蹤（Lineage）

元資料層：記錄每個資料檔案的 hash、大小、路徑與版本標籤。與 Git commit 關聯，確保程式碼與資料版本同步。

遠端儲存層：實際資料以內容定址（Content-addressable）方式存放。相同內容只存一份（去重），節省儲存空間。

版本目錄（Catalog）：提供人類可讀的版本瀏覽介面，支援標籤、搜尋與版本比較。

血統追蹤：記錄資料從來源到最終版本的完整轉換歷程，支援影響分析與根因追溯。

DVC vs LakeFS

DVC 適合以程式碼為中心的 ML 團隊。它將資料版本追蹤嵌入 Git 工作流，學習曲線低，適合中小規模資料集。

LakeFS 適合資料工程團隊管理大規模資料湖。它提供 Git-like 的分支、合併操作，支援原子性的多檔案提交，適合 PB 級資料的版控需求。

應用場景

ML 實驗可重現性

資料科學家需要精確記錄每次實驗使用的資料版本。透過 DVC + Git，可以建立「程式碼 commit + 資料版本 + 模型權重」的完整快照，實現任意實驗的精確重現。

資料回滾

當上游資料管線出錯污染了資料集，版控系統允許快速回滾到已知正確的版本，將故障影響降到最低。

A/B 測試資料管理

不同實驗組可能需要不同版本的特徵資料。版控系統支援分支機制，讓多個實驗並行操作不同資料版本而互不干擾。

合規與審計追蹤

法規要求能夠證明模型訓練使用了哪些資料。版控的不可變歷史（Immutable History）天然滿足審計需求。
