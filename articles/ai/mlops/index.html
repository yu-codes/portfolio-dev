MLOps

機器學習維運與自動化

MLOps（Machine Learning Operations）是將機器學習模型從實驗環境可靠地部署到生產環境，並持續維運的工程實踐。它結合了軟體工程的 DevOps 理念與機器學習特有的挑戰——資料版本管理、模型再訓練、效能監控與漂移偵測。成熟的 MLOps 實踐能將模型從「筆記本中的實驗」轉化為「穩定運行的業務系統」。

核心概念

MLOps 的動機

傳統軟體開發中，程式碼是唯一的變動源。但 ML 系統有三個變動軸：

  • 程式碼變動：模型架構、特徵邏輯、推論服務的程式碼更新
  • 資料變動：訓練資料的分佈隨時間漂移，新資料持續累積
  • 模型變動：重新訓練產生新版本模型，超參數調整影響行為

這三重變動使得 ML 系統的可重現性、測試與部署比傳統軟體複雜得多。

MLOps 成熟度等級

等級描述特徵
Level 0手動流程手動訓練、手動部署、無監控
Level 1ML 管線自動化自動訓練管線、持續訓練、特徵儲存
Level 2CI/CD 管線自動化自動測試、自動部署、自動監控與回滾

核心元件

實驗追蹤（Experiment Tracking）：記錄每次訓練的超參數、指標、程式碼版本與資料版本。工具如 MLflow、Weights & Biases、Neptune。

模型註冊（Model Registry）：集中管理模型版本，支援階段標記（Staging → Production → Archived）、審批流程與血緣追蹤。

特徵儲存（Feature Store）：統一管理特徵定義、計算邏輯與服務，確保訓練與推論的一致性。

模型服務（Model Serving）：將模型封裝為 API 或嵌入應用，支援即時推論、批次推論與串流推論。

持續訓練（CT）

持續訓練是 MLOps 相較於傳統 DevOps 最獨特的概念：

# 持續訓練觸發條件
triggers:
  - schedule: "每週一自動重訓"
  - data_drift: "輸入分佈偏移超過閾值"
  - performance_decay: "線上指標低於 SLA"
  - new_data_volume: "新資料累積超過 N 筆"
  - manual: "資料科學家手動觸發"

架構總覽

MLOps 的完整架構可分為以下子系統：

資料管理系統：資料版本控制（DVC）、資料品質驗證（Great Expectations）、資料標註管理。確保訓練資料的可追溯性與品質。

訓練自動化系統：編排訓練管線（Airflow、Kubeflow Pipelines）、分散式訓練基礎設施、超參數搜尋服務。支援一鍵觸發端到端訓練流程。

CI/CD 系統：程式碼測試（單元測試、整合測試）、模型品質驗證（指標閾值、公平性檢查）、自動部署（藍綠部署、金絲雀發布）。

監控系統：模型效能監控（準確率、延遲）、資料漂移偵測（PSI、KS 檢定）、基礎設施監控（CPU/GPU 使用率、記憶體）。異常時自動告警或觸發回滾。

治理系統：模型文件、審計日誌、合規報告、存取控制。滿足企業治理與法規要求。

批次推論 vs 即時推論

批次推論適合對延遲不敏感的場景（如每日推薦列表更新），架構簡單且成本低。即時推論滿足低延遲需求（如詐欺偵測），但需考慮服務可用性、自動擴展與快取策略。

應用場景

  1. 自動化訓練管線：資料到達 → 品質驗證 → 特徵計算 → 模型訓練 → 評估 → 部署的端到端自動化
  2. 模型 CI/CD：每次程式碼提交自動觸發訓練、評估與部署，包含品質閘門
  3. 效能監控與告警：即時追蹤模型線上表現，偵測漂移並自動觸發重訓或回滾
  4. 實驗管理：團隊協作的實驗追蹤，支援結果比較、最佳模型選擇與知識傳承
  5. 重訓練管線：基於排程或觸發條件的自動模型更新，確保模型保持新鮮度
  6. 合規與審計：記錄模型決策依據、資料血緣、版本歷史以滿足法規要求
