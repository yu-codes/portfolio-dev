ML CI/CD

機器學習持續整合與部署

機器學習 CI/CD（Continuous Integration / Continuous Deployment）是將傳統軟體工程的持續整合與持續部署理念，延伸至機器學習生命週期的實踐方法。不同於傳統 CI/CD 僅處理程式碼的建置與部署，ML CI/CD 還需涵蓋資料驗證、模型訓練、評估與上線等額外步驟，使整個流程更加複雜且具有挑戰性。

核心概念

ML CI/CD 的核心在於將機器學習工作流程標準化與自動化。其主要概念包含以下幾個面向：

持續整合（CI）層面涵蓋了程式碼品質檢查、單元測試、資料驗證以及模型訓練的可重現性測試。每當資料科學家提交程式碼變更時，系統會自動執行資料格式檢查、特徵工程邏輯測試、以及小規模的模型訓練驗證，確保變更不會破壞現有流程。

持續部署（CD）層面則關注模型的自動化部署流程。當模型通過所有品質門檻後，系統會自動將模型封裝為可服務的格式（如 Docker 容器或 ONNX 模型），並部署至預定環境。部署策略可能包含金絲雀發布（Canary Release）、藍綠部署（Blue-Green Deployment）或影子模式（Shadow Mode）等。

**持續訓練（CT）**是 ML CI/CD 特有的概念，指當資料分布發生變化或模型效能下降時，系統能自動觸發重新訓練流程，確保模型始終反映最新的資料模式。

版本控制在 ML CI/CD 中不僅限於程式碼，還包括資料版本（Data Versioning）、模型版本（Model Versioning）以及管線配置版本，形成完整的可追溯性鏈條。

架構總覽

典型的 ML CI/CD 架構由以下層級組成：

┌─────────────────────────────────────────────────┐
│              觸發層（Trigger Layer）                │
│   Git Push / Schedule / Data Drift Alert        │
└─────────────────┬───────────────────────────────┘
                  ▼
┌─────────────────────────────────────────────────┐
│            管線編排層（Orchestration）              │
│   Kubeflow / Airflow / GitHub Actions           │
└─────────────────┬───────────────────────────────┘
                  ▼
┌────────┬────────┬────────┬────────┬─────────────┐
│ 資料驗證 │ 特徵工程 │ 模型訓練 │ 模型評估 │  模型註冊   │
└────────┴────────┴────────┴────────┴─────────────┘
                  ▼
┌─────────────────────────────────────────────────┐
│            部署層（Deployment Layer）              │
│   Model Serving / A-B Testing / Monitoring      │
└─────────────────────────────────────────────────┘

在觸發層中，系統可由多種事件啟動：程式碼推送（Git Push）、排程觸發（Cron Schedule）、資料變更偵測或模型效能警報。管線編排層負責協調各步驟的執行順序與相依性。核心處理層包含資料驗證、特徵工程、模型訓練、模型評估與模型註冊等步驟。最終的部署層處理模型上線、流量分配與即時監控。

工具比較：

  • GitHub Actions + DVC：適合小型團隊，整合簡單，但大規模訓練任務管理能力有限
  • Kubeflow Pipelines：適合 Kubernetes 生態，可擴展性強，但學習曲線陡峭
  • MLflow + Jenkins：企業級方案，功能完整，但部署與維護成本較高
  • Vertex AI Pipelines：GCP 原生整合，託管服務減少維運負擔，但供應商鎖定風險較高

應用場景

場景一：推薦系統的持續更新——電商平台的推薦模型需要每日根據新的用戶行為資料進行增量訓練。ML CI/CD 管線在每日排程中自動擷取前一天的互動資料、驗證資料品質、執行增量訓練、評估新模型指標，並在通過品質門檻後以金絲雀方式逐步部署。

場景二：自然語言處理模型的版本迭代——當 NLP 團隊改進了文本前處理邏輯或更換了預訓練模型，CI 流程會自動在標準資料集上評估新模型的各項指標（F1、BLEU 等），與基準模型進行比較，並生成詳細的效能報告供審查。

場景三：即時欺詐偵測系統——金融機構的欺詐偵測模型需要快速回應新型態攻擊。當監控系統偵測到模型精確率下降時，自動觸發重訓練管線，在新標註資料上微調模型，並在影子模式下驗證後快速上線。

場景四：多模型協同部署——在複雜的 AI 系統中，多個模型之間存在相依關係。ML CI/CD 需要管理模型間的版本相容性，確保上游模型的輸出格式與下游模型的輸入介面一致。
