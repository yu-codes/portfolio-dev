推論服務架構（Model Serving Architecture）是將機器學習模型以高效能、高可用的方式暴露為線上服務的系統設計。一個優秀的推論服務架構需要在延遲、吞吐量、可靠性與成本之間取得平衡。

核心概念

推論模式

根據延遲需求與資料特性，推論服務可分為：

  • 即時推論（Real-time Inference）：毫秒級回應，適用於互動式應用
  • 近即時推論（Near Real-time）：秒級回應，適用於準即時處理
  • 批次推論（Batch Inference）：分鐘到小時級，適用於大量離線預測

服務模式

  • 同步服務：客戶端等待推論結果回傳（REST/gRPC）
  • 非同步服務：客戶端提交請求後輪詢結果（Message Queue）
  • 串流服務：持續輸出推論結果（WebSocket/SSE，如 LLM 生成）

模型伺服器（Model Server）

專為模型推論優化的服務框架，提供：

  • 模型載入與生命週期管理
  • 請求批次化（Dynamic Batching）
  • 多模型共存與路由
  • 硬體加速整合（GPU、TPU）
  • 健康檢查與指標匯出

推論圖（Inference Graph）

複雜應用通常涉及多個模型的串聯或並聯：

  • 前處理模型 → 主模型 → 後處理模型
  • 集成模型的平行推論與結果合併
  • 條件路由（根據輸入特徵選擇模型）

動態批次化（Dynamic Batching）

將短時間內到達的多個請求合併為一個批次進行推論：

  • 降低 GPU 呼叫次數，提升硬體利用率
  • 設定最大等待時間（max latency budget）
  • 設定最大批次大小（max batch size）
  • 在延遲與吞吐量之間動態平衡

模型排程

多模型環境下的資源排程策略：

  • 獨佔模式：每個模型佔用專屬 GPU
  • 共享模式：多模型共用 GPU，透過時間片輪轉
  • 動態載入：按需載入模型到 GPU 記憶體
  • 優先級排程：根據 SLA 優先服務高優先級請求

模型版本管理與流量控制

  • 金絲雀部署：新版本先接收少量流量
  • 藍綠部署：瞬間切換全部流量
  • 影子模式：新模型平行推論但不回傳結果
  • A/B 測試：按規則分配流量到不同版本

應用場景

大語言模型服務

LLM 推論具有高延遲、高記憶體需求的特點。需要特化的服務架構支援：KV-Cache 管理、連續批次處理（Continuous Batching）、張量平行（Tensor Parallelism），以及 Token 級串流輸出。

推薦系統即時推論

電商推薦需要在毫秒級內完成使用者特徵提取、候選召回、精排模型推論與結果融合。架構上通常採用多級串聯的推論圖，每一級都有嚴格的延遲預算。

電腦視覺即時處理

影像辨識、物體偵測等 CV 任務需要高吞吐量處理。透過 TensorRT 優化與 Triton 的動態批次化，實現在單張 GPU 上達到數千 QPS 的推論效能。

多模型 A/B 測試平台

同時服務多個模型版本進行線上實驗。推論服務架構需要支援細粒度的流量路由、各版本獨立的指標收集，以及安全的版本切換機制。
