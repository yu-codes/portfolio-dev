Prompt Engineering

Prompt Engineering（提示工程）是設計與優化輸入提示詞以引導大型語言模型產生高品質輸出的技術。在不修改模型權重的前提下，精心設計的 Prompt 能顯著提升 LLM 在各類任務上的表現，是使用 AI 系統最具成本效益的方式。

核心概念

Prompt 的組成要素

一個有效的 Prompt 通常包含以下元素：

要素說明範例
角色設定（Role）定義模型的身份與行為「你是一位資深的後端工程師」
任務指令（Instruction）明確說明要完成的任務「請分析以下程式碼的效能問題」
上下文（Context）提供背景資訊系統架構說明、限制條件
輸入資料（Input）待處理的具體內容程式碼片段、文件
輸出格式（Format）指定回應的結構「以 JSON 格式回傳」
示例（Examples）Few-shot 範例輸入-輸出對

基本設計原則

  1. 明確具體：避免模糊指令，提供清晰的任務定義
  2. 結構化：使用分隔符、標題、編號組織 Prompt
  3. 漸進引導：複雜任務拆解為步驟
  4. 限制範圍：明確告知模型不應做什麼
  5. 迭代優化：根據輸出持續調整

架構總覽

Zero-Shot Prompting

不提供任何示例，直接描述任務：

請將以下句子翻譯為英文：
「今天天氣真好」

適用於模型已有良好能力的簡單任務。

Few-Shot Prompting

提供少量輸入-輸出示例，讓模型學習模式：

將情感分類為正面或負面：
文本：「這部電影太精彩了！」→ 正面
文本：「服務態度很差」→ 負面
文本：「今天的晚餐非常美味」→

要點：

  • 通常 3-5 個示例即可
  • 示例應涵蓋不同情況
  • 保持格式一致

Chain-of-Thought（CoT）

引導模型展示推理過程，顯著提升複雜推理任務的準確率：

問題：Roger 有 5 顆網球，他又買了 2 罐網球，每罐有 3 顆。
請問他現在有幾顆網球？

讓我們一步步思考：
1. Roger 原本有 5 顆網球
2. 他買了 2 罐，每罐 3 顆，所以買了 2 × 3 = 6 顆
3. 總共 5 + 6 = 11 顆

答案：11 顆

Zero-Shot CoT：只需加上「Let’s think step by step」即可觸發推理。

Tree-of-Thought（ToT）

讓模型探索多條推理路徑，自我評估後選擇最佳方案：

探索三種不同的方法來解決這個問題：
方法 1：...
方法 2：...
方法 3：...

評估每種方法的優缺點，選擇最佳方案。

Self-Consistency

多次生成 CoT 推理，取多數投票作為最終答案：

  • 設定較高的 temperature 以產生多樣化推理路徑
  • 聚合多次回答中的最常見結論
  • 在數學和邏輯推理任務上特別有效

System Prompt 設計模式

模式用途適用情境
角色扮演定義專家身份專業領域問答
規則約束限制輸出行為安全合規場景
格式模板統一輸出結構API 回應、報表生成
多步工作流定義處理流程複雜分析任務

Prompt 優化技巧

技巧效果實作方式
分隔符使用避免注入攻擊使用 ```、—、XML tags
正面指令比否定句更有效「請回答問題」而非「不要閒聊」
輸出錨定引導格式在 Prompt 尾部開始輸出格式
溫度調控控制創意度事實類 temp=0，創意類 temp=0.7+
Token 限制控制長度「用 100 字以內回答」

應用場景

程式開發輔助

  • 程式碼生成：明確指定語言、框架、輸入輸出格式
  • 程式碼審查：提供 checklist 讓模型逐項檢查
  • Debug 協助：提供錯誤訊息和上下文

內容創作

  • 結構化寫作：提供大綱 → 逐段展開
  • 風格遷移：提供目標風格的範例
  • 多語言翻譯：加入術語表和風格指引

資料處理

  • 資訊擷取：定義 schema 和輸出格式
  • 分類與標記：Few-shot + 明確類別定義
  • 摘要生成：指定長度、重點和受眾

AI Agent 系統

  • ReAct 模式：結合推理與行動
  • 工具選擇：描述可用工具讓模型決定何時調用
  • 多步任務分解：將複雜目標拆為子任務

安全與對齊

  • 防止 Prompt Injection：使用分隔符與驗證
  • 內容過濾：定義拒絕回答的情境
  • 事實性要求：要求引用來源、標注不確定性
