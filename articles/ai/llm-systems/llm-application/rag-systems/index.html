RAG 檢索增強生成

RAG（Retrieval-Augmented Generation）是一種結合資訊檢索與文本生成的架構，透過在生成前檢索相關知識，讓 LLM 能夠基於最新、準確的外部資訊回答問題。RAG 有效解決了 LLM 的知識截止、幻覺與領域知識不足等問題，是企業 AI 應用最主流的架構之一。

核心概念

為什麼需要 RAG

LLM 面臨的限制：

  • 知識截止：訓練資料有時間限制，無法回答最新問題
  • 幻覺問題：缺乏知識時可能虛構答案
  • 領域不足：對企業內部知識缺乏理解
  • 不可驗證：無法追溯回答的資訊來源

RAG 的解決方式：在生成前檢索相關文件作為上下文，讓模型基於事實生成回答。

RAG 基本流程

使用者問題
    ↓
[檢索階段] Query → 向量化 → 相似度搜尋 → Top-K 文件
    ↓
[增強階段] 問題 + 檢索結果 → 組合 Prompt
    ↓
[生成階段] LLM 基於上下文生成答案
    ↓
回答（附引用來源）

架構總覽

Naive RAG（基礎版）

最簡單的 RAG 實作：

  1. 文件處理：載入 → 切分 → 嵌入 → 存入向量資料庫
  2. 查詢處理：問題嵌入 → 相似度搜尋 → 取 Top-K
  3. 生成回答：將檢索結果塞入 Prompt → LLM 回答

限制：切分品質差、檢索不精確、缺乏後處理。

Advanced RAG（進階版）

在 Naive RAG 基礎上加入多項優化：

Pre-Retrieval（檢索前）：

  • Query Rewriting：改寫或擴展查詢
  • HyDE：生成假設文件再檢索
  • Query Decomposition：將複雜問題拆解

Retrieval（檢索中）：

  • Hybrid Search：結合向量搜尋 + 關鍵字搜尋（BM25）
  • Re-ranking：使用 Cross-Encoder 重新排序
  • Metadata Filtering：依時間、來源等篩選

Post-Retrieval（檢索後）：

  • Context Compression：去除無關段落
  • Lost-in-the-Middle 處理：重新排列重要文件位置
  • Citation Generation：標記引用來源

Modular RAG（模組化）

將 RAG 拆為可組合的模組：

[Routing] → 判斷是否需要檢索
    ↓
[Retrieval] → 多源檢索（向量DB、圖DB、API）
    ↓
[Reranking] → 相關性重排序
    ↓
[Compression] → 上下文壓縮
    ↓
[Generation] → 帶引用的生成
    ↓
[Validation] → 幻覺檢測、事實驗證

RAG vs Fine-tuning vs Long Context

維度RAGFine-tuningLong Context
知識更新即時（更新索引）需重新訓練即時（放入上下文）
成本中（檢索基礎設施）高（訓練成本）高（Token 消耗）
準確性高（可引用來源）中（可能幻覺）高（全部上下文）
規模數十億文件有限受限於 Context Window
延遲中（檢索 + 生成）低（僅生成）高（長上下文處理）
可解釋性高（可追溯來源）低中

應用場景

企業知識庫問答

  • 內部文件（SOP、技術文檔、會議記錄）的智能檢索
  • 客服系統回答產品相關問題
  • 法規遵循：確保回答有依據且可追溯

程式碼助手

  • 基於程式碼庫的上下文回答開發問題
  • API 文件查詢與程式碼範例生成
  • Bug 分析：檢索相似的歷史問題

研究與分析

  • 論文檢索與摘要
  • 市場研究報告生成
  • 專利分析與先前技術搜尋

多模態 RAG

  • 圖表理解：檢索相關圖表並分析
  • 影音內容：語音/影片轉文字後建索引
  • PDF 處理：解析表格、圖片與文字

對話式 RAG

  • 多輪對話中維護上下文
  • 追問機制：發現資訊不足時主動檢索
  • 個人化：根據使用者歷史偏好調整檢索
