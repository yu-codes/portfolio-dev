LLM 評估（LLM Evaluation）是系統化衡量大型語言模型品質與可靠性的關鍵流程。隨著 LLM 被廣泛應用於生產環境，如何客觀、全面地評估模型表現——從基礎能力到安全性、從離線測試到線上監控——已成為 AI 工程中不可或缺的一環。本文將全面探討 LLM 評估的方法論、指標體系、工具生態與最佳實踐。

核心概念

為什麼 LLM 評估特別困難

傳統機器學習模型（分類、回歸）有明確的正確答案，評估相對簡單。但 LLM 的生成式特性帶來獨特挑戰：

挑戰說明影響
開放式輸出同一問題可能有多種合理回答無法用簡單正確率衡量
多維度能力知識、推理、創意、安全等面向單一指標不足以描述
主觀性語言品質涉及人類偏好自動指標與人類判斷可能脫節
任務多樣性從翻譯到程式碼生成不存在通用評估方案
動態環境模型持續更新，知識會過時需持續監控而非一次性測試

評估的三個層次

  1. 能力評估（Capability Evaluation）：模型能做什麼？使用標準基準測試衡量知識、推理、語言能力。
  2. 品質評估（Quality Evaluation）：模型做得好不好？評估輸出的有用性、正確性、流暢性。
  3. 安全評估（Safety Evaluation）：模型是否可信？檢測幻覺、偏見、有害輸出、資料洩漏等。

評估指標分類

自動化指標：

  • 精確匹配指標：Accuracy、Exact Match、F1
  • 文本相似度指標：BLEU、ROUGE、BERTScore
  • 程式碼指標：pass@k、CodeBLEU
  • RAG 指標：Faithfulness、Context Relevance、Answer Relevance

基於模型的指標：

  • LLM-as-Judge：使用 GPT-4 等強模型評分
  • NLI-based：使用自然語言推理模型檢測事實性
  • Reward Model：使用 RLHF 訓練的獎勵模型評分

人工評估指標：

  • Likert Scale：1-5 分量表評分
  • A/B 比較：兩個回答的偏好選擇
  • ELO Rating：基於多輪配對比較的排名

架構總覽

離線評估流水線

典型的離線評估包含以下步驟：

定義評估目標
    ↓
[測試集準備] → 收集/建構測試案例（含 ground truth）
    ↓
[推理執行] → 模型在測試集上生成回答
    ↓
[指標計算] → 自動指標 + LLM Judge + 人工抽檢
    ↓
[結果分析] → 按類別細分、錯誤分析、信心區間
    ↓
[決策] → 上線 / 回退 / 優化

線上評估架構

生產環境中的持續監控：

使用者請求 → LLM 回答 → 品質監控
                            ├── 延遲與吞吐量
                            ├── 幻覺檢測（即時）
                            ├── 用戶反饋（👍👎）
                            ├── 安全過濾觸發率
                            └── A/B 測試指標

評估驅動的開發循環

構建 → 評估 → 分析 → 改進 → 重新評估
         ↑                        |
         └────────────────────────┘

此循環適用於：Prompt 優化、RAG 調整、微調策略選擇等。

LLM-as-Judge 的優缺點

優點：

  • 可大規模自動化，快速迭代
  • 評估維度靈活（可自定義標準）
  • 比傳統指標（BLEU、ROUGE）更貼近人類判斷

缺點：

  • 存在位置偏見（傾向選第一個回答）
  • 冗長偏見（偏好較長的回答）
  • 自我偏見（GPT-4 可能偏好 GPT-4 的回答）
  • 成本隨評估量增長

緩解策略：

  • 隨機化回答順序
  • 使用多個 Judge 模型取平均
  • 定期與人工評估校準

應用場景

模型選型與比較

在專案初期選擇合適模型：

  • 建立與業務相關的測試集（50-200 題）
  • 在候選模型上評估（GPT-4o、Claude、Gemini、開源模型）
  • 綜合考慮品質、延遲、成本三角

RAG 系統品質把關

RAG 系統的核心評估維度：

  • Context Relevance：檢索到的文件是否相關？
  • Faithfulness：回答是否忠於檢索內容？
  • Answer Relevance：回答是否真正回應了問題？
  • Groundedness：每個聲明是否有來源支持？

Prompt 工程迭代

使用評估驅動 Prompt 優化：

  • 定義期望行為的測試案例
  • 修改 Prompt 後自動回歸測試
  • 追蹤各版本 Prompt 的表現變化

微調效果驗證

在微調前後進行比較評估：

  • 確認微調後模型在目標任務上有提升
  • 檢查通用能力是否退化（Catastrophic Forgetting）
  • 驗證安全性未受影響

持續品質監控

生產環境中的品質守護：

  • 設定品質指標的警戒線
  • 自動偵測回答品質下降（Drift Detection）
  • 用戶反饋驅動的評估閉環

安全與合規審計

定期進行安全評估：

  • 紅隊測試（Prompt Injection、Jailbreak）
  • 偏見檢測（性別、種族、文化等）
  • 隱私洩漏測試（PII Extraction）
  • 有害內容生成測試
