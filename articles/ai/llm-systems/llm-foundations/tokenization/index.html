Tokenization 與詞彙表

Tokenization（分詞）是自然語言處理的第一步，決定了文本如何被轉換為模型可處理的數字序列。良好的 Tokenizer 設計直接影響模型的語言理解能力、訓練效率與多語言支援。本文將探討主流的子詞分詞方法及其設計考量。

核心概念

為什麼需要 Tokenization

語言模型的輸入必須是數字序列，Tokenization 負責建立文字與數字之間的映射：

"Hello, world!" → [15496, 11, 995, 0] → 模型處理 → [預測 token ID] → "..."

分詞粒度的權衡

粒度優點缺點
字元級（Character）詞彙表小、無 OOV序列極長、語義薄弱
詞級（Word）語義明確詞彙表巨大、OOV 問題嚴重
子詞級（Subword）平衡詞彙大小與序列長度需訓練分詞模型

現代 LLM 幾乎全部採用子詞分詞，在詞彙表大小（通常 32K-128K）與序列長度之間取得最佳平衡。

架構總覽

Byte Pair Encoding（BPE）

BPE 是最廣泛使用的子詞分詞演算法，由 GPT 系列、LLaMA、Mistral 等採用。

訓練流程：

  1. 初始化詞彙表為所有單一字元（或 byte）
  2. 統計訓練語料中所有相鄰 token 對的頻率
  3. 合併頻率最高的 token 對為新 token
  4. 重複步驟 2-3 直到達到目標詞彙表大小

編碼流程：

  1. 將文本拆分為字元
  2. 反覆應用學到的合併規則（按訓練時的順序）
  3. 得到最終的 token 序列

範例：

"lowest" → ["l", "o", "w", "e", "s", "t"]
         → ["lo", "w", "e", "s", "t"]    （合併 l+o）
         → ["low", "e", "s", "t"]         （合併 lo+w）
         → ["low", "est"]                  （合併 e+st）

WordPiece

WordPiece 由 Google 用於 BERT，與 BPE 類似但合併策略不同：

  • 選擇合併後能最大化語料似然度的 token 對
  • 使用 ## 前綴標記非詞首的子詞片段
  • 編碼時使用最長匹配（Greedy Longest-Match-First）

範例：

"playing" → ["play", "##ing"]
"unaffable" → ["un", "##aff", "##able"]

Unigram Language Model

由 SentencePiece 框架實現，T5 和部分模型使用：

  1. 從一個很大的候選詞彙表開始
  2. 計算每個 token 的 Unigram 語言模型機率
  3. 移除對整體似然度影響最小的 token
  4. 重複直到達到目標大小

特點：可產生多種合法的分詞結果（對正則化有幫助）

Byte-Level BPE

GPT-2 引入的改進，直接在 byte 層級操作：

  • 初始詞彙表為 256 個 byte 值
  • 無需預處理或 Unicode 正規化
  • 天然支援任何語言和字元
  • tiktoken（OpenAI）和 LLaMA tokenizer 均採用此方法

Tokenization 對中文的影響

中文沒有天然空格分隔，不同 Tokenizer 表現差異大：

  • GPT-2 Tokenizer：每個中文字可能被拆為 2-3 個 byte token（效率差）
  • LLaMA Tokenizer：中文字通常為獨立 token
  • ChatGLM：專門優化中文的詞彙表，中文壓縮率高

應用場景

LLM 預訓練

  • 訓練前需先在語料上訓練 Tokenizer
  • 詞彙表設計影響模型的多語言能力
  • tiktoken（OpenAI）支援高效的平行編碼

多語言模型

  • mBERT、XLM-R 使用共享詞彙表支援 100+ 語言
  • 需平衡各語言的 token 分配（Fertility）
  • 低資源語言可能被過度切分

程式碼模型

  • 程式碼需特殊處理（縮排、運算符、關鍵字）
  • StarCoder 使用專門的程式碼 Tokenizer
  • 空格和換行的編碼方式影響程式碼生成品質

長文本處理

  • Token 數量直接決定 context window 的有效利用
  • 高壓縮率的 Tokenizer 能在相同 token 限制下處理更多文字
  • Byte-level 方法對特殊字元和格式有更好的處理能力

推理優化

  • Token 數量直接影響推理成本（按 token 計費）
  • 選擇壓縮率高的 Tokenizer 可降低 API 調用成本
  • Speculative decoding 需要 Draft Model 使用相同 Tokenizer
